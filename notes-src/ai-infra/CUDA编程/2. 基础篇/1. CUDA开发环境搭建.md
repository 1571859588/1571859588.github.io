## 核心概念（新人友好讲解）

### 1. GPU 驱动与 CUDA Toolkit 的关系

很多初学者搞不清"驱动"和"CUDA Toolkit"到底是什么关系。打个比方：

- **驱动** = 手机的操作系统固件——它让硬件能正常工作
- **CUDA Toolkit** = 在这个操作系统上安装的开发工具包——提供编译器、库和调试器，让你能写出在 GPU 上跑的程序

### 1.1 版本兼容性规则

NVIDIA 采用**前向兼容**设计：新版驱动可以运行旧版 CUDA Toolkit 编译的程序，但反过来不行。核心规则：

- 每个 CUDA Toolkit 版本都有**最低驱动版本要求**
- 驱动版本 ≥ Toolkit 要求即可，不必精确匹配
- `nvidia-smi` 显示的 "CUDA Version" 是驱动**最高支持**的 Toolkit 版本，**不是**当前安装的 Toolkit 版本（新手必踩坑）

| CUDA Toolkit 版本 | 最低 Linux 驱动版本 | 最低 Windows 驱动版本 |
|-------------------|--------------------|--------------------|
| CUDA 12.6 | ≥ 560.28 | ≥ 560.70 |
| CUDA 12.4 | ≥ 550.54 | ≥ 551.61 |
| CUDA 12.2 | ≥ 535.54 | ≥ 536.25 |
| CUDA 11.8 | ≥ 520.61 | ≥ 520.06 |

💡 **提示**：始终优先升级驱动到最新稳定版，它既能向后兼容旧 Toolkit，也为后续升级 Toolkit 留出空间。

### 1.2 CUDA Toolkit 包含的组件

| 📦 组件 | 📝 作用 | 你会用到的场景 |
|---------|---------|--------------|
| nvcc | CUDA C/C++ 编译器 | 编译所有 .cu 文件 |
| cuBLAS | 线性代数运算库 | 矩阵乘法基准对比 |
| cuDNN | 深度学习原语库（需单独下载） | PyTorch/TensorFlow 后端 |
| cuFFT | 快速傅里叶变换库 | 信号处理 |
| cuRAND | 随机数生成库 | 随机初始化 |
| Nsight Systems | 系统级性能分析工具 | 找 GPU idle gap |
| Nsight Compute | Kernel 级性能分析工具 | 分析单个 kernel 瓶颈 |
| CUDA Runtime (libcudart) | 运行时 API 库 | 所有 CUDA 程序运行时依赖 |
| cuda-gdb | GPU 调试器 | 调试 kernel |

### 1.3 新人术语速查表

| 术语 | 全称 | 通俗解释 |
|------|------|----------|
| Host | - | CPU 端（主机），负责控制逻辑 |
| Device | - | GPU 端（设备），负责并行计算 |
| Kernel | - | 在 GPU 上执行的函数（核函数） |
| SM | Streaming Multiprocessor | GPU 的"计算核心"，一块 GPU 有多个 SM |
| Compute Capability | 计算能力 | GPU 架构版本号，如 8.0、8.9，决定支持的特性 |
| SASS | - | GPU 的最终机器码 |
| PTX | Parallel Thread Execution | NVIDIA 的中间指令集（类似 Java 字节码） |

### 步骤 1：验证已有环境（✅ 无需 sudo，你直接执行）

这是你最先要做的事——确认服务器上的 CUDA 真的能用。

```bash
# 检查驱动
nvidia-smi
```

**预期输出**（具体数字因你的 GPU 而异）：

```

|-----------------------------------------+------------------------+----------------------+
| GPU  Name                 Persistence-M | Bus-Id          Disp.A | Volatile Uncorr. ECC |
| Fan  Temp   Perf          Pwr:Usage/Cap |           Memory-Usage | GPU-Util  Compute M. |
|=========================================+========================+======================|
|   0  NVIDIA GeForce RTX 4090       Off | 00000000:01:00.0  Off |                  Off |
|  0%   35C    P8              12W / 450W |       1MiB / 24564MiB |      0%      Default |
+-----------------------------------------+------------------------+----------------------+
```

**关注三个信息**：
- **Driver Version**：当前驱动版本
- **CUDA Version**：驱动**最高支持**的 CUDA 版本（不是已安装的 Toolkit 版本！）
- **Memory-Usage**：显存使用情况

```bash
# 检查 CUDA Toolkit（编译器）
nvcc --version
```

**预期输出**：

```
nvcc: NVIDIA (R) Cuda compiler driver
Copyright (c) 2005-2024 NVIDIA Corporation
Built on ...
Cuda compilation tools, release 12.6, V12.6.20
Build cuda_12.6.r12.6/compiler.34431801_0
```

```bash
# 确认 nvcc 路径
which nvcc
```

**预期输出**：

```
/usr/local/cuda/bin/nvcc
```

📌 **关键判断**：如果上面三条命令都正常输出，说明你的环境完整可用，**直接跳到步骤 5（nvcc 编译）开始写代码**。只有当某条命令报 "command not found" 时，才需要关心下面的安装步骤。

### 步骤 2：安装 NVIDIA 驱动（⚠️ 需要 sudo，你跳过即可）

> 你的服务器已装好驱动，本节仅供了解。如果将来要给自己的机器装驱动，参考这里。

```bash
# 方法 1：使用 apt 包管理器（推荐）
sudo apt update
sudo apt install nvidia-driver-560

# 安装完成后重启
sudo reboot

# 验证安装
nvidia-smi
```

```bash
# 方法 2：使用官方 .run 安装包
# 先禁用 nouveau 开源驱动
sudo bash -c "echo 'blacklist nouveau' >> /etc/modprobe.d/blacklist.conf"
sudo update-initramfs -u
sudo reboot

# 下载并执行安装脚本
chmod +x NVIDIA-Linux-x86_64-560.35.03.run
sudo ./NVIDIA-Linux-x86_64-560.35.03.run
```

⚠️ **注意**：使用 `.run` 安装包前必须禁用 `nouveau` 驱动并切换到文本模式（tty），否则安装会失败。对于生产服务器，建议使用包管理器安装以便后续自动更新。

### 步骤 3：安装 CUDA Toolkit（⚠️ 需要 sudo / ✅ conda 无 sudo）

#### 3.1 系统级安装（⚠️ 需要 sudo，你跳过）

```bash
# 下载 CUDA 12.6 runfile
wget https://developer.download.nvidia.com/compute/cuda/12.6.0/local_installers/cuda_12.6.0_560.28.03_linux.run

# 执行安装（注意：取消勾选 Driver，避免覆盖已有驱动）
sudo sh cuda_12.6.0_560.28.03_linux.run
```

安装时选择界面中：
- ❌ 取消 Driver（已经单独安装过）
- ✅ 勾选 CUDA Toolkit
- ✅ 勾选 CUDA Samples（学习用，可选）

💡 **提示**：如果你用 `apt` 安装，可以用 `sudo apt install cuda-toolkit-12-6`，这样不会捆绑驱动安装。

#### 3.2 conda 安装（✅ 无需 sudo，多版本管理推荐）

这是**无 sudo 权限时最实用的方案**——在 conda 环境里装独立的 CUDA Toolkit，不碰系统目录：

```bash
# 仅安装 CUDA 运行时和编译器（不含驱动）
conda install -c nvidia cuda-toolkit=12.6
```

这种方式适合在已有系统驱动的前提下，为特定 conda 环境配置独立的 CUDA 版本。例如项目 A 用 CUDA 11.8，项目 B 用 CUDA 12.6，可以分别建两个 conda 环境互不干扰。

📌 **你的场景**：服务器系统级 CUDA 已装好，你直接用系统的即可。只有在需要测试不同 CUDA 版本编译行为时，才用 conda 装额外版本。

### 步骤 4：环境变量配置（✅ 无需 sudo，你直接执行）

安装完成后，需要将 CUDA 的路径加入系统环境变量。这一步改的是你自己的 `~/.bashrc`，不需要 sudo。

在 `~/.bashrc` 或 `~/.zshrc` 中添加：

```bash
# CUDA 路径配置
export CUDA_HOME=/usr/local/cuda-12.6
export PATH=$CUDA_HOME/bin:$PATH
export LD_LIBRARY_PATH=$CUDA_HOME/lib64:$LD_LIBRARY_PATH
```

```bash
# 使配置生效
source ~/.bashrc

# 验证
nvcc --version
which nvcc
```

**环境变量参数解释**（新人必看）：

| 变量 | 作用 | 不配置会怎样 |
|------|------|------------|
| `CUDA_HOME` | 指向 CUDA 安装根目录 | 部分工具找不到 CUDA |
| `PATH` | 系统查找可执行文件的路径列表 | shell 找不到 `nvcc` 命令 |
| `LD_LIBRARY_PATH` | 动态链接库查找路径 | 运行时找不到 `libcudart.so` |

💡 **提示**：`/usr/local/cuda` 通常是个软链接，指向 `/usr/local/cuda-12.6` 这样的具体版本目录。用软链接的好处是切换版本时只改链接，不用改环境变量。

### 步骤 5：nvcc 编译工具链（✅ 无需 sudo）

nvcc 是 CUDA 程序的编译器驱动程序。它不是一个独立的编译器，而是一个**编译协调器**：将 `.cu` 文件中的设备代码和主机代码分离，分别用 NVIDIA PTX 编译器和系统 C++ 编译器处理，最终链接成可执行文件。

#### 5.1 编译流程

```
.cu 源文件
   │
   ▼
nvcc 前端
   │
   ├──► 设备代码 (GPU) ──► ptxas ──► cubin ──┐
   │                                          ├──► 链接器 ──► 可执行文件
   └──► 主机代码 (CPU) ──► g++/MSVC ──► .o ──┘
```

#### 5.2 常用编译选项（新人参数速查）

```bash
# 基本编译
nvcc hello.cu -o hello

# 指定 GPU 架构（为 RTX 4090 的 sm_89 编译）
nvcc -arch=sm_89 hello.cu -o hello

# 生成多架构兼容的 fatbinary
nvcc -gencode arch=compute_80,code=sm_80 \
     -gencode arch=compute_89,code=sm_89 \
     hello.cu -o hello

# 开启调试信息（用于 cuda-gdb）
nvcc -g -G hello.cu -o hello_debug

# 开启优化
nvcc -O3 hello.cu -o hello_opt

# 显示编译过程的详细信息
nvcc --verbose hello.cu -o hello
```

#### 5.3 关键编译选项说明（新人必看）

| 选项 | 作用 | 什么时候用 |
|------|------|----------|
| `-arch=sm_XX` | 指定目标 GPU 架构的计算能力 | 生产环境指定目标 GPU |
| `-gencode` | 生成多架构代码，适配不同 GPU | 发布给多种 GPU 用户的库 |
| `-g` | 生成主机代码调试信息 | 用 gdb 调试 CPU 代码 |
| `-G` | 生成设备代码调试信息 | 用 cuda-gdb 调试 kernel |
| `-lineinfo` | 在优化代码中保留行号信息 | 性能分析（ncu）时定位代码行 |
| `-Xcompiler` | 将后续选项传递给主机编译器 | 传 C++ 标准等选项给 g++ |
| `-std=c++17` | 指定 C++ 标准版本 | 用 C++17 特性时 |
| `-maxrregcount=N` | 限制每个线程使用的寄存器数量 | 优化 occupancy 时 |
| `--ptxas-options=-v` | 显示每个 kernel 的寄存器和共享内存用量 | 分析性能瓶颈必用 |

#### 5.4 GPU 架构代号速查（新人必看）

| 架构代号 | 代表 GPU | 计算能力 | 发布年份 |
|---------|---------|---------|---------|
| sm_70 | V100 | 7.0 | 2017 |
| sm_75 | T4, RTX 2080 | 7.5 | 2018 |
| sm_80 | A100 | 8.0 | 2020 |
| sm_86 | RTX 3090 | 8.6 | 2020 |
| sm_89 | RTX 4090, L40 | 8.9 | 2022 |
| sm_90 | H100 | 9.0 | 2022 |

📌 **如何查你的 GPU 架构**：

```bash
# 方法 1：nvidia-smi 看 GPU 型号，再对照上表
nvidia-smi --query-gpu=name,compute_cap --format=csv
```

**预期输出**：

```
name, compute_cap
NVIDIA GeForce RTX 4090, 8.9
```

```bash

```

**预期输出**：

```

```

### 步骤 6：CMake 集成 CUDA（✅ 无需 sudo）

在实际项目中，很少直接手动调用 `nvcc`，而是使用 CMake 作为构建系统来管理编译过程。CMake 从 3.8 版本开始原生支持 CUDA 作为一等语言。

#### 6.1 最小 CMakeLists.txt

```cmake
cmake_minimum_required(VERSION 3.18)
project(MyCUDAProject LANGUAGES CXX CUDA)

# 设置 C++ 和 CUDA 标准
set(CMAKE_CXX_STANDARD 17)
set(CMAKE_CUDA_STANDARD 17)

# 指定目标 GPU 架构
set(CMAKE_CUDA_ARCHITECTURES 80 89 90)

# 添加可执行文件
add_executable(main src/main.cu)

# 链接 CUDA 运行时库
target_link_libraries(main PRIVATE CUDA::cudart)
```

#### 6.2 构建和运行

```bash
mkdir build && cd build
cmake .. -DCMAKE_BUILD_TYPE=Release
make -j$(nproc)
./main
```

💡 **提示**：`CMAKE_CUDA_ARCHITECTURES` 变量代替在代码中硬编码 `-arch`，切换目标 GPU 时只需改一处配置。`make -j$(nproc)` 中 `$(nproc)` 表示用满所有 CPU 核心加速编译。

### 步骤 7：Hello World：第一个 CUDA 程序（✅ 无需 sudo）

现在万事俱备，让我们写一个完整的 Hello World 来验证整个工具链。

#### 7.1 源代码 `hello_cuda.cu`

```cpp
#include <cstdio>

// __global__ 修饰符表明这是一个 GPU 上执行的 kernel 函数
__global__ void helloKernel() {
    // threadIdx.x 是当前线程在 Block 内的编号
    printf("Hello from GPU thread %d in block %d!\n",
           threadIdx.x, blockIdx.x);
}

int main() {
    printf("Launching kernel...\n");

    // <<<2, 4>>> 表示启动 2 个 Block，每个 Block 有 4 个线程
    helloKernel<<<2, 4>>>();

    // 等待 GPU 上所有操作完成
    cudaDeviceSynchronize();

    printf("Done!\n");
    return 0;
}
```

#### 7.2 编译并运行

```bash
nvcc hello_cuda.cu -o hello_cuda
./hello_cuda
```

**预期输出**：

```
Launching kernel...
Hello from GPU thread 0 in block 0!
Hello from GPU thread 1 in block 0!
Hello from GPU thread 2 in block 0!
Hello from GPU thread 3 in block 0!
Hello from GPU thread 0 in block 1!
Hello from GPU thread 1 in block 1!
Hello from GPU thread 2 in block 1!
Hello from GPU thread 3 in block 1!
Done!
```

📌 **关键点**：线程的实际执行顺序并不确定——两个 Block 的输出可能交替出现。并行程序的执行顺序由 GPU 硬件调度器决定。

#### 7.3 代码解析（新人参数解释）

| 要素 | 含义 | 详细说明 |
|------|------|----------|
| `__global__` | 标记函数为 kernel | 由 CPU 调用、GPU 执行，返回类型必须是 void |
| `<<<2, 4>>>` | 启动配置 | 第一个数 = Block 数量(Grid)，第二个数 = 每 Block 线程数 |
| `threadIdx.x` | 线程在 Block 内的索引 | 从 0 开始，这里是 0~3 |
| `blockIdx.x` | Block 在 Grid 内的索引 | 从 0 开始，这里是 0~1 |
| `cudaDeviceSynchronize()` | CPU 等待 GPU 完成 | kernel 启动是异步的，不等待则 printf 可能还没执行就退出了 |

> 🔧 **为什么需要 `cudaDeviceSynchronize()`**：CUDA kernel 启动是**异步**的——CPU 下完命令立刻继续执行下一行，不等 GPU 干完。如果不加这行，`main` 里的 `printf("Done!")` 会先打印，然后程序退出，GPU 上的 `printf` 可能根本来不及输出。

### 步骤 8：多版本 CUDA 管理（✅ 用户态方案 / ⚠️ 系统级方案）

实际开发中，你可能需要同时维护多个 CUDA 版本（比如项目 A 用 CUDA 11.8，项目 B 用 CUDA 12.6）。NVIDIA 的设计允许多版本共存。

#### 8.1 多版本安装的目录结构

每个版本安装到独立目录：

```
/usr/local/cuda-11.8/
/usr/local/cuda-12.4/
/usr/local/cuda-12.6/
```

#### 8.2 系统级符号链接切换（⚠️ 需要 sudo，你用不了）

```bash
# 切换到 CUDA 12.6
sudo rm -f /usr/local/cuda
sudo ln -s /usr/local/cuda-12.6 /usr/local/cuda

# 切换到 CUDA 11.8
sudo rm -f /usr/local/cuda
sudo ln -s /usr/local/cuda-11.8 /usr/local/cuda
```

环境变量统一指向 `/usr/local/cuda`（软链接）：

```bash
export CUDA_HOME=/usr/local/cuda
export PATH=$CUDA_HOME/bin:$PATH
export LD_LIBRARY_PATH=$CUDA_HOME/lib64:$LD_LIBRARY_PATH
```

#### 8.3 用户态切换脚本（✅ 无需 sudo，你用这个）

既然不能改 `/usr/local/cuda` 软链接，你可以写一个只改自己环境变量的脚本——效果一样，且不需要 sudo：

```bash
#!/bin/bash
# 保存为 ~/bin/switch-cuda.sh
# 用法：source switch-cuda.sh 12.6

CUDA_VERSION=$1
if [ -d "/usr/local/cuda-$CUDA_VERSION" ]; then
    export CUDA_HOME=/usr/local/cuda-$CUDA_VERSION
    export PATH=$CUDA_HOME/bin:$PATH
    export LD_LIBRARY_PATH=$CUDA_HOME/lib64:$LD_LIBRARY_PATH
    echo "Switched to CUDA $CUDA_VERSION"
    nvcc --version
else
    echo "CUDA $CUDA_VERSION not found!"
fi
```

使用方式（必须用 `source` 而不是直接执行，否则环境变量改不到当前 shell）：

```bash
source ~/bin/switch-cuda.sh 12.6
```

**预期输出**：

```
Switched to CUDA 12.6
nvcc: NVIDIA (R) Cuda compiler driver
Copyright (c) 2005-2024 NVIDIA Corporation
Built on ...
Cuda compilation tools, release 12.6, V12.6.20
```

#### 8.4 使用 module 系统（✅ 无需 sudo，集群环境常用）

在 HPC 集群上通常使用 Environment Modules，这是最优雅的多版本管理方式：

```bash
module avail cuda          # 查看可用的 CUDA 版本
module load cuda/12.6      # 加载某个版本
module list                # 查看当前加载的模块
module switch cuda/12.6 cuda/11.8  # 切换版本
```

📌 **你的场景判断**：如果你的服务器有 `module` 命令，优先用它；没有的话用上面的 `source` 脚本。

### 步骤 9：编译运行官方 deviceQuery（✅ 无需 sudo，可选验证）


## 面试回答（可直接口述的版本）

**问：请描述一下 CUDA 开发环境的组成和搭建流程。**

答：CUDA 开发环境由三层组成：最底层是 NVIDIA 驱动，让 GPU 硬件能被系统识别；中间层是 CUDA Toolkit，提供 nvcc 编译器、运行时库和性能分析工具；上层是环境变量配置，让 shell 能找到这些工具。搭建流程是：先装驱动确保 `nvidia-smi` 正常，再装 Toolkit，注意安装时取消勾选 Driver 避免覆盖已有驱动，然后配置 `CUDA_HOME`、`PATH`、`LD_LIBRARY_PATH` 三个环境变量，最后编译运行 deviceQuery 验证。版本兼容上要注意：驱动向下兼容 Toolkit，`nvidia-smi` 显示的是驱动最高支持的 CUDA 版本，不是实际安装的 Toolkit 版本。

## 深入追问（可能被追问的点）

- **Q1: `nvidia-smi` 显示的 CUDA 版本和 `nvcc -V` 显示的不一样怎么办？**
  A: 这是正常的。`nvidia-smi` 显示驱动**最高支持**的 CUDA 版本，`nvcc -V` 显示**实际安装**的 Toolkit 版本。只要 Toolkit 版本 ≤ 驱动支持的版本就能用。

- **Q2: 编译时不指定 `-arch` 会怎样？**
  A: nvcc 会用默认架构（通常是较旧的 compute_52），生成的代码能在大多数 GPU 上跑但用不到新特性。生产环境应显式指定目标架构。

- **Q3: `-arch=sm_89` 和 `-gencode arch=compute_89,code=sm_89` 有什么区别？**
  A: `-arch` 是简写形式，等价于 `-gencode arch=compute_89,code=sm_89`。`-gencode` 可以多次指定生成多架构 fatbinary，`-arch` 只能指定一个。

- **Q4: 为什么 kernel 启动是异步的？**
  A: 为了让 CPU 和 GPU 并行工作——CPU 下完 kernel 启动命令后可以继续准备下一个任务的数据，不用等 GPU 执行完。这叫做"异步执行模型"，是 GPU 高性能的关键之一。

- **Q5: 没有 sudo 怎么装 CUDA Toolkit？**
  A: 三种方式：① `conda install -c nvidia cuda-toolkit=12.6` 装到自己环境；② 用 runfile 的 `--toolkit --installprefix=$HOME/cuda-12.6` 装到家目录；③ 用 `module load` 加载集群已装好的版本。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| 驱动 vs Toolkit | 驱动管硬件，Toolkit 是开发工具 | 驱动=手机固件，Toolkit=App |
| `nvidia-smi` CUDA 版本 vs `nvcc -V` 版本 | 前者是驱动支持上限，后者是实际装的 | smi=能力上限，nvcc=实际版本 |
| `-G` vs `-g` | `-G` 调试 GPU 代码，`-g` 调试 CPU 代码 | 大写=Device(GPU)，小写=host(CPU) |
| `-arch` vs `-gencode` | `-arch` 单架构简写，`-gencode` 多架构 | arch 简单，gencode 灵活 |
| conda cuda-toolkit vs 系统 cuda-toolkit | conda 装在环境里隔离，系统装全局共享 | conda=项目级，系统=全局级 |
| `sudo ln -s` vs `source switch-cuda.sh` | 前者改系统软链接需 root，后者改环境变量不需 | 无 sudo 用 source 脚本 |

## 常见报错速查表

| ❌ 报错 | 原因 | ✅ 解决 |
|--------|------|--------|
| `nvidia-smi: command not found` | 驱动未安装或未加入 PATH | 检查驱动是否安装（`lspci \| grep NVIDIA`） |
| `nvidia-smi: driver/library version mismatch` | 驱动被更新但未重启 | `sudo reboot`（你需联系管理员） |
| `nvcc: command not found` | Toolkit 未安装或环境变量未配置 | 检查 `which nvcc`、`echo $PATH` |
| 编译报 `unsupported gpu architecture sm_XX` | nvcc 版本太旧不支持你的 GPU | 升级 Toolkit 到支持目标架构的版本 |
| 链接报 `cannot find -lcudart` | `LD_LIBRARY_PATH` 未包含 CUDA lib64 | `export LD_LIBRARY_PATH=/usr/local/cuda/lib64:$LD_LIBRARY_PATH` |
| CMake 报 `No CUDA toolset found` | CMake 找不到 nvcc | 设置 `-DCMAKE_CUDA_COMPILER=/usr/local/cuda/bin/nvcc` |
| 运行报 `CUDA error: no CUDA-capable device` | 驱动问题或 GPU 未被识别 | `lspci \| grep NVIDIA` 检查硬件 |
| 运行报 `CUDA error: insufficient driver` | Toolkit 版本高于驱动支持 | 降级 Toolkit 或升级驱动 |
