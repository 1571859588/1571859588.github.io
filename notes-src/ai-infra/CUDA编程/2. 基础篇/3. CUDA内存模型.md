## 一句话结论

GPU 的算力远大于带宽——绝大多数 Kernel 瓶颈是内存访问。CUDA 内存分五层（寄存器/共享内存/全局内存/常量内存/纹理内存+统一内存），核心优化原则是：**把数据放在离计算最近的地方，全局内存必须合并访问，Block 内协作用共享内存（Tiling）**。

## 核心概念（新人友好讲解）

### 1. 为什么内存模型如此重要

GPU 的计算能力极强——一块 A100 的峰值算力为 312 TFLOPS（FP16），但显存带宽只有约 2 TB/s。做道算术：一次浮点乘加需要读 2 个 float（8 Bytes）产生 2 FLOP，那么 2 TB/s 带宽最多喂饱 500 GFLOPS——**不到峰值算力的 0.2%**。

这意味着：**绝大多数 Kernel 的性能瓶颈不是计算，而是内存访问**。理解 CUDA 多层内存，学会把数据放在"离计算最近的地方"，是优化 GPU 程序的第一课。

### 2. 内存层次总览

![CUDA 内存模型层次](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/cuda内存模型.png)

GPU 的内存层次像城市的交通系统——寄存器是办公桌上的文件（最快但最小），共享内存是同一办公室的文件柜（快速且可共享），全局内存是城市外的大仓库（容量大但来回需要时间）。

#### 2.1 内存层次架构图（ASCII 版）

```
                    GPU 芯片
        ┌─────────────────────────────────────┐
        │           每个 SM (Streaming         │
        │           Multiprocessor) 内部       │
        │  ┌─────────────────────────────┐    │
        │  │  寄存器 (Registers)          │ ◄── 单线程私有，最快
        │  │  ~256KB/SM                  │    │
        │  ├─────────────────────────────┤    │
        │  │  共享内存 (Shared Memory)    │ ◄── Block 内共享
        │  │  48~228KB/SM                │    │
        │  ├─────────────────────────────┤    │
        │  │  L1 Cache                   │ ◄── 自动管理
        │  │  128~256KB/SM               │    │
        │  └─────────────────────────────┘    │
        └─────────────────────────────────────┘
                          │
                          ▼
        ┌─────────────────────────────────────┐
        │  L2 Cache (所有 SM 共享)             │ ◄── 自动管理
        │  6~50MB                              │
        └─────────────────────────────────────┘
                          │
                          ▼
        ┌─────────────────────────────────────┐
        │  全局内存 (HBM, 芯片外)              │ ◄── 所有线程可见
        │  16~80GB                             │
        └─────────────────────────────────────┘
```

#### 2.2 各级内存对比（新人必背）

| 📊 内存类型 | 位置 | 容量 | 延迟 | 带宽 | 作用域 | 生命周期 |
|------------|------|------|------|------|--------|---------|
| 寄存器 | SM 芯片上 | ~256KB/SM | 1 cycle | 最高 | 单线程 | 线程生命周期 |
| 共享内存 | SM 芯片上 | 48~228KB/SM | ~20 cycles | ~19 TB/s | Block 内 | Block 生命周期 |
| L1 Cache | SM 芯片上 | 128~256KB/SM | ~30 cycles | ~19 TB/s | 自动 | 自动管理 |
| L2 Cache | GPU 芯片上 | 6~50MB | ~200 cycles | ~5 TB/s | 所有 SM | 自动管理 |
| 全局内存 (HBM) | 芯片外 | 16~80GB | ~400 cycles | 1~3.4 TB/s | 全局 | 应用生命周期 |
| 常量内存 | 芯片外+缓存 | 64KB | 1~400 cycles | 有缓存时极高 | 全局只读 | 应用生命周期 |

📌 **记忆技巧**：越靠近计算单元的存储越快、越小、越贵。寄存器 > 共享内存 > L1 > L2 > 全局内存。

### 3. 全局内存（Global Memory / HBM）

![合并访存与非合并访存对比](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/合并访存与非合并访存对比.png)

全局内存是 GPU 的"主存"，对应物理上的 HBM（高带宽内存）。容量最大但延迟最高，是 CPU-GPU 数据交换的枢纽。

#### 3.1 基本操作（新人 API 速查）

```cpp
// 分配设备内存
float* d_data;
size_t size = N * sizeof(float);
cudaMalloc(&d_data, size);

// Host → Device 传输
cudaMemcpy(d_data, h_data, size, cudaMemcpyHostToDevice);

// Device → Host 传输
cudaMemcpy(h_result, d_data, size, cudaMemcpyDeviceToHost);

// 释放设备内存
cudaFree(d_data);
```

**API 参数解释**（新人必看）：

| API | 参数 | 说明 |
|-----|------|------|
| `cudaMalloc(&ptr, size)` | `&ptr` 接收设备指针；`size` 字节数 | 类似 `malloc`，但分配在显存 |
| `cudaMemcpy(dst, src, size, kind)` | `dst`/`src` 目标/源指针；`size` 字节数；`kind` 方向 | `kind` 取值见下表 |
| `cudaFree(ptr)` | `ptr` 设备指针 | 释放显存 |

**cudaMemcpy 的 `kind` 参数**（新人易混淆）：

| kind 值 | 含义 |
|---------|------|
| `cudaMemcpyHostToDevice` | CPU → GPU |
| `cudaMemcpyDeviceToHost` | GPU → CPU |
| `cudaMemcpyDeviceToDevice` | GPU → GPU（同卡或跨卡） |

#### 3.2 合并访问（Coalesced Access）—— 最重要的优化原则

全局内存的访问效率取决于**一个 Warp 内 32 个线程的访问模式是否连续对齐**。GPU 以 32B/64B/128B 的粒度从全局内存取数据（一次"事务"），如果 Warp 内线程访问的地址连续，一次事务就能满足所有线程——这就是**合并访问**。

```cpp
// ✅ 合并访问：连续线程访问连续地址
// Thread 0 → data[0], Thread 1 → data[1], ...
float val = data[threadIdx.x + blockIdx.x * blockDim.x];

// ❌ 跨步访问：效率低
// Thread 0 → data[0], Thread 1 → data[stride], ...
float val = data[(threadIdx.x + blockIdx.x * blockDim.x) * stride];
```

**合并访问图解**：

```
合并访问 ✅：Warp 内 32 个线程访问连续地址
  T0   T1   T2   T3  ...  T31
  ▼    ▼    ▼    ▼         ▼
 [0]  [1]  [2]  [3]  ... [31]   ← 一次 128B 事务搞定

非合并访问 ❌：Warp 内 32 个线程访问分散地址
  T0   T1   T2   T3  ...  T31
  ▼    ▼    ▼    ▼         ▼
 [0] [128][256][384] ... [3968]  ← 需要 32 次事务，带宽利用率 3%
```

#### 3.3 访问模式对性能的影响

| 访问模式 | 事务数（Warp 访问 32 个 float） | 有效带宽利用率 |
|---------|-------------------------------|--------------|
| 完美合并（连续对齐） | 1 次 128B 事务 | 100% |
| 连续但未对齐 | 2 次 128B 事务 | 50% |
| 随机散列访问 | 最多 32 次 32B 事务 | ~3% |

💡 **提示**：设计数据结构时，优先选择 **Structure of Arrays (SoA)** 而非 **Array of Structures (AoS)**，因为 SoA 更容易实现合并访问。

#### 3.4 SoA vs AoS 示例（新人必看）

```cpp
// ❌ AoS（Array of Structures）— 相邻线程访问不连续
struct Particle {
    float x, y, z;
    float vx, vy, vz;
};
Particle particles[N];
// Thread i 访问 particles[i].x → 间隔 sizeof(Particle) = 24 bytes

// ✅ SoA（Structure of Arrays）— 相邻线程访问连续
struct Particles {
    float x[N], y[N], z[N];
    float vx[N], vy[N], vz[N];
};
Particles p;
// Thread i 访问 p.x[i] → 间隔 sizeof(float) = 4 bytes，完美合并
```

### 4. 共享内存（Shared Memory）

共享内存位于 SM 芯片上，48~228KB/SM，是 Block 内所有线程共享的高速暂存。把它想象成**一个团队的白板**——团队成员都可以快速读写，但其他团队看不到。

#### 4.1 声明与使用

**静态声明**（编译时确定大小）：

```cpp
__global__ void sharedMemDemo(float* input, float* output, int N) {
    // 静态声明：编译时确定大小
    __shared__ float tile[256];

    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < N) {
        // 将全局内存数据加载到共享内存
        tile[threadIdx.x] = input[idx];

        // 同步：确保所有线程都完成加载
        __syncthreads();

        // 现在可以安全地访问 tile 中其他线程加载的数据
        int partner = blockDim.x - 1 - threadIdx.x;
        output[idx] = tile[partner];
    }
}
```

**动态声明**（运行时确定大小）：

```cpp
// 动态声明：运行时确定大小
__global__ void dynamicSharedMem(float* data, int N) {
    extern __shared__ float sharedData[];  // 大小在启动时指定

    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < N) {
        sharedData[threadIdx.x] = data[idx];
    }
    __syncthreads();
    // ...
}

// 启动时通过第三个参数指定动态共享内存大小
int sharedMemSize = blockSize * sizeof(float);
dynamicSharedMem<<<grid, block, sharedMemSize>>>(data, N);
```

**参数解释**（新人必看）：

| 关键字 | 含义 |
|--------|------|
| `__shared__` | 声明共享内存变量，位于 SM 芯片上 |
| `extern __shared__` | 动态共享内存，大小在 kernel 启动时指定 |
| `__syncthreads()` | Block 内线程同步屏障，所有线程到达后继续 |

> 🔧 **为什么必须 `__syncthreads()`**：共享内存写入是异步的，如果一个线程还没写完，另一个线程就读，会读到旧值。`__syncthreads()` 确保所有线程都完成写入后再继续。

#### 4.2 Bank Conflict（新人重点理解）

共享内存被划分为 32 个 **Bank**（对应 32 个 Warp 线程），每个 Bank 宽度 4 bytes。当一个 Warp 中多个线程同时访问同一个 Bank 的不同地址时，访问必须串行化——这就是 **Bank Conflict**。

```
Bank 编号 = (地址 / 4 bytes) % 32
```

```
Bank:    0   1   2   3   ...  31
       ┌───┬───┬───┬───┬─────┬───┐
地址:  │ 0 │ 4 │ 8 │12 │ ... │124│  ← 第 0 行
       ├───┼───┼───┼───┼─────┼───┤
       │128│132│136│140│ ... │252│  ← 第 1 行
       └───┴───┴───┴───┴─────┴───┘
```

```cpp
// ✅ 无 Bank Conflict：每个线程访问不同 Bank
float val = shared[threadIdx.x];         // 线程 i → Bank i

// ❌ 2-way Bank Conflict：步长为 2，线程 0 和 16 冲突
float val = shared[threadIdx.x * 2];     // 线程 0 → Bank 0, 线程 16 → Bank 0

// ✅ 广播无冲突：所有线程读同一地址
float val = shared[0];                   // 广播，无冲突
```

#### 4.3 避免 Bank Conflict 的技巧（Padding）

对于矩阵转置等场景，经典方法是添加 padding：

```cpp
// 原始：2D 共享内存，列访问时会有 Bank Conflict
__shared__ float tile[32][32];

// 优化：加 1 列 padding，错开 Bank 对齐
__shared__ float tile[32][32 + 1];  // 33 列
```

#### 4.4 共享内存配置

现代 GPU（sm_80+）的 L1 Cache 和共享内存**共享同一块物理 SRAM**，可以通过 API 配置比例：

```cpp
// 优先分配更多共享内存（最大 164KB on A100）
cudaFuncSetAttribute(myKernel,
    cudaFuncAttributeMaxDynamicSharedMemorySize, 164 * 1024);

// 设置 L1/Shared Memory 偏好
cudaFuncSetCacheConfig(myKernel, cudaFuncCachePreferShared);
```

### 5. 寄存器与局部内存

#### 5.1 寄存器

寄存器是每个线程私有的最快存储。Kernel 中的局部变量默认存放在寄存器中：

```cpp
__global__ void compute(float* data, int N) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;  // 存在寄存器
    float temp = 0.0f;                                  // 存在寄存器
    float acc = 0.0f;                                   // 存在寄存器

    for (int i = 0; i < 10; i++) {
        temp = data[idx + i * N];
        acc += temp * temp;
    }
    data[idx] = acc;
}
```

#### 5.2 寄存器溢出（Register Spilling）

每个 SM 的寄存器总量有限（A100 为 65536 个 32-bit 寄存器/SM）。当一个线程使用的寄存器超过 SM 能分配给它的数量时，多出的变量会"溢出"到**局部内存**（实际是全局内存的一块区域，有 L1/L2 缓存加速但延迟仍然高）。

```bash
# ✅ 无需 sudo：查看 Kernel 的寄存器用量
nvcc --ptxas-options=-v mykernel.cu
# 预期输出: ptxas info: Used 48 registers, 0 bytes smem

# 限制最大寄存器数（可能导致溢出但提高占用率）
nvcc --maxrregcount=32 mykernel.cu
```

**预期输出**：

```
ptxas info    : Used 48 registers, 0 bytes smem, 368 bytes cmem[0]
```

#### 5.3 寄存器使用建议

| ✅ 推荐做法 | ❌ 不推荐做法 |
|-------------|--------------|
| 复用临时变量 | 声明大量独立变量 |
| 循环展开适度（`#pragma unroll`） | 展开因子过大导致寄存器爆炸 |
| 用 `--ptxas-options=-v` 监控 | 盲目追求零溢出 |

⚠️ **注意**：寄存器使用是**权衡**——用更多寄存器可以减少内存访问，但也会降低 SM 上能同时驻留的 Warp 数量（降低占用率）。

### 6. 常量内存（Constant Memory）

常量内存是一块 64KB 的**只读**全局内存区域，配有专用常量缓存。当 Warp 内所有线程读取同一地址时，一次缓存读取就能广播给 32 个线程。

#### 6.1 声明与使用

```cpp
// 在全局作用域声明常量内存
__constant__ float coefficients[256];

// 主机端拷贝数据到常量内存
float h_coeff[256];
// ... 填充 h_coeff ...
cudaMemcpyToSymbol(coefficients, h_coeff, sizeof(h_coeff));

// Kernel 中直接读取（像全局变量一样）
__global__ void applyFilter(float* data, float* output, int N) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;
    if (idx < N) {
        float result = 0.0f;
        for (int i = 0; i < 256; i++) {
            result += data[idx + i] * coefficients[i];  // 所有线程读同一个 coefficients[i]
        }
        output[idx] = result;
    }
}
```

**API 解释**（新人必看）：

| 关键字/API | 含义 |
|-----------|------|
| `__constant__` | 声明常量内存变量，全局只读，64KB 上限 |
| `cudaMemcpyToSymbol` | 从主机拷贝数据到设备端的全局/常量符号 |

#### 6.2 适用场景

| ✅ 适合常量内存 | ❌ 不适合常量内存 |
|----------------|-----------------|
| 所有线程读相同值（卷积核、系数表） | 每个线程读不同地址 |
| 数据量 ≤ 64KB | 数据量超过 64KB |
| 数据在 Kernel 执行期间不变 | 数据需要被 GPU 修改 |

📌 **关键点**：如果 Warp 内不同线程访问常量内存的不同地址，访问会被串行化为 32 次读取，性能反而比全局内存更差。

### 7. 纹理内存（Texture Memory）

纹理内存通过专用纹理缓存访问全局内存，针对**二维空间局部性**优化。深度学习时代使用较少，但图像处理和插值场景仍有价值。

#### 7.1 特点

- 针对 2D 空间局部性优化的缓存策略
- 支持硬件插值（线性/双线性）
- 支持自动边界处理（Clamp/Wrap）
- 对不规则访问模式比全局内存表现更好

#### 7.2 现代替代

CUDA 12+ 的 L1/L2 缓存已覆盖大部分纹理内存场景。新代码建议：

- 规则访问 → 直接用全局内存 + L1 缓存
- 只读数据 → 用 `__ldg()` 内置函数走只读缓存路径
- 需要硬件插值 → 仍用纹理

```cpp
// __ldg() 走只读缓存路径，无需配置纹理对象
float val = __ldg(&input[idx]);
```

### 8. 统一内存（Unified Memory）

统一内存（CUDA 6.0 引入）提供**单一地址空间**，CPU 和 GPU 都可以通过相同指针访问数据，系统自动在两者之间迁移页面。

#### 8.1 基本用法

```cpp
// 分配统一内存
float* data;
cudaMallocManaged(&data, N * sizeof(float));

// CPU 端初始化——直接用指针
for (int i = 0; i < N; i++) {
    data[i] = (float)i;
}

// GPU 端计算——同一个指针
myKernel<<<grid, block>>>(data, N);
cudaDeviceSynchronize();

// CPU 端读取结果——还是同一个指针
printf("Result: %f\n", data[0]);

// 释放
cudaFree(data);
```

#### 8.2 页面迁移机制（新人理解重点）

统一内存的"魔法"背后是操作系统级别的**按需页面迁移**：

1. CPU 首次写入时，页面驻留在系统内存
2. GPU Kernel 访问时触发缺页中断，驱动将页面迁移到显存
3. Kernel 执行完后 CPU 再次访问，页面迁回系统内存

#### 8.3 性能优化：预取

自动按需迁移有延迟开销，可以通过预取提前触发迁移：

```cpp
// 在 Kernel 启动前，预取数据到 GPU
int device;
cudaGetDevice(&device);
cudaMemPrefetchAsync(data, N * sizeof(float), device);

myKernel<<<grid, block>>>(data, N);
cudaDeviceSynchronize();

// Kernel 完成后，预取结果到 CPU
cudaMemPrefetchAsync(data, N * sizeof(float), cudaCpuDeviceId);
```

#### 8.4 适用场景分析

| ✅ 适合统一内存 | ❌ 不适合统一内存 |
|----------------|-----------------|
| 原型开发和快速迭代 | 极致性能要求的生产代码 |
| CPU/GPU 交替访问的复杂数据结构 | 大块数据的单向传输 |
| 链表、树等指针结构 | 简单的输入→计算→输出模式 |
| 不确定 GPU 会访问哪些数据 | 明确知道数据使用模式 |

⚠️ **注意**：统一内存简化编程但可能牺牲性能。性能敏感代码用显式 `cudaMemcpy` 通常更快。

### 9. 内存选型决策指南

面对一个新 Kernel，如何决定把数据放在哪里？

```
数据是否在 Kernel 中被修改?
   │
   ├─ 只读 ──► 所有线程读同一值?
   │            ├─ 是 且 ≤64KB ──► 常量内存
   │            └─ 否 ──────────► 全局内存 + __ldg()
   │
   └─ 读写 ──► 是否需要 Block 内共享?
                ├─ 是 ──► 共享内存
                └─ 否 ──► 全局内存 (合并访问)
```

#### 9.1 快速参考卡片

| 需求 | 首选方案 | 次选方案 |
|------|---------|---------|
| Block 内线程间数据交换 | 共享内存 + `__syncthreads()` | Warp Shuffle（同 Warp 内） |
| 全局只读广播 | 常量内存 | `__ldg()` |
| 大数组逐元素处理 | 全局内存（合并访问） | — |
| 局部累加器/计数器 | 寄存器 | — |
| 复杂数据结构快速原型 | 统一内存 | — |
| 减少全局内存事务 | Tiling + 共享内存 | L1 缓存 |

#### 9.2 典型优化模式：Tiling（最重要）

Tiling 是最经典的内存优化模式——将全局内存数据分块加载到共享内存，在共享内存上做多次计算，从而用少量全局内存访问换取大量快速的共享内存访问：

```cpp
__global__ void matmulTiled(float* A, float* B, float* C,
                            int M, int N, int K) {
    __shared__ float As[TILE][TILE];
    __shared__ float Bs[TILE][TILE];

    int row = blockIdx.y * TILE + threadIdx.y;
    int col = blockIdx.x * TILE + threadIdx.x;
    float sum = 0.0f;

    for (int t = 0; t < (K + TILE - 1) / TILE; t++) {
        // 协作加载 tile 到共享内存
        if (row < M && t * TILE + threadIdx.x < K)
            As[threadIdx.y][threadIdx.x] = A[row * K + t * TILE + threadIdx.x];
        else
            As[threadIdx.y][threadIdx.x] = 0.0f;

        if (col < N && t * TILE + threadIdx.y < K)
            Bs[threadIdx.y][threadIdx.x] = B[(t * TILE + threadIdx.y) * N + col];
        else
            Bs[threadIdx.y][threadIdx.x] = 0.0f;

        __syncthreads();

        // 在共享内存上计算
        for (int k = 0; k < TILE; k++) {
            sum += As[threadIdx.y][k] * Bs[k][threadIdx.x];
        }
        __syncthreads();
    }

    if (row < M && col < N) {
        C[row * N + col] = sum;
    }
}
```

💡 **提示**：Tiling 的核心思想——"搬一次数据，用多次"——在 FlashAttention、GEMM 优化等高级场景中反复出现，是 GPU 编程最重要的优化范式之一。

## 面试回答（可直接口述的版本）

**问：请描述 CUDA 的内存层次和优化原则。**

答：CUDA 内存分五层：寄存器（单线程私有，最快）、共享内存（Block 内共享，~20 周期）、L1/L2 缓存（自动管理）、全局内存（HBM，~400 周期）、常量内存（64KB 只读+缓存）。优化核心原则有三条：① 全局内存必须合并访问——Warp 内 32 个线程访问连续地址，一次事务完成；② Block 内协作用共享内存做 Tiling，搬一次数据用多次，减少全局内存访问；③ 数据结构用 SoA 而非 AoS，保证连续访问。算力远大于带宽，所以内存访问通常是瓶颈，Tiling 是最通用的优化范式，在 GEMM 和 FlashAttention 中反复出现。

## 深入追问（可能被追问的点）

- **Q1: 共享内存和 L1 缓存有什么关系？**
  A: 现代 GPU（sm_80+）它们共享同一块物理 SRAM，可通过 `cudaFuncSetCacheConfig` 配置比例。共享内存程序员显式管理，L1 缓存硬件自动管理。

- **Q2: 寄存器溢出（spill）一定不好吗？**
  A: 不一定。少量溢出到局部内存（有 L1/L2 缓存）影响不大，但大量溢出会显著降速。用 `--ptxas-options=-v` 看 spill stores 数量判断。

- **Q3: 统一内存为什么慢？**
  A: 按需页面迁移有缺页中断开销，且首次访问时数据不在本地。用 `cudaMemPrefetchAsync` 预取可以缓解。

- **Q4: Bank Conflict 怎么检测？**
  A: 用 Nsight Compute 的 `l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld.sum` 指标，或用 `cuda-gdb` 调试。

- **Q5: 为什么 Tiling 能加速？**
  A: 全局内存延迟 ~400 周期，共享内存 ~20 周期。Tiling 把数据搬到共享内存后多次复用，把 20 次全局访问变成 1 次全局 + 20 次共享，大幅减少平均延迟。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| 共享内存 vs 全局内存 | 前者 Block 内芯片上(~20cyc)，后者全局芯片外(~400cyc) | shared=白板，global=仓库 |
| `__shared__` vs `__constant__` | 前者可读写 Block 内共享，后者全局只读 | shared=团队白板，constant=公告栏 |
| AoS vs SoA | 前者结构体数组(不合并)，后者数组结构体(合并) | SoA=Suitable for SIMD |
| 静态 vs 动态共享内存 | 前者编译时定大小，后者启动时指定 | 静态简单，动态灵活 |
| `cudaMalloc` vs `cudaMallocManaged` | 前者显式管理，后者统一内存自动迁移 | Managed=省心但慢 |
| Bank Conflict vs Warp Divergence | 前者内存访问冲突串行化，后者分支执行串行化 | Bank=内存，Divergence=计算 |

## 常见报错速查表

| ❌ 报错 | 原因 | ✅ 解决 |
|--------|------|--------|
| `out of memory` | 显存不足 | 减小数据量或检查是否有内存泄漏 |
| `illegal memory access` | 访问越界或主机指针当设备指针用 | 检查 `cudaMalloc` 和索引边界 |
| `shared memory too large` | 静态共享内存超过 SM 上限 | 改用动态共享内存或减小 tile |

| 统一内存卡顿 | 页面迁移开销 | 用 `cudaMemPrefetchAsync` 预取 |
