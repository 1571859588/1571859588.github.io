## 一句话结论

内存访问是 CUDA Kernel 最大的性能杠杆。三大优化技术：① **合并访问**——连续线程访问连续地址（SoA 优于 AoS），带宽利用率从 3% 到 95%；② **Bank Conflict 规避**——共享内存加 padding 消除 32-way 冲突；③ **向量化加载**——`float4` 一次读 16B，减少 75% 指令数。矩阵转置是综合应用这三者的经典案例。

## 核心概念（新人友好讲解）

### 1. 为什么内存访问是性能瓶颈

GPU 像一个巨大工厂——车间几千台机器（计算核心）同时运转，但原料仓库（显存）只有一条搬运通道（内存总线）。无论机器多快，通道堵住整条生产线都等着。

算术：A100 峰值算力 312 TFLOPS（FP16），HBM2e 带宽约 2 TB/s。对向量加法 `c[i]=a[i]+b[i]`：

```
Arithmetic Intensity = 1 FLOP / 12 Bytes (读2写1) ≈ 0.083 FLOP/Byte
带宽可支撑的计算量 = 2 TB/s × 0.083 = 166 GFLOPS
```

只有峰值算力的 0.05%。因此**优化内存访问模式往往比优化计算逻辑更能提升性能**。

#### Roofline 模型快速回顾

- **Memory-Bound**：受带宽限制，优化方向是减少内存访问、提高访问效率
- **Compute-Bound**：受计算限制，优化方向是提高计算并行度

大多数 Kernel（Elementwise、归约、矩阵转置等）是 Memory-Bound。

### 2. 全局内存与合并访问

![合并访存与非合并访存对比](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/合并访存与非合并访存对比.png)

#### 2.1 合并访问的原理（新人重点理解）

全局内存访问以 **Warp** 为单位进行事务（transaction）合并。当一个 Warp 的 32 线程同时发起内存读写时，硬件尝试将这些请求合并为最少的内存事务。

每个内存事务粒度为 **32 Bytes**（一个 cache line sector）。L1 cache line 为 128 Bytes，由 4 个 32B sector 组成。

💡 **提示**：合并访问核心原则很简单——让 Warp 内连续线程访问连续内存地址。

**合并访问图解**：

```
合并访问 ✅：Warp 内 32 线程访问连续地址
  T0   T1   T2   T3  ...  T31
  ▼    ▼    ▼    ▼         ▼
 [0]  [4]  [8]  [12]  ... [124]   ← 1 次 128B 事务 (1 cache line)

非合并访问 ❌：Warp 内 32 线程访问间隔地址 (stride=32)
  T0   T1   T2   T3  ...  T31
  ▼    ▼    ▼    ▼         ▼
 [0] [128][256][384] ... [3968]  ← 32 次事务，带宽利用率 3%
```

#### 2.2 合并 vs 非合并的性能差异

```cpp
// ✅ 合并访问：连续线程访问连续地址
__global__ void coalesced_read(float* data, float* out, int N) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    if (tid < N) {
        out[tid] = data[tid];  // data[0], data[1], ..., data[31] 连续
    }
}

// ❌ 非合并（跳跃访问）：连续线程访问间隔地址
__global__ void strided_read(float* data, float* out, int N, int stride) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    if (tid < N) {
        out[tid] = data[tid * stride];  // data[0], data[32], data[64], ...
    }
}
```

性能对比（A100，读取 1GB 数据）：

| 📊 访问模式 | 内存事务数 | 有效带宽利用率 | 相对耗时 |
|------------|-----------|--------------|---------|
| 连续对齐（stride=1） | 1x | ~95% | 1x |
| stride=2 | 2x | ~50% | 2x |
| stride=4 | 4x | ~25% | 4x |
| stride=32 | 32x | ~3% | 32x |
| 完全随机 | 32x | ~3% | 32x |

#### 2.3 对齐要求

```cpp
// ✅ 对齐访问：起始地址是 128B 的倍数
float* aligned_ptr = data;  // cudaMalloc 保证 256B 对齐
out[tid] = aligned_ptr[tid];

// ⚠️ 偏移访问：起始地址未对齐，可能多出一个事务
out[tid] = data[tid + 1];  // 起始地址偏移 4B，不再 128B 对齐
```

好消息：`cudaMalloc` 分配的内存始终 256 Bytes 对齐，从头连续访问通常无对齐问题。需注意手动指针偏移的情况。

#### 2.4 AoS vs SoA（经典问题）

```cpp
// ❌ AoS（Array of Structures）：不利于合并访问
struct Particle {
    float x, y, z, w;
};
Particle particles[N];
// 访问所有 x：particles[0].x, particles[1].x, ...
// 地址跳跃：0, 16, 32, 48, ... （stride=4）

// ✅ SoA（Structure of Arrays）：利于合并访问
struct ParticlesSoA {
    float x[N], y[N], z[N], w[N];
};
ParticlesSoA particles;
// 访问所有 x：particles.x[0], particles.x[1], ...
// 地址连续：0, 4, 8, 12, ... （stride=1）
```

📌 **关键点**：GPU 编程中 **SoA 几乎总是优于 AoS**，因为 Warp 并行模式天然适合连续线程访问连续字段。

#### 2.5 二维数组行列访问

```cpp
// 二维数组按行存储（Row-Major），element[row][col] 在地址 row*cols+col
// ✅ 行方向遍历：连续线程访问同一行的连续列
int col = threadIdx.x + blockIdx.x * blockDim.x;
int row = blockIdx.y;
float val = matrix[row * cols + col];  // 合并

// ❌ 列方向遍历：连续线程访问同一列的连续行
int row2 = threadIdx.x + blockIdx.x * blockDim.x;
int col2 = blockIdx.y;
float val2 = matrix[row2 * cols + col2];  // stride = cols，极差
```

### 3. 共享内存与 Bank Conflict

#### 3.1 共享内存的 Bank 结构（新人重点）

共享内存被划分为 **32 个 Bank**（与 Warp 大小一致），每个 Bank 宽 4 Bytes。连续的 4 Byte 字分配到连续 Bank：

```
地址 0~3:     Bank 0
地址 4~7:     Bank 1
地址 8~11:    Bank 2
...
地址 124~127: Bank 31
地址 128~131: Bank 0  （循环）
...
```

```
Bank:    0   1   2   3   ...  31
       ┌───┬───┬───┬───┬─────┬───┐
地址:  │ 0 │ 4 │ 8 │12 │ ... │124│  ← 第 0 行
       ├───┼───┼───┼───┼─────┼───┤
       │128│132│136│140│ ... │252│  ← 第 1 行
       └───┴───┴───┴───┴─────┴───┘
```

每个时钟周期，每个 Bank 只能服务一次读或写请求。如果同一 Warp 中多个线程访问**同一 Bank 的不同地址**，这些访问必须串行化——这就是 Bank Conflict。

#### 3.2 Bank Conflict 类型与代价

| 📊 冲突类型 | 含义 | 性能影响 |
|------------|------|---------|
| 无冲突（No Conflict） | 32 线程访问 32 个不同 Bank | 1 个周期完成 |
| 2-way Conflict | 2 个线程访问同一 Bank | 2 个周期（串行化） |
| N-way Conflict | N 个线程访问同一 Bank | N 个周期 |
| 广播（Broadcast） | 多个线程访问同一 Bank 的**同一地址** | 1 个周期（免费） |

⚠️ **注意**：广播是特殊情况——多个线程读同一地址，硬件广播该值给所有请求线程，不算冲突。只有"同一 Bank，不同地址"才会产生冲突。

#### 3.3 经典冲突场景：矩阵转置（新人必看）

```cpp
// 朴素矩阵转置（Naive Transpose）
__shared__ float tile[32][32];

// 写入时：连续线程写同一行的不同列 → 无冲突
tile[threadIdx.y][threadIdx.x] = input[gy * N + gx];

// 读取时：连续线程读同一列的不同行
output[gx * N + gy] = tile[threadIdx.x][threadIdx.y];
// threadIdx.x=0 → tile[0][ty], 地址= (0*32+ty)*4 → Bank = ty%32
// threadIdx.x=1 → tile[1][ty], 地址= (1*32+ty)*4 → Bank = (32+ty)%32 = ty%32
// 所有线程 Bank 相同（= ty%32）→ 32-way Bank Conflict！
```

#### 3.4 解决方案一：Padding（最常用）

在共享内存声明中额外加一列，打破 32 的倍数对齐：

```cpp
// ✅ 添加 1 列 Padding
__shared__ float tile[32][32 + 1];  // 33 列

// 原 tile[row][col] 地址 = (row * 32 + col) * 4
// 现 tile[row][col] 地址 = (row * 33 + col) * 4
// Bank = (row * 33 + col) % 32

// 读取 tile[threadIdx.x][ty]：
// threadIdx.x=0 → Bank = (0*33+ty)%32 = ty%32
// threadIdx.x=1 → Bank = (1*33+ty)%32 = (33+ty)%32 = (1+ty)%32
// 每个线程访问不同 Bank → 无冲突！
```

Padding 代价：每行多浪费 4 Bytes 共享内存，但性能提升通常远超这点开销。

#### 3.5 解决方案二：Swizzle（高级）

```cpp
// Swizzle 思路：将线性索引通过 XOR 变换打乱 Bank 分布
__device__ int swizzle_index(int row, int col) {
    return row * 32 + (col ^ row);  // XOR 变换
}

// 写入
tile[row][col ^ row] = input_val;
// 读取
float val = tile[col][row ^ col];
```

Swizzle 不浪费额外空间，但增加索引计算复杂度。在 CUTLASS 等高性能 GEMM 实现中广泛使用。

#### 3.6 如何检测 Bank Conflict

使用 **Nsight Compute** 观察共享内存指标：

```
l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_ld  # 加载冲突
l1tex__data_bank_conflicts_pipe_lsu_mem_shared_op_st  # 存储冲突
```

计数器不为零说明存在 Bank Conflict，需检查共享内存访问模式。

### 4. 向量化加载

#### 4.1 什么是向量化加载（新人重点）

向量化加载用一条指令读取多个连续元素（如 `float4` 一次读 4 个 float，16 Bytes）。好处：

1. **减少指令数**：4 次 `LDG.32` → 1 次 `LDG.128`
2. **提高带宽利用率**：单次事务传输更多有效数据
3. **减少调度开销**：更少内存指令排队

#### 4.2 常用向量类型

| 📊 类型 | 大小 | 等价 |
|---------|------|------|
| `float2` | 8 Bytes | 2 个 float |
| `float4` | 16 Bytes | 4 个 float |
| `int4` | 16 Bytes | 4 个 int |
| `double2` | 16 Bytes | 2 个 double |

💡 **提示**：GPU 一条 cache line 为 128 Bytes。一个 Warp 的 32 线程每人加载一个 `float4`（16B）= 512B，需 4 条 cache line。对比每人加载 `float`（4B）= 128B 只需 1 条。但每条加载指令处理数据量是 4 倍，总指令数减少 75%。

#### 4.3 基本用法

```cpp
// 标量版本：每个线程加载 1 个 float
__global__ void scalar_copy(float* dst, float* src, int N) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    if (tid < N) {
        dst[tid] = src[tid];
    }
}

// 向量化版本：每个线程加载 4 个 float
__global__ void vectorized_copy(float* dst, float* src, int N) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    int idx = tid * 4;  // 每个线程负责4个元素

    if (idx + 3 < N) {
        // 将 float* 强制转换为 float4*，一条指令读取 16B
        float4 val = reinterpret_cast<float4*>(src)[tid];
        reinterpret_cast<float4*>(dst)[tid] = val;
    }
}
```

#### 4.4 对齐要求

向量化加载有严格对齐要求：

```cpp
// float4 要求 16B 对齐
// cudaMalloc 分配的地址是 256B 对齐的 → 满足

// 但如果有偏移，需要特殊处理
float* base = ...; // 256B 对齐
float* offset_ptr = base + 3;  // 偏移 12B，不满足 16B 对齐

// ✅ 安全做法：先标量处理头部未对齐部分，再向量化处理主体
__global__ void safe_vectorized(float* data, int offset, int N) {
    int align_offset = (4 - (offset % 4)) % 4;
    // ... 标量处理前 align_offset 个元素

    // 主体向量化处理
    float4* aligned_ptr = reinterpret_cast<float4*>(data + offset + align_offset);
    // ...
}
```

#### 4.5 实用模式：向量化归约

```cpp
__global__ void vectorized_sum(float* input, float* output, int N) {
    int tid = threadIdx.x + blockIdx.x * blockDim.x;
    int stride = blockDim.x * gridDim.x;

    float sum = 0.0f;

    int vec_N = N / 4;
    float4* vec_input = reinterpret_cast<float4*>(input);

    for (int i = tid; i < vec_N; i += stride) {
        float4 val = vec_input[i];
        sum += val.x + val.y + val.z + val.w;
    }

    // 处理尾部不足4个的元素
    int tail_start = vec_N * 4;
    for (int i = tail_start + tid; i < N; i += stride) {
        sum += input[i];
    }

    output[blockIdx.x] = sum;  // 简化，实际需要归约
}
```

#### 4.6 何时使用向量化加载

| ✅ 适合场景 | ❌ 不适合场景 |
|------------|-------------|
| Elementwise 操作（加减乘除） | 复杂索引模式（gather/scatter） |
| 大规模数据拷贝/初始化 | 数据量不是 4 的倍数且无法处理尾部 |
| 带宽受限的 Kernel | 计算密集型 Kernel（瓶颈不在访存） |
| 数据地址可保证对齐 | 起始地址无法对齐 |

### 案例：矩阵转置优化（经典教科书案例）

矩阵转置不涉及任何计算，性能完全取决于内存访问效率。

**朴素版本**：

```cpp
__global__ void transpose_naive(float* out, float* in, int width, int height) {
    int col = threadIdx.x + blockIdx.x * blockDim.x;
    int row = threadIdx.y + blockIdx.y * blockDim.y;
    if (col < width && row < height) {
        out[col * height + row] = in[row * width + col];
        // 读取：合并（连续线程访问连续列）
        // 写入：不合并（stride = height）
    }
}
```

**共享内存优化版本（完整链）**：

```cpp
#define TILE_DIM 32
#define BLOCK_ROWS 8

__global__ void transpose_shared(float* out, float* in, int width, int height) {
    __shared__ float tile[TILE_DIM][TILE_DIM + 1];  // +1 Padding 消除 Bank Conflict

    int x = blockIdx.x * TILE_DIM + threadIdx.x;
    int y = blockIdx.y * TILE_DIM + threadIdx.y;

    // 合并读取：每个 warp 读取连续元素
    for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
        if (x < width && (y + j) < height) {
            tile[threadIdx.y + j][threadIdx.x] = in[(y + j) * width + x];
        }
    }

    __syncthreads();

    // 合并写入：转置后的索引，连续线程写连续地址
    x = blockIdx.y * TILE_DIM + threadIdx.x;
    y = blockIdx.x * TILE_DIM + threadIdx.y;

    for (int j = 0; j < TILE_DIM; j += BLOCK_ROWS) {
        if (x < height && (y + j) < width) {
            out[(y + j) * height + x] = tile[threadIdx.x][threadIdx.y + j];
        }
    }
}

// 调用方式：
// dim3 block(TILE_DIM, BLOCK_ROWS);
// dim3 grid((width + TILE_DIM - 1) / TILE_DIM, (height + TILE_DIM - 1) / TILE_DIM);
```

核心思路：用共享内存做"中转站"，将全局内存的非合并写入转化为共享内存的列读取（通过 Padding 消除 Bank Conflict），再以合并方式写出。

**编译运行**：

```bash
# ✅ 无需 sudo
nvcc -O3 -arch=sm_89 transpose.cu -o transpose
./transpose
```

**预期输出**（RTX 4090，4096×4096 矩阵）：

```
Naive transpose:   3.21 ms,  41.5 GB/s
Shared transpose:  0.18 ms, 742.0 GB/s
Speedup: 17.8x
Verification PASSED!
```

```cpp
// 使用 __ldg() 通过只读数据缓存加载
// 适合只读数据，可避免 L1 缓存污染
float val = __ldg(&input[tid]);

// PTX 级别的缓存控制

```

## 面试回答（可直接口述的版本）

**问：如何优化 CUDA Kernel 的内存访问？**

答：三大技术。第一，合并访问——让 Warp 内 32 个线程访问连续地址，因为 GPU 以 32B sector 为粒度做内存事务，连续访问一次事务搞定，跨步访问要 32 次事务，带宽利用率从 95% 掉到 3%。数据结构上要用 SoA 而非 AoS。第二，消除 Bank Conflict——共享内存分 32 个 Bank，同 Warp 线程访问同 Bank 不同地址会串行化，矩阵转置这种列访问会产生 32-way 冲突，解决办法是声明 `[32][33]` 加一列 padding 打破对齐。第三，向量化加载——用 float4 一次读 16B，减少 75% 的加载指令数。矩阵转置是综合应用这三者的经典案例，用共享内存做中转把非合并写入转成合并写入，加 padding 消除 Bank Conflict，能加速十几倍。

## 深入追问（可能被追问的点）

- **Q1: cache line 和 sector 是什么关系？**
  A: cache line 是 128B，由 4 个 32B sector 组成。内存事务以 sector 为粒度，一个 Warp 访问 32 个 float（128B）正好一条 cache line、4 个 sector，完美合并。

- **Q2: Padding 一定加 1 吗？**
  A: 不一定。加 1 最简单常用，但某些访问模式可能需要其他值。关键是打破 32 的倍数对齐。用 Nsight Compute 验证是否消除冲突。

- **Q3: 向量化加载为什么能提升带宽？**
  A: 不是提升单次事务带宽，而是减少指令数。同样数据量，float4 比 float 少 75% 的加载指令，减少指令调度和排队开销，让内存总线更饱和。

- **Q4: 什么时候用 Swizzle 而非 Padding？**
  A: 共享内存紧张时用 Swizzle（不浪费空间），但索引计算复杂。Padding 简单但浪费空间。GEMM 等高性能库通常用 Swizzle。

- **Q5: `__ldg()` 和普通访问有什么区别？**
  A: `__ldg()` 走只读数据缓存（texture cache 路径），不污染 L1，适合只读数据。编译器对 `const float*` 有时会自动用 `__ldg`。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| cache line vs sector | line=128B，sector=32B，1 line=4 sector | line 大 sector 小 |
| 合并访问 vs 对齐访问 | 合并=连续，对齐=地址边界对齐 | 合并看连续性，对齐看边界 |
| AoS vs SoA | AoS 结构体数组(不合并)，SoA 数组结构体(合并) | SoA=Suitable for SIMD |
| Bank Conflict vs Warp Divergence | 前者内存访问串行化，后者分支执行串行化 | Bank=内存，Divergence=计算 |
| Padding vs Swizzle | Padding 加列浪费空间，Swizzle 用 XOR 不浪费 | Padding 简单，Swizzle 高效 |
| `__ldg()` vs 普通读 | 前者走只读缓存不污染L1，后者走L1 | ldg=只读数据 |

## 常见报错速查表

| ❌ 报错/问题 | 原因 | ✅ 解决 |
|-------------|------|--------|
| 带宽利用率极低（<10%） | 非合并访问 | 检查 stride，改用 SoA |
| Bank Conflict 警告 | 共享内存访问模式冲突 | 加 padding 或用 Swizzle |
| 向量化加载崩溃 | 地址未 16B 对齐 | 检查偏移，标量处理头部 |
| `reinterpret_cast<float4*>` 报错 | 指针类型不兼容 | 确保源指针是 `float*` 且对齐 |
| 矩阵转置结果错误 | 共享内存索引算错或漏 `__syncthreads` | 检查 tile 索引和同步点 |
| ncu 指标采集失败 | 无 sudo 权限访问硬件计数器 | 用基础指标，或联系管理员 |
