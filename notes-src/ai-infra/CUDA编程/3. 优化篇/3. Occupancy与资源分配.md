## 一句话结论

Occupancy = SM 上活跃 Warp 数 / SM 最大 Warp 数，衡量"停车场利用率"。三大限制因素：寄存器、共享内存、Block 大小。但**高 Occupancy 不等于高性能**——GEMM、FlashAttention 等计算密集型 Kernel 故意用更多寄存器和共享内存换数据复用，Occupancy 低反而更快。真正的目标是在延迟隐藏与资源利用间找平衡。

## 核心概念（新人友好讲解）

### 1. Occupancy 是什么

想象一个停车场有 64 个车位（SM 最大可驻留的 Warp 数），如果当前只有 32 辆车停着，停车场利用率就是 50%。Occupancy 就是 GPU 中 SM 的"停车场利用率"。

#### 1.1 正式定义

```
Occupancy = 每个 SM 上活跃 Warp 数 / 每个 SM 支持的最大 Warp 数
```

例如在 A100（Compute Capability 8.0）上：
- 每个 SM 最大支持 64 个 Warp（2048 个线程）
- 如果你的 Kernel 由于资源限制只能在每个 SM 上运行 32 个 Warp
- 那么 Occupancy = 32/64 = 50%

#### 1.2 为什么 Occupancy 重要

核心价值在于**延迟隐藏**（Latency Hiding）。GPU 执行模型依赖 Warp 切换来掩盖内存访问延迟：

```
时间线：
Warp 0: [计算] [等待内存...400cycles...] [计算]
Warp 1:        [计算] [等待内存...400cycles...] [计算]
Warp 2:               [计算] [等待内存...400cycles...]
...

如果活跃 Warp 足够多，调度器总能找到就绪 Warp 来填充等待期
```

需要多少 Warp 才能完全隐藏延迟？粗略估算：

```
所需 Warp 数 ≥ 内存延迟(cycles) / 每条指令执行周期 = 400 / 4 = 100
```

但一个 SM 最多 64 个 Warp，实际中无法完全隐藏延迟——这正是为什么要让 Occupancy 尽可能高（但不是唯一目标）。

### 2. SM 的资源清单

每个 SM 有一组固定硬件资源，所有驻留其上的 Block 共享。

#### 2.1 主要架构资源对比（新人速查）

| 📊 资源 | Ampere (A100) | Ada (RTX 4090) | Hopper (H100) |
|---------|---------------|----------------|---------------|
| 每 SM 最大线程数 | 2048 | 1536 | 2048 |
| 每 SM 最大 Warp 数 | 64 | 48 | 64 |
| 每 SM 最大 Block 数 | 32 | 16 | 32 |
| 每 SM 寄存器总数 | 65536 | 65536 | 65536 |
| 每线程最大寄存器数 | 255 | 255 | 255 |
| 每 SM 共享内存上限 | 164 KB | 100 KB | 228 KB |
| 每 Block 最大共享内存 | 163 KB | 99 KB | 227 KB |
| 每 Block 最大线程数 | 1024 | 1024 | 1024 |

#### 2.2 资源分配的层级关系

```
SM 总资源池
   │
   ├──► Block 0 ──► Warp 0-7 ──► Thread 0-255
   ├──► Block 1 ──► Warp 8-15 ──► Thread 256-511
   └──► Block N ──► ...
```

📌 **关键点**：资源以 **Block 为单位**分配。一个 Block 要么完整进入 SM，要么完全不进入——不存在半个 Block 驻留。因此，如果一个 Block 占用资源刚好超过 SM 剩余资源一半多一点，整个剩余空间就浪费了。

### 3. 影响 Occupancy 的三大因素

#### 3.1 因素一：寄存器用量（最常见限制）

每个线程使用的寄存器数量是最常见的 Occupancy 限制因素。

**计算逻辑**：

```
每 SM 可驻留线程数 = ⌊SM 寄存器总数 / 每线程寄存器数⌋
```

但实际分配要满足粒度约束。寄存器以 **256 个为一组**（Allocation Granularity）分配给一个 Warp。

**示例**（A100，65536 寄存器/SM）：

| 每线程寄存器数 | 每 Warp 实际分配 | SM 可容纳 Warp 数 | Occupancy |
|--------------|-----------------|-------------------|-----------|
| 32 | 32×32=1024 | 65536/1024=64 | 100% |
| 48 | 48×32=1536 | 65536/1536=42 | 42/64≈65% |
| 64 | 64×32=2048 | 65536/2048=32 | 50% |
| 128 | 128×32=4096 | 65536/4096=16 | 25% |
| 255 | 255×32=8160→对齐8192 | 65536/8192=8 | 12.5% |

💡 **提示**：可用 `__launch_bounds__` 或 `maxrregcount` 编译选项限制寄存器用量：

```cpp
// 方式一：__launch_bounds__(maxThreadsPerBlock, minBlocksPerMultiprocessor)
__global__ void __launch_bounds__(256, 2)  // 每 Block 最多256线程，目标每SM 2个Block
my_kernel(...) { ... }

// 编译器行为：
// SM 寄存器总量 = 65536
// 需同时驻留：256线程 × 2block = 512线程
// 每线程最大寄存器：65536 / 512 = 128个
// 编译器将寄存器上限设为 128，溢出的变量 spill 到 Local Memory

// 方式二：编译时全局限制
// nvcc -maxrregcount=32 my_kernel.cu
```

⚠️ **注意**：强制限制寄存器数可能导致编译器将变量溢出到局部内存（register spilling），反而降低性能。需用 profiler 实际验证。

#### 3.2 因素二：共享内存用量

每个 Block 声明的共享内存总量（静态 + 动态）会限制 SM 能容纳的 Block 数。

```
每 SM 可驻留 Block 数 = ⌊SM 共享内存总量 / 每 Block 共享内存用量⌋
```

**示例**（A100，164KB 共享内存/SM）：

| 每 Block 共享内存 | SM 可容纳 Block 数 | 每 Block 256线程时总线程 | Occupancy |
|-----------------|-------------------|------------------------|-----------|
| 8 KB | 20（但受最大 Block 数 32 限制） | min(20,32)×256→cap 2048 | 100% |
| 32 KB | 5 | 5×256=1280 | 62.5% |
| 64 KB | 2 | 2×256=512 | 25% |
| 164 KB | 1 | 1×256=256 | 12.5% |

#### 3.3 因素三：Block 大小

Block 大小同时受两个限制：① 每 Block 最大 1024 线程；② 每 SM 最大 Block 数量。

| Block 大小 | 每 SM Block 数上限 | 总 Warp 数 | Occupancy |
|-----------|-------------------|-----------|-----------|
| 32 (1 Warp) | 32（Block 数限制） | 32 | 50% |
| 64 (2 Warps) | 32（Block 数限制） | 64 | 100% |
| 128 (4 Warps) | 16（2048/128） | 64 | 100% |
| 256 (8 Warps) | 8（2048/256） | 64 | 100% |
| 512 (16 Warps) | 4（2048/512） | 64 | 100% |
| 1024 (32 Warps) | 2（2048/1024） | 64 | 100% |

📌 **关键点**：Block 太小（如 32 线程）会被"每 SM 最大 Block 数"限制，导致 Occupancy 上不去。建议至少 128 或 256。**256 是不错的默认选择**。

### 4. Occupancy 计算实例

已知条件（A100）：
- Kernel 每线程使用 40 个寄存器
- Block 大小：256 线程 = 8 个 Warp
- 每 Block 使用 16 KB 共享内存

**步骤一：寄存器限制**

```
每 Warp 寄存器 = 40 × 32 = 1280
SM 可容纳 Warp = ⌊65536 / 1280⌋ = 51 Warps
```

**步骤二：共享内存限制**

```
SM 可容纳 Block = ⌊164KB / 16KB⌋ = 10 Blocks
对应 Warp 数 = 10 × 8 = 80 Warps
```

**步骤三：Block 数限制**：每 SM 最多 32 Block，10 未触达上限。

**步骤四：线程数限制**：每 SM 最多 2048 线程 = 64 Warps。

**综合取最小值**：

```
实际 Warp 数 = min(51, 80, 64) = 51 Warps
```

但 Block 整体分配（每 Block 8 Warp）：

```
⌊51 / 8⌋ × 8 = 6 × 8 = 48 Warps
Occupancy = 48 / 64 = 75%
```

| 步骤 | 是否需要 sudo | 你的情况 |

### 查看 Kernel 资源使用量

```bash
# ✅ 无需 sudo
nvcc -Xptxas -v my_kernel.cu -o my_kernel
```

**预期输出**：

```
ptxas info    : Compiling entry function 'my_kernel' for 'sm_89'
ptxas info    : Function properties for my_kernel
    0 bytes stack frame, 0 bytes spill stores, 0 bytes spill loads
ptxas info    : Used 40 registers, 16384 bytes smem, 380 bytes cmem[0]
```

**输出解读**（新人必看）：

| 输出项 | 含义 |
|--------|------|
| `Used 40 registers` | 每线程使用 40 个寄存器 |
| `16384 bytes smem` | 每 Block 共享内存 16384 字节 = 16KB |
| `380 bytes cmem[0]` | 常量内存 380 字节 |
| `0 bytes spill stores` | 无寄存器溢出（好现象） |
| `0 bytes stack frame` | 无栈帧（kernel 不调用函数） |

### 使用 CUDA Occupancy API

```cpp
#include <cuda_runtime.h>

int main() {
    int blockSize = 256;
    int minGridSize, gridSize;

    // 自动计算最佳 Block 大小
    cudaOccupancyMaxPotentialBlockSize(
        &minGridSize, &blockSize,
        my_kernel,  // kernel 函数指针
        0,          // 动态共享内存大小
        0           // Block 大小上限（0=无限制）
    );

    // 查询给定配置的 Occupancy
    int maxActiveBlocks;
    cudaOccupancyMaxActiveBlocksPerMultiprocessor(
        &maxActiveBlocks,
        my_kernel,
        blockSize,
        0  // 动态共享内存
    );

    int device;
    cudaGetDevice(&device);
    cudaDeviceProp prop;
    cudaGetDeviceProperties(&prop, device);

    float occupancy = (float)(maxActiveBlocks * blockSize) /
                      prop.maxThreadsPerMultiProcessor;
    printf("Optimal blockSize: %d\n", blockSize);
    printf("Occupancy: %.1f%%\n", occupancy * 100);
    return 0;
}
```

**预期输出**：

```
Optimal blockSize: 256
Occupancy: 75.0%
```

**API 参数解释**（新人必看）：

| API | 作用 |
|-----|------|
| `cudaOccupancyMaxPotentialBlockSize` | 自动算出使 occupancy 最大的 blockSize |
| `cudaOccupancyMaxActiveBlocksPerMultiprocessor` | 查询给定 blockSize 时每 SM 最多活跃 Block 数 |
| `cudaGetDeviceProperties` | 获取设备属性（maxThreadsPerMultiProcessor 等） |

```bash
# ✅ 基础功能无需 sudo
ncu --set basic --target-processes all ./my_kernel
```

Nsight Compute 会显示 Occupancy 的瓶颈来源：

Occupancy Limiters:

| 指标 | 含义 |

## 5. Occupancy 不是越高越好（重要认知）

### 5.1 反直觉的例子

| 📊 版本 | 寄存器/线程 | 共享内存/Block | Occupancy | 实际 GFLOPS |
|---------|------------|--------------|-----------|------------|
| Version A | 32 | 8 KB | 100% | 800 |
| Version B | 96 | 48 KB | 33% | 1200 |

Version B 的 Occupancy 更低，但性能更高！原因：它用更多寄存器和共享内存**增加数据复用**，每次从全局内存加载的数据被反复使用多次，减少了总内存访问量。

### 5.2 高 Occupancy 的代价

盲目追求高 Occupancy 可能导致：

1. **寄存器溢出**：强制限制 `maxrregcount`，编译器将变量溢出到局部内存，速度慢数十倍
2. **共享内存复用不足**：用更少共享内存意味着 Tile 更小，数据复用率降低，总全局内存访问反而增加
3. **缓存抖动**：太多活跃 Warp 争抢有限 L1/L2 缓存，cache miss 率上升

### 5.3 何时 Occupancy 是关键

对**内存访问密集且复用度低**的 Kernel 最重要：
- ✅ Elementwise 操作（向量加法、激活函数）：几乎无数据复用，纯靠延迟隐藏
- ✅ 归约操作：每元素只被读一次
- ✅ 简单 Stencil 操作

### 5.4 何时可接受低 Occupancy

- ✅ GEMM / 矩阵乘法：大量数据复用，寄存器和共享内存换计算效率
- ✅ FlashAttention：用更多共享内存减少 HBM 访问
- ✅ 计算密集型 Kernel：瓶颈在计算而非访存

### 5.5 黄金法则

```
性能 = f(Occupancy, ILP, 数据复用率, 访存效率, ...)
```

💡 **提示**：不要设定固定 Occupancy 目标（如"必须 75%"），用 profiler 实测吞吐量。如果降低 Occupancy 但提升了数据复用或 ILP，整体性能可能更好。

## 面试回答（可直接口述的版本）

**问：什么是 Occupancy？高 Occupancy 一定好吗？**

答：Occupancy 是 SM 上活跃 Warp 数与最大 Warp 数的比值，衡量 SM 资源利用率。它受三大因素限制：寄存器用量（每线程寄存器越多，SM 容纳线程越少）、共享内存用量（每 Block 共享内存越大，SM 容纳 Block 越少）、Block 大小（太小会被每 SM 最大 Block 数限制）。高 Occupancy 的价值在于延迟隐藏——更多活跃 Warp 让调度器在等内存时有事可做。但高 Occupancy 不等于高性能：GEMM、FlashAttention 这类计算密集型 Kernel 故意用更多寄存器和共享内存换数据复用，Occupancy 低到 33% 反而比 100% 快，因为减少的全局内存访问量远超延迟隐藏的收益。调优原则是：Memory-bound Kernel 追求高 Occupancy，Compute-bound Kernel 优先保证数据复用，用 Nsight Compute 实测决定。

## 深入追问（可能被追问的点）

- **Q1: `__launch_bounds__` 的两个参数怎么用？**
  A: `__launch_bounds__(maxThreadsPerBlock, minBlocksPerMultiprocessor)`。第一个限制每 Block 最大线程数，第二个提示编译器目标每 SM 驻留多少 Block，编译器据此压缩寄存器用量。第二个参数是提示不是硬约束。

- **Q2: Register Spilling 怎么检测和避免？**
  A: 用 `nvcc -Xptxas -v` 看 `spill stores` 数量，非零说明有溢出。避免方法：减少临时变量、适度展开循环、不要过度限制 `maxrregcount`。少量溢出（有 L1 缓存）影响不大，大量溢出需重构。

- **Q3: 为什么 Block 大小选 256？**
  A: 256 是 32 的倍数（Warp 对齐），是 8 个 Warp（SM 容纳 8 个 256 线程的 Block 达满 2048 线程），寄存器压力适中。太小（如 32）被 Block 数限制，太大（如 1024）灵活性差。

- **Q4: GEMM 为什么故意降低 Occupancy？**
  A: 用更大 Tile（更多寄存器和共享内存）让每次加载的数据被复用更多次，减少全局内存访问。虽然 Occupancy 从 100% 降到 25%，但计算量不变而访存量减半，总性能更高。

- **Q5: 如何判断 Kernel 是 Memory-bound 还是 Compute-bound？**
  A: 用 Nsight Compute 的 SOL（Speed of Light）面板，对比 `compute throughput` 和 `memory throughput`，低的那个是瓶颈。或算 Roofline：算术强度 < 峰值算力/峰值带宽 则 Memory-bound。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| Occupancy vs 性能 | 前者是资源利用率，后者是实际吞吐 | 高Occupancy≠高性能 |
| 寄存器 vs 共享内存限制 | 前者按线程算，后者按 Block 算 | 寄存器=线程级，共享=Block级 |
| `__launch_bounds__` vs `maxrregcount` | 前者是函数级提示，后者是全局编译选项 | launch_bounds 灵活，maxrregcount 全局 |
| Memory-bound vs Compute-bound | 前者瓶颈在带宽，后者在算力 | 算术强度低=Memory-bound |
| 理论 Occupancy vs 实际 Occupancy | 前者是资源算出的上限，后者是运行时实测 | 理论≥实际 |
| Allocation Granularity vs Block 整体分配 | 前者寄存器按 256 一组，后者 Block 要么全进要么不进 | 寄存器粒度，Block 整体 |

## 常见报错速查表

| ❌ 报错/问题 | 原因 | ✅ 解决 |
|-------------|------|--------|
| `too many resources requested for launch` | 寄存器或共享内存超限 | 减少 blockSize 或用 `__launch_bounds__` |
| 性能远低于预期 | 寄存器溢出到 local memory | `nvcc -Xptxas -v` 查 spill，调整 `maxrregcount` |
| Occupancy 50% 无法提升 | 寄存器用量是瓶颈 | 减少临时变量或接受现状（Compute-bound 可能无所谓） |
| Occupancy 100% 但性能仍差 | 瓶颈不在延迟隐藏 | 检查访存效率、ILP、数据复用 |
| ncu 报 Occupancy 限制器显示 Registers | 寄存器是瓶颈 | 用 `__launch_bounds__` 压缩寄存器，验证无 spill |
| 动态共享内存超过默认上限 | 默认只给 48KB | `cudaFuncSetAttribute` 设置 `MaxDynamicSharedMemorySize` |
