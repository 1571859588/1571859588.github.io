## 一句话结论

GEMM（矩阵乘法）是计算密集型算子，与 Reduce（Memory-Bound）相反。优化核心是**提升算术强度**（从 Naive 的 0.25 到 Block Tiling 的 32 FLOP/Byte）和**隐藏延迟**。关键链：Block Tiling（共享内存复用）→ Thread Tiling（寄存器外积）→ float4 向量化 → Bank Conflict 消除 → Double Buffering（流水线）→ cp.async（Ampere 异步）。最终可达 95%+ cuBLAS 性能，且 GEMM 故意用低 Occupancy（25%）换高数据复用。

## 核心概念（新人友好讲解）

### 1. GEMM 定义与性能指标

![GEMM 矩阵乘法定义](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/gemm-1.png)

```
C = A × B
A: M×K, B: K×N, C: M×N
总浮点运算量 = 2 × M × N × K FLOPs
```

**性能计算**：

```
TFLOPS = 2 × M × N × K / (Kernel 执行时间(秒) × 10¹²)
```

### 2. Roofline 模型与算术强度（核心概念）

```
算术强度 = 浮点运算量(FLOPs) / 数据搬运量(Bytes)
```

当算术强度超过**平衡点**（= 峰值算力 / 峰值带宽）时 Compute-Bound，否则 Memory-Bound。V100 平衡点约 17.4 FLOPs/Byte。

**Naive 实现的算术强度**：计算一个 C[i][j]，运算量 2K FLOPs，数据搬运 8K Bytes（读 A 一行 + B 一列）：

```
算术强度 = 2K / 8K = 0.25 FLOP/Byte  ← 远低于平衡点，Memory-Bound
```

**Block Tiling 后的算术强度**（BM=BN=128）：

```
算术强度 = BM × BN / (2 × (BM + BN)) = 128×128 / (2×256) = 32 FLOP/Byte  ← Compute-Bound
```

📌 **关键认知**：GEMM 优化的本质是**通过分块提升算术强度**，把 Memory-Bound 变成 Compute-Bound。

### 3. 三级数据搬运链

```
Global Memory ──Block 协作加载 BM×BK / BK×BN──► Shared Memory
                                                      │
                                    Thread 独立加载 TM+TN 个 float
                                                      ▼
                                                  Register
                                                      │
                                              TM×TN 次 FMA
                                                      ▼
                                                   计算结果
```

### 4. 关键概念速查（新人必看）

| 概念 | 解释 |
|------|------|
| **Tiling（分块）** | 把 C 划分为 BM×BN 分块，每 Block 负责一块，沿 K 维以 BK 步长迭代 |
| **Block Tiling** | 用共享内存做 Block 级数据复用 |
| **Thread Tiling** | 用寄存器做线程级数据复用（外积） |
| **外积 vs 内积** | 外积先加载 A 的 TM 行和 B 的 TN 列到寄存器，再做 TM×TN 次 FMA，复用率更高 |
| **float4 向量化** | 一条指令加载 4 个 float，减少 75% 指令数 |
| **Double Buffering** | 两套缓冲区，一边计算一边加载，隐藏延迟 |
| **cp.async** | Ampere 引入的异步拷贝指令，Global→Shared 无需经过寄存器 |
| **Wave 模型** | GPU 同一时刻能并行执行的 Block 集合，影响 L2 命中率 |

### 5. 推荐参数表（实战速查）

| 📊 参数 | ✅ 推荐值 | 📝 说明 |
|---------|-----------|---------|
| BM | 128 | 过小算术强度不足，过大共享内存放不下 |
| BN | 128 | 同上 |
| BK | 8 | 影响共享内存用量和加载指令数 |
| TM | 8 | 每线程负责 8 行 |
| TN | 8 | 每线程负责 8 列 |
| Block Size | 256 | = (128/8)×(128/8) = 16×16 |
| Warp 形状 | 4×8 | 平衡 A/B 访问效率 |
| Registers/Thread | 120-128 | 用 `__launch_bounds__` 控制 |
| Shared Mem/Block | 16-32 KB | 含双缓冲 |

### 1. 朴素实现（Naive，Baseline）

```cpp
__global__ void sgemm_naive(const float* A, const float* B, float* C,
                            int M, int N, int K) {
    int row = blockIdx.y * blockDim.y + threadIdx.y;
    int col = blockIdx.x * blockDim.x + threadIdx.x;

    if (row < M && col < N) {
        float sum = 0.0f;
        for (int k = 0; k < K; k++) {
            sum += A[row * K + k] * B[k * N + col];
        }
        C[row * N + col] = sum;
    }
}
```

**问题**：每计算一个 C 元素都要读 A 的一行和 B 的一列，算术强度 0.25，严重 Memory-Bound。

**性能：6%~11% cuBLAS**

### 2. Thread Block 级 Tiling（共享内存复用）

![Thread Block 级 Tiling 分块](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/gemm-2.png)

```cpp
template <int BM, int BN, int BK, int BLOCK_SIZE>
__global__ void sgemm_block_tiling(float* A, float* B, float* C,
                                   int M, int K, int N) {
    __shared__ float As[BM][BK];
    __shared__ float Bs[BK][BN];

    int r0 = blockIdx.y * BM;
    int c0 = blockIdx.x * BN;
    int tid = threadIdx.x;

    // 加载 tileA 时的线程重排
    constexpr int A_BLOCK_X = BK;
    constexpr int A_BLOCK_Y = BLOCK_SIZE / A_BLOCK_X;
    int a_thread_x = tid % A_BLOCK_X;
    int a_thread_y = tid / A_BLOCK_X;

    // 加载 tileB 时的线程重排
    constexpr int B_BLOCK_X = 32;
    constexpr int B_BLOCK_Y = BLOCK_SIZE / B_BLOCK_X;
    int b_thread_x = tid % B_BLOCK_X;
    int b_thread_y = tid / B_BLOCK_X;

    // 计算 tileC 时的线程排布
    constexpr int C_BLOCK_X = 16;
    constexpr int C_BLOCK_Y = BLOCK_SIZE / C_BLOCK_X;
    int c_thread_x = tid % C_BLOCK_X;
    int c_thread_y = tid / C_BLOCK_X;

    constexpr int Tm = BM / C_BLOCK_Y;
    constexpr int Tn = BN / C_BLOCK_X;
    float Ct[Tm][Tn] = {0.0f};

    for (int k = 0; k < K; k += BK) {
        // 协作加载 tileA
        #pragma unroll
        for (int i = a_thread_y; i < BM; i += A_BLOCK_Y) {
            int r = r0 + i, c = k + a_thread_x;
            As[i][a_thread_x] = (r < M && c < K) ? A[r * K + c] : 0.0f;
        }

        // 协作加载 tileB
        #pragma unroll
        for (int j = b_thread_x; j < BN; j += B_BLOCK_X) {
            int r = k + b_thread_y, c = c0 + j;
            Bs[b_thread_y][j] = (r < K && c < N) ? B[r * N + c] : 0.0f;
        }

        __syncthreads();

        // 外积方式计算 As × Bs
        #pragma unroll
        for (int p = 0; p < BK; p++) {
            for (int i = 0; i < Tm; i++) {
                int row = c_thread_y + i * C_BLOCK_Y;
                for (int j = 0; j < Tn; j++) {
                    int col = c_thread_x + j * C_BLOCK_X;
                    Ct[i][j] += As[row][p] * Bs[p][col];
                }
            }
        }

        __syncthreads();
    }

    // 写回结果
    for (int i = 0; i < Tm; i++) {
        int r = r0 + c_thread_y + i * C_BLOCK_Y;
        for (int j = 0; j < Tn; j++) {
            int c = c0 + c_thread_x + j * C_BLOCK_X;
            if (r < M && c < N) C[r * N + c] = Ct[i][j];
        }
    }
}
```

**核心思想**：把 C 划分为 BM×BN 分块，沿 K 维以 BK 步长迭代，每次把 A 的 BM×BK 和 B 的 BK×BN 子块协作搬到共享内存，在共享内存上做局部乘累加。算术强度从 0.25 提升到 32。

**性能：25%~40% cuBLAS**

### 3. Thread 级 Tiling（寄存器外积）

![Thread 级 Tiling 寄存器外积](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/gemm-3.png)

```cpp
template <int BM, int BN, int BK, int TM, int TN>
__global__ void sgemm_thread_tiling(const float* A, const float* B, float* C,
                                    int M, int N, int K) {
    __shared__ float As[BM][BK];
    __shared__ float Bs[BK][BN];

    const int thread_num = BM / TM * BN / TN;
    int tid = threadIdx.x;
    int thread_row = (tid / (BN / TN)) * TM;
    int thread_col = (tid % (BN / TN)) * TN;

    // 寄存器存储
    float a_frag[TM];
    float b_frag[TN];
    float c_frag[TM][TN] = {0.0f};

    int by = blockIdx.y, bx = blockIdx.x;

    for (int bk = 0; bk < K; bk += BK) {
        // 协作加载 A、B 到 Shared Memory（省略边界检查）
        load_tile_A(A, As, by, bk, tid, M, K);
        load_tile_B(B, Bs, bx, bk, tid, K, N);
        __syncthreads();

        // 外积累加
        for (int k = 0; k < BK; k++) {
            // 从 Shared Memory 加载到寄存器
            for (int i = 0; i < TM; i++)
                a_frag[i] = As[thread_row + i][k];
            for (int j = 0; j < TN; j++)
                b_frag[j] = Bs[k][thread_col + j];

            // 寄存器上做外积
            for (int i = 0; i < TM; i++)
                for (int j = 0; j < TN; j++)
                    c_frag[i][j] += a_frag[i] * b_frag[j];
        }
        __syncthreads();
    }

    // 写回 Global Memory
    for (int i = 0; i < TM; i++)
        for (int j = 0; j < TN; j++)
            C[(by * BM + thread_row + i) * N + bx * BN + thread_col + j] = c_frag[i][j];
}
```

**外积 vs 内积**（新人重点理解）：

```
// 内积：每线程独立累加一个 C 元素 → 复用率低
for k: sum += A[row][k] * B[k][col]

// 外积：每线程负责 TM×TN 个 C 元素
for k:
    load A_frag[TM] from smem  // 读一次 A 的 TM 个元素
    load B_frag[TN] from smem  // 读一次 B 的 TN 个元素
    for i in TM:
        for j in TN:
            C_frag[i][j] += A_frag[i] * B_frag[j]  // TM×TN 次 FMA
```

外积把共享内存访问比值从 1 提升到 4（TM=TN=8 时 64/16=4）。

**性能：50%~65% cuBLAS**

### 4. 向量化访存（float4）

```cpp
#define FLOAT4(ptr) (reinterpret_cast<float4*>(&(ptr))[0])

// 一次加载 4 个连续 float
float4 tmp = FLOAT4(A[row * K + col]);
```

**转置 A 存储策略**（利用 float4 连续加载）：

```cpp
// 从 Global Memory 加载 A 的一行（连续），转置存入 Shared Memory
float4 a_val = FLOAT4(A[global_row * K + bk + load_col]);
As[load_col + 0][local_row] = a_val.x;
As[load_col + 1][local_row] = a_val.y;
As[load_col + 2][local_row] = a_val.z;
As[load_col + 3][local_row] = a_val.w;

// 从 Shared Memory 读取时可以用 float4（连续地址）
float4 a_frag_vec = FLOAT4(As[k][thread_row]);
```

**性能：60%~75% cuBLAS**（减少 75% load/store 指令）

### 5. Bank Conflict 消除

四种方案：

```cpp
// 方案一：Padding（tileA 转置存储）
__shared__ float As[BK][BM + 4];  // 错开 bank 映射

// 方案二：XOR Swizzle
As[k][row ^ (k * 4)] = value;           // 写入时 swizzle
float val = As[k][target_row ^ (k * 4)]; // 读取时反算

// 方案三：4×4 子块拆分（让 quarter-warp 内线程读取地址连续铺满 bank）

![8x8 计算区域拆分为 4x4 子块](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/gemm-7.png)

// 方案四：Z-order 线程映射
int lane_id = threadIdx.x % 32;
int lane_row = lane_id % 2 + (lane_id / 16) * 2;    // 4 行
int lane_col = (lane_id % 16) / 2;                  // 8 列
```

![Z-order 线程映射](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/gemm-8.png)

| 方案 | tileA transactions | tileB transactions | 合计 |
|------|-------------------|-------------------|------|
| 2×16 warp（基线） | 有冲突 | 有冲突 | 较差 |
| 4×8 warp + 1×8 QW | 2 | 8 | 10 |
| 4×8 warp + 4×4 子块 | 2 | 4 | 6 |
| 4×8 warp + Z-order | 2 | 2 | 4 |

![float4 下 warp(4x8) 的 Transaction 分析](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/gemm-5.png)

**性能：70%~80% cuBLAS**

### 6. Warp 级分块

![Warp 级分块形状](https://raw.githubusercontent.com/caomaolufei/AIInfraGuide/main/AIInfraGuide/images/gemm-4.png)

```cpp
int warp_id = tid >> 5;            // tid / 32
int lane_id = tid & 31;           // tid % 32

// warp 在 Block 中的位置（4×2 排列）
int warp_row = warp_id >> 1;      // M 方向：0~3
int warp_col = warp_id & 1;       // N 方向：0~1

// lane 在 warp 内的位置（4×8 排列，行主序）
int lane_row = lane_id >> 3;      // M 方向：0~3
int lane_col = lane_id & 7;       // N 方向：0~7

int row_c = (warp_row * 4 + lane_row) * TM;
int col_c = (warp_col * 8 + lane_col) * TN;
```

> 🔧 **为什么 4×8 优于 1×32**：Warp 形状影响 A/B 的访问效率。4×8 让 quarter-warp（8 线程）读 A 和 B 时都能触发广播，减少事务数。

**性能：80%~87% cuBLAS**

### 7. 双缓冲与流水线（Double Buffering）

**思路**：两套共享内存缓冲区，一边用 Buffer 0 计算，一边往 Buffer 1 加载。

```cpp
template <int BM, int BN, int BK, int TM, int TN>
__global__ void sgemm_double_buffer(const float* A, const float* B, float* C,
                                    int M, int N, int K) {
    __shared__ float As[2][BK][BM];  // 两个 buffer
    __shared__ float Bs[2][BK][BN];

    float a_frag[2][TM], b_frag[2][TN];
    float c_frag[TM][TN] = {0.0f};
    float ldg_a[4], ldg_b[4];  // 暂存 Global Memory 加载结果

    int buf_idx = 0;

    // Prologue: 加载第一个 tile
    load_global_to_shared(A, B, As[0], Bs[0], 0);
    __syncthreads();
    load_shared_to_reg(As[0], Bs[0], a_frag[0], b_frag[0], 0);

    for (int tile_k = BK; tile_k < K + BK; tile_k += BK) {
        int next_buf = 1 - buf_idx;

        #pragma unroll
        for (int k = 0; k < BK; k++) {
            // 预取下一个 k 步的寄存器数据
            if (k + 1 < BK) {
                load_shared_to_reg(As[buf_idx], Bs[buf_idx],
                                   a_frag[1 - k % 2], b_frag[1 - k % 2], k + 1);
            }

            // 在第一个 k 步时，发射下一 tile 的 Global Memory 加载
            if (k == 0 && tile_k < K) {
                issue_ldg(A, B, ldg_a, ldg_b, tile_k);
            }

            // 执行 TM×TN 的外积计算
            outer_product(a_frag[k % 2], b_frag[k % 2], c_frag);
        }

        // 将 LDG 结果写入下一 buffer
        if (tile_k < K) {
            store_reg_to_shared(ldg_a, ldg_b, As[next_buf], Bs[next_buf]);
        }
        __syncthreads();

        buf_idx = next_buf;
        load_shared_to_reg(As[buf_idx], Bs[buf_idx], a_frag[0], b_frag[0], 0);
    }

    store_c(c_frag, C);
}
```

> 🔧 **关键**：先发射 LDG 再做计算——LDG 是异步的，300+ cycle 延迟被 FFMA 计算隐藏。主循环只需一次 `__syncthreads()`。

**性能：90%~97% cuBLAS**

### 8. Ampere 异步拷贝（cp.async）

```cpp
#include <cuda/pipeline>
#include <cooperative_groups.h>

auto group = cooperative_groups::this_thread_block();
__shared__ cuda::pipeline_shared_state<cuda::thread_scope_block, 2> pipe_state;
auto pipe = cuda::make_pipeline(group, &pipe_state);

// 发射异步拷贝（Global → Shared，不经寄存器）
pipe.producer_acquire();
cuda::memcpy_async(group, smem_ptr, gmem_ptr, sizeof(float) * BK * BM, pipe);
pipe.producer_commit();

// 等待数据就绪
pipe.consumer_wait();
// ... 计算 ...
pipe.consumer_release();
```

> 🔧 **cp.async 优势**：传统方式是 Global→Register→Shared（两步），cp.async 直接 Global→Shared（一步），减少寄存器占用和中间步骤。

### 9. SASS 级优化（指令交错）

```sass
LDS R0, [R_addr]      // 发射 smem 读取，20-30 cycle 延迟
FFMA R10, R4, R5, R10 // 利用等待时间做计算（需足够多独立 FFMA）
FFMA R11, R4, R6, R11
FFMA R12, R4, R7, R12
...
FFMA R28, R2, R9, R28
// R0 此时已就绪
FFMA R14, R0, R5, R14 // 使用 LDS 结果
```

**写法优化**：

```cpp
// ❌ 慢：除法取模
int row = threadIdx.x / BN_TILE;
int col = threadIdx.x % BN_TILE;

// ✅ 快：位运算（BN_TILE 为 2 的幂）
int row = threadIdx.x >> 3;  // BN_TILE = 8
int col = threadIdx.x & 7;
```

```cpp
// ✅ #pragma unroll 正确写法（循环变量从 0 开始）
#pragma unroll
for (int i = 0; i < TM; i++) {
    int idx = threadIdx.x * TM + i;
    ...
}
```

**性能：95%~99% cuBLAS**

| SASS 手调 | 95%~99% | 寄存器调度 |

### 寄存器预算分析（TM=TN=8）

```
c_frag:    8×8 = 64 个寄存器
a_frag+b_frag: 8+8 = 16 个寄存器
索引/临时变量: ~30 个
合计: ~110-128 个寄存器/线程

V100: 65536 寄存器/SM
128 寄存器 × 256 线程/Block = 32768 寄存器/Block
每 SM 容纳 2 Block = 512 线程 = 16 Warps
Occupancy = 16/64 = 25%
```

> 💡 25% Occupancy 看似很低，但 GEMM 有极高 ILP（每线程 64 个独立 FMA 链），足以隐藏延迟。这正是 `3. Occupancy与资源分配` 说的"高 Occupancy 不等于高性能"。

## 面试回答（可直接口述的版本）

**问：如何优化 CUDA GEMM？**

答：GEMM 是 Compute-Bound 算子，优化核心是提升算术强度和隐藏延迟。第一步 Block Tiling：把 C 分成 128×128 的块，协作加载 A 和 B 的子块到共享内存，算术强度从 0.25 提升到 32，把 Memory-Bound 变成 Compute-Bound。第二步 Thread Tiling：每线程用寄存器负责 8×8 的子块，用外积代替内积——先加载 A 的 8 行和 B 的 8 列到寄存器，再做 64 次 FMA，共享内存复用率提升 4 倍。第三步 float4 向量化加载，一条指令读 16B，减少 75% 指令数。第四步消除 Bank Conflict，用 Padding 或 Swizzle。第五步 Double Buffering，两套缓冲区一边计算一边加载，LDG 延迟被 FFMA 隐藏。Ampere 架构还能用 cp.async 直接 Global→Shared 不经寄存器。最终能达到 95%+ cuBLAS 性能。注意 GEMM 故意用 25% 低 Occupancy 换高数据复用，因为 ILP 足够隐藏延迟。

## 深入追问（可能被追问的点）

- **Q1: 为什么 GEMM 用外积而 Reduce 不用？**
  A: GEMM 每线程计算 TM×TN 个输出，外积让 A 和 B 各加载一次复用 TM×TN 次。Reduce 是多输入单输出，没有矩阵乘法的外积结构。

- **Q2: Block Tiling 后算术强度 32 怎么算的？**
  A: BM×BN 个输出用 BM×BK + BK×BN 的输入数据，算术强度 = BM×BN×2×BK / ((BM×BK+BK×BN)×4) = BM×BN/(2×(BM+BN))。BM=BN=128 时为 32。

- **Q3: Double Buffering 为什么能隐藏延迟？**
  A: LDG 是异步的，发射后 300+ cycle 才返回数据。双缓冲在等 Buffer 0 数据时用 Buffer 1 计算，等计算时加载 Buffer 0，两个操作重叠。

- **Q4: cp.async 比传统 LDG+STS 好在哪？**
  A: 传统方式 Global→Register→Shared 占用寄存器且有两次访存。cp.async 直接 Global→Shared，减少寄存器占用，且是异步的能更好地与计算重叠。

- **Q5: 为什么 GEMM 的 Occupancy 只有 25% 却很快？**
  A: GEMM 每线程有 64 个独立 FMA 链（8×8 子块），ILP 极高，足以隐藏延迟。用更多寄存器换数据复用，减少的全局内存访问远超延迟隐藏的收益。

- **Q6: Wave 和 L2 Cache 有什么关系？**
  A: Wave 是 GPU 同时执行的 Block 集合。Block 调度顺序影响 L2 命中率——相邻 Block 共享 A/B 数据时 L2 命中率高。大矩阵可产生 10%+ 性能差异。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| Reduce vs GEMM | 前者 Memory-Bound(算术强度0.25)，后者 Compute-Bound(32) | Reduce 瓶颈带宽，GEMM 瓶颈算力 |
| Block Tiling vs Thread Tiling | 前者共享内存复用，后者寄存器复用 | Block=共享，Thread=寄存器 |
| 内积 vs 外积 | 内积每线程1输出，外积每线程TM×TN输出 | 外积复用率高 |
| Double Buffer vs cp.async | 前者软件双缓冲，后者硬件异步拷贝 | cp.async 是 Ampere 硬件特性 |
| Padding vs Swizzle | Padding 加列浪费空间，Swizzle 用 XOR 不浪费 | Padding 简单，Swizzle 高效 |
| 高 Occupancy vs 低 Occupancy | 前者延迟隐藏好，后者数据复用高 | GEMM 选低 Occupancy |

## 常见报错速查表

| ❌ 报错/问题 | 原因 | ✅ 解决 |
|-------------|------|--------|
| `too many resources requested for launch` | 寄存器超限 | 用 `__launch_bounds__` 限制，减小 TM/TN |
| 结果错误 | 边界检查遗漏或索引算错 | 检查 `if (r < M && c < N)` |
| 性能远低于预期 | Bank Conflict 或未向量化 | 用 ncu 检查，加 Padding/Swizzle |
| float4 对齐错误 | 指针未 16B 对齐 | 确保从对齐地址开始，处理偏移 |
| Double Buffer 死锁 | `__syncthreads()` 位置错 | 确保每轮迭代同步一次 |
| cp.async 编译错误 | 架构不支持或头文件缺失 | 需 sm_80+，`#include <cuda/pipeline>` |
| cuBLAS 对比链接失败 | 未链接 cublas 库 | 编译加 `-lcublas` |
