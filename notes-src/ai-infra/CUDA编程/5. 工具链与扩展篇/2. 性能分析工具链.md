## 一句话结论

性能优化的前提是**精准诊断**，不会用 Profiler 的优化都是盲猜。**Nsight Systems (nsys)** 看 CPU-GPU 全链路，找 GPU idle gap（Kernel Launch 过密、CPU 瓶颈、通信等待）；**Nsight Compute (ncu)** 下钻到单个 Kernel，看 SOL（Speed of Light）面板判断 Memory-Bound 还是 Compute-Bound；**nvcc --ptxas-options=-v** 编译时看寄存器/Shared Memory 占用；**cuobjdump/nvdisasm** 看 SASS 指令。**sudo 注意**：nsys/ncu 基础功能✅无需 sudo，但部分硬件性能计数器⚠️受 `NVreg_RestrictProfilingToAdminUsers` 限制，非 root 用户能跑但采集不到部分 counter。

## 背景与定位

- **这篇讲什么**：四大性能分析工具的安装、命令、输出解读、典型场景。重点讲 nsys 和 ncu 两个 GUI/CLI 双模工具。

## ⚠️ sudo 权限说明（重要，先读）

### 基础功能✅无需 sudo

| 工具 | 基础功能 | 是否需要 sudo |
|------|---------|--------------|
| `nsys` | 抓 CPU-GPU trace、Kernel 时间线、CUDA API 调用 | ❌ 不需要 |
| `ncu` | 看 SOL 面板、Memory/Compute 比例、Warp State | ❌ 不需要 |
| `ncu --set full` | 详细 Kernel 报告 | ❌ 不需要 |
| `nvcc --ptxas-options=-v` | 编译时资源分析 | ❌ 不需要 |
| `cuobjdump` / `nvdisasm` | 反汇编 PTX/SASS | ❌ 不需要 |

### 高级功能⚠️可能需要 root

NVIDIA 驱动有个内核参数 `NVreg_RestrictProfilingToAdminUsers`，默认值为 1（仅 admin 可用硬件性能计数器）。受此影响：

| 高级功能 | 是否受限制 | 表现 |
|---------|----------|------|
| `ncu --metrics` 采集**硬件性能计数器**（如 `smsp__inst_executed.sum`） | ⚠️ 可能受限 | 报错 `ERR_NVGPUCTRPERM` 或返回 0 |
| `ncu --target-processes all` 抓其他用户进程 | ⚠️ 需要 root | 报权限错误 |
| `nsys` 采集 NVTX range、CUDA Graph | ❌ 不受限 | 正常 |

### 三种解决方案（按推荐度排序）

**方案 1：装 sudo 时改驱动参数**（最彻底，但需要一次 root）：
```bash
# 编辑 GRUB 或 modprobe 配置
sudo bash -c 'echo "options nvidia NVreg_RestrictProfilingToAdminUsers=0" > /etc/modprobe.d/nvidia.conf'
sudo reboot
# 重启后所有用户都能用硬件计数器
```

**方案 2：给可执行文件加 CAP_SYS_ADMIN 能力**（不需要 root 改驱动）：
```bash
sudo setcap cap_sys_admin+ep $(which ncu)
sudo setcap cap_sys_admin+ep $(which nsys)
# 之后 ncu/ncu --metrics 不需要 sudo 也能跑
```

**方案 3：不用 sudo，避开需要计数器的 metric**（最实用）：
```bash
# ❌ 需要 sudo 的写法（采集特定硬件 counter）
ncu --metrics smsp__inst_executed.sum,sm__warps_active.avg.pct_of_peak_sustained_active ./my_app

# ✅ 不需要 sudo 的写法（用预设 set）
ncu --set full ./my_app              # 详细报告，用内置 metric 集合
ncu --set roofline ./my_app          # Roofline 图
ncu --set memory ./my_app            # Memory 分析
```

### 检测当前环境

```bash
# 检查驱动是否限制了 counter
cat /proc/driver/nvidia/params | grep RestrictProfiling
# 输出：RmProfilingAdminOnly = 1  ← 受限
# 或：  RmProfilingAdminOnly = 0  ← 不受限

# 测试 ncu 是否能采集 counter
ncu --metrics smsp__cycles_active.avg.pct_of_peak_sustained_active \
    --target-processes all ./my_app 2>&1 | grep -i "ERR_NVGPUCTRPERM"
# 如果有输出，说明受限
```

📌 **结论**：本笔记所有命令默认**无 sudo 环境**，能用 `--set full` 等预设集就别用 `--metrics`。

## 核心概念（新人友好讲解）

### 1. 四大工具的层次关系

```
应用层       │ 我的 CUDA 程序 / PyTorch 训练
            │
系统级 ◀── nsys（Nsight Systems）：CPU-GPU 全链路时间线
            │   ─ 找 GPU idle gap
            │   ─ 找 Kernel Launch 过密
            │   ─ 找 CPU 瓶颈
            │
Kernel 级 ◀── ncu（Nsight Compute）：单个 Kernel 深度分析
            │   ─ SOL 面板（Memory/Compute 比例）
            │   ─ Warp State（stall 原因）
            │   ─ Occupancy
            │   ─ Source View（哪行代码慢）
            │
编译时 ◀── nvcc --ptxas-options=-v：寄存器、Shared Memory 占用
            │   ─ 算 Occupancy 上限
            │   ─ 检查 spill 到 Local Memory
            │
指令级 ◀── cuobjdump / nvdisasm：看 PTX / SASS 指令
                ─ 看编译器生成了什么指令
                ─ 看是否用了 vector load / WMMA
```

**什么时候用哪个**：

| 症状 | 用哪个 |
|------|--------|
| 训练整体慢，不知道哪里慢 | nsys |
| GPU 利用率低（nvidia-smi 显示 50%） | nsys |
| 某个具体 Kernel 慢 | ncu |
| 想知道 Kernel 是 Memory-Bound 还是 Compute-Bound | ncu（SOL 面板） |
| 改了代码 Occupancy 变了 | nvcc --ptxas-options=-v |
| 怀疑编译器没生成 vector load | cuobjdump |

### 2. Nsight Systems (nsys)：系统级分析

#### 2.1 nsys 解决什么问题

**典型场景**：训练一个 Transformer，GPU 利用率只有 60%，剩下的 40% 时间 GPU 在干嘛？可能是：
- CPU 在准备数据（DataLoader 慢）
- CPU 在 Launch Kernel（Kernel Launch Overhead）
- GPU 在等通信（多卡 allreduce）
- Kernel 之间有空隙（前一个 Kernel 结束到下一个开始）

nsys 抓一份 trace，所有时间线一目了然。

#### 2.2 nsys 基础命令

```bash
# 抓一份 trace（CLI）
nsys profile -o my_profile ./my_cuda_app

# 抓 PyTorch 训练（带 Python 栈）
nsys profile -o pytorch_profile \
    --trace=cuda,nvtx,osrt,cudnn,cublas \
    python train.py

# 关键参数：
#   -o <name>        输出文件名（生成 .nsys-rep）
#   --trace=<...>    抓哪些 trace（cuda/nvtx/osrt/cudnn/cublas）
#   --duration=<s>   抓多少秒
#   --kill=<sig>     抓完发什么信号终止程序
#   --stats=true     抓完输出统计摘要到 stdout

# 看 trace（CLI 摘要）
nsys stats my_profile.nsys-rep
# 输出多个表格：CUDA API 统计、Kernel 统计、内存统计等

# 用 GUI 看（推荐）
# 把 .nsys-rep 文件下载到本地，用 Nsight Systems GUI 打开
```

#### 2.3 nsys 输出解读

**CLI 摘要示例**（`nsys stats`）：

```
Time (s)  Duration (%)  Name
────────  ────────────  ─────────────────────────────
1.234567       12.34    cudaMemcpyAsync
1.456789        8.76    kernel_softmax
1.567890       15.23    kernel_matmul
...
```

**关键看**：
1. **CUDA Kernel 时间线**：哪些 Kernel 占总时间长
2. **CUDA API 时间线**：`cudaMemcpy`、`cudaLaunchKernel` 的开销
3. **GPU Idle Gap**：两个 Kernel 之间的空白
4. **NVTX Range**：你在代码里标记的范围（见下文）

#### 2.4 用 NVTX 标记范围

NVTX（NVIDIA Tools Extension）让你在代码里标记一段范围，nsys trace 里会高亮显示。

```cpp
// C++/CUDA
#include <nvtx3/nvToolsExt.h>

void train_one_iteration() {
    nvtxRangePushA("forward");
    forward();
    nvtxRangePop();

    nvtxRangePushA("backward");
    backward();
    nvtxRangePop();

    nvtxRangePushA("optimizer_step");
    optimizer_step();
    nvtxRangePop();
}
```

```python
# Python（torch.profiler 内置 NVTX）
import torch
with torch.autograd.profiler.emit_nvtx():
    with torch.autograd.profiler.profile():
        train_one_iteration()
```

nsys 抓出来的 trace 里能直接看到「forward」「backward」「optimizer_step」三段，每段包含哪些 Kernel。

#### 2.5 GPU Idle Gap 的典型成因

| 现象 | 成因 | 解决 |
|------|------|------|
| Kernel 之间有空白，CPU API 调用密集 | Kernel Launch Overhead | 用 CUDA Graph 批量 Launch |
| Kernel 之间有空白，CPU 在 Python 解释器 | Python 解释器开销 | torch.compile |
| 等数据加载 | DataLoader 慢 | `num_workers` / `pin_memory` |
| 等通信 | 多卡 allreduce 慢 | 梯度分桶、计算通信重叠 |
| 等同步 | `cudaDeviceSynchronize` 滥用 | 异步化、去掉不必要的 sync |

### 3. Nsight Compute (ncu)：Kernel 级分析

#### 3.1 ncu 解决什么问题

**典型场景**：nsys 告诉你 `kernel_softmax` 占了 30% 时间，现在要让它变快。它是 Memory-Bound 还是 Compute-Bound？哪里慢？ncu 给出 Kernel 内部数据。

#### 3.2 ncu 基础命令

```bash
# 详细报告（推荐，无需 sudo）
ncu --set full -o my_kernel_report ./my_cuda_app

# 只分析特定 Kernel
ncu --kernel-name regex:softmax --set full -o softmax_report ./my_cuda_app

# 只抓前 N 次调用
ncu --launch-count 5 --kernel-name regex:matmul --set full ./my_cuda_app

# 抓 PyTorch 训练里的某个 Kernel
ncu --target-processes all \
    --kernel-name regex:softmax \
    --launch-skip 10 --launch-count 1 \
    --set full \
    -o softmax_report \
    python train.py

# CLI 看摘要
ncu --import my_kernel_report.ncu-rep --print-summary per-kernel
```

**关键参数**：
- `--set full`：详细报告（SOL、Warp State、Occupancy、Source 全部）
- `--set basic`：基础报告（轻量）
- `--set roofline`：Roofline 图
- `--kernel-name regex:<name>`：只抓特定 Kernel
- `--launch-skip N`：跳过前 N 次调用（避免 warmup）
- `--launch-count N`：只抓 N 次

#### 3.3 SOL 面板：判断 Memory/Compute Bound

**SOL（Speed of Light）面板**是 ncu 报告的第一块，也是最关键的指标。

```
✓ cuCUDA Kernel: softmax_kernel
  Duration: 1.23 ms
  Grid: (256, 1, 1)    Block: (128, 1, 1)

  Speed of Light:
    SM Frequency: 1.41 GHz
    Duration: 1.234 ms

    SOL Memory:    85.4%   ◀── 内存带宽利用率
    SOL Compute:   12.3%   ◀── 计算单元利用率
    SOL L1/TEX:    45.2%
    SOL L2:        78.9%
```

**判定规则**：

```
SOL Memory > 60%  且  SOL Compute < 30%   →  Memory-Bound
SOL Compute > 60% 且  SOL Memory < 30%    →  Compute-Bound
两者都 > 60%                              →  已经很好，难再优化
两者都 < 30%                              →  延迟限制（Latency-Bound）
```

**Softmax 的典型情况**：SOL Memory ~80%，SOL Compute ~15%，Memory-Bound。优化方向是减少内存访问（fuse 操作、向量化加载）。

**GEMM 的典型情况**：SOL Compute ~70%，SOL Memory ~30%，Compute-Bound。优化方向是提高算术强度（Tiling）。

#### 3.4 Memory 面板：内存带宽利用率

```
Memory:
  DRAM Frequency:     6.50 GHz   ◀── HBM2e 频率
  Theoretical BW:     1638 GB/s  ◀── 理论带宽（A100）

  Total Bytes Read:   12.3 GB
  Total Bytes Write:  6.1 GB
  Total Bytes:        18.4 GB

  Achieved BW:        1489 GB/s  ◀── 实际带宽
  BW Utilization:     90.9%      ◀── 利用率
```

**判定**：
- BW Utilization > 80% → 已经吃满带宽，优化空间小
- BW Utilization < 50% → 访问模式有问题（Bank Conflict、非合并访问）

#### 3.5 Warp State 面板：stall 原因

```
Warp State:
  Active Cycles:      1.2e9
  Eligible Warps:     2.3  ◀── 平均每周期可执行的 Warp 数（理想接近 4）

  Stall Reasons (avg cycles per warp):
    Stall Memory:      45.6%   ◀── 等内存
    Stall Compute:     12.3%   ◀── 等计算单元
    Stall Barrier:     8.7%    ◀── 等 __syncthreads
    Stall Not Selected: 5.4%   ◀── Warp 就绪但没被选
    Stall Wait:        3.2%    ◀── 等长延时操作（如 ld.global）
    ...
```

**Stall 原因解读**：

| Stall 原因 | 含义 | 优化方向 |
|-----------|------|---------|
| `Stall Memory` | 等内存加载完成 | 提高数据复用、向量化、prefetch |
| `Stall Barrier` | 等 `__syncthreads` | 减少 barrier、用 warp-level ops |
| `Stall Wait` | 等长延时操作（global memory load） | 用 Shared Memory 缓存 |
| `Stall Not Selected` | Warp 就绪但调度器没选它 | 提高 Occupancy |
| `Stall Short Scoreboard` | 等共享内存或 L1（短延时） | 减少共享内存竞争 |

#### 3.6 Occupancy 面板

```
Occupancy:
  Theoretical Active Warps:    48 / 64  ◀── 理论上限（受寄存器/Shared Mem 限制）
  Achieved Active Warps:       42 / 64  ◀── 实际值
  Theoretical Occupancy:       75%
  Achieved Occupancy:          65.6%
```

**重要认知**：
- **理论 Occupancy** 由 Kernel 用的寄存器数和 Shared Memory 决定（用 `nvcc --ptxas-options=-v` 看）
- **实际 Occupancy** 通常小于理论值（受 warp 调度、block 大小影响）
- **Occupancy 高不等于性能好**——GEMM 故意用低 Occupancy（25%）换高数据复用

#### 3.7 Source View：哪行代码慢

ncu 能关联到源代码，告诉你哪一行 CUDA 代码耗时最高：

```
Source View (kernel_softmax):
  Line 23:  row = tl.load(input + offsets, mask=mask);    ◀── 45% 时间
  Line 26:  row_max = tl.max(row, axis=0);                ◀── 12% 时间
  Line 27:  numerator = tl.exp(row - row_max);            ◀── 8% 时间
  Line 30:  tl.store(output + offsets, result, mask=mask);◀── 25% 时间
```

**用法**：编译时加 `-lineinfo`（保留下源码信息）：
```bash
nvcc -O3 -lineinfo my_kernel.cu -o my_kernel
# 之后 ncu 报告能关联到源码行号
```

### 4. nvcc --ptxas-options=-v：编译时资源分析

#### 4.1 命令

```bash
nvcc -O3 --ptxas-options=-v my_kernel.cu -o my_kernel
# 或
nvcc -O3 -Xptxas -v my_kernel.cu -o my_kernel
```

#### 4.2 输出解读

```
ptxas info    : Compiling entry function 'softmax_kernel' for 'sm_80'
ptxas info    : Function properties for softmax_kernel
                0 bytes stack frame, 0 bytes spill stores, 0 bytes spill loads
ptxas info    : Used 24 registers, 4096 bytes smem, 32 bytes cmem[0]
```

**关键指标**：

| 字段 | 含义 | 看什么 |
|------|------|--------|
| `Used N registers` | 每个 Thread 用 N 个寄存器 | 太大会降低 Occupancy |
| `bytes smem` | 每个 Block 用多少 Shared Memory | 太大会降低 Occupancy |
| `bytes spill stores / spill loads` | 寄存器溢出到 Local Memory | ❌ 不应该有，有就要优化 |
| `bytes stack frame` | 栈空间 | 复杂函数才有 |
| `bytes cmem[0]` | 常量内存 | 一般是参数和字面量 |

#### 4.3 用寄存器数算 Occupancy

以 A100（SM 80）为例：
- 每 SM 65536 个寄存器
- 每 SM 最多 64 个 Warp（2048 个 Thread）

```
Kernel 用 24 寄存器/Thread：
  Warp 用 24 × 32 = 768 寄存器
  每 SM 可放 65536 / 768 = 85 个 Warp
  但受 64 Warp/SM 上限 → 实际 64 Warp
  Occupancy = 64 / 64 = 100%

Kernel 用 80 寄存器/Thread：
  Warp 用 80 × 32 = 2560 寄存器
  每 SM 可放 65536 / 2560 = 25 个 Warp
  Occupancy = 25 / 64 = 39%
```

#### 4.4 spill 的处理

**spill**：寄存器不够用，编译器把数据存到 Local Memory（实际是 DRAM），性能大幅下降。

**判断**：
```bash
nvcc --ptxas-options=-v my_kernel.cu -o my_kernel
# 如果看到 "32 bytes spill stores, 32 bytes spill loads" → 发生 spill
```

**解决**：
1. 减少 Kernel 内的局部变量
2. 把 `if-else` 分支合并（减少活跃变量）
3. 用 `--maxrregcount=N` 限制寄存器（但会触发 spill，反向操作）
4. 把 Kernel 拆成多个小 Kernel

### 5. cuobjdump / nvdisasm：指令级分析

#### 5.1 cuobjdump：看 PTX 和 SASS

```bash
# 列出可执行文件里的所有 Kernel
cuobjdump --list-elf my_kernel

# 看 PTX（中间表示）
cuobjdump --dump-ptx my_kernel

# 看 SASS（实际 GPU 指令）
cuobjdump --dump-sass my_kernel
```

#### 5.2 nvdisasm：反汇编 SASS

```bash
# 反汇编 SASS（更详细）
nvdisasm -S my_kernel.cubin

# 打印控制流图
nvdisasm -cfg my_kernel.cubin
```

#### 5.3 PTX vs SASS

```
源代码 → PTX（虚拟 ISA，跨架构） → SASS（实际指令，特定架构）
                  ↑                          ↑
            cuobjdump --dump-ptx       cuobjdump --dump-sass
```

**PTX 示例**：
```
.visible .entry softmax_kernel(
    .param .u64 softmax_kernel_param_0,
    ...
)
{
    .reg .f32 %f<5>;
    .reg .b32 %r<3>;
    .reg .b64 %rd<8>;

    ld.param.u64 %rd1, [softmax_kernel_param_0];
    cvta.to.global.u64 %rd2, %rd1;
    ...
    ld.global.f32 %f1, [%rd2];
    ...
}
```

**SASS 示例**（A100）：
```
/*0000*/    IADD3 R1, R0, c[0x0][0x28], RZ ;     /* 加法 */
/*0010*/    LDG.E.128 R4, [R2] ;                 /* 128位 global load（float4） */
/*0020*/    IMMA.8816.S8.S8 R8, R4, R6, R10 ;    /* Tensor Core 指令 */
```

#### 5.4 什么时候看 SASS

- 验证编译器是否生成了 vector load（`LDG.E.128` = float4）
- 验证是否用了 Tensor Core（`IMMA` / `HMMA` 指令）
- 排查编译器是否做了奇怪的优化

## 动手实践

### 实验 1：用 nsys 找 GPU idle gap

**前置**：装好 Nsight Systems（CUDA Toolkit 自带）。

**测试代码**：
```python
# nsys_demo.py
import torch
import time

def bad_pipeline():
    """Kernel 之间有大段空隙"""
    x = torch.randn(1024, 1024, device='cuda')
    results = []
    for _ in range(100):
        # 每次都要 CPU 决策 + Launch，有空隙
        y = torch.relu(x)
        z = torch.sigmoid(y)
        w = torch.tanh(z)
        results.append(w)
    return results

torch.cuda.synchronize()
t0 = time.perf_counter()
bad_pipeline()
torch.cuda.synchronize()
print(f"耗时: {(time.perf_counter() - t0) * 1000:.2f} ms")
```

**抓 trace**：
```bash
nsys profile -o nsys_demo \
    --trace=cuda,nvtx \
    --force-overwrite=true \
    python nsys_demo.py
```

**看摘要**：
```bash
nsys stats nsys_demo.nsys-rep
```

**预期输出**（关键部分）：
```
CUDA Kernel Statistics:
  Time (%)  Total Time   Instances   Avg         Min         Max         Name
  ────────  ───────────  ──────────  ──────────  ──────────  ──────────  ──────────
     35.2    4.234 ms         100     0.042 ms    0.038 ms    0.051 ms    void at::native::...
     33.1    3.981 ms         100     0.040 ms    0.036 ms    0.048 ms    void at::native::...
```

**用 GUI 看**：把 `nsys_demo.nsys-rep` 下载到本地，用 Nsight Systems GUI 打开，能看到：
- CPU 时间线（Python 解释器在跑）
- GPU 时间线（Kernel 一个接一个）

**优化前后对比**：用 `torch.compile` 编译后空隙会消失。

### 实验 2：用 ncu 分析 Softmax Kernel

**测试代码**：
```python
# ncu_demo.py
import torch

def softmax_naive(x):
    """eager softmax，多次读写"""
    row_max = x.max(dim=1, keepdim=True)[0]
    exp_x = torch.exp(x - row_max)
    row_sum = exp_x.sum(dim=1, keepdim=True)
    return exp_x / row_sum

x = torch.randn(4096, 4096, device='cuda')

# Warmup
for _ in range(10):
    _ = softmax_naive(x)
torch.cuda.synchronize()

# Benchmark
for _ in range(50):
    _ = softmax_naive(x)
torch.cuda.synchronize()
```

**抓 ncu 报告**：
```bash
ncu --target-processes all \
    --kernel-name regex:softmax \
    --launch-skip 10 --launch-count 1 \
    --set full \
    -o softmax_report \
    --force-overwrite=true \
    python ncu_demo.py
```

**CLI 看摘要**（无需 sudo）：
```bash
ncu --import softmax_report.ncu-rep \
    --print-summary per-kernel
```

**预期输出**（关键部分）：
```
Softmax Kernel Report:
  Duration: 0.42 ms
  Theoretical Occupancy: 75.0%
  Achieved Occupancy:    72.3%

  Speed of Light:
    SOL Memory:    82.3%   ◀── Memory-Bound
    SOL Compute:   15.6%

  Memory:
    Achieved BW:    1492 GB/s
    BW Utilization: 91.0%

  Warp State:
    Stall Memory:      52.3%   ◀── 主要 stall 原因
    Stall Not Selected: 8.1%
```

**判定**：
- SOL Memory 82% >> SOL Compute 15% → Memory-Bound
- BW Utilization 91% → 接近带宽极限
- Stall Memory 52% → 主要在等内存

**优化方向**：写一个 fused softmax（Triton 或手写 CUDA）一次读一行，融合 max+exp+sum+div，减少读写次数。

### 实验 3：用 ptxas 看 GEMM 优化前后寄存器变化

**测试代码**（参考 `2. CUDA-GEMM算子性能优化算子性能优化.md`）：
```bash
# 编译朴素 GEMM
nvcc -O3 -lineinfo --ptxas-options=-v naive_gemm.cu -o naive_gemm

# 编译优化 GEMM（Block Tiling + 寄存器外积）
nvcc -O3 -lineinfo --ptxas-options=-v opt_gemm.cu -o opt_gemm
```

**朴素 GEMM 输出**：
```
ptxas info : Used 16 registers, 0 bytes smem, 0 bytes spill stores
```

**优化 GEMM 输出**：
```
ptxas info : Used 124 registers, 8192 bytes smem, 0 bytes spill stores
```

**对比分析**：
- 朴素版寄存器少（16），Shared Memory 0，Occupancy 高但性能差
- 优化版寄存器多（124），Shared Memory 8KB，Occupancy 低（25%）但性能好
- **GEMM 优化的反直觉**：用低 Occupancy 换高数据复用，总性能更高

## 面试回答

### Q1：你用过哪些 CUDA 性能分析工具？

**标准回答（30 秒）**：

主要用四类：
1. **Nsight Systems (nsys)**：系统级，抓 CPU-GPU 全链路 trace，找 GPU idle gap 和 Kernel Launch Overhead
2. **Nsight Compute (ncu)**：Kernel 级，看 SOL 面板判断 Memory/Compute Bound，看 Warp State 找 stall 原因
3. **nvcc --ptxas-options=-v**：编译时看寄存器和 Shared Memory 占用，算理论 Occupancy
4. **cuobjdump / nvdisasm**：看 PTX 和 SASS 指令，验证编译器生成的指令

典型优化流程：先用 nsys 找最慢的 Kernel，再用 ncu 分析这个 Kernel 是 Memory-Bound 还是 Compute-Bound，根据瓶颈类型针对性优化，最后用 ncu 对比优化前后报告。

### Q2：怎么判断一个 Kernel 是 Memory-Bound 还是 Compute-Bound？

**标准回答**：

用 Nsight Compute 的 SOL（Speed of Light）面板：
- `SOL Memory > 60%` 且 `SOL Compute < 30%` → Memory-Bound（如 Softmax、LayerNorm）
- `SOL Compute > 60%` 且 `SOL Memory < 30%` → Compute-Bound（如优化后的 GEMM）
- 两者都 < 30% → Latency-Bound（通常 Occupancy 太低或 stall 严重）

也可以用 Roofline 模型：算 Kernel 的算术强度（FLOP/Byte），对比硬件平衡点（A100 约 74 FLOP/Byte for FP16）。低于平衡点 Memory-Bound，高于则 Compute-Bound。

### Q3：用 nsys/ncu 时遇到权限问题怎么办？

**标准回答**：

nsys/ncu 基础功能✅不需要 sudo，但采集硬件性能计数器（`--metrics`）会受 NVIDIA 驱动参数 `NVreg_RestrictProfilingToAdminUsers` 限制，默认值为 1，报错 `ERR_NVGPUCTRPERM`。

三种解决：
1. 改驱动参数 `NVreg_RestrictProfilingToAdminUsers=0`（需要一次 root + 重启）
2. `sudo setcap cap_sys_admin+ep $(which ncu)` 给 ncu 加能力
3. **不用 sudo**：避开 `--metrics`，用 `--set full` / `--set roofline` 等预设集（涵盖 90% 场景）

## 深入追问

### Q1：SOL 面板的 Memory 和 Compute 比例为什么不是 1:1？

因为 GPU 的算力和带宽不是 1:1。A100 FP16 算力 312 TFLOPS，HBM 带宽 2 TB/s，平衡点 = 312/2 = 156 FLOP/Byte。一个 Kernel 算术强度 50 FLOP/Byte 时，受限于带宽（只能跑 100 TFLOPS），SOL Compute 显示 32%，SOL Memory 显示 80%。

### Q2：Occupancy 高为什么不一定性能好？

Occupancy 是「SM 上活跃 Warp 数 / 最大 Warp 数」。高 Occupancy 意味着调度器有更多 Warp 可选，能隐藏延迟。但：
1. 如果 Kernel 不是 Latency-Bound（如 Compute-Bound 的 GEMM），高 Occupancy 没用
2. 高 Occupancy 往往意味着每个 Thread 用的资源少（寄存器/Shared Memory），可能数据复用差
3. GEMM 故意用 25% Occupancy 换更多寄存器做 Thread Tiling，性能反而更好

### Q3：nsys 和 ncu 能同时用吗？

不能同时抓，但可以串行：
```bash
# 先用 nsys 找慢 Kernel
nsys profile -o step1 ./my_app
# 看报告，找到最慢的 Kernel 名字

# 再用 ncu 分析这个 Kernel
ncu --kernel-name regex:<慢Kernel名> --set full -o step2 ./my_app
```

### Q4：Stall Not Selected 是什么？怎么优化？

意思是「Warp 就绪，但调度器选了别的 Warp」。原因通常是 Occupancy 太低，调度器没有足够的 Warp 可选。优化方向是提高 Occupancy（减少寄存器、缩小 Block 尺寸、用 `--maxrregcount`）。

但如果 Kernel 是 Compute-Bound，Stall Not Selected 高也无所谓——说明计算单元已经满了。

### Q5：ptxas 报了 spill，怎么定位是哪个变量溢出？

ptxas 不直接告诉你哪个变量溢出。定位方法：
1. 编译时加 `-Xptxas -dlcm=ca -v`，看 spill 大小
2. 拆分 Kernel，把内层循环抽出来，看哪个 Kernel 还 spill
3. 减少 Kernel 内的局部变量（用 array 替代多个 scalar）
4. 用 `--ptxas-options=-O3` 启用激进优化（可能减少 spill）

## 易混淆点对比

### 1. nsys vs ncu

| 维度 | nsys（Nsight Systems） | ncu（Nsight Compute） |
|------|----------------------|---------------------|
| 层次 | 系统级（CPU-GPU 全链路） | Kernel 级 |
| 时间范围 | 整个程序（秒到分钟） | 单次 Kernel 调用（毫秒） |
| 主要问题 | GPU idle gap、Kernel Launch Overhead | Memory/Compute Bound、stall 原因 |
| 输出 | 时间线 trace（.nsys-rep） | Kernel 详细报告（.ncu-rep） |
| 性能开销 | 小（<5%） | 大（10–100x，因为单 Kernel 反复跑采集 counter） |
| 典型场景 | 「训练整体慢」「GPU 利用率低」 | 「这个 Kernel 怎么优化」 |

### 2. PTX vs SASS

| 维度 | PTX | SASS |
|------|-----|------|
| 全称 | Parallel Thread Execution | Streaming Assembler |
| 层次 | 虚拟 ISA | 实际硬件指令 |
| 跨架构 | ✅ 是（编译到不同 sm_xx） | ❌ 否（每个架构一份） |
| 可读性 | 较好 | 较差 |
| 用途 | 学习编译器优化思路 | 验证实际指令（如 Tensor Core） |
| 工具 | `cuobjdump --dump-ptx` | `cuobjdump --dump-sass` |

### 3. ncu 的 `--set` 选项

| Set | 内容 | 用途 |
|-----|------|------|
| `--set basic` | 基础指标 | 快速概览 |
| `--set full` | 完整报告（SOL + Warp + Occupancy + Source） | 详细分析（推荐） |
| `--set roofline` | Roofline 图 | 算术强度可视化 |
| `--set memory` | Memory 详细分析 | 内存瓶颈 |
| `--set compute` | Compute 详细分析 | 计算瓶颈 |

## 常见报错速查表

| 报错 / 现象 | 原因 | 解决 |
|------------|------|------|
| `ERR_NVGPUCTRPERM` | 受 `NVreg_RestrictProfilingToAdminUsers` 限制 | 用 `--set full` 代替 `--metrics`，或改驱动参数 |
| `No kernels were reprofiled` | `--kernel-name regex:` 没匹配到 | 检查 Kernel 名字，用 `ncu --list-kernels` 列出 |
| `Target process (PID xxx) is not allowed` | `--target-processes all` 需要 root | 加 `--target-processes all` 前确认权限，或只 profile 自己的进程 |
| `==WARNING== No CUPTI events found` | 没抓到任何 CUDA 活动 | 检查程序是否真的调用了 CUDA（不要只跑 CPU 部分） |
| `nvcc: ptxas fatal: ...` | ptxas 编译失败 | 检查 `--ptxas-options` 参数语法 |
| spill stores > 0 | 寄存器溢出 | 减少局部变量、`--maxrregcount` |
| nsys 抓不到 Python 栈 | 没装 PyTorch NVTX | 用 `torch.autograd.profiler.emit_nvtx()` |
| ncu 抓 PyTorch Kernel 失败 | `--target-processes` 没设 | 加 `--target-processes all` |
| Source View 没源码 | 编译时没加 `-lineinfo` | `nvcc -lineinfo ...` |
| nsys 报告打开是空的 | 程序运行时间太短 | 用 `--duration` 或加长训练循环 |
