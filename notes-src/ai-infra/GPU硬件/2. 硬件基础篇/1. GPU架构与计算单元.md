## 一句话结论

GPU 用「少量控制单元 + 海量 ALU」换吞吐，硬件按 **GPC → TPC → SM → Warp** 层级组织——SM 是调度基本单位、Warp（32 线程）是执行最小单位，Tensor Core 是 AI 加速核心引擎。理解这三层就理解了 GPU 为什么快。

## 核心概念（新人友好讲解）

### 1. CPU vs GPU：为什么深度学习选择了 GPU

#### 1.1 架构哲学的根本差异

想象你要批改 10000 份选择题试卷。CPU 的做法是请一位经验丰富的阅卷老师，他速度快、能处理各种复杂题型，但一次只能改一份；GPU 的做法是请 10000 个临时工，每人只会对答案这一件事，但所有人同时开工——1 秒就全部改完了。

这就是**延迟导向**（Latency-oriented）与**吞吐导向**（Throughput-oriented）的核心区别：

| 对比维度 | CPU | GPU |
|----------|-----|-----|
| 设计目标 | 低延迟处理复杂任务 | 高吞吐处理大量简单任务 |
| 核心数量 | 几个到几十个"大核" | 数千到上万个"小核" |
| 单核能力 | 强（复杂分支预测、乱序执行） | 弱（简单 ALU，按序执行） |
| 缓存占比 | 芯片面积的大部分 | 芯片面积的小部分 |
| 控制逻辑 | 复杂（乱序执行、分支预测器） | 简单（大量核心共享控制单元） |
| 内存带宽 | 较低（DDR5 ~100 GB/s） | 极高（HBM3 ~3 TB/s） |
| 适合任务 | 串行逻辑、操作系统、网络服务 | 矩阵运算、数据并行、图形渲染 |

用芯片面积分配来直观感受：

```
CPU 芯片面积分配                    GPU 芯片面积分配
┌──────────────────────┐          ┌──────────────────────┐
│  ████████████  控制逻辑  │          │  ██  控制逻辑            │
│  ████████████  +缓存    │          │  ██  +缓存              │
│  ████  ALU              │          │  ████████████████  ALU  │
│  ████  (少量计算单元)    │          │  ████████████████  (海量)│
└──────────────────────┘          └──────────────────────┘
  少量强核 + 大缓存                  海量简单核 + 小缓存
  → 擅长复杂串行任务                 → 擅长简单并行任务
```

#### 1.2 深度学习为什么天然适合 GPU

深度学习拆到最底层就是海量的矩阵乘法：

- **前向传播**：每一层 `Y = XW + b`，本质是矩阵乘法
- **反向传播**：计算梯度依然是矩阵乘法
- **Attention**：`softmax(QK^T / √d_k)V`，核心还是矩阵乘法

这些运算有两个关键特征：
1. **数据并行度极高**：矩阵中每个元素的计算相互独立，天然可分配给不同核心
2. **计算模式规则**：不需要复杂分支判断，所有线程执行相同指令

这恰好是 GPU 的长项。

> 💡 **数量级对比**：一块 H100 GPU 的 FP16 算力约 989 TFLOPS，而高端服务器 CPU 的 FP16 算力通常在个位数 TFLOPS 级别。差距在两个数量级以上。

### 2. GPU 硬件架构剖析：GPC → TPC → SM → Warp

![GPU hardware architecture diagram](./1. GPU%E6%9E%B6%E6%9E%84%E4%B8%8E%E8%AE%A1%E7%AE%97%E5%8D%95%E5%85%83.assets/GPU%20hardware%20architecture%20diagram.png)

NVIDIA GPU 的计算核心按层级组织，就像一家大公司的组织架构：

```
GPU 芯片
├── GPC (Graphics Processing Cluster)  ← "事业部"
│   ├── TPC (Texture Processing Cluster)  ← "部门"
│   │   ├── SM (Streaming Multiprocessor)  ← "小组"，GPU 的基本调度单元
│   │   │   ├── CUDA Core × N  ← "组员"，执行 FP32/INT32 运算
│   │   │   ├── Tensor Core × M  ← "专家"，执行矩阵乘加
│   │   │   ├── SFU (Special Function Unit)  ← 执行 sin/cos/exp 等
│   │   │   ├── Register File  ← 寄存器堆（速度最快的存储）
│   │   │   ├── Shared Memory  ← 共享内存（SM 内所有线程共享）
│   │   │   └── L1 Cache
│   │   └── SM ...
│   └── TPC ...
├── GPC ...
├── L2 Cache（全局共享）
└── Memory Controller → HBM（显存）
```

#### 2.1 新人术语速查

| 术语 | 全称 | 一句话解释 |
|------|------|-----------|
| **GPC** | Graphics Processing Cluster | GPU 顶层划分，一个芯片有多个 GPC |
| **TPC** | Texture Processing Cluster | GPC 下的子单元，含多个 SM |
| **SM** | Streaming Multiprocessor | GPU 基本调度单元，所有线程调度在此发生 |
| **CUDA Core** | — | 通用计算核心，一个周期执行一次标量乘加（FMA） |
| **Tensor Core** | — | 矩阵乘加专用单元，一个周期完成一个小矩阵的 D=A×B+C |
| **SFU** | Special Function Unit | 执行超越函数（sin/cos/exp/sqrt）的单元 |
| **Warp** | — | 32 个连续线程，GPU 执行的最小单位 |
| **SIMT** | Single Instruction, Multiple Threads | NVIDIA 的执行模型：同一条指令驱动多个线程 |

#### 2.2 SM：GPU 的基本调度单元

SM 是理解 GPU 架构的关键。以 H100 的 SM 为例：

| 组件 | 数量 | 功能 |
|------|------|------|
| CUDA Core（FP32） | 128 | 执行浮点和整数运算 |
| Tensor Core（第四代） | 4 | 执行矩阵乘累加（MMA）运算 |
| Load/Store Unit | 32 | 负责内存读写 |
| SFU | 16 | 执行超越函数 |
| Warp Scheduler | 4 | 每周期调度 4 个 Warp |
| Register File | 256 KB | 线程私有寄存器 |
| Shared Memory / L1 Cache | 228 KB（可配置） | SM 内共享的高速存储 |

> 📌 **关键点**：SM 是资源分配的最小粒度。编写 CUDA Kernel 时，一个 Thread Block 会被调度到一个 SM 上执行。SM 内的寄存器、共享内存由这个 SM 上的所有 Block 共同分配。

#### 2.3 Warp：GPU 执行的最小单位

GPU 不是一个线程一个线程执行的，而是以 **Warp**（线程束）为单位，32 个线程锁步执行同一条指令。就像军队齐步走——32 个士兵听同一声号令，迈出同一只脚。

```
Warp（32 个线程）
├── Thread 0:  执行 add r1, r2, r3
├── Thread 1:  执行 add r1, r2, r3  (同一条指令，不同数据)
├── ...
└── Thread 31: 执行 add r1, r2, r3
```

这种执行模式叫做 **SIMT**（Single Instruction, Multiple Threads）。

> ⚠️ **Warp Divergence**：如果 Warp 内的线程遇到分支（if-else），走不同分支的线程会被掩码（mask），导致部分线程空转。这是 GPU 编程中需要极力避免的性能杀手。详见 `../../CUDA编程/3. 优化篇/1. Warp与执行模型.md`。

#### 2.4 软件层次如何映射硬件

CUDA 编程模型的 Grid/Block/Thread 映射到硬件的 GPC/SM/Core：

| 软件层次（CUDA） | 硬件层次 | 说明 |
|-----------------|---------|------|
| Grid | 整个 GPU（所有 GPC） | 一次 kernel 启动的所有线程 |
| Block | 一个 SM | Block 被调度到一个 SM 上执行 |
| Thread | CUDA Core | 最小执行单元 |
| Warp | SM 的 Warp Scheduler | 32 个连续线程，硬件隐式分组 |

> 💡 **深入索引计算和 Block 大小选择**：详见 `../../CUDA编程/2. 基础篇/2. CUDA编程模型.md`。本系列只讲硬件视角的大框架，不展开编程细节。

### 3. Tensor Core：AI 加速的核心引擎

#### 3.1 CUDA Core vs Tensor Core

CUDA Core 是通用计算核心，一个时钟周期执行一次 FMA（fused multiply-add，融合乘加）运算。Tensor Core 是专门为矩阵运算设计的加速单元——一个时钟周期可以完成一个小矩阵的乘累加。

```
CUDA Core（标量计算器）：
  一个周期：c = a × b + c    （1 次乘加）

Tensor Core（矩阵乘加专用电路）：
  一个周期：D[16×8] = A[16×16] × B[16×8] + C[16×8]
  = 16 × 8 × 16 = 2048 次乘加！
```

> 🔧 **MMA（Matrix Multiply-Accumulate）**：Tensor Core 执行的基本操作 `D = A × B + C`，其中 A、B、C、D 是小尺寸矩阵。具体尺寸取决于精度和架构代次。

#### 3.2 精度格式演进

不同精度格式在算力、精度、显存占用之间做了不同取舍：

| 精度 | 位宽 | 动态范围 | 首次 Tensor Core 支持 | 典型应用 |
|------|------|---------|---------------------|---------|
| FP32 | 32 bit | 高 | — | 基准精度、loss 累加 |
| TF32 | 19 bit | 同 FP32 | Ampere（A100） | 训练（Ampere 上默认 FP32 路径） |
| FP16 | 16 bit | 较窄 | Volta（V100） | 混合精度训练（需 loss scaling） |
| BF16 | 16 bit | 同 FP32 | Ampere（A100） | 混合精度训练（推荐，无需 loss scaling） |
| FP8（E4M3/E5M2） | 8 bit | 可选 | Hopper（H100） | 训练 + 推理 |
| INT8 | 8 bit | 整数 | Turing（T4） | 推理量化 |
| FP4 | 4 bit | 极窄 | Blackwell（B200） | 推理量化 |

> 💡 **BF16 vs FP16**：都是 16 位，但 BF16 保留了和 FP32 相同的 8 位指数位，因此动态范围更大，训练时更不容易溢出。现代大模型训练几乎都用 BF16 而非 FP16。

精度位宽结构对比：

```
FP32:  [1 符号] [8 指数] [23 尾数]  = 32 bit  范围大、精度高
TF32:  [1 符号] [8 指数] [10 尾数]  = 19 bit  范围同 FP32、精度同 FP16
BF16:  [1 符号] [8 指数] [7  尾数]  = 16 bit  范围同 FP32、精度低
FP16:  [1 符号] [5 指数] [10 尾数]  = 16 bit  范围窄、精度高
FP8:   [1 符号] [4或5指数][3或2尾数] = 8 bit  范围/精度可选
```

#### 3.3 Tensor Core 的算力倍增

以 H100 SXM 为例，Tensor Core 对比 CUDA Core 的性能提升：

| 精度 | CUDA Core 算力 | Tensor Core 算力 | 加速比 |
|------|---------------|-----------------|--------|
| FP32 | 67 TFLOPS | 495 TFLOPS（TF32） | ~7.4× |
| FP16 | 134 TFLOPS | 989 TFLOPS | ~7.4× |
| FP8 | — | 1,979 TFLOPS | — |

> 📌 **关键点**：要触发 Tensor Core 加速，矩阵的维度必须满足对齐要求（通常是 8 或 16 的倍数）。如果模型的 hidden_size 不是 8 的倍数，Tensor Core 可能无法被充分利用。

#### 3.4 精度演进时间线

```
Volta(2017)    Turing(2018)    Ampere(2020)     Hopper(2022)    Blackwell(2024)
   │               │                │                │                │
 FP16           INT8            TF32              FP8              FP4
                INT4            BF16          Transformer Engine   双芯
                                                              第五代 TC
```

每代架构的核心精度突破详见 `../3. 架构演进篇/2. NVIDIA-GPU架构演进史.md`。

## 面试回答（可直接口述的版本）

**问：为什么深度学习选择 GPU 而不是 CPU？**

答：核心是架构哲学的差异。CPU 是延迟导向——少量强核、大缓存、复杂分支预测，擅长串行复杂任务；GPU 是吞吐导向——海量简单核、小缓存、多线程隐藏延迟，擅长并行简单任务。深度学习的计算核心是矩阵乘法，数据并行度高、计算模式规则、分支少，天然适合 GPU 的大规模并行架构。一块 H100 的 FP16 算力约 989 TFLOPS，比高端 CPU 高两个数量级。

**问：SM 和 Warp 分别是什么？为什么 Warp 是 32？**

答：SM（Streaming Multiprocessor）是 GPU 的基本调度单元，内部包含 CUDA Core、Tensor Core、寄存器、共享内存等资源。一个 Thread Block 被调度到一个 SM 上执行。Warp 是 GPU 执行的最小单位，32 个连续线程锁步执行同一条指令（SIMT 模型）。Warp 是 32 是 NVIDIA 的硬件设计选择——32 是平衡了调度开销和并行度的折中点，太小调度开销大，太大分支分歧惩罚重。

**问：Tensor Core 和 CUDA Core 区别？为什么能加速 AI？**

答：CUDA Core 是通用标量计算核心，一个周期执行一次乘加（FMA）。Tensor Core 是矩阵乘加专用单元，一个周期可以完成一个小矩阵的 D=A×B+C（如 16×8×16 的 FP16 运算，等价于 2048 次乘加）。AI 计算的核心是矩阵乘法，Tensor Core 专门优化了这个操作，相比 CUDA Core 有约 7 倍加速。代价是矩阵维度需要对齐（8 或 16 的倍数）。

## 深入追问（可能被追问的点）

- **Q1: Warp Divergence 是什么？影响多大？**
  A: Warp 内 32 个线程遇到 if-else 分支时，走不同分支的线程被掩码空转，两个分支串行执行。最坏情况（每个线程走不同分支）性能降低 32 倍。详见 `../../CUDA编程/3. 优化篇/1. Warp与执行模型.md`。

- **Q2: SIMT 和 SIMD 有什么区别？**
  A: SIMD（如 CPU AVX）是完全锁步，所有 lane 执行完全相同的操作。SIMT 是 NVIDIA 的扩展——32 个线程执行同一条指令，但每个线程有独立的 PC（程序计数器）和寄存器，理论上可以走不同分支（只是有 divergence 代价）。

- **Q3: 为什么矩阵维度必须是 8 或 16 的倍数才能用 Tensor Core？**
  A: Tensor Core 的 MMA 操作处理固定大小的矩阵块（如 16×8×16），如果输入矩阵维度不是块大小的倍数，需要 padding 补齐，否则无法对齐到 Tensor Core 的计算单元。

- **Q4: 一个 SM 能同时运行多少个 Block？**
  A: 取决于 SM 资源（寄存器、共享内存）和 Block 的资源需求。SM 有最大活跃 Block 数限制（如 16~32），实际能运行多少个由资源分配决定。详见 `../../CUDA编程/3. 优化篇/3. Occupancy与资源分配.md`。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| CUDA Core vs Tensor Core | 标量乘加 vs 矩阵块乘加 | 计算器 vs 矩阵专用电路 |
| SM vs Warp | SM 是调度单位（资源容器）vs Warp 是执行单位（32线程） | SM=车间，Warp=一个班组 |
| SIMT vs SIMD | 线程有独立 PC/寄存器 vs 完全锁步 | SIMT 允许分歧，SIMD 不允许 |
| Block vs Warp | Block 是逻辑协作单位（程序员定）vs Warp 是物理执行单位（硬件定） | Block 可跨 Warp，Warp 不可拆分 |
| CPU 多核 vs GPU 多核 | 几个强核（复杂控制+大缓存）vs 上万弱核（简单 ALU） | 延迟优先 vs 吞吐优先 |
| FMA vs MMA | FMA=标量乘加 c=a×b+c vs MMA=矩阵乘加 D=A×B+C | CUDA Core 做 FMA，Tensor Core 做 MMA |

  > **答**：Warp 内 32 个线程遇到 if-else 分支时，走不同分支的线程被掩码（mask）空转，两个分支只能串行执行。最坏情况（每线程走不同分支）性能降低 32 倍。所以 GPU 编程要尽量避免 Warp 内分支——可以用三元运算符或 predicated execution 替代 if-else。→ 详见 §2.3 ⚠️ + 深入追问 Q1

- [x] 能说出 SIMT 的含义和与 SIMD 的区别
  > **答**：SIMT（Single Instruction, Multiple Threads）= 32 个线程执行同一条指令，但每个线程有独立的 PC（程序计数器）和寄存器，理论上可以走不同分支（只是有 divergence 代价）。SIMD（如 CPU AVX）是完全锁步，所有 lane 执行完全相同的操作，不允许分歧。一句话：SIMT 允许分歧（有代价），SIMD 不允许分歧。→ 详见 §2.1 术语表 + 深入追问 Q2

- [x] 能解释 Tensor Core 和 CUDA Core 的本质区别
  > **答**：CUDA Core 是通用标量计算核心，一个周期执行一次乘加（FMA）：`c = a × b + c`。Tensor Core 是矩阵乘加专用单元，一个周期完成一个小矩阵的 `D = A × B + C`（如 16×8×16 的 FP16 运算 = 2048 次乘加）。类比：CUDA Core 像计算器（一次算一个数），Tensor Core 像矩阵专用电路（一次算一整块）。加速比约 7 倍。→ 详见 §3.1 + 面试回答 Q3

- [x] 能说出 FP32/TF32/FP16/BF16/FP8 各自的位宽和适用场景
  > **答**：
  > | 精度 | 位宽 | 适用场景 |
  > |------|------|---------|
  > | FP32 | 32 bit | 基准精度、loss 累加 |
  > | TF32 | 19 bit | 训练（Ampere 默认开启，不用改代码） |
  > | FP16 | 16 bit | 混合精度训练（需 loss scaling） |
  > | BF16 | 16 bit | 混合精度训练（推荐，无需 loss scaling） |
  > | FP8 | 8 bit | 训练+推理（Hopper 引入） |
  > → 详见 §3.2 精度格式表

- [x] 能解释为什么 BF16 比 FP16 更适合大模型训练
  > **答**：BF16 和 FP32 一样有 8 位指数位，动态范围大（最大约 3.4×10³⁸），训练时不容易溢出，不需要 loss scaling。FP16 只有 5 位指数位，范围窄（最大约 65504），小梯度容易下溢变成 0，必须配合 GradScaler 使用。现代大模型训练几乎都用 BF16。→ 详见 §3.2 💡 + `../3. 架构演进篇/2. NVIDIA-GPU架构演进史.md` §5.4

- [x] 能说出矩阵维度对齐对 Tensor Core 加速的影响
  > **答**：Tensor Core 的 MMA 操作处理固定大小的矩阵块（如 16×8×16），所以输入矩阵的维度必须是 8 或 16 的倍数才能对齐到 Tensor Core 的计算单元。如果 hidden_size 不是 8 的倍数，需要 padding 补齐，否则 Tensor Core 无法被充分利用——等于花了买 Tensor Core 的钱却没用上。→ 详见 §3.3 📌 + 深入追问 Q3
