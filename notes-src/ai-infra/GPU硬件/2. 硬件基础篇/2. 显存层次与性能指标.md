## 一句话结论

GPU 的存储按「Register → Shared Memory → L1/L2 Cache → HBM」金字塔分层，越靠近计算核心越快越小；AI 场景下**带宽往往比算力先成为瓶颈**——用 Roofline 模型的算术强度（FLOPs/Byte）判断任务是 Compute-bound 还是 Memory-bound，是做性能优化的第一步。

## 核心概念（新人友好讲解）

### 1. GPU 存储层级金字塔

GPU 的计算再快，如果数据"喂"不上来也白搭。GPU 内部的存储按层级组织，和 CPU 类似——越靠近计算核心，速度越快但容量越小：

```
存储金字塔（从快到慢、从小到大）

         ┌───────────┐
         │ Register  │  ← 每线程 ~255 个，~0 cycle，最快但极小
         │  (寄存器)  │
         ├───────────┤
         │  Shared   │  ← 每 SM ~228 KB，~30 cycle，SM 内线程共享
         │  Memory   │     程序员显式管理（__shared__）
         ├───────────┤
         │  L1 Cache │  ← 与 Shared Memory 共享芯片面积
         │           │     硬件自动管理
         ├───────────┤
         │  L2 Cache │  ← 整卡 ~50 MB，~200 cycle，所有 SM 共享
         ├───────────┤
         │   HBM     │  ← 80 GB，~600 cycle，主存储
         │  (显存)    │     带宽决定吞吐上限
         └───────────┘
```

#### 1.1 各层级参数对比

| 层级 | 容量 | 延迟（cycle） | 带宽量级 | 作用域 | 说明 |
|------|------|--------------|---------|--------|------|
| **Register**（寄存器） | 每线程 ~255 个 | ~0 | 最高 | 单线程私有 | 最快，但容量极有限 |
| **Shared Memory** | 每 SM ~228 KB | ~30 | 高 | SM 内所有线程 | 程序员显式管理的"L1 级别"缓存 |
| **L1 Cache** | 与 Shared Memory 共享 | ~30 | 高 | SM 内 | 硬件自动管理 |
| **L2 Cache** | 整卡 ~50 MB | ~200 | 中 | 全局 | 所有 SM 共享 |
| **HBM**（显存） | 80 GB | ~600 | 3+ TB/s | 全局 | 主存储，带宽决定吞吐上限 |

> 📌 **关键点**：从 Register 到 HBM，延迟差了 600 倍以上。GPU 靠**大量线程并发隐藏延迟**——当一部分线程在等 HBM 数据时，SM 切换到另一部分线程执行计算，让计算单元不闲着。这就是 GPU 吞吐导向设计的核心体现。

#### 1.2 为什么需要金字塔而不是全部用 HBM

| 存储类型 | 带宽 | 容量 | 成本 |
|---------|------|------|------|
| Register | ~10s TB/s | ~KB 级 | 极贵（芯片面积） |
| Shared Memory / L1 | ~数 TB/s | ~百 KB 级 | 贵 |
| L2 Cache | ~数 TB/s | ~几十 MB | 中 |
| HBM | 3+ TB/s | 80 GB | 相对便宜 |

> 💡 **类比**：Register 像你手里的笔记本（秒看但只有几页），Shared Memory 像办公桌上的参考书（几步就能拿到），L2 像楼层的图书角（走几步取书），HBM 像图书馆（书最多但要走一趟）。GPU 不会把所有数据都放笔记本上——太贵了，而是靠层级调度，让最常用的数据待在最近的地方。

> ⚠️ **Register Spilling**：当线程使用的变量超过寄存器上限时，编译器会把多余的变量"溢出"到 Local Memory——但 Local Memory 实际存储在 HBM 上，访问速度和 Global Memory 一样慢。这是需要警惕的性能陷阱。详见 `../../CUDA编程/2. 基础篇/3. CUDA内存模型.md`。

### 2. HBM：高带宽显存

#### 2.1 为什么需要 HBM

传统 GDDR 显存就像一条单车道公路——传输速率有限。HBM（High Bandwidth Memory）把几千条"车道"堆叠在一起，通过硅中介层（Silicon Interposer）直接和 GPU 芯片紧贴互联，实现数倍的带宽提升。

```
传统 GDDR 显存（如 RTX 4090）：

  GPU 芯片 ──── PCB 走线 ──── GDDR6X 颗粒（焊在 PCB 上）
              距离远，信号损耗大，带宽受限


HBM 显存（如 A100 / H100）：

  ┌──────────────────────────┐
  │     GPU 芯片（逻辑裸片）     │
  │  ████████████████████████ │
  │  │ 硅中介层（Silicon Interposer）│ │  ← 高密度细线互连
  │  │  ┌────┐  ┌────┐        │ │
  │  │  │HBM │  │HBM │  ...   │ │  ← 显存堆叠在旁边
  │  │  │堆叠│  │堆叠│        │ │
  │  │  └────┘  └────┘        │ │
  │  └────────────────────────┘ │
  └──────────────────────────┘
  距离极近（毫米级），带宽极高
```

#### 2.2 HBM 代际对比

| 显存类型 | 代表 GPU | 带宽 | 容量 | 堆叠层数 | 发布年 |
|---------|---------|------|------|---------|-------|
| GDDR6X | RTX 4090 | 1,008 GB/s | 24 GB | — | 2022 |
| HBM2e | A100 | 2,039 GB/s | 80 GB | 4 层 | 2020 |
| HBM3 | H100 | 3,350 GB/s | 80 GB | 8 层 | 2022 |
| HBM3e | B200 | 8,000 GB/s | 192 GB | 12 层 | 2024 |

> 💡 **趋势**：从 A100 到 B200，显存带宽增长了近 4 倍。这对 LLM 推理意义重大——推理阶段的 decode 是典型的 **Memory-bound** 任务，带宽每提升 1 倍，理论吞吐量就能接近翻倍。显存容量从 80GB → 192GB 也让单卡能装下更大的模型。

> 🔧 **新人术语速查**：
> - **HBM**（High Bandwidth Memory）：高带宽显存，通过 3D 堆叠 + 硅中介层实现极高带宽
> - **Silicon Interposer**（硅中介层）：连接 GPU 芯片和 HBM 颗粒的高密度互连基板
> - **TSV**（Through-Silicon Via）：硅通孔，HBM 堆叠层之间的垂直互连技术
> - **Memory-bound**：任务性能受限于显存带宽而非算力
> - **Compute-bound**：任务性能受限于算力而非显存带宽

### 3. Roofline 模型：算力 vs 带宽的平衡点

#### 3.1 算术强度

判断一个计算任务是 Compute-bound 还是 Memory-bound，核心指标是**算术强度**（Arithmetic Intensity）：

```
                浮点运算量 (FLOPs)
算术强度 = ─────────────────────────
              内存访问量 (Bytes)

单位：FLOPs / Byte
```

算术强度越高，说明每读取 1 Byte 数据做了越多计算——越偏 Compute-bound；反之越偏 Memory-bound。

#### 3.2 Roofline 模型

Roofline 模型把算力和带宽画在一张图上，直观展示性能上限：

```
性能 (FLOPS)
    ↑
    │         ╱  ← Compute Peak（算力上限）
    │        ╱
    │       ╱
    │      ╱
    │     ╱
    │    ╱  ← Memory Bandwidth Ceiling（带宽上限）
    │   ╱
    │  ╱
    │ ╱
    │╱
    └──────────────────────→ 算术强度 (FLOPs/Byte)
         ↑
    ops:byte 比
    （拐点 = 算力 / 带宽）
```

- **拐点左侧**（算术强度 < ops:byte 比）：Memory-bound，性能受带宽限制
- **拐点右侧**（算术强度 > ops:byte 比）：Compute-bound，性能受算力限制

#### 3.3 H100 的 Roofline 实例

以 H100 SXM 为例：

| 参数 | 值 |
|------|-----|
| FP16 算力 | 989 TFLOPS |
| HBM3 带宽 | 3,350 GB/s |
| **ops:byte 比** | 989 × 10¹² / 3,350 × 10⁹ ≈ **295 FLOPs/Byte** |

> 📌 **关键点**：H100 的 ops:byte 比约 295。这意味着——你的 Kernel 每从显存读取 1 Byte 数据，需要执行至少 295 次 FP16 浮点运算，才能让算力充分跑满。否则 GPU 会"闲着等数据"。

#### 3.4 常见算子的算术强度

| 算子 | 算术强度 (FLOPs/Byte) | 类型 | 说明 |
|------|----------------------|------|------|
| **GEMM（矩阵乘法）** | ~200~500+ | Compute-bound（大矩阵） | O(N³) 计算量 / O(N²) 数据量 |
| **Attention（标准）** | ~50~100 | Memory-bound | KV cache 读取量大 |
| **FlashAttention** | ~200~300 | Compute-bound（优化后） | 减少显存访问，提升算术强度 |
| **LayerNorm** | ~5~10 | Memory-bound | 每元素只需几次运算 |
| **激活函数（ReLU/GELU）** | ~1~3 | Memory-bound | 逐元素操作，几乎纯访存 |
| **Softmax** | ~5~15 | Memory-bound | 需要两遍扫描（max + exp + sum） |

> 💡 **优化启示**：大多数深度学习算子是 Memory-bound。FlashAttention、Kernel Fusion（算子融合）等技术的核心思路就是**减少显存访问次数、提升算术强度**，把 Memory-bound 变成 Compute-bound。

> 🔧 **新人参数解释**：
> - **ops:byte 比**：GPU 的算力除以带宽，即每读取 1 Byte 数据需要执行多少次运算才能"吃满"算力
> - **Kernel Fusion（算子融合）**：把多个小算子合并成一个大 kernel，中间结果不写回 HBM，减少显存访问
> - **FlashAttention**：通过分块计算和重算，将 Attention 的 HBM 访问从 O(N²) 降至 O(N²/√M)，显著提升算术强度

### 4. GPU 关键性能指标

#### 4.1 四大核心指标

评估一块 GPU 的 AI 计算能力，需要关注以下核心指标：

| 指标 | 含义 | 决定什么 | 典型值（H100） |
|------|------|---------|---------------|
| **算力**（TFLOPS） | 每秒浮点运算次数 | Compute-bound 任务上限 | 989 TFLOPS（FP16） |
| **显存带宽**（GB/s） | 每秒显存读写数据量 | Memory-bound 任务上限 | 3,350 GB/s |
| **显存容量**（GB） | 可存储的数据总量 | 能跑多大模型 | 80 GB |
| **互联带宽**（GB/s） | GPU 间通信速度 | 多卡并行扩展效率 | 900 GB/s（NVLink） |

> 📌 **关键点**：四个指标中，**显存容量决定能不能跑**，**带宽决定跑得多快**，**算力决定计算密集型任务的上限**，**互联带宽决定多卡扩展效率**。选卡时这四个维度缺一不可。

#### 4.2 算力计算公式

GPU 的理论峰值算力可以通过硬件参数推算：

```
FP16 算力 = SM 数量 × 每 SM 的 FP16 Core 数 × 2 × 时钟频率

其中 ×2 是因为每个 FP Core 每周期执行一次 FMA（Fused Multiply-Add），
包含一次乘法 + 一次加法 = 2 次浮点运算。
```

以 H100 SXM 为例：

| 参数 | 值 |
|------|-----|
| SM 数量 | 132 |
| 每 SM 的 FP16 Core 数 | 512（128 FP32 × 2，FP16 是 FP32 双发） |
| 时钟频率 | ~1.83 GHz |
| 理论 FP16 算力 | 132 × 512 × 2 × 1.83 ≈ 247 TFLOPS（CUDA Core） |
| 含 Tensor Core 的 FP16 | 989 TFLOPS（Tensor Core 加速 ~4×） |

> 💡 **提示**：厂商标称的 TFLOPS 通常是带 Tensor Core 的**稀疏（Sparsity）模式峰值**，实际应用中的有效算力通常只有峰值的 30%~60%。做性能分析时应该用实测值而非标称值。

> 🔧 **新人术语速查**：
> - **FMA**（Fused Multiply-Add）：融合乘加，c = a × b + c，一次完成，等价 2 次浮点运算
> - **稀疏（Sparsity）**：矩阵中 50% 元素为零时，硬件跳过零元素计算，算力翻倍
> - **稠密（Dense）**：不考虑稀疏性的算力，是实际应用的保守估计

```bash

nvidia-smi --query-gpu=memory.total,memory.used,memory.free --format=csv
```

#### 5.4 测量显存带宽（使用 PyTorch）

```python
import torch

# ✅ 无需 sudo —— 用 PyTorch 测量显存读带宽
if torch.cuda.is_available():
    gpu = torch.cuda.get_device_properties(0)
    print(f"GPU: {gpu.name}")
    print(f"显存: {gpu.total_memory / 1024**3:.1f} GB")

    # 测量 HBM 读带宽：大数组逐元素读取
    N = 256 * 1024 * 1024  # 1 GB of float32
    x = torch.randn(N, device='cuda', dtype=torch.float32)

    # 预热
    _ = x.sum()
    torch.cuda.synchronize()

    # 计时
    import time
    torch.cuda.synchronize()
    start = time.perf_counter()
    for _ in range(10):
        s = x.sum()
    torch.cuda.synchronize()
    elapsed = (time.perf_counter() - start) / 10

    # 读 1GB 数据，reduce 到 1 个值
    bandwidth = (N * 4) / elapsed / 1e9  # float32 = 4 bytes
    print(f"实测读带宽: {bandwidth:.0f} GB/s")
    print(f"理论峰值带宽: ~3350 GB/s (H100)")
    print(f"利用效率: {bandwidth / 3350 * 100:.1f}%")
```

> ⚠️ **注意**：PyTorch 测量的是粗略有效带宽，实际峰值带宽需要用 NVIDIA 的 `bandwidthTest` 工具或 `nccl-tests` 精确测量。详见 `../../CUDA编程/5. 工具链与扩展篇/2. 性能分析工具链.md`。

## 面试回答（可直接口述的版本）

**问：GPU 的存储层级是怎样的？为什么需要分层？**

答：GPU 存储按 Register → Shared Memory → L1/L2 Cache → HBM 分层，越靠近计算核心越快越小。从 Register 到 HBM 延迟差了 600 倍以上。分层的原因是成本——Register 极快但极贵（占芯片面积），HBM 便宜但慢。GPU 靠大量线程并发隐藏延迟：一部分线程在等 HBM 数据时，SM 切换到另一部分线程执行，让计算单元不闲着。这就是吞吐导向设计的核心。

**问：什么是 Roofline 模型？怎么判断一个算子是 Compute-bound 还是 Memory-bound？**

答：Roofline 模型用算术强度（FLOPs/Byte）来判断。算术强度 = 浮点运算量 / 内存访问量。GPU 有一个 ops:byte 比 = 算力 / 带宽，比如 H100 是 989 TFLOPS / 3,350 GB/s ≈ 295 FLOPs/Byte。如果算子的算术强度低于 295，说明每读 1 Byte 数据做的运算太少，算力有富余，就是 Memory-bound；高于 295 就是 Compute-bound。大多数深度学习算子（LayerNorm、激活函数、标准 Attention）是 Memory-bound，所以 FlashAttention 和 Kernel Fusion 的核心思路就是减少显存访问、提升算术强度。

**问：GPU 的四大性能指标是什么？选卡时怎么权衡？**

答：四大指标是算力（TFLOPS）、显存带宽（GB/s）、显存容量（GB）、互联带宽（GB/s）。显存容量决定能不能跑——7B 模型 BF16 需要 14GB 参数 + 优化器状态约 126GB，单卡 80GB 装不下就需要分片。带宽决定跑多快——推理 decode 阶段是 Memory-bound，带宽翻倍吞吐量接近翻倍。算力决定计算密集型任务上限——GEMM 是 Compute-bound。互联带宽决定多卡扩展效率。选卡时这四个维度缺一不可，比如 H100 各项均衡，L40S 带宽只有 864 GB/s 偏推理。

**问：厂商标称的 TFLOPS 和实际有效算力差多少？为什么？**

答：厂商标称通常是 Tensor Core 的稀疏模式峰值，实际有效算力通常只有峰值的 30%~60%。原因包括：一是稀疏模式要求矩阵 50% 元素为零才生效，实际模型不一定满足；二是访存延迟、分支分歧等导致计算单元空转；三是 kernel 启动、同步等额外开销。所以做性能分析要用实测值，可以用 NCU（NVIDIA Nsight Compute）测量实际 SOL（Speed of Light）指标。

## 深入追问（可能被追问的点）

- **Q1: Register Spilling 是什么？怎么检测？**
  A: 当线程使用的寄存器超过 SM 分配上限时，编译器把多余变量溢出到 Local Memory——但 Local Memory 实际存储在 HBM 上，访问和 Global Memory 一样慢。用 `nvcc --ptxas-options=-v` 编译时可以看到寄存器使用量，用 NCU 分析可以看到 Local Memory load/store 计数。详见 `../../CUDA编程/3. 优化篇/3. Occupancy与资源分配.md`。

- **Q2: FlashAttention 怎么提升算术强度的？**
  A: 标准 Attention 需要把 N×N 的 attention matrix 写回 HBM 再读回来做 Softmax，算术强度很低。FlashAttention 把 Q/K/V 分块（tiling），在 Shared Memory 中完成小块的 attention 计算，中间结果不写回 HBM，只写最终输出。HBM 访问从 O(N²) 降至 O(N²/√M)，算术强度大幅提升。详见 `../../CUDA编程/4. 算子实战篇/2. CUDA-GEMM算子性能优化.md`。

- **Q3: 为什么 GEMM 是 Compute-bound 而不是 Memory-bound？**
  A: GEMM 的计算量是 O(N³)，数据量是 O(N²)，算术强度 = N³/N² = N，随矩阵尺寸增长。对于 4096×4096 的矩阵，算术强度约 2730 FLOPs/Byte，远超 H100 的 ops:byte 比 295，所以是 Compute-bound。但小矩阵（如 128×128）算术强度只有 128，是 Memory-bound。

- **Q4: 为什么标称算力要区分稀疏和稠密？**
  A: NVIDIA 从 Ampere 开始支持结构化稀疏（2:4 sparsity）——矩阵每 4 个元素中最多 2 个非零，硬件跳过零元素计算，算力翻倍。但实际模型需要先做剪枝才能满足 2:4 格式，通用场景下不一定满足。所以稀疏峰值是理论最大，稠密峰值是保守估计，实际介于两者之间。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| Compute-bound vs Memory-bound | 受算力限制 vs 受带宽限制 | 看算术强度 vs ops:byte 比 |
| Register vs Shared Memory | 单线程私有 vs SM 内共享 | Register=手里笔记本，Shared=桌上参考书 |
| L1 Cache vs Shared Memory | 硬件自动管理 vs 程序员显式管理 | L1=自动挡，Shared=手动挡 |
| L2 Cache vs HBM | 全局缓存（~50MB）vs 主存储（80GB） | L2=楼层图书角，HBM=图书馆 |
| 稀疏算力 vs 稠密算力 | 矩阵含零可跳过 vs 不考虑稀疏 | 稀疏=理想峰值，稠密=保守估计 |
| 算力 vs 带宽 | 每秒算多少（FLOPS）vs 每秒搬多少（GB/s） | 算力=CPU速度，带宽=马路宽度 |
| HBM2e vs HBM3 | 2,039 GB/s vs 3,350 GB/s | 代际升级，带宽翻倍 |
| 理论算力 vs 有效算力 | 规格表峰值 vs 实际跑出来的 | 有效通常 30%~60% 理论 |

## 自测清单（含答案）

- [x] 能画出 GPU 存储金字塔（Register → Shared Memory → L1/L2 → HBM）并说出各级延迟量级
  > **答**：
  > ```
  > Register（~0 cycle）→ Shared Memory（~30 cycle）
  >   → L1 Cache（~30 cycle）→ L2 Cache（~200 cycle）→ HBM（~600 cycle）
  > ```
  > 越靠近计算核心越快越小，从 Register 到 HBM 延迟差了 600 倍以上。→ 详见 §1 存储金字塔图

- [x] 能解释为什么 GPU 需要存储分层（成本 + 延迟隐藏）
  > **答**：两层原因：①成本——Register 极快但极贵（占芯片面积），HBM 便宜但慢，全用 Register 造不起，全用 HBM 太慢；②延迟隐藏——GPU 靠大量线程并发隐藏延迟，一部分线程在等 HBM 数据时，SM 切换到另一部分线程执行，让计算单元不闲着。类比：你手里只有几页笔记本（Register），但桌上放了参考书（Shared Memory），楼下有图书角（L2），街对面有图书馆（HBM）——常用的东西放手边，偶尔用的放远处。→ 详见 §1.2 成本表 + 📌

- [x] 能说出 Register Spilling 是什么以及为什么影响性能
  > **答**：当线程使用的寄存器超过 SM 分配上限时，编译器把多余变量"溢出"到 Local Memory——但 Local Memory 实际存储在 HBM 上，访问速度和 Global Memory 一样慢（~600 cycle）。本来 0 cycle 的操作变成了 600 cycle，严重拖慢性能。检测方法：`nvcc --ptxas-options=-v` 看寄存器使用量，NCU 看 Local Memory load/store 计数。→ 详见 §1.2 ⚠️ + 深入追问 Q1

- [x] 能解释 HBM 的原理（3D 堆叠 + 硅中介层）和与传统 GDDR 的区别
  > **答**：HBM 通过 3D 堆叠 + 硅中介层（Silicon Interposer）把显存颗粒紧贴 GPU 芯片（毫米级距离），实现极高带宽。传统 GDDR 焊在 PCB 板上，距离远、信号损耗大、带宽受限。类比：GDDR 像快递送货（路远慢），HBM 像东西就在桌上（手边快）。HBM 用 TSV（硅通孔）做层间垂直互连，层数越多带宽越高。→ 详见 §2.1 对比图

- [x] 能说出 HBM2e/HBM3/HBM3e 各代带宽和代表 GPU
  > **答**：
  > | 类型 | 带宽 | 容量 | 代表 GPU |
  > |------|------|------|---------|
  > | HBM2e | 2,039 GB/s | 80 GB | A100 |
  > | HBM3 | 3,350 GB/s | 80 GB | H100 |
  > | HBM3e | 8,000 GB/s | 192 GB | B200 |
  > → 详见 §2.2 HBM 代际对比表

- [x] 能用算术强度公式判断一个算子是 Compute-bound 还是 Memory-bound
  > **答**：算术强度 = FLOPs / Bytes（每读 1 Byte 数据做了多少次浮点运算）。GPU 有一个 ops:byte 比 = 算力 / 带宽。如果算子的算术强度 < ops:byte 比 → Memory-bound（带宽先满，算力有空余）；> ops:byte 比 → Compute-bound（算力先满，带宽有空余）。→ 详见 §3.1-3.2

- [x] 能算出 H100 的 ops:byte 比（~295 FLOPs/Byte）并解释含义
  > **答**：H100 ops:byte 比 = 989 TFLOPS / 3,350 GB/s ≈ 295 FLOPs/Byte。含义：你的 kernel 每从显存读 1 Byte 数据，需要执行至少 295 次 FP16 浮点运算才能让算力跑满。否则 GPU 会"闲着等数据"——算力有富余但数据搬不过来。→ 详见 §3.3

- [x] 能说出 GEMM、Attention、LayerNorm 各自是 Compute-bound 还是 Memory-bound
  > **答**：
  > - **GEMM**：Compute-bound（大矩阵算术强度 ~200-500+，远超 295）
  > - **Attention（标准）**：Memory-bound（~50-100，KV cache 读取量大）
  > - **LayerNorm**：Memory-bound（~5-10，每元素只需几次运算）
  > FlashAttention 通过分块重算把算术强度提到 ~200-300，变成了 Compute-bound。→ 详见 §3.4 算子算术强度表

- [x] 能解释 FlashAttention 和 Kernel Fusion 为什么能提升算术强度
  > **答**：两者核心思路都是**减少显存访问次数**。FlashAttention 把 Q/K/V 分块（tiling），在 Shared Memory 中完成小块计算，中间结果不写回 HBM，HBM 访问从 O(N²) 降至 O(N²/√M)。Kernel Fusion 把多个小算子合并成一个大 kernel，中间结果不写回 HBM。显存访问少了，算术强度（FLOPs/Bytes）的分母变小，值自然变大。→ 详见 §3.4 💡 + 深入追问 Q2

- [x] 能说出 GPU 四大核心指标及其含义
  > **答**：①算力（TFLOPS）——决定 Compute-bound 任务上限；②显存带宽（GB/s）——决定 Memory-bound 任务上限；③显存容量（GB）——决定能跑多大模型；④互联带宽（GB/s）——决定多卡扩展效率。一句话：容量决定"能不能跑"，带宽决定"跑多快"，算力决定"算多猛"，互联决定"几卡一起跑多快"。→ 详见 §4.1 四大指标表

- [x] 能用算力计算公式粗算一块 GPU 的理论 FP16 算力
  > **答**：公式 = SM 数 × 每 SM FP16 Core 数 × 2 × 时钟频率（×2 因为 FMA = 乘+加 = 2 次运算）。以 H100 为例：132 × 512 × 2 × 1.83 GHz ≈ 247 TFLOPS（CUDA Core），含 Tensor Core 约 989 TFLOPS。→ 详见 §4.2 算力计算

- [x] 能解释厂商标称 TFLOPS 和实际有效算力的差距原因
  > **答**：标称通常是 Tensor Core 稀疏模式峰值，实际有效只有 30%~60%。三个原因：①稀疏模式要求矩阵 50% 元素为零才生效，实际模型不一定满足；②访存延迟、分支分歧导致计算单元空转；③kernel 启动、同步等额外开销。做性能分析要用实测值（NCU 测 SOL 指标）。→ 详见 §4.2 💡 + 面试回答 Q4

- [x] 能用 `nvidia-smi` 查看显存容量和使用情况
  > **答**：`nvidia-smi --query-gpu=memory.total,memory.used,memory.free --format=csv`。完整信息用 `nvidia-smi -q` 查看 Product Name / Compute Capability / FB Memory Usage 等。→ 详见 §5.1-5.2 命令与输出示例

- [x] 能用 PyTorch 粗测显存读带宽
  > **答**：在 GPU 上分配大数组（如 1GB float32），用 `x.sum()` 做 reduce 操作（读全部数据算一个值），计时后带宽 = 数据量 / 耗时。注意先 warmup 再计时。→ 详见 §5.4 完整代码示例
