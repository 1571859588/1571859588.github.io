## 一句话结论

训练显存主要花在「参数 + 梯度 + 优化器状态 + 激活值」四块，优化靠 ZeRO/梯度检查点/Offload；多卡互联按「机内 NVLink（900 GB/s 级）→ 机间 InfiniBand（50~400 GB/s）」分层，用 `nvidia-smi topo -m` 看拓扑、`nvidia-smi nvlink -s` 看状态。

## 核心概念（新人友好讲解）

### 1. 显存管理：AI Infra 核心挑战

#### 1.1 训练时显存都用在哪里

以 Adam 优化器 + FP16 混合精度训练为例，每个参数需要的显存：

| 组成部分 | 每参数字节 | 说明 |
|----------|-----------|------|
| FP16 参数 | 2 B | 前向/反向计算使用 |
| FP32 参数副本 | 4 B | Adam 需要维护 FP32 master weights |
| FP32 梯度 | 4 B | 反向传播产生 |
| Adam 一阶动量（m） | 4 B | 梯度的指数移动平均 |
| Adam 二阶动量（v） | 4 B | 梯度平方的指数移动平均 |
| **合计** | **18 B** | — |

> 🔧 **新人参数解释**：
> - **master weights**：FP32 精度的参数副本，用于参数更新（FP16 精度不够，累加小梯度会丢失）
> - **Adam 一阶动量 m**：梯度的指数移动平均（动量），帮助加速收敛
> - **Adam 二阶动量 v**：梯度平方的指数移动平均，用于自适应学习率调整

**显存估算实例：7B 模型**

```
固定显存 = 7 × 10^9 × 18 = 126 GB

加上激活值（Activation，取决于 batch size 和序列长度）：
  - batch_size=4, seq_len=2048 → 约 20~40 GB 激活值
  - 总计约 146~166 GB → 单卡 A100 80GB 装不下！
```

> 📌 **关键点**：7B 模型训练需要约 126GB 固定显存 + 激活值，远超单卡 80GB。这就是为什么需要 ZeRO 分片或梯度检查点等优化策略。

#### 1.2 常用显存优化策略

| 策略 | 原理 | 显存节省 | 代价 |
|------|------|---------|------|
| **混合精度训练** | FP16/BF16 做前向反向，FP32 做参数更新 | ~50% 参数显存 | 需要 loss scaling（FP16） |
| **梯度累积** | 多个 micro-batch 累积梯度，等效大 batch | 减少激活值峰值 | 增加训练步数 |
| **梯度检查点** | 前向只保留部分激活，反向重新计算 | 激活显存降至 O(√N) | 增加约 33% 计算量 |
| **ZeRO 优化** | 将优化器状态/梯度/参数分片到多卡 | 线性降低每卡显存 | 增加通信量 |
| **Offloading** | 将部分数据卸载到 CPU 内存或 NVMe | 突破单卡显存上限 | PCIe 带宽成瓶颈 |

##### ZeRO 三个阶段详解

ZeRO（Zero Redundancy Optimizer）将训练状态分片到多卡，逐步消除冗余：

```
                    每卡存储的内容
                ┌──────────────────────────────────┐
   ZeRO-1      │  参数(完整) + 梯度(完整) + 优化器(分片) │  省 4x 优化器显存
                └──────────────────────────────────┘
                ┌──────────────────────────────────┐
   ZeRO-2      │  参数(完整) + 梯度(分片) + 优化器(分片) │  再省梯度显存
                └──────────────────────────────────┘
                ┌──────────────────────────────────┐
   ZeRO-3      │  参数(分片) + 梯度(分片) + 优化器(分片) │  最省显存，通信最多
                └──────────────────────────────────┘
```

| 阶段 | 分片内容 | 显存节省 | 通信开销 | 适用场景 |
|------|---------|---------|---------|---------|
| ZeRO-1 | 优化器状态 | ~4× | 与普通 DP 相同 | 入门，通信不增加 |
| ZeRO-2 | 优化器状态 + 梯度 | ~8× | 少量增加（reduce-scatter） | 中大模型训练（推荐） |
| ZeRO-3 | 优化器状态 + 梯度 + 参数 | ~N×（N 卡数） | 大量增加（all-gather 参数） | 超大模型（单卡装不下参数） |

> 💡 **类比**：ZeRO 就像把一本大账本拆给三个人分别记——一个记参数、一个记梯度、一个记优化器状态，每人只需保管三分之一。用的时候再凑齐。

> 💡 **实践建议**：ZeRO Stage 2 + 混合精度 + 梯度检查点是训练中大模型（7B~70B）的常见配置。

### 2. 多卡互联：从单机到集群

#### 2.1 互联技术对比

| 互联技术 | 带宽 | 场景 | 说明 |
|----------|------|------|------|
| PCIe 5.0 | 128 GB/s（双向） | 跨卡（通用） | 普遍但带宽较低 |
| NVLink 4.0（Hopper） | 900 GB/s（全双工） | 机内 GPU 互联 | 专用高速互联 |
| NVLink 5.0（Blackwell） | 1,800 GB/s（全双工） | 机内 GPU 互联 | B200 使用 |
| NVSwitch | 连接所有 NVLink | 机内全互联 | 实现 All-to-All 全带宽 |
| InfiniBand NDR | 400 Gb/s（50 GB/s） | 跨机互联 | 大规模集群标配 |
| RoCE | 200~400 Gb/s | 跨机互联 | 以网承载 RDMA |

> 🔧 **新人术语速查**：
> - **NVLink**：NVIDIA 专用的 GPU 间高速互联，点对点
> - **NVSwitch**：NVLink 交换芯片，实现机内所有 GPU 全互联（任意两卡满带宽）
> - **InfiniBand（IB）**：专为高性能计算设计的网络协议，低延迟、高带宽
> - **RoCE**：RDMA over Converged Ethernet，在以太网上实现 RDMA
> - **RDMA**（Remote Direct Memory Access）：远程直接内存访问，绕过 CPU 直接读写远端内存
> - **GPUDirect**：GPU 间直接通信（绕过 CPU 内存），减少拷贝
> - **P2P**（Peer-to-Peer）：GPU 间点对点直接访问显存
> - **allreduce**：分布式训练中的集合通信原语，所有 GPU 的梯度求和后广播

#### 2.2 带宽层级

```
带宽量级对比（从快到慢）：

HBM 显存          8,000 GB/s     ████████████████████████████  ← 单卡内部
NVLink 5.0        1,800 GB/s     ██████                        ← 机内 GPU 间
NVLink 4.0          900 GB/s     ███                           ← 机内 GPU 间
NVLink 3.0          600 GB/s     ██                            ← 机内 GPU 间
PCIe 5.0            128 GB/s     ▌                             ← 机内（慢）
InfiniBand NDR       50 GB/s     ▏                             ← 跨机（最慢）
```

> 📌 **关键点**：机内 NVLink 几百 GB/s，机间 IB 几十 GB/s，差一个数量级。这就是多机训练的瓶颈所在。

#### 2.3 典型多卡拓扑

以 DGX H100（8 卡）为例：

```
┌───────────────────────────────────────────────────┐
│                    DGX H100 节点                    │
│                                                    │
│   GPU0 ←─→ GPU1 ←─→ GPU2 ←─→ GPU3                 │
│    ↕  ╲╱   ↕  ╲╱   ↕  ╲╱   ↕  ╲╱                  │
│   GPU4 ←─→ GPU5 ←─→ GPU6 ←─→ GPU7                 │
│         ↕          ↕          ↕          ↕         │
│    ┌─────────────────────────────────┐             │
│    │        NVSwitch（全互联）          │  ← 任意两卡 900 GB/s
│    └─────────────────────────────────┘             │
│         │          │          │          │         │
│    ┌────┴────┐ ┌───┴────┐ ┌───┴────┐ ┌───┴────┐  │
│    │ IB 网卡  │ │ IB 网卡 │ │ IB 网卡 │ │ IB 网卡 │  │ ← 跨节点
│    └────┬────┘ └───┬────┘ └───┬────┘ └───┬────┘  │
└─────────┼──────────┼──────────┼──────────┼────────┘
          ↓          ↓          ↓          ↓
              InfiniBand 交换机 → 其他节点
```

> 📌 **并行策略设计原则**：通信量大的并行维度（如张量并行）放在高带宽的机内 NVLink 上，通信量小的维度（如数据并行、流水线并行）放在机间 IB 上。

### 3. 动手实践：查看 GPU 拓扑与 NVLink 状态

#### 3.1 查看 GPU 拓扑

```bash
# ✅ 无需 sudo —— 查看 GPU 之间的拓扑连接关系
nvidia-smi topo -m
```

**预期输出**（8×A100 SXM 典型输出，实际取决于硬件配置）：

```
        GPU0  GPU1  GPU2  GPU3  GPU4  GPU5  GPU6  GPU7  NUMA
GPU0     X    NV12  NV12  NV12  NV12  NV12  NV12  NV12  0
GPU1    NV12   X    NV12  NV12  NV12  NV12  NV12  NV12  0
GPU2    NV12  NV12   X    NV12  NV12  NV12  NV12  NV12  0
GPU3    NV12  NV12  NV12   X    NV12  NV12  NV12  NV12  0
GPU4    NV12  NV12  NV12  NV12   X    NV12  NV12  NV12  0
GPU5    NV12  NV12  NV12  NV12  NV12   X    NV12  NV12  0
GPU6    NV12  NV12  NV12  NV12  NV12  NV12   X    NV12  0
GPU7    NV12  NV12  NV12  NV12  NV12  NV12  NV12   X    0
```

**连接类型解读**（新人必看）：

| 标记 | 含义 | 带宽量级 |
|------|------|---------|
| **NV12** | 12 条 NVLink 连接 | 最高（~600 GB/s for A100） |
| **NVL** | NVLink 连接（未指明数量） | 高 |
| **NVB** | NVBridge（通过 NVSwitch） | 高 |
| **PIX** | 同一 PCIe 桥接下 | 中（PCIe 带宽） |
| **NODE** | 同一 NUMA 节点但不同 PCIe 桥 | 中低 |
| **SYS** | 跨 NUMA 节点（走 QPI/UPI） | 低 |
| **X** | 同一 GPU（自身） | — |

> 💡 **解读技巧**：
> - 看到 **NV12/NVL** = 走 NVLink，带宽最高，适合放张量并行
> - 看到 **SYS** = 跨 NUMA 节点，带宽最低，尽量避免高频通信
> - 全是 **NV12** = 通过 NVSwitch 全互联（如 DGX），任意两卡满带宽

> ⚠️ **WSL 注意**：`nvidia-smi topo -m` 在 WSL 中可能无法返回真实硬件拓扑（输出受限或全是 SYS/PIX）。建议在真实 Linux GPU 服务器上运行以获取准确拓扑信息。

## 面试回答（可直接口述的版本）

**问：训练 7B 模型显存怎么估算？为什么需要 ZeRO？**

答：7B 模型用 Adam + FP16 混合精度训练，每参数约 18 字节（FP16 参数 2B + FP32 master 4B + FP32 梯度 4B + Adam m/v 各 4B），固定显存约 126GB。加上激活值（batch_size=4、seq_len=2048 约 20~40GB），总计约 146~166GB。单卡 A100 80GB 装不下，所以需要 ZeRO 分片——ZeRO-2 把优化器状态和梯度分片到多卡，8 卡时每卡固定显存降到约 20GB，就能训练了。

**问：NVLink 和 InfiniBand 的区别？为什么多机训练慢？**

答：NVLink 是机内 GPU 间专用互联，H100 达 900 GB/s；InfiniBand 是跨机网络，NDR 最高 50 GB/s。相差约 18 倍。多机训练时 All-Reduce 通信走 IB，带宽远低于机内 NVLink，所以多机扩展效率远低于单机多卡。设计并行策略时要把通信量大的维度（张量并行）放机内 NVLink，通信量小的（数据并行/流水线并行）放机间 IB。

**问：ZeRO 三个阶段分别切什么？**

答：ZeRO-1 切优化器状态（省 4 倍显存，通信不增加）；ZeRO-2 在 ZeRO-1 基础上切梯度（再省梯度显存，通信略增）；ZeRO-3 切参数+梯度+优化器状态（最省，每卡显存随卡数线性下降，但通信大量增加因为每次前向/反向都要 all-gather 参数）。实践中 ZeRO-2 是性价比最高的选择。

**问：`nvidia-smi topo -m` 输出里的 SYS 和 NVL 分别什么意思？**

答：NVL 表示两卡通过 NVLink 直连，带宽最高（A100 约 600 GB/s）。SYS 表示两卡跨 NUMA 节点，通信要走 CPU 间的 QPI/UPI 总线，带宽最低。设计并行策略时，通信频繁的 GPU 对应该在 NVL 连接上，避免在 SYS 连接上做高频通信。

## 深入追问（可能被追问的点）

- **Q1: 梯度检查点和 Offload 的区别？什么时候用哪个？**
  A: 梯度检查点是重算换显存——前向只存部分激活，反向重新计算，增加约 33% 计算量但激活显存降到 O(√N)。Offload 是把数据搬到 CPU 内存——突破 GPU 显存上限但受 PCIe 带宽限制（~32 GB/s）。优先用梯度检查点（GPU 内部），不够再加 Offload。

- **Q2: 为什么不直接用 ZeRO-3？**
  A: ZeRO-3 每次前向/反向都要 all-gather 参数，通信量巨大。如果单卡能装下参数（用 ZeRO-2），就不要用 ZeRO-3。只有模型参数本身单卡装不下时才用 ZeRO-3。DeepSpeed/FSDP 会根据模型大小自动选择。

- **Q3: NVSwitch 和 NVLink 有什么区别？**
  A: NVLink 是点对点互联——两卡之间直连。NVSwitch 是交换芯片——把所有 GPU 的 NVLink 汇聚到一个交换芯片，实现任意两卡满带宽通信。8 卡 DGX 用 NVSwitch 实现全互联，没有 NVSwitch 的话每张卡只能和少数几张直连。

- **Q4: allreduce 通信量怎么算？**
  A: Ring All-Reduce 的通信量 = 2 × (N-1)/N × 参数量大小，N 为 GPU 数。例如 8 卡 7B 模型 BF16（14GB），allreduce 通信量 ≈ 2 × 7/8 × 14 ≈ 24.5 GB。在 900 GB/s NVLink 上耗时约 27ms。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| NVLink vs NVSwitch | 点对点直连 vs 交换芯片全互联 | NVLink=直连，NVSwitch=交换机 |
| NVLink vs InfiniBand | 机内（900 GB/s）vs 机间（50 GB/s） | 机内高速，机间网络 |
| ZeRO-1/2/3 | 切优化器/切梯度/切参数 | 逐步切更多，通信也更多 |
| 梯度检查点 vs Offload | 重算换显存 vs 搬到 CPU | GPU 内部 vs 跨设备 |
| 带宽 vs 延迟 | 吞吐量（GB/s）vs 往返时间（μs） | 带宽看总量，延迟看单次 |
| 激活值 vs 梯度 | 前向中间结果 vs 反向计算结果 | 激活值随序列长度增长 |

  > - ZeRO-2：切优化器状态 + 梯度 → 再省梯度显存，通信略增（reduce-scatter）
  > - ZeRO-3：切参数 + 梯度 + 优化器状态 → 最省（N 卡线性降低），但通信大量增加（每次前向/反向 all-gather 参数）
  > 类比：把一本大账本拆给三个人记——一个记参数、一个记梯度、一个记优化器状态。实践推荐 ZeRO-2（性价比最高）。→ 详见 §1.2 ZeRO 表 + ASCII 图

- [x] 能解释梯度检查点的工作原理和代价
  > **答**：前向传播时只保留部分激活值（丢弃中间层），反向传播时重新计算被丢弃的部分。效果：激活显存从 O(N) 降至 O(√N)。代价：增加约 33% 计算量（前向算两遍）。类比：考试时不抄全部笔记，只记目录，要用时翻书重读——省了纸（显存）但费了时间（算力）。→ 详见 §1.2 优化策略表 + 深入追问 Q1

- [x] 能说出 NVLink/NVSwitch/InfiniBand 的带宽量级
  > **答**：
  > - NVLink 4.0（H100）：900 GB/s（机内 GPU 间）
  > - NVLink 5.0（B200）：1,800 GB/s（机内 GPU 间）
  > - NVSwitch：不是新带宽，而是交换芯片——让机内所有 GPU 任意两卡满 NVLink 带宽
  > - InfiniBand NDR：50 GB/s（跨机互联）
  > 机内 NVLink 比机间 IB 高约 18 倍。→ 详见 §2.1 互联技术表 + §2.2 带宽层级图

- [x] 能解释为什么多机训练比单机多卡慢
  > **答**：机内 NVLink 900 GB/s vs 机间 IB 50 GB/s，差约 18 倍。多机训练时 All-Reduce 通信走 IB，带宽远低于机内 NVLink，所以多机扩展效率远低于单机多卡。这就是并行策略设计要把通信量大的维度（张量并行）放机内、通信量小的（数据并行/流水线并行）放机间的原因。→ 详见 §2.2 + 面试回答 Q2

- [x] 能用 `nvidia-smi topo -m` 查看 GPU 拓扑
  > **答**：✅ 无需 sudo。命令 `nvidia-smi topo -m` 输出一个矩阵，显示任意两个 GPU 之间的连接类型（NV12/NVL/SYS/PIX 等）。⚠️ WSL 中可能输出受限。→ 详见 §3.1 命令 + 预期输出

- [x] 能解读 topo 输出中的 NVL/SYS/PIX 含义
  > **答**：
  > - **NV12/NVL**：走 NVLink 直连，带宽最高（A100 约 600 GB/s）
  > - **PIX**：同一 PCIe 桥接下，中等带宽
  > - **NODE**：同一 NUMA 节点但不同 PCIe 桥，中低
  > - **SYS**：跨 NUMA 节点（走 CPU 间 QPI/UPI），带宽最低
  > 设计并行策略时：通信频繁的 GPU 对应该在 NVL 连接上，避免在 SYS 上做高频通信。→ 详见 §3.1 连接类型解读表

- [x] 能用 `nvidia-smi nvlink -s` 查看 NVLink 状态
  > **答**：✅ 无需 sudo。命令 `nvidia-smi nvlink -s` 显示每个 NVLink 通道的速率。H100 SXM 有 18 条 NVLink，每条 25 GB/s（单向），18 × 50（双向）= 900 GB/s。如果某条显示 `Disabled` 或 `Error`，说明该通道有故障。⚠️ WSL 中可能输出受限。→ 详见 §3.2 命令 + 预期输出

- [x] 能说出并行策略设计的带宽分配原则
  > **答**：通信量大的并行维度放高带宽的机内 NVLink 上（如张量并行——每层都要 all-reduce），通信量小的维度放机间 IB 上（如数据并行——只在各步 all-reduce 梯度、流水线并行——只在阶段边界通信）。一句话：贵的带宽（NVLink）给大户用，便宜的带宽（IB）给小户用。→ 详见 §2.3 📌 + 面试回答 Q2
