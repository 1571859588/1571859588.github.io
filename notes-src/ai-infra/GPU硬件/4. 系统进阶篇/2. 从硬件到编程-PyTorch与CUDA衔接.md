## 核心概念（新人友好讲解）

### 1. CUDA 编程模型：硬件视角概览

![CUDA programming model](./2. %E4%BB%8E%E7%A1%AC%E4%BB%B6%E5%88%B0%E7%BC%96%E7%A8%8B-PyTorch%E4%B8%8ECUDA%E8%A1%94%E6%8E%A5.assets/CUDA%20programming%20model.png)

CUDA 是 NVIDIA 提供的并行计算编程框架，连接"软件算法"和"硬件执行"。即使你日常用 PyTorch 而不直接写 CUDA，理解编程模型也有助于理解底层发生了什么。

#### 1.1 软件层次映射硬件

CUDA 程序以 **Kernel** 为单位在 GPU 上执行。Kernel 的线程按三级层次组织：

| 软件层次 | 说明 | 硬件对应 | 类比 |
|---------|------|---------|------|
| **Grid** | 一次 Kernel 启动的所有线程 | 整个 GPU（所有 GPC） | 学校 |
| **Block** | 一组协作的线程（最多 1024 个） | 调度到一个 SM | 班级 |
| **Thread** | 最小执行单元 | CUDA Core | 学生 |
| **Warp** | 32 个连续线程（隐式分组） | SM 的 Warp Scheduler | 小组 |

#### 1.2 内存模型映射显存层次

CUDA 编程中不同的变量声明对应不同的硬件存储：

| CUDA 内存类型 | 声明方式 | 硬件位置 | 作用域 | 生命周期 |
|--------------|---------|---------|--------|---------|
| Register | 局部变量 | SM 寄存器堆 | 单线程 | Kernel |
| Local Memory | 溢出的局部变量 | HBM（显存） | 单线程 | Kernel |
| Shared Memory | `__shared__` | SM 片上存储 | Block 内 | Kernel |
| Global Memory | `__device__` / cudaMalloc | HBM（显存） | 所有线程 | 应用 |
| Constant Memory | `__constant__` | HBM + 缓存 | 所有线程 | 应用 |

> 💡 **映射关系**：Global Memory → HBM，Shared Memory → L1 片上存储，Register → SM 寄存器堆。详见 `../../CUDA编程/2. 基础篇/3. CUDA内存模型.md`。

> 🔧 **新人术语速查**：
> - **Kernel**：在 GPU 上执行的函数，用 `__global__` 修饰
> - **H2D / D2H**：Host-to-Device / Device-to-Host，CPU↔GPU 数据搬运
> - **kernel launch**：启动 kernel 执行（`<<<grid, block>>>` 语法）
> - **warmup**：首次 GPU 调用耗时明显高于后续（lazy init + JIT 编译）
> - **异步执行**：kernel launch 是异步的，CPU 启动后立即返回，不等 GPU 执行完

### 2. PyTorch 感受 GPU 计算

大多数 AI 工程师日常通过 PyTorch 间接调用 GPU。理解底层 CUDA 有助于写出更高效的代码。

```python
import torch
import time

# 检查 GPU 信息
if torch.cuda.is_available():
    gpu = torch.cuda.get_device_properties(0)
    print(f"GPU: {gpu.name}")
    print(f"SM 数量: {gpu.multi_processor_count}")
    print(f"显存: {gpu.total_memory / 1024**3:.1f} GB")

# CPU vs GPU 矩阵乘法性能对比
N = 4096
A_cpu = torch.randn(N, N)
B_cpu = torch.randn(N, N)
A_gpu = A_cpu.cuda()
B_gpu = B_cpu.cuda()

# 预热（第一次 GPU 调用会慢，详见后文 warmup 说明）
torch.mm(A_gpu, B_gpu)
torch.cuda.synchronize()

# 计时
start = time.perf_counter()
C_cpu = torch.mm(A_cpu, B_cpu)
cpu_time = time.perf_counter() - start

torch.cuda.synchronize()
start = time.perf_counter()
C_gpu = torch.mm(A_gpu, B_gpu)
torch.cuda.synchronize()
gpu_time = time.perf_counter() - start

print(f"CPU: {cpu_time*1000:.2f} ms | GPU: {gpu_time*1000:.2f} ms | 加速比: {cpu_time/gpu_time:.1f}x")
```

**预期输出**（以 RTX 4090 为例，实际数值取决于 CPU 和 GPU 型号）：

```
GPU: NVIDIA GeForce RTX 4090
SM 数量: 128
显存: 23.7 GB
CPU: 345.67 ms | GPU: 1.23 ms | 加速比: 281.0x
```

**预期输出**（以 A100 SXM 为例）：

```
GPU: NVIDIA A100-SXM4-80GB
SM 数量: 108
显存: 79.2 GB
CPU: 312.45 ms | GPU: 0.85 ms | 加速比: 367.6x
```

> 🔧 **新人参数解释**：
> - `torch.cuda.get_device_properties(0)`：获取 0 号 GPU 的硬件属性
> - `gpu.multi_processor_count`：SM 数量
> - `gpu.total_memory`：显存总量（字节）
> - `torch.mm`：矩阵乘法（调用 cuBLAS，底层用 Tensor Core）
> - `torch.cuda.synchronize()`：等待 GPU 完成所有工作（kernel launch 是异步的，不 sync 计时不准）

#### 2.1 `.to('cuda')` 背后发生了什么

```python
A_gpu = A_cpu.cuda()  # 等价于 A_cpu.to('cuda')
```

这一行代码背后：
1. 在 GPU HBM 上分配一块和 `A_cpu` 同样大小的显存
2. 通过 PCIe/NVLink 将 CPU 内存的数据拷贝到 GPU 显存（H2D 传输）
3. 返回一个指向 GPU 显存的张量引用

> 💡 **H2D 传输开销**：CPU→GPU 数据拷贝受 PCIe 带宽限制（~32 GB/s for PCIe 4.0），大张量传输很慢。实践中应尽量减少 H2D 传输——在 GPU 上初始化数据、在 GPU 上做预处理。

#### 2.2 为什么第一次 GPU 调用慢（warmup）

```python
# 第一次调用（慢）
torch.cuda.synchronize()
start = time.perf_counter()
torch.mm(A_gpu, B_gpu)  # 第一次
torch.cuda.synchronize()
first_call = time.perf_counter() - start  # 可能 50ms+

# 第二次调用（快）
start = time.perf_counter()
torch.mm(A_gpu, B_gpu)  # 第二次
torch.cuda.synchronize()
second_call = time.perf_counter() - start  # 可能 1ms
```

**原因**：
1. **CUDA 上下文初始化**：首次使用 GPU 时 PyTorch 会初始化 CUDA 上下文（加载驱动、分配资源）
2. **JIT 编译**：某些 kernel 首次运行时需要即时编译
3. **cuBLAS handle 创建**：首次调用 cuBLAS 会创建 handle

> 💡 **实践建议**：性能测试时一定要先 warmup（跑一次再计时），否则首次开销会严重扭曲结果。

### 3. 第一个 CUDA kernel：vecAdd（硬件视角）

下面是一个**完整可运行**的向量加法 CUDA 程序。别被长度吓到——核心 kernel 只有 5 行，其余都是"准备数据 → 搬到 GPU → 启动 kernel → 搬回结果 → 验证"的模板代码：

```c
#include <stdio.h>
#include <stdlib.h>
#include <cuda_runtime.h>

// ========== Kernel：GPU 上执行的函数 ==========
__global__ void vecAdd(float *A, float *B, float *C, int N) {
    int idx = blockIdx.x * blockDim.x + threadIdx.x;  // 算出我是第几个线程
    if (idx < N) {                                    // 边界检查（线程可能多于元素）
        C[idx] = A[idx] + B[idx];                     // 一个线程负责一个元素
    }
}

// ========== 主机端（CPU）代码 ==========
int main() {
    int N = 1000000;
    size_t size = N * sizeof(float);

    // ① CPU 分配内存 + 初始化数据
    float *h_A = (float *)malloc(size);
    float *h_B = (float *)malloc(size);
    float *h_C = (float *)malloc(size);
    for (int i = 0; i < N; i++) {
        h_A[i] = (float)i;
        h_B[i] = (float)(i * 2);
    }

    // ② GPU 分配显存
    float *d_A, *d_B, *d_C;
    cudaMalloc((void **)&d_A, size);
    cudaMalloc((void **)&d_B, size);
    cudaMalloc((void **)&d_C, size);

    // ③ 把数据从 CPU 搬到 GPU（H2D）
    cudaMemcpy(d_A, h_A, size, cudaMemcpyHostToDevice);
    cudaMemcpy(d_B, h_B, size, cudaMemcpyHostToDevice);

    // ④ 启动 Kernel
    int threadsPerBlock = 256;
    int blocksPerGrid = (N + threadsPerBlock - 1) / threadsPerBlock;  // 向上取整
    vecAdd<<<blocksPerGrid, threadsPerBlock>>>(d_A, d_B, d_C, N);

    // ⑤ 等待 GPU 算完（kernel launch 是异步的！）
    cudaDeviceSynchronize();

    // ⑥ 把结果从 GPU 搬回 CPU（D2H）
    cudaMemcpy(h_C, d_C, size, cudaMemcpyDeviceToHost);

    // ⑦ 验证结果
    int errors = 0;
    for (int i = 0; i < N; i++) {
        if (h_C[i] != h_A[i] + h_B[i]) errors++;
    }
    printf("Vector addition of %d elements\n", N);
    printf("%s\n", errors == 0 ? "✅ Result is correct!" : "❌ Errors found!");

    // ⑧ 释放资源
    cudaFree(d_A); cudaFree(d_B); cudaFree(d_C);
    free(h_A); free(h_B); free(h_C);
    return 0;
}
```

> 🔧 **新人逐段讲解**：
> - `__global__`：告诉编译器"这个函数在 GPU 上执行，从 CPU 调用"。这是 CUDA 的函数修饰符
> - `blockIdx.x * blockDim.x + threadIdx.x`：线程全局索引公式——"我属于第几个 Block × 每 Block 多少线程 + Block 内第几个 = 全局第几号"。类比：你在第 3 班（blockIdx），每班 256 人（blockDim），你是班内第 42 号（threadIdx），那你的学号就是 3×256+42 = 810
> - `<<<blocksPerGrid, threadsPerBlock>>>`：启动语法——告诉 GPU 开多少个 Block、每 Block 多少线程
> - `cudaMalloc` / `cudaMemcpy`：GPU 版的 malloc / memcpy，多了个方向参数（`cudaMemcpyHostToDevice` = CPU→GPU，`cudaMemcpyDeviceToHost` = GPU→CPU）
> - `cudaDeviceSynchronize()`：等 GPU 算完。因为 kernel 启动是异步的——CPU 发出指令后立刻返回，不等 GPU 执行完

#### 3.1 怎么编译和运行

**前提**：机器上有 NVIDIA GPU + CUDA Toolkit。先确认环境：

```bash
# ✅ 无需 sudo —— 检查 CUDA 编译器是否存在
nvcc --version
# 如果输出 nvcc 版本信息，说明 CUDA Toolkit 已安装

# ✅ 无需 sudo —— 检查 GPU 是否被识别
nvidia-smi
# 能看到 GPU 表格就 OK
```

> ⚠️ **如果 `nvcc --version` 报 command not found**：说明 CUDA Toolkit 未安装或未加入 PATH。在 Linux 服务器上通常装在 `/usr/local/cuda/bin/`，试试 `export PATH=/usr/local/cuda/bin:$PATH`。WSL 下需要安装 Windows 版 CUDA Toolkit（不是 WSL 内的）。详见 `../../CUDA编程/2. 基础篇/1. CUDA开发环境搭建.md`。

**编译 + 运行**（就两条命令）：

```bash
# 1. 把上面的代码保存为 vecAdd.cu（注意后缀是 .cu 不是 .c）
# 2. 编译
nvcc vecAdd.cu -o vecAdd

# 3. 运行
./vecAdd
```

**预期输出**：

```
Vector addition of 1000000 elements
✅ Result is correct!
```

> 💡 **第一次编译可能慢**：nvcc 首次编译会做一些初始化，后续编译会快很多。如果看到 warning 信息通常可以忽略（除非是 error）。

##### 常见编译报错排查

**报错 1：`unsupported GNU version! gcc versions later than 13 are not supported!`**

这是最常见的问题。原因：nvcc 编译 `.cu` 文件时，会调用系统（或 conda）的 g++ 作为"主机编译器"来编译非 GPU 部分。CUDA 12.4 只支持 GCC ≤ 13，但如果你的 conda 环境装了 GCC 14/15，nvcc 就会用它，然后报错。

```bash
# 诊断：看看 nvcc 用的是哪个 g++
which g++
# 如果输出 /mnt/xxx/conda/envs/xxx/bin/g++ ← conda 的 g++，版本太新

g++ --version
# 如果输出 15.x.0 ← 就是这个问题
```

**方案 A：用 `-ccbin` 指定系统自带的旧版 g++（临时编译推荐）**

```bash
# ✅ 无需 sudo —— 让 nvcc 用系统 g++ 而不是 conda 的
nvcc -ccbin /usr/bin/g++ vecAdd.cu -o vecAdd
```

**方案 B：降级 conda 环境的 GCC 到 13（长期开发推荐）**

一次性把 conda 环境的 g++ 降级到 CUDA 支持的版本，后续 `nvcc` 直接用 conda 的 g++ 即可，不需要每次加 `-ccbin`：

```bash
# 在目标环境中降级到 GCC 13（CUDA 12.4 支持 ≤ 13）
conda install -y -p <env路径> -c conda-forge "gxx=13" "gcc=13"

# 降级后直接编译，无需 -ccbin
nvcc vecAdd.cu -o vecAdd
```

> 📝 **方案 B 详细说明**（头文件 vs 运行时、CUDA-GCC 版本对照表、方案对比）见排错笔记：[Conda 环境安装指定版本编译器（CUDA 兼容）](../../troubleshooting/Python与Conda/Conda%20环境安装指定版本编译器（CUDA%20兼容）.md)

> 🔧 **原理讲解**：nvcc 的工作分两步——① 把 GPU 代码（`__global__` 函数）编译成 PTX → SASS；② 把 CPU 代码（`main` 函数等）交给主机编译器（g++）编译。`-ccbin` 就是指定第 ② 步用哪个编译器。conda 环境经常装了很新的 GCC（14/15），CUDA 12.x 不认识它的头文件，所以要用系统的旧 g++。

> 💡 **怎么找到系统 g++ 的路径**：
> ```bash
> # 方法 1：直接看 /usr/bin/ 下有哪些 g++
> ls /usr/bin/g++*
> # 可能输出：/usr/bin/g++  /usr/bin/g++-11  /usr/bin/g++-12
>
> # 方法 2：临时退出 conda 环境再看
> conda deactivate
> which g++
> ```

> ⚠️ **不要用 `-allow-unsupported-compiler`**：这个 flag 只是跳过版本检查，但 GCC 15 的 C++ 标准库头文件（`type_traits` 等）用了 nvcc 不认识的新语法（如 `__is_array`、`__is_pointer` 等），即使跳过版本检查也会报 48 个错误。正确的做法是换一个旧版编译器，不是强行用新版。

**报错 2：`error: no kernel image is available for execution on the device`**

说明编译时指定的 GPU 架构和你实际的 GPU 不匹配。加 `-arch` 参数指定：

```bash
# RTX 4090 (Ada, compute 8.9)
nvcc -ccbin /usr/bin/g++ -arch=sm_89 vecAdd.cu -o vecAdd

# A100 (Ampere, compute 8.0)
nvcc -ccbin /usr/bin/g++ -arch=sm_80 vecAdd.cu -o vecAdd

# H100 (Hopper, compute 9.0)
nvcc -ccbin /usr/bin/g++ -arch=sm_90 vecAdd.cu -o vecAdd
```

**报错 3：`nvcc command not found`**

CUDA Toolkit 未安装或未加入 PATH：

```bash
# 临时生效（当前终端）
export PATH=/usr/local/cuda/bin:$PATH

# 永久生效（写入 ~/.bashrc）
echo 'export PATH=/usr/local/cuda/bin:$PATH' >> ~/.bashrc
source ~/.bashrc
```

> 📌 **一图总结编译排查流程**：
> ```
> nvcc vecAdd.cu -o vecAdd 报错？
> │
> ├─ "unsupported GNU version" → conda 的 g++ 太新
> │   ├─ 方案 A（临时）：nvcc -ccbin /usr/bin/g++ vecAdd.cu -o vecAdd
> │   └─ 方案 B（长期）：conda install -c conda-forge "gxx=13" "gcc=13" 降级
> │
> ├─ "no kernel image" → GPU 架构不匹配
> │   └─ 解决：nvcc -ccbin /usr/bin/g++ -arch=sm_XX vecAdd.cu -o vecAdd
> │
> └─ "command not found" → CUDA Toolkit 没装或没加 PATH
>     └─ 解决：export PATH=/usr/local/cuda/bin:$PATH
> ```

#### 3.2 执行流程

理解代码后，整个程序的执行流程如下：

```
Host (CPU)                    Device (GPU)
   │                              │
   ├─ ① malloc + 初始化数据         │
   ├─ ② cudaMalloc(分配显存)        │
   ├─ ③ cudaMemcpy H2D ────────►  │  (数据到 HBM)
   ├─ ④ vecAdd<<<grid,block>>>()  │
   │  ──启动（异步）──►            ├─ Block 分配到各 SM
   │                              ├─ SM 内按 Warp(32) 调度执行
   │                              ├─ CUDA Core 执行 C[idx]=A[idx]+B[idx]
   ├─ ⑤ cudaDeviceSynchronize()  │  (等 GPU 算完)
   │  ◄────────── ⑥ cudaMemcpy D2H│  (结果回 CPU)
   ├─ ⑦ 验证结果                   │
   └─ ⑧ free / cudaFree            │
```

#### 3.3 把 vecAdd 的每个概念对应回硬件

| vecAdd 代码 | 对应硬件 | 说明 |
|------------|---------|------|
| `blocksPerGrid = 3907` | 3907 个 Block 分配到 SM | A100 有 108 个 SM，每个 SM 可运行多个 Block |
| `threadsPerBlock = 256` | 每 Block 256 线程 | 256 / 32 = 8 个 Warp |
| `threadIdx.x` | CUDA Core 执行 | 每个线程在一个 CUDA Core 上运行 |
| `blockIdx.x * blockDim.x` | Block 在 SM 上的索引 | 决定这组线程处理哪段数据 |
| `A[idx]` / `B[idx]` | HBM 全局内存访问 | 从显存读数据到寄存器 |
| `C[idx] = A[idx] + B[idx]` | CUDA Core 的 FMA 单元 | 在寄存器中做加法，写回 HBM |
| `cudaMalloc` | 在 HBM 上分配显存 | 对应 Global Memory |
| `cudaMemcpy H2D` | CPU 内存 → GPU HBM | 走 PCIe 总线（~32 GB/s） |

> 💡 **vecAdd 是 Memory-Bound**：每读 2 个 float（8 Byte）只做 1 次加法（1 FLOP），算术强度 = 1/8 = 0.125 FLOP/Byte，远低于任何 GPU 的平衡点。所以 vecAdd 永远卡在带宽上，Tensor Core 也帮不上忙（它只加速矩阵乘）。

## 面试回答（可直接口述的版本）

**问：Grid/Block/Thread 怎么映射到 GPU 硬件？**

答：Grid 映射到整个 GPU——一次 kernel 启动的所有线程。Block 映射到一个 SM——Block 被调度到 SM 上执行，Block 内线程可共享内存和同步。Thread 映射到 CUDA Core——最小执行单元。还有隐式的 Warp——32 个连续线程，SM 的 Warp Scheduler 以 Warp 为单位调度执行。这个三层层次既方便程序员思考（逻辑分组），又贴合硬件物理布局（SM/调度单元）。

**问：为什么 PyTorch 第一次 GPU 调用慢？**

答：三个原因：① CUDA 上下文初始化——首次使用 GPU 时 PyTorch 加载驱动、分配 CUDA 上下文；② JIT 编译——部分 kernel 首次运行时即时编译；③ cuBLAS handle 创建。所以性能测试一定要先 warmup 跑一次再计时，否则首次开销会严重扭曲结果。

**问：`.to('cuda')` 做了什么？数据搬运的开销怎么避免？**

答：`.to('cuda')` 做了三件事：①在 GPU HBM 上分配显存；②通过 PCIe 将 CPU 内存数据拷贝到 GPU（H2D 传输）；③返回 GPU 显存引用。H2D 传输受 PCIe 带宽限制（~32 GB/s），大张量很慢。避免方法：在 GPU 上直接初始化数据（如 `torch.randn(..., device='cuda')`）、在 GPU 上做预处理、避免频繁的 CPU↔GPU 数据来回。

## 深入追问（可能被追问的点）

- **Q1: kernel launch 为什么是异步的？**
  A: 如果同步等待 GPU 执行完，CPU 会空等。异步启动让 CPU 可以继续准备下一个 kernel 的数据、启动下一个 kernel，实现 CPU 和 GPU 的流水线并行。用 `torch.cuda.synchronize()` 或 CUDA Event 来显式等待。

- **Q2: vecAdd 为什么用不上 Tensor Core？**
  A: Tensor Core 只加速矩阵乘累加（D=A×B+C），vecAdd 是逐元素加法（C[i]=A[i]+B[i]），由 CUDA Core 的 ALU 执行。而且 vecAdd 算术强度极低（0.125 FLOP/Byte），是 Memory-Bound，即使算力翻倍也没用——瓶颈在带宽。

- **Q3: Block 大小为什么通常选 256？**
  A: 256 是 32 的倍数（8 个完整 Warp，无浪费），且不会占用过多寄存器/共享内存导致 SM 能容纳的 Block 数减少。详见 `../../CUDA编程/2. 基础篇/2. CUDA编程模型.md` 的 Block 大小选择策略。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|----------|
| Host vs Device | CPU vs GPU | Host=主机，Device=设备 |
| 同步 vs 异步 | 阻塞等待 vs 立即返回 | kernel launch=异步，synchronize=同步 |
| warmup vs steady | 首次开销 vs 稳定耗时 | warmup 后才测真实性能 |
| 逻辑层次 vs 物理层次 | Grid/Block/Thread（程序员定）vs GPC/SM/Warp/Core（硬件定） | 逻辑映射物理 |
| H2D vs D2H | CPU→GPU vs GPU→CPU | H=Host, D=Device |

- [x] 能说出 Grid/Block/Thread 到 GPC/SM/Core 的映射关系
  > **答**：
  > | 软件（CUDA） | 硬件 | 类比 |
  > |-------------|------|------|
  > | Grid | 整个 GPU（所有 GPC） | 学校 |
  > | Block | 一个 SM | 班级 |
  > | Thread | CUDA Core | 学生 |
  > | Warp（隐式） | SM 的 Warp Scheduler | 小组 |
  > Block 被调度到一个 SM 上执行，Block 内线程可共享内存和同步。→ 详见 §1.1 映射表

- [x] 能解释 CUDA 内存类型到硬件存储的映射（Global→HBM, Shared→L1, Register→寄存器）
  > **答**：
  > | CUDA 内存 | 硬件位置 | 说明 |
  > |----------|---------|------|
  > | Register | SM 寄存器堆 | 局部变量，最快 |
  > | Shared Memory | SM 片上存储 | `__shared__` 声明，Block 内共享 |
  > | Global Memory | HBM（显存） | `cudaMalloc` 分配，所有线程可见 |
  > | Local Memory | HBM（显存） | Register 溢出，实际在 HBM 上 |
  > | Constant Memory | HBM + 缓存 | `__constant__` 声明，只读 |
  > → 详见 §1.2 内存模型表

- [x] 能解释 `.to('cuda')` 背后发生了什么
  > **答**：做了三件事：①在 GPU HBM 上分配一块和输入同样大小的显存；②通过 PCIe 将 CPU 内存数据拷贝到 GPU（H2D 传输，受 PCIe 带宽限制 ~32 GB/s）；③返回指向 GPU 显存的张量引用。所以大张量搬移很慢，应尽量在 GPU 上直接初始化数据。→ 详见 §2.1 + 面试回答 Q3

- [x] 能解释为什么第一次 GPU 调用慢（warmup）
  > **答**：三个原因：①CUDA 上下文初始化——首次使用 GPU 时 PyTorch 加载驱动、分配 CUDA 上下文；②JIT 编译——部分 kernel 首次运行时即时编译；③cuBLAS handle 创建。所以性能测试一定要先 warmup（跑一次再计时），否则首次开销会严重扭曲结果。→ 详见 §2.2 + 面试回答 Q2

- [x] 能解释 kernel launch 为什么是异步的
  > **答**：如果同步等待 GPU 执行完，CPU 会空等。异步启动让 CPU 可以继续准备下一个 kernel 的数据、启动下一个 kernel，实现 CPU 和 GPU 的流水线并行。用 `torch.cuda.synchronize()` 或 CUDA Event 来显式等待 GPU 完成。类比：你让助手去做一件事，不会站在原地等他做完，而是继续干自己的活，需要结果时再问他要。→ 详见 深入追问 Q1

- [x] 能读懂 vecAdd 代码并解释每个概念对应硬件
  > **答**：
  > | 代码 | 对应硬件 | 说明 |
  > |------|---------|------|
  > | `blocksPerGrid = 3907` | Block 分配到 SM | A100 有 108 个 SM，每 SM 可运行多个 Block |
  > | `threadsPerBlock = 256` | 256/32 = 8 个 Warp | 每 Block 8 个 Warp |
  > | `threadIdx.x` | CUDA Core 执行 | 每线程在一个 CUDA Core 上 |
  > | `A[idx]` / `B[idx]` | HBM 全局内存访问 | 从显存读数据到寄存器 |
  > | `C[idx] = A[idx] + B[idx]` | CUDA Core 的 ALU | 寄存器中做加法，写回 HBM |
  > → 详见 §3.1 对应表

- [x] 能解释 vecAdd 为什么是 Memory-Bound
  > **答**：每读 2 个 float（8 Byte）只做 1 次加法（1 FLOP），算术强度 = 1/8 = 0.125 FLOP/Byte，远低于任何 GPU 的 ops:byte 比（H100 是 295）。瓶颈在带宽而不是算力，即使算力翻倍也没用——数据搬不过来。Tensor Core 也帮不上忙（它只加速矩阵乘，不加速逐元素加法）。→ 详见 §3.1 💡 + 深入追问 Q2

- [x] 能说出减少 H2D 传输的方法
  > **答**：①在 GPU 上直接初始化数据（`torch.randn(..., device='cuda')`）；②在 GPU 上做预处理（如数据增强）；③避免频繁的 CPU↔GPU 数据来回；④用 pinned memory 加速 PCIe 传输。核心原则：数据一旦上了 GPU 就别再拿回 CPU。→ 详见 §2.1 💡 + 面试回答 Q3

- [x] 能用 PyTorch 查看 GPU 的 SM 数量和显存
  > **答**：`gpu = torch.cuda.get_device_properties(0)` → `gpu.multi_processor_count`（SM 数量）、`gpu.total_memory`（显存总量字节）、`gpu.name`（GPU 名称）。→ 详见 §2 代码示例
