## 一句话结论

> **一句话结论**：长序列训练的核心矛盾是 Attention 计算量和激活显存的 $O(s^2)$ 增长——序列长度 $s$ 翻倍，计算和显存翻 $4$ 倍。上下文并行（Context Parallelism, CP）通过**沿序列维度切分 Attention 计算**来解决这个问题，三种主流方案代表了不同的切分哲学：Ring Attention 沿序列切 Q/K/V 后环形传递、Ulysses 沿 Head 切然后用 All-to-All 重排、Megatron CP 沿序列切后配合 FlashAttention 分块计算。选择 CP 方案本质是在通信模式（P2P vs All-to-All）和模型约束（Head 数 vs 序列长度）之间做权衡。

## 背景与现象

### 1. 这篇讲什么

- **序列长度墙**：为什么 2K → 128K 序列膨胀让单卡 OOM，以及为什么 TP/PP/DP 都解决不了这个问题
- **FlashAttention 的局限**：FlashAttention 降低了 $O(s^2)$ 显存但并没有减少 $O(s^2)$ 的**计算量**，当序列长到计算也单卡跑不动时需要进一步的分布式方案
- **三种上下文并行方案**：Ring Attention（环形传递 KV）、DeepSpeed-Ulysses（All-to-All 重排）、Megatron CP（序列切分+分块计算）
- **方案对比与选型**：各自的通信模式、限制条件、适用场景的对比

### 2. 为什么重要

- 2024-2025 年，LLM 的上下文窗口从 2K/4K 暴涨到 128K（GPT-4）、200K（Claude 3）、1M（Gemini 1.5 Pro）
- 长上下文是 Agent、RAG（检索增强生成）、长文档分析、代码库理解等场景的基础
- 训练长上下文模型必须先解决"单卡装不下一条序列"的问题
- 上下文并行是 3D 并行之外的"第四维度"，是大模型训练的必修课

### 3. 前置知识

- Attention 公式 $\text{Attention}(Q,K,V)=\text{softmax}(QK^T/\sqrt d)V$，以及 $Q,K,V\in\mathbb R^{s\times d}$
- FlashAttention 的 Tiling + Online Softmax 原理
- 集合通信原语：AllGather、All-to-All（参考 `..\集合通信基础\3. 通信原语篇\2. 集合通信五大原语.md`）
- SP（序列并行）与 CP（上下文并行）的区别：SP 切非 Attention 区域的激活，CP 切 Attention 计算本身

### 4. 在分布式训练学习链路中的位置

这是第 9 章，属于"进阶组合"中的第一环。前面你已经学完了五大并行策略的独立用法（DP、ZeRO、TP、PP），这一章引入第 4 个切分维度——序列维度。学完本章后你可以处理长上下文大模型的训练。

## 核心概念（新人友好讲解）

### 1. 序列长度墙：为什么 $s$ 翻倍一切翻 4 倍

Attention 的计算核心是 $S = QK^T$，这是一个 $s \times s$ 的矩阵乘法：

| 项目 | 复杂度 | 含义 |
|------|-------|------|
| 注意力分数矩阵 $S$ 大小 | $O(s^2)$ | $s\times s$ 的矩阵，存储所有 token 对之间的注意力分数 |
| 计算量 | $O(s^2 d)$ | 两次矩阵乘法：$QK^T$ 是 $O(s^2 d)$，$PV$ 是 $O(s^2 d)$ |
| 显存（无 FlashAttention） | $O(s^2)$ | $S$ 和 $P$ 各 $s\times s$ |
| 显存（有 FlashAttention） | $O(s)$ | 分块计算，不存 $s\times s$ 中间矩阵 |
| KV Cache（推理） | $O(s \cdot n_{\text{layers}} \cdot d)$ | 每层都要缓存 |

**关键数据**：假设 $d=128$（head dim），$s$ 从 4K 到 128K 的变化：

| 序列长度 | $S$ 矩阵大小（FP16） | 单 head Attention 计算量 | 影响 |
|---------|---------------------|----------------------|------|
| 4K | 32 MB | 4M FLOPs | 单卡轻松 |
| 32K | 2 GB | 256M FLOPs | 单卡吃力 |
| 128K | 32 GB | 4B FLOPs | 单卡 OOM + 算不过来 |
| 1M | 2 TB | 250B FLOPs | 必须分布式 |

### 2. 为什么 TP/PP/DP 都解决不了"单条序列太长"

| 策略 | 为什么解决不了 |
|------|--------------|
| TP（张量并行） | TP 沿 hidden dim 切矩阵，$s\times s$ 的注意力矩阵**维度不变**，每卡仍看到完整的 $s\times s$ |
| PP（流水线并行） | PP 沿层切，但每一层内部 Attention 的 $s\times s$ **维度不变**，单层还是 OOM |
| DP（数据并行） | DP 切 Batch（多口气跑多条序列），但**每条序列的长度不变**，单条序列仍 OOM |
| SP（序列并行） | SP 切 Dropout/LayerNorm 的激活，**不切 Attention 计算本身** |

**结论**：要处理超长序列，必须沿序列维度 $s$ 切分 Attention 计算——这就是上下文并行（CP）。

### 3. 上下文并行的基本思想

把一条序列的 token 分给多张 GPU，每张 GPU 只负责部分 token 的 Attention 计算：

```
                    原始序列（长度 s）
                    [t0, t1, t2, t3, t4, t5, t6, t7]
                              │
                   沿序列维度切分，分配给 4 张 GPU
                              │
         ┌─────────┬─────────┼─────────┬─────────┐
         ▼         ▼         ▼         ▼         ▼
      GPU 0     GPU 1     GPU 2     GPU 3
    [t0, t1]  [t2, t3]  [t4, t5]  [t6, t7]
       │
       └────► 每张 GPU 只负责自己那部分 token 的 Attention
              但需要拿到其他 GPU 的 K/V 才能完成完整的 Attention
              → 这就是不同 CP 方案解决的核心问题
```

三种方案的核心区别在于：**怎么让每张 GPU 拿到其他 GPU 的 K/V**。

## 深入讲解

## 第一部分：序列长度墙与 FlashAttention 回顾

### 1.1 Attention 计算量推导

标准 Scaled Dot-Product Attention：

$$
\text{Attention}(Q,K,V) = \text{softmax}\left(\frac{QK^T}{\sqrt d}\right)V
$$

其中 $Q, K, V \in \mathbb{R}^{s \times d}$（单头）。

**计算量分解**：

| 步骤 | 操作 | 计算量 |
|------|------|-------|
| ① | $S = QK^T$ | $2s^2 d$ FLOPs |
| ② | $P = \text{softmax}(S)$ | $O(s^2)$ FLOPs |
| ③ | $O = PV$ | $2s^2 d$ FLOPs |
| **合计** | | $4s^2 d$ FLOPs（约等于） |

**显存分解**（无 FlashAttention）：

| 数据 | 大小 | 格式 |
|------|------|------|
| $Q, K, V$ | $3sd$ | FP16 |
| $S = QK^T$ | $s^2$ | FP32（稳定 softmax） |
| $P = \text{softmax}(S)$ | $s^2$ | FP16 |
| $O$ | $sd$ | FP16 |
| **合计** | $2s^2 + 4sd$ | |

当 $s=128\text{K}, d=128$ 时：$S$ 和 $P$ 各需 $(128\times 1024)^2 \times 2 \approx 32$ GB——单张 A100 40G 已被吃光。

### 1.2 FlashAttention 的能力边界

FlashAttention 通过 Tiling + Online Softmax 把显存从 $O(s^2)$ 降到了 $O(s)$，但它**没有减少计算量**——$O(s^2 d)$ 的 FLOPs 仍然存在。

```
              FlashAttention 解决了什么？
              
  ┌─────────────────┬─────────────────────┐
  │  显存：O(s²)    │  计算量：O(s²d)      │
  │  FlashAttn → O(s) │  仍然 O(s²d)      │
  │  ✅ 不 OOM       │  ❌ 单卡算不过来     │
  └─────────────────┴─────────────────────┘
  
  当 s 长到计算也单卡跑不动的时候 → 需要上下文并行（CP）
```

**生活类比**：FlashAttention 是把仓库管理优化了（不用存整个 $s\times s$ 中间矩阵了），但**运送货物的总量没变**（计算量 $O(s^2 d)$ 不变）。当货物多到一个人搬不完时，就需要多个人一起搬——这就是 CP。

## 第二部分：Ring Attention

### 2.1 核心思想

Ring Attention 是三种方案中**最直观**的：沿序列维度切分 Q/K/V，然后让 K/V 在 GPU 之间**环形传递**，每张卡每收到一块 K/V 就计算局部 Attention。

```
  关键直觉：Attention 中 Q 和 K/V 的角色不对称
  - Q 不需要在卡间传递（每张卡用自己的 Q）
  - K/V 需要在卡间传递（计算时需要别人的 K/V）
  - 所以只需要环形传递 K/V，Q 原地不动
```

### 2.2 数据流图解（重点）

假设 4 张 GPU，序列切为 4 段。每张 GPU 有自己的 $Q_i$, $K_i$, $V_i$（$i \in \{0,1,2,3\}$）：

```
                        环形 KV 传递过程
                        
      GPU 0                  GPU 1                  GPU 2                  GPU 3
   [Q₀,K₀,V₀]             [Q₁,K₁,V₁]             [Q₂,K₂,V₂]             [Q₃,K₃,V₃]
        │                       │                       │                       │
        │    ★ 第 1 步 ★        │                       │                       │
        │  用自己 K/V 算局部 QKᵀ │                      │                       │
        │  O₀ += Q₀K₀ᵀ × V₀    │  O₁ += Q₁K₁ᵀ × V₁    │  O₂ += Q₂K₂ᵀ × V₂    │  O₃ += Q₃K₃ᵀ × V₃
        │                       │                       │                       │
        │   发送 K₀,V₀ ──────────► 发送 K₁,V₁ ──────────► 发送 K₂,V₂ ──────────► 发送 K₃,V₃
        │                  (环形右传)              │                       │
        │                       │                       │                       │
        │    ★ 第 2 步 ★        │                       │                       │
        │  收到 K₃,V₃（从GPU3）  │  收到 K₀,V₀（从GPU0）  │  收到 K₁,V₁（从GPU1）  │  收到 K₂,V₂（从GPU2）
        │  O₀ += Q₀K₃ᵀ × V₃    │  O₁ += Q₁K₀ᵀ × V₀    │  O₂ += Q₂K₁ᵀ × V₁    │  O₃ += Q₃K₂ᵀ × V₂
        │                       │                       │                       │
        │   发送 K₃,V₃ ──────────► 发送 K₀,V₀ ──────────► 发送 K₁,V₁ ──────────► 发送 K₂,V₂
        │                       │                       │                       │
        │    ★ 第 3 步 ★        │                       │                       │
        │  收到 K₂,V₂（从GPU3）  │  收到 K₃,V₃（从GPU0）  │  收到 K₀,V₀（从GPU1）  │  收到 K₁,V₁（从GPU2）
        │  O₀ += Q₀K₂ᵀ × V₂    │  O₁ += Q₁K₃ᵀ × V₃    │  O₂ += Q₂K₀ᵀ × V₀    │  O₃ += Q₃K₁ᵀ × V₁
        │                       │                       │                       │
        │   发送 K₂,V₂ ──────────► 发送 K₃,V₃ ──────────► 发送 K₀,V₀ ──────────► 发送 K₁,V₁
        │                       │                       │                       │
        │    ★ 第 4 步 ★        │                       │                       │
        │  收到 K₁,V₁（从GPU3）  │  收到 K₂,V₂（从GPU0）  │  收到 K₃,V₃（从GPU1）  │  收到 K₀,V₀（从GPU2）
        │  O₀ += Q₀K₁ᵀ × V₁    │  O₁ += Q₁K₂ᵀ × V₂    │  O₂ += Q₂K₃ᵀ × V₃    │  O₃ += Q₃K₀ᵀ × V₀
        │                       │                       │                       │
        ▼                       ▼                       ▼                       ▼
   O₀ 完成 = Q₀(K₀+K₁+K₂+K₃)ᵀV   O₁ 完成             O₂ 完成             O₃ 完成
```

**关键点**：
- 经过 $N-1$ 次传递（$N$ 张卡），每张卡收集到了所有其他卡的 K/V
- 每张卡计算了 $N$ 轮局部 Attention，每次都是小块的 $Q_i \times K_{\text{received}}^T$
- 传输与计算可以**重叠**（收到即可算，不等待所有 K/V 到齐）

### 2.3 Online Softmax 合并

Ring Attention 在合并多块 Attention 结果时，用的是与 FlashAttention 相同的 Online Softmax 递推公式。

设已处理 $j$ 块 K/V 后的状态：局部最大值 $m^{(j)}$，局部指数和 $\ell^{(j)}$，局部输出 $O^{(j)}$。第 $j+1$ 块到来后：

$$
S^{(j+1)} = Q \times K_{j+1}^T / \sqrt d
$$

$$
m^{(j+1)} = \max(m^{(j)}, \text{rowmax}(S^{(j+1)}))
$$

$$
P^{(j+1)} = \exp(S^{(j+1)} - m^{(j+1)})
$$

$$
\ell^{(j+1)} = \ell^{(j)} \cdot e^{m^{(j)} - m^{(j+1)}} + \text{rowsum}(P^{(j+1)})
$$

$$
O^{(j+1)} = \left(O^{(j)} \cdot \ell^{(j)} \cdot e^{m^{(j)} - m^{(j+1)}} + P^{(j+1)} \times V_{j+1}\right) / \ell^{(j+1)}
$$

⚠️ **注意**：合并结果与标准 Attention 在数学上**完全等价**（除浮点舍入误差），不是近似。

### 2.4 计算与通信重叠

Ring Attention 的核心工程优化：**当在计算局部 Attention 的同时，可以在后台异步传输下一块 K/V**。

```
  时间轴 →
  
  GPU 0:
  │── load K₁,V₁ ──│── compute Q₀K₁ᵀ ──│── load K₂,V₂ ──│── compute Q₀K₂ᵀ ──│
                     ├── recv K₂,V₂ ────┤                 ├── recv K₃,V₃ ────┤
                     (异步加载下一块)                        (异步加载下一块)
  
  理想情况：传输时间 ≤ 计算时间，传输完全被计算掩盖
```

**生活类比**：厨房在做当前这道菜的同时，助手已经把下一道菜的食材备好了。菜一道接一道，厨房不停转。

### 2.5 因果掩码下的负载不均衡

**问题**：带因果掩码（Causal Mask）的 Attention 中，每个 token 只能看到自己和之前的 token。当序列按 token 位置切分到不同 GPU 时：

```
  序列切分（4 GPU）：
  
  GPU 0: [t0, t1, t2]     t0 能看 t0        → 需要全部 3 个位置 K/V
                            t1 能看 t0,t1      → 需要前 2 个位置 K/V
                            t2 能看 t0,t1,t2   → 需要全部 3 个位置 K/V
  
  GPU 1: [t3, t4, t5]     t3 能看 t0..t3     → 需要 GPU0 的 K/V + GPU1 的第 1 个
                            t4 能看 t0..t4     → 需要 GPU0 的 K/V + GPU1 的前 2 个
                            t5 能看 t0..t5     → 需要 GPU0 的 K/V + GPU1 的全部
  
  GPU 2: [t6, t7, t8]     需要 GPU0 + GPU1 的全部 K/V
  GPU 3: [t9, t10, t11]   需要 GPU0 + GPU1 + GPU2 的全部 K/V
  
  → GPU 编号越大，需要等待的 K/V 越多，计算量也越大！
```

**优化方案**：

1. **Zigzag 排序**：将 token 按 Zigzag 顺序分配给 GPU，使每张 GPU 分到的"需要前面 token 数"大致均衡

2. **Striped Attention**：不是连续切分，而是交错切分：
   ```
   GPU 0: [t0, t4, t8, ...]    → 每张卡都分布在整个序列中
   GPU 1: [t1, t5, t9, ...]    → 每张卡都要差不多数量的前面 token
   GPU 2: [t2, t6, t10, ...]
   GPU 3: [t3, t7, t11, ...]
   ```
   缺点：破坏了数据局部性，增加通信开销

3. **计算不对称调度**：让负载轻的卡多做些额外计算或提前开始通信

### 2.6 Ring Attention 伪代码

```python
def ring_attention(q, k, v, cp_rank, cp_size):
    """
    q, k, v: [seq_chunk, n_heads, head_dim]
    cp_rank: 当前 GPU 的 CP 编号
    cp_size: CP 组中的 GPU 数量
    """
    seq_len = q.shape[0]
    head_dim = q.shape[-1]
    
    # 初始化统计量
    m = torch.full((seq_len,), -float('inf'), device=q.device)  # 行最大值
    l = torch.zeros((seq_len,), device=q.device)                 # 指数和
    O = torch.zeros_like(q)                                       # 输出
    
    # 确定相邻卡的 rank（环形拓扑）
    prev_rank = (cp_rank - 1) % cp_size
    next_rank = (cp_rank + 1) % cp_size
    
    # 当前持有的 K/V（初始为自己的）
    curr_k, curr_v = k, v
    
    for step in range(cp_size):
        # ① 计算当前 K/V 块的注意力分数
        scale = 1.0 / math.sqrt(head_dim)
        s = torch.matmul(q, curr_k.transpose(-2, -1)) * scale  # [seq_chunk, seq_chunk]
        
        # ② Online Softmax 合并（参考 FlashAttention 的递推公式）
        m_new = torch.max(m, s.max(dim=-1).values)
        p = torch.exp(s - m_new.unsqueeze(-1))  # 安全 softmax
        l_new = l * torch.exp(m - m_new) + p.sum(dim=-1)
        
        # ③ 更新输出
        O = (O * l.unsqueeze(-1) * torch.exp(m - m_new).unsqueeze(-1) + 
             torch.matmul(p, curr_v)) / l_new.unsqueeze(-1)
        
        # ④ 更新统计量
        m = m_new
        l = l_new
        
        # ⑤ 异步发送自己的 K/V 给下一张卡，同时接收上一张卡的 K/V
        if step < cp_size - 1:
            send_kv = (k, v)  # 始终发送原始的 K/V
            recv_k, recv_v = p2p_send_recv(
                send_kv, next_rank, prev_rank
            )
            curr_k, curr_v = recv_k, recv_v
    
    return O
```

### 2.7 Ring Attention 的通信量分析

每张卡：
- 发送 $N-1$ 次自己的 K/V：$(N-1) \times s/N \times d \times 2$（K 和 V）元素
- 接收 $N-1$ 次别人的 K/V：同量
- 总通信量：$2(N-1) \times s/N \times d \times 2 = 4(N-1)sd/N$ 元素
- 当 $N$ 大时，约等于 $4sd$

**关键结论**：Ring Attention 的通信量与**序列长度 $s$ 成正比**，与 CP 并行度 $N$ **无关**（$N$ 大时趋于常数 $4sd$）。

## 第三部分：DeepSpeed-Ulysses

### 3.1 核心思想

Ulysses 走了一条"绕路"的路线：**先沿 Head 维度切分 → 用 All-to-All 重排为序列维度切分**。

```
  为什么不直接沿序列切？
  → 因为"沿 Head 切"不需要修改 Attention 计算本身，刚好利用 All-to-All 完成维度转置
```

**生活类比**：Ulysses 像邮局的"分拣→重新打包→分发"流程。先把一堆信（tokens）按收件地址（Head）分到不同人手上，然后每个人用 All-to-All 把自己手上的信按街道（序列）重新分拣打包。

### 3.2 All-to-All 重排过程图解

```
  ★ Ulysses 的核心：两次 All-to-All 完成 Head↔序列的维度重排 ★
  
  ┌──────────────────────────────────────────────────────────────────┐
  │  初始状态：每个 GPU 持有全部序列长度，但只有部分 Head            │
  └──────────────────────────────────────────────────────────────────┘
  
  Q: [seq_full, n_heads/cp_size, head_dim]  ← 每个 GPU 的 Head 不同
  
  ┌──────────────────────────────────────────────────────────────┐
  │  ★ 第一次 All-to-All：沿 Head → 沿序列                       │
  │                                                              │

```

### 3.3 Ulysses 为什么需要 All-to-All？

All-to-All 的语义是：**每个节点把自己的数据分片发给所有节点，同时从所有节点接收对应分片的数据**。

```
  All-to-All 的维度转置视角：
  
  输入：GPU i 持有 [全部s, Head_i, d]  ← "我有全部序列，但只有部分 Head"
  
  All-to-All 的规则：
  - GPU i 把自己数据的第 j 段序列发给 GPU j
  - GPU i 接收所有卡发来的第 i 段序列
  
  输出：GPU i 持有 [Seq_i, 全部Head, d]  ← "我有全部 Head，但只有部分序列"
```

这一过程**恰好完成了一次维度转置**：从 `(全部序列, 部分Head)` 转换为 `(部分序列, 全部Head)`。

### 3.4 Ulysses 的通信量分析

两次 All-to-All，每次传输 $s \cdot N_h \cdot d / N$：

- 第一次：$s \cdot N_h \cdot d / N$（前向）
- 第二次：$s \cdot N_h \cdot d / N$（前向）
- 总通信量：$2 \cdot s \cdot N_h \cdot d / N$

**关键结论**：Ulysses 的通信量与**序列长度 $s$ 和 Head 总数 $N_h$ 都成正比**，但除以 CP 并行度 $N$。

**vs Ring Attention**：当 $N_h$（Head 数）大时，Ulysses 通信量更大；但当 $s$ 固定、$N_h$ 不大时，Ulysses 的通信量与 $N$ 成反比，可扩展性更好。

### 3.5 Ulysses 的关键限制

```
  ⚠️ Ulysses 的核心限制：Head 数必须 ≥ CP 并行度
  
  原因：沿 Head 维度切分时，每个 GPU 至少需要 1 个 Head
  → CP_size ≤ n_heads
  → 如果模型只有 8 个 Head，最多只能开 CP=8
```

**哪些模型受限制？**

| 模型 | Head 数 | 最大 CP 度 |
|------|--------|-----------|
| LLaMA-7B | 32 | 32（足够） |
| LLaMA-70B | 64 | 64 |
| Qwen-7B | 32 | 32 |
| GPT-2 Small | 12 | 12 |
| Phi-2 | 32 | 32 |

**生活类比**：Ulysses 像"按班级分组"，如果只有 12 个班级（Head），最多只能分成 12 组（CP=12），不能更多。

### 3.6 Ulysses 伪代码

```python
def ulysses_attention(q, k, v, cp_group):
    """
    q, k, v: [seq_len, n_heads_local, head_dim]
    每个 GPU 持有全部序列长度，但只有 n_heads/cp_size 个 Head
    """
    cp_size = cp_group.size()
    
    # ① 第一次 All-to-All：Head 切分 → 序列切分
    # 从 [seq, h_local, d] 变为 [seq/cp_size, h_local * cp_size, d]
    q_seq = all_to_all(q, cp_group)  # 沿 Head 维重排为序列维
    k_seq = all_to_all(k, cp_group)
    v_seq = all_to_all(v, cp_group)
    
    # ② 本地标准 Attention（每个 GPU 有全部 Head + 部分序列）
    scale = 1.0 / math.sqrt(q_seq.shape[-1])
    attn = torch.matmul(q_seq, k_seq.transpose(-2, -1)) * scale
    attn = torch.softmax(attn, dim=-1)
    out_seq = torch.matmul(attn, v_seq)
    
    # ③ 第二次 All-to-All：序列切分 → Head 切分（逆变换）
    out = all_to_all(out_seq, cp_group)
    
    return out
```

## 第四部分：Megatron Context Parallel

### 4.1 核心思想

Megatron CP 沿序列维度切分 Q/K/V，但它不环形传递 KV，而是通过**AllGather** 收集所有人的 K/V 或通过**P2P** 交换对应分片。Megatron CP 深度融合在 Megatron-LM 的架构中，与 TP/SP 协同工作。

```
  Megatron CP 的独特之处：
  
  1. 它是在 Megatron-LM 框架内实现的，和 TP/SP 原生配合
  2. 使用 AllGather 或 P2P 通信，不是 Ring 模式
  3. 与 FlashAttention 分块计算无缝集成
  4. 支持因果 Attention 的优化
```

### 4.2 与 TP/SP 的协同

在 Megatron-LM 中，一个 Transformer Block 的切分可以同时包含 TP + SP + CP：

```
  Megatron-LM 中 TP + SP + CP 的 Transformer Block：
  
  ┌─────────────────────────────────────────────────────────────┐
  │                   Transformer Layer                          │
  │                                                             │
  │   输入：[B, s, H]  (Batch, Seq, Hidden)                     │
  │                                                             │
  │   ★ 如果是 CP 第一段序列：                                   │
  │                                                     │
  │   ┌─── Self-Attention ──────────────────────────┐   │
  │   │  ┌──────────────┐  ┌──────────────────┐     │   │
  │   │  │ Column Parallel │      │  CP Attention    │     │   │
  │   │  │ QKV 投影     │      │  序列切分+AllGather │     │   │
  │   │  │ (TP 沿 H 切) │      │  (CP 沿 s 切)     │     │   │
  │   │  └──────┬───────┘  └────────┬─────────┘     │   │
  │   │         │                   │                │   │
  │   │         └───────┬───────────┘                │   │
  │   │                 ▼                            │   │
  │   │         ┌──────────────┐                     │   │
  │   │         │ FlashAttn 分块│                    │   │
  │   │         │ + TP AllReduce│                    │   │
  │   │         └──────────────┘                     │   │
  │   └──────────────────────────────────────────────┘   │
  │                                                       │
  │   ┌─── MLP ────────────────────────────────────┐     │
  │   │  Column Parallel + Row Parallel (TP)        │     │
  │   │  SP: 切 Dropout/LayerNorm 的激活            │     │
  │   └──────────────────────────────────────────────┘     │
  └─────────────────────────────────────────────────────────┘
```

### 4.3 Megatron CP 的通信模式

```
  ★ Megatron CP 的 AllGather 通信模式 ★
  
  初始化：序列 s 沿 CP 维度切分
  GPU 0: Q[0:s/N]  GPU 1: Q[s/N:2s/N]  GPU 2: Q[2s/N:3s/N] ...
  
  ┌─────────────────────────────────────────────────────┐
  │  前向传播：                                          │
  │                                                     │
  │  ① 各卡本地计算 QKV 投影（TP 可能已经切了）          │
  │  ② AllGather K 和 V（所有卡收集全部序列的 K/V）      │
  │  ③ 每卡用全部 K/V 和自己的 Q 计算 Attention          │
  │  ④ FlashAttention 分块计算（不存 s×s 矩阵）          │
  └─────────────────────────────────────────────────────┘
```

**AllGather 通信量**：每次收集 $s \times d$ 的 K 和 $s \times d$ 的 V，通信量 $\propto s \cdot d$。

## 第五部分：三种方案深入对比

### 5.1 核心差异总览

| 维度 | Ring Attention | DeepSpeed-Ulysses | Megatron CP |
|------|---------------|-------------------|-------------|
| **切分方式** | 沿序列切 Q/K/V | 先沿 Head 切，All-to-All 重排为序列切 | 沿序列切 |
| **通信原语** | P2P（环形传递） | All-to-All（2 次） | AllGather / P2P |
| **通信次数** | $N-1$ 次 P2P | 2 次 All-to-All | 1 次 AllGather |
| **通信量与 s** | $\propto s$ | $\propto s \cdot N_h$ | $\propto s$ |
| **通信量 vs N** | 与 N 无关 | $\propto 1/N$ | 与 N 无关 |
| **限制条件** | 无 | Head 数 ≥ CP 度 | Megatron 生态 |
| **因果掩码优化** | Zigzag/Striped | 天然均衡 | 天然均衡 |
| **计算通信重叠** | 可重叠 | 难重叠（全局同步） | 难重叠 |
| **框架支持** | 自实现/FlashAttention-3 | DeepSpeed | Megatron-LM |

### 5.2 通信量精确对比

假设 $s$ = 序列长度，$N_h$ = Head 总数，$N_{\text{cp}}$ = CP 并行度，$d$ = Head 维度：

| 方案 | 总通信量（每 GPU） | 简化 |
|------|-------------------|------|
| Ring Attention | $4(N_{\text{cp}}-1) \cdot s \cdot d / N_{\text{cp}}$ | $\approx 4sd$（N 大时） |
| Ulysses | $2 \cdot s \cdot N_h \cdot d / N_{\text{cp}}$ | $\propto s N_h / N_{\text{cp}}$ |
| Megatron CP | $2 \cdot s \cdot d$（AllGather K+V） | $2sd$ |

**关键差异**：
- Ring Attention 通信量不含 $N_h$（Head 数无关），适合 Head 多的模型
- Ulysses 通信量含 $N_h$，随 Head 数线性增长，但随 $N_{\text{cp}}$ 减小
- 在 Head 数小（如 8-16）且 CP 度高的场景，Ulysses 可能通信更少

### 5.3 选型决策树

```
  选择上下文并行方案：
  
  ┌── 你用 Megatron-LM 框架？
  │    └─ 是 → Megatron CP（原生支持 TP+SP+CP 协同）
  │
  ├── Head 数 ≤ CP 并行度？
  │    └─ 是 → 只能用 Ring Attention（Ulysses 有 Head 数限制）
  │
  ├── 序列非常长（128K+）？
  │    └─ 是 → Ring Attention（通信与 N 无关，可扩展性好）
  │        └─ 是因果 Attention？
  │            └─ 需要 Zigzag/Striped 优化来处理负载不均衡
  │
  ├── Head 数多（64+）且 CP 度高？
  │    └─ 是 → Ulysses（通信量 ∝ 1/N，高并行度时通信更少）
  │
  └── 你更愿意自己造轮子？
       └─ Ring Attention（概念简单，代码量小）
```

## 面试回答

### 面试题：什么是上下文并行（CP）？它解决了什么问题？

**标准回答**：

> 上下文并行（Context Parallelism, CP）是沿序列维度切分 Attention 计算的并行策略。标准的 Attention 计算量是 $O(s^2 d)$，当序列长度 $s$ 增长到 128K 甚至 1M 时，单卡无论在显存还是算力上都扛不住。TP（张量并行）、PP（流水线并行）、DP（数据并行）都解决不了这个问题——TP 沿 Hidden dim 切，PP 沿层切，DP 沿 Batch 切，没有一个碰序列维度。CP 通过把一条长序列的 tokens 分给多张 GPU，每张 GPU 只负责计算部分 tokens 的 Attention 输出来解决这个问题。三种主流实现是 Ring Attention（环形传递 KV）、DeepSpeed-Ulysses（All-to-All 维度重排）和 Megatron CP（序列切分+AllGather）。选择哪种方案取决于 Head 数、序列长度、框架生态。

## 深入追问

### 追问 1：Ring Attention 和 Ulysses 各自的优劣是什么？

- **Ring Attention 的优势**：① 无 Head 数限制，任何模型都能用；② 通信量与 CP 度 $N$ 无关（$4sd$），可扩展性好；③ 支持计算与通信重叠。**劣势**：① 因果掩码下负载不均衡，高编号的 GPU 要等更多 K/V；② 需要改算子实现，不能直接复用标准 Attention。
- **Ulysses 的优势**：① 本地是标准 Attention 计算，可直接复用 FlashAttention；② 通信量随 CP 度减小（$\propto 1/N_{\text{cp}}$），高并行度时通信更高效；③ 因果掩码下负载天然均衡。**劣势**：① **Head 数必须 ≥ CP 并行度**的限制；② All-to-All 是全局同步操作，难以掩盖通信延迟；③ 通信量随 $N_h$ 线性增长。

### 追问 2：SP（序列并行）和 CP（上下文并行）的区别是什么？

SP 切的是**非 Attention 区域**的激活（如 Dropout、LayerNorm 等），这些操作沿序列维度独立，不需要通信即可完成，只在跨序列边界时需要 AllGather 传递少量数据。CP 切的是 **Attention 计算本身**，需要通信来完成完整的 Attention。SP 通常和 CP 配合使用：SP 处理 Attention 前后的非 Attention 计算，CP 处理 Attention 本身。

### 追问 3：因果 Attention 下 Ring Attention 的负载不均衡怎么解决？

三种方法：① **Zigzag 排序**——把 token 按一定模式交替分配给 GPU，让每个 GPU 的"需要前面 token 数量"大致均衡；② **Striped Attention**——交错切分而非连续切分，每张卡分布在整个序列中；③ **非对称调度**——让计算量小的卡提前开始通信或承担额外计算。最实用的方案是 Zigzag + Striped 的组合，实践中负载不均衡通常控制在 10% 以内。

### 追问 4：为什么 Ulysses 的因果掩码负载天然均衡？

因为 Ulysses 在 Attention 计算前已经通过 All-to-All 变成了"部分序列 + 全部 Head"，Attention 计算是在分好的序列段内完成的。因果掩码只是限制了 token 之间能否互看（在段内），不改变计算量在各个 GPU 之间的分布。

## 易混淆点

| 易混淆对 | 本质区别 |
|---------|---------|
| **SP vs CP** | SP 切非 Attention 区域的激活（Dropout/LayerNorm），CP 切 Attention 计算本身。SP 的通信很小（只在序列边界）；CP 的通信是 Attention 正确性的核心 |
| **Ring Attention vs Ring AllReduce** | Ring Attention 是环形传递 **K/V 分块**（数据内容传递）；Ring AllReduce 是环形传递**规约后的结果**（加总/求平均）。两者都用了环拓扑，但目的完全不同 |
| **CP vs TP** | CP 沿**序列维度**切，切出后每张卡能看到部分 token 的完整 hidden；TP 沿**hidden 维度**切，切出后每张卡看到全部 token 的部分 hidden |
| **All-to-All vs AllGather** | AllGather：每张卡把自己数据发给所有人，结果每张卡有全部数据。All-to-All：每张卡把自己的 N 份分片分别发给 N 张卡，结果每张卡拿到不同来源的数据。Ulysses 用 All-to-All 恰好完成了一次**维度转置** |
| **Ulysses vs Ring Attention 通信量** | Ring Attention 的通信量不含 $N_h$（Head 数），Ulysses 含 $N_h$。Head 多（64+）且 CP 度高时 Ulysses 可能通信更少；Head 少时 Ulysses 通信量更小但受限制 |

    cp_size: 模拟的 GPU 数量
    """
    seq_len = q.shape[0]
    chunk_size = seq_len // cp_size
    head_dim = q.shape[-1]
    
    # 模拟各 GPU 的数据：列表，每个元素是一张卡的 Q/K/V
    q_chunks = [q[i*chunk_size:(i+1)*chunk_size] for i in range(cp_size)]
    k_chunks = [k[i*chunk_size:(i+1)*chunk_size] for i in range(cp_size)]
    v_chunks = [v[i*chunk_size:(i+1)*chunk_size] for i in range(cp_size)]
    
    # 标准 Attention 完整结果（ground truth）
    scale = 1.0 / math.sqrt(head_dim)
    s_full = torch.matmul(q, k.T) * scale
    p_full = torch.softmax(s_full, dim=-1)
    o_full = torch.matmul(p_full, v)
    
    # Ring Attention 模拟
    o_ring = []
    
    for rank in range(cp_size):
        # 每个 rank 维护自己的统计量
        m = torch.full((chunk_size,), -float('inf'))
        l = torch.zeros((chunk_size,))
        O = torch.zeros(chunk_size, head_dim)
        
        q_local = q_chunks[rank]
        
        for step in range(cp_size):
            # 当前迭代收到的 K/V（环形偏移）
            send_rank = (rank - step) % cp_size
            k_recv = k_chunks[send_rank]
            v_recv = v_chunks[send_rank]
            
            # 计算局部注意力
            s_local = torch.matmul(q_local, k_recv.T) * scale
            
            # Online Softmax 合并
            m_new = torch.max(m, s_local.max(dim=-1).values)
            p = torch.exp(s_local - m_new.unsqueeze(-1))
            l_new = l * torch.exp(m - m_new) + p.sum(dim=-1)
            
            O = (O * l.unsqueeze(-1) * torch.exp(m - m_new).unsqueeze(-1) + 
                 torch.matmul(p, v_recv)) / l_new.unsqueeze(-1)
            
            m = m_new
            l = l_new
        
        o_ring.append(O)
    
    o_ring = torch.cat(o_ring, dim=0)  # 拼接所有 rank 的输出
    
    # 验证：Ring Attention 结果应与标准 Attention 一致
    diff = (o_ring - o_full).abs().max().item()
    print(f'Ring Attention vs Standard Attention 最大误差: {diff:.2e}')
    assert diff < 1e-4, f'误差过大: {diff}'
    print('测试通过！Ring Attention 结果与标准 Attention 一致')

# 运行测试
torch.manual_seed(42)

seq_len = 2048
cp_size = 4

q = torch.randn(seq_len, head_dim)
k = torch.randn(seq_len, head_dim)
v = torch.randn(seq_len, head_dim)

ring_attention_sim(q, k, v, cp_size)
