## 一句话结论

> **一句话结论**：真实的大模型训练从不是"只用一个并行策略"，而是 TP×PP×DP 的 3D 并行（加上 EP/CP 后扩展到 4D/5D）。组合的三大原则：① 各维度切不同的东西，天然正交；② `world_size = TP × PP × DP`（× EP × CP）；③ 按通信频率从高到低映射到带宽从高到低——TP 放机内 NVLink，PP 可跨机 InfiniBand，DP 带宽要求最低可走最慢的链路。设计并行方案是"先装得下，再跑得快"的约束优化过程。

## 核心概念（新人友好讲解）

### 1. 为什么单一策略不够用

| 策略 | 能解决什么 | 天花板在哪 |
|------|-----------|-----------|
| **DP** | 加速训练（B 并行） | **不省参数显存**——每张卡仍存完整模型 |
| **ZeRO/FSDP** | 省优化器/梯度/参数 | 纯 ZeRO-3 通信量大（~3Ψ），大模型训练慢 |
| **TP** | 省激活显存（切隐藏维度） | **不能跨机**——通信量太大（每层都 AllReduce） |
| **PP** | 省参数显存（分层层存储） | **有 Bubble**——GPU 空闲等待 |
| **SP/CP** | 处理长序列 | 只解决序列维度，不解决其他维度 |

**结论**：大模型训练是"总有一款天花板适合你"——每种策略都有用但也都有局限，必须组合。

### 2. 混合并行的三大原则

#### 原则 1：维度正交性

```
  ★ 各维度切的是不同的东西，它们天然正交（互不干扰）★
  
  一条输入数据 [B, S, H]  → 一个 Transformer Block
  
  ┌────────────────────────────────────────────────────────┐
  │                                                        │
  │  DP 沿 Batch 维切 ──► 不同数据在同一模型副本上跑        │
  │                                                        │
  │  CP 沿 Sequence 维切 ──► 同一序列的不同 token 在不同卡    │
  │                                                        │
  │  TP 沿 Hidden 维切 ──► 同一矩阵乘法的列/行在不同卡      │
  │                                                        │
  │  PP 沿 Layer 维切 ──► 不同层在不同卡                    │
  │                                                        │
  │  EP 沿 Expert 维切 ──► 不同 Expert 在不同卡             │
  │                                                        │
  └────────────────────────────────────────────────────────┘
  
  它们之间没有"互斥"关系 → 可以同时开
```

#### 原则 2：world_size 约束

```
  ★ 基本恒等式 ★
  
  总 GPU 数量 = DP × TP × PP  （3D 并行）
  总 GPU 数量 = DP × TP × PP × EP  （加上 MoE → 4D）
  总 GPU 数量 = DP × TP × PP × EP × CP  （加上长序列 → 5D）
```

#### 原则 3：通信频率到硬件带宽的映射

```
  ★ 黄金法则：通信越频繁的，放带宽越高的硬件上 ★
  
  通信频率从高到低：        硬件带宽从高到低：
  
  TP：每层都通信（最频繁）   → NVLink（机内, 900GB/s A100）
  CP/SP：Attention 内通信    → NVLink（机内）
  EP：每层 MoE All-to-All   → NVLink 或 IB（看规模）
  PP：每个 micro-batch 传激活→ InfiniBand（跨机, ~50GB/s）
  DP：每步一次梯度同步        → InfiniBand / 以太网（最慢）
  
  ┌───────────────────────────────────────────────────────────┐
  │                                                           │
  │   单机内（NVLink 900GB/s）                                 │
  │   ┌─────────────────────────────────────────┐            │
  │   │  TP 组：8 卡内做张量并行                    │            │
  │   │  CP 组：如需 CP，通常也和 TP 放在同一机内    │            │
  │   └─────────────────────────────────────────┘            │
  │                                                           │
  │   跨机（InfiniBand ~50GB/s）                               │
  │   ┌─────────────────────────────────────────┐            │
  │   │  PP：跨机传激活和梯度                      │            │
  │   │  DP：跨机 AllReduce 梯度                   │            │
  │   └─────────────────────────────────────────┘            │
  │                                                           │
  │   跨机（以太网 ~10-25GB/s，作为补充）                       │
  │   ┌─────────────────────────────────────────┐            │
  │   │  大 DP 组可能跨越多节点                    │            │
  │   └─────────────────────────────────────────┘            │
  └───────────────────────────────────────────────────────────┘
```

## 深入讲解

## 第一部分：通信域划分详解

### 1.1 TP 为什么必须机内

**核心原因**：TP 每层 Transformer 都需要 AllReduce，通信频率最高。

```
  ★ TP 的通信频率 ★
  
  一个 Transformer Block 中 TP 的通信点：
  
  ┌──────────────────────────────────────────────────┐
  │  Self-Attention:                                 │
  │    QKV 投影（Column Parallel: f  AllReduce  g）  │
  │    Output 投影（Row Parallel:    g  AllReduce  f）│
  │                                                  │
  │  一个 Block 中至少 2 次 AllReduce                 │
  │  40 层 Transformer = 2 × 40 = 80 次 AllReduce    │
  │  1 个 micro-batch 就要 80 次往返通信               │
  └──────────────────────────────────────────────────┘
  
  如果 TP 跨机（InfiniBand, ~50 GB/s, ~2μs latency）：
  - 80 次 AllReduce × 2μs = 160μs 纯延迟
  - 每次 AllReduce 的数据量 ≈ H × 字节/元素
  - H=12288, FP16: 24KB × 80 次 = ~2MB
  - 在 50 GB/s 下：2MB / 50GB/s ≈ 40μs
  - 总计 ~200μs 通信 + kernel launch overhead
  - 而单层计算 ~几百μs → 通信开销不可接受
  
  如果 TP 机内（NVLink, 900 GB/s, <1μs latency）：
  - NVSwitch 直连：延迟 <1μs
  - 带宽 900 GB/s
  - → 通信开销可控（通常是计算 bound，不是通信 bound）
```

### 1.2 PP 为什么可以跨机

PP 的通信模式是：每个 micro-batch 的前向结束时把激活发给下一段，反向开始时把梯度发回上一段。

```
  ★ PP 通信模式分析 ★
  
  Stage 0                Stage 1                Stage 2
  [层 0-13]  ──激活──►  [层 14-27]  ──激活──►  [层 28-39]
             ◄──梯度──              ◄──梯度──
  
  通信量：
  - 每次 P2P 传输 = B × micro-size × H（bf16: 2 byte）
  - 假设 B=1, micro=1, H=12288:
    激活: ~24KB, 梯度: ~24KB
  - 每个 micro-batch 2 次 P2P
  - 总共: 4 × 24KB = 96KB/micro-batch
  
  vs PP 计算量（单 stage）:
  - 每层 ~8H² FLOPs, 14 层
  - 总计 ~8 × 14 × 12288² ≈ 16.9G FLOPs
  - 计算时间 >> 通信时间
  
  → PP 通信频率低（只在段边界）+ 通信量相对小 → 可跨机
```

### 1.3 DP 为什么通信要求最低

DP 只在每步（after backward）同步一次梯度，通信频率 = 1/步 vs TP = 每层。

```
  ★ 各策略通信量比较（以 7B, H=4096, 32 层为例）★
  
  ┌──────────┬────────────┬──────────────┬──────────┐
  │  策略    │ 通信频率    │ 每次通信量    │ 相对开支 │
  ├──────────┼────────────┼──────────────┼──────────┤
  │  TP      │ 每层 2 次   │ 2 × H × 2B   │ 最高     │
  │          │ (前+后=4次) │ ≈ 64KB      │          │
  ├──────────┼────────────┼──────────────┼──────────┤
  │  CP      │ Attention  │ ∝ s×d       │ 很高     │
  │          │ 每层 1-2 次 │              │          │
  ├──────────┼────────────┼──────────────┼──────────┤
  │  EP      │ 每 MoE 层   │ 2 × 2 ×      │ 高       │
  │          │ 2 次 A2A    │ M×K×H       │          │
  ├──────────┼────────────┼──────────────┼──────────┤
  │  PP      │ 段边界      │ B×S×H       │ 中       │
  │          │ 每个 micro  │              │          │
  ├──────────┼────────────┼──────────────┼──────────┤
  │  DP      │ 每步 1 次   │ 2Ψ           │ 低       │
  │          │             │ (模型大小×2) │          │
  └──────────┴────────────┴──────────────┴──────────┘
```

## 第二部分：Rank 编排

### 2.1 Rank 的多维坐标映射

在混合并行中，每个 GPU（rank）需要知道自己属于哪个 TP 组、PP 组、DP 组。这相当于给每个 rank 分配了一个**三维坐标**：

```
  ★ rank → (tp_rank, pp_rank, dp_rank) ★
  
  给定：TP=4, PP=2, DP=3 → world_size = 4×2×3 = 24
  （注意实际是虚拟的 DP，真实 DP 是数据并行）
  
  维度排列顺序（关键设计决策）：
  
  方案 A：tp 维度在最内层
  rank 0:  (0, 0, 0)  │ rank 12: (0, 0, 1)
  rank 1:  (1, 0, 0)  │ rank 13: (1, 0, 1)
  rank 2:  (2, 0, 0)  │ rank 14: (2, 0, 1)
  rank 3:  (3, 0, 0)  │ rank 15: (3, 0, 1)
  rank 4:  (0, 1, 0)  │ rank 16: (0, 1, 1)
  rank 5:  (1, 1, 0)  │ rank 17: (1, 1, 1)
  ...                   ...
  
  含义：
  - rank 0-3: 同一个 PP stage，同一个 DP 分组，不同 TP rank
    → 它们在同一台机器上做 TP（物理上相邻）
  - rank 4-7: 下一个 PP stage 的同一 DP 组
    → 它们在另一台机器上（PP 跨机）
  
  方案 B：pp 维度在最内层
  rank 0:  (0, 0, 0)
  rank 1:  (0, 1, 0)  ← PP 相邻 → 跨机
  rank 2:  (0, 0, 1)  ← DP 分组
  ...
  
  方案 A（tp 最内层）通常是最佳选择：
  → TP 组内 rank 连续 → 在同一台机器上
```

### 2.2 维度排序的物理意义

```
  ★ 维度排列顺序决定了哪些 rank 物理上相邻 ★
  
  规则：第一个维度（最内层）的 rank 应该在同一台机器上（NVLink 互联）
  
  ┌─────────────────────────────────────────────────┐
  │  维度排序              │  物理布局               │
  ├────────────────────────┼────────────────────────┤
  │  tp 最内              │  TP 组在同一节点上      │
  │  pp 次内              │  不同 PP stage 依次排布  │
  │  dp 最外              │  不同 DP 副本分散        │
  ├────────────────────────┼────────────────────────┤
  │  或（Megatron 常用）  │                        │
  │  tp 最内              │  同 TP 组在同一节点     │
  │  dp 次内              │  同一 stage 的 DP 在同一节点 │
  │  pp 最外              │  不同 stage 跨节点      │
  └────────────────────────┴────────────────────────┘
```

### 2.3 Megatron-LM 的 `initialize_model_parallel`

Megatron-LM 的初始化过程是理解 rank 编排的最佳入口：

```python
# Megatron-LM 核心初始化逻辑（简化版）

def initialize_model_parallel(
    tensor_model_parallel_size,  # TP 并行度
    pipeline_model_parallel_size,  # PP 并行度
):
    world_size = torch.distributed.get_world_size()
    
    # 计算 DP 并行度
    data_parallel_size = world_size // (
        tensor_model_parallel_size * pipeline_model_parallel_size
    )
    
    # ★ 核心：rank → 三维坐标映射 ★
    for rank in range(world_size):
        # 维度展开顺序（Megatron 默认：tp 最内，pp 次外，dp 最外）
        tp_rank = rank % tensor_model_parallel_size
        pp_rank = (rank // tensor_model_parallel_size) % pipeline_model_parallel_size
        dp_rank = rank // (tensor_model_parallel_size * pipeline_model_parallel_size)
        
        # 创建各通信组的 communicator
        # ...
    
    # 创建 TP 通信组：tp_rank 不同，pp_rank 相同，dp_rank 相同的 rank
    for dp in range(data_parallel_size):
        for pp in range(pipeline_model_parallel_size):
            tp_group_ranks = [
                dp * tp_size * pp_size + pp * tp_size + tp
                for tp in range(tp_size)
            ]
            # 创建 NCCL communicator
            tp_group = dist.new_group(tp_group_ranks)
    
    # 创建 PP 通信组：pp_rank 不同，tp_rank/dp_rank 相同
    # → PP 组是跨节点的（tp 粒度对齐）
    for dp in range(data_parallel_size):
        for tp in range(tensor_model_parallel_size):
            pp_group_ranks = [
                dp * tp_size * pp_size + pp * tp_size + tp
                for pp in range(pp_size)
            ]
            pp_group = dist.new_group(pp_group_ranks)
    
    # 创建 DP 通信组：dp_rank 不同，tp_rank 相同，pp_rank 相同
    for pp in range(pipeline_model_parallel_size):
        for tp in range(tensor_model_parallel_size):
            dp_group_ranks = [
                dp * tp_size * pp_size + pp * tp_size + tp
                for dp in range(dp_size)
            ]
            dp_group = dist.new_group(dp_group_ranks)
```

## 第三部分：拓扑设计实战

### 3.1 64 卡集群设计（8 节点 × 8 GPU）

这是面试中最常问的问题。假设 NVLINK 机内（900 GB/s），IB 跨机（50 GB/s）：

```
  ★ 场景：64 × A100 40G，训练一个 70B 模型 ★
  
  模型参数：
  - Hidden dim H = 8192
  - FFN intermediate = 28672
  - 80 层 Transformer
  - 70B 参数, BF16 ~140GB 模型权重
  - 单卡 40GB → 至少需要 140/40 ≈ 4 张卡放参数（不算优化器）
  - 算上优化器 ~140 × 3 = 420GB → 至少 11 张卡
  
  方案设计 → 满足 "world_size × GPU_mem > total_mem_needed"
  → 先从 TP 开始（机内 8 卡上限），再用 PP 跨机，最后 DP 填满
```

#### 方案 A：TP=8, PP=4, DP=2

```
  ★ 方案 A: TP=8, PP=4, DP=2 ★
  
  物理拓扑：
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  Node 0 (GPU 0-7)    Node 1 (GPU 8-15)                  │
  │  ★ TP=8 (机内)         ★ TP=8 (机内)                     │
  │  PP stage 0, DP 0     PP stage 0, DP 1                  │
  │                                                          │
  │  Node 2 (GPU 16-23)   Node 3 (GPU 24-31)                │
  │  ★ TP=8 (机内)         ★ TP=8 (机内)                     │
  │  PP stage 1, DP 0     PP stage 1, DP 1                  │
  │                                                          │
  │  Node 4 (GPU 32-39)   Node 5 (GPU 40-47)                │
  │  ★ TP=8 (机内)         ★ TP=8 (机内)                     │
  │  PP stage 2, DP 0     PP stage 2, DP 1                  │
  │                                                          │
  │  Node 6 (GPU 48-55)   Node 7 (GPU 56-63)                │
  │  ★ TP=8 (机内)         ★ TP=8 (机内)                     │
  │  PP stage 3, DP 0     PP stage 3, DP 1                  │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
  
  PP 阶段分配（80 层 / 4 段 = 每段 20 层）：
  Stage 0: 层 0-19
  Stage 1: 层 20-39
  Stage 2: 层 40-59
  Stage 3: 层 60-79
  
  TP=8 分析：
  ✅ TP=8 充分利用了单机 8 卡的 NVLink 带宽
  ✅ 单卡 hidden: 8192/8 = 1024 → 很小，TP 通信效率高
  ⚠️ 但 TP=8 可能导致单卡计算量太小 → GPU 利用率下降
  
  PP=4 分析：
  ✅ 每段 20 层，大小适中
  ⚠️ 4 段有 Bubble  
  1F1B 调度下 Bubble = (PP-1)/micro_batches ≈ (4-1)/M
  需要足够多的 micro-batches 来掩盖
```

#### 方案 B：TP=4, PP=8, DP=2

```
  ★ 方案 B: TP=4, PP=8, DP=2 ★
  
  物理拓扑：
  ┌──────────────────────────────────────────────────────────┐
  │                                                          │
  │  Node 0 (GPU 0-3: TP组0, GPU 4-7: TP组1)                │
  │    TP组0: PP stage 0, DP 0                               │
  │    TP组1: PP stage 1, DP 0                               │
  │                                                          │
  │  Node 1 (GPU 8-11: TP组2, GPU 12-15: TP组3)             │
  │    TP组2: PP stage 2, DP 0                               │
  │    TP组3: PP stage 3, DP 0                               │
  │                                                          │
  │  Node 2-3: PP stage 4-7, DP 0                           │
  │  Node 4-7: PP stage 0-7, DP 1                           │
  │                                                          │
  └──────────────────────────────────────────────────────────┘
  
  PP 阶段分配（80 层 / 8 段 = 每段 10 层）：
  Stage 0: 层 0-9,  Stage 1: 层 10-19, ...
  
  TP=4 分析：
  ✅ 单卡 hidden: 8192/4 = 2048 → 适中
  ✅ 每节点 2 个 TP 组，NVLink 带宽充足
  ⚠️ 但机内 2 个 TP 组共享 NVLink → 可能有竞争
  
  PP=8 分析：
  ⚠️ Bubble = (8-1)/M，需要更多 micro-batches
  ⚠️ 更多段 = 更多边界通信开销
  ✅ 每段只有 10 层，单卡激活显存小
```

#### 方案对比总结

| 维度 | 方案 A (TP=8, PP=4) | 方案 B (TP=4, PP=8) |
|------|---------------------|---------------------|
| **TP 利用率** | 充分利用 NVLink | 机内 2 组共享，可能有竞争 |
| **PP Bubble** | $3/M$（更小） | $7/M$（更大） |
| **单卡计算量** | 偏小（1/8 参数量） | 适中（1/4 参数量） |
| **单卡显存** | 更少（TP 切得更细） | 更多 |
| **通信效率** | TP 机内效率高 | TP 机内+跨 stage 更多 |
| **推荐场景** | 层数少、单层大 | 层数多、单层小 |

**选择建议**：
- 层数少（<40 层）+ 单层大（H>8192）→ 选方案 A（多 TP，少 PP）
- 层数多（>60 层）+ 单层小（H<4096）→ 选方案 B（少 TP，多 PP）

### 3.2 千卡集群配置思路

```
  ★ 千卡集群 (1024 GPU × A100 40G) ★
  
  场景：训练 175B 模型（GPT-3 规模）
  
  参数估算：
  - 模型参数: 175B × 2 (BF16) = 350 GB
  - 优化器状态: 175B × 4 (FP32) × 3 (m, v, master) = 2100 GB
  - 静态显存总计: ~2450 GB (最少)
  - 单卡 40G → 至少 62 张卡（仅静态显存）
  - 加上激活显存 → 至少 128+ 张卡
  
  推荐配置（Megatron 风格）：
  - TP = 8（单机 8 卡 NVLink → 充分利用机内带宽）
  - PP = 8（80-96 层 → 每段 10-12 层，大小适中）
  - DP = 1024 / (8 × 8) = 16
  
  每卡负载分析：
  ┌────────────────────────────────────────────────────────┐
  │  模型参数（BF16）     350 GB                             │
  │  TP 切分后（/8）     ~44 GB  每卡（BP 需要）             │
  │  PP 切分后（/8）     ~5.5 GB 每卡（前向存储）            │
  │  优化器状态（/TP/PP）~33 GB   每卡                       │
  │  静态总计           ~38.5 GB  每卡 ← 40GB 刚好够         │
  │  激活显存（含 AC）   ~1-2 GB   每卡                      │
  │  总计               ~40 GB    满负载                    │
  └────────────────────────────────────────────────────────┘
```

### 3.3 调参方法论

```
  ★ "先装得下，再跑得快" ★
  
  Step 1：确定最小 GPU 数量（能装下）
  ┌───────────────────────────────────────────────────┐
  │  总显存需求 = 参数(BF16) + 梯度 + 优化器 + 激活     │
  │  最小 GPU = ceil(总显存需求 / 单卡显存)            │
  │                                                   │
  │  对于 70B + AdamW:                                │
  │  - 参数: 140GB (BF16)                             │
  │  - 优化器: 840GB (FP32 × 3)                       │
  │  - 静态总计: 980GB → 至少 25 张 A100 40G           │
  │  - 加上激活 → 至少 32 张                           │
  └───────────────────────────────────────────────────┘
  
  Step 2：确定 TP（机内 8 卡上限）
  ┌───────────────────────────────────────────────────┐
  │  TP 越大 → 单卡模型越小 → 越省显存                   │
  │  但 TP > 8 → 必须跨机 → 性能急剧下降                │
  │  所以 TP 上限 = 单机 GPU 数                         │
  └───────────────────────────────────────────────────┘
  
  Step 3：确定 PP（根据层数）
  ┌───────────────────────────────────────────────────┐
  │  PP = 层数 / 每段层数                               │
  │  每段层数不宜太少（<4 层 → Bubble 比计算还多）       │
  │  也不宜太多（>30 层 → 单段太重，激活 OOM）          │
  │  一般 8-20 层/段是较好的范围                         │
  └───────────────────────────────────────────────────┘
  
  Step 4：DP 填满剩余 GPU
  ┌───────────────────────────────────────────────────┐
  │  DP = world_size / (TP × PP)                        │
  │  DP 越大 → 全局 Batch Size 越大 → 需调整 LR         │
  │  LR scaling: new_lr = base_lr × sqrt(DP_new/DP_old)│
  └───────────────────────────────────────────────────┘
  
  Step 5：微调优化
  ┌───────────────────────────────────────────────────┐
  │  - 尝试不同 TP 组合（4 vs 8）                       │
  │  - 调整 PP schedule（1F1B vs Interleaved）          │
  │  - 加入 Activation Checkpointing（省激活显存）       │
  │  - 加入 ZeRO-1/2（省优化器显存）                     │
  │  - 如果还 OOM → 减少 micro-batch size 或加更多 GPU │
  └───────────────────────────────────────────────────┘
```

## 面试回答

### 面试题：给你 64 张 A100 40G，训练一个 70B 模型，怎么配置并行策略？

**标准回答**：

> 首先做显存估算。70B 模型在 BF16 下约 140GB，FP32 优化器状态约 840GB（3 倍参数量），静态显存总计约 980GB。64 张 40G 卡总计 2560GB 显存，足够。然后设计并行配置：
>
> ① **TP 优先机内**（单机 8 卡上限），选 TP=8，充分利用 NVLink 900GB/s 带宽；
> ② **PP 根据层数**（假设 80 层），选 PP=4，每段 20 层，用 1F1B 调度减少 Bubble；
> ③ **DP 填满**：DP = 64/(8×4) = 2；
>
> 配置：TP=8, PP=4, DP=2。每卡显存约：参数 140GB/64 ≈ 2.2GB（BF16） + 优化器 840GB/64 ≈ 13GB + 激活 ~5GB = ~20GB per card，不到 40GB 上限的一半，有充足余量支持更大的 batch size 或更长的序列。
>
> 如果需要支持更大的 batch size 或更长的序列，可以降低 DP=1（只用 32 卡），或者减少 TP=4（释放更多 NVLink 给其他通信）。

## 深入追问

### 追问 1：TP=8 会不会使单卡计算量太小？

会。当 TP=8 时，单卡的计算量只有原来的 1/8。如果每卡分到的计算量太小，GPU 的计算单元（SM/Tensor Core）利用率会下降。可以用以下公式估算：

**单卡计算量** = 总计算量 / (TP × PP) ≈ $B \times S \times H^2 / TP$ 

当 $H/TP$ 太小时（如 < 1024），矩阵乘法的 GPU 利用率下降。所以 TP 不是越大越好——需要在"省显存"和"计算效率"之间折中。

参考经验值：
- H=8192, TP=8 → 单卡 hidden=1024 → 能接受
- H=4096, TP=8 → 单卡 hidden=512 → 偏小，建议 TP=4

### 追问 2：DP 和 ZeRO 能同时用吗？

可以，但通常 DP + ZeRO-1/2 就够了，不需要 ZeRO-3（因为 TP+PP 已经分担了参数）。如果不用 PP（纯 TP+DP），通常配合 ZeRO-2 或 ZeRO-3 来分担优化器状态。

### 追问 3：如果再加 MoE，怎么编排？

加上 EP（Expert Parallelism）后变成 4D 并行。总 GPU 数 = TP × PP × DP × EP。关键是把 EP 的 All-to-All 通信也尽量放在机内（或使用深度优化的跨机 All-to-All 如 DeepEP）。常见做法是 EP 组在机内（和 TP 共用 NVLink），DP 和 PP 跨机。

### 追问 4：PP 的 Bubble 在 3D 并行中还是问题吗？

是的，但可以通过多 micro-batch + 1F1B/Interleaved 调度来缓解。如果 micro-batches 数 M >> PP，Bubble 占比 ≈ (PP-1)/M 会很小。另一个选择是 Interleaved 1F1B（把一个 stage 分成多个 virtual stage），Bubble 进一步减小为 (PP-1)/(PP×M_virtual)。

## 易混淆点

| 易混淆对 | 本质区别 |
|---------|---------|
| **TP 跨机 vs 机内** | 跨机 TP 理论上可行但实践中几乎不用——通信频率太高以至于性能不堪。TP 必须机内是工程现实，不是理论限制 |
| **DP vs ZeRO** | DP 是策略（多份模型独立跑，同步梯度），ZeRO 是优化（切分优化器/梯度/参数以省显存）。两者经常一起用：DP 负责加速，ZeRO 负责省显存 |
| **PP Bubble vs TP 通信开销** | PP Bubble 是**空闲等待**（GPU 什么都不做），TP 通信开销是**额外操作**（GPU 在忙但部分资源在通信）。两者都是效率损失但形态不同 |
| **Rank 编号 vs 物理 GPU** | Rank 编号是逻辑编号（进程号），物理 GPU 是硬件编号（PCIe/NVLink 拓扑决定）。Megatron 默认 TP 最内层编排确保同一 TP 组的 rank 编号连续，对应到物理上同一节点的 GPU |
| **Micro-batch vs Global Batch** | Micro-batch 是单卡单次前向的数据量，Global Batch 是所有 DP 卡一次迭代的总数据量。$B_{\text{global}} = B_{\text{micro}} \times \text{gradient\_accumulation\_steps} \times DP$ |

### 实验 1：手算 3D 并行方案

```python
# ✅ 无需 sudo：并行方案计算器
def design_parallel_config(
    model_params_B,      # 模型参数量 (Billion)
    num_layers,          # 模型层数
    hidden_dim,          # 隐藏维度
    total_gpus,          # 总 GPU 数
    gpu_memory_GB,       # 单卡显存 (GB)
    gpus_per_node,       # 单节点 GPU 数（NVLink 互联）
):
    """为给定模型和集群设计并行方案"""
    
    # Step 1: 估算显存需求
    param_GB_BF16 = model_params_B * 2        # BF16: 2 bytes/param
    opt_GB_FP32 = model_params_B * 4 * 3      # FP32: 4 bytes, ×3 (m, v, master)
    total_static_GB = param_GB_BF16 + opt_GB_FP32
    total_GB_with_activation = total_static_GB * 1.15  # 加 15% 激活
    
    print(f"=== 显存估算 ===")
    print(f"模型参数(BF16):       {param_GB_BF16:.0f} GB")
    print(f"优化器状态(FP32×3):    {opt_GB_FP32:.0f} GB")
    print(f"静态显存总计:          {total_static_GB:.0f} GB")
    print(f"含激活(~15%):          {total_GB_with_activation:.0f} GB")
    print(f"最少需要 GPU 数:       {total_GB_with_activation / gpu_memory_GB:.0f} ({gpu_memory_GB}GB×卡)")
    
    # Step 2: 生成候选方案
    print(f"\n=== 候选并行方案 ===")
    
    schemes = []
    for tp in range(1, min(gpus_per_node, 8) + 1):
        if total_gpus % tp != 0:
            continue
        for pp in range(1, min(num_layers, 64) + 1):
            if total_gpus % (tp * pp) != 0:
                continue
            dp = total_gpus // (tp * pp)
            if dp < 1:
                continue
            
            # 每卡显存（粗估）
            mem_per_card = total_GB_with_activation / (tp * pp)
            
            # 每段层数
            layers_per_stage = num_layers / pp
            
            # PP Bubble 占比
            # 假设 micro_batches = 16
            M = 16
            bubble_ratio = (pp - 1) / M if pp > 1 else 0
            
            # TP 计算效率分数（hidden_dim/tp 越大越好）
            hidden_per_card = hidden_dim / tp
            tp_efficiency = min(hidden_per_card / 1024, 1.0)  # 1024 作为"好的起点"
            
            if mem_per_card < gpu_memory_GB and layers_per_stage >= 1:
                schemes.append({
                    'TP': tp, 'PP': pp, 'DP': dp,
                    'Mem/GPU': mem_per_card,
                    'Layers/Stage': layers_per_stage,
                    'Bubble%': bubble_ratio * 100,
                    'TP効率': tp_efficiency,
                    'TP Cross-Node?': tp > gpus_per_node,
                    'Score': 0  # 后面计算
                })
    
    # Step 3: 评分
    for s in schemes:
        # 得分 = 越接近 40% 显存利用率（留余量给 batch）+ 低 Bubble + 高 TP 效率
        mem_score = min(s['Mem/GPU'] / (gpu_memory_GB * 0.5), 1.0)  # 目标是 50% 利用率
        bubble_score = 1.0 - s['Bubble%'] / 100
        tp_score = s['TP効率']
        cross_node_penalty = 0.3 if s['TP Cross-Node?'] else 0
        
        s['Score'] = (mem_score * 0.3 + bubble_score * 0.3 + 
                      tp_score * 0.3 + (1.0 - cross_node_penalty) * 0.1)
    
    # 按分数排序
    schemes.sort(key=lambda x: x['Score'], reverse=True)
    
    # 输出 Top 5
    for i, s in enumerate(schemes[:5]):
        print(f"\n方案 {i+1}: TP={s['TP']}, PP={s['PP']}, DP={s['DP']}")
        print(f"  每卡显存:    {s['Mem/GPU']:.1f} GB / {gpu_memory_GB} GB ({s['Mem/GPU']/gpu_memory_GB*100:.0f}%)")
        print(f"  每段层数:    {s['Layers/Stage']:.0f}")
        print(f"  PP Bubble:   {s['Bubble%']:.1f}%")
        print(f"  单卡 Hidden:  {hidden_dim/s['TP']:.0f}")
        print(f"  TP 跨节点:    {'是 ⚠️' if s['TP Cross-Node?'] else '否 ✅'}")
        print(f"  综合评分:    {s['Score']:.3f}")


# 示例：70B 模型 + 64 张 A100 40G
print("=" * 60)
print("场景: 70B 模型, 64 × A100 40G (8 节点 × 8 GPU)")
print("=" * 60)
```

```python
# ✅ 无需 sudo：模拟 Megatron rank 编排
def simulate_rank_layout(tp, pp, dp):
    """打印 Megatron-LM 风格的 rank 编排矩阵"""
    world_size = tp * pp * dp
    
    print(f"\n=== Rank 编排 (TP={tp}, PP={pp}, DP={dp}) ===")
    print(f"World Size: {world_size}")
    print()
    
    # 按 tp 最内, pp 次内, dp 最外 打印
    header = "    " + " ".join([f"DP={d:2d}  " for d in range(dp)])
    print(header)
    
    for pp_r in range(pp):
        for tp_r in range(tp):
            row = f"PP={pp_r},TP={tp_r}:"
            for dp_r in range(dp):
                rank = dp_r * tp * pp + pp_r * tp + tp_r
                row += f"  {rank:4d}"
            print(row)
    
    print()
    print("--- 通信组划分 ---")
    
    # TP 组
    print(f"\nTP 组 ({tp} 卡/组, 共 {pp*dp} 组):")
    for dp_r in range(dp):
        for pp_r in range(pp):
            ranks = [dp_r * tp * pp + pp_r * tp + t for t in range(tp)]
            print(f"  DP{pp_r},PP{pp_r}: {ranks}")
    
    # PP 组
    print(f"\nPP 组 ({pp} 卡/组, 共 {tp*dp} 组):")
    for dp_r in range(dp):
        for tp_r in range(tp):
            ranks = [dp_r * tp * pp + p * tp + tp_r for p in range(pp)]
            print(f"  DP{pp_r},TP{tp_r}: {ranks}")
    
    # DP 组
    print(f"\nDP 组 ({dp} 卡/组, 共 {tp*pp} 组):")
    for pp_r in range(pp):
        for tp_r in range(tp):
            ranks = [d * tp * pp + pp_r * tp + tp_r for d in range(dp)]
            print(f"  PP{pp_r},TP{tp_r}: {ranks}")
```
