# 2.1 集合通信原语详解

> **一句话结论**：所有并行策略（DDP、ZeRO/FSDP、TP、MoE）的核心都是"切分 + 通信"。切分决定显存省不省，通信决定速度快不快。而通信的具体执行靠的是**6个集合通信原语**——AllReduce、ReduceScatter、AllGather、Broadcast、All-to-All、send/recv。它们就像一套固定的"传球战术"，吃透它们，看任何并行策略都能立刻看懂数据在 GPU 之间怎么流动。其中最重要的是记住**恒等式 AllReduce = ReduceScatter + AllGather**——它是 Ring AllReduce 的实现蓝图，也是理解 ZeRO/FSDP 通信量的基础。

## 背景与现象

### 为什么通信原语值得单独讲

假设你在训练一个 7B 参数的模型，用了 8 张 A100。你会遇到一个问题：多张卡之间怎么"通气"？

- 用 DDP 训练时，8张卡各自算了不同的梯度，怎么保证最后每张卡用同样的梯度更新参数？
- 用 FSDP/ZeRO-3 训练时，每张卡只存了 1/8 的参数，要用到某层的时候怎么"临时拼"出完整参数？
- 用 MoE 训练时，token 要根据路由结果跳到不同的 GPU 上的专家，怎么高效发过去？

这些问题的答案都落到同一件事上：**多张卡之间高效地交换数据**。而这是一套有固定名字、固定语义的操作——"集合通信原语"（Collective Communication Primitives）。

### 类比：球队的固定战术

打个比方，通信原语就像一支足球队的几套固定战术配合——传中、回传、二过一。球员（GPU）平时各管一摊，但到了关键时刻必须按既定套路把球（数据）准确传到位。把这几套"战术"吃透，后面看任何阵型（并行策略）都能立刻看懂球是怎么流动的。

### 一张全景表建立直觉

先看下面这张表，知道"哪个策略用哪个原语"，然后我们再逐个展开：

| 通信原语 | 典型使用它的并行策略 |
|---------|---------------------|
| **AllReduce** | DDP 梯度同步、TP 输出合并 |
| **ReduceScatter** | ZeRO-2/3、FSDP 反向 |
| **AllGather** | ZeRO-3 前向、FSDP 重组参数 |
| **All-to-All** | MoE 专家并行、Ulysses 序列并行 |
| **Broadcast** | 初始化时同步参数 |
| **send / recv** | 流水线并行传激活值（PP） |

> **重要提示**：通信发生在不同层级的硬件上，带宽差一个数量级。（关于硬件带宽的详细介绍，参见 `../集合通信基础/2. 硬件篇/1. 单机卡间通信-NVLink与NVSwitch.md`）机内 NVLink 可达数百 GB/s，机间 InfiniBand 约 50 GB/s 量级，PCIe 更低。**同样一个 AllReduce，放机内还是跨机，耗时可能差十倍**。这是后续所有"哪个策略放哪里"决策的物理根源。

## 核心概念

### 先搞懂几个基础名词

| 名词 | 含义 | 直觉理解 |
|------|------|----------|
| **Rank** | 参与通信的进程编号 | 球衣号码，每张 GPU 一个号 |
| **World Size（$N$）** | 总进程数 | 场上总共有几个球员 |
| **集合通信** | 一组进程共同参与、有固定语义的数据交换 | 固定的战术配合，不是乱跑 |
| **点对点通信（P2P）** | 两个进程之间收发数据 | 两个球员之间的一次传球 |

### 六大通信原语一图胜千言

| 原语 | 一句话解释 | 类比 | 输入→输出 |
|------|-----------|------|-----------|
| **Broadcast** | 一份数据复制给所有人 | 老师在黑板上写答案，全班看到 | root 持有 → 所有人拿到 |
| **Reduce** | 所有数据汇总到一个 rank | 课代表收集全班作业 | 所有卡 → 只汇总到一张卡 |
| **AllReduce** | 汇总后人人有份 | 课代表统计完，把结果复印发给每人 | 所有卡 → 所有人拿到相同结果 |
| **ReduceScatter** | 汇总后每人只留一片 | 每个人统计不同的题，最后每人只负责一道题的结果 | 所有卡 → 每人拿到一个分片 |
| **AllGather** | 每人一片，拼成完整数据 | 每人搜一个书架，最后互相通报，全馆完整清单 | 每人一片 → 所有人拿到完整数据 |
| **All-to-All** | 人人给人人发不同数据 | 交换座位游戏：每个人同时和所有人交换不同物品 | 每人发 N 份 → 每人收 N 份 |

### 最核心的恒等式（贯穿全书）

$$
\text{AllReduce} = \text{ReduceScatter} + \text{AllGather}
$$

直观理解：
- **ReduceScatter**：让每张卡拿到"自己那片数据的全局求和结果"
- **AllGather**：让各卡把自己的那片结果共享给所有人，拼成完整结果
- 两者合起来 = AllReduce

这个拆解不仅是理论恒等式，更是 **Ring AllReduce 的实现蓝图**，也是 ZeRO 能"用两个半步通信代替一次 AllReduce"的根本依据。

## 深入讲解

### 3.1 Broadcast：一份数据复制给所有人

一个根 rank 持有数据，复制到所有 rank。

```mermaid

        A0["rank0: A"]
        A1["rank1: —"]
        A2["rank2: —"]
    end
    subgraph after["Broadcast 后"]
        B0["rank0: A"]
        B1["rank1: A"]
        B2["rank2: A"]
    end
    before --> after
```

```
    ASCII 备用图：
    Broadcast 前:    Broadcast 后:
    GPU0: [A]        GPU0: [A]
    GPU1: [  ]       GPU1: [A]
    GPU2: [  ]       GPU2: [A]
```

换成具体数据看——根 rank（GPU0）持有 [1,2,3]，其余卡为空，Broadcast 后所有卡都拿到同一份拷贝：

```
操作前:          Broadcast 后:
GPU0: [1,2,3]    GPU0: [1,2,3]
GPU1: [ ]        GPU1: [1,2,3]
GPU2: [ ]        GPU2: [1,2,3]
GPU3: [ ]        GPU3: [1,2,3]
```

**典型用途**：训练开始时，把 rank 0 的初始模型参数广播给所有进程，保证各卡起点一致（DDP 初始化就是这么做的）。

### 3.2 Scatter / Gather：分发与收集

- **Scatter**：根 rank 把一个大张量切成 $N$ 份，第 $i$ 份发给 rank $i$（分发）
- **Gather**：每个 rank 把自己的数据交给根 rank，拼成完整张量（收集，Scatter 的逆操作）

以 4 卡为例，根 rank（GPU0）持有一个长度为 4 的张量，Scatter 把它逐段拆给各卡；Gather 则是反过来把各卡的数据收回根 rank 拼成完整张量：

```
Scatter（GPU0 分发）:        Gather（收集回 GPU0）:
操作前         操作后        操作前       操作后
GPU0:[a,b,c,d] GPU0:[a]      GPU0:[a]     GPU0:[a,b,c,d]
GPU1:[ ]       GPU1:[b]      GPU1:[b]     GPU1:[b]
GPU2:[ ]       GPU2:[c]      GPU2:[c]     GPU2:[c]
GPU3:[ ]       GPU3:[d]      GPU3:[d]     GPU3:[d]
```

注意 Scatter 与 Broadcast 的区别：Broadcast 把**同一份完整数据**复制给所有卡，Scatter 把数据**切开分发**，每卡只拿到其中一段。

**典型用途**：早期 DataParallel 用 Scatter 分发 mini-batch、Gather 收集输出；现在更多用于数据预处理阶段的切分。

### 3.3 Reduce：规约到一个 rank

所有 rank 的数据按某种运算（求和、求最大等）规约，结果只放到根 rank。

$$
\text{rank}_{\text{root}} \leftarrow \sum_{i=0}^{N-1} x_i
$$

具体到向量上——4 张卡各持一个长度为 3 的向量，Reduce(sum) 后只有根 rank（GPU0）拿到逐元素求和的结果，其余卡数据不变：

```
操作前:          Reduce(sum) 到 GPU0:
GPU0: [1,2,3]    GPU0: [10,20,30]
GPU1: [2,4,6]    GPU1: [2,4,6]    (不变)
GPU2: [3,6,9]    GPU2: [3,6,9]    (不变)
GPU3: [4,8,12]   GPU3: [4,8,12]   (不变)
```

> **易错点**：Reduce 与 AllReduce 的差别只在"结果给谁"——Reduce 只汇总到根 rank，AllReduce 则让**所有 rank 都拿到**同一份结果。这也是为什么 $\text{AllReduce} = \text{Reduce} + \text{Broadcast}$。

### 3.4 AllReduce：规约结果人人有份（本章最核心原语）

最重要的原语。所有 rank 的数据规约后，**结果广播回每一个 rank**。语义上等价于 `Reduce` 之后再 `Broadcast`，但实现上远比"先归约再广播"高效（见 Ring AllReduce 算法）。

```mermaid
graph LR
    subgraph before["AllReduce(sum) 前"]
        A0["rank0: 1"]
        A1["rank1: 2"]
        A2["rank2: 3"]
    end
    subgraph after["AllReduce 后"]
        B0["rank0: 6"]
        B1["rank1: 6"]
        B2["rank2: 6"]
    end
    before --> after
```

```
    ASCII 备用图：
    AllReduce 前:    AllReduce 后:
    rank0: 1         rank0: 6
    rank1: 2         rank1: 6
    rank2: 3         rank2: 6
```

如果把数据换成更贴近梯度的向量，过程会更直观——4 张卡各持一个长度为 3 的向量，AllReduce(sum) 后每张卡都拿到逐元素求和的结果：

```
操作前:          AllReduce(sum) 后:
GPU0: [1,2,3]    GPU0: [10,20,30]
GPU1: [2,4,6]    GPU1: [10,20,30]
GPU2: [3,6,9]    GPU2: [10,20,30]
GPU3: [4,8,12]   GPU3: [10,20,30]
```

注意求和是**按元素对齐**的：第一个分量 1+2+3+4=10，第二个分量 2+4+6+8=20，第三个分量 3+6+9+12=30。操作完成后四张卡的内容完全一致——这正是 DDP 里"所有卡梯度同步"的含义。

**典型用途**：DDP 同步梯度——每卡算出本地梯度后，AllReduce 求和（再除以 $N$ 取平均），使所有卡拿到一致的全局平均梯度；TP（张量并行）中也用它合并切分矩阵乘的部分和。

### 3.5 ReduceScatter：规约后只留自己那份

规约（求和）之后，每个 rank **只保留结果的一个分片**，而不是完整结果。

```mermaid

    subgraph before["ReduceScatter 前 (每 rank 持 N 个分片)"]
        A0["rank0: a0,a1,a2"]
        A1["rank1: b0,b1,b2"]
        A2["rank2: c0,c1,c2"]
    end
    subgraph after["ReduceScatter 后"]
        B0["rank0: a0+b0+c0"]
        B1["rank1: a1+b1+c1"]
        B2["rank2: a2+b2+c2"]
    end
    before --> after
```

```
    ASCII 备用图：
    ReduceScatter 前:          ReduceScatter 后:
    rank0: [a0, a1, a2]        rank0: [a0+b0+c0]
    rank1: [b0, b1, b2]        rank1: [a1+b1+c1]
    rank2: [c0, c1, c2]        rank2: [a2+b2+c2]
```

换成数值看更直观——4 张卡各持一个长度为 4 的向量（4 个分片），ReduceScatter(sum) 先把所有卡按元素求和，再让第 $i$ 张卡只保留第 $i$ 个分片：

```
操作前:            ReduceScatter(sum) 后:
GPU0: [1,2,3,4]    GPU0: [4]    (第 0 片求和: 1+1+1+1)
GPU1: [1,2,3,4]    GPU1: [8]    (第 1 片求和: 2+2+2+2)
GPU2: [1,2,3,4]    GPU2: [12]   (第 2 片求和: 3+3+3+3)
GPU3: [1,2,3,4]    GPU3: [16]   (第 3 片求和: 4+4+4+4)
```

**典型用途**：ZeRO-2/3 和 FSDP 反向传播——梯度规约后，每卡只需要自己负责更新的那一分片，正好用 ReduceScatter 一步到位。

### 3.6 AllGather：分片拼成完整数据

每个 rank 持有一个分片，操作后**每个 rank 都拿到拼接起来**的完整数据。它不做规约，只做拼接。

```mermaid
graph LR
    subgraph before["AllGather 前"]
        A0["rank0: a"]
        A1["rank1: b"]
        A2["rank2: c"]
    end
    subgraph after["AllGather 后"]
        B0["rank0: a,b,c"]
        B1["rank1: a,b,c"]
        B2["rank2: a,b,c"]
    end
    before --> after
```

```
    ASCII 备用图：
    AllGather 前:    AllGather 后:
    rank0: [a]       rank0: [a, b, c]
    rank1: [b]       rank1: [a, b, c]
    rank2: [c]       rank2: [a, b, c]
```

每张卡各持一个分片，AllGather 后所有卡都集齐了完整序列（只拼接、不求和）：

```
操作前:       AllGather 后:
GPU0: [A]     GPU0: [A,B,C,D]
GPU1: [B]     GPU1: [A,B,C,D]
GPU2: [C]     GPU2: [A,B,C,D]
GPU3: [D]     GPU3: [A,B,C,D]
```

> **直觉理解**：把这张图和 ReduceScatter 的图叠起来看——ReduceScatter 让每卡"从完整数据收缩到一个求和分片"，AllGather 让每卡"从一个分片扩张回完整数据"，二者正好互为逆向。

**典型用途**：ZeRO-3 / FSDP 前向，每卡只存参数分片，计算前用 AllGather 临时拼出完整层参数。

### 3.7 All-to-All：人人给人人发不同数据

最"全连接"的原语：每个 rank 都给其他每个 rank 发送一份**各不相同**的数据。可以理解为一次分布式的矩阵转置。

以 4 卡为例，记 $D_{ij}$ 表示"原本在 GPU$i$、要发给 GPU$j$"的那块数据。操作前每张卡按**目的地**排好 4 块数据；操作后每张卡收到的是所有卡发给它的数据，相当于把数据矩阵做了一次转置（行列互换）：

```
操作前(每卡按目的地排):       All-to-All 后(每卡收齐发给自己的):
GPU0: [D00,D01,D02,D03]       GPU0: [D00,D10,D20,D30]
GPU1: [D10,D11,D12,D13]       GPU1: [D01,D11,D21,D31]
GPU2: [D20,D21,D22,D23]       GPU2: [D02,D12,D22,D32]
GPU3: [D30,D31,D32,D33]       GPU3: [D03,D13,D23,D33]
```

可以看到第 $j$ 列（所有卡发给 GPU$j$ 的块）在操作后变成了 GPU$j$ 的那一行——这正是"矩阵转置"的含义。

**典型用途**：MoE 专家并行——token 要根据路由结果发往不同卡上的专家，正是 All-to-All；Ulysses 序列并行也用它在序列维和注意力头维之间切换切分方式。

### Ring AllReduce 算法原理

#### 4.1 朴素 AllReduce 的问题

最直接的实现：所有卡把数据发给一个中心 rank，中心求和后再广播回去。问题在于**中心节点成了带宽瓶颈**——$N$ 张卡的数据都要挤过它的一条网络链路，卡越多越堵。它的通信时间随 $N$ 线性增长，完全无法扩展。

打个比方：这就像全班作业都交给一个课代表统计再挨个发回，人越多课代表越累，效率越低。

#### 4.2 Ring：把所有卡连成一个环

Ring AllReduce 的思路是**去中心化**：把 $N$ 张卡逻辑上连成一个环，每张卡只和左右邻居通信，没有中心节点。它分两个阶段，正好对应恒等式 $\text{AllReduce} = \text{ReduceScatter} + \text{AllGather}$。

设每张卡持有大小为 $\Psi$ 的数据（如梯度），先把它切成 $N$ 个分片。

**阶段一：Reduce-Scatter（$N-1$ 步）**

每一步，每张卡把自己的一个分片发给下一个邻居，同时接收上一个邻居发来的分片并累加。经过 $N-1$ 步后，**每张卡上都有一个分片是"全局求和完成"的**（不同卡完成的是不同分片）。

**阶段二：All-Gather（$N-1$ 步）**

现在每张卡手里有一个"已求和好"的分片，再沿环传递 $N-1$ 步，把这些完整分片传遍所有卡。结束后每张卡都集齐了全部 $N$ 个求和后的分片，即完整的 AllReduce 结果。

```mermaid
graph LR
    R0["rank0"] --> R1["rank1"]
    R1 --> R2["rank2"]
    R2 --> R3["rank3"]
    R3 --> R0
```

```
    ASCII 备用图（Ring 拓扑）：
    rank0 → rank1 → rank2 → rank3
      ↑                      │
      └──────────────────────┘
```

#### 4.3 通信量推导：为什么与卡数无关

这是 Ring 算法的精髓。数据被切成 $N$ 份，每份大小 $\frac{\Psi}{N}$。两个阶段各 $N-1$ 步，每步每张卡发送一个分片：

$$
\text{每卡发送量} = \underbrace{(N-1)\cdot\frac{\Psi}{N}}_{\text{Reduce-Scatter}} + \underbrace{(N-1)\cdot\frac{\Psi}{N}}_{\text{All-Gather}} = 2\cdot\frac{N-1}{N}\Psi
$$

接收量与发送量相同。关键看这个系数：

$$
2\cdot\frac{N-1}{N}\Psi = 2\Psi\left(1 - \frac{1}{N}\right) \xrightarrow{N \text{ 较大}} 2\Psi
$$

> **核心结论**：当卡数 $N$ 较大时，每卡收发量趋近于 $2\Psi$，**与卡数 $N$ 无关**！这意味着无论用 8 卡还是 1024 卡做 DDP，每张卡每步的通信量都稳定在约 $2\Psi$，不会随规模膨胀——这正是 Ring AllReduce 能支撑大规模数据并行的根本原因。

#### 4.4 Ring vs Tree：什么时候用哪个

| 特性 | Ring AllReduce | Tree AllReduce |
|------|---------------|----------------|
| **步数** | $2(N-1)$，随 $N$ 线性增长 | $O(\log N)$，低延迟 |
| **带宽效率** | 大数据量时能逼近带宽上限 | 节点切换有开销 |
| **适用场景** | 梯度同步等大块数据（$\Psi$ 大） | 小张量、控制信号 |
| **延迟** | 随 $N$ 增长 | 低延迟 |

实际中 **NCCL 会根据数据量、卡数、拓扑自动选择** Ring / Tree 等算法（更多细节参见 `../集合通信基础/5. NCCL实战篇/1. NCCL通信库基础与PyTorch使用.md`），无需手动指定——这也是为什么用 `NCCL_DEBUG=INFO` 能看到它"选了哪种算法"。

### 通信量的量化方法

#### 5.1 统一记号

| 符号 | 含义 |
|------|------|
| $\Psi$ | 参数量（或待通信张量的元素数） |
| $N$ | 参与通信的 rank 数 |
| $\alpha$ | 单次通信的固定延迟（latency） |
| $B$ | 链路带宽（Bytes/s） |

#### 5.2 三类原语的每卡通信量

基于 Ring 实现，常用原语每卡的收发总量（大 $N$ 近似）：

| 原语 | 每卡通信量 | 记忆法 |
|------|-----------|--------|
| **AllReduce** | $2\Psi$ | ReduceScatter + AllGather |
| **ReduceScatter** | $\Psi$ | AllReduce 的一半 |
| **AllGather** | $\Psi$ | AllReduce 的另一半 |

> **关键点**：这张表解释了一个常见困惑——为什么 FSDP（ZeRO-3）每步通信量是 $3\Psi$ 而 DDP 只有 $2\Psi$。因为 FSDP 前向要 AllGather（$\Psi$）、反向要 AllGather（$\Psi$）+ ReduceScatter（$\Psi$），合计 $3\Psi$，比 DDP 多 50%——这正是它省显存付出的通信代价。

#### 5.3 通信时间估算模型（$\alpha$-$\beta$ 模型）

一次通信的耗时可以用经典的 $\alpha$-$\beta$ 模型估算：

$$
T \approx \alpha + \frac{\text{数据量}}{B}
$$

- **延迟项 $\alpha$**：每次通信的固定开销（建立连接、同步），与数据量无关。小数据量时它主导——这就是小张量适合用低延迟的 Tree 算法的原因。
- **带宽项 $\frac{\text{数据量}}{B}$**：数据量越大、带宽越低，耗时越长。大数据量时它主导——这就是大梯度适合用带宽最优的 Ring 的原因。

> **提示**：估算时务必用对带宽。机内 NVLink（数百 GB/s）和机间 IB（约 50 GB/s）代入同一个公式会得到差十倍的结果——这定量解释了"为什么 TP 必须放机内"。

#### 5.4 计算与通信重叠

最后一个关键优化思想：通信不必"干等"。现代框架会让**通信和计算并行进行**（overlap）——比如 DDP 在反向传播算后面层梯度的同时，就把前面已算好的梯度发出去做 AllReduce。

> **核心概念**：理想情况下，如果通信能完全被计算"盖住"，那么通信几乎"免费"。这就是 DDP 的 Bucket 机制、ZeRO 的 prefetch 等优化的共同目标——把 $T_{\text{comm}}$ 尽可能藏到计算时间里。

关于通信与计算 Overlap 的更多细节，参考 `../集合通信基础/4. 算法篇/3. 通信与计算Overlap.md`。

### 点对点通信：一切的基础

集合通信再花哨，底层都是由最朴素的"两个进程之间收发数据"组合而成的。这就是点对点通信（Point-to-Point, P2P）。

- `send(tensor, dst)`：把张量发给目标 rank
- `recv(tensor, src)`：从源 rank 接收张量

流水线并行（PP）的 Stage 之间传递激活值，靠的就是它。

**死锁陷阱**：

```python
import torch.distributed as dist

# ❌ 危险写法：两个进程都先 send 再 recv，可能死锁
# rank 0 和 rank 1 都卡在 send 等对方 recv
dist.send(tensor, dst=peer)
dist.recv(tensor, src=peer)
```

> **注意**：如果两个进程都用阻塞式 `send` 先发后收，且底层缓冲区不足以暂存，双方就会都卡在 `send` 上等对方来 `recv`，形成**死锁**。

**解决方案**：

```python
# 方案 A：错开顺序——一方先发后收，另一方先收后发
if rank == 0:
    dist.send(tensor, dst=1)
    dist.recv(tensor, src=1)
else:
    dist.recv(tensor, src=0)
    dist.send(tensor, dst=0)

# 方案 B：用异步原语 isend / irecv，先发起再统一等待
reqs = [dist.isend(send_buf, dst=peer), dist.irecv(recv_buf, src=peer)]
for r in reqs:
    r.wait()
```

> **关键点**：实际框架里更推荐 `dist.batch_isend_irecv()` 一次性提交一批收发请求，由后端统一调度，既避免死锁又能重叠通信。PP 的实现大量依赖这种异步批量收发。

### 动手实验

理论之后，用 `torch.distributed` 亲手验证这些原语的语义。下面的脚本用 `torchrun --nproc_per_node=4 collective_demo.py` 运行。

```python
# collective_demo.py
# 目的：亲手验证 AllReduce、AllGather、ReduceScatter 的输入输出语义
# 运行方式：NCCL_DEBUG=INFO torchrun --nproc_per_node=4 collective_demo.py
import os
import torch
import torch.distributed as dist


def main():
    dist.init_process_group(backend="nccl")
    rank = dist.get_rank()
    world_size = dist.get_world_size()
    torch.cuda.set_device(rank)
    device = torch.device(f"cuda:{rank}")

    # ---- 实验 1: AllReduce(sum) ----
    # 每卡持有自己的 rank 值，求和后应所有卡都得到 0+1+2+3=6
    x = torch.tensor([float(rank)], device=device)
    dist.all_reduce(x, op=dist.ReduceOp.SUM)
    print(f"[AllReduce] rank{rank} -> {x.item()}")  # 全部输出 6.0

    # ---- 实验 2: AllGather ----
    # 每卡持有一个分片，收集后每卡都拿到 [0,1,2,3]
    src = torch.tensor([float(rank)], device=device)
    gathered = [torch.zeros(1, device=device) for _ in range(world_size)]
    dist.all_gather(gathered, src)
    print(f"[AllGather] rank{rank} -> {[t.item() for t in gathered]}")

    # ---- 实验 3: ReduceScatter ----
    # 每卡提供 world_size 个分片，规约后各卡只留自己那份
    inputs = [torch.tensor([float(rank * 10 + i)], device=device)
              for i in range(world_size)]
    out = torch.zeros(1, device=device)
    dist.reduce_scatter(out, inputs, op=dist.ReduceOp.SUM)
    print(f"[ReduceScatter] rank{rank} -> {out.item()}")

    dist.destroy_process_group()


if __name__ == "__main__":
    main()
```

配合 `NCCL_DEBUG=INFO` 运行，还能观察 NCCL 实际选了哪种算法：

```bash
# ✅ 无需 sudo
NCCL_DEBUG=INFO torchrun --nproc_per_node=4 collective_demo.py 2>&1 | grep -i "Ring\|Tree\|Algorithm"
```

**建议练习**：把张量大小从 1 个元素逐步加大到几百 MB，用 `torch.cuda.Event` 计时，对比 AllReduce 实测带宽——你会看到小数据量时延迟主导、大数据量时逼近链路带宽，正好印证 $T \approx \alpha + \text{数据量}/B$ 模型。

## 面试回答

### 面试官问："请讲讲集合通信原语有哪些，分别用在什么场景？"

**口述回答**：

集合通信原语是多 GPU 间交换数据的固定模式，最常用的有 6 个：

1. **Broadcast**：一份数据复制给所有人。训练开始时 rank 0 把模型参数广播给所有卡，保证起点一致。
2. **AllReduce**：所有人数据求和/平均后，结果给所有人。DDP 用它同步梯度——每卡算出本地梯度后 AllReduce 取平均，所有卡拿到一致的梯度来更新参数。
3. **ReduceScatter**：求和后每卡只留一片。FSDP/ZeRO 用它——因为每卡只更新自己那 1/N 参数，只需拿到对应分片的聚合梯度，不用完整梯度。
4. **AllGather**：每人一片，拼成完整数据。FSDP 用它在计算前把散落的参数分片临时拼成完整参数。
5. **All-to-All**：人人给人人发不同数据。MoE 专家并行用它在 GPU 之间路由 token。
6. **send/recv**：两个进程间收发。流水线并行（PP）用它在 Stage 间传递激活值。

最重要的是记住 **AllReduce = ReduceScatter + AllGather**。这个恒等式解释了 Ring AllReduce 的实现方式（两个阶段各 N-1 步），也解释了 FSDP 比 DDP 多 50% 通信量的原因（FSDP 的通信量 3Ψ = 前向 AllGather + 反向 AllGather + 反向 ReduceScatter）。

## 深入追问

以下是面试中可能被追问的问题：

1. **Ring AllReduce 的单卡通信量怎么推导？为什么与卡数无关？**
   - 数据切 N 份，每份 Ψ/N，两个阶段各 N-1 步，每卡发送 2×(N-1)×Ψ/N = 2Ψ(N-1)/N ≈ 2Ψ（N 大时）。

2. **Ring 和 Tree AllReduce 各适合什么场景？为什么 NCCL 要同时支持两种？**
   - Ring 适合大数据量（带宽最优），Tree 适合小数据量（低延迟）。NCCL 根据数据量自动选择。

3. **FSDP 每步通信量 3Ψ 比 DDP 的 2Ψ 多出 50%，这额外的 Ψ 从哪来？**
   - 前向多一次 AllGather。DDP 只有反向一次 AllReduce(2Ψ)，FSDP 是前向 AllGather(Ψ) + 反向 AllGather(Ψ) + 反向 ReduceScatter(Ψ) = 3Ψ。

4. **Compute-Communication Overlap 的原理？Bounds Check / Bucket 机制？**
   - 反向从后往前算，靠后层梯度先就绪，不等全部算完就发起通信，让通信和前面的计算重叠。DDP 把梯度打包成 Bucket，一桶满了就触发 AllReduce。

5. **α-β 通信模型是什么？什么时候延迟主导，什么时候带宽主导？**
   - T ≈ α + 数据量/B。小数据量 α 主导，用 Tree；大数据量带宽项主导，用 Ring。

6. **All-to-All 的实现复杂度？它和 AllGather/ReduceScatter 的通信量有何不同？**
   - All-to-All 也可以 Ring 实现，每卡通信量也是 Ψ（与数据量成正比），但每个 step 发送的数据都不同，不像 AllReduce 是求和聚合。

7. **点对点 send/recv 死锁的条件和解决方法？异步 isend/irecv 的工作原理？**
   - 双方都用阻塞 send 先发后收 → 死锁。解决：错开顺序，或用 isend/irecv 异步发起再统一等待。

## 易混淆点

| 易混淆对 | 区别 | 记忆诀窍 |
|----------|------|----------|
| **AllReduce vs Reduce** | AllReduce 人人有份，Reduce 只有根 rank | "All" 就是"大家都有" |
| **ReduceScatter vs AllReduce** | ReduceScatter 每卡只留一片，AllReduce 每卡拿全部 | Scatter = 散射 = 每卡只有散片 |
| **AllGather vs Broadcast** | AllGather 拼接的是不同数据，Broadcast 复制的是同一份 | Gather = 收集不同碎片 |
| **AllGather vs Scatter** | AllGather 碎片→完整（收集分片），Scatter 完整→碎片（切开分发） | Gather 是收集，Scatter 是分发 |
| **All-to-All vs AllGather** | All-to-All 每个 rank 发不同数据给不同 rank，AllGather 发相同数据 | 矩阵转置 vs 拼接 |
| **Ring vs Tree AllReduce** | Ring 带宽最优（大 Ψ），Tree 低延迟（小 Ψ） | Ring=大，Tree=小 |

## 踩坑记录

| 坑 | 现象 | 原因 | 解决 |
|----|------|------|------|
| send/recv 死锁 | 训练卡住不动，GPU 占用 100% 但无进展 | 两进程都用阻塞 send 先发后收 | 错开发送顺序，或用 isend/irecv |
| 不同 rank 间调用不一致 | 部分 rank 卡住 | 如只在 rank 0 调用了 `dist.barrier()` | 集合通信必须所有 rank 同步调用 |
| 网络的带宽被低估 | DDP 通信超时或极慢 | 机内 NVLink 和跨机 IB 带宽差 10 倍 | 根据拓扑选择并行策略：高带宽放频繁通信 |
| find_unused_parameters | DDP 构造时报 Warning | 模型有未参与前向计算的参数 | 设置 `find_unused_parameters=True`，但有开销 |
| NCCL 超时 | 长时间通信无响应，训练 hang 住 | 节点间网络不通、防火墙阻断 | 运行 nccl-tests 测连通性 |
