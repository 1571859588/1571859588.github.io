# 4.2 PyTorch 数据并行从原理到实战

> **一句话结论**：本篇章是 4.1 原理篇的实战落地——用可运行的代码把 DDP 和 FSDP 讲清楚。DDP 适合单卡装得下的模型，核心只有三步：`init_process_group` → 包装 `DDP(model)` → 用 `DistributedSampler`；FSDP 适合单卡装不下的模型，额外需要配置分片策略和自动包装策略。多机训练用 `torchrun` 一键启动。踩坑集中在死锁（条件分支/barrier 不对称）、显存估算不准、模型保存方式（FSDP 要用 DCP）。

## 核心概念

### 先搞懂 5 个基本名词

| 概念 | 含义 | 类比 | 代码中怎么获取 |
|------|------|------|-------------|
| **World Size** | 参与训练的总进程数 | 总共有几位老师 | `dist.get_world_size()` |
| **Rank** | 每个进程的全局唯一编号 | 老师的工号（0 到 N-1） | `dist.get_rank()` |
| **Local Rank** | 进程在本机内的编号 | 本办公室内的座位号 | `os.environ["LOCAL_RANK"]` |
| **Backend** | 底层通信库 | 用什么语言交流 | `dist.init_process_group(backend="nccl")` |
| **Process Group** | 一组参与通信的进程 | 所有老师构成的群组 | 默认所有进程属于同一组 |

### 新手必读：这些 API 是什么

在深入代码之前，先理解 3 个核心 API 的作用：

- **`torch.distributed.init_process_group`**：分布式训练的"开机键"。它告诉 PyTorch："我这里有 N 个进程，它们之间要通过 NCCL 通信"。**所有分布式训练代码的第一行都必须调用它**。

- **`DistributedSampler`**：数据分发器。没有它，8 张卡都会读到完全相同的 8 份数据——等于 8 个老师批同一份卷子，数据并行就成笑话了。它保证每个 rank 拿到不重叠的数据分片。

- **`torchrun`**：PyTorch 自带的"多进程启动器"。你的训练脚本还是单卡的逻辑，`torchrun` 帮你复制 N 份、每份指定不同的 GPU、自动设置 `RANK`/`WORLD_SIZE` 等环境变量。

## 深入讲解

### 通信后端选择

PyTorch 支持三种通信后端，GPU 训练只用 NCCL：

| 后端 | 适用硬件 | 支持操作 | 说明 |
|------|---------|---------|------|
| **NCCL** | NVIDIA GPU | 几乎所有集合通信 | GPU 训练首选，性能最优 |
| **Gloo** | CPU / GPU | 大部分集合通信 | 跨平台，CPU 训练首选 |
| **MPI** | CPU / GPU | 取决于 MPI 实现 | 需单独安装 MPI 库 |

> **关键点**：GPU 分布式训练几乎都用 NCCL 后端。NCCL（NVIDIA Collective Communications Library）针对 NVLink、NVSwitch、InfiniBand 等 NVIDIA 互联做了深度优化。

### 集合通信原语速查表

分布式训练中的数据同步依赖集合通信操作：

| 原语 | 作用 | DDP/FSDP 中的用途 |
|------|------|------------------|
| **Broadcast** | 一个进程的数据发给所有进程 | 初始化时同步模型参数 |
| **AllReduce** | 所有进程数据聚合后发给所有进程 | DDP 梯度同步 |
| **ReduceScatter** | 先 Reduce 再 Scatter——聚合后分片 | FSDP 梯度同步 |
| **AllGather** | 每个进程的分片收集到所有进程 | FSDP 前向重组参数 |
| **Barrier** | 等所有进程到同一点再继续 | 同步控制（下载数据集等） |

> **提示**：AllReduce = ReduceScatter + AllGather 是理解 DDP 和 FSDP 通信差异的钥匙。详细原理见 `1. 数据并行详解-原理与通信量推导.md`。

### 初始化通信

所有分布式训练的第一步都是初始化进程组。

```python
import torch.distributed as dist
# 导入 PyTorch 的分布式通信库，dist 是 distributed 的缩写

# 方式一：通过环境变量初始化（最常用，配合 torchrun）
# torchrun 启动时会自动设置 MASTER_ADDR、MASTER_PORT、RANK、WORLD_SIZE、LOCAL_RANK
# 所以代码中不需要手动指定，一行搞定
dist.init_process_group(backend="nccl")
# backend="nccl" 表示使用 NCCL 通信库（NVIDIA GPU 专用）

# 方式二：显式指定参数（用于调试或特殊场景）
# 需要手动指定主节点地址、端口、全局进程数和当前进程号
dist.init_process_group(
    backend="nccl",                          # 使用 NCCL 后端
    init_method="tcp://192.168.1.1:29500",   # 主节点的 IP:端口，所有节点通过它"会合"
    world_size=8,                             # 总进程数 = 节点数 × 每节点 GPU 数
    rank=0,                                   # 当前进程的全局编号
)
```

### DDP：数据并行的标准方案

#### DDP 工作原理回顾

DDP（DistributedDataParallel）的核心机制：

1. **初始化**：将 rank 0 的模型参数 Broadcast 到所有进程，确保起点一致
2. **前向 + 反向**：每个进程在自己的数据分片上独立计算梯度
3. **梯度同步**：反向传播过程中，DDP 通过 **AllReduce** 将所有进程的梯度取平均

> **注意**：DDP 的梯度同步不是等反向传播完全结束后才开始的。它使用 Bucket 机制：将参数梯度按反向计算顺序分成若干 Bucket，每个 Bucket 的梯度就绪后立即发起 AllReduce，实现**计算与通信重叠**。详见 `1. 数据并行详解-原理与通信量推导.md`。

#### 完整 DDP 训练代码

下面是一个可直接运行的 DDP 训练示例。每个关键步骤都有注释解释"这行代码在干什么"：

```python
# ============================================================
# 完整的 DDP 训练脚本（train_ddp.py）
# 启动方式：torchrun --nproc_per_node=4 train_ddp.py
# ============================================================

import os
import torch
import torch.nn as nn
import torch.distributed as dist
# 上面两行：导入 PyTorch 核心模块和分布式通信模块
from torch.nn.parallel import DistributedDataParallel as DDP
# DDP 包装器：把普通模型变成分布式模型，自动处理梯度同步
from torch.utils.data import DataLoader, DistributedSampler
# DistributedSampler：确保每个 rank 读到不重叠的数据分片
from torchvision import datasets, transforms
# torchvision 提供 MNIST 数据集和图像预处理


def setup():
    """初始化分布式环境——每卡一个进程的第一件事"""
    dist.init_process_group(backend="nccl")
    # 初始化 NCCL 通信组，所有进程开始互相"认识"
    local_rank = int(os.environ["LOCAL_RANK"])
    # 从 torchrun 设置的环境变量中读取本机内的 GPU 编号
    torch.cuda.set_device(local_rank)
    # 告诉 PyTorch "这个进程只能用第 local_rank 块 GPU"
    return local_rank


def cleanup():
    """清理分布式环境——训练结束/崩溃时必须调用"""
    dist.destroy_process_group()
    # 释放 NCCL 通信资源，不调用可能导致下次启动异常


def main():
    local_rank = setup()
    # 获取本地 GPU 编号
    rank = dist.get_rank()
    # 获取全局进程编号（0, 1, 2, ...）
    world_size = dist.get_world_size()
    # 获取总进程数

    # ---- 1. 构建模型并包装为 DDP ----
    model = nn.Sequential(
        nn.Linear(784, 256),   # 输入 784 (28×28 MNIST)，输出 256
        nn.ReLU(),              # 激活函数
        nn.Linear(256, 10),    # 输出 10 类
    ).to(local_rank)
    # .to(local_rank) 把模型放到当前进程对应的 GPU 上
    model = DDP(model, device_ids=[local_rank])
    # DDP 包装：自动处理梯度 AllReduce + Bucket 优化
    # device_ids 指定模型在哪块 GPU 上

    # ---- 2. 构建数据加载器（必须使用 DistributedSampler） ----
    transform = transforms.Compose([
        transforms.ToTensor(),                        # PIL 图片 → Tensor
        transforms.Lambda(lambda x: x.view(-1)),     # 把 28×28 展平成 784 维向量
    ])
    dataset = datasets.MNIST("./data", train=True, download=True, transform=transform)
    # 下载 MNIST 数据集到 ./data 目录（只在 rank 0），其他 rank 会等待

    sampler = DistributedSampler(
        dataset,                         # 要切分的数据集
        num_replicas=world_size,         # 总共 N 个 rank
        rank=rank,                       # 当前 rank 编号
        shuffle=True                     # 不同 rank 的 shuffle 种子不同
    )
    # DistributedSampler 自动把数据集切成 world_size 份
    # rank 0 拿前 1/N 条，rank 1 拿下 1/N 条...

    dataloader = DataLoader(
        dataset,
        batch_size=64,         # 每个 rank 的 local batch size
        sampler=sampler,       # ⚠️ 注意：有了 sampler 就不能设 shuffle=True
        num_workers=2          # 用 2 个子进程加载数据
    )

    # ---- 3. 优化器和损失函数 ----
    optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
    # Adam 优化器，学习率 0.001
    criterion = nn.CrossEntropyLoss()
    # 分类任务的标准损失函数

    # ---- 4. 训练循环 ----
    model.train()  # 切换到训练模式（启用 Dropout/BatchNorm）
    for epoch in range(5):  # 训练 5 个 epoch
        sampler.set_epoch(epoch)
        # ⚠️ 关键：每个 epoch 调用一次，让每个 rank 的数据切分重新洗牌
        # 不调用的话，所有 epoch 数据分配完全一样
        total_loss = 0.0
        for batch_idx, (data, target) in enumerate(dataloader):
            data, target = data.to(local_rank), target.to(local_rank)
            # 把数据搬到当前进程的 GPU 上

            optimizer.zero_grad()   # 清零上一轮的梯度
            output = model(data)     # 前向传播
            loss = criterion(output, target)  # 计算损失
            loss.backward()         # 反向传播——DDP 在这里自动进行 AllReduce 梯度同步
            # backward() 里发生了什么：
            # 1. PyTorch 正常计算每个参数的梯度
            # 2. 梯度就绪后，DDP 的 Bucket 机制触发 AllReduce
            # 3. AllReduce 后，所有 rank 拿到相同的平均梯度
            optimizer.step()        # 用同步后的梯度更新参数

            total_loss += loss.item()

        if rank == 0:
            # ⚠️ 只在 rank 0 打印和保存，避免 8 个进程同时刷屏
            avg_loss = total_loss / len(dataloader)
            print(f"Epoch {epoch}: avg_loss = {avg_loss:.4f}")

    cleanup()


if __name__ == "__main__":
    main()
```

#### DDP 使用要点

**1. 必须使用 DistributedSampler**

```python
# sampler 把数据集均匀切给各 rank，确保每卡读到不同数据
sampler = DistributedSampler(
    dataset,
    num_replicas=world_size,  # 总共 world_size 份
    rank=rank,                 # 当前 rank 拿第 rank 份
    shuffle=True               # shuffle 种子 = epoch + rank，保证各 rank 不重复
)
dataloader = DataLoader(dataset, batch_size=64, sampler=sampler)
# ⚠️ 用了 sampler 就不能设 DataLoader 的 shuffle=True，否则会冲突
```

如果不使用 `DistributedSampler`，每个进程会读到相同的数据，数据并行就失去了意义。

**2. 每个 epoch 必须调用 `sampler.set_epoch(epoch)`**

```python
for epoch in range(num_epochs):
    sampler.set_epoch(epoch)  # 关键：打乱不同 epoch 的数据分配
    # 原理：DistributedSampler 用 epoch 作为随机种子的一部分
    # 不调用 = 所有 epoch 各 rank 数据完全相同
    for batch in dataloader:
        ...
```

不调用 `set_epoch` 会导致每个 epoch 各进程的数据分配完全相同。

**3. 只在 rank 0 上做日志和保存**

```python
if dist.get_rank() == 0:
    print(f"Loss: {loss.item()}")
    # 保存模型时注意：DDP 给参数名加了 "module." 前缀
    # 要用 model.module.state_dict() 去掉前缀再保存
    torch.save(model.module.state_dict(), "checkpoint.pt")
```

> **注意**：保存模型时要用 `model.module.state_dict()` 而不是 `model.state_dict()`。DDP 会给参数名加上 `module.` 前缀，直接保存会导致加载时 key 不匹配。

#### DDP 的局限

DDP 简单高效，但有一个根本性限制：**每块 GPU 都持有完整的模型副本**。对 7B 以上模型，单卡 80 GB 显存往往不够——这就是 FSDP 要解决的问题。

---

### FSDP：参数分片的显存优化方案

FSDP（FullyShardedDataParallel）是 PyTorch 对 DeepSpeed ZeRO 思想的原生实现——核心理念是**不让每块卡都存一份完整的模型，而是把参数、梯度、优化器状态分片存储**。

#### 从 ZeRO 理解 FSDP

| ZeRO 阶段 | 分片内容 | 每卡显存 | 对应 FSDP 策略 |
|-----------|---------|---------|---------------|
| **Stage 1** | 优化器状态 | $4\Psi + \frac{12\Psi}{N}$ | `SHARD_GRAD_OP` 的一部分 |
| **Stage 2** | + 梯度 | $2\Psi + \frac{14\Psi}{N}$ | `SHARD_GRAD_OP` |
| **Stage 3** | + 参数 | $\frac{16\Psi}{N}$ | `FULL_SHARD` |

以 7B 模型、8 卡为例：
- DDP：每卡 ~112 GB（装不下 80 GB 的 A100）
- FSDP FULL_SHARD：每卡 ~112/8 = ~14 GB（轻松装下）

#### FSDP 工作流程

1. **初始状态**：每块 GPU 只存储自己负责的参数分片
2. **前向传播**：通过 **AllGather** 收集分片 → 重组完整参数 → 计算 → 释放非本地分片
3. **反向传播**：AllGather 重组参数 → 计算梯度 → **ReduceScatter** 分片回各 GPU → 释放完整参数
4. **参数更新**：每块 GPU 只更新自己分片的参数和优化器状态

> **关键点**：FSDP 用**通信换显存**。每层的前向和反向都需要一次 AllGather + 一次 ReduceScatter，通信量比 DDP 更大（$3\Psi$ vs $2\Psi$），但显存占用大幅降低。

#### FSDP 分片策略速查

| 策略 | 行为 | 显存节省 | 通信开销 | 适用场景 |
|------|------|---------|---------|---------|
| `FULL_SHARD` | 参数+梯度+优化器全部分片 | 最高 | 最高 | 大模型，显存紧张 |
| `SHARD_GRAD_OP` | 梯度+优化器分片，参数不分片 | 中等 | 中等 | 中等模型 |
| `HYBRID_SHARD` | 机内全分片，机间数据并行 | 高 | 机间较低 | 多机训练首选 |
| `NO_SHARD` | 不分片（等价 DDP） | 无 | 最低 | 调试或小模型 |

> **提示**：`HYBRID_SHARD` 是多机训练的推荐策略。机内走 FSDP（利用高带宽 NVLink），机间走数据并行（适应较低的 InfiniBand 带宽），是通信和显存的最佳平衡。

#### FSDP 完整训练代码

```python
# ============================================================
# 完整的 FSDP 训练脚本
# 启动方式：torchrun --nproc_per_node=4 train_fsdp.py
# ============================================================

import os
import functools
import torch
import torch.nn as nn
import torch.distributed as dist
from torch.distributed.fsdp import (
    FullyShardedDataParallel as FSDP,  # FSDP 包装器
    MixedPrecision,                     # 混合精度配置（前向 BF16，优化器 FP32）
    ShardingStrategy,                   # 分片策略枚举（FULL_SHARD, SHARD_GRAD_OP...）
)
from torch.distributed.fsdp.wrap import size_based_auto_wrap_policy
# 自动包装策略：参数量超过阈值的子模块会被单独分片
# 这是性能关键——不是整个模型打一个包，而是每个子模块（如 Linear 层）各自分片
from torch.utils.data import DataLoader, DistributedSampler
from torchvision import datasets, transforms


def setup():
    """初始化分布式环境"""
    dist.init_process_group(backend="nccl")
    # 初始化 NCCL 通信组
    local_rank = int(os.environ["LOCAL_RANK"])
    # 从 torchrun 环境变量读本机 GPU 编号
    torch.cuda.set_device(local_rank)
    # 绑定当前进程到指定 GPU
    return local_rank


def cleanup():
    """清理分布式环境"""
    dist.destroy_process_group()
    # 释放 NCCL 资源


class SimpleMLP(nn.Module):
    """多层感知机模型，用于演示 FSDP 的使用"""
    def __init__(self, input_dim=784, hidden_dim=2048, num_layers=8, output_dim=10):
        super().__init__()
        layers = []
        for i in range(num_layers):
            in_d = input_dim if i == 0 else hidden_dim
            # 第一层输入 784，其余层输入 hidden_dim
            out_d = output_dim if i == num_layers - 1 else hidden_dim
            # 最后一层输出 10，其余层输出 hidden_dim
            layers.append(nn.Linear(in_d, out_d))
            # 添加 Linear 层
            if i < num_layers - 1:
                layers.append(nn.ReLU())
                # 除最后一层外都加 ReLU 激活
        self.net = nn.Sequential(*layers)
        # 把所有层打包成 Sequential

    def forward(self, x):
        return self.net(x)


def main():
    local_rank = setup()
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    # ---- 1. 创建模型 ----
    model = SimpleMLP().to(local_rank)

    # ---- 2. 配置混合精度 ----
    bf16_policy = MixedPrecision(
        param_dtype=torch.bfloat16,    # 前向和反向计算的参数用 BF16（省显存、加速）
        reduce_dtype=torch.bfloat16,   # 梯度 AllReduce 通信也走 BF16（省带宽）
        buffer_dtype=torch.bfloat16,   # BatchNorm 的 buffer 也用 BF16
    )
    # MixedPrecision 的意义：
    # - 前向/反向算得再快也只是 BF16，最终优化器更新还是 FP32
    # - 这保证了精度安全，同时大幅减少显存和通信量

    # ---- 3. 配置自动包装策略 ----
    wrap_policy = functools.partial(
        size_based_auto_wrap_policy,
        min_num_params=1_000_000,  # 参数量超过 100 万的子模块单独分片
    )
    # 为什么要自动包装？
    # 如果不分模块，FSDP 会把整个模型当成一个大的 flat tensor 来分片
    # 这样每次前向都要整个 AllGather 一次——效率极低
    # 按子模块包装 = 每个子模块独立分片 = 粒度更细 = 通信和计算可以更好地重叠

    # ---- 4. 用 FSDP 包装模型 ----
    model = FSDP(
        model,
        sharding_strategy=ShardingStrategy.FULL_SHARD,
        # FULL_SHARD：参数 + 梯度 + 优化器状态全部分片（ZeRO-3 级别）
        mixed_precision=bf16_policy,
        # 前向 BF16 + 优化器 FP32
        auto_wrap_policy=wrap_policy,
        # 按子模块粒度分片
        device_id=local_rank,
        # 模型放在哪块 GPU 上
    )

    # ---- 5. 数据加载 ----
    transform = transforms.Compose([
        transforms.ToTensor(),
        transforms.Lambda(lambda x: x.view(-1)),  # 展平 28×28 → 784
    ])
    dataset = datasets.MNIST(
        "./data", train=True,
        download=(rank == 0),  # 只让 rank 0 下载数据集，避免 8 个进程抢下载
        transform=transform
    )
    dist.barrier()
    # 等 rank 0 下载完成，所有 rank 到齐再继续
    # barrier 就像一个"集合点"：所有人到齐了才能走

    sampler = DistributedSampler(
        dataset,
        num_replicas=world_size,
        rank=rank,
        shuffle=True
    )
    dataloader = DataLoader(
        dataset,
        batch_size=64,
        sampler=sampler,      # 用 DistributedSampler 分配数据
        num_workers=2
    )

    # ---- 6. 优化器 ----
    optimizer = torch.optim.AdamW(model.parameters(), lr=1e-3)
    # 注意：FSDP 内部已做参数分片优化，不需要特殊处理 optimizer
    criterion = nn.CrossEntropyLoss()

    # ---- 7. 训练循环 ----
    model.train()
    for epoch in range(5):
        sampler.set_epoch(epoch)
        # 每个 epoch 重新洗牌
        total_loss = 0.0
        for data, target in dataloader:
            data, target = data.to(local_rank), target.to(local_rank)

            optimizer.zero_grad()   # 清零梯度
            output = model(data)     # 前向——FSDP 在此自动做 AllGather
            # forward() 里发生了什么（对每个 FSDP unit）：
            # 1. 检测到需要该 unit 的完整参数
            # 2. 触发 AllGather：所有卡贡献自己的分片，拼出完整参数
            # 3. 用完整参数完成前向计算
            # 4. 立即释放非本地分片（显存回落）
            loss = criterion(output, target)
            loss.backward()         # 反向——FSDP 在此自动做 AllGather + ReduceScatter
            # backward() 里发生了什么（对每个 FSDP unit）：
            # 1. 再次触发 AllGather 拼出完整参数
            # 2. 正常算出梯度
            # 3. 触发 ReduceScatter：梯度聚合 + 分片回各卡
            # 4. 释放完整参数和完整梯度
            optimizer.step()
            # optimizer.step() 只更新本地分片对应的参数

            total_loss += loss.item()

        if rank == 0:
            print(f"Epoch {epoch}: avg_loss = {total_loss / len(dataloader):.4f}")

    cleanup()


if __name__ == "__main__":
    main()
```

#### FSDP 模型保存与加载

FSDP 的模型保存比 DDP 复杂，因为参数是分片存储的。推荐使用 PyTorch 的 **Distributed Checkpoint**（DCP）：

```python
import torch.distributed.checkpoint as dcp
# PyTorch 的分布式 checkpoint 库：正确处理分片状态的保存和恢复
from torch.distributed.checkpoint.state_dict import (
    get_state_dict,   # 从 FSDP 模型中抽取完整的分片状态字典
    set_state_dict,   # 把加载的状态字典写回 FSDP 模型的分片
)

# ============================================================
# 保存（所有 rank 必须共同参与调用）
# ============================================================
# get_state_dict 会正确处理 FSDP 的参数分片，抽取完整状态
model_sd, optim_sd = get_state_dict(model, optimizer)
dcp.save(
    {"model": model_sd, "optim": optim_sd},
    checkpoint_id="./checkpoints/epoch_5",
    # checkpoint_id 是保存目录，会自动创建
)

# ============================================================
# 加载（所有 rank 必须共同参与调用）
# ============================================================
# 先获取当前分片的状态占位
model_sd, optim_sd = get_state_dict(model, optimizer)
dcp.load(
    {"model": model_sd, "optim": optim_sd},
    checkpoint_id="./checkpoints/epoch_5",
)
# 把加载好的状态写回模型和优化器
set_state_dict(
    model, optimizer,
    model_state_dict=model_sd,
    optim_state_dict=optim_sd,
)
# set_state_dict 把完整的参数拆成 N 份、各 rank 各拿自己的分片
# 这支持 resharding：保存时 8 卡、恢复时 4 卡也能正常加载
```

> **注意**：不要直接用 `torch.save(model.state_dict())` 保存 FSDP 模型。FSDP 的 `state_dict` 行为取决于 `state_dict_type` 配置，直接保存可能只落盘了本地分片。用 DCP + `get_state_dict` / `set_state_dict` 才能正确处理分片的保存与恢复。

#### FSDP2：下一代 API

PyTorch 正在推进新一代的 FSDP API——`torch.distributed.fsdp.fully_shard`（通常称为 FSDP2）。与 FSDP1 的 module-wrapper 方式不同，FSDP2 采用**可组合的 per-parameter 分片**方式：

```python
from torch.distributed.fsdp import fully_shard
# FSDP2 的入口：fully_shard 替代了 FSDP() 包装器

# FSDP2 风格：直接对每个子模块调用 fully_shard
for layer in model.layers:
    fully_shard(layer)
    # 每个 layer 独立分片，粒度更细
fully_shard(model)
# 最后对整体再调用一次，处理没有被 loop 覆盖的参数

# 优势：
# - 分片粒度到单个参数，更灵活
# - 基于 DTensor (Distributed Tensor)，与 TP 组合时接口统一
# - 更清晰的初始化与 checkpoint 语义
```

> **提示**：截至 PyTorch 2.x，FSDP2 已可用但仍在快速迭代中。新项目可以尝试 FSDP2，但生产环境中 FSDP1 仍然是更稳定的选择。

### DDP vs FSDP：如何选择

| 对比维度 | DDP | FSDP |
|---------|-----|------|
| **核心思路** | 每卡完整模型，梯度 AllReduce | 参数分片，按需 AllGather |
| **每卡显存** | 完整参数+梯度+优化器 | $\frac{1}{N}$ 参数+梯度+优化器 |
| **通信量/步** | AllReduce 梯度（$2\Psi$） | AllGather + ReduceScatter（$3\Psi$） |
| **通信模式** | 一次 AllReduce | 每层：AllGather(前向)+AllGather+ReduceScatter(反向) |
| **适用规模** | 单卡能装下 | 单卡装不下 |
| **代码复杂度** | 低（改动极小） | 中（需配置分片策略+包装策略） |
| **吞吐量** | 通信少，吞吐高 | 通信多，吞吐略低 |
| **模型保存** | `model.module.state_dict()` | 推荐 DCP |

**选择建议**：

- 模型参数+优化器状态能装进单卡显存 → 用 **DDP**
- 单卡装不下完整模型 → 用 **FSDP**
- 多机训练 + 大模型 → 用 **FSDP + HYBRID_SHARD**
- 超大模型（百亿级以上）→ 考虑 FSDP + TP + PP

### torchrun 与多机训练实战

#### torchrun 简介

`torchrun` 是 PyTorch 推荐的分布式训练启动工具（替代已废弃的 `torch.distributed.launch`）。核心能力：

- **自动设置环境变量**：`RANK`、`WORLD_SIZE`、`LOCAL_RANK`、`MASTER_ADDR`、`MASTER_PORT`
- **弹性训练支持**：支持节点动态加入/退出
- **容错机制**：Worker 进程失败时可自动重启

#### 单机多卡启动

```bash
# ✅ 无需 sudo
# 单机 4 卡
torchrun --nproc_per_node=4 train.py --batch_size 64

# 指定 master 端口（避免端口冲突，默认 29500）
torchrun --nproc_per_node=4 --master_port=29501 train.py
```

#### 多机多卡启动

多机训练需要在每台机器上分别启动 `torchrun`，指定相同的 rendezvous（会合）信息：

```bash
# ✅ 无需 sudo
# ============================================================
# 机器 0（master 节点）——先在 master 节点上运行
# ============================================================
torchrun \
    --nnodes=2 \
    # --nnodes：总共有几台机器参与训练
    --nproc_per_node=4 \
    # --nproc_per_node：每台机器启动几个进程（通常等于 GPU 数）
    --node_rank=0 \
    # --node_rank：当前机器的编号（master=0，其他机器=1,2,...）
    --rdzv_backend=c10d \
    # --rdzv_backend：会合（rendezvous）后端
    # c10d 是 PyTorch 内置的，不需要额外的 etcd/consul 等外部服务
    --rdzv_endpoint=192.168.1.100:29500 \
    # --rdzv_endpoint：master 节点的 IP:端口
    # 所有节点都向这个地址报到，等待全员到齐后一起开始训练
    train.py

# ============================================================
# 机器 1——在第二台机器上运行
# ============================================================
torchrun \
    --nnodes=2 \
    --nproc_per_node=4 \
    --node_rank=1 \
    --rdzv_backend=c10d \
    --rdzv_endpoint=192.168.1.100:29500 \
    # rdzv_endpoint 必须和 master 节点一致
    train.py
```

参数速查表：

| 参数 | 含义 | 示例 |
|------|------|------|
| `--nnodes` | 总机器数 | `--nnodes=2` |
| `--nproc_per_node` | 每台机器的进程数（通常=GPU 数） | `--nproc_per_node=4` |
| `--node_rank` | 当前机器的编号（从 0 开始） | `--node_rank=0` |
| `--rdzv_backend` | 会合后端（`c10d` 无需外部依赖） | `--rdzv_backend=c10d` |
| `--rdzv_endpoint` | 会合地址（master 节点的 IP:端口） | `--rdzv_endpoint=192.168.1.100:29500` |

#### 弹性训练

torchrun 支持弹性训练——允许节点数量在指定范围内动态变化：

```bash
# ✅ 无需 sudo
# 最少 1 台、最多 4 台机器
torchrun \
    --nnodes=1:4 \
    # --nnodes=min:max 表示最少和最多的机器数
    --nproc_per_node=4 \
    --rdzv_id=my_training_job \
    # --rdzv_id：训练任务的唯一标识，弹性训练时必须设置
    # 同一个 rdzv_id 的节点才能互相"会合"
    --rdzv_backend=c10d \
    --rdzv_endpoint=192.168.1.100:29500 \
    train.py
```

> **注意**：弹性训练要求训练代码能正确处理 checkpoint 的保存和恢复。当节点加入或退出时，torchrun 会重启所有 Worker，训练脚本需要从最近的 checkpoint 恢复状态。

#### 训练代码中获取分布式信息

使用 torchrun 启动时，训练代码中通过环境变量获取分布式信息：

```python
import os

# 以下环境变量由 torchrun 自动设置，无需手动配置
local_rank = int(os.environ["LOCAL_RANK"])
# LOCAL_RANK：本机内的 GPU 编号（0, 1, 2, 3）
# 用来绑定进程到具体 GPU：torch.cuda.set_device(local_rank)

rank = int(os.environ["RANK"])
# RANK：全局进程编号（跨所有机器的唯一编号）
# 两机各 4 卡：rank 0-3 在机器 0 上，rank 4-7 在机器 1 上

world_size = int(os.environ["WORLD_SIZE"])
# WORLD_SIZE：总进程数 = nnodes × nproc_per_node

local_world_size = int(os.environ["LOCAL_WORLD_SIZE"])
# LOCAL_WORLD_SIZE：本机的进程数 = nproc_per_node
```

## 分布式训练常见踩坑指南

### 死锁（Hang）

分布式训练最常见的问题就是"卡住不动"：

| 问题 | 现象 | 原因 | 解决 |
|------|------|------|------|
| 条件分支不同 | 某 rank 卡住，其他正常 | 某 rank 走了不同代码路径，跳过了 forward/backward | 确保所有 rank 执行相同的计算路径 |
| barrier 不对称 | 所有卡停住 | `if rank == 0: dist.barrier()` 只有 rank 0 调用 | 集合通信必须所有 rank 共同调用 |
| 数据大小不一致 | 某 rank 先结束 | 不同 rank 数据条数不同 | `DistributedSampler(drop_last=True)` |
| NCCL 超时 | 长时间卡住 | 网络不通或某 rank 计算太慢 | `NCCL_DEBUG=INFO` 排查 |
| find_unused_params | DDP 卡住 | 有条件分支的参数没参与前向，梯度永不就绪 | 设置 `DDP(model, find_unused_parameters=True)` |

### 常用调试环境变量

```bash
# ✅ 无需 sudo，在运行 torchrun 前设置即可

# 开启 NCCL 调试日志——死锁排查第一步
export NCCL_DEBUG=INFO
# NCCL_DEBUG=INFO 打印通信初始化信息（拓扑、算法选择等）
export NCCL_DEBUG_SUBSYS=ALL
# NCCL_DEBUG_SUBSYS=ALL 打印所有子系统的详细日志

# 增加集合通信超时时间（NCCL 后端默认 10 分钟）
# 在代码中设置：
# dist.init_process_group(..., timeout=datetime.timedelta(seconds=3600))

# 看门狗心跳监控超时：监控线程发现集合通信卡死多久后强制中止进程
# 注意它管的是"卡死检测/中止"，不是延长单次通信的超时时间
export TORCH_NCCL_HEARTBEAT_TIMEOUT_SEC=600

# PyTorch 分布式调试——更详细的错误信息
export TORCH_DISTRIBUTED_DEBUG=DETAIL
# DETAIL 级别会检查集合通信调用是否在所有 rank 上对称

# 指定可见 GPU，避免 GPU 编号错位（如机器上有 8 卡但只用 4 卡）
export CUDA_VISIBLE_DEVICES=0,1,2,3
# 这行在 torchrun 之前设置，确保只有指定的 GPU 被进程看到
```

### 随机种子一致性

分布式训练中的随机性管理需要特别注意：

```python
import torch
import random
import numpy as np

def set_seed(seed, rank):
    # 模型初始化需要一致的种子（保证 DDP 各 rank 参数一致）
    torch.manual_seed(seed)
    # 设置 PyTorch 的随机种子，保证模型权重初始化相同
    np.random.seed(seed)
    # 设置 NumPy 的随机种子
    random.seed(seed)
    # 设置 Python 标准库的随机种子

    # 数据增强等需要不同的种子（避免各 rank 数据完全相同）
    torch.manual_seed(seed + rank)
    # ⚠️ 重点：
    # - 模型初始化用固定种子 seed（所有 rank 初始化完全相同的参数）
    # - 数据增强/Dropout 用 seed + rank（让每个 rank 的随机行为不同）
    # - 如果 Dropout 也用相同种子，各 rank 的 dropout mask 完全一样
    #   就失去了"不同数据路径"带来的正则化效果
```

> **关键点**：DDP 要求所有 rank 的初始参数一致（DDP 构造时会从 rank 0 Broadcast），但数据增强、Dropout 等应该在不同 rank 上有不同的随机行为——否则各 rank 的"数据并行"就失去了多样性。

### 学习率缩放

数据并行的有效 batch size = 单卡 batch size × world_size。当 world_size 增大时，通常需要同比放大学习率：

$$
lr_{\text{effective}} = lr_{\text{base}} \times \text{world\_size}
$$

但这个线性缩放在 world_size 很大时可能导致训练不稳定。实践中常配合**学习率预热**（Warmup）使用：

```python
# ============================================================
# 学习率线性缩放 + Warmup
# ============================================================
base_lr = 1e-4                           # 单卡的基础学习率
effective_lr = base_lr * world_size      # 线性缩放：卡数翻倍，学习率也翻倍
# 为什么线性缩放？
# 每张卡的 local batch 不变，但 AllReduce 后梯度是 N 张卡的平均
# 平均值不随 N 变化，所以当 local batch 不变时，学习率也不需要调整
# 但实际上有效 batch size = local_batch × N 变大了
# 更大的有效 batch 通常可以承受更大的学习率
warmup_steps = 1000                      # 预热步数

for step in range(total_steps):
    if step < warmup_steps:
        # 预热阶段：学习率从 0 线性增长到 effective_lr
        lr = effective_lr * (step / warmup_steps)
    else:
        # 预热结束后保持 effective_lr（后续可加 cosine decay）
        lr = effective_lr
    for param_group in optimizer.param_groups:
        param_group["lr"] = lr
```

> **提示**：线性缩放学习率是经验法则，不是绝对规则。大规模训练中（数百卡以上），可能需要用 $\sqrt{\text{world\_size}}$ 缩放（即 LAMB/LARS 优化器的思路）来保持稳定性。

## 面试回答

### 面试官问："从单卡训练改 DDP，代码需要改哪些地方？"

**口述回答**：

核心改动只有 4 处：

1. **进程初始化**：在代码最前面加 `dist.init_process_group(backend="nccl")`，用 `torchrun` 启动时它自动读环境变量。
2. **模型包装**：`model = DDP(model.to(local_rank), device_ids=[local_rank])` 替代原来的 `model.to(device)`。
3. **数据加载**：`DataLoader` 的 `sampler` 换成 `DistributedSampler(dataset, num_replicas=world_size, rank=rank)`，每个 epoch 要调 `sampler.set_epoch(epoch)`。
4. **日志/保存**：只在 `rank == 0` 时打印和保存，保存模型用 `model.module.state_dict()`（去掉 DDP 加的 `module.` 前缀）。

其他部分（前向/反向/optimizer.step/loss 计算）完全不用改。启动时用 `torchrun --nproc_per_node=N train.py`。

   - c10d rendezvous：所有节点向 master 报到，到齐后开始；弹性训练：节点离开/加入触发全部重启，从 checkpoint 恢复。

## 易混淆点

| 易混淆对 | 区别 | 记忆诀窍 |
|----------|------|----------|
| `model.state_dict()` vs `model.module.state_dict()` | DDP 加了 `module.` 前缀，加载时 key 不匹配 | DDP 保存用 `.module` |
| `DistributedSampler` vs 普通随机切分 | Sampler 保证不重叠，普通随机可能重复 | Sampler=保证每人拿不同的 |
| `shuffle=True` in DataLoader vs Sampler | 用 Sampler 时 DataLoader 不能设 shuffle | sampler 已经管了 shuffle |
| `torch.save` vs `dcp.save` | torch.save 只存一个 rank 的状态，dcp.save 正确处理分片 | FSDP 必须用 dcp |
| FSDP1 vs FSDP2 | FSDP1 module-wrapper，FSDP2 per-parameter + DTensor | FSDP2 = 更细粒度 + 更好组合 |
| NCCL vs Gloo | NCCL 是 GPU 专用，Gloo 跨平台 | GPU 训练只用 NCCL |

## 踩坑记录

| 坑 | 现象 | 原因 | 解决 |
|----|------|------|------|
| 忘记 `sampler` | 每卡读到相同数据，loss 不下降 | DataLoader 没设 sampler | 用 `DistributedSampler` |
| 忘记 `set_epoch` | 每个 epoch 数据分配一样 | sampler 种子不变 | `sampler.set_epoch(epoch)` |
| `find_unused_parameters` | DDP 卡住不动 | 条件分支导致参数没参与前向 | 设 `find_unused_parameters=True` |
| 不调 barrier 下载数据 | 多卡同时下载，网络拥塞 | 每个 rank 都试图下载 | `download=(rank==0)` + `dist.barrier()` |
| FSDP 直接用 `state_dict()` | 加载失败或 key 不匹配 | 只保存了本地分片 | 用 `DCP + get_state_dict/set_state_dict` |
| NCCL 超时 | 训练 hang 住 | 节点间网络不通、防火墙 | `NCCL_DEBUG=INFO` + nccl-tests 测连通 |
