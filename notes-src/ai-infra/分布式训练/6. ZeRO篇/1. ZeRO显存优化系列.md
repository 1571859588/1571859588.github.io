# ZeRO 显存优化系列：逐层消除训练冗余

> **一句话结论**：ZeRO 的核心洞察是——DDP 中每张卡都冗余存储了完整的优化器状态、梯度和参数，把这些冗余切分到 N 张卡上，显存占用就能从每卡 16Ψ 降到最低 Ψ/N × 16Ψ，代价是通信量增加约 50%。ZeRO-1 切优化器状态、ZeRO-2 加切梯度、ZeRO-3 全切，三级递进。

---

## 背景与现象

### 为什么需要 ZeRO？

在学完数据并行（DDP）后，你可能会想：8 张 A100 40G，每卡 40GB 显存，8 卡共 320GB，应该能训练很大的模型了吧？

但现实是——用 DDP 训练一个 7B 模型时，单卡就需要约 112GB 显存，远超 40GB。8 张卡加起来 320GB 足够，但 DDP 要求每卡都存完整副本，所以单卡装不下就没法训。

问题出在哪里？**冗余**。8 张卡每张都存了完全一样的优化器状态、梯度和参数，其中 7/8 的存储是浪费的。

ZeRO（Zero Redundancy Optimizer）就是 DeepSpeed 团队提出的解决方案：**把冗余切掉，每卡只存 1/N**。

### 前置知识：显存账本

在阅读本章前，你需要知道混合精度训练中每张卡存了什么（详见 [3.1 优化器原理与显存开销](../4. 优化器篇/1. 优化器原理与显存开销分析.md)）：

以一个参数量为 Ψ 的模型为例，使用 AdamW + FP16 混合精度训练：

| 组成 | 精度 | 大小 | 占比 |
|------|------|------|------|
| 参数（FP16） | FP16 | 2Ψ Bytes | 12.5% |
| 梯度（FP16） | FP16 | 2Ψ Bytes | 12.5% |
| **优化器状态** | **FP32** | **12Ψ Bytes** | **75%** |
| - FP32 参数副本 | FP32 | 4Ψ Bytes | |
| - 一阶动量（Adam m） | FP32 | 4Ψ Bytes | |
| - 二阶动量（Adam v） | FP32 | 4Ψ Bytes | |
| **合计** | | **16Ψ Bytes** | **100%** |

> **新人提示**：这里 Ψ 是参数数量。比如 7B 模型，Ψ = 7×10⁹，16Ψ = 112GB。优化器状态占 75% 是因为 Adam 需要存 FP32 副本 + 两个动量，都是 4 字节/参数。

## 核心概念：ZeRO 的"切分-聚合"范式

### 生活类比

想象一个 8 人团队，每人桌上都放了一份完整的 100 页项目文档（冗余）。现在要省空间：

- **ZeRO-1**：优化器状态相当于文档的"修订记录"。8 人各存 12.5 页修订记录，需要时互相借阅
- **ZeRO-2**：在 1 的基础上，每人的"批注"也只存 1/8
- **ZeRO-3**：连正文也只存 1/8，谁要看谁的章节就向对应的人借

## 深入讲解

### 1. ZeRO-1：切分优化器状态

#### 1.1 核心思想

优化器状态占 75% 显存但每步只用一次（在 `optimizer.step()` 时）。ZeRO-1 的做法：

- **平时**：每卡只存 1/N 的优化器状态（自己负责的那一片参数的 Adam m、v 和 FP32 副本）
- **参数更新时**：每卡用自己分片的梯度更新自己分片的优化器状态和参数
- **更新后**：通过 AllGather 把更新后的 FP16 参数同步给所有卡

#### 1.2 切分示意图

```
假设 N=4 张卡，模型有 8 个参数（P1~P8）

DDP（每卡存完整优化器状态）：
  GPU 0: [P1 P2 P3 P4 P5 P6 P7 P8] 的 m,v,FP32副本（全部）
  GPU 1: [P1 P2 P3 P4 P5 P6 P7 P8] 的 m,v,FP32副本（全部）← 冗余！
  GPU 2: [P1 P2 P3 P4 P5 P6 P7 P8] 的 m,v,FP32副本（全部）← 冗余！
  GPU 3: [P1 P2 P3 P4 P5 P6 P7 P8] 的 m,v,FP32副本（全部）← 冗余！

ZeRO-1（优化器状态按分片存储）：
  GPU 0: 优化器状态只负责 [P1 P2]
  GPU 1: 优化器状态只负责 [P3 P4]
  GPU 2: 优化器状态只负责 [P5 P6]
  GPU 3: 优化器状态只负责 [P7 P8]

  参数和梯度仍然是每卡完整的（冗余但暂时不切）
```

#### 1.3 执行流程

```
ZeRO-1 单步训练流程：

┌──────────────────────────────────────────────────┐
│ 1. 前向传播：每卡用完整 FP16 参数做前向（同 DDP） │
│                                                   │
│ 2. 反向传播：每卡计算梯度 → AllReduce 梯度        │
│    （每卡得到完整的 FP16 梯度，同 DDP）            │
│                                                   │
│ 3. 参数更新（关键变化）：                         │
│    a. 每卡只更新自己负责的 1/N 参数分片            │
│       GPU 0 只更新 P1,P2 的 Adam 状态和参数        │
│    b. optimizer.step() 只更新分片                  │
│                                                   │
│ 4. 参数同步：AllGather 更新后的参数                │
│    GPU 0 广播 P1,P2 的新值给所有卡                 │
│    GPU 1 广播 P3,P4 的新值给所有卡                 │
│    ...                                            │
│    最终每卡都有完整的更新后参数                    │
└──────────────────────────────────────────────────┘
```

#### 1.4 显存分析

| 项目 | DDP | ZeRO-1 |
|------|-----|--------|
| 参数（FP16） | 2Ψ | 2Ψ |
| 梯度（FP16） | 2Ψ | 2Ψ |
| 优化器状态（FP32） | 12Ψ | 12Ψ/N |
| **合计** | **16Ψ** | **4Ψ + 12Ψ/N** |

以 7B 模型为例，N=8：

- DDP：16 × 7B = 112GB / 卡
- ZeRO-1：4 × 7B + 12 × 7B / 8 = 28 + 10.5 = **38.5GB / 卡**（刚好装进 40G A100！）

#### 1.5 通信量分析

| 步骤 | DDP | ZeRO-1 |
|------|-----|--------|
| 梯度同步 | AllReduce = 2Ψ | AllReduce = 2Ψ |
| 参数同步 | 无（各卡独立更新相同副本） | AllGather = 2Ψ |
| **总通信量** | **2Ψ** | **2Ψ + 2Ψ = 4Ψ**（但 AllGather 可与计算 overlap） |

> **新人提示**：AllReduce = AllGather + ReduceScatter，通信量是 2Ψ。ZeRO-1 的额外 AllGather 传的是更新后的参数（2Ψ），但这一步可以和下一层的计算重叠，实际开销很小。

#### 1.6 DeepSpeed 配置示例

```json
{
  "zero_optimization": {
    "stage": 1,
    "reduce_bucket_size": 5e8
  }
}
```

### 2. ZeRO-2：进一步切分梯度

#### 2.1 核心思想

ZeRO-1 切了优化器状态后，梯度仍然是每卡完整存储的（2Ψ）。但仔细想——梯度在反向传播时是逐层产生的，而且每卡只需要更新自己负责的 1/N 参数分片，那**非自己分片的梯度用完就可以丢弃**。

ZeRO-2 的做法：用 **ReduceScatter** 替代 AllReduce，每卡只保留自己负责分片的聚合梯度。

#### 2.2 通信模式对比

```
DDP / ZeRO-1 的梯度同步（AllReduce）：

  AllReduce = ReduceScatter + AllGather

  GPU 0: 梯度 [g1,g2,g3,g4,g5,g6,g7,g8]
  GPU 1: 梯度 [g1,g2,g3,g4,g5,g6,g7,g8]
  GPU 2: 梯度 [g1,g2,g3,g4,g5,g6,g7,g8]
  GPU 3: 梯度 [g1,g2,g3,g4,g5,g6,g7,g8]

  Step 1: ReduceScatter → 每卡只保留 1/N 的聚合梯度
  GPU 0: [G1,G2]  ← 聚合了所有卡对 P1,P2 的梯度
  GPU 1: [G3,G4]
  GPU 2: [G5,G6]
  GPU 3: [G7,G8]

  Step 2: AllGather → 把聚合结果广播给所有卡（ZeRO-1 需要完整梯度）
  GPU 0: [G1,G2,G3,G4,G5,G6,G7,G8]  ← 完整但冗余
  GPU 1: [G1,G2,G3,G4,G5,G6,G7,G8]  ← 完整但冗余
  ...

  ────────────────────────────────────
  ZeRO-2：只做 ReduceScatter，跳过 AllGather！

  GPU 0: [G1,G2]  ← 只保留自己的分片，其余丢弃
  GPU 1: [G3,G4]
  GPU 2: [G5,G6]
  GPU 3: [G7,G8]
```

> **新人提示**：ReduceScatter 是 AllReduce 的前半部分。AllReduce = ReduceScatter + AllGather。ZeRO-2 聪明地发现"后半部分的 AllGather 是多余的"——因为每卡只需要更新自己分片的参数，不需要完整梯度。

#### 2.3 显存分析

| 项目 | ZeRO-1 | ZeRO-2 |
|------|--------|--------|
| 参数（FP16） | 2Ψ | 2Ψ |
| 梯度（FP16） | 2Ψ | **2Ψ/N** |
| 优化器状态（FP32） | 12Ψ/N | 12Ψ/N |
| **合计** | **4Ψ + 12Ψ/N** | **2Ψ + 14Ψ/N** |

以 7B 模型为例，N=8：

- ZeRO-1：38.5GB
- ZeRO-2：2 × 7B + 14 × 7B / 8 = 14 + 12.25 = **26.25GB / 卡**

#### 2.4 通信量分析

| 步骤 | ZeRO-1 | ZeRO-2 |
|------|--------|--------|
| 梯度同步 | AllReduce = 2Ψ | ReduceScatter = Ψ |
| 参数同步 | AllGather = 2Ψ | AllGather = 2Ψ |
| **总通信量** | **4Ψ** | **3Ψ** |

> ZeRO-2 通信量反而比 ZeRO-1 少！因为省掉了 AllReduce 中冗余的 AllGather 步骤。但注意，这里的"通信量减少"是因为梯度不需要在所有卡上有完整副本。

#### 2.5 反向传播中的逐层 ReduceScatter

```
ZeRO-2 反向传播流程（逐层通信）：

Layer L (最后一层)
  │
  ├─→ 反向计算梯度 g_L
  ├─→ ReduceScatter: 只保留自己分片的梯度 ──┐
  │                                        │
Layer L-1                                │ 梯度已分片
  │                                      │ 存储
  ├─→ 反向计算梯度 g_{L-1}               │
  ├─→ ReduceScatter: 累加到分片 ─────────┘
  │
  ... (逐层向上)

优点：梯度产生后立即分片存储，不需要等全部算完
      → 可以和反向计算 overlap
```

  }

### 3. ZeRO-3：连参数也切分

#### 3.1 核心思想

当模型参数本身（2Ψ）就超过单卡显存时，ZeRO-1 和 ZeRO-2 都无能为力——因为它们仍然每卡存完整参数。ZeRO-3 把参数也切了：

- **平时**：每卡只存 1/N 的参数、梯度、优化器状态
- **前向传播时**：逐层 AllGather 重组当前层的完整参数 → 计算 → 释放
- **反向传播时**：逐层 AllGather 重组参数 → 计算梯度 → ReduceScatter 分片梯度 → 释放参数

这就是 FSDP（Fully Sharded Data Parallel）的核心思想。

#### 3.2 切分示意图

```
ZeRO-3：参数也分片（以 N=4 为例）

  平时存储（每卡只存 1/N）：
  ┌──────────────────────────────────┐
  │ GPU 0: P1,P2 | g1,g2 | opt_1,2  │
  │ GPU 1: P3,P4 | g3,g4 | opt_3,4  │
  │ GPU 2: P5,P6 | g5,g6 | opt_5,6  │
  │ GPU 3: P7,P8 | g7,g8 | opt_7,8  │
  └──────────────────────────────────┘

  前向传播第 3 层时（需要 P3,P4）：
  ┌──────────────────────────────────┐
  │ Step 1: AllGather 重组           │
  │   GPU 0 ← GPU 1: 给我 P3,P4     │
  │   GPU 0 现在有完整的 P3,P4      │
  │                                  │
  │ Step 2: 前向计算                 │
  │   output = input @ P3,P4        │
  │                                  │
  │ Step 3: 释放 P3,P4               │
  │   GPU 0: 不再需要，丢弃          │
  └──────────────────────────────────┘

  反向传播第 3 层时（需要 P3,P4 的梯度）：
  ┌──────────────────────────────────┐
  │ Step 1: AllGather 重组 P3,P4     │
  │                                  │
  │ Step 2: 计算梯度 g3,g4           │
  │                                  │
  │ Step 3: ReduceScatter 梯度        │
  │   GPU 1 保留 G3,G4（自己的分片） │
  │   其余卡丢弃 G3,G4              │
  │                                  │
  │ Step 4: 释放 P3,P4               │
  └──────────────────────────────────┘
```

#### 3.3 显存分析

| 项目 | ZeRO-2 | ZeRO-3 |
|------|--------|--------|
| 参数（FP16） | 2Ψ | **2Ψ/N** |
| 梯度（FP16） | 2Ψ/N | 2Ψ/N |
| 优化器状态（FP32） | 12Ψ/N | 12Ψ/N |
| **合计** | **2Ψ + 14Ψ/N** | **16Ψ/N** |

以 7B 模型为例，N=8：

- ZeRO-2：26.25GB
- ZeRO-3：16 × 7B / 8 = **14GB / 卡**（大模型线性缩放！）

以 70B 模型为例：

- DDP：16 × 70B = 1120GB / 卡（完全不可能）
- ZeRO-3：16 × 70B / 8 = **140GB / 卡**（需要多节点，但单卡只需 1/8）

#### 3.4 通信量分析

ZeRO-3 的通信量是三阶段中最高的：

| 步骤 | 通信操作 | 通信量 | 说明 |
|------|---------|--------|------|
| 前向 | AllGather（每层参数） | Ψ | 逐层重组参数 |
| 反向 | AllGather（每层参数） | Ψ | 反向也需要参数 |
| 反向 | ReduceScatter（梯度） | Ψ | 分片存储梯度 |
| **合计** | | **3Ψ** | 比 DDP（2Ψ）多 50% |

#### 3.5 通信量对比总览

```
通信量对比（每步总量）：

  DDP:     ████████████████████  2Ψ（AllReduce）
  ZeRO-1:  ████████████████████████████████  4Ψ（AllReduce + AllGather）
  ZeRO-2:  ██████████████████████████  3Ψ（ReduceScatter + AllGather）
  ZeRO-3:  ██████████████████████████████  3Ψ（2×AllGather + ReduceScatter）

  注意：ZeRO-1 通信量看似最高，但它的 AllReduce 和 AllGather
  可以与计算重叠，实际通信开销不一定比 ZeRO-2 高多少。
  ZeRO-3 的通信量与 ZeRO-2 相同（3Ψ），但通信模式不同
  （需要逐层 AllGather，对延迟更敏感）。
```

#### 3.6 逐层 AllGather 的 overlap

ZeRO-3 的关键优化是**逐层通信与计算重叠**：

```
不重叠（慢）：
  AllGather L1 → 前向 L1 → AllGather L2 → 前向 L2 → ...

重叠（快）：
  AllGather L1 → 前向 L1 ──────────────────→
                 AllGather L2 → 前向 L2 ───→
                                AllGather L3 → 前向 L3 →
  ↑ 当前层计算时，同时预取下一层的参数
```

```python
# ✅ 无需 sudo —— FSDP 中逐层 AllGather 的简化伪代码
class ZeRO3Layer:
    def forward(self, x):
        # 1. AllGather 重组当前层完整参数
        full_weight = all_gather(self.weight_shard)
        # 2. 计算
        output = x @ full_weight
        # 3. 释放完整参数（只保留分片）
        del full_weight
        return output

    def backward(self, grad_output):
        # 1. AllGather 重组参数
        full_weight = all_gather(self.weight_shard)
        # 2. 计算梯度
        grad_input = grad_output @ full_weight.T
        grad_weight = grad_output.T @ self.input
        # 3. ReduceScatter 梯度（分片存储）
        self.grad_shard = reduce_scatter(grad_weight)
        # 4. 释放
        del full_weight
        return grad_input
```

#### 3.7 PyTorch FSDP 使用示例

```python
# ✅ 无需 sudo
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP
from torch.distributed.fsdp import ShardingStrategy

# FSDP 的三种分片策略：
# - FULL_SHARD:     等价于 ZeRO-3（参数+梯度+优化器全切）
# - SHARD_GRAD_OP:  等价于 ZeRO-2（只切梯度+优化器）
# - NO_SHARD:       等价于 DDP（不切）

model = FSDP(
    model,
    sharding_strategy=ShardingStrategy.FULL_SHARD,  # ZeRO-3
    device_id=torch.cuda.current_device(),
    use_orig_params=True,  # 保留原始参数名，方便检查
)

# 训练循环与普通 DDP 完全一致
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)
for batch in dataloader:
    loss = model(batch)
    loss.backward()
    optimizer.step()
    optimizer.zero_grad()
```

### 4. ZeRO-Offload 与 ZeRO-Infinity

#### 4.1 ZeRO-Offload：CPU 卸载

当 GPU 显存不够但你有大内存 CPU 服务器时，可以把优化器状态放到 CPU：

```
ZeRO-Offload 架构：

  ┌─────────── GPU（计算密集）──────────┐
  │  FP16 参数    FP16 梯度              │
  │  前向传播     反向传播               │
  └──────────┬──────────────────────────┘
             │ PCIe 4.0 (~32 GB/s)
  ┌──────────▼──────────────────────────┐
  │           CPU（大内存）              │
  │  FP32 参数副本   FP32 一阶动量      │
  │  FP32 二阶动量   FP32 梯度          │
  │  参数更新（AdamW step）              │
  └─────────────────────────────────────┘

  数据流：
  1. GPU 反向传播 → 产出 FP16 梯度
  2. 梯度 ReduceScatter 到 GPU 分片 → 转 FP32 → 传到 CPU（PCIe）
  3. CPU 用 FP32 梯度更新优化器状态和参数
  4. 更新后的 FP32 参数 → 转 FP16 → 传回 GPU → AllGather 同步
```

**适用场景**：1-4 卡训练大模型（如单卡 A100 训练 10B 模型）

**瓶颈**：CPU-GPU 传输带宽。PCIe 4.0 理论 32GB/s，实际约 20-25GB/s，远低于 NVLink 的 300-600GB/s。

```json
{
  "zero_optimization": {
    "stage": 2,
    "offload_optimizer": {
      "device": "cpu",
      "pin_memory": true
    }
  },
  "zero_stage": 2
}
```

#### 4.2 ZeRO-Infinity：NVMe 卸载

当 CPU 内存也不够时（万亿参数），进一步利用 NVMe SSD：

```
ZeRO-Infinity 三级存储：

  ┌─────────────── GPU HBM ────────────────┐
  │ 当前层的参数/梯度（少量，计算中需要）   │
  │ 带宽：~2 TB/s（A100 HBM）              │
  └──────────┬─────────────────────────────┘
             │ NVLink (~600 GB/s)
  ┌──────────▼─────────────────────────────┐
  │            CPU DRAM                     │
  │ 优化器状态（FP32）+ 近期要用的参数分片  │
  │ 带宽：~100 GB/s                         │
  └──────────┬─────────────────────────────┘
             │ PCIe 4.0 (~32 GB/s)
  ┌──────────▼─────────────────────────────┐
  │            NVMe SSD                     │
  │ 完整模型参数/优化器状态的持久化存储     │
  │ 带宽：~7 GB/s（PCIe 4.0 NVMe）          │
  └─────────────────────────────────────────┘

  关键技术：分块预取（prefetch）
  - 当前层计算时，后台异步预取下一层的参数到 GPU
  - 计算与 I/O 重叠，掩盖 SSD 的高延迟
```

```json
{
  "zero_optimization": {
    "stage": 3,
    "offload_optimizer": {
      "device": "nvme",
      "nvme_path": "/local/nvme"
    },
    "offload_param": {
      "device": "nvme",
      "nvme_path": "/local/nvme"
    }
  }
}
```

#### 4.3 存储层级对比

| 层级 | 设备 | 带宽 | 容量 | 延迟 |
|------|------|------|------|------|
| L1 | GPU HBM | ~2 TB/s | 40-80 GB | ~ns |
| L2 | CPU DRAM | ~100 GB/s | 256-2048 GB | ~100ns |
| L3 | NVMe SSD | ~7 GB/s | 4-16 TB | ~10μs |

### 6. ZeRO 与 FSDP 的关系

#### 6.1 对照表

| 维度 | ZeRO（DeepSpeed） | FSDP（PyTorch 原生） |
|------|-------------------|---------------------|
| 来源 | 微软 DeepSpeed 论文+实现 | PyTorch 1.11+ 原生 |
| ZeRO-1 等价 | `stage=1` | `SHARD_GRAD_OP`（近似） |
| ZeRO-2 等价 | `stage=2` | `SHARD_GRAD_OP` |
| ZeRO-3 等价 | `stage=3` | `FULL_SHARD` |
| CPU Offload | ✅ `offload_optimizer` | ✅ `CPUOffloadPolicy` |
| NVMe Offload | ✅ ZeRO-Infinity | ❌ |
| 混合精度 | 内置 | 需配合 AMP |
| 配置方式 | JSON 配置文件 | Python API |
| 生态集成 | HuggingFace `accelerate` | PyTorch 原生 |
| 调试难度 | 中（配置文件+日志） | 低（Python API） |

#### 6.2 怎么选？

```
选 FSDP 还是 DeepSpeed？

  ┌─ PyTorch 生态内 → FSDP
  │  - 简单直接，torch.distributed.fsdp
  │  - 调试方便，Python 原生
  │  - 适合从 DDP 迁移
  │
  ├─ 需要 NVMe Offload → DeepSpeed ZeRO-Infinity
  │  - FSDP 不支持 NVMe
  │
  ├─ 已有 DeepSpeed 配置 → 继续 DeepSpeed
  │  - 迁移成本高
  │
  └─ HuggingFace accelerate → 两者都支持
     - 可以通过 accelerate 配置切换
```

## 面试回答

### 版本一：1 分钟简答

> ZeRO 是 DeepSpeed 提出的显存优化方法，核心思想是消除数据并行中的冗余存储。它分三个阶段递进：ZeRO-1 切分优化器状态（占 75% 显存），ZeRO-2 进一步切分梯度，ZeRO-3 把参数也切了。每张卡只存 1/N 的训练状态，显存从 16Ψ 降到 16Ψ/N。通信代价从 DDP 的 2Ψ 增加到 ZeRO-3 的 3Ψ，多了 50%，但可以通过通信与计算重叠来掩盖。FSDP 的 `FULL_SHARD` 就是 PyTorch 原生实现的 ZeRO-3。

### 版本二：3 分钟详解

> 先说问题：DDP 中每张卡都存了完整的模型参数、梯度和优化器状态，N 张卡就有 (N-1)/N 的冗余。以 7B 模型为例，AdamW + FP16 训练每卡需要 16Ψ = 112GB，远超 40GB 显存。
>
> ZeRO 的解法是"切分-聚合"——平时每卡只存 1/N，需要时通过通信获取完整数据，用完即弃。
>
> **ZeRO-1** 切优化器状态（12Ψ → 12Ψ/N），因为优化器状态占 75% 显存，切它性价比最高。每卡显存从 16Ψ 降到 4Ψ + 12Ψ/N。通信量比 DDP 多一个 AllGather 参数（2Ψ），但可以 overlap。
>
> **ZeRO-2** 在 1 的基础上切梯度，用 ReduceScatter 替代 AllReduce，每卡只保留自己分片的梯度。显存降到 2Ψ + 14Ψ/N，通信量反而减少到 3Ψ（省了 AllReduce 中的 AllGather 部分）。
>
> **ZeRO-3** 把参数也切了，前向和反向时逐层 AllGather 重组参数。显存降到 16Ψ/N，实现线性缩放。通信量 3Ψ（前向 AllGather Ψ + 反向 AllGather Ψ + ReduceScatter Ψ），比 DDP 多 50%。
>
> 选型上：参数单卡装得下用 ZeRO-2，装不下用 ZeRO-3/FSDP。少卡大模型用 ZeRO-Offload 把优化器状态卸载到 CPU。

## 深入追问

### Q1: ZeRO-3 的通信量是 3Ψ，比 DDP 的 2Ψ 多 50%，为什么实际训练速度降幅不大？

**A**：关键在于**通信与计算重叠**。ZeRO-3 采用逐层 AllGather——当前层计算时，后台异步预取下一层的参数。只要通信时间 < 计算时间，通信就被完全掩盖。实际中 NVLink 带宽很高（600GB/s），大部分场景能实现良好重叠。只有在通信带宽不足（如跨机以太网）时才会明显变慢。

### Q2: ZeRO-2 和 ZeRO-3 通信量都是 3Ψ，为什么 ZeRO-3 更慢？

**A**：虽然总量相同，但**通信模式不同**：
- ZeRO-2：参数完整存在每卡，前向不需要通信，只在反向梯度同步和参数更新后需要通信
- ZeRO-3：每层前向和反向都需要 AllGather，通信频率高，对延迟敏感

ZeRO-3 的通信是"小批量多次"（逐层 AllGather），ZeRO-2 是"大批量少次"。前者对延迟更敏感。

### Q3: ZeRO-Offload 的 CPU 计算会不会成为瓶颈？

**A**：会，但可以缓解。ZeRO-Offload 的策略是：
1. CPU 只做参数更新（AdamW step），不做前向/反向
2. 参数更新用多线程并行 + SIMD 指令优化
3. 梯度从 GPU→CPU 传输可以和反向计算 overlap
4. 参数从 CPU→GPU 传输可以和下一层前向 overlap

但如果模型特别大，CPU 参数更新的计算量（12Ψ 浮点运算）会成为瓶颈。此时可考虑 ZeRO-Infinity 的 NVMe + 预取。

### Q4: ZeRO-3 和 FSDP 的 FULL_SHARD 有什么区别？

**A**：算法层面完全等价，但实现细节有差异：
- **参数分片粒度**：DeepSpeed 按 parameter group 分片，FSDP 按 Module（`FlatParameter`）分片
- **通信优化**：FSDP 支持 `forward_prefetch` 和 `backward_prefetch`，DeepSpeed 有 `contiguous_gradients`
- **Offload 支持**：DeepSpeed 支持 NVMe，FSDP 只支持 CPU
- **调试**：FSDP 的错误信息更 Pythonic，DeepSpeed 的 JSON 配置容易出错但更声明式

### Q5: ZeRO-3 能完全替代 TP（张量并行）吗？

**A**：不能完全替代，各有优势：
- **ZeRO-3**：适合"参数显存"问题，通信在层间（可以 overlap），适合跨机
- **TP**：适合"激活值显存"问题（如 FFN 中间维度很大），通信在层内（延迟敏感），适合单机 NVLink

千亿模型通常 ZeRO-3 + TP 组合使用：TP 在机内（8卡 NVLink），ZeRO-3 在机间。

## 易混淆点

### 1. ZeRO-1 的 AllGather vs DDP 的 AllReduce

| 维度 | DDP AllReduce | ZeRO-1 AllGather |
|------|--------------|------------------|
| 时机 | 反向传播后（梯度同步） | 参数更新后（参数同步） |
| 数据 | 梯度 | 更新后的参数 |
| 通信量 | 2Ψ | 2Ψ |
| 可否 overlap | 可以（Bucket 分桶） | 可以（与下一层计算） |

### 2. ZeRO-3 的 AllGather 与 ZeRO-1 的 AllGather

- **ZeRO-1 的 AllGather**：在 `optimizer.step()` 之后，广播更新后的参数，**一次性的大批量通信**
- **ZeRO-3 的 AllGather**：在前向/反向传播中，逐层重组参数，**多次的小批量通信**

### 3. FSDP 的三种 ShardingStrategy

| 策略 | 等价 ZeRO | 参数分片 | 梯度分片 | 优化器分片 |
|------|----------|---------|---------|-----------|
| `NO_SHARD` | DDP | ❌ | ❌ | ❌ |
| `SHARD_GRAD_OP` | ZeRO-2 | ❌ | ✅ | ✅ |
| `FULL_SHARD` | ZeRO-3 | ✅ | ✅ | ✅ |

### 4. 显存节省 vs 通信量增加

```
  显存节省                通信量
  ZeRO-1 ──→ ZeRO-2 ──→ ZeRO-3    ZeRO-1 ──→ ZeRO-2 ──→ ZeRO-3
  多 ←───────────────────→ 少     少 ←───────────────────→ 多

  注意：ZeRO-2 的通信量（3Ψ）反而比 ZeRO-1（4Ψ）少
       因为 ZeRO-2 省掉了 AllReduce 中的冗余 AllGather
       但 ZeRO-2 的显存节省不如 ZeRO-3
```
