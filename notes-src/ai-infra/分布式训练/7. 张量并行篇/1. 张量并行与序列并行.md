# 张量并行与序列并行：层内矩阵切分

> **一句话结论**：张量并行（TP）把单个矩阵乘法 $Y=XA$ 按列或按行切分到多卡并行计算，是唯一能解决"单层前向就超显存"的并行策略。Column Parallel 切权重列（每卡算不同输出列），Row Parallel 切权重行（每卡算部分和再 AllReduce）。序列并行（SP）进一步切分 LayerNorm/Dropout 的激活冗余，把 AllReduce 拆成 ReduceScatter+AllGather，不增加通信量但省激活显存。TP 对延迟极敏感，通常只在单机 NVLink 内使用（8 卡以内）。

---

## 背景与现象

### 为什么需要张量并行？

在学完 ZeRO/FSDP 后，你可能会想：既然 ZeRO-3 能把参数切到 N 张卡，那千亿模型不也能训了吗？

但实际中会遇到两类问题：

**问题 1：单层激活值超显存**

Transformer 的 FFN 层有一个中间维度 `intermediate_size`，通常是 hidden_size 的 4 倍。以 GPT-3 175B 为例：

```
hidden_size = 12288
intermediate_size = 49152  (4 × 12288)
batch_size = 8, seq_len = 2048

FFN 第一层激活值 = batch × seq_len × intermediate_size × 2 bytes (FP16)
                = 8 × 2048 × 49152 × 2
                ≈ 1.6 GB  （仅一层的一次前向）
```

ZeRO-3 只切参数，不切激活值。如果单层激活值就几 GB，几十层下来就爆了。

**问题 2：ZeRO-3 的通信在跨机时太慢**

ZeRO-3 需要逐层 AllGather 参数，如果跨机（以太网/IB），通信延迟高，重叠困难。但 TP 把切分放在层内，每层通信一次，频率和 PP 类似。

### TP 的定位

```
┌─────────────────────────────────────────────────────┐
│              并行策略的切分维度                      │
│                                                     │
│  ZeRO/DP:  切"训练状态"（参数、梯度、优化器）        │
│  TP:       切"层内计算"（权重矩阵的行/列）           │
│  PP:       切"层"（不同层在不同卡）                  │
│  SP/CP:    切"序列维度"（序列长度方向）              │
│                                                     │
│  关键区别：                                          │
│  ZeRO 不切激活 → 激活显存仍可能超限                  │
│  TP 切权重+激活 → 每卡只存 1/N 的权重和激活          │
└─────────────────────────────────────────────────────┘
```

### 带宽约束：为什么 TP 只在单机内用

TP 每层都需要通信（AllReduce），通信频率远高于 ZeRO（逐层 vs 逐步）：

| 策略 | 通信频率 | 通信操作 | 推荐互联 |
|------|---------|---------|---------|
| DDP/ZeRO | 每步一次 | AllReduce | 跨机 IB 可接受 |
| TP | 每层两次 | AllReduce | 必须 NVLink（机内） |
| PP | 每 Stage 边界 | P2P send/recv | 跨机 IB 可接受 |

> **新人提示**：NVLink 带宽 300-600 GB/s，远高于 InfiniBand 的 25-50 GB/s。TP 的每层通信需要低延迟高带宽，所以只在单机 8 卡内使用。跨机用 ZeRO 或 PP。

参考：[1.1 单机卡间通信-NVLink与NVSwitch](../集合通信基础/2. 硬件篇/1. 单机卡间通信-NVLink与NVSwitch.md)

## 核心概念：矩阵乘法怎么切

### 生活类比

想象一个矩阵乘法 $Y = XA$，其中 $X$ 是输入（学生成绩表），$A$ 是权重（评分标准表），$Y$ 是输出（最终评分）。

```
完整计算（单卡）：
  X (m×k)  ×  A (k×n)  =  Y (m×n)
  成绩表       评分标准      最终评分

  一个老师算所有的，太累了
```

怎么分给多个老师？

**方式 1：按列切评分标准**（Column Parallel）
- 每个老师只看一部分评分维度（比如老师 1 看语文、老师 2 看数学）
- 每个老师需要完整的成绩表
- 每个老师算出自己那部分评分

**方式 2：按行切评分标准**（Row Parallel）
- 评分标准按行切，每个老师只看一部分评分项
- 成绩表也按列切，每个老师只拿到对应的成绩分片
- 每个老师算出部分分数，最后加起来

## 深入讲解

### 1. Column Parallel Linear（按列切分）

#### 1.1 数学推导

给定 $Y = XA$，其中 $X \in \mathbb{R}^{m \times k}$，$A \in \mathbb{R}^{k \times n}$，$Y \in \mathbb{R}^{m \times n}$。

把 $A$ 按列切分：$A = [A_1, A_2, \ldots, A_N]$，其中 $A_i \in \mathbb{R}^{k \times n/N}$。

则：
$$Y = XA = X[A_1, A_2, \ldots, A_N] = [XA_1, XA_2, \ldots, XA_N]$$

每张卡计算 $Y_i = X \cdot A_i$，得到 $Y$ 的不同列分片。

#### 1.2 切分示意图

```
Column Parallel Linear (N=2)

完整计算：
         A (k×n)
         ┌──────────────────┐
         │   A_1    │  A_2   │    按列切分
         │ (k×n/2)  │(k×n/2)│
         └──────────────────┘
X (m×k)  ↑    ↑
 ┌─────┐ │    │
 │     │ │    │     Y (m×n)
 │  X  │ ×    │  =  ┌──────────────────┐
 │     │ │    │     │   Y_1    │  Y_2   │
 │     │ │    │     │ (m×n/2) │(m×n/2)│
 └─────┘ │    │     └──────────────────┘
         │    │           ↑       ↑
       GPU 0  GPU 1      GPU 0    GPU 1

  X 完整复制到每张卡        Y 分片在每张卡上

  GPU 0: Y_1 = X @ A_1   (得到 Y 的前 n/2 列)
  GPU 1: Y_2 = X @ A_2   (得到 Y 的后 n/2 列)
```

#### 1.3 通信需求

- **输入 $X$**：每卡都需要完整的 $X$ → 通过 **Broadcast** 复制（或各卡已有完整输入）
- **输出 $Y$**：各卡得到列分片 $Y_i$，**不需要通信**（后续可直接输入到 Row Parallel）

#### 1.4 应用位置

- FFN 的第一个线性层（`hidden_size → intermediate_size`）：切中间维度
- Attention 的 QKV 投影（`hidden_size → 3 × hidden_size`）：切多头维度

---

### 2. Row Parallel Linear（按行切分）

#### 2.1 数学推导

给定 $Y = XA$，把 $A$ 按行切分：$A = \begin{bmatrix} A_1 \\ A_2 \\ \vdots \\ A_N \end{bmatrix}$，其中 $A_i \in \mathbb{R}^{k/N \times n}$。

输入 $X$ 也按列切分：$X = [X_1, X_2, \ldots, X_N]$，其中 $X_i \in \mathbb{R}^{m \times k/N}$。

则：
$$Y = XA = [X_1, X_2, \ldots, X_N] \begin{bmatrix} A_1 \\ A_2 \\ \vdots \\ A_N \end{bmatrix} = X_1 A_1 + X_2 A_2 + \cdots + X_N A_N = \sum_{i=1}^{N} X_i A_i$$

每张卡计算 $Y_i = X_i \cdot A_i$（部分结果），再通过 **AllReduce** 求和得到完整 $Y$。

#### 2.2 切分示意图

```
Row Parallel Linear (N=2)

完整计算：
  X (m×k)         A (k×n)
  ┌──────┬──────┐  ┌──────┐
  │  X_1  │  X_2 │  │  A_1 │    按行切分 A
  │(m×k/2)│(m×k/2)│  │(k/2×n)│
  └──────┴──────┘  ├──────┤
       ↑    ↑      │  A_2 │
     GPU 0 GPU 1   │(k/2×n)│
                    └──────┘
                    ↑    ↑
                  GPU 0  GPU 1

  GPU 0: Y_0 = X_1 @ A_1   (部分结果, m×n)
  GPU 1: Y_1 = X_2 @ A_2   (部分结果, m×n)

  Y = Y_0 + Y_1  ← AllReduce (求和)

         ┌─────────────┐
         │   Y (m×n)   │  完整结果（每卡相同）
         └─────────────┘
           ↑         ↑
         GPU 0     GPU 1
         (相同)    (相同)
```

#### 2.3 通信需求

- **输入 $X$**：按列分片，每卡只需 $X_i$（与 Column Parallel 的输出对接）
- **输出 $Y$**：各卡的部分结果 $Y_i$ 需要 **AllReduce**（求和）得到完整 $Y$

#### 2.4 应用位置

- FFN 的第二个线性层（`intermediate_size → hidden_size`）：合并中间维度
- Attention 的输出投影（`hidden_size → hidden_size`）：合并多头输出

---

### 3. Column + Row 的黄金组合

Megatron-LM 的关键洞察：**Column Parallel 后接 Row Parallel，中间不需要通信**。

```
FFN 的两层线性层组合 (N=2)

  输入 X (完整)
     │
     ├──────────────────────────────────┐
     │ Column Parallel (切中间维度)       │
     │ GPU 0: Y1 = X @ A1              │ GPU 1: Y2 = X @ A2
     │ Y1 (m×n/2)                      │ Y2 (m×n/2)
     ├──────────────────────────────────┤
     │ 激活函数 (GELU/SiLU)              │
     │ GPU 0: Y1 = σ(Y1)               │ GPU 1: Y2 = σ(Y2)
     ├──────────────────────────────────┤
     │ Row Parallel (合并中间维度)       │
     │ GPU 0: Z1 = Y1 @ B1             │ GPU 1: Z2 = Y2 @ B2
     │ Z1 (m×p) 部分结果                │ Z2 (m×p) 部分结果
     ├──────────────────────────────────┤
     │ AllReduce                        │
     │ Z = Z1 + Z2                      │
     └──────────────────────────────────┘
     │
     输出 Z (完整)

  通信次数：1 次 AllReduce（在 Row Parallel 之后）
  中间的 Column → Row 传递是分片到分片，无需通信！
```

> **新人提示**：这就是 Megatron-LM 的核心设计——Column Parallel 的输出恰好是 Row Parallel 需要的分片输入，两者"拼积木"一样接上，中间不需要任何通信。只有 Row Parallel 输出时才需要一次 AllReduce。

#### 数学等价性验证

$$Y = \text{GeLU}(X \cdot A) \cdot B$$

切分后：

$$Y_i = \text{GeLU}(X \cdot A_i) \cdot B_i$$

$$Y = \sum_{i=1}^{N} Y_i = \sum_{i=1}^{N} \text{GeLU}(X \cdot A_i) \cdot B_i$$

因为 GeLU 是逐元素的（element-wise），所以 $\text{GeLU}(X \cdot A_i)$ 只作用于 $XA$ 的第 $i$ 列分片，与整体 $\text{GeLU}(XA)$ 的对应分片完全一致。因此上述分解是精确等价的。

### 4. Transformer Block 的 TP 切分方案

#### 4.1 完整切分图

```
Transformer Block 在 TP=2 下的切分：

  输入 h (b×s×hidden)     ← 每卡完整（来自上一个 AllReduce 或输入）
  │
  ├─ LayerNorm            ← 每卡完整（非 TP 区域，有冗余）
  │  h_norm (b×s×hidden)
  │
  ├─ QKV 投影 (Column Parallel)
  │  GPU 0: Q1,K1,V1 = h_norm @ Wqkv_1   (b×s×hidden/2 各)
  │  GPU 1: Q2,K2,V2 = h_norm @ Wqkv_2   (b×s×hidden/2 各)
  │  ← 每卡负责 hidden/N 个 head
  │
  ├─ Attention 计算（每卡独立）
  │  GPU 0: attn_out_1 = Attn(Q1,K1,V1)  (b×s×hidden/2)
  │  GPU 1: attn_out_2 = Attn(Q2,K2,V2)  (b×s×hidden/2)
  │  ← 多头天然可切分！每个 head 独立计算
  │
  ├─ Output 投影 (Row Parallel)
  │  GPU 0: out_1 = attn_out_1 @ Wo_1     (b×s×hidden, 部分结果)
  │  GPU 1: out_2 = attn_out_2 @ Wo_2     (b×s×hidden, 部分结果)
  │
  ├─ AllReduce  ← 第 1 次通信
  │  out = out_1 + out_2  (b×s×hidden, 每卡完整)
  │
  ├─ Residual + Dropout
  │  h = h + out                          ← 每卡完整
  │
  ├─ LayerNorm            ← 每卡完整（非 TP 区域）
  │  h_norm (b×s×hidden)
  │
  ├─ FFN 第 1 层 (Column Parallel)
  │  GPU 0: h1_1 = h_norm @ W1_1  (b×s×intermediate/2)
  │  GPU 1: h1_2 = h_norm @ W1_2  (b×s×intermediate/2)
  │  ← 切中间维度
  │
  ├─ 激活函数 (GELU/SiLU)
  │  GPU 0: h1_1 = σ(h1_1)
  │  GPU 1: h1_2 = σ(h1_2)
  │
  ├─ FFN 第 2 层 (Row Parallel)
  │  GPU 0: out_1 = h1_1 @ W2_1   (b×s×hidden, 部分结果)
  │  GPU 1: out_2 = h1_2 @ W2_2   (b×s×hidden, 部分结果)
  │
  ├─ AllReduce  ← 第 2 次通信
  │  out = out_1 + out_2  (b×s×hidden)
  │
  └─ Residual + Dropout
     h = h + out                          ← 每卡完整

  通信次数：每层 2 次 AllReduce（Attention + FFN）
```

#### 4.2 通信量分析

每个 Transformer Block 前向需要 2 次 AllReduce，每次传输的数据量是 $b \times s \times h$（batch × seq_len × hidden_size）：

$$\text{前向通信量} = 2 \times 2bsh = 4bsh \text{ bytes}$$

（AllReduce 通信量 = 2 × 数据量，因为是 ReduceScatter + AllGather）

反向传播的 AllReduce 与前向对称：

$$\text{反向通信量} = 4bsh \text{ bytes}$$

$$\text{总通信量（每层每步）} = 8bsh \text{ bytes}$$

> **对比 DDP**：DDP 每步通信量是 $2\Psi$（所有参数梯度）。TP 每层通信量是 $8bsh$，两者量级不同——TP 传的是激活值（小），DDP 传的是梯度（大）。

#### 4.3 Megatron-LM 代码示例

```python
# ✅ 无需 sudo —— Megatron-LM 风格的 TP 实现（简化版）
import torch
import torch.nn as nn
import torch.distributed as dist

class ColumnParallelLinear(nn.Module):
    """按列切分权重的线性层"""
    def __init__(self, in_features, out_features, tp_size):
        super().__init__()
        self.tp_size = tp_size
        self.tp_rank = dist.get_rank()
        assert out_features % tp_size == 0
        self.out_features_per_partition = out_features // tp_size
        # 每卡只存 1/N 的权重
        self.weight = nn.Parameter(torch.empty(
            self.out_features_per_partition, in_features
        ))
        nn.init.normal_(self.weight, std=0.02)

    def forward(self, x):
        # x: (b, s, in_features) —— 每卡完整输入
        # output: (b, s, out_features/N) —— 每卡得到分片输出
        return x @ self.weight.T  # 不需要通信！

class RowParallelLinear(nn.Module):
    """按行切分权重的线性层"""
    def __init__(self, in_features, out_features, tp_size):
        super().__init__()
        self.tp_size = tp_size
        assert in_features % tp_size == 0
        self.in_features_per_partition = in_features // tp_size
        # 每卡只存 1/N 的权重（按行切分）
        self.weight = nn.Parameter(torch.empty(
            out_features, self.in_features_per_partition
        ))
        nn.init.normal_(self.weight, std=0.02)

    def forward(self, x):
        # x: (b, s, in_features/N) —— 每卡分片输入（来自 Column Parallel）
        # output: (b, s, out_features) —— 部分结果
        output = x @ self.weight.T
        # AllReduce 求和
        dist.all_reduce(output, op=dist.ReduceOp.SUM)
        return output


class TPEncoderLayer(nn.Module):
    """Megatron-LM 风格的 TP Transformer Block"""
    def __init__(self, hidden_size, num_heads, intermediate_size, tp_size):
        super().__init__()
        self.tp_size = tp_size
        num_heads_per_partition = num_heads // tp_size
        hidden_per_partition = hidden_size // tp_size
        inter_per_partition = intermediate_size // tp_size

        # Attention: QKV 用 Column Parallel, Output 用 Row Parallel
        self.qkv = ColumnParallelLinear(hidden_size, 3 * hidden_size, tp_size)
        self.attn_out = RowParallelLinear(hidden_size, hidden_size, tp_size)

        # FFN: 第1层 Column Parallel, 第2层 Row Parallel
        self.ffn_in = ColumnParallelLinear(hidden_size, intermediate_size, tp_size)
        self.ffn_out = RowParallelLinear(intermediate_size, hidden_size, tp_size)

        self.norm1 = nn.LayerNorm(hidden_size)  # 非 TP 区域，每卡完整
        self.norm2 = nn.LayerNorm(hidden_size)

    def forward(self, x):
        # --- Attention ---
        h = self.norm1(x)
        qkv = self.qkv(h)          # (b, s, 3*hidden/N) per GPU
        # ... attention 计算 ...
        attn_out = self.attn_out(attn_result)  # AllReduce 在这里
        x = x + attn_out

        # --- FFN ---
        h = self.norm2(x)
        h = self.ffn_in(h)         # (b, s, intermediate/N) per GPU
        h = torch.nn.functional.gelu(h)
        ffn_out = self.ffn_out(h)  # AllReduce 在这里
        x = x + ffn_out
        return x
```

### 5. 序列并行（Sequence Parallelism, SP）

#### 5.1 问题：非 TP 区域的激活冗余

在标准 TP 中，LayerNorm 和 Dropout 不参与 TP 切分，它们的输入/输出在每卡上是完整的：

```
标准 TP 的激活值分布（TP=2）：

  ┌─────────────────────────────────────┐
  │ LayerNorm: h (b×s×hidden)          │  ← 每卡完整（冗余！）
  │ QKV/Attn/Out: h (b×s×hidden/N)    │  ← 分片
  │ Residual: h (b×s×hidden)           │  ← 每卡完整
  │ LayerNorm: h (b×s×hidden)          │  ← 每卡完整（冗余！）
  │ FFN: h (b×s×intermediate/N)       │  ← 分片
  │ Residual: h (b×s×hidden)           │  ← 每卡完整
  └─────────────────────────────────────┘

  问题：非 TP 区域（LN + Dropout + Residual）的激活值在 N 卡上冗余存储
        层数越多，冗余越大
```

对于大模型（如 175B，96 层），非 TP 区域的激活值冗余可达数十 GB。

#### 5.2 解决方案：沿序列维度切分

SP 的核心思想：在非 TP 区域，把激活值沿序列维度（seq_len）切分，每卡只存 $s/N$。

```
序列并行的激活值分布（TP=2, SP=2）：

  ┌─────────────────────────────────────────────┐
  │ LayerNorm: h (b×s/N×hidden)                │  ← 分片！不冗余
  │                                              │
  │     AllGather (序列维度) → 恢复完整          │  ← 进入 TP 区域前
  │                                              │
  │ QKV/Attn/Out: h (b×s×hidden/N)             │  ← 分片（TP 切分）
  │                                              │
  │     ReduceScatter (序列维度) → 切分           │  ← 离开 TP 区域时
  │                                              │
  │ LayerNorm: h (b×s/N×hidden)                │  ← 分片！不冗余
  │     AllGather → TP 区域                     │
  │ FFN: h (b×s×intermediate/N)               │  ← 分片
  │     ReduceScatter → 非 TP 区域              │
  │ Residual: h (b×s/N×hidden)                │  ← 分片
  └─────────────────────────────────────────────┘
```

#### 5.3 通信变化：AllReduce → ReduceScatter + AllGather

```
标准 TP 的通信：
  TP 区域出口: AllReduce(h)  → 完整结果到每卡

  AllReduce = ReduceScatter + AllGather
  通信量 = 2 × bsh  （传输量 = bsh 的两倍）

序列并行的通信：
  TP 区域出口: ReduceScatter(h) → 每卡得到 b×s/N×hidden 分片
  TP 区域入口: AllGather(h)    → 恢复完整 b×s×hidden

  总通信量 = ReduceScatter(bsh) + AllGather(bsh) = 2 × bsh

  结论：通信量完全相同！AllReduce 本来就等于 RS + AG
       但 SP 把激活值从完整 bsh 变成了 bsh/N → 省了显存
```

> **新人提示**：AllReduce 的底层实现就是 ReduceScatter + AllGather。SP 没有增加通信量，只是把两步拆开，中间插入非 TP 计算而已。这是"免费的午餐"——同样的通信量，省了激活显存。

#### 5.4 SP 的显存收益

| 组件 | 标准 TP | 序列并行 SP | 节省 |
|------|---------|------------|------|
| LayerNorm 输入/输出 | bsh | **bsh/N** | (N-1)/N |
| Dropout 输入/输出 | bsh | **bsh/N** | (N-1)/N |
| Residual | bsh | **bsh/N** | (N-1)/N |
| Attention 激活 | bsh/N | bsh/N | 0 |
| FFN 激活 | bsh/N | bsh/N | 0 |

以 GPT-3 175B 为例（96 层，hidden=12288，batch=8，seq=2048，FP16）：

- 非 TP 激活（每层）: 2 × 8 × 2048 × 12288 × 2 ≈ 0.8 GB
- 96 层 × 2（Attention + FFN 两个 LN）≈ 153 GB（TP=1 时）
- TP=8, SP 省掉 7/8 → 约 134 GB 节省

### 6. GQA/MQA 下的 TP 切分

#### 6.1 问题：KV Head 不足

标准 MHA（Multi-Head Attention）中，Q/K/V 的 head 数相同，TP 天然按 head 切分。

但 GQA（Grouped-Query Attention）中，KV Head 数量 < Q Head 数量：

```
MHA (8 heads):
  Q: 8 heads → TP=8, 每卡 1 head ✅
  K: 8 heads → TP=8, 每卡 1 head ✅
  V: 8 heads → TP=8, 每卡 1 head ✅

GQA (Q=8, KV=2):
  Q: 8 heads → TP=8, 每卡 1 head ✅
  K: 2 heads → TP=8 ??? 每卡分不到 1 head ❌
  V: 2 heads → TP=8 ??? 每卡分不到 1 head ❌

  KV Head = 2 < TP = 8 → 无法均匀分配
```

#### 6.2 解决策略

**策略 1：复制 KV Head**

```
GQA (Q=8, KV=4), TP=8:
  Q: 8 heads → 每卡 1 head
  K: 4 heads → 前 4 卡各 1 head, 后 4 卡复制
  V: 4 heads → 前 4 卡各 1 head, 后 4 卡复制

  GPU 0: Q_head0, K_head0, V_head0
  GPU 1: Q_head1, K_head1, V_head1
  GPU 2: Q_head2, K_head2, V_head2
  GPU 3: Q_head3, K_head3, V_head3
  GPU 4: Q_head4, K_head0, V_head0  ← 复制 KV
  GPU 5: Q_head5, K_head1, V_head1  ← 复制 KV
  GPU 6: Q_head6, K_head2, V_head2  ← 复制 KV
  GPU 7: Q_head7, K_head3, V_head3  ← 复制 KV

  优点：实现简单
  缺点：KV Head 被复制，显存不均匀
```

**策略 2：部分 rank 共享（分组）**

```
GQA (Q=8, KV=4), TP=8:
  分成 4 组，每组 2 个 rank 共享同一组 KV Head
  GPU 0,1: Q_head0,1, K_head0, V_head0
  GPU 2,3: Q_head2,3, K_head1, V_head1
  GPU 4,5: Q_head4,5, K_head2, V_head2
  GPU 6,7: Q_head6,7, K_head3, V_head3

  优点：KV 不复制
  缺点：组内需要额外同步 K,V 的计算结果
```

### 7. TP 与其他并行策略的关系

#### 7.1 TP vs ZeRO 对比

| 维度 | TP | ZeRO |
|------|-----|------|
| 切分对象 | 权重 + 激活 | 参数 + 梯度 + 优化器状态 |
| 通信频率 | 每层 2 次 | 每步 1-2 次 |
| 通信操作 | AllReduce（小量激活） | AllReduce/AllGather（大量参数/梯度） |
| 激活切分 | ✅（含 SP） | ❌ |
| 互联要求 | NVLink（机内） | IB/以太网可接受（跨机） |
| 典型规模 | 单机 8 卡 | 跨多机 |

#### 7.2 为什么大模型需要 TP + ZeRO 组合

```
175B 模型训练（32 节点 × 8 A100 = 256 GPU）

  机内 TP=8:
    - 切权重和激活，解决单层超显存问题
    - 利用 NVLink 高带宽做 AllReduce

  机间 ZeRO-3 (DP=32):
    - 切参数/梯度/优化器状态
    - 跨机只做每步一次的 AllReduce/AllGather

  通信分解:
    TP 通信: 机内 NVLink (600 GB/s) → 每层 AllReduce → 快
    ZeRO 通信: 机间 IB (50 GB/s) → 每步一次 → 可接受
```

## 面试回答

### 版本一：1 分钟简答

> 张量并行是把单个矩阵乘法切分到多卡。Megatron-LM 提出了 Column Parallel 和 Row Parallel 两种切分方式：Column Parallel 把权重按列切，每卡用完整输入乘自己的列分片，得到输出的不同列；Row Parallel 把权重按行切，每卡用分片输入乘自己的行分片，得到部分和再 AllReduce。关键设计是 Column→Row 的组合中间不需要通信——Column 的输出正好是 Row 的分片输入。一个 Transformer Block 前向需要 2 次 AllReduce（Attention 和 FFN 各一次）。序列并行进一步把 LayerNorm/Dropout 的激活沿序列维度切分，把 AllReduce 拆成 ReduceScatter+AllGather，通信量不变但省了激活显存。TP 每层都要通信，对延迟敏感，所以只在单机 NVLink 内使用。

### 版本二：3 分钟详解

> TP 解决的是"单层前向就超显存"的问题。ZeRO 只切参数不切激活，但 FFN 的中间激活可能就有几 GB。
>
> Megatron-LM 的核心设计是两种切分方式的组合。**Column Parallel** 把权重 $A$ 按列切为 $[A_1, \ldots, A_N]$，每卡算 $Y_i = X \cdot A_i$，得到输出的不同列分片，不需要通信。**Row Parallel** 把权重按行切，每卡算部分和 $Y_i = X_i \cdot A_i$，需要 AllReduce 求和。两者组合时，Column 的列分片输出恰好是 Row 需要的分片输入，所以中间不需要通信——只有 Row 输出时才做一次 AllReduce。
>
> 一个 Transformer Block 的 TP 切分：QKV 用 Column Parallel（多头天然可切），输出投影用 Row Parallel，一次 AllReduce；FFN 第一层 Column Parallel（切中间维度），第二层 Row Parallel，再一次 AllReduce。共 2 次 AllReduce，通信量是 $4bsh$（前向）。
>
> 序列并行解决了非 TP 区域（LayerNorm、Dropout）的激活冗余。把 AllReduce 拆成 ReduceScatter（出口）+ AllGather（入口），在非 TP 区域只存 $1/N$ 的激活。通信量不变（AllReduce 本来就等于 RS+AG），但激活显存大幅降低。
>
> TP 的局限是每层都要通信，对延迟极敏感，所以只在单机 NVLink 内用。大模型通常 TP×DP（ZeRO）组合：机内 TP=8，机间 ZeRO-3。

## 深入追问

### Q1: 为什么 Column Parallel 不需要通信，但输入 X 要完整？

**A**：Column Parallel 中 $Y_i = X \cdot A_i$，每卡需要完整的 $X$（因为 $X$ 的所有行都参与 $A_i$ 的计算）。但 $X$ 通常来自上一个 AllReduce 的输出或输入嵌入层，本来每卡就有完整副本，所以不需要额外通信。如果 $X$ 是上层的分片输出，则需要先 AllGather 再做 Column Parallel。

### Q2: Attention 的 softmax 在 TP 下怎么处理？

**A**：Attention 的 Q、K、V 按 head 切分后，每个 head 的 attention 计算（$QK^T$ → softmax → $\times V$）是完全独立的。每卡只算自己负责的 head，不需要其他 head 的数据。只有最后输出投影（Row Parallel）时才需要 AllReduce。这正是 GQA 的设计灵感——KV head 可以被多个 Q head 共享。

### Q3: SP 把 AllReduce 拆成 RS+AG，真的没有额外代价吗？

**A**：通信量确实不变（AllReduce = RS + AG），但有两个细微差异：
1. **延迟**：两个独立通信操作比一个 AllReduce 略慢（多一次通信启动开销）
2. **overlap 机会**：拆开后，RS 后的非 TP 计算可以和 AllGather overlap，实际上可能更快

在 Megatron-LM 的实现中，SP 通常有轻微的通信开销增加，但激活显存的节省远大于这个代价。

### Q4: TP 能跨机使用吗？

**A**：技术上可以，但不推荐。TP 每层 2 次 AllReduce，如果跨机（IB 25-50 GB/s），通信延迟会成为严重瓶颈。实际中 TP 限制在单机 NVLink 内（带宽 300-600 GB/s）。跨机用 PP 或 ZeRO。这也是为什么大模型训练用 TP×PP×DP 的 3D 并行——TP 机内，PP 机间。

### Q5: TP 和 ZeRO 能同时切同一组参数吗？

**A**：可以，但需要小心。TP 把权重按列/行切分到机内 N 卡，ZeRO-3 把参数分片到所有卡（包括跨机）。实际操作中，ZeRO 的分片粒度通常是 TP group 级别的——即整个 TP group 共享一份参数分片，而不是每张卡单独分片。参考 [11.1 3D 并行](../12. 混合并行篇/1. 3D并行与混合并行策略.md) 的通信域设计。

## 易混淆点

### 1. Column Parallel vs Row Parallel

| 维度 | Column Parallel | Row Parallel |
|------|----------------|-------------|
| 权重切分 | 按列切 $A = [A_1, \ldots, A_N]$ | 按行切 $A = [A_1; \ldots; A_N]$ |
| 输入要求 | 完整 $X$ | 分片 $X_i$ |
| 输出形式 | 分片 $Y_i$（不同列） | 部分和 $Y_i$（需要求和） |
| 通信 | 无（输出不需要同步） | AllReduce（求和） |
| 典型应用 | FFN 第 1 层、QKV 投影 | FFN 第 2 层、输出投影 |

### 2. TP 的 AllReduce vs DDP 的 AllReduce

| 维度 | TP AllReduce | DDP AllReduce |
|------|-------------|---------------|
| 传什么 | 激活值（部分和） | 梯度 |
| 数据量 | $bsh$（小） | $2\Psi$（大） |
| 频率 | 每层 2 次 | 每步 1 次 |
| 互联要求 | NVLink | IB 可接受 |

### 3. SP 的 AllGather/ReduceScatter vs ZeRO 的 AllGather/ReduceScatter

| 维度 | SP | ZeRO-3 |
|------|-----|--------|
| 传什么 | 激活值（序列分片） | 参数/梯度 |
| 数据量 | $bsh$（小） | $\Psi$（大） |
| 频率 | 每层 2 次（进出 TP 区域） | 每步每层 1 次 |
| 目的 | 省激活显存 | 省参数显存 |

### 4. TP 的通信频率 vs PP 的通信频率

- **TP**：每层 2 次 AllReduce → 通信与计算交替
- **PP**：每个 Stage 边界 1 次 P2P → 通信只在 Stage 交接处
- TP 通信频率远高于 PP，所以 TP 对带宽要求更高
