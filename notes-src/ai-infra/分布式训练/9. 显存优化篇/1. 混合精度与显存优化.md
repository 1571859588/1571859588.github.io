# 混合精度与显存优化：把硬件用到极致

> **一句话结论**：混合精度用低位宽（BF16/FP16）做前向反向计算，FP32 维护主权重，显存减半且 Tensor Core 算力翻倍；大模型首选 BF16（范围同 FP32 不用 Loss Scaling）。梯度累积用 $K$ 步小 batch 模拟大 batch，配合 `no_sync()` 跳过中间步通信。Activation Checkpointing 只存每层输入、反向时重算，激活显存从 $O(L)$ 降到 $O(\sqrt{L})$，代价是约 33% 额外计算量。这三项与并行维度正交，几乎所有训练都会用到。

---

## 背景与现象

### 为什么需要混合精度？

前面学完 ZeRO、TP、PP 后，并行策略解决了"多卡协作"的问题。但单卡内部还有大量显存可以省，计算速度也可以更快。

以 7B 模型为例，FP32 训练：

```
FP32 训练 7B 模型（单卡）：
  参数:     7B × 4 bytes = 28 GB
  梯度:     7B × 4 bytes = 28 GB
  优化器:   7B × 12 bytes = 84 GB (AdamW)
  激活:     ~10 GB
  合计:     ~150 GB → 单卡 40GB A100 完全装不下

用 FP16/BF16 混合精度后：
  参数:     7B × 2 bytes = 14 GB   ← 减半！
  梯度:     7B × 2 bytes = 14 GB   ← 减半！
  优化器:   7B × 12 bytes = 84 GB  (仍 FP32)
  激活:     ~5 GB                   ← 减半！
  合计:     ~117 GB → 还是不够，但已经省了 33GB
```

**两个好处**：
1. **省显存**：参数和梯度从 4 bytes 降到 2 bytes，激活也减半
2. **提速**：Tensor Core 对 FP16/BF16 矩阵乘法的算力是 FP32 的 2-8 倍

参考：[1.1 GPU 架构与计算单元](../GPU硬件/2. 硬件基础篇/1. GPU架构与计算单元.md) 中的 Tensor Core 详解

## 核心概念：浮点数格式

### 浮点数的结构

一个浮点数由三部分组成：符号位（Sign）、指数位（Exponent）、尾数位（Mantissa）。

```
浮点数格式：  [Sign] [Exponent] [Mantissa]

  Sign（1 bit）：正负号
  Exponent（E bits）：决定数值范围（能表示多大/多小）
  Mantissa（M bits）：决定精度（有效数字多少位）

  数值 = (-1)^sign × 1.mantissa × 2^(exponent - bias)
```

### 生活类比

把浮点数想象成**科学计数法**：

```
科学计数法：  1.2345 × 10^6

  指数（6）   → 决定数量级（百万级）
  尾数（1.2345）→ 决定精度（5位有效数字）

浮点数一样：
  指数位多 → 能表示的数值范围大（不易溢出）
  尾数位多 → 能表示的精度高（不易舍入误差）
```

### 四种格式对比

```
FP32 (单精度)：
┌───┬──────────┬─────────────────────────┐
│ S │ Exponent │       Mantissa           │
│1b │  8 bits  │        23 bits          │
└───┴──────────┴─────────────────────────┘
  范围: ±3.4×10^38    精度: ~7位有效数字

FP16 (半精度)：
┌───┬──────┬──────────────┐
│ S │ Exp  │  Mantissa    │
│1b │ 5 bits│  10 bits    │
└───┴──────┴──────────────┘
  范围: ±65504          精度: ~3位有效数字
  问题：范围太小！梯度 > 65504 就溢出

BF16 (Brain Float 16)：
┌───┬──────────┬─────────┐
│ S │ Exponent │Mantissa │
│1b │  8 bits  │ 7 bits  │
└───┴──────────┴─────────┘
  范围: ±3.4×10^38     精度: ~2位有效数字
  优点：范围和 FP32 一样！只是精度低
  → 不用担心溢出，但精度损失略大

FP8 (E4M3 格式)：
┌───┬──────┬───────┐
│ S │ Exp  │ Mant  │
│1b │ 4 bits│ 3 bits│
└───┴──────┴───────┘
  范围: ±448            精度: ~1-2位
  优点：显存减半 + 算力再翻倍
  限制：范围极小，需要动态缩放
```

### 详细对比表

| 格式 | 总位宽 | 符号位 | 指数位 | 尾数位 | 最大值 | 最小正值 | 有效数字 | 硬件支持 |
|------|--------|--------|--------|--------|--------|---------|---------|---------|
| FP32 | 32 | 1 | 8 | 23 | 3.4×10³⁸ | 1.2×10⁻³⁸ | ~7 | 所有 GPU |
| FP16 | 16 | 1 | 5 | 10 | 65504 | 5.9×10⁻⁸ | ~3 | V100+ |
| BF16 | 16 | 1 | 8 | 7 | 3.4×10³⁸ | 1.2×10⁻³⁸ | ~2 | A100+ |
| FP8 (E4M3) | 8 | 1 | 4 | 3 | 448 | 1.95×10⁻³ | ~1-2 | H100+ |
| FP8 (E5M2) | 8 | 1 | 5 | 2 | 57344 | 6.1×10⁻⁵ | ~1 | H100+ |

> **新人提示**：BF16 是 Google Brain 团队设计的格式，核心思想是"保留 FP32 的指数位（8位），砍尾数位（23→7）"。这样数值范围与 FP32 完全相同，只是精度低一些。对于深度学习来说，精度低一点问题不大，但溢出是大问题——所以 BF16 成了大模型训练的首选。

### FP16 vs BF16：为什么大模型选 BF16

```
FP16 的问题（指数位只有 5）：
  最大值 = 65504
  梯度 > 65504 → 溢出变成 inf → 训练崩溃
  梯度 < 6e-8  → 下溢出变成 0 → 梯度消失

  → 需要 Loss Scaling 把梯度放大到安全范围
  → 但调 Loss Scaling 很麻烦，动态 scaling 有时不稳定

BF16 的优势（指数位 8，同 FP32）：
  最大值 = 3.4×10^38
  梯度几乎不可能溢出！
  → 不需要 Loss Scaling
  → 大模型训练更稳定

  代价：精度低（只有 7 位尾数 vs FP16 的 10 位）
  → 但深度学习对精度不敏感，训练完全 OK
```

| 维度 | FP16 | BF16 |
|------|------|------|
| 范围 | ±65504 | ±3.4×10³⁸（同 FP32） |
| 精度 | 较高（10 位尾数） | 较低（7 位尾数） |
| 溢出风险 | 高 | 极低 |
| Loss Scaling | 必须 | 不需要 |
| 硬件要求 | V100+ | A100+ |
| 大模型训练 | 旧方案 | **首选** |

## 深入讲解

### 1. BF16 混合精度训练流程

#### 1.1 完整流程图

```
BF16 混合精度训练流程：

  ┌─────────────────────────────────────────────────────┐
  │                    初始化                             │
  │  FP32 Master Weight (主权重，4 bytes/param)         │
  │  ↓ to BF16                                          │
  │  BF16 Model Parameters (2 bytes/param)              │
  └─────────────────────┬───────────────────────────────┘
                        │
  ┌─────────────────────▼───────────────────────────────┐
  │                   前向传播                           │
  │                                                      │
  │  输入 (BF16) → 每层权重 (BF16) → 输出 (BF16)        │
  │  所有矩阵乘法用 BF16 → Tensor Core 加速             │
  │                                                      │
  │  Loss = f(BF16 params, BF16 activations)            │
  │  Loss 是 FP32（归约操作提升精度）                    │
  └─────────────────────┬───────────────────────────────┘
                        │
  ┌─────────────────────▼───────────────────────────────┐
  │                   反向传播                           │
  │                                                      │
  │  梯度计算用 BF16（Tensor Core 加速）                │
  │  ∂L/∂W (BF16, 2 bytes/param)                        │
  │                                                      │
  │  注意：梯度是 BF16，精度有限                         │
  └─────────────────────┬───────────────────────────────┘
                        │
  ┌─────────────────────▼───────────────────────────────┐
  │                 梯度同步（分布式）                    │
  │                                                      │
  │  AllReduce 梯度 (BF16) 或转 FP32 后 AllReduce        │
  │  → 聚合后的梯度 (BF16 或 FP32)                      │
  └─────────────────────┬───────────────────────────────┘
                        │
  ┌─────────────────────▼───────────────────────────────┐
  │                参数更新（关键！）                     │
  │                                                      │
  │  1. BF16 梯度 → FP32 梯度（精度提升）                │
  │  2. FP32 Master Weight 更新：                       │
  │     W_fp32 = W_fp32 - lr × grad_fp32                │
  │  3. 更新后的 FP32 Master Weight → BF16 参数         │
  │     W_bf16 = to_bf16(W_fp32)                        │
  │                                                      │
  │  为什么要 FP32 Master Weight？                       │
  │  → 更新量 lr × grad 通常很小（如 1e-5 × 1e-3 = 1e-8）│
  │  → BF16 的精度只有 7 位尾数，这么小的更新会被舍入掉  │
  │  → 用 FP32 累积更新，避免精度丢失                    │
  └─────────────────────────────────────────────────────┘
```

> **新人提示**：Master Weight 是混合精度的核心——它是一个 FP32 的参数副本，用于精确累加微小的更新量。如果直接用 BF16 参数做更新，每次更新的量（~1e-8）远小于 BF16 的精度（~1e-3），会被完全舍入掉，模型就学不动了。这与 AdamW 优化器状态用 FP32 存储是同一个原因。参考 [3.1 优化器原理与显存开销](../4. 优化器篇/1. 优化器原理与显存开销分析.md)。

#### 1.2 为什么需要 FP32 Master Weight

```
精度问题示意：

  假设参数 W = 1.0（BF16），学习率 lr = 1e-4，梯度 grad = 0.01

  更新量 = lr × grad = 1e-4 × 0.01 = 1e-6

  直接用 BF16 更新：
    W_new = W - 1e-6 = 1.0 - 0.000001 = ???
    BF16 的精度在 1.0 附近约为 2^(-7) ≈ 0.0078
    → 1e-6 远小于 0.0078 → 被舍入为 0
    → W_new = 1.0（没变！）→ 模型不学习

  用 FP32 Master Weight 更新：
    W_fp32 = 1.0（FP32，精度 ~1e-7）
    W_fp32_new = 1.0 - 1e-6 = 0.999999（FP32 能表示）
    → 精确保存更新
    W_bf16_new = to_bf16(0.999999) ≈ 0.9995（BF16 近似）
    → 虽然前向用 BF16 有精度损失，但累积更新在 FP32 中正确保存
```

#### 1.3 PyTorch AMP 代码示例

```python
# ✅ 无需 sudo —— PyTorch Automatic Mixed Precision (AMP)
import torch
from torch.amp import autocast, GradScaler

model = MyModel().cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

# BF16 不需要 GradScaler（范围大，不会溢出）
# FP16 需要 GradScaler 做 Loss Scaling
use_bf16 = True  # A100+ 推荐 BF16

for batch in dataloader:
    batch = {k: v.cuda() for k, v in batch.items()}

    # 前向传播用 autocast 自动选择精度
    with autocast(device_type='cuda', dtype=torch.bfloat16):
        loss = model(**batch)

    # 反向传播
    loss.backward()

    # 参数更新（optimizer 内部用 FP32 Master Weight）
    optimizer.step()
    optimizer.zero_grad()
```

#### 1.4 BF16 的显存收益

| 组件 | FP32 | BF16 混合精度 | 节省 |
|------|------|-------------|------|
| 参数（前向/反向用） | 4Ψ | 2Ψ | 50% |
| 梯度 | 4Ψ | 2Ψ | 50% |
| FP32 Master Weight | 0 | 4Ψ | -4Ψ |
| 优化器状态（AdamW） | 8Ψ | 8Ψ | 0 |
| 激活值 | 4Ψ | 2Ψ | 50% |
| **合计** | **20Ψ** | **18Ψ** | **10%** |

> 注意：混合精度不会让总显存减半，因为 FP32 Master Weight 和优化器状态仍然占很大比例。真正的显存节省来自激活值减半 + 参数/梯度的显存节省。配合 ZeRO 后效果更明显。

### 2. FP16 与 Loss Scaling

#### 2.1 FP16 的溢出问题

```
FP16 的梯度分布（典型深度学习模型）：

  梯度值
  10^2 ┤                              ← 大梯度，OK
       │
  10^0 ┤                    ■■■       ← 大部分梯度在这里
       │              ■■■■■
  10^-2┤        ■■■■■
       │    ■■■
  10^-4┤■■■                     ← 小梯度
       │
  10^-6┤                          ← 更小，可能下溢
       │
  10^-8┤                          ← FP16 最小正值 ~6e-8
       │                              以下变成 0！

  问题：大部分梯度在 10^-2 到 10^-4 范围
        但有些重要的小梯度 < 6e-8 会丢失
```

#### 2.2 Loss Scaling 原理

```
Loss Scaling 的做法：

  1. 计算 Loss 后，乘以一个大的 scale factor S
     scaled_loss = loss × S

  2. 反向传播时，所有梯度自动乘以 S
     scaled_grad = grad × S

  3. 更新参数前，梯度除以 S
     grad = scaled_grad / S

  效果：
  原始梯度 1e-7 → 缩放后 1e-7 × 2^16 = 6.5e-3 → 在 FP16 安全范围！
  原始梯度 1e-2 → 缩放后 1e-2 × 2^16 = 655 → 仍在 FP16 范围内（< 65504）

  关键：选一个足够大的 S 把小梯度拉到安全区，
        但又不能太大导致大梯度溢出
```

#### 2.3 动态 Loss Scaling

```python
# ✅ 无需 sudo —— FP16 动态 Loss Scaling
from torch.amp import GradScaler

model = MyModel().cuda()
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

# GradScaler 自动管理 scale factor
scaler = GradScaler()

for batch in dataloader:
    optimizer.zero_grad()

    with autocast(device_type='cuda', dtype=torch.float16):
        loss = model(**batch)

    # Scale loss → backward → unscale → step
    scaler.scale(loss).backward()
    scaler.step(optimizer)
    scaler.update()  # 自动调整 scale factor

    # GradScaler 的动态调整逻辑：
    # - 如果发现 inf/nan 梯度 → scale 太大，减半
    # - 如果连续 N 步无溢出 → scale 可以翻倍
    # - 自动找到合适的 scale，不需要手动调
```

#### 2.4 BF16 为什么不需要 Loss Scaling

```
BF16 的数值范围 = FP32 的数值范围

  最大值: 3.4 × 10^38
  最小正值: 1.2 × 10^-38

  深度学习的梯度范围: 10^-8 到 10^2
  → 完全在 BF16 的范围内 → 不需要缩放

  → 这就是大模型训练选 BF16 的核心原因：省去了 Loss Scaling 的麻烦
```

### 3. FP8 训练（H100+）

#### 3.1 FP8 格式

H100 GPU 引入了 FP8 支持，有两种格式：

```
FP8 两种格式：

  E4M3（精度优先）：
  ┌───┬──────┬───────┐
  │ S │ Exp  │ Mant  │
  │1b │ 4 bits│ 3 bits│
  └───┴──────┴───────┘
  范围: ±448
  精度: 3 位尾数
  用途: 前向传播（权重和激活）

  E5M2（范围优先）：
  ┌───┬──────┬───────┐
  │ S │ Exp  │ Mant  │
  │1b │ 5 bits│ 2 bits│
  └───┴──────┴───────┘
  范围: ±57344
  精度: 2 位尾数
  用途: 反向传播（梯度，范围更大防溢出）
```

#### 3.2 FP8 训练流程

```
FP8 混合精度训练（H100 Transformer Engine）：

  ┌─────────────────────────────────────────────┐
  │ 前向传播：FP8 (E4M3)                         │
  │  - 权重: FP8                                 │
  │  - 激活: FP8                                 │
  │  - 矩阵乘法: FP8 × FP8 → FP32 累加          │
  │  - per-tensor 动态缩放（防止溢出）            │
  │                                              │
  │ 反向传播：FP8 (E5M2) for 梯度                │
  │  - 梯度: FP8                                 │
  │  - per-tensor 动态缩放                        │
  │                                              │
  │ 参数更新：FP32 Master Weight                 │
  │  - 同 BF16 混合精度                          │
  └─────────────────────────────────────────────┘

  显存收益（相比 BF16）：
  - 参数: 2 bytes → 1 byte (FP8)
  - 激活: 2 bytes → 1 byte (FP8)
  - 算力: H100 FP8 算力是 BF16 的 2 倍
```

#### 3.3 FP8 的动态缩放

```
FP8 的范围只有 ±448（E4M3），远小于 BF16 的 ±3.4×10^38
→ 需要 per-tensor 动态缩放

  每个 tensor 计算前：
  1. 找到 tensor 中的最大绝对值 max_val
  2. 计算 scale = 448 / max_val（让最大值映射到 FP8 上限）
  3. tensor 乘以 scale 转为 FP8
  4. 计算后结果除以 scale 恢复

  → 这是 per-tensor 的，不同 tensor 有不同 scale
  → Transformer Engine 自动管理
```

#### 3.4 FP8 的适用场景与风险

| 维度 | BF16 | FP8 |
|------|------|-----|
| 显存 | 2 bytes | 1 byte |
| 算力 | 基准 | 2× BF16 |
| 精度 | 7 位尾数 | 2-3 位尾数 |
| 硬件 | A100+ | H100+ |
| 稳定性 | 高 | 中（需要调参） |
| 适用 | 大模型首选 | 推理/前向、大batch训练 |

> **风险提示**：FP8 精度低（2-3 位有效数字），对某些任务（如小 batch、高精度要求的微调）可能导致训练不稳定。通常 FP8 用于推理或大 batch 训练，训练时前向用 FP8、反向梯度用 BF16 的混合策略更安全。

参考：[1.1 GPU 架构与计算单元](../GPU硬件/2. 硬件基础篇/1. GPU架构与计算单元.md) 中 Tensor Core 对 FP8 的支持

### 4. 梯度累积（Gradient Accumulation）

#### 4.1 核心思想

当显存不够用大 batch size 时，用多个小 batch 的梯度累加来等效大 batch：

```
梯度累积（K=4）：

  正常训练（batch=32）：
  ┌──────────────────────┐
  │ 前向(batch=32)       │
  │ 反向(梯度 g)         │
  │ optimizer.step(g)   │ ← 更新
  └──────────────────────┘

  梯度累积（K=4, batch=8）：
  ┌──────────────────────┐
  │ 前向(batch=8)  第1步  │
  │ 反向(梯度 g1)        │ ← 累加到 g
  │ 前向(batch=8)  第2步  │
  │ 反向(梯度 g2)        │ ← g = g1 + g2
  │ 前向(batch=8)  第3步  │
  │ 反向(梯度 g3)        │ ← g = g1 + g2 + g3
  │ 前向(batch=8)  第4步  │
  │ 反向(梯度 g4)        │ ← g = g1 + g2 + g3 + g4
  │ optimizer.step(g/K) │ ← 更新（除以 K 等效平均）
  └──────────────────────┘

  等效 batch size = 8 × 4 = 32（与正常训练一样）
```

#### 4.2 Effective Batch Size

$$\text{Effective Batch Size} = \text{batch\_size} \times \text{accumulation\_steps} \times \text{world\_size}$$

```
示例：8 张 GPU，每卡 batch=4，累积 4 步
  Effective Batch Size = 4 × 4 × 8 = 128
```

#### 4.3 与 DDP 的配合：no_sync()

```python
# ✅ 无需 sudo —— 梯度累积 + DDP 的标准写法
import torch
import torch.distributed as dist
from torch.nn.parallel import DistributedDataParallel as DDP

model = MyModel().cuda()
model = DDP(model, device_ids=[dist.get_rank()])
optimizer = torch.optim.AdamW(model.parameters(), lr=1e-4)

accumulation_steps = 4

for i, batch in enumerate(dataloader):
    # 关键：中间步跳过 DDP 的梯度同步
    is_last_step = (i + 1) % accumulation_steps == 0

    if not is_last_step:
        # no_sync 上下文：不触发 AllReduce
        with model.no_sync():
            loss = model(batch)
            loss = loss / accumulation_steps  # 除以 K
            loss.backward()
    else:
        # 最后一步：正常前向+反向，触发 AllReduce
        loss = model(batch)
        loss = loss / accumulation_steps
        loss.backward()

        # 这一步的梯度会自动 AllReduce
        # 累积的梯度 + 当前梯度 = 完整梯度
        optimizer.step()
        optimizer.zero_grad()
```

> **新人提示**：DDP 的 `no_sync()` 是梯度累积的关键。如果不跳过中间步的 AllReduce，每步都会做梯度同步（通信），但中间步的梯度还没累积完，同步是浪费的。`no_sync()` 让前 K-1 步只在本地累积梯度，第 K 步才做一次 AllReduce。

#### 4.4 与 PP 的配合

PP 的 micro-batch 本质就是一种梯度累积——多个 micro-batch 的梯度累积后才更新参数：

```
PP 的梯度累积：

  M=8 个 micro-batch 流过 P=4 个 Stage
  每个 micro-batch 的梯度累积
  最后统一做 optimizer.step()

  → PP 天然包含梯度累积，不需要额外配置
  → Effective Batch Size = micro_batch × M × DP
```

#### 4.5 注意事项

```
梯度累积的关键陷阱：Loss 必须除以 K

  错误写法：
    for i, batch in enumerate(dataloader):
        loss = model(batch)           # ← 没有除以 K
        loss.backward()
        if (i+1) % K == 0:
            optimizer.step()
            optimizer.zero_grad()

    → 梯度 = g1 + g2 + g3 + g4（是正常梯度的 K 倍！）
    → 学习率等效放大 K 倍 → 训练不稳定

  正确写法：
    for i, batch in enumerate(dataloader):
        loss = model(batch) / K       # ← 除以 K（等价于梯度平均）
        loss.backward()
        if (i+1) % K == 0:
            optimizer.step()
            optimizer.zero_grad()

    → 梯度 = (g1+g2+g3+g4) / K = 平均梯度
    → 与大 batch 训练数学等价
```

| 注意点 | 说明 |
|--------|------|
| Loss / K | 梯度累积的 Loss 要除以 K，否则等效学习率放大 |
| BatchNorm | 梯度累积时 BN 的统计量只用了 1/K 的数据，可能不准 |
| Dropout | 每步的 dropout mask 不同，累积时需注意 |
| DDP no_sync | 中间步用 `no_sync()` 跳过 AllReduce |

### 5. Activation Checkpointing（激活重计算）

#### 5.1 问题：激活显存随层数线性增长

```
不用 Checkpointing 的激活显存：

  每层激活值 ≈ b × s × h × 几个中间结果 × 2 bytes
  L 层总激活 ≈ L × 单层激活

  GPT-3 175B (96层, b=8, s=2048, h=12288)：
    单层激活 ≈ 8 × 2048 × 12288 × 4 × 2 ≈ 1.6 GB
    96 层 ≈ 154 GB → 激活显存远超参数显存！
```

#### 5.2 核心思想：用计算换显存

```
Activation Checkpointing 原理：

  不用 Checkpointing：
  ┌─────────────────────────────────────────────┐
  │ 前向：存 L1输入, L1输出, L2输入, L2输出, ...│  ← 全部保存
  │                                              │
  │ 反向：用存的激活值计算梯度                   │  ← 直接用
  └─────────────────────────────────────────────┘
  显存: O(L) × 单层激活

  Full Checkpointing：
  ┌─────────────────────────────────────────────┐
  │ 前向：只存 L1输入, L2输入, L3输入, ...      │  ← 只存每层输入
  │      (中间激活用完即弃)                      │
  │                                              │
  │ 反向：重算 L3前向 → 得到中间激活 → 算梯度   │  ← 重新前向
  │      重算 L2前向 → 得到中间激活 → 算梯度   │
  │      ...                                     │
  └─────────────────────────────────────────────┘
  显存: O(L) × 单层输入  ← 只有输入，远小于全部中间激活
  代价: 前向计算量 × 2（反向时重算一遍）
```

#### 5.3 数学分析：为什么是 $O(\sqrt{L})$

```
Checkpoint 的分层策略：

  把 L 层分成 √L 个组，每组 √L 层：
  ┌─────────┬─────────┬─────────┐
  │ 组1     │ 组2     │ ...     │
  │ L1-L√L  │ L(√L+1)-│ L-L层   │
  │         │ L(2√L)  │         │
  └─────────┴─────────┴─────────┘

  前向时：
  - 存每个组的输入：√L 份 checkpoint
  - 组内的中间激活用完即弃

  反向时：
  - 重算当前组的 √L 层前向 → 得到中间激活
  - 用中间激活算梯度
  - 下一个组同理

  显存分析：
  - checkpoint 存储：√L × 单层输入 = O(√L)
  - 重算时临时激活：√L × 单层激活 = O(√L)
  - 总计：O(√L)

  原来是 O(L) × 单层激活，现在是 O(√L)
  → 如果 L=96, √L ≈ 10, 省了约 10 倍
```

#### 5.4 计算代价

```
不用 Checkpointing：
  前向: 1 × L 层
  反向: 1 × L 层（用存的激活算梯度）
  总计: 2L 层计算量

Full Checkpointing：
  前向: 1 × L 层（只存输入，丢弃中间激活）
  反向: 1 × L 层（算梯度）+ 1 × L 层（重算前向得到中间激活）
  总计: 3L 层计算量

  额外计算: L 层 / 3L 总计 = 33%

  → 代价是增加约 33% 的前向计算量
  → 但省了 O(L) → O(√L) 的激活显存
  → 对于大模型，显存是瓶颈，33% 的计算代价完全值得
```

#### 5.5 Selective Checkpointing

不是所有层都值得重算——有些层激活大但计算量小，重算很划算；有些层计算量大但激活小，重算不划算。

```
Selective Checkpointing 策略：

  Transformer Block 内部：

  ┌──────────────────────────────────────────┐
  │ LayerNorm        ← 激活: b×s×h           │
  │ QKV Linear       ← 激活: b×s×3h          │
  │ QK^T (attn)     ← 激活: b×h×s×s ← 大！ │  ← 值得 checkpoint
  │ Softmax          ← 激活: b×h×s×s ← 大！ │  ← 值得 checkpoint
  │ AV (attn)       ← 激活: b×s×h           │
  │ Output Linear    ← 激活: b×s×h           │
  │ LayerNorm        ← 激活: b×s×h           │
  │ FFN Linear 1     ← 激活: b×s×4h ← 大    │
  │ GELU             ← 激活: b×s×4h ← 大    │  ← 值得 checkpoint
  │ FFN Linear 2     ← 激活: b×s×h           │
  └──────────────────────────────────────────┘

  Selective 策略：
  - 只 checkpoint Attention 的 softmax 和 FFN 的 GELU
  - 这些操作计算量小但激活值大
  - 重算代价低，收益高

  Megatron-LM 的默认策略：
  - Full Checkpoint: 整个 Block 重算
  - Selective Checkpoint: 只重算选定的操作（推荐）
```

| 策略 | 激活显存 | 计算代价 | 适用 |
|------|---------|---------|------|
| 不用 Checkpoint | $O(L)$ | 0 | 小模型 |
| Full Checkpoint | $O(\sqrt{L})$ | +33% | 大模型，显存紧张 |
| Selective Checkpoint | $O(L) \times 0.3$ | +10-15% | **推荐，性价比最高** |

#### 5.6 PyTorch 代码示例

```python
# ✅ 无需 sudo —— PyTorch Activation Checkpointing
from torch.utils.checkpoint import checkpoint

class TransformerLayer(nn.Module):
    def __init__(self, config):
        super().__init__()
        self.attention = Attention(config)
        self.ffn = FFN(config)
        self.norm1 = nn.LayerNorm(config.hidden_size)
        self.norm2 = nn.LayerNorm(config.hidden_size)

    def forward(self, x):
        # 方式1: 不用 checkpoint（正常模式）
        # h = self.norm1(x)
        # x = x + self.attention(h)
        # h = self.norm2(x)
        # x = x + self.ffn(h)
        # return x

        # 方式2: Full Checkpoint（整个 Block 重算）
        def custom_forward(x):
            h = self.norm1(x)
            x = x + self.attention(h)
            h = self.norm2(x)
            x = x + self.ffn(h)
            return x

        # use_reentrant=False 是 PyTorch 2.x 推荐的新实现
        return checkpoint(custom_forward, x, use_reentrant=False)


# 更简洁的用法：直接装饰
from torch.utils.checkpoint import checkpoint_sequential

# 对多层 Sequential 模型做 checkpoint
# 每 checkpoint_layer_num 层存一个 checkpoint
checkpoint_sequential(layers, num_layers // checkpoint_every, input)
```

#### 5.7 与并行策略的配合

```
Activation Checkpointing + 各并行策略：

  + DDP/ZeRO: 无冲突，ZeRO 切参数，Checkpointing 省激活 → 天然配合
  + TP: SP 已减少激活冗余，Checkpointing 进一步省 → 配合使用
  + PP: 1F1B 峰值激活 = P 份，Checkpointing 把每份从 O(L) 降到 O(√L)
        → Megatron-LM 默认 PP + selective checkpointing

  典型大模型配置：
    ZeRO-3 + TP + PP + BF16 + Selective Checkpointing + 梯度累积
    → 把每种优化都用上，硬件利用到极致
```

## 面试回答

### 版本一：1 分钟简答

> 混合精度训练用 BF16 做前向反向计算，FP32 维护 Master Weight 做参数更新。BF16 的指数位和 FP32 相同（8位），数值范围一样大不会溢出，所以不需要 Loss Scaling，大模型训练首选 BF16。FP16 范围小（最大 65504）容易溢出，需要动态 Loss Scaling 把梯度放大到安全范围。FP8 是 H100 引入的更低位宽，显存再减半、算力再翻倍，但精度低需要 per-tensor 动态缩放，通常前向用 FP8、反向用 BF16。梯度累积用 K 步小 batch 累加梯度等效大 batch，配合 DDP 的 `no_sync()` 跳过中间步通信。Activation Checkpointing 只存每层输入、反向时重算前向，激活显存从 O(L) 降到 O(√L)，代价是约 33% 额外计算量。这三项与并行策略正交，几乎所有训练都会用到。

### 版本二：3 分钟详解

> 混合精度的核心是两套精度——计算用低位宽（省显存+提速），更新用 FP32（保精度）。BF16 的指数位与 FP32 完全相同（8位），数值范围 ±3.4×10^38，梯度不会溢出，所以不需要 Loss Scaling。相比之下 FP16 指数位只有 5 位，最大值 65504，梯度容易溢出，必须用动态 Loss Scaling 把梯度放大到安全范围。所以 A100 之后的大模型训练普遍用 BF16。
>
> 为什么需要 FP32 Master Weight？因为每次参数更新量 $lr \times grad$ 通常很小（~1e-6），BF16 的精度（~1e-3）会把这个更新量舍入为零。用 FP32 累积更新，避免精度丢失。这与 AdamW 的优化器状态用 FP32 是同一个原因。
>
> 梯度累积解决"显存不够用大 batch"的问题。连续 K 步前向+反向累积梯度，每 K 步更新一次。Effective Batch Size = batch × K × world_size。配合 DDP 时用 `no_sync()` 跳过前 K-1 步的 AllReduce，只在第 K 步同步。注意 Loss 要除以 K。
>
> Activation Checkpointing 解决"激活显存随层数线性增长"的问题。Full Checkpointing 只存每层输入，反向时重算前向得到中间激活。显存从 O(L) 降到 O(√L)（把 L 层分成 √L 组），代价是约 33% 额外计算。Selective Checkpointing 更聪明——只重算"计算量小但激活大"的操作（如 Attention 的 softmax），代价只有 10-15%。Megatron-LM 默认用 selective + PP + 1F1B。
>
> 这些技术与 ZeRO/TP/PP 正交，典型大模型训练配置是 ZeRO-3 + TP + PP + BF16 + Selective Checkpointing + 梯度累积。

## 深入追问

### Q1: BF16 精度低（只有 7 位尾数），训练效果会不会变差？

**A**：理论上会有一点点精度损失，但实践中影响很小：
1. 深度学习对精度不敏感——梯度的方向比大小更重要，BF16 足以表示方向
2. 关键操作（参数更新、Loss 归约）仍然用 FP32
3. 大量实验证明 BF16 训练的模型效果与 FP32 接近

唯一需要注意的是某些精度敏感的操作（如 LayerNorm 中的方差计算），PyTorch AMP 会自动用 FP32 处理这些操作。

### Q2: FP8 训练现在成熟吗？

**A**：FP8 是 H100 之后的新技术，正在快速成熟：
- **推理**：已广泛使用 FP8（如 TensorRT-LLM）
- **训练**：大 batch 训练可用，小 batch/微调可能不稳定
- **典型策略**：前向用 FP8（E4M3），反向梯度用 BF16，参数更新用 FP32
- **风险**：精度低（2-3 位尾数），需要 per-tensor 动态缩放，调试成本高

建议：除非对速度要求极高，否则 BF16 是更稳妥的选择。

### Q3: 梯度累积和 PP 的 micro-batch 有什么区别？

**A**：本质上是同一种技术，但应用场景不同：
- **梯度累积**：解决"单卡显存不够用大 batch"的问题，单卡内做 K 步累加
- **PP 的 micro-batch**：解决"流水线 Bubble"的问题，把 mini-batch 切成 M 个 micro-batch 流过流水线

两者可以叠加：PP 的每个 micro-batch 可以再做梯度累积。Effective Batch Size = micro_batch × M × DP × K。

### Q4: Activation Checkpointing 的 33% 计算代价能接受吗？

**A**：对于大模型来说，显存是瓶颈而不是计算量：
- 如果不用 Checkpointing，显存不够 → 无法训练
- 用了 Checkpointing，多算 33% 但能训练 → 值得

而且实际上 33% 是上界——如果用 Selective Checkpointing（只重算轻操作），代价只有 10-15%。再加上通信与计算 overlap，实际减速不到 10%。

### Q5: 混合精度和 ZeRO 怎么配合？

**A**：天然配合，互不冲突：
- ZeRO 切分的是"参数+梯度+优化器状态"，混合精度影响的是"精度（位宽）"
- ZeRO-3 分片后，每卡存 1/N 的 BF16 参数 + 1/N 的 FP32 Master Weight + 1/N 的优化器状态
- 通信时传 BF16（2 bytes），比传 FP32（4 bytes）省一半带宽

所以 ZeRO + BF16 是标准组合，几乎每个大模型训练都这么配。

## 易混淆点

### 1. Master Weight vs 优化器状态

| 概念 | 是什么 | 为什么用 FP32 |
|------|--------|-------------|
| Master Weight | 参数的 FP32 副本 | 精确累加微小更新量 |
| 优化器状态 | Adam 的 m、v | 精确累积梯度统计量 |

> 两者都是 FP32 副本，但 Master Weight 是参数的副本，优化器状态是动量的副本。在 AdamW 中，FP32 Master Weight 就是优化器的 `params`，m 和 v 是优化器状态。

### 2. Loss Scaling vs 学习率缩放

| 概念 | 做什么 | 为什么 |
|------|--------|--------|
| Loss Scaling | 梯度乘大数再除 | 防止 FP16 小梯度下溢 |
| 学习率缩放 | Effective batch 变大时调 lr | 保持训练稳定性 |

> Loss Scaling 是精度问题（FP16 范围不够），学习率缩放是优化问题（大 batch 需要大 lr）。两者无关。

### 3. 梯度累积 vs micro-batch

| 概念 | 场景 | 通信 |
|------|------|------|
| 梯度累积 | 单卡内，K 步小 batch 累加 | DDP 用 `no_sync()` 跳过 |
| micro-batch (PP) | 多卡 PP，M 个 micro-batch 流水线 | Stage 边界 P2P |

### 4. Full vs Selective Checkpointing

| 策略 | 重算什么 | 显存节省 | 计算代价 |
|------|---------|---------|---------|
| Full | 整个 Block | O(√L) | +33% |
| Selective | 只重算轻操作 | O(L)×0.7 | +10-15% |
