# 推理加速方向 - 24 周实战项目学习路线

> **主线方向**：`GPU Kernel/算子 → 单机推理引擎 → serving 调度/KV cache → 量化/编译优化 → 分布式推理/K8s 生产化`
>
> 训练、RAG、MLOps 需要懂，但不是主战场；它们主要用来构造真实 workload、评估效果和部署系统。

---

## 一、24 周实战路线

| 阶段 | 项目重点 | 推荐仓库 | 你要做出的成果 |
|---|---|---|---|
| 0-2 周 | 推理性能基础 | [meta-pytorch/gpt-fast](https://github.com/meta-pytorch/gpt-fast) | 跑通 Llama/Qwen 小模型推理，记录 TTFT、TPOT、tokens/s、显存、带宽；复现 `torch.compile`、int8/int4、speculative decoding、tensor parallel 的收益。gpt-fast 本身就是 `<1000 LOC` 的 PyTorch-native 推理项目，适合入门拆解。 |
| 3-5 周 | Triton/CUDA kernel | [gpu-mode/Triton-Puzzles](https://github.com/gpu-mode/Triton-Puzzles)、[triton-lang/triton](https://github.com/triton-lang/triton)、[Dao-AILab/flash-attention](https://github.com/Dao-AILab/flash-attention) | 实现并 benchmark `RMSNorm / RoPE / softmax / matmul / attention`，写出 profiler 报告。Triton Puzzles 从内存 load/store 到 FlashAttention、量化神经网络都有练习，很适合打基础。 |
| 6-9 周 | 从零做小型 LLM 推理引擎 | [GeeeekExplorer/nano-vllm](https://github.com/GeeeekExplorer/nano-vllm)、[sgl-project/mini-sglang](https://github.com/sgl-project/mini-sglang)、[jmaczan/tiny-vllm](https://github.com/jmaczan/tiny-vllm) | 做一个自己的 `mini-inference-engine`：支持 prefill/decode、KV cache、continuous batching、PagedAttention、OpenAI API。`nano-vllm` 约 1200 行，`mini-sglang` 约 5000 行并含 radix cache/chunked prefill/overlap scheduling，`tiny-vllm` 更硬核，C++/CUDA 从零实现。 |
| 10-12 周 | 主流 serving 框架压测 | [vllm-project/vllm](https://github.com/vllm-project/vllm)、[sgl-project/sglang](https://github.com/sgl-project/sglang)、[InternLM/lmdeploy](https://github.com/InternLM/lmdeploy)、[NVIDIA/TensorRT-LLM](https://github.com/NVIDIA/TensorRT-LLM) | 做 `LLM Serving Benchmark Lab`：同一模型、同一 workload 下比较 vLLM/SGLang/LMDeploy/TensorRT-LLM 的吞吐、p95 latency、TTFT、TPOT、KV cache 命中率。vLLM 覆盖 PagedAttention、continuous batching、prefix cache、chunked prefill、量化、spec decoding 等核心能力。 |
| 13-15 周 | 量化与低精度推理 | [mit-han-lab/llm-awq](https://github.com/mit-han-lab/llm-awq)、[mit-han-lab/smoothquant](https://github.com/mit-han-lab/smoothquant)、[ModelCloud/GPTQModel](https://github.com/ModelCloud/GPTQModel)、[pytorch/ao](https://github.com/pytorch/ao) | 做 `Quantization Tradeoff Report`：AWQ/GPTQ/SmoothQuant/TorchAO 对比 FP16/FP8/INT8/INT4 的速度、显存、困惑度或简单 eval。AWQ 有 4-bit CUDA kernel 和 TinyChat，SmoothQuant 是 W8A8 PTQ 经典路线。 |
| 16-19 周 | KV cache / 调度 / 小众高价值方向 | [LMCache/LMCache](https://github.com/LMCache/LMCache)、[kvcache-ai/Mooncake](https://github.com/kvcache-ai/Mooncake)、[microsoft/sarathi-serve](https://github.com/microsoft/sarathi-serve)、[microsoft/vidur](https://github.com/microsoft/vidur)、[punica-ai/punica](https://github.com/punica-ai/punica) | 做 `KV-cache-aware Serving System`：实现 prefix-aware routing、KV cache offload/reuse、chunked prefill、multi-LoRA batching 的实验。LMCache 专注把 KV cache 变成可持久化、可观测、可跨 engine 复用的层，Mooncake 是 KVCache-centric disaggregated serving，很容易聊出深度。 |
| 20-24 周 | 分布式推理生产化 | [vllm-project/production-stack](https://github.com/vllm-project/production-stack)、[llm-d/llm-d](https://github.com/llm-d/llm-d)、[ai-dynamo/dynamo](https://github.com/ai-dynamo/dynamo) | 做最终项目：K8s 部署 vLLM/SGLang，多副本路由，Grafana 监控 TTFT、latency、pending/running requests、GPU KV usage/cache hit rate；再加入 KV-aware routing 或 prefill/decode disaggregation。vLLM production-stack 已经提供 K8s/Grafana/router 参考。 |

## 二、小众但面试官会感兴趣的项目

优先看这几个：

1. **[jmaczan/tiny-vllm](https://github.com/jmaczan/tiny-vllm)**：C++/CUDA 从零做 vLLM-like engine，覆盖 Safetensors、KV cache、continuous batching、online softmax、PagedAttention。比"我调过 vLLM 参数"更有说服力。
2. **[microsoft/vidur](https://github.com/microsoft/vidur)**：LLM inference simulator，适合做调度策略、batching、资源配置实验，不烧很多 GPU。
3. **[microsoft/sarathi-serve](https://github.com/microsoft/sarathi-serve)**：chunked prefill / low-latency serving 方向，适合和 vLLM/SGLang 做调度对比。
4. **[punica-ai/punica](https://github.com/punica-ai/punica)**：multi-tenant LoRA serving，面试里可以聊多 adapter batching、kernel 复用、服务隔离。
5. **[kvcache-ai/Mooncake](https://github.com/kvcache-ai/Mooncake)**：KVCache-centric disaggregated architecture，是比较新的系统方向，和长上下文、agent、RAG workload 非常贴。
6. **[skyzh/tiny-llm](https://github.com/skyzh/tiny-llm)**：如果你有 Apple Silicon，也可以走 Metal/本地推理路线；它的 roadmap 覆盖 KV cache、quantized matmul、FlashAttention、continuous batching、chunked prefill、PagedAttention、MoE、speculative decoding。

## 三、最终简历三件套

> 建议最后只把 2-3 个项目写进简历，别把所有仓库都堆上去。

### 项目 1：LLM Inference Benchmark & Optimization Lab

**简历写法**：构建多后端 LLM 推理压测平台，对比 vLLM/SGLang/TensorRT-LLM/LMDeploy 在不同 batch、prompt length、并发和量化配置下的 TTFT、TPOT、吞吐、p95 latency 与显存占用，定位 KV cache、prefill/decode、kernel 选择对性能的影响。

### 项目 2：Mini LLM Serving Engine from Scratch

**简历写法**：从零实现轻量 LLM 推理引擎，支持 Safetensors 加载、prefill/decode、KV cache、continuous batching、PagedAttention/OpenAI-compatible API，并通过 Nsight/PyTorch profiler 分析 kernel 与显存瓶颈。

### 项目 3：KV-cache-aware Distributed Serving System

**简历写法**：基于 vLLM/SGLang + LMCache/Mooncake 搭建 KV cache 复用与 prefix-aware routing 原型，支持长上下文/多轮对话 workload，接入 Grafana 指标并验证 TTFT、吞吐和 GPU KV 使用率改善。

## 四、其他方向要学到什么程度

- **训练**：懂 Transformer、attention、RoPE、GQA/MQA、MoE、LoRA 原理即可，不必主攻训练框架。
- **RAG/Agent**：不用深做应用，但要会构造长上下文、多轮、共享 prefix 的真实 workload。
- **MLOps/K8s**：要会 Docker、K8s、Helm、Prometheus/Grafana，因为推理加速最终一定要落到服务吞吐和成本。
- **编译器**：先掌握 Triton、torch.compile、CUDA graph、FlashAttention 思路；TVM/MLC-LLM 可作为进阶，不必一开始硬啃。
