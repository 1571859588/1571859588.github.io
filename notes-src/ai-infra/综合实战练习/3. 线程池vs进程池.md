# 练习 3：线程池 vs 进程池执行 I/O 任务和 CPU 任务

> 关联笔记：`综合实战：跨层思维与自检练习.md` 5.1 练习 3
> 知识点：GIL、线程 vs 进程、I/O 密集 vs CPU 密集、concurrent.futures
> 难度：中等

---

## 一句话结论

I/O 密集任务（网络/磁盘等待）用线程池即可——GIL 在 I/O 阻塞时释放，线程能真正并行等待；CPU 密集任务（纯计算）必须用进程池——GIL 阻止多线程同时执行 Python 字节码，多进程各自有独立 GIL 才能真正并行计算。

---

## 题目

分别用线程池和进程池执行 I/O 等待任务和纯 Python CPU 任务，记录并解释耗时差异。

---

## 知识点

| 概念 | 说明 |
|------|------|
| GIL（全局解释器锁） | CPython 同一时刻只允许一个线程执行 Python 字节码 |
| I/O 密集 | 大部分时间在等网络/磁盘/键盘，CPU 空闲，GIL 在 I/O 阻塞时释放 |
| CPU 密集 | 大部分时间在做计算，GIL 被持有，多线程无法并行 |
| `ThreadPoolExecutor` | 线程池，共享内存空间，受 GIL 限制 |
| `ProcessPoolExecutor` | 进程池，独立内存空间，各自有 GIL，有进程创建开销 |

---

## 一步步实现

### 第 1 步：I/O 密集任务对比

```python
# io_benchmark.py
import time
import os
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def io_task(dummy_arg):
    """模拟 I/O 等待（如网络请求、磁盘读写）"""
    time.sleep(0.5)           # I/O 阻塞，GIL 释放
    return dummy_arg

def run_io_benchmark():
    tasks = list(range(8))    # 8 个任务，每个等 0.5 秒
    print(f"I/O 任务：{len(tasks)} 个，每个 sleep(0.5s)")

    # 串行（基准）
    t0 = time.perf_counter()
    for t in tasks:
        io_task(t)
    t_serial = time.perf_counter() - t0
    print(f"  串行:       {t_serial:.3f}s")

    # 线程池
    t0 = time.perf_counter()
    with ThreadPoolExecutor(max_workers=8) as pool:
        list(pool.map(io_task, tasks))
    t_thread = time.perf_counter() - t0
    print(f"  线程池(8):  {t_thread:.3f}s   加速比: {t_serial/t_thread:.1f}x")

    # 进程池
    t0 = time.perf_counter()
    with ProcessPoolExecutor(max_workers=8) as pool:
        list(pool.map(io_task, tasks))
    t_process = time.perf_counter() - t0
    print(f"  进程池(8):  {t_process:.3f}s   加速比: {t_serial/t_process:.1f}x")

run_io_benchmark()
```

#### 预期结果

```
I/O 任务：8 个，每个 sleep(0.5s)
  串行:       4.002s
  线程池(8):  0.522s   加速比: 7.7x
  进程池(8):  0.616s   加速比: 6.5x
```

**解释**：
- 串行：8 × 0.5s = 4s
- 线程池：8 个线程同时 sleep，GIL 在 `time.sleep` 时释放，所以真正并行等待，≈ 0.5s
- 进程池：也能并行，但进程创建有额外开销（fork/spawn），比线程池略慢

### 第 2 步：CPU 密集任务对比

```python
# cpu_benchmark.py
import time
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

def cpu_task(n):
    """纯 Python 计算：累加 0 到 n-1"""
    total = 0
    for i in range(n):
        total += i
    return total

def run_cpu_benchmark():
    n = 10_000_000            # 每个任务计算 1000 万次
    tasks = [n] * 4           # 4 个任务
    print(f"CPU 任务：{len(tasks)} 个，每个累加 {n:,} 次")

    # 串行（基准）
    t0 = time.perf_counter()
    for t in tasks:
        cpu_task(t)
    t_serial = time.perf_counter() - t0
    print(f"  串行:       {t_serial:.3f}s")

    # 线程池
    t0 = time.perf_counter()
    with ThreadPoolExecutor(max_workers=4) as pool:
        list(pool.map(cpu_task, tasks))
    t_thread = time.perf_counter() - t0
    print(f"  线程池(4):  {t_thread:.3f}s   加速比: {t_serial/t_thread:.1f}x")

    # 进程池
    t0 = time.perf_counter()
    with ProcessPoolExecutor(max_workers=4) as pool:
        list(pool.map(cpu_task, tasks))
    t_process = time.perf_counter() - t0
    print(f"  进程池(4):  {t_process:.3f}s   加速比: {t_serial/t_process:.1f}x")

run_cpu_benchmark()
```

#### 预期结果

```
CPU 任务：4 个，每个累加 10,000,000 次
  串行:       1.234s
  线程池(4):  1.198s   加速比: 1.0x
  进程池(4):  0.351s   加速比: 3.5x
```

**解释**：
- 串行：4 × 0.31s ≈ 1.23s
- 线程池：**几乎没有加速**（≈ 1.0x）！因为 `cpu_task` 是纯 Python 循环，GIL 全程被持有，4 个线程只能轮流执行，不比串行快
- 进程池：**明显加速**（≈ 3.5x）。每个进程有独立 GIL，4 个进程真正并行计算。不到 4x 是因为进程创建和数据传输有开销

### 第 3 步：完整对比脚本（一键运行两个场景）

```python
# full_benchmark.py
import time
from concurrent.futures import ThreadPoolExecutor, ProcessPoolExecutor

# ===== I/O 密集任务 =====
def io_task(x):
    time.sleep(0.5)
    return x

# ===== CPU 密集任务 =====
def cpu_task(n):
    total = 0
    for i in range(n):
        total += i
    return total

def benchmark(name, func, args, workers):
    """通用基准测试函数"""
    # 串行
    t0 = time.perf_counter()
    for a in args:
        func(a)
    t_serial = time.perf_counter() - t0

    # 线程池
    t0 = time.perf_counter()
    with ThreadPoolExecutor(max_workers=workers) as pool:
        list(pool.map(func, args))
    t_thread = time.perf_counter() - t0

    # 进程池
    t0 = time.perf_counter()
    with ProcessPoolExecutor(max_workers=workers) as pool:
        list(pool.map(func, args))
    t_process = time.perf_counter() - t0

    print(f"\n{'='*50}")
    print(f"{name}（{len(args)} 个任务，{workers} 并发）")
    print(f"{'='*50}")
    print(f"  串行:    {t_serial:.3f}s")
    print(f"  线程池:  {t_thread:.3f}s  (加速 {t_serial/t_thread:.1f}x)")
    print(f"  进程池:  {t_process:.3f}s  (加速 {t_serial/t_process:.1f}x)")

if __name__ == "__main__":
    benchmark("I/O 密集", io_task, list(range(8)), workers=8)
    benchmark("CPU 密集", cpu_task, [10_000_000]*4, workers=4)
```

#### 预期结果

```
==================================================
I/O 密集（8 个任务，8 并发）
==================================================
  串行:    4.001s
  线程池:  0.530s  (加速 7.6x)
  进程池:  0.562s  (加速 7.1x)

==================================================
CPU 密集（4 个任务，4 并发）
==================================================
  串行:    0.935s
  线程池:  1.038s  (加速 0.9x)
  进程池:  0.301s  (加速 3.1x)
```

> 注意：实际数值因 CPU 核心数和负载而异，但**趋势不变**：I/O 密集→线程池≈进程池（线程池更快）；CPU 密集→进程池 >> 线程池。

## 逐行解释

### 为什么 I/O 密集时线程池能并行？

```python
def io_task(x):
    time.sleep(0.5)    # ← 关键在这里
```

`time.sleep()` 是一个系统调用（`nanosleep`），它让当前线程进入睡眠状态。CPython 在调用系统级 I/O 函数前会**释放 GIL**，让其他线程可以获取 GIL 执行。当 sleep 结束后，线程重新获取 GIL 继续。

同理，`requests.get()`、`socket.recv()`、`file.read()` 等 I/O 操作都会在阻塞时释放 GIL。

### 为什么 CPU 密集时线程池不能并行？

```python
def cpu_task(n):
    total = 0
    for i in range(n):     # ← 纯 Python 字节码循环
        total += i
```

这个循环不涉及任何 I/O，每条字节码指令执行前都需要持有 GIL。CPython 的 GIL 调度策略是：每执行 100 条字节码（或 `sys.getswitchinterval()` 时间到期）后释放 GIL 让其他线程执行。但因为是 CPU 密集，线程 A 释放 GIL 后线程 B 立即抢到，但 B 也只是在执行 Python 字节码，A 很快又抢回去——**来回切换反而有开销**，所以线程池不比串行快，甚至可能更慢。

### 为什么进程池能并行 CPU 任务？

每个进程有自己独立的 Python 解释器实例和 GIL。4 个进程的 4 个 GIL 互不影响，可以真正同时执行 Python 字节码。代价是：
- 进程创建有开销（fork 或 spawn）
- 进程间不共享内存，参数需要序列化（pickle）传输

### 为什么进程池的加速比不到 4x？

理论 4x，实际 3.5x，差额来自：
1. **进程创建开销**：4 个子进程需要 fork/spawn + 初始化 Python 解释器
2. **数据传输**：`pool.map` 要把参数 pickle 后通过管道发给子进程，结果也要 pickle 传回
3. **CPU 竞争**：操作系统调度、缓存失效等

数据量越大、计算越重，进程创建开销占比越小，加速比越接近核心数。

## 决策表

| 任务类型 | 特征 | 推荐 | 原因 |
|----------|------|------|------|
| I/O 密集 | 网络请求、文件读写、数据库查询 | 线程池 | GIL 在 I/O 时释放，线程能并行等待，开销比进程小 |
| CPU 密集 | 数学计算、图像处理、数据压缩 | 进程池 | 绕过 GIL，真正并行计算 |
| 混合型 | 先读数据再计算 | 进程池 or 拆分 | I/O 部分线程池够，计算部分要进程池 |
| 需要共享大量内存 | 多个任务共享大数组 | 线程池 + C 扩展 | 进程间共享内存复杂；C 扩展可用 `gil_scoped_release` 绕过 GIL |

---

## 常见错误

| 错误 | 原因 | 修复 |
|------|------|------|
| 进程池报 `PicklingError` | 传入的函数或参数不可序列化（lambda、闭包、自定义类未实现 `__reduce__`） | 用模块级函数，不用 lambda；参数用可 pickle 的类型 |
| 线程池跑 CPU 任务不加速 | GIL 阻止并行执行 Python 字节码 | 改用进程池，或用 C 扩展 + `gil_scoped_release` |
| 进程池在 Windows/macOS 上行为不同 | Windows 默认 spawn（重新导入模块），Linux 默认 fork | 确保模块顶层代码有 `if __name__ == "__main__":` 保护 |
| `max_workers` 设太高 | 进程/线程数超过 CPU 核心数，上下文切换开销增大 | CPU 密集：`max_workers = os.cpu_count()`；I/O 密集可适当多设 |
