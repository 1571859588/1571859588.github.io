## 一句话结论

用 `cProfile` 采样函数调用，用 `pstats` 按 `cumtime`（累计耗时）排序找瓶颈——先测量再优化，别凭直觉改代码。

---

## 题目

用 cProfile 分析一个有性能问题的脚本，找出累计耗时最高的三个函数，优化其中一个并重新测量对比。

---

## 知识点

| 概念 | 说明 |
|------|------|
| `cProfile` | Python 内置的性能分析器，记录每个函数的调用次数和耗时 |
| `pstats` | 解析 cProfile 输出的统计库，可按不同列排序 |
| `tottime` | 函数自身执行时间（不含子调用） |
| `cumtime` | 函数累计执行时间（含所有子调用）——**找瓶颈看这个** |
| `ncalls` | 函数被调用次数 |

---

## 一步步实现

### 第 1 步：写一个有性能问题的脚本

```python
# slow_script.py
"""一个故意写得很慢的脚本，用于 cProfile 分析"""
import time

def load_data(n):
    """模拟数据加载（慢：用列表拼接）"""
    result = []
    for i in range(n):
        result = result + [i]       # ← 瓶颈1：每次创建新列表，O(n²)
    return result

def process_data(data):
    """模拟数据处理（慢：用 += 拼接字符串）"""
    summary = ""
    for item in data:
        summary += str(item) + ","  # ← 瓶颈2：字符串拼接 O(n²)
    return summary

def save_data(summary):
    """模拟保存（正常快）"""
    time.sleep(0.1)                  # 模拟 I/O 等待
    with open("/tmp/output.txt", "w") as f:
        f.write(summary[:1000])      # 只写前 1000 字符

def main():
    data = load_data(5000)           # 加载 5000 个元素
    summary = process_data(data)     # 处理
    save_data(summary)               # 保存

if __name__ == "__main__":
    main()
```

### 第 2 步：用 cProfile 分析

```bash
# 方式一：命令行直接分析
uv run python -m cProfile -o profile.out slow_script.py

# 方式二：在 Python 里分析

```

### 第 4 步：解读结果

看 `cumtime` 列（累计耗时），从上到下：

| 排名 | 函数 | cumtime | 含义 |
|------|------|---------|------|
| 1 | `main()` | 1.834s | 总入口，等于所有子调用之和 |
| 2 | `process_data()` | 1.234s | **瓶颈 1**：字符串拼接 O(n²) |
| 3 | `load_data()` | 0.490s | **瓶颈 2**：列表拼接 O(n²) |
| 4 | `time.sleep()` | 0.100s | 模拟 I/O，无法优化 |

再看 `tottime`（自身耗时，不含子调用）：
- `{method 'join' of 'str' objects}` tottime=1.196s —— 字符串 `+=` 内部调用了 `join`，每次拼接都创建新字符串
- `load_data` 自身 tottime=0.026s，但 cumtime=0.490s —— 说明耗时在子调用（列表 `+` 运算）里

### 第 5 步：优化

```python
# fast_script.py
"""优化后的版本"""
import time

def load_data(n):
    """优化：用 append 代替列表拼接，O(n)"""
    result = []
    for i in range(n):
        result.append(i)             # ← O(1) 均摊
    return result

def process_data(data):
    """优化：用 join 代替 +=，O(n)"""
    parts = []
    for item in data:
        parts.append(str(item))      # 先收集
    return ",".join(parts)           # ← 一次性 join，O(n)

def save_data(summary):
    time.sleep(0.1)
    with open("/tmp/output.txt", "w") as f:
        f.write(summary[:1000])

def main():
    data = load_data(5000)
    summary = process_data(data)
    save_data(summary)

if __name__ == "__main__":
    main()
```

| 指标 | 优化前 | 优化后 | 提升 |
|------|--------|--------|------|

## 逐行解释

### pstats 输出各列含义

```
   ncalls  tottime  percall  cumtime  percall filename:lineno(function)
```

| 列 | 含义 | 优化时看哪列 |
|----|------|-------------|
| `ncalls` | 函数被调用次数 | 调用次数异常多→检查是否在循环里重复调用 |
| `tottime` | 函数自身耗时（不含子调用） | 找"哪个函数本身最慢"→看这列 |
| `percall` (第一个) | tottime / ncalls | 每次调用的平均自身耗时 |
| `cumtime` | 累计耗时（含所有子调用） | **找瓶颈→看这列**（从 main 往下看） |
| `percall` (第二个) | cumtime / ncalls | 每次调用的平均累计耗时 |

### 为什么 `result = result + [i]` 是 O(n²)？

```python
result = result + [i]    # 每次创建全新的列表
```

每次 `+` 都会：
1. 创建一个新列表
2. 把 `result` 的所有元素拷贝过去
3. 把 `[i]` 的元素追加

第 k 次循环拷贝 k 个元素，总共拷贝 0+1+2+...+n-1 = n(n-1)/2 次，即 O(n²)。

`result.append(i)` 是在原列表末尾追加，均摊 O(1)，总共 O(n)。

### 为什么 `summary += str(item) + ","` 是 O(n²)？

Python 字符串是不可变的。每次 `+=` 都会：
1. 创建新字符串
2. 拷贝左操作数的所有字符
3. 拷贝右操作数的所有字符

第 k 次循环拷贝约 k×（平均数字长度）个字符，总共 O(n²)。

`",".join(parts)` 只创建一次新字符串，一次性拷贝所有元素，总共 O(n)。

## 常见错误

| 错误 | 原因 | 修复 |
|------|------|------|
| 看了 `tottime` 而不是 `cumtime` | `tottime` 不含子调用，可能漏掉真正的瓶颈 | 先按 `cumtime` 排序看全局，再按 `tottime` 找具体函数 |
| 优化了不重要的函数 | 没按 `cumtime` 排序，改了耗时占比小的函数 | 按 `cumtime` 降序，从最大的开始优化 |
| cProfile 本身影响性能 | 分析器有开销，绝对时间不准 | **关注相对比例**和**优化前后对比**，不看绝对值 |
| 优化后没重新测量 | "我觉得变快了"不可靠 | 优化后重新跑 cProfile，对比 `cumtime` |
