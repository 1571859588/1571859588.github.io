# 练习 6：堆越界与 AddressSanitizer

> 关联笔记：`综合实战：跨层思维与自检练习.md` 5.2 练习 6
> 知识点：未定义行为（UB）、AddressSanitizer、内存安全
> 难度：入门

---

## 一句话结论

C++ 的数组越界是**未定义行为（UB）**——普通编译可能"正常运行"（静默写坏内存），但 AddressSanitizer 能在运行时当场捕获并报告精确的调用栈。

---

## 题目

故意制造堆越界（`vector` 下标越界），分别用普通编译和 ASan 编译运行，观察输出差异。

---

## 知识点

| 概念 | 说明 |
|------|------|
| 未定义行为（UB） | 标准不规定行为，编译器可以做任何事（崩溃、静默错误、看似正常） |
| `vector::operator[]` | 不做边界检查，越界是 UB |
| `vector::at()` | 做边界检查，越界抛 `std::out_of_range` |
| AddressSanitizer (ASan) | GCC/Clang 内置的内存错误检测器，编译时加 `-fsanitize=address` |
| ASan 工作原理 | 在内存周围插"红区"（poison zone），访问红区时报错 |

---

## 一步步实现

### 第 1 步：写有 bug 的程序

```cpp
// bug.cpp
#include <vector>
#include <iostream>

int main() {
    std::vector<int> v(4);          // 4 个元素，下标 0-3

    for (int i = 0; i <= 4; ++i) {  // ← bug：i 从 0 到 4（5 次），越界 1 次
        v[i] = i * 10;
        std::cout << "v[" << i << "] = " << v[i] << std::endl;
    }

    std::cout << "程序结束" << std::endl;
    return 0;
}
```

### 第 2 步：普通编译运行（UB 可能静默通过）

```bash
g++ -std=c++20 -g bug.cpp -o bug
./bug
```

#### 预期结果（可能的情况之一）

```
v[0] = 0
v[1] = 10
v[2] = 20
v[3] = 30
v[4] = 40
程序结束
```

**关键理解**：程序"正常"退出了！`v[4]` 写到了 vector 分配的堆内存之后的区域。这块内存可能是：
- vector 预留的额外空间（capacity > size 时）
- 其他堆分配的元数据
- 未使用的堆内存

UB 的可怕之处：**它不保证崩溃**。你的程序可能默默写坏了别的数据，在很久之后才表现出问题，或者永远不被发现。这就是为什么不能依赖"能跑就是对的"。

### 第 3 步：ASan 编译运行（当场捕获错误）

```bash
g++ -std=c++20 -g -fsanitize=address -fno-omit-frame-pointer bug.cpp -o bug_asan
./bug_asan
```

> `-g` 生成调试信息（ASan 报告里显示行号）
> `-fsanitize=address` 启用 ASan
> `-fno-omit-frame-pointer` 保留帧指针，让调用栈更准确

#### 预期结果

```

v[1] = 10
v[2] = 20
v[3] = 30

    #0 0x... in main /path/to/bug.cpp:8        ← 精确到行号！
    #1 0x... in __libc_start_main
    #2 0x... in _start

0x602000000030 is located 0 bytes to the right of 16-byte region [0x602000000030,0x602000000040)

    #0 0x... in operator new[]
```

### 第 4 步：对比两种结果

| 维度 | 普通编译 | ASan 编译 |
|------|----------|-----------|
| 是否检测到越界 | 否（UB 静默通过） | 是（当场报错并终止） |
| 程序是否继续执行 | 是（可能写坏了内存） | 否（ABORTING 终止） |
| 错误信息 | 无 | 精确到文件名和行号 |
| 性能 | 正常 | 慢 2-3x（内存和速度开销） |

### 第 5 步：修复——用 at() 替代 operator[]

```cpp
// fixed.cpp
#include <vector>
#include <iostream>

int main() {
    std::vector<int> v(4);

    for (int i = 0; i <= 4; ++i) {
        v.at(i) = i * 10;              // ← at() 做边界检查
        std::cout << "v[" << i << "] = " << v[i] << std::endl;
    }

    return 0;
}
```

```bash
g++ -std=c++20 -g fixed.cpp -o fixed
./fixed
```

#### 预期结果

```
v[0] = 0
v[1] = 10
v[2] = 20
v[3] = 30
terminate called after throwing an instance of 'std::out_of_range'
  what():  vector::_M_range_check: __n (which is 4) >= this->size() (which is 4)
Aborted (core dumped)
```

`at()` 在越界时抛 `std::out_of_range` 异常，可以 try-catch 处理，比 UB 安全得多。

## 逐行解释

### ASan 报告怎么读

```
==12345==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x602000000034
```
- `12345` = 进程 PID
- `heap-buffer-overflow` = 错误类型：堆缓冲区溢出
- `0x602000000034` = 被访问的非法地址

```
WRITE of size 4 at 0x602000000034 thread T0
    #0 0x... in main /path/to/bug.cpp:8
```
- `WRITE of size 4` = 写了 4 字节（int 的大小）
- `thread T0` = 主线程
- `bug.cpp:8` = 第 8 行，即 `v[i] = i * 10;`

```
0x602000000030 is located 0 bytes to the right of 16-byte region [0x602000000030,0x602000000040)
```
- `0 bytes to the right` = 紧挨着合法区域的右边
- `16-byte region` = vector 分配了 16 字节（4 个 int × 4 字节）
- 地址 `0x...34` = 第 16 字节之后的位置（4×4=16，下标 4 就是第 17-20 字节）

```
Shadow bytes around the buggy address:
  fa fa fd fd fa fa fd fd ...
  ↑
  fa = heap-right-redzone（ASan 插入的红区）
  fd = heap-left-redzone
```
ASan 在每块堆内存周围插入"红区"（poison zone），访问红区就报错。

### 为什么普通编译不崩溃？

`std::vector` 的 `operator[]` 等价于 `*(data_ptr + i)`，即指针运算 + 解引用。只要 `data_ptr + 4` 指向的地址是**可写的合法地址**（不是未映射页），就不会触发段错误。

vector 的堆分配通常会有额外的 capacity 空间（比如 size=4 但 capacity=4 或更多），越界 1 个元素可能恰好落在已分配但未使用的堆内存里，所以不崩溃。但如果越界更多，或者堆分配器的元数据被写坏，程序可能在后续操作中崩溃——而且崩溃点远离 bug 点，极难调试。

### ASan 的性能开销

| 维度 | 开销 |
|------|------|
| 内存 | 约 2-3x（每个 8 字节内存对应 1 字节 shadow memory） |
| 速度 | 约 2-3x 慢 |
| 适用 | 开发/测试环境，不要用于生产 |

---

## ASan 能检测的内存错误类型

| 错误类型 | 说明 | 示例 |
|----------|------|------|
| heap-buffer-overflow | 堆溢出 | `v[v.size()]` |
| stack-buffer-overflow | 栈溢出 | `int arr[4]; arr[4] = 0;` |
| use-after-free | 释放后使用 | `delete p; p->x = 1;` |
| double-free | 重复释放 | `delete p; delete p;` |
| memory leaks | 内存泄漏 | `new` 了不 `delete`（需加 `-fsanitize=leak`） |
| global-buffer-overflow | 全局变量溢出 | `int g[4]; g[5] = 0;` |

---

## 常见错误

| 错误 | 原因 | 修复 |
|------|------|------|
| ASan 程序运行不了 | 没装 libasan | `apt install libasan6`（Ubuntu）或 `g++ -fsanitize=address` 会自动链接 |
| ASan 报 `ASAN_OPTIONS` 警告 | 环境变量配置问题 | 按提示设置 `ASAN_OPTIONS=detect_leaks=1` |
| 生产环境开了 ASan | 性能开销大 | ASan 只用于开发/测试/CI，生产用普通编译 |
| 越界没崩溃就不管 | "能跑就行"心态 | UB 不保证崩溃，用 ASan 或 `at()` 提前发现 |
