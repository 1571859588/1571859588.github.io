# 练习 8：用 ldd 和 nm 分析 Python 原生扩展

> 关联笔记：`综合实战：跨层思维与自检练习.md` 5.3 练习 8
> 知识点：动态库依赖、符号表、Python C 扩展机制
> 难度：中等

---

## 一句话结论

Python 的 `.so` 文件本质上是一个动态库，用 `ldd` 看它依赖什么库，用 `nm -C` 看它导出了什么符号——每个 C 扩展必须导出一个 `PyInit_模块名` 入口函数。

---

## 题目

找一个 Python 原生扩展（如 NumPy 的核心库），用 `ldd` 分析其动态库依赖，用 `nm -C` 分析其导出符号。

---

## 知识点

| 工具 | 作用 | 关键选项 |
|------|------|----------|
| `ldd` | 查看动态库的依赖链 | `ldd file.so` |
| `nm` | 查看目标文件的符号表 | `-C` 解码 C++ 符号名，`-D` 查动态符号表 |
| `readelf -d` | 查看 ELF 文件的动态段 | `RPATH`/`RUNPATH`、`NEEDED` |
| `objdump` | 更强大的反汇编/分析工具 | `-T` 动态符号表 |

---

## 一步步实现

### 第 1 步：找到 Python 扩展的 .so 文件

```bash
# 找 NumPy 的核心扩展库
python -c "import numpy.core; print(numpy.core.__file__)"
# 输出类似：/home/user/.venv/lib/python3.11/site-packages/numpy/core/_multiarray_umath.cpython-311-x86_64-linux-gnu.so
```

> 如果没装 NumPy，先 `pip install numpy`。
> 文件名格式：`模块名.cpython-版本-平台.so`，例如 `cpython-311-x86_64-linux-gnu` 表示 Python 3.11 + x86_64 Linux。

为了方便后续操作，把路径存到变量：

```bash
SO_PATH=$(python -c "import numpy.core; print(numpy.core.__file__)")
echo "$SO_PATH"
```

### 第 2 步：用 ldd 查看动态库依赖

```bash
ldd "$SO_PATH"
```

#### 预期结果

```
        linux-vdso.so.1 (0x00007ffd23bf6000)
        libopenblas.so.0 => /home/user/.venv/lib/python3.11/site-packages/numpy/core/../libs/libopenblas64_p-r0-...so (0x00007f...)
        libm.so.6 => /lib/x86_64-linux-gnu/libm.so.6 (0x00007f...)
        libpthread.so.0 => /lib/x86_64-linux-gnu/libpthread.so.0 (0x00007f...)
        libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x00007f...)
        libstdc++.so.6 => /lib/x86_64-linux-gnu/libstdc++.so.6 (0x00007f...)
        libgcc_s.so.1 => /lib/x86_64-linux-gnu/libgcc_s.so.1 (0x00007f...)
        /lib64/ld-linux-x86-64.so.2 (0x00007f...)
        libgfortran.so.5 => /lib/x86_64-linux-gnu/libgfortran.so.5 (0x00007f...)
```

### 第 3 步：解读 ldd 输出

| 依赖库 | 含义 | 为什么要链接它 |
|--------|------|---------------|
| `libopenblas.so.0` | 线性代数库（BLAS/LAPACK） | NumPy 的矩阵运算底层用 OpenBLAS |
| `libm.so.6` | 数学库 | `sin`/`cos`/`sqrt` 等数学函数 |
| `libpthread.so.0` | POSIX 线程库 | OpenBLAS 多线程并行计算 |
| `libc.so.6` | C 标准库 | `malloc`/`free`/`memcpy` 等基础函数 |
| `libstdc++.so.6` | C++ 标准库 | NumPy 部分核心用 C++ 编写 |
| `libgcc_s.so.1` | GCC 运行时 | 异常处理、栈展开 |
| `libgfortran.so.5` | Fortran 运行时 | BLAS/LAPACK 原始用 Fortran 编写 |

> `linux-vdso.so.1` 是虚拟动态库（内核映射到每个进程的地址空间），不是真实文件。
> `/lib64/ld-linux-x86-64.so.2` 是动态链接器（loader），负责加载所有其他 .so。

### 第 4 步：用 nm 查看导出符号

```bash
# -D 查看动态符号表（运行时可见的符号）
# -C 解码 C++ 符号名（name demangling）
# grep ' T ' 只看 .text 段中有定义的导出符号
nm -CD "$SO_PATH" | grep ' T ' | head -30
```

#### 预期结果

```
00000000000a1234 T PyInit__multiarray_umath       ← Python 扩展入口！
00000000000b5678 T PyArray_New
00000000000b6abc T PyArray_NewFromDescr
00000000000c1234 T PyArray_FromAny
00000000000c2345 T PyArray_ContiguousFromObject
00000000000d3456 T PyArray_DIM
00000000000d4567 T PyArray_DATA
0000000000567890 T cblas_saxpy                    ← OpenBLAS 接口
00000000005678a0 T cblas_dgemm
...
```

### 第 5 步：重点看 PyInit 入口函数

```bash
# 每个 Python C 扩展必须导出 PyInit_模块名 函数
nm -CD "$SO_PATH" | grep PyInit
```

#### 预期结果

```
00000000000a1234 T PyInit__multiarray_umath
```

**为什么这个函数名是 `PyInit__multiarray_umath`（双下划线）？**

Python 的 C 扩展入口函数命名规则：`PyInit_` + 模块名。模块名是 `_multiarray_umath`（带前导下划线表示内部模块），所以入口函数是 `PyInit__multiarray_umath`（`PyInit_` + `_multiarray_umath` = 两个下划线）。

Python 导入 `.so` 文件时：
1. 从文件名解析模块名（去掉 `.cpython-311-...so` 后缀）
2. 在 .so 的符号表里找 `PyInit_模块名`
3. 调用这个函数初始化模块

### 第 6 步：用 readelf 查看动态段

```bash
# 查看 NEEDED（依赖的库）和 RUNPATH（搜索路径）
readelf -d "$SO_PATH" | grep -E 'NEEDED|RUNPATH|RPATH'
```

#### 预期结果

```
 0x0000000000000001 (NEEDED)   Shared library: [libopenblas.so.0]
 0x0000000000000001 (NEEDED)   Shared library: [libm.so.6]
 0x0000000000000001 (NEEDED)   Shared library: [libpthread.so.0]
 0x0000000000000001 (NEEDED)   Shared library: [libc.so.6]
 0x0000000000000001 (NEEDED)   Shared library: [libstdc++.so.6]
 0x0000000000000001 (NEEDED)   Shared library: [libgcc_s.so.1]
 0x0000000000000001 (NEEDED)   Shared library: [libgfortran.so.5]
 0x000000000000001d (RUNPATH)  Library runpath: [/home/user/.venv/lib/python3.11/site-packages/numpy/core/../libs]
```

**RUNPATH** 告诉动态链接器去哪里找 `libopenblas.so.0`——NumPy 把自己的 OpenBLAS 库打包在 `numpy/core/../libs/` 目录里，通过 RUNPATH 让 .so 能自动找到它，不需要设置 `LD_LIBRARY_PATH`。

### 第 7 步：对比——看一个 pybind11 扩展

如果你做了练习 7 或综合实战的 `fast_scale`：

```bash
# 找到 fast_scale 的 .so
ls build/fast_scale*.so

ldd build/fast_scale*.so
#         libpython3.11.so.1.0 => /lib/x86_64-linux-gnu/libpython3.11.so.1.0 (0x...)
#         libstdc++.so.6 => /lib/x86_64-linux-gnu/libstdc++.so.6 (0x...)
#         libgcc_s.so.1 => /lib/x86_64-linux-gnu/libgcc_s.so.1 (0x...)
#         libc.so.6 => /lib/x86_64-linux-gnu/libc.so.6 (0x...)
#         /lib64/ld-linux-x86-64.so.2 (0x...)

nm -CD build/fast_scale*.so | grep ' T '
# 0000000000001234 T PyInit_fast_scale           ← pybind11 的入口
# 0000000000002345 T pybind11_init...
```

对比 NumPy 和 fast_scale：
- NumPy 依赖 OpenBLAS/Fortran（因为做数值计算）
- fast_scale 只依赖 libpython + C++ 运行时（因为只做简单缩放）
- 两者都必须导出 `PyInit_模块名`

## nm 符号类型速查

| 符号类型 | 含义 |
|----------|------|
| `T` | .text 段，有定义的全局导出符号（函数） |
| `t` | .text 段，局部符号（static 函数） |
| `D` | .data 段，已初始化的全局变量 |
| `B` | .bss 段，未初始化的全局变量 |
| `U` | 未定义符号（需要从其他库链接进来） |
| `W` | 弱符号（可被覆盖） |

```bash
# 查看未定义符号（需要从外部链接的）
nm -CD "$SO_PATH" | grep ' U ' | head -20
# 输出示例：
#                  U PyFloat_AsDouble         ← 需要 libpython 提供
#                  U PyList_New               ← 同上
#                  U malloc                   ← 需要 libc 提供
#                  U cblas_sgemm              ← 需要 libopenblas 提供
```

## 常见错误

| 错误 | 原因 | 修复 |
|------|------|------|
| `ldd: exited with unknown exit code (135)` | .so 文件架构不匹配（如在 x86 上看 ARM 的 .so） | 确认文件架构和系统一致 |
| `nm: format not recognized` | 文件不是 ELF 格式（可能是 Windows 的 .pyd） | Windows 用 `dumpbin /exports file.pyd` |
| `not a dynamic executable` | 文件是静态链接的，没有动态依赖 | 用 `nm` 而非 `ldd` |
| 找不到 `.so` 文件 | 模块是纯 Python（没有 C 扩展） | 换一个有 C 扩展的模块（如 numpy、torch） |
