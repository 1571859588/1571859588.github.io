# 1.2 多机间通信：InfiniBand 与 RoCE

## 一句话结论

单机 8 卡用 NVLink，跨机用 InfiniBand（IB）或 RoCE。RDMA 是核心——绕过 CPU 直达显存，延迟从 50-100μs 降到 1-2μs。⚠️ WSL 通常没有 IB 网卡，`ibstat` 跑不了，只能在真实服务器看；GPUDirect RDMA 需 root 权限，无 sudo 不可用。对上层 PyTorch 代码来说 IB 和 RoCE 完全透明，NCCL 自动选择。

## 背景与定位

单机 8 卡是 AI 训练的基本单元，但要训练百亿乃至千亿参数的大模型，往往需要几十甚至上百台机器组成集群。这时候机器之间的网络互联就成了关键。本篇承接 1.1 的单机互联，讲跨机网络，为后续理解 NCCL 传输后端和跨机并行策略打基础。

## ⚠️ 环境与 sudo 权限说明

| 命令/操作 | 权限 | 说明 |
|-----------|:----:|------|
| `ibstat` | ⚠️ WSL 不可用 | 需真实 IB 网卡，WSL 通常没有 |
| `ibv_devinfo` | ⚠️ WSL 不可用 | 同上 |
| `show_gids` | ⚠️ WSL 不可用 | 同上 |

**替代方案**：WSL 下跑不了 IB 命令，本篇贴出标准 `ibstat` 输出供你对照阅读，理解概念即可。可以用 `ip link show` 查看普通以太网卡作为可跑的替代实验。

## 核心概念（新人友好讲解）

### 1. 传统以太网的局限

普通以太网（TCP/IP）虽然成本低、无处不在，但用在大模型训练中有几个硬伤：

> **生活类比**：TCP/IP 就像两家公司之间**寄快递**——要经过前台收件、登记入库、安检分拣等一系列流程，包裹在各个环节排队等候。

**以太网三大硬伤**：

| 硬伤 | 说明 | 影响 |
|------|------|------|
| 延迟高 | 数据要经过操作系统内核的协议栈处理 | 端到端延迟 50-100 微秒 |
| CPU 开销大 | 数据拷贝和协议处理占用 CPU | 挤占数据预处理算力 |
| 有效带宽受限 | 协议头开销导致带宽打不满 | 100GbE 硬件实际可用 < 100Gb/s |

大模型训练的梯度同步是高频操作（每个 training step 都要执行），高延迟和低带宽会成为训练速度的严重拖累。

### 2. RDMA：绕过 CPU 直达显存

RDMA（Remote Direct Memory Access，远程直接内存访问）是解决上述问题的核心技术。

> **生活类比**：RDMA 就像两栋楼之间架了一条**专用传送带**——物品直接从 A 楼仓库传到 B 楼仓库，谁都不用经手，不用登记不用安检。

```
传统 TCP/IP 数据路径：
  GPU显存 → CPU内存(拷贝1) → 内核协议栈(处理) → CPU内存(拷贝2) → 网卡 → 网络
                                  ↑ CPU 参与全程

RDMA 数据路径：
  GPU显存 → 网卡 → 网络 → 网卡 → GPU显存
                    ↑ CPU 完全不参与
```

**RDMA 三大优势**：

| 优势 | 说明 | 对比 TCP/IP |
|------|------|------------|
| 零拷贝（Zero-Copy） | 数据不需要在用户态和内核态之间来回搬运 | TCP/IP 要拷贝 2 次 |
| 内核旁路（Kernel Bypass） | 绕过操作系统内核，消除上下文切换 | TCP/IP 要经过完整协议栈 |
| 微秒级延迟 | 端到端延迟 1-2 微秒 | 比 TCP/IP 快 50-100 倍 |

### 3. InfiniBand：AI 集群的主流选择

InfiniBand（简称 IB）**无限带宽互联技术**是当前大规模 AI 训练集群中最主流的高性能网络方案，原生支持 RDMA。

#### IB 速率代际演进（新人必背）

| 标准 | 单链路速率 | 4x 带宽（常用配置） | 等效字节带宽 | 典型应用 |
|------|:---------:|:------------------:|:-----------:|---------|
| HDR | 50 Gb/s | 200 Gb/s | 25 GB/s | A100 集群 |
| NDR | 100 Gb/s | 400 Gb/s | 50 GB/s | H100 集群 |
| XDR | 200 Gb/s | 800 Gb/s | 100 GB/s | B200 集群 |

**新人参数讲解**：
- **"4x"是什么意思**：IB 网卡可以聚合多条物理通道（1x/4x/12x），4x 是最常用的配置，把 4 条通道绑在一起提升带宽。
- **"Gb/s vs GB/s"换算**：1 Byte = 8 bits，所以 200 Gb/s ÷ 8 = 25 GB/s。带宽用 bit 标称是网络行业惯例，但算实际数据搬运时间要用 Byte。
- **目前主流**：大型 AI 集群普遍使用 NDR 400Gb/s 或更新的 XDR 800Gb/s。

#### IB 的关键特点

- 原生 RDMA 支持，协议栈效率极高
- 拥塞控制由**硬件**完成，大规模集合通信场景表现稳定
- 需要专用交换机和网卡（HCA），成本较高
- NVIDIA（收购 Mellanox 后）是主要供应商，与 GPU 生态深度绑定

### 4. RoCE：以太网上的 RDMA

RoCE（RDMA over Converged Ethernet）让标准以太网硬件也能支持 RDMA，是 InfiniBand 之外的另一个选择。

| 版本 | 协议层 | 路由能力 | 状态 |
|------|--------|---------|:----:|
| RoCE v1 | 以太网二层（链路层） | 不可跨子网路由 | 已淘汰 |
| RoCE v2 | UDP/IP 三层（网络层） | 可路由，更通用 | ✅ 主流 |

> **新人参数讲解**：
> - **"二层 vs 三层"**：二层（链路层）只能在一个子网内通信，三层（网络层）可以跨子网路由。RoCE v2 封装在 UDP/IP 包里，能像普通网络流量一样路由，部署更灵活。
> - **现在说 RoCE 默认指 v2**，v1 基本已淘汰。

### 5. InfiniBand 与 RoCE v2 选型对比

| 维度 | InfiniBand | RoCE v2 |
|------|-----------|---------|
| 延迟 | ~1 μs | ~2-5 μs |
| 带宽 | 400-800 Gb/s | 100-400 Gb/s |
| 拥塞控制 | 硬件原生实现 | 依赖 ECN/PFC 软件配置 |
| 成本 | 高（专用设备） | 中（复用以太网基础设施） |
| 大规模稳定性 | 成熟 | 需要仔细调优 |
| 适用用户 | 大型 AI Lab、万卡集群 | 云厂商、中小规模集群 |

**选型建议**：
- 追求极致性能和大规模稳定性（万卡训练）→ **InfiniBand**
- 预算受限或需要复用现有网络 → **RoCE v2**
- 两者对上层训练代码（PyTorch / DeepSpeed）完全透明，NCCL 会自动选择可用的传输方式

### 6. GPUDirect RDMA（新人拓展）

> **新人参数讲解**：GPUDirect RDMA 是让 GPU 显存直接通过网卡传输，不经过 CPU 内存中转的技术。没有它，数据路径是 GPU显存 → CPU内存 → 网卡 → 网络；有了它，直接 GPU显存 → 网卡 → 网络，省一次拷贝。

⚠️ **GPUDirect RDMA 需要 root 权限加载内核模块**（如 `nvidia-peermem`），无 sudo 的环境不可用。NCCL 会自动检测，如果不可用则回退到经过 CPU 内存中转的路径（性能略低但能跑）。

## 动手实践

### 实验 1：查看 IB 网卡状态（⚠️ WSL 不可用，贴标准输出）

```bash
# 查看 IB 网卡基本信息
ibstat
```

**预期输出**（标准 H100 集群的 Mellanox ConnectX-7 网卡）：

```
CA 'mlx5_0'
    CA type: MT4129
    Number of ports: 1
    Firmware version: 28.39.1002
    Hardware version: 0
    Node GUID: 0xa0b8d30f00e8b8a0
    System image GUID: 0xa0b8d30f00e8b8a0
    Port 1:
        State: Active
        Physical state: LinkUp
        Rate: 400               # ← NDR 400 Gb/s
        Base lid: 1
        LMC: 0
        SM lid: 1
        Capability mask: 0x9edc8a44
        Capability mask Extended: 0x6c68e0
        Port GUID: 0xa2b8d30f00e8b8a1
        Link layer: InfiniBand
```

**如何读懂这个输出**：
1. `CA 'mlx5_0'`：网卡设备名（NCCL 环境变量 `NCCL_IB_HCA=mlx5_0` 就填这个）
2. `State: Active`：网卡已激活，可以正常工作
3. `Rate: 400`：当前速率为 400 Gb/s（NDR）
4. `Link layer: InfiniBand`：链路层是 IB（如果是 RoCE 会显示 Ethernet）

## 面试回答（30 秒口述版）

> 跨机通信主流用 InfiniBand 或 RoCE，核心都是 RDMA 技术——绕过 CPU 内核直接访问远端内存，延迟从 TCP/IP 的 50-100 微秒降到 1-2 微秒。IB 是专用网络，NDR 400Gb/s 带宽，硬件拥塞控制，万卡集群首选但成本高；RoCE v2 在以太网上跑 RDMA，可路由、成本低，中小规模集群常用。两者对上层透明，NCCL 自动选择。GPUDirect RDMA 让 GPU 显存直接走网卡不经 CPU 内存，但需要 root 权限加载内核模块。选型上追求极致性能选 IB，预算受限选 RoCE。

## 深入追问

**Q1：RDMA 为什么能绕过 CPU？**
RDMA 网卡（RNIC）内置了完整的协议处理引擎，可以直接访问注册的内存区域。应用程序提前把内存区域注册给网卡（memory registration），网卡通过 DMA 直接读写这块内存，CPU 只在发起和完成时收到通知，中间全程不参与。

**Q2：IB 和 RoCE 在上层代码层面有区别吗？**
没有。PyTorch/NCCL 通过统一的接口调用，NCCL 自动探测可用传输方式。开发者写 `dist.init_process_group(backend="nccl")` 即可，不需要关心底层是 IB 还是 RoCE。区别只在运维层面（网络配置、拥塞控制调优）。

**Q3：为什么 RoCE 延迟比 IB 高？**
RoCE 要在以太网帧里封装 RDMA 数据，经过以太网交换机时可能有缓冲和调度延迟。IB 是专用协议，交换机硬件直接处理，延迟更低。另外 RoCE 的拥塞控制依赖 PFC（Priority Flow Control）和 ECN，配置不当会导致重传，增加延迟。

**Q4：4x 是什么意思？为什么不用 12x？**
"4x"指聚合 4 条物理通道。4x 是成本和带宽的平衡点——更多通道需要更多线缆和交换机端口，成本急剧上升。12x 存在但极少用，因为 4x NDR 已经达到 400 Gb/s，对大多数场景够用。

**Q5：没有 IB 网卡，多机训练能跑吗？**
能跑，但性能差。NCCL 会回退到 TCP Socket 传输，延迟高、带宽低。对于小规模实验（2-4 机）可以接受，大规模训练必须用 IB 或 RoCE。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|---------|
| RDMA vs TCP/IP | RDMA 绕过内核，TCP/IP 经过内核 | 传送带 vs 寄快递 |
| IB vs RoCE | IB 专用网络硬件，RoCE 复用以太网 | 专线 vs 普通路上跑快车 |
| RoCE v1 vs v2 | v1 二层不可路由，v2 三层可路由 | v2 是主流，v1 已淘汰 |
| HDR/NDR/XDR | 200/400/800 Gb/s | A100用HDR，H100用NDR，B200用XDR |
| Gb/s vs GB/s | 1 GB/s = 8 Gb/s | 网络标 bit，存储标 Byte |
| GPUDirect RDMA vs 普通 RDMA | GPUDirect 是 GPU显存直传，普通是 CPU内存直传 | GPUDirect 省 GPU↔CPU 拷贝 |

## 常见报错速查表

| ❌ 现象/报错 | 原因 | ✅ 解决 |
|-------------|------|---------|
| `ibstat: command not found` | 系统未安装 IB 工具包 | `conda install -c conda-forge rdma-core`（✅ 无需 sudo）；或本篇对照阅读 |
| WSL 下看不到 mlx5 网卡 | WSL 虚拟化不暴露 IB 硬件 | 正常，在真实服务器验证；用 `ip link` 看以太网卡 |
| NCCL 日志显示 `NET/IB` 未找到 | 无 IB 网卡或驱动未装 | NCCL 自动回退到 Socket；不影响单机训练 |
| NCCL 日志 `GPUDirect RDMA disabled` | 无 root 权限加载 peermem 模块 | NCCL 自动回退到经 CPU 内存路径；性能略低但能跑 |
| RoCE 网络丢包导致训练卡顿 | PFC/ECN 拥塞控制配置不当 | 需网络运维调整交换机配置（超出个人权限范围） |
