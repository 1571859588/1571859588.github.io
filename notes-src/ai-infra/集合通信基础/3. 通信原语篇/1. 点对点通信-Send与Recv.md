## 一句话结论

Send/Recv 是最基础的通信模式——一个进程发，一个进程收，就像两个人**打电话**，一对一、定向传递。✅ 单机用 `torchrun --nproc_per_node=2` 即可跑通示例，无需 sudo。流水线并行（PP）中 micro-batch 激活值的跨机传递就靠它。当需要"多方同时协作"时，才升级到集合通信原语。

## 背景与定位

在进入集合通信原语（下一节）之前，先理解最基础的通信模式：点对点通信。它是所有分布式通信的"原子操作"，理解了它，再看集合通信就是"多个点对点的组合优化"。

## ⚠️ 环境与 sudo 权限说明

| 命令/操作 | 权限 | 说明 |
|-----------|:----:|------|
| `torchrun --nproc_per_node=2` | ✅ 无需 sudo | 单机多进程，WSL 可跑 |
| `dist.send` / `dist.recv` | ✅ 无需 sudo | PyTorch API，单机可跑 |
| 跨机点对点 | ⚠️ 需配置 | 需 SSH 免密 + 多机环境 |

**核心说明**：单机双进程完全可以跑通本篇示例，理解概念足够。跨机需要多台机器+SSH 配置，但 API 代码完全一样。

## 核心概念（新人友好讲解）

### 1. Send/Recv 语义

点对点通信就是**一个进程发送数据，另一个进程接收**。

> **生活类比**：就像两个人**打电话**——只有发送方和接收方两个人参与，定向传递，不涉及第三方。

```
rank 0 (发送方)                    rank 1 (接收方)
    │                                  │
    │──── dist.send(tensor, dst=1) ───▶│
    │                                  │  dist.recv(tensor, src=0)
    │                                  │
```

与集合通信（AllReduce 等）的区别：
- 点对点：**一对一**，只有两个进程参与
- 集合通信：**多对多**，所有进程同时参与

### 2. torch.distributed API

PyTorch 通过 `torch.distributed` 模块提供点对点通信 API：

| API | 作用 | 阻塞特性 |
|-----|------|---------|
| `dist.send(tensor, dst)` | 发送张量到目标 rank | 阻塞（等对方收完） |
| `dist.recv(tensor, src)` | 从源 rank 接收张量 | 阻塞（等数据到） |
| `dist.isend(tensor, dst)` | 异步发送 | 非阻塞，返回 work 对象 |
| `dist.irecv(tensor, src)` | 异步接收 | 非阻塞，返回 work 对象 |

**新人参数讲解**：
- **`dst`（destination）**：目标进程的 rank 编号。`dst=1` 表示发给 rank 1。
- **`src`（source）**：源进程的 rank 编号。`src=0` 表示从 rank 0 收。
- **`rank`**：进程在通信组中的编号，从 0 开始。`dist.get_rank()` 获取自己的 rank。
- **`world_size`**：总进程数。`dist.get_world_size()` 获取。

### 3. 同步 vs 异步发送

| 模式 | API | 行为 | 适用场景 |
|------|-----|------|---------|
| 同步（阻塞） | `dist.send` | 发送方等接收方收完才返回 | 简单、可靠，适合学习 |
| 异步（非阻塞） | `dist.isend` | 立即返回 work 对象，后台发送 | 性能优化，配合 `work.wait()` |

> **新人建议**：先学同步 `send/recv` 理解概念，实际工程中用异步 `isend/irecv` 提升 Overlap。

### 4. 与集合通信的分工

| 通信模式 | 典型 API | 适用并行策略 | 原因 |
|---------|---------|------------|------|
| 点对点 | `send/recv` | 流水线并行（PP） | PP 只需相邻 stage 传激活值，一对一 |
| 集合通信 | `all_reduce/all_gather` | 数据并行（DP）、张量并行（TP） | 需要所有卡同步梯度/参数，多对多 |

> 💡 **关键点**：PP 用点对点是因为每个 stage 只需把激活值传给下一个 stage，不需要所有卡都参与。DP/TP 用集合通信是因为梯度/参数需要所有卡同步。

## 动手实践

### 实验 1：单机双进程 Send/Recv（✅ 无需 sudo）

将以下代码保存为 `p2p_demo.py`：

```python
import os
import torch
import torch.distributed as dist

def setup(rank, world_size):
    # ⚠️ 不要手动设 MASTER_ADDR / MASTER_PORT！
    # torchrun 会自动设置这些环境变量并启动 TCPStore。
    # 手动覆盖会导致进程连错端口，等 600 秒超时报错：
    #   "The client socket has timed out after 600000ms while trying to connect to (localhost, 12355)"
    torch.cuda.set_device(rank)  # ← 必须在 init_process_group 之前！NCCL 初始化时就要绑定 GPU
    dist.init_process_group(backend="nccl", rank=rank, world_size=world_size)

def cleanup():
    dist.destroy_process_group()

def main():
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    if rank == 0:
        # rank 0 发送数据
        tensor = torch.tensor([1.0, 2.0, 3.0]).cuda()
        print(f"[rank {rank}] 发送前: {tensor.cpu().tolist()}")
        dist.send(tensor, dst=1)
        print(f"[rank {rank}] 发送完成")
    else:
        # rank 1 接收数据
        tensor = torch.zeros(3).cuda()
        print(f"[rank {rank}] 接收前: {tensor.cpu().tolist()}")
        dist.recv(tensor, src=0)
        print(f"[rank {rank}] 接收后: {tensor.cpu().tolist()}")

if __name__ == "__main__":
    rank = int(os.environ.get("RANK", 0))
    world_size = int(os.environ.get("WORLD_SIZE", 1))
    setup(rank, world_size)
    main()
    cleanup()
```

> ⚠️ **torchrun 与环境变量的关系**（踩坑高发区）：
>
> `torchrun` 启动时会**自动设置** `MASTER_ADDR`、`MASTER_PORT`、`RANK`、`WORLD_SIZE`、`LOCAL_RANK` 等环境变量，并在指定端口启动 TCPStore 作为进程发现的"会合点"。
>
> | 写法 | 效果 | 与 torchrun 兼容？ |
> |------|------|:---:|
> | `os.environ["MASTER_PORT"] = "12355"` | **强制覆盖** torchrun 设的值 | ❌ 进程连错端口，600 秒超时 |
> | `os.environ.setdefault("MASTER_PORT", "12355")` | 只在未设时才设，不覆盖 | ✅ 安全 |
> | （不写这行） | 完全交给 torchrun | ✅ 最推荐 |
>
> 简记：**用 torchrun 就别碰 MASTER_ADDR / MASTER_PORT**，让它自己管。

**运行命令**（需要至少 2 张 GPU，单 GPU 可改 backend 为 gloo 用 CPU 跑）：

```bash
# 方式1: 用 torchrun 启动（推荐，✅ 无需 sudo）
torchrun --nproc_per_node=2 p2p_demo.py
```

如果只有 1 张 GPU，改用 gloo 后端在 CPU 上跑：

```bash
# 方式2: 单卡用 gloo 后端（CPU 通信，✅ 无需 sudo）
# 把代码里 backend="nccl" 改成 backend="gloo"，.cuda() 改成 .cpu()
torchrun --nproc_per_node=2 p2p_demo.py
```

**预期输出**：

```
[rank 0] 发送前: [1.0, 2.0, 3.0]
[rank 1] 接收前: [0.0, 0.0, 0.0]
[rank 0] 发送完成
[rank 1] 接收后: [1.0, 2.0, 3.0]
```

> 💡 **解读**：rank 0 的 `[1,2,3]` 通过 `dist.send` 传到了 rank 1，rank 1 原来的 `[0,0,0]` 被覆盖成了 `[1,2,3]`。这就是点对点通信。

### 实验 2：异步 Send/Recv（✅ 无需 sudo）

```python
import os
import torch
import torch.distributed as dist

def main():
    rank = dist.get_rank()

    if rank == 0:
        tensor = torch.tensor([10.0, 20.0]).cuda()
        # 异步发送，立即返回 work 对象
        work = dist.isend(tensor, dst=1)
        print(f"[rank {rank}] isend 已提交，做其他事...")
        work.wait()  # 等待实际发送完成
        print(f"[rank {rank}] isend 完成")
    else:
        tensor = torch.zeros(2).cuda()
        work = dist.irecv(tensor, src=0)
        print(f"[rank {rank}] irecv 已提交，做其他事...")
        work.wait()  # 等待数据到达
        print(f"[rank {rank}] irecv 完成，收到: {tensor.cpu().tolist()}")

if __name__ == "__main__":
    # setdefault 只在环境变量不存在时才设值，不会覆盖 torchrun 已设的值（安全）
    # 对比：os.environ["MASTER_ADDR"] = "localhost" 会强制覆盖，导致 torchrun 的 TCPStore 连不上
    os.environ.setdefault("MASTER_ADDR", "localhost")
    os.environ.setdefault("MASTER_PORT", "12356")
    rank = int(os.environ.get("RANK", 0))
    world_size = int(os.environ.get("WORLD_SIZE", 1))
    torch.cuda.set_device(rank)  # ← 必须在 init_process_group 之前！
    dist.init_process_group(backend="nccl", rank=rank, world_size=world_size)
    main()
    dist.destroy_process_group()
```

**运行**：

```bash
torchrun --nproc_per_node=2 p2p_async_demo.py
```

**预期输出**：

```
[rank 0] isend 已提交，做其他事...
[rank 1] irecv 已提交，做其他事...
[rank 0] isend 完成
[rank 1] irecv 完成，收到: [10.0, 20.0]
```

> 💡 **异步的价值**：`isend`/`irecv` 提交后立即返回，你可以在 `work.wait()` 之前做其他计算，实现通信与计算的 Overlap。这就是流水线并行能把通信藏进计算里的基础。

## 面试回答（30 秒口述版）

> 点对点通信是最基础的通信模式，Send/Recv 实现一对一的数据定向传递。PyTorch 提供 `dist.send/recv`（同步）和 `dist.isend/irecv`（异步）两套 API。流水线并行用点对点传递 micro-batch 的激活值，因为每个 stage 只需传给下一个 stage；而数据并行和张量并行用集合通信，因为需要所有卡同步。异步 API 配合 `work.wait()` 可以实现通信与计算的 Overlap。

## 深入追问

**Q1：send 和 recv 必须配对吗？**
是的。`dist.send` 必须有对应的 `dist.recv`，否则发送方会一直阻塞等待。如果只有 send 没有 recv，会死锁。

**Q2：send 的 tensor 和 recv 的 tensor 大小必须一致吗？**
必须一致。发送方发 3 个元素，接收方也必须准备 3 元素的 tensor。类型也要匹配（都用 float32）。

**Q3：isend 提交后修改 tensor 会怎样？**
会出问题。`isend` 提交后到 `work.wait()` 返回前，tensor 的底层内存正在被读取发送，此时修改会导致发送的数据不一致。必须在 `wait()` 之后才能修改。

**Q4：点对点通信能跨机吗？**
能。底层走 NCCL（GPU 间）或 gloo（CPU 间），跨机时自动走网络。API 代码完全一样，只是 `MASTER_ADDR` 要设成主机的 IP。

**Q5：PP 为什么不用集合通信？**
PP 的每个 stage 只需把激活值传给相邻的下一个 stage，是一对一的关系。用集合通信（如 AllGather）会让所有卡都收到数据，浪费带宽。点对点更精准。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|---------|
| send vs isend | send 阻塞，isend 非阻塞 | i=immediate(立即返回) |
| recv vs irecv | recv 阻塞，irecv 非阻塞 | 同上 |
| 点对点 vs 集合通信 | 一对一 vs 多对多 | 打电话 vs 全班对答案 |
| src vs dst | src=来源, dst=目标 | s=source, d=destination |
| rank vs world_size | rank=自己的编号, world_size=总数 | 学号 vs 班级人数 |

## 常见报错速查表

| ❌ 现象/报错 | 原因 | ✅ 解决 |
|-------------|------|---------|
| **torchrun 启动后卡住不动**（无任何 print 输出） | `torch.cuda.set_device(rank)` 写在 `init_process_group()` 后面，NCCL 初始化时两个进程都绑 GPU 0 导致死锁 | **把 `set_device` 移到 `init_process_group` 之前**；也可加 `NCCL_DEBUG=INFO` 环境变量看日志定位 |
| **`socket timed out after 600000ms` 连接 (localhost, 12355) 超时** | 代码用 `os.environ["MASTER_PORT"] = "12355"` **覆盖**了 torchrun 自动设的端口，进程去连一个没有 TCPStore 的端口 | **删掉手动设 `MASTER_ADDR`/`MASTER_PORT` 的行**，torchrun 自动处理；如需保留备用，用 `setdefault()` 而非直接赋值 |
| 程序卡住不动（已有 print 输出） | send/recv 没配对，死锁 | 检查 rank 0 发给 rank 1，rank 1 必须有对应 recv |
| `RuntimeError: CUDA error` | 单卡跑 nccl 后端 | 改用 `backend="gloo"` + `.cpu()` |
| tensor 大小不匹配 | send 和 recv 的 tensor shape 不同 | 确保两边 shape 和 dtype 一致 |
| `Connection refused` | 跨机时 MASTER_ADDR/PORT 配错 | 确认主节点 IP 可达、端口未被占用 |
| isend 后修改 tensor | wait() 前修改了正在发送的数据 | 在 `work.wait()` 之后再操作 tensor |
