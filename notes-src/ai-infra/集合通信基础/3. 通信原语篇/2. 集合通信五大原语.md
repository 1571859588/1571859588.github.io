## 一句话结论

集合通信有五大核心原语：**Broadcast**（一对多）、**Reduce**（多对一）、**AllReduce**（全归约，所有人拿到相同结果）、**AllGather**（全收集）、**ReduceScatter**（归约分片）。✅ 全部可用 `torchrun --nproc_per_node=2` 单机跑通。最重要的一句话：**AllReduce = ReduceScatter + AllGather**，这一分解正是 Ring 8AllReduce 算法的核心思想。

## 背景与定位

分布式训练的本质是让多张 GPU 协同完成一个大任务。GPU 之间的数据交换不是随意的，而是遵循一套标准化的集合通信原语。掌握这些原语，就能理解各种并行策略中"数据到底怎么流动"。

本篇是通信原语篇的核心，后续的 Ring AllReduce（3.1）、通信 Overlap（3.3）、并行策略分析（5.1）都建立在这些原语之上。

## 核心概念（新人友好讲解）

### 1. Broadcast（广播）：一对多发送

一个 GPU 把数据发给所有其他 GPU。

> **生活类比**：老师把同一份试卷发给全班同学——一个人发，所有人收。

```
操作前:              Broadcast 后:
GPU0: [ABCD]         GPU0: [ABCD]  ← 发送方
GPU1: [    ]         GPU1: [ABCD]  ← 收到
GPU2: [    ]         GPU2: [ABCD]  ← 收到
GPU3: [    ]         GPU3: [ABCD]  ← 收到
```

**典型用途**：分发初始模型参数（rank 0 把模型广播到所有卡）。

### 2. Reduce（归约）：多对一聚合

所有 GPU 的数据按元素做聚合运算（通常是求和），结果汇总到一个 GPU 上。

> **生活类比**：课代表把全班同学的作业收齐后，把每道题的答案加起来汇总到一张表上——所有人交，一个人汇总。

```
操作前:              Reduce(sum) 到 GPU0:
GPU0: [1,2,3]        GPU0: [10,20,30]  ← 汇总结果
GPU1: [2,4,6]        GPU1: [2,4,6]     ← 未变
GPU2: [3,6,9]        GPU2: [3,6,9]     ← 未变
GPU3: [4,8,12]       GPU3: [4,8,12]    ← 未变
```

> 💡 **"归约"是什么意思**：把多个数据通过某种运算（加/乘/取最大值）合并成一个。`ReduceOp.SUM` 是最常用的，即求和。

**典型用途**：Loss 汇总（各卡算出 loss 后求和到 rank 0）。

### 3. AllReduce（全归约）：所有人聚合后所有人都拿到结果

等价于 Reduce + Broadcast，但实际实现更高效。

> **生活类比**：全班同学先把各自的答案汇总求平均，再把最终答案发给**每个人**——所有人参与，所有人都拿到结果。

```
操作前:              AllReduce(sum) 后:
GPU0: [1,2,3]        GPU0: [10,20,30]  ← 全部相同
GPU1: [2,4,6]        GPU1: [10,20,30]  ← 全部相同
GPU2: [3,6,9]        GPU2: [10,20,30]  ← 全部相同
GPU3: [4,8,12]       GPU3: [10,20,30]  ← 全部相同
```

**核心用途**：**数据并行训练中同步梯度**——每张卡计算各自的梯度，AllReduce 后所有卡得到相同的梯度均值。

> 📌 **这是分布式训练最常用的原语**，没有之一。DDP 每个 step 都要执行一次 AllReduce 来同步梯度。

### 4. AllGather（全收集）：每人拿一片，最后人人拿全部

每个 GPU 各持有一部分数据，收集后每人都拥有完整数据。

> **生活类比**：拼图游戏——每人手里有一块拼图，大家把各自的拼图收集到一起，每个人都得到完整的拼图。

```
操作前:              AllGather 后:
GPU0: [A]            GPU0: [A,B,C,D]  ← 全部完整
GPU1: [B]            GPU1: [A,B,C,D]  ← 全部完整
GPU2: [C]            GPU2: [A,B,C,D]  ← 全部完整
GPU3: [D]            GPU3: [A,B,C,D]  ← 全部完整
```

**核心用途**：ZeRO-3 在前向传播前用 AllGather 收集完整的模型参数（平时参数分片存储，用到时再收集）。

### 5. ReduceScatter（归约-分发）：归约后分片，每人拿一片

先对所有 GPU 的数据做归约（如求和），然后把结果分片，每个 GPU 只拿其中一片。

> **生活类比**：课代表把全班作业收齐后求和，然后把汇总结果撕成 4 份，每人拿一份——先汇总再分发。

```
操作前:              ReduceScatter(sum) 后:
GPU0: [1,2,3,4]      GPU0: [4]     ← 第 0 片归约结果 (1+1+1+1=4)
GPU1: [1,2,3,4]      GPU1: [8]     ← 第 1 片归约结果 (2+2+2+2=8)
GPU2: [1,2,3,4]      GPU2: [12]    ← 第 2 片归约结果 (3+3+3+3=12)
GPU3: [1,2,3,4]      GPU3: [16]    ← 第 3 片归约结果 (4+4+4+4=16)
```

**核心用途**：ZeRO 系列中将归约后的梯度分片存储到各卡（每卡只存一部分梯度，省显存）。

### 💡 核心分解：AllReduce = ReduceScatter + AllGather

这是本篇最重要的结论：

```
AllReduce 的效果 = 先做 ReduceScatter，再做 AllGather

步骤1 ReduceScatter:  归约 + 分片，每卡拿到 1/N 的归约结果
步骤2 AllGather:      把各卡的 1/N 结果拼成完整结果，人人都有全部

两步合起来 = 所有卡都拿到了完整的归约结果 = AllReduce
```

> 📌 **为什么要分解**？直接做 AllReduce（中心化方案）会让 rank 0 成为瓶颈。分解成 ReduceScatter + AllGather 后，可以用 **Ring 算法**让所有卡同时工作，带宽最优。这就是 3.1 篇 Ring AllReduce 的核心思想。

### 6. 通信量对比一览

| 🔧 原语 | 📤 每 rank 发送量 | 📥 每 rank 接收量 | 典型用途 |
|--------|:----------------:|:----------------:|---------|
| Broadcast | M（root）/ 0 | 0（root）/ M | 参数初始化广播 |
| Reduce | M | M（root）/ 0 | Loss 汇总 |
| AllReduce | ≈ 2M | ≈ 2M | DDP 梯度同步 |
| AllGather | (N-1)M/N | (N-1)M/N | ZeRO-3 参数收集 |
| ReduceScatter | (N-1)M/N | (N-1)M/N | ZeRO 梯度分片 |

**新人参数讲解**：
- **M**：总数据量（比如一个 1GB 的梯度张量，M = 1GB）
- **N**：参与节点数（比如 8 张卡，N = 8）
- **(N-1)M/N**：当 N=8 时，(8-1)/8 × M ≈ 0.875M，比 M 略小。这是因为分片后每卡只发自己那份。
- **AllReduce ≈ 2M**：因为 AllReduce = ReduceScatter + AllGather，每步约 (N-1)M/N，两步加起来约 2(N-1)M/N，N 大时趋近 2M。

### 7. 集合通信与并行策略的映射关系

| 并行策略 | 主要通信原语 | 通信频率 | 通信数据量 | 为什么 |
|---------|------------|---------|-----------|--------|
| 数据并行 (DDP) | AllReduce | 每层反向后 | 梯度大小 | 同步梯度 |
| ZeRO-1/2 | ReduceScatter + AllReduce | 每层反向后 | 梯度大小 | 梯度分片 |
| ZeRO-3 | AllGather + ReduceScatter | 每层前向 + 反向 | 参数大小 | 参数分片 |
| 张量并行 (TP) | AllReduce / AllGather | 每层前向 + 反向 | 激活大小 | 同步激活 |
| 流水线并行 (PP) | 点对点 Send/Recv | 每个 micro-batch | 激活大小 | 传激活值 |

> 📌 **关键点**：TP 的通信频率最高（每层都要），而且通信量与激活大小相关——这就是为什么 **TP 必须放在 NVLink 互联的卡上**。PP 虽然跨机，但通信频率和数据量都较小，IB 网络完全能够承载。

## 动手实践

### 实验 1：AllReduce（✅ 无需 sudo）

```python
import os
import torch
import torch.distributed as dist

def main():
    rank = dist.get_rank()
    # 每张卡初始化不同的值
    tensor = torch.tensor([float(rank + 1)] * 4).cuda()
    print(f"[rank {rank}] AllReduce 前: {tensor.cpu().tolist()}")

    # 所有卡求和，结果每张卡都拿到
    dist.all_reduce(tensor, op=dist.ReduceOp.SUM)
    print(f"[rank {rank}] AllReduce 后: {tensor.cpu().tolist()}")

if __name__ == "__main__":
    os.environ.setdefault("MASTER_ADDR", "localhost")
    os.environ.setdefault("MASTER_PORT", "12357")
    rank = int(os.environ.get("RANK", 0))
    world_size = int(os.environ.get("WORLD_SIZE", 1))
    dist.init_process_group(backend="nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)
    main()
    dist.destroy_process_group()
```

**运行**（4 卡）：

```bash
torchrun --nproc_per_node=4 allreduce_demo.py
```

**预期输出**：

```
[rank 0] AllReduce 前: [1.0, 1.0, 1.0, 1.0]
[rank 1] AllReduce 前: [2.0, 2.0, 2.0, 2.0]
[rank 2] AllReduce 前: [3.0, 3.0, 3.0, 3.0]
[rank 3] AllReduce 前: [4.0, 4.0, 4.0, 4.0]
[rank 0] AllReduce 后: [10.0, 10.0, 10.0, 10.0]
[rank 1] AllReduce 后: [10.0, 10.0, 10.0, 10.0]
[rank 2] AllReduce 后: [10.0, 10.0, 10.0, 10.0]
[rank 3] AllReduce 后: [10.0, 10.0, 10.0, 10.0]
```

> 💡 **解读**：4 张卡分别持有 [1,2,3,4]，AllReduce(SUM) 后所有卡都拿到 1+2+3+4=10。这就是 DDP 梯度同步的原理。

### 实验 2：AllGather（✅ 无需 sudo）

```python
import os
import torch
import torch.distributed as dist

def main():
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    # 每张卡持有一片数据
    local_chunk = torch.tensor([float(rank + 1)] * 2).cuda()  # rank 0: [1,1], rank 1: [2,2]...
    print(f"[rank {rank}] AllGather 前 local: {local_chunk.cpu().tolist()}")

    # 准备接收所有卡的数据
    gathered = [torch.zeros(2, device=f"cuda:{rank}") for _ in range(world_size)]
    dist.all_gather(gathered, local_chunk)

    result = torch.cat(gathered)
    print(f"[rank {rank}] AllGather 后: {result.cpu().tolist()}")

if __name__ == "__main__":
    os.environ.setdefault("MASTER_ADDR", "localhost")
    os.environ.setdefault("MASTER_PORT", "12358")
    rank = int(os.environ.get("RANK", 0))
    world_size = int(os.environ.get("WORLD_SIZE", 1))
    dist.init_process_group(backend="nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)
    main()
    dist.destroy_process_group()
```

**运行**：

### 实验 3：ReduceScatter（✅ 无需 sudo）

```python
import os
import torch
import torch.distributed as dist

def main():
    rank = dist.get_rank()
    world_size = dist.get_world_size()

    # 每张卡持有完整数据 [1,2,3,4]（4个元素，4卡各分1片）
    input_tensor = torch.tensor([1.0, 2.0, 3.0, 4.0]).cuda()
    print(f"[rank {rank}] ReduceScatter 前: {input_tensor.cpu().tolist()}")

    # 输出：每卡只拿到 1 个元素的归约结果
    output = torch.zeros(1, device=f"cuda:{rank}")
    input_chunks = list(input_tensor.chunk(world_size))  # 切成 4 片
    dist.reduce_scatter(output, input_chunks, op=dist.ReduceOp.SUM)
    print(f"[rank {rank}] ReduceScatter 后: {output.cpu().tolist()}")

if __name__ == "__main__":
    os.environ.setdefault("MASTER_ADDR", "localhost")
    os.environ.setdefault("MASTER_PORT", "12359")
    rank = int(os.environ.get("RANK", 0))
    world_size = int(os.environ.get("WORLD_SIZE", 1))
    dist.init_process_group(backend="nccl", rank=rank, world_size=world_size)
    torch.cuda.set_device(rank)
    main()
    dist.destroy_process_group()
```

**运行**：

```bash
torchrun --nproc_per_node=4 reducescatter_demo.py
```

**预期输出**：

```
[rank 0] ReduceScatter 前: [1.0, 2.0, 3.0, 4.0]
[rank 1] ReduceScatter 前: [1.0, 2.0, 3.0, 4.0]
[rank 2] ReduceScatter 前: [1.0, 2.0, 3.0, 4.0]
[rank 3] ReduceScatter 前: [1.0, 2.0, 3.0, 4.0]
[rank 0] ReduceScatter 后: [4.0]     ← 第0片: 1+1+1+1=4
[rank 1] ReduceScatter 后: [8.0]     ← 第1片: 2+2+2+2=8
[rank 2] ReduceScatter 后: [12.0]    ← 第2片: 3+3+3+3=12
[rank 3] ReduceScatter 后: [16.0]    ← 第3片: 4+4+4+4=16
```

> 💡 **解读**：每卡原本有完整数据 [1,2,3,4]，ReduceScatter 后每卡只拿到 1 个元素的归约结果。rank 0 拿第 0 片的归约值 4，rank 1 拿第 1 片的 8...如果接着做一次 AllGather，所有人就都拿到 [4,8,12,16]——这就等价于 AllReduce！

## 面试回答（30 秒口述版）

> 集合通信有五大原语：Broadcast 一对多广播参数，Reduce 多对一汇总 loss，AllReduce 全归约让所有人拿到相同的梯度均值（DDP 最常用），AllGather 全收集让每人拿到完整数据（ZeRO-3 用），ReduceScatter 归约后分片让每人拿一片（ZeRO 用）。核心关系是 AllReduce = ReduceScatter + AllGather，这一分解是 Ring AllReduce 带宽最优的基础。不同并行策略用不同原语：DP 用 AllReduce，TP 用 AllReduce/AllGather（频率高所以必须 NVLink），PP 用点对点 Send/Recv。

## 深入追问

**Q1：AllReduce 和 Reduce+Broadcast 有什么区别？**
效果一样，但实现不同。Reduce+Broadcast 是串行两步，rank 0 先汇总再广播，rank 0 是瓶颈。AllReduce（Ring 实现）让所有卡同时工作，带宽最优。所以 AllReduce ≠ 简单的 Reduce+Broadcast，而是有更高效的算法。

**Q2：AllReduce 通信量为什么是 ≈2M 而不是 M？**
因为 AllReduce = ReduceScatter + AllGather，每步每卡约发 (N-1)M/N，两步共约 2(N-1)M/N，N 大时趋近 2M。这是算法层面的通信量，不是"数据被传了两遍"。

**Q3：ReduceScatter 的 "scatter" 和 AllGather 的 "gather" 是反操作吗？**
是的。Scatter 是把一份完整数据拆成 N 份分给 N 个卡；Gather 是把 N 个卡的数据收集成一份完整数据。ReduceScatter = 先 Reduce 再 Scatter。

**Q4：ZeRO-3 为什么前向和反向都要通信？**
ZeRO-3 把参数分片存储，每卡只存 1/N 的参数。前向传播时需要完整参数，所以用 AllGather 收集；反向传播算完梯度后，用 ReduceScatter 归约梯度并分片存储。所以通信量比 DDP 大（参数比梯度大），但省显存。

**Q5：TP 为什么用 AllReduce 而不是 AllGather？**
TP 的每一层前向/反向需要把各卡计算的**部分结果求和**成完整结果，这是 AllReduce 的语义。某些 TP 实现也会用 ReduceScatter + AllGather 的组合来优化通信量，具体取决于实现。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|---------|
| Reduce vs AllReduce | Reduce 只 root 拿结果，AllReduce 所有人都拿 | All=所有人都拿 |
| AllGather vs AllReduce | AllGather 是拼接（不运算），AllReduce 是归约（求和等） | Gather=拼图，Reduce=求和 |
| ReduceScatter vs Reduce | ReduceScatter 结果分片每人拿一片，Reduce 只 root 拿全部 | Scatter=分片 |
| Broadcast vs AllGather | Broadcast 是一人发给所有人，AllGather 是每人各拿一片拼成全部 | Broadcast 一对多，AllGather 多对多 |
| AllReduce vs Reduce+Broadcast | 效果相同，AllReduce 实现更高效（Ring） | AllReduce 是优化版 |

## 常见报错速查表

| ❌ 现象/报错 | 原因 | ✅ 解决 |
|-------------|------|---------|
| `RuntimeError: NCCL error` | 单卡跑 nccl 后端 | 改 `backend="gloo"` + `.cpu()` |
| all_gather 输出 list 大小不对 | gathered list 长度不等于 world_size | `[torch.zeros(...) for _ in range(world_size)]` |
| reduce_scatter chunk 数不匹配 | input_chunks 数量 ≠ world_size | `tensor.chunk(world_size)` 确保切成 N 片 |
| 结果全是 0 | 没初始化 input 或 output tensor | 确保 input 有值、output 用 `torch.zeros` 预分配 |
| 死锁卡住 | 不同 rank 执行了不同的集合通信 | 确保所有 rank 执行相同的操作（all_reduce 等是集体操作） |
