## 一句话结论

nccl-tests 是 NVIDIA 官方的通信性能测试工具，能测出 AllReduce/AllGather 等操作的实际带宽。✅ 编译无需 sudo（用 conda 装 CUDA/NCCL/MPI，make 指向 conda 路径）。**最重要的概念**：输出里有两个带宽——`algbw`（算法带宽）和 `busbw`（总线带宽），对照硬件理论带宽时**看 busbw**。8 卡 H100 单机 AllReduce 的 busbw 应接近 ~850 GB/s。

## 背景与定位

4.1 篇讲了 NCCL 的概念和环境变量，但怎么知道你的集群通信性能达标？nccl-tests 就是标准答案——它是 NVIDIA 提供的基准测试工具，能测出不同通信操作的实际带宽。本篇教你编译、运行、读懂输出、排查性能问题。

## 核心概念（新人友好讲解）

### 1. nccl-tests 是什么

nccl-tests 是 NVIDIA 官方的通信性能基准测试工具集，包含多个测试程序：

| 测试程序 | 测什么 | 对应原语 |
|---------|--------|---------|
| `all_reduce_perf` | AllReduce 带宽 | DDP 梯度同步 |
| `all_gather_perf` | AllGather 带宽 | ZeRO-3 参数收集 |
| `reduce_scatter_perf` | ReduceScatter 带宽 | ZeRO 梯度分片 |
| `broadcast_perf` | Broadcast 带宽 | 参数初始化 |
| `sendrecv_perf` | 点对点 Send/Recv 带宽 | PP 激活值传递 |
| `alltoall_perf` | AllToAll 带宽 | 专家并行（MoE） |

> 💡 **最常用的是 `all_reduce_perf`**，因为 AllReduce 是 DDP 最频繁的通信操作。

### 2. 编译方法（✅ 无 sudo 方案）

#### 步骤 1：用 conda 安装依赖

```bash
# 创建独立环境（可选，推荐）
conda create -n nccl-test python=3.11 -y
conda activate nccl-test

# 安装 CUDA toolkit、NCCL、MPI（✅ 全部无需 sudo）
conda install -c nvidia nccl -y
conda install -c conda-forge openmpi -y
conda install -c nvidia cuda-toolkit -y
```

#### 步骤 2：克隆并编译

```bash
git clone https://github.com/NVIDIA/nccl-tests.git
cd nccl-tests

# 编译，指向 conda 路径（✅ 无需 sudo）
make MPI=1 \
     MPI_HOME=$CONDA_PREFIX \
     CUDA_HOME=$CONDA_PREFIX \
     NCCL_HOME=$CONDA_PREFIX
```

**新人参数讲解**：
- **`MPI=1`**：启用 MPI 支持（多机测试需要）。不启用只能单机测试。
- **`MPI_HOME`**：MPI 安装路径。conda 装的 MPI 在 `$CONDA_PREFIX`。
- **`CUDA_HOME`**：CUDA 安装路径。conda 装的在 `$CONDA_PREFIX`。
- **`NCCL_HOME`**：NCCL 安装路径。conda 装的在 `$CONDA_PREFIX`。
- **`$CONDA_PREFIX`**：当前 conda 环境的根目录，如 `~/miniconda3/envs/nccl-test`。

> 💡 **编译成功后**，可执行文件在 `build/` 目录下，如 `build/all_reduce_perf`。

### 3. 运行测试

#### 单机测试

```bash
# 单机 8 卡 AllReduce 带宽测试
./build/all_reduce_perf -b 8 -e 256M -f 2 -g 8
```

**新人参数讲解**：
- **`-b 8`**：begin，最小测试数据大小 8 字节
- **`-e 256M`**：end，最大测试数据大小 256MB
- **`-f 2`**：factor，每次测试数据大小乘以 2（8 → 16 → 32 → ... → 256M）
- **`-g 8`**：GPU 数量为 8

> 💡 数据从 8 字节测到 256MB，覆盖小消息（延迟敏感）到大消息（带宽敏感）的全范围。

#### 多机测试

```bash
# 多机测试（通过 MPI 启动，需配置 hosts 文件 + SSH 免密）
mpirun -np 16 --hostfile hosts \
  -x NCCL_DEBUG=INFO \
  -x NCCL_IB_HCA=mlx5_0 \
  ./build/all_reduce_perf -b 8 -e 256M -f 2 -g 8
```

**新人参数讲解**：
- **`-np 16`**：总进程数 16（2 机 × 8 卡）
- **`--hostfile hosts`**：指定机器列表文件
- **`-x NCCL_DEBUG=INFO`**：传递环境变量给所有进程
- **`-x NCCL_IB_HCA=mlx5_0`**：指定 IB 网卡

**hosts 文件示例**（每行一台机器的 IP 或主机名）：
```
192.168.1.1
192.168.1.2
```

### 4. 读懂输出：algbw vs busbw（本篇重点）

**预期输出**（单机 8 卡 AllReduce）：

```
#  size(B)    count   type    redop    time(us)  algbw(GB/s)  busbw(GB/s)
   8388608  2097152   float     sum      285.4       29.4         51.5
  67108864 16777216   float     sum     1823.0       36.8         64.4
 268435456 67108864   float     sum     5210.0       51.5         90.1
```

#### 输出列含义

| 列 | 含义 | 说明 |
|----|------|------|
| `size(B)` | 测试数据大小（字节） | 从 8B 到 256MB |
| `count` | 元素个数 | size / sizeof(type) |
| `type` | 数据类型 | float = FP32 |
| `redop` | 归约操作 | sum = 求和 |
| `time(us)` | 耗时（微秒） | 实际测量值 |
| `algbw(GB/s)` | 算法带宽 | = size / time |
| `busbw(GB/s)` | 总线带宽 | = algbw × 2(N-1)/N |

#### algbw vs busbw 详细讲解（新人必懂）

**两个带宽的区别**：

```
algbw（算法带宽）= 数据量 / 时间
  → 反映"应用层"看到的吞吐：我给了你 M 字节，你多快传完

busbw（总线带宽）= algbw × 2(N-1)/N
  → 反映"硬件链路"实际利用率：每条物理链路实际跑了多少数据
```

**为什么要乘 2(N-1)/N？**

回忆 3.1 篇的 Ring AllReduce 通信量公式：每 rank 通信量 = 2(N-1)/N × M。

- `algbw` 把 M 当作"有用数据"算带宽，但实际每 rank 传了 2(N-1)/N × M
- `busbw` 修正了这个倍数，反映**每条物理链路的实际利用率**
- N=8 时，2(N-1)/N = 14/8 = 1.75，所以 busbw = algbw × 1.75

**对照硬件理论带宽时看哪个？**

> 📌 **看 busbw！** busbw 才是能和硬件理论带宽直接对比的指标。

```
8 卡 H100 NVLink 4.0 理论双向带宽 = 900 GB/s
实测 busbw 应接近 ~850 GB/s（~95% 利用率）

如果 busbw 只有 400 GB/s，说明 NVLink 拓扑或 NCCL 配置有问题
```

**algbw 和 busbw 的数值关系示例**（N=8）：

| size(B) | algbw(GB/s) | busbw(GB/s) | 关系 |
|---------|:-----------:|:-----------:|------|
| 8MB | 29.4 | 51.5 | busbw ≈ algbw × 1.75 |
| 64MB | 36.8 | 64.4 | busbw ≈ algbw × 1.75 |
| 256MB | 51.5 | 90.1 | busbw ≈ algbw × 1.75 |

> 💡 N=8 时 2(N-1)/N = 1.75，所以 busbw ≈ algbw × 1.75。如果是 N=4，倍数是 1.5。

### 5. 经验参考值

| 硬件配置 | 操作 | 理论带宽 | busbw 经验值 | 达标率 |
|---------|------|:--------:|:-----------:|:------:|
| 8× H100 SXM | AllReduce | 900 GB/s | ~850 GB/s | ~95% |
| 8× A100 SXM | AllReduce | 600 GB/s | ~550 GB/s | ~92% |
| 8× H100 PCIe | AllReduce | 64 GB/s | ~55 GB/s | ~86% |
| 2 机 IB NDR | AllReduce | 50 GB/s | ~45 GB/s | ~90% |

> ⚠️ 小数据量（< 1MB）的 busbw 会远低于大消息，因为延迟占主导。这是正常的，看大消息（> 64MB）的 busbw 判断是否达标。

### 6. 性能不达标排查

如果实测 busbw 远低于经验值，按以下顺序排查：

| 排查步骤 | 命令 | 看什么 |
|---------|------|--------|
| 1. 看传输方式 | `NCCL_DEBUG=INFO` | `via P2P/IPC`（NVLink）还是 `via SHM`（回退） |
| 2. 看拓扑 | `nvidia-smi topo -m` | GPU 间是否 NV#（NVLink 连通） |
| 3. 看算法 | `NCCL_DEBUG=INFO` | Ring 还是 Tree，大消息应该 Ring |
| 4. 强制算法 | `NCCL_ALGO=Ring` | 对比强制 Ring 和自动选择 |
| 5. 禁用 P2P 排障 | `NCCL_P2P_DISABLE=1` | 如果禁用后反而正常，说明 P2P 链路有问题 |
| 6. 看 GPU 利用率 | `nvidia-smi dmon` | 测试时 GPU 利用率应接近 100% |

## 面试回答（30 秒口述版）

> nccl-tests 是 NVIDIA 官方通信性能测试工具，最常用 all_reduce_perf 测 AllReduce 带宽。编译用 conda 装 CUDA/NCCL/MPI 无需 sudo，make 指向 conda 路径。输出有两个带宽：algbw 是算法带宽（数据量/时间），busbw 是总线带宽（algbw×2(N-1)/N），对照硬件理论带宽时看 busbw。8 卡 H100 的 busbw 应接近 850 GB/s。性能不达标先看 NCCL_DEBUG=INFO 的传输方式（via P2P 还是 SHM），再看拓扑和算法选择。

## 深入追问

**Q1：为什么 busbw 有时超过理论带宽？**
测量误差、CUDA 事件计时精度、NCCL 内部优化（如 LL128 协议）。略超 5% 以内正常。如果超太多，检查测量方法是否正确。

**Q2：小消息 algbw 很低怎么办？**
小消息（< 4KB）延迟占主导，algbw 低是正常的。NCCL 会自动用 LL/LL128 协议降低延迟。如果仍然低，检查 NCCL_PROTO 是否被强制设成了 Simple。

**Q3：多机测试怎么配 SSH 免密？**
```bash
# 在主节点生成密钥
ssh-keygen -t rsa
# 复制到所有从节点
ssh-copy-id user@worker1
ssh-copy-id user@worker2
# 验证
ssh user@worker1 echo ok   # 不应要求密码
```

**Q4：all_gather_perf 的 busbw 倍数和 all_reduce 一样吗？**
不一样。AllReduce 的倍数是 2(N-1)/N，AllGather 的倍数是 (N-1)/N。因为 AllGather 只有一个阶段（收集），AllReduce 有两个阶段（ReduceScatter + AllGather）。nccl-tests 会自动用正确的倍数。

**Q5：WSL 下能跑 nccl-tests 吗？**
能编译能跑，但性能数据不代表真实硬件能力。WSL 走 SHM，busbw 会远低于 NVLink 理论值。在裸机 Linux 服务器上测才有参考价值。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|---------|
| algbw vs busbw | algbw=应用层吞吐，busbw=硬件链路利用率 | 对照理论带宽看 busbw |
| busbw 倍数 | AllReduce: 2(N-1)/N，AllGather: (N-1)/N | AllReduce有两阶段所以×2 |
| 小消息 vs 大消息 | 小消息看延迟，大消息看带宽 | 小用LL协议，大用Simple |
| 单机 vs 多机测试 | 单机走 P2P/SHM，多机走 IB/Socket | 多机需 hosts + SSH |
| MPI_HOME vs CUDA_HOME | MPI_HOME=MPI路径，CUDA_HOME=CUDA路径 | conda装都在$CONDA_PREFIX |

## 常见报错速查表

| ❌ 现象/报错 | 原因 | ✅ 解决 |
|-------------|------|---------|
| `make: *** No rule to make target` | Makefile 找不到依赖 | 确认 `CUDA_HOME`/`NCCL_HOME` 路径正确，`echo $CONDA_PREFIX` |
| `fatal error: nccl.h: No such file` | NCCL 未安装或路径不对 | `conda install -c nvidia nccl`；确认 `NCCL_HOME=$CONDA_PREFIX` |
| `fatal error: mpi.h: No such file` | MPI 未安装 | `conda install -c conda-forge openmpi`；或 `make MPI=0` 不用 MPI（仅单机） |
| busbw 远低于理论值 | 传输方式回退到 SHM/Socket | `NCCL_DEBUG=INFO` 看 via 什么；确保 NVLink/IB 可用 |
| `mpirun: command not found` | MPI 未安装或未加入 PATH | `conda install -c conda-forge openmpi`；`which mpirun` 验证 |
| 多机测试卡住 | SSH 免密未配置 | `ssh-copy-id` 配置免密；检查 hosts 文件 IP 正确 |
| `NCCL WARN : timeout` | 某个节点没加入 | 检查所有节点 NCCL 版本一致；防火墙放行端口 |
