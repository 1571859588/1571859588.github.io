## 一句话结论

从通信视角看：**TP 每层 4 次 AllReduce 必须走 NVLink（不跨机）**，DP 用 AllReduce 同步梯度（大模型跨机耗时显著），PP 用点对点传激活值（通信量小可跨机但有气泡）。气泡占比公式 = (P-1)/(P-1+M)，增大 micro-batch 数 M 能缩小气泡。本篇是全系列的综合应用，附通信瓶颈排查清单。

## 背景与定位

前面 10 篇讲了硬件（NVLink/IB）、原语（AllReduce/AllGather 等）、算法（Ring/Tree）、工具（NCCL/nccl-tests）。本篇把它们串起来，从"数据搬运"角度解释分布式训练中几个常见的架构选择——为什么 TP 不跨机、DP 通信开销多大、PP 为什么有气泡。这是面试高频考点。

## 核心概念（新人友好讲解）

### 1. 张量并行为什么只能待在机内

张量并行（TP）在 Transformer 的每一层中都需要执行 AllReduce：

```
一层前向: 输入 → Linear1 → AllReduce → 激活 → Linear2 → AllReduce → 输出
一层反向: 梯度 → AllReduce → ... → AllReduce → ...
```

每个 Transformer 层包含 Attention 和 MLP 两个子模块，各自前向、反向各 1 次，合计 **4 次 AllReduce**。

一个 80 层的大模型意味着 **320 次 AllReduce**。

#### 通信耗时估算

假设每次 AllReduce 搬运 1GB 数据：

| 互联方式 | 带宽 | 单次耗时 | 320 次总耗时 | 差距 |
|---------|:----:|:--------:|:----------:|:----:|
| NVLink 4.0（H100） | 900 GB/s | ~1.1 ms | ~0.35 秒 | 1x |
| IB NDR 4x | 50 GB/s | ~20 ms | ~6.4 秒 | **18x** |

> 📌 **结论**：走 IB 比 NVLink 慢 18 倍，训练速度断崖式下降。这就是 **"TP 不跨机"的物理原因**——TP 的通信频率太高（每层 4 次），必须待在 NVLink 带宽范围内。

### 2. 数据并行的通信开销

数据并行（DP）中 AllReduce 的数据量 = 模型梯度大小：

| 模型参数 | FP16 梯度大小 | 8 卡 Ring AllReduce 每卡通信量 | NVLink 4.0 耗时 | IB NDR 耗时 |
|---------|:------------:|:----------------------------:|:--------------:|:-----------:|
| 7B | 14 GB | ~28 GB（2×1.75×8GB） | ~31 ms | ~560 ms |
| 70B | 140 GB | ~280 GB | ~311 ms | ~5.6 秒 |

> 💡 **解读**：
> - 7B 模型在 NVLink 下 31ms 可接受，但跨机 IB 要 560ms
> - 70B 模型跨机 IB 要 5.6 秒——每个 step 多花 5.6 秒通信，训练效率极低
> - 这就是 **ZeRO 系列要把梯度和参数切碎分片**的动机：用更多次小通信替代一次大通信，让通信可以 Overlap

### 3. 流水线并行的气泡问题

流水线并行（PP）把模型按层切分到不同机器，机器间通过**点对点 Send/Recv** 传递激活值。通信量远小于 TP 和 DP，IB 网络完全够用，但代价是**流水线气泡**（GPU 空闲等待的时间段）。

#### 气泡占比公式（必背）

$$
\text{气泡占比} = \frac{P-1}{P-1+M}
$$

**新人参数讲解**：
- **P**：流水线级数（Pipeline stages），即模型被切分成几段
- **M**：micro-batch 数量（一个 batch 被切成 M 个小 batch 依次送入流水线）
- **气泡**：流水线启动和排空阶段，部分 GPU 空闲等待的时间
- **分子 P-1**：流水线填满需要 P-1 个 micro-batch 的启动时间
- **分母 P-1+M**：总时间 = 启动时间 + 正常流水时间

#### 气泡占比计算示例

| P（级数） | M（micro-batch） | 气泡占比 | 说明 |
|:---------:|:----------------:|:--------:|------|
| 4 | 1 | 3/4 = 75% | 只有 1 个 micro-batch，气泡极大 |
| 4 | 4 | 3/7 ≈ 43% | 4 个 micro-batch，气泡减小 |
| 4 | 16 | 3/19 ≈ 16% | 16 个 micro-batch，气泡较小 |
| 4 | 64 | 3/67 ≈ 4.5% | 64 个 micro-batch，气泡很小 |
| 8 | 64 | 7/71 ≈ 9.9% | 级数增加，气泡略增 |

> 💡 **结论**：增大 M（micro-batch 数）能缩小气泡，但会增加显存占用（要同时保存 M 份激活值）。这是 PP 的核心 trade-off。

#### 气泡可视化（ASCII 图）

```
P=4, M=4 的流水线（每个格子是一个 micro-batch 在某个 stage 的计算）:

         t0   t1   t2   t3   t4   t5   t6   t7
Stage 0  [B1] [B2] [B3] [B4]                      ← 填满后空闲（气泡）
Stage 1       [B1] [B2] [B3] [B4]                 ← 等启动（气泡）+ 填满后空闲
Stage 2            [B1] [B2] [B3] [B4]
Stage 3                 [B1] [B2] [B3] [B4]       ← 等启动最长（气泡最大）

气泡 = 启动阶段(P-1=3个时间槽) + 排空阶段(P-1=3个时间槽)
正常流水 = M=4 个时间槽
总时间 = P-1+M = 3+4 = 7 个时间槽
气泡占比 = (P-1)/(P-1+M) = 3/7 ≈ 43%
```

### 4. 三种并行通信特征对比

| 维度 | 张量并行 (TP) | 数据并行 (DP) | 流水线并行 (PP) |
|------|:-----------:|:------------:|:-------------:|
| 通信原语 | AllReduce/AllGather | AllReduce | 点对点 Send/Recv |
| 通信频率 | **每层 4 次**（最高） | 每层反向后 1 次 | 每个 micro-batch 1 次 |
| 通信数据量 | 激活大小 | 梯度大小 | 激活大小（小） |
| 通信带宽需求 | **极高**（必须 NVLink） | 高（大模型跨机慢） | 低（IB 够用） |
| 能否跨机 | ❌ 不能 | ⚠️ 大模型不推荐 | ✅ 可以 |
| 主要代价 | 通信开销大 | 梯度同步慢 | 流水线气泡 |

> 📌 **选型口诀**：
> - **TP 放机内**（NVLink 范围，通信频率太高）
> - **DP 可跨机**（但大模型梯度大，用 ZeRO 优化）
> - **PP 放跨机**（通信量小，IB 够用，代价是气泡）

### 5. 常见通信瓶颈排查清单

| 现象 | 可能原因 | 排查方法 |
|------|---------|---------|
| 训练 step 时间远超预期 | 通信带宽不足 | `nccl-tests` 测实际带宽（4.2 篇） |
| GPU 利用率长期偏低 | 数据加载或 CPU 预处理拖后腿 | Nsight Systems 看 CPU-GPU 交互 |
| 不同卡速度不一致 | NVLink 拓扑不对称 | `nvidia-smi topo -m` 检查（1.1 篇） |
| 多机扩展效率差 | IB 网络配置问题 | `NCCL_DEBUG=INFO` 查看传输路径（4.1 篇） |
| 通信带宽远低于理论值 | 算法选择不优或缓冲区过小 | `NCCL_ALGO` 强制切换算法测试（4.1 篇） |
| 显存 OOM | KV Cache / 激活 / 优化器状态超限 | 逐项计算显存占用 |

## 动手实践

### 实验 1：通信耗时估算计算

**场景**：70B 模型，80 层 Transformer，8 卡 H100。

```python
# TP 通信耗时估算（假设每次 AllReduce 1GB 激活）
tp_layers = 80
tp_comm_per_layer = 4  # 每层 4 次 AllReduce
tp_data_per_comm = 1  # GB
nvlink_bw = 900  # GB/s
ib_bw = 50  # GB/s

tp_time_nvlink = tp_layers * tp_comm_per_layer * tp_data_per_comm / nvlink_bw
tp_time_ib = tp_layers * tp_comm_per_layer * tp_data_per_comm / ib_bw

print(f"TP 走 NVLink: {tp_time_nvlink:.2f}s")   # 0.36s
print(f"TP 走 IB: {tp_time_ib:.2f}s")            # 6.40s
print(f"差距: {tp_time_ib/tp_time_nvlink:.1f}x")  # 18x
```

**预期输出**：

```
TP 走 NVLink: 0.36s
TP 走 IB: 6.40s
差距: 18.0x
```

### 实验 2：气泡占比计算

```python
def bubble_ratio(P, M):
    """计算流水线气泡占比"""
    return (P - 1) / (P - 1 + M)

print("P=4:")
for M in [1, 4, 16, 64]:
    print(f"  M={M:2d}: 气泡占比 = {bubble_ratio(4, M):.1%}")

print("P=8:")
for M in [1, 4, 16, 64]:
    print(f"  M={M:2d}: 气泡占比 = {bubble_ratio(8, M):.1%}")
```

**预期输出**：

```
P=4:
  M= 1: 气泡占比 = 75.0%
  M= 4: 气泡占比 = 42.9%
  M=16: 气泡占比 = 15.8%
  M=64: 气泡占比 = 4.5%
P=8:
  M= 1: 气泡占比 = 87.5%
  M= 4: 气泡占比 = 63.6%
  M=16: 气泡占比 = 30.4%
  M=64: 气泡占比 = 9.9%
```

> 💡 **解读**：P=4 时 M=64 能把气泡压到 4.5%，但 P=8 时 M=64 还有 9.9%。级数越多需要的 micro-batch 越多才能填满流水线。

### 实验 3：排查清单使用指引

当遇到通信性能问题时，按以下顺序排查（所有命令 ✅ 无需 sudo）：

```bash

# 步骤2: 看拓扑

# 步骤4: 看 GPU 利用率

# 预期: 利用率接近 100%（如果低说明通信阻塞了计算）

# 步骤5: 强制算法测试
NCCL_ALGO=Ring NCCL_DEBUG=INFO python train.py
# 对比自动选择和强制 Ring 的性能差异
```

## 面试回答（30 秒口述版）

> 从通信视角看三种并行：TP 每层 4 次 AllReduce，80 层模型 320 次，走 NVLink 约 0.35 秒，走 IB 约 6.4 秒差 18 倍，所以 TP 必须在机内 NVLink 范围。DP 用 AllReduce 同步梯度，7B 模型 NVLink 下 31ms 可接受，70B 跨机 IB 要 5.6 秒所以需要 ZeRO 分片优化。PP 用点对点传激活值通信量小可跨机，但有流水线气泡，占比 (P-1)/(P-1+M)，增大 micro-batch 数 M 能缩小气泡但增显存。排查通信问题先看 NCCL_DEBUG=INFO 的传输方式，再测 nccl-tests 带宽，最后看 GPU 利用率。

## 深入追问

**Q1：3D 并行是什么？**
3D 并行 = TP + DP + PP 同时使用。通常 TP 放机内（NVLink），PP 放跨机（IB），DP 在剩余维度扩展。例如 8 机 64 卡：每机 8 卡做 TP（8路），8 机做 PP（8级），整体再做 DP。这样三种并行各取所长。

**Q2：ZeRO 怎么降低 DP 通信开销？**
ZeRO-1/2 把梯度分片（ReduceScatter 替代 AllReduce），每卡只存 1/N 梯度，通信量从 2M 降到 2M/N。ZeRO-3 进一步把参数分片，用 AllGather 在需要时收集。代价是通信次数增加，但单次通信量小，更容易 Overlap。

**Q3：气泡能完全消除吗？**
不能完全消除，但可以用 1F1B（一个前向一个反向交替）等调度策略减小。Interleaved 1F1B（交错调度）能进一步压缩气泡，Megatron-LM 有实现。理论上 M→∞ 时气泡→0，但显存限制 M 不能太大。

**Q4：为什么不用全 TP？**
TP 只能放机内（最多 8 卡），无法扩展到更多卡。大模型训练需要 TP+DP+PP 组合才能用上千卡。而且 TP 通信量大，层内切分太细会通信开销过大。

**Q5：通信开销和计算开销怎么平衡？**
理想情况是通信时间 = 计算时间（完全 Overlap）。如果通信 > 计算，说明网络瓶颈，需要更快的网络或更优的并行策略。如果计算 > 通信，说明网络有余量，可以增加并行度。用 Profiler（nsys）测量通信和计算占比。

## 易混淆点对比

| 易混概念 | 区别 | 记忆技巧 |
|---------|------|---------|
| TP vs DP 通信 | TP 每层4次最高频，DP 每层反向1次 | TP频率最高 |
| TP vs PP 跨机 | TP不能跨机，PP可以 | TP要NVLink，PP通信小 |
| 气泡 vs 通信开销 | 气泡是PP的代价，通信开销是TP/DP的代价 | PP气泡，TP/DP通信 |
| P vs M 对气泡影响 | P增大气泡增，M增大气泡减 | 级数多气泡大，micro多气泡小 |
| ZeRO-1/2 vs ZeRO-3 | 1/2分梯度，3还分参数 | 3分得最彻底 |

## 常见报错速查表

| ❌ 现象 | 原因 | ✅ 排查方法 |
|---------|------|-----------|
| 训练速度远慢于预期 | 通信带宽不足 | `nccl-tests` 测带宽，对照经验值 |
| GPU 利用率 < 50% | 通信阻塞或数据加载慢 | `nvidia-smi dmon`；nsys 分析 |
| 多机扩展性差（加机器不加速） | IB 配置或拓扑问题 | `NCCL_DEBUG=INFO` 看传输路径 |
| TP 跨机后训练极慢 | TP 通信走 IB 而非 NVLink | 检查 `nvidia-smi topo -m`；TP 组必须在 NVLink 卡上 |
| 大模型 DP 跨机 OOM | 梯度+参数+优化器状态太大 | 用 ZeRO-1/2/3 分片 |
| PP 训练 GPU 空闲多 | 气泡占比高 | 增大 micro-batch 数 M；用 1F1B 调度 |
