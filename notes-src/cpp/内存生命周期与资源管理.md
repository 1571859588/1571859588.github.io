## 一句话结论

C++ 区别于 Python/Java 的核心：**你必须亲手管理内存和资源的生命周期**。现代 C++ 的解法是 **RAII**——把资源绑定到对象的生命周期上，让作用域替你自动释放。记住：能用 `std::vector` 就别 `new`，能用 `unique_ptr` 就别裸指针，能用 `const` 就别不加。

## 零、为什么 C++ 要操心内存？

先建立直觉——对比三种语言的内存管理：

| 语言 | 谁管内存 | 怎么回收 | 你的责任 |
|------|---------|---------|---------|
| **Python** | 解释器 | 引用计数 + GC | 几乎不用管 |
| **Java** | JVM | 垃圾回收器（GC） | 不用管，但可能 STW |
| **C++** | **你** | 手动 `delete` 或 RAII 自动释放 | **必须管，否则内存泄漏或崩溃** |

```
Python:  x = [1,2,3]        → 不用管，GC 自动回收
Java:    int[] x = {1,2,3}; → 不用管，JVM 自动回收
C++:     int* x = new int[3]{1,2,3};  → 你必须 delete[] x;
         std::vector<int> x = {1,2,3}; → vector 帮你管（RAII）
```

**核心问题**：C++ 没有垃圾回收器。你 `new` 出来的东西必须自己 `delete`，否则内存永远不释放（内存泄漏）。如果 `delete` 了两次（重复释放），程序可能崩溃。

现代 C++（C++11 以后）给出了答案：**RAII + 智能指针**，让你写起来像 Python 一样省心，但性能像 C 一样快。

## 一、值、地址、指针与引用

### 1.1 三个核心概念

先看一段代码，建立直觉：

```cpp
#include <iostream>

int main() {
    int value = 42;    // 在内存中创建一个 int，值为 42
    int* ptr = &value; // ptr 保存 value 的地址（指针）
    int& ref = value;  // ref 是 value 的别名（引用）

    *ptr = 7;          // 通过指针找到 value，改成 7
    ref += 1;          // 通过引用找到 value，加 1

    std::cout << value << '\n';  // 输出 8
}
```

**逐行拆解**：

| 代码 | 含义 | 类比 |
|------|------|------|
| `int value = 42;` | 在内存中创建一个变量，名字叫 value，值为 42 | 一个盒子里装了 42 |
| `&value` | 取 value 的内存地址 | 盒子在仓库里的坐标 |
| `int* ptr = &value;` | ptr 是一个指针，保存 value 的地址 | 一张纸条写着"42号盒子在A3货架" |
| `int& ref = value;` | ref 是 value 的**别名**，两者是同一个东西 | 给 42号盒子贴了个小名"小四" |
| `*ptr = 7;` | 解引用指针：顺着地址找到 value，改成 7 | 按纸条找到盒子，把内容换成 7 |
| `ref += 1;` | 通过别名修改原变量 | 叫"小四"把内容 +1 |

### 1.2 指针 vs 引用——选哪个？

| 概念 | 能否为空 | 能否改指向 | 声明时必须初始化 | 常见用途 |
|------|---------|-----------|----------------|---------|
| **值 `T`** | 否 | 不适用 | 是 | 拥有一个独立对象 |
| **指针 `T*`** | 可以是 `nullptr` | **可以** | 否（但建议初始化） | 可选对象、数组、底层接口 |
| **引用 `T&`** | **不能为空** | **不能改指向** | **必须** | 必须存在的别名、函数参数 |
| **常量引用 `const T&`** | 不能为空 | 不能改指向 | 必须 | 只读借用，避免复制 |

**关键区别**：

```cpp
int a = 1, b = 2;

// 指针可以改指向
int* ptr = &a;
ptr = &b;        // ✅ ptr 现在指向 b

// 引用不能改指向
int& ref = a;
ref = b;         // ⚠️ 这不是"让 ref 引用 b"！
                 // 这是把 b 的值赋给 a，结果是 a=2, b=2
                 // ref 仍然引用 a
```

### 1.3 函数参数怎么传？——接口意图

```cpp
void consume(T value);          // 按值传递：函数获得自己的副本（复制一份进去）
void update(T& value);          // 引用传递：函数会修改调用方的原始对象
void inspect(const T& value);   // const 引用：只读，不复制，不改
void optional(T* value);        // 指针：可能传 nullptr（"没有"）
```

**完整示例**：

```cpp
#include <iostream>
#include <string>

// 按值：函数内部修改不影响外部
void consume(std::string s) {
    s += " (modified)";  // 修改的是副本
}

// 可变引用：函数内部修改直接影响外部
void update(std::string& s) {
    s += " (modified)";
}

// const 引用：只读，不复制
void inspect(const std::string& s) {
    std::cout << "内容: " << s << '\n';
    // s += "x";  // ❌ 编译错误：const 不能修改
}

// 指针：可能为空，调用时要判断
void optional(const std::string* s) {
    if (s == nullptr) {
        std::cout << "没有传入字符串\n";
        return;
    }
    std::cout << "内容: " << *s << '\n';
}

int main() {
    std::string name = "hello";

    consume(name);
    std::cout << "consume 后: " << name << '\n';        // hello（没变）

    update(name);
    std::cout << "update 后: " << name << '\n';          // hello (modified)

    inspect(name);                                        // 内容: hello (modified)
    optional(&name);                                      // 内容: hello (modified)
    optional(nullptr);                                    // 没有传入字符串
}
```

**什么时候用哪个？**（决策树）

```
参数需要修改调用方的对象吗？
│
├─ 是 → 用 T&（引用）
│
├─ 否 → 对象小吗？（int, float, 指针等 ≤ 8 字节）
│       │
│       ├─ 小 → 按值传 T（最简单，复制成本可忽略）
│       │
│       └─ 大（string, vector, 自定义类）→ 用 const T&（避免复制）
│
└─ 可能不传（可选参数）→ 用 T*（指针，可能为 nullptr）
```

> **新手原则**：小型标量（int、float）按值传；大对象用 `const T&`；需要修改用 `T&`；可能为空用 `T*`。不要机械地把所有参数都改成引用。

## 二、数组、指针与连续内存

### 2.1 原生数组为什么会"退化"

在 C++ 中，原生数组在大多数情况下会自动变成"指向首元素的指针"：

```cpp
void print_first(int arr[]) {
    // ⚠️ 在这里，arr 不再是数组，而是 int* 指针
    // 你完全不知道 arr 有多少个元素！
    std::cout << arr[0] << '\n';
    std::cout << sizeof(arr) << '\n'; // sizeof(arr) 在这里是 sizeof(int*) = 8（64位系统），不是数组大小
}

int main() {
    int data[] = {10, 20, 30, 40, 50};
    // sizeof(data) = 20（5个int × 4字节）
    print_first(data);  // 数组退化为指针，长度信息丢失
}
```

**为什么会这样**：C 语言时代的历史遗留设计。数组传参时只传首地址，不传长度，所以函数内部不知道数组有多大。

### 2.2 解决方案：用能表达长度的类型

**方法一：传长度参数（C 风格）**

```cpp
void scale(float* data, std::size_t size, float factor) {
    for (std::size_t i = 0; i < size; ++i) {
        data[i] *= factor;
    }
}

int main() {
    float data[] = {1.0f, 2.0f, 3.0f, 4.0f};
    scale(data, 4, 2.0f);  // 每个元素乘 2
    // data 现在是 {2.0, 4.0, 6.0, 8.0}
}
```

**方法二：用 `std::span`（C++20，推荐）**

```cpp
#include <span>
#include <array>
#include <vector>
#include <iostream>

// std::span = 指针 + 长度，但封装好了，不用手动传 size
void scale(std::span<float> values, float factor) {
    for (float& value : values) {  // 可以用范围 for 循环
        value *= factor;
    }
}

void print(std::span<const float> values) {
    for (float v : values) {
        std::cout << v << ' ';
    }
    std::cout << '\n';
}

int main() {
    // span 可以接受多种来源：原生数组、vector、array
    float raw[] = {1.0f, 2.0f, 3.0f};
    std::vector<float> vec = {4.0f, 5.0f, 6.0f, 7.0f};
    std::array<float, 2> arr = {8.0f, 9.0f};

    scale(raw, 2.0f);   // 原生数组也能用
    scale(vec, 2.0f);   // vector 也能用
    scale(arr, 2.0f);   // array 也能用

    print(raw);  // 2 4 6
    print(vec);  // 8 10 12 14
    print(arr);  // 16 18
}
```

> **新人疑问：`std::array<float, 2>` 中的 `2` 是什么意思？**
>
> `std::array` 是 C++ 的**固定大小数组**，模板参数有两个：`std::array<元素类型, 大小>`。
>
> - `float` = 元素类型
> - `2` = 数组大小（**编译期常量，运行时不能改**）
>
> 所以 `std::array<float, 2> arr = {8.0f, 9.0f};` 表示"创建一个有 2 个 float 的固定数组"。
>
> 对比三种数组：
>
> | 类型 | 大小 | 大小何时确定 | 内存位置 | 示例 |
> |------|------|------------|---------|------|
> | 原生数组 `float[]` | 固定 | 编译期 | 栈 | `float arr[] = {1,2,3};` |
> | `std::array<T,N>` | 固定 | 编译期 | 栈 | `std::array<float,2> arr = {8,9};` |
> | `std::vector<T>` | 可变 | 运行期 | 堆 | `std::vector<float> v = {1,2,3};` |
>
> `std::array` 相比原生数组的好处：知道自己大小、能用范围 for、不会退化为指针。`N` 必须是编译期常量（字面量、`constexpr` 变量等）。

> **常见报错：`fatal error: span: No such file or directory`**
>
> `<span>` 是 **C++20** 才引入的头文件。如果编译器太老或没用 C++20 标准，就会报这个错。
>
> **解决方法**：
> 1. **编译时指定 C++20 标准**：`g++ -std=c++20 main.cpp -o main`
> 2. **用足够新的编译器**：GCC ≥ 10、Clang ≥ 10、MSVC ≥ 19.28
> 3. **在线编译器推荐**：[Godbolt](https://godbolt.org/)（选 GCC 12+，加 `-std=c++20`） 或者 [[C++\] gcc 13.2.0 - Wandbox](https://wandbox.org/)
>
> **如果编译器不支持 C++20**，用 C++17 的替代写法：
>
> ```cpp
> // C++17 替代方案一：传 vector 引用
> void scale(std::vector<float>& values, float factor) {
>     for (float& value : values) { value *= factor; }
> }
>
> // C++17 替代方案二：指针 + 长度（最通用）
> void scale_raw(float* values, std::size_t size, float factor) {
>     for (std::size_t i = 0; i < size; ++i) { values[i] *= factor; }
> }
> ```

**`std::span` 的核心特点**：
- **不拥有数据**：它只是一个"视图"，像 Python 的 slice，但底层不复制
- **携带长度**：不用额外传 size 参数
- **轻量**：内部就是 `指针 + 长度`，sizeof = 16 字节（64位系统）
- **注意**：调用期间底层内存必须仍然有效（不能传一个已销毁的局部数组给 span）

> **`sizeof = 16 字节` 是怎么算出来的？**
>
> `std::span` 内部只有两个成员：
>
> | 成员 | 类型 | 64 位系统大小 | 作用 |
> |------|------|-------------|------|
> | 指针 | `T*` | **8 字节** | 指向数据首地址 |
> | 长度 | `size_t` | **8 字节** | 元素个数 |
>
> ```
> 8 字节（指针）+ 8 字节（长度）= 16 字节
> ```
>
> **为什么都是 8 字节**：64 位系统中地址总线是 64 位的，所以任何指针（`int*`、`float*`、`T*`）都是 8 字节（要存 64 位地址）；`size_t` 是表示大小的无符号整数，在 64 位系统上定义为 `uint64_t`，也是 8 字节。
>
> **验证**：
> ```cpp
> std::cout << sizeof(std::span<float>);   // 16
> std::cout << sizeof(float*);             // 8
> std::cout << sizeof(std::size_t);        // 8
> ```
>
> **32 位系统对比**：指针和 `size_t` 都是 4 字节，所以 `sizeof(std::span<float>) = 8`。
>
> **为什么不存 `end` 指针而存 `size`？**
> - 存 `size` 的好处：想知道长度时 O(1) 直接读，不用算 `end - begin`
> - 存 `end` 的好处：切片操作稍微方便
> - 标准库选择了存 `size`，所以是 `指针 + size_t = 16 字节`

### 2.3 二维矩阵的线性存储

在内存中，二维矩阵是**一维线性存储**的。行主序（Row-major）矩阵 `A[M][N]` 的 `(row, col)` 对应一维偏移：

```
offset = row × N + col
```

**图解**（3行4列矩阵）：

```
逻辑视角：              物理存储（连续内存）：
  col:  0   1   2   3   
row 0: [A] [B] [C] [D]    →  [A][B][C][D][E][F][G][H][I][J][K][L]
row 1: [E] [F] [G] [H]         ↑              ↑              ↑
row 2: [I] [J] [K] [L]         offset=0       offset=4       offset=8

访问 A[1][2]（第2行第3列，即 G）：
offset = 1 × 4 + 2 = 6  →  内存第7个位置 = G ✅
```

**代码实现**：

```cpp
#include <vector>
#include <iostream>

class Matrix {
public:
    Matrix(std::size_t rows, std::size_t cols)
        : rows_(rows), cols_(cols), data_(rows * cols, 0.0f) {}

    // 访问元素
    float& operator()(std::size_t row, std::size_t col) {
        return data_[row * cols_ + col];  // 核心公式
    }

    float operator()(std::size_t row, std::size_t col) const {
        return data_[row * cols_ + col];
    }

    std::size_t rows() const { return rows_; }
    std::size_t cols() const { return cols_; }

private:
    std::size_t rows_;
    std::size_t cols_;
    std::vector<float> data_;  // 一维存储
};

int main() {
    Matrix m(3, 4);
    m(0, 0) = 1.0f;
    m(1, 2) = 5.0f;
    m(2, 3) = 9.0f;

    std::cout << m(0, 0) << '\n';  // 1
    std::cout << m(1, 2) << '\n';  // 5
    std::cout << m(2, 3) << '\n';  // 9
}
```

> **新人疑问：为什么有两个 `operator()`？`&` 和 `const` 各是什么意思？**
>
> 这叫 **const 重载**——两个函数同名同参数，但一个用于"可修改"场景，一个用于"只读"场景。
>
> **逐个拆解**：
>
> ```cpp
> // 版本一：可变版本（用于非 const 对象）
> float& operator()(std::size_t row, std::size_t col) {
>     return data_[row * cols_ + col];
> }
> //       ^ 返回 float 的引用
> //  ↑ 没有const → 可以修改对象
> ```
> - 返回类型是 `float&`（引用），意味着返回的是**元素本身的别名**
> - 调用者可以通过这个引用**修改**矩阵里的值：`m(0,0) = 1.0f;`
> - 没有 `const` 修饰 → 只能对**非 const 对象**调用
>
> ```cpp
> // 版本二：只读版本（用于 const 对象）
> float operator()(std::size_t row, std::size_t col) const {
>     return data_[row * cols_ + col];
> }
> //      ^ 返回 float 的值（拷贝）
> //                                   ^^^^^ const 成员函数
> ```
> - 返回类型是 `float`（值），返回的是**元素的拷贝**，调用者不能通过它修改矩阵
> - 末尾的 `const` 表示"这个函数不会修改对象状态" → 可以对 **const 对象**调用
>
> **编译器怎么选**：
>
> ```cpp
> Matrix m(3, 4);          // 非 const 对象
> const Matrix cm(3, 4);   // const 对象
>
> m(0, 0) = 1.0f;          // 选版本一（非const对象 → 调非const版本）✅
>                           // 返回引用，可以赋值
>
> float v = cm(0, 0);      // 选版本二（const对象 → 只能调const版本）✅
>                           // 返回值，不能写 cm(0,0) = 1.0f
> // cm(0, 0) = 1.0f;      // ❌ 编译错误！const 对象不能修改
> ```
>
> **为什么不能只写一个 `float&` 版本？**
> ```cpp
> // 假设只有这一个版本：
> float& operator()(std::size_t row, std::size_t col) {
>     return data_[row * cols_ + col];
> }
>
> const Matrix cm(3, 4);
> cm(0, 0);  // ❌ 编译错误！
> // 原因：非 const 成员函数不能在 const 对象上调用
> // 编译器担心你会通过返回的引用修改 const 对象
> ```
>
> **为什么不能只写一个返回值的版本？**
> ```cpp
> // 假设只有这一个版本：
> float operator()(std::size_t row, std::size_t col) const {
>     return data_[row * cols_ + col];
> }
>
> Matrix m(3, 4);
> m(0, 0) = 1.0f;  // ❌ 语法错误！
> // 原因：返回的是值的拷贝，给拷贝赋值没有意义
> // 等价于：float temp = data_[...]; temp = 1.0f; ← 没改到矩阵
> ```
>
> **总结：两个版本缺一不可**
>
> | 版本 | 返回类型 | const 修饰 | 何时用 | 能否赋值 |
> |------|---------|-----------|--------|---------|
> | 可变版本 | `float&`（引用） | 无 | 非 const 对象 | ✅ `m(0,0)=1` |
> | 只读版本 | `float`（值） | 有 | const 对象 | ❌ 只能读 |
>
> **记忆口诀**：
> - `&` 在返回类型前面 → 返回引用 → 能改（像 Python 的 `__setitem__`）
> - `const` 在函数后面 → 只读 → 不能改（像 Python 的 `@property`）

> **为什么要知道这个**：CUDA 编程中 GPU 显存的访问模式直接依赖这个公式。连续访问（按行顺序）比跳着访问（按列顺序）快很多倍，因为 CPU 缓存和 GPU 合并访存（coalesced access）都依赖访问的连续性。

## 三、对象存储期与生命周期

### 3.1 "栈"和"堆"——先用通俗说法

```
栈（Stack）                     堆（Heap）
┌──────────────┐               ┌──────────────┐
│ 局部变量      │               │ new 出来的    │
│ 函数参数      │               │ 动态分配的    │
│ 离开作用域    │               │ 必须手动 delete│
│ 自动销毁      │               │ 否则泄漏！    │
└──────────────┘               └──────────────┘
   快（移动指针）                 慢（找空闲块）
   小（默认 1-8MB）               大（几乎全部物理内存）
   自动管理                       手动管理
```

### 3.2 代码示例

```cpp
#include <iostream>

void example() {
    // ① 自动存储期（栈）：离开 {} 就自动销毁
    int local = 1;
    {
        int inner = 2;
        // inner 在这里有效
    }  // ← inner 在这里自动销毁
    // inner 已经不存在了

    // ② 动态存储期（堆）：必须手动 delete
    auto buffer = new float[1024];  // 在堆上分配 1024 个 float
    buffer[0] = 3.14f;

    // ③ 如果忘记 delete 就返回 → 内存泄漏！
    delete[] buffer;  // 必须手动释放

    // ④ 如果 delete 后还用 buffer → 悬空指针（未定义行为）！
    // buffer[0] = 0.0f;  // ❌ 危险！buffer 已被释放

    // ⑤ 如果 delete 两次 → 重复释放（可能崩溃）！
    // delete[] buffer;   // ❌ 危险！
}

int main() {
    example();
}
```

### 3.3 手动 new/delete 的五大陷阱

| 陷阱 | 代码示例 | 后果 |
|------|---------|------|
| **内存泄漏** | `new int` 后忘记 `delete` | 内存永远不释放，程序越跑越占 |
| **重复释放** | `delete p; delete p;` | 可能崩溃（堆被破坏） |
| **悬空指针** | `delete p; *p = 5;` | 未定义行为，可能静默出错 |
| **形式不匹配** | `new int[10]` 但用 `delete`（不是 `delete[]`） | 未定义行为 |
| **异常路径泄漏** | `new` 后函数中途抛异常，没走到 `delete` | 内存泄漏 |

**异常路径泄漏示例**：

```cpp
void dangerous() {
    auto data = new float[1024];  // 分配内存

    do_something();  // ⚠️ 如果这里抛异常？
    // 后面的 delete[] 永远不会执行 → 内存泄漏！

    delete[] data;
}

// 正确做法：用 RAII（下一节讲）
void safe() {
    std::vector<float> data(1024);  // vector 在析构时自动释放

    do_something();  // 即使抛异常，data 的析构也会执行 → 自动释放
}
```

> **核心认知**：手动 `new/delete` 的问题不是"难写"，而是**异常安全**——你很难保证每条退出路径都正确释放。现代 C++ 的解法是 RAII。

## 四、RAII：把资源绑定到对象生命周期

### 4.1 什么是 RAII

**RAII（Resource Acquisition Is Initialization）**：
- **资源获取即初始化**：在构造函数中获取资源
- **资源释放即析构**：在析构函数中释放资源
- 核心思想：**让资源由对象管理，让对象由作用域管理**

```
                    对象的构造函数
                    ┌─────────────┐
                    │ 获取资源     │ ← new / fopen / malloc / cudaMalloc
                    └──────┬──────┘
                           │
                    对象在使用中
                           │
                    对象离开作用域（或被析构）
                    ┌─────────────┐
                    │ 释放资源     │ ← delete / fclose / free / cudaFree
                    └─────────────┘
                    ↑
              无论正常返回还是抛异常，析构都会执行
```

**对比 Python**：
```python
# Python 的 with 语句就是 RAII 的精神
with open("file.txt") as f:    # 获取资源
    data = f.read()
# 离开 with 块，f.close() 自动调用  ← 释放资源

# C++ 的 RAII 更彻底：所有局部对象都自动这样做
{
    std::ifstream f("file.txt");  // 构造时打开文件
    std::string line;
    std::getline(f, line);
}  // 离开作用域，f 的析构自动关闭文件
```

### 4.2 完整示例：自己实现 RAII 类

```cpp
#include <cstdio>
#include <iostream>
#include <stdexcept>

class File {
public:
    // 构造函数：获取资源（打开文件）
    explicit File(const char* path)
        : handle_(std::fopen(path, "rb")) {
        if (handle_ == nullptr) {
            throw std::runtime_error(std::string("无法打开文件: ") + path);
        }
        std::cout << "文件已打开\n";
    }

    // 析构函数：释放资源（关闭文件）
    ~File() {
        if (handle_ != nullptr) {
            std::fclose(handle_);
            std::cout << "文件已关闭\n";
        }
    }

    // 删除拷贝（防止两个对象指向同一个文件句柄）
    File(const File&) = delete;
    File& operator=(const File&) = delete;

    // 提供使用接口
    int read() {
        return std::fgetc(handle_);
    }

private:
    std::FILE* handle_;
};
int main() {
    {
        File f("test.txt");  // 构造：打开文件
        int ch = f.read();   // 使用
        std::cout << "读到: " << ch << '\n';
    }  // 离开作用域 → 析构自动关闭文件
    // 即使上面 read() 抛异常，析构也会执行 ✅
}
```

> **新人疑问：`std::FILE` 是哪来的？和我们自定义的 `File` 类有关系吗？**
>
> **完全没关系**，只是名字看起来像。区分一下：
>
> | 名字 | 来源 | 是什么 | 用法 |
> |------|------|--------|------|
> | `File` | **我们自己定义的类**（本笔记里的 RAII 示例） | C++ 类 | `File f("test.txt");` |
> | `FILE` | **C 标准库** `<cstdio>` 里定义的结构体类型 | C 语言时代的文件句柄 | `std::FILE* handle = std::fopen(...);` |
>
> **`std::FILE` 是什么**：
> - C 标准库 `<stdio.h>` 里有一个 `FILE` 结构体，代表"打开的文件"
> - C++ 把 C 标准库搬进 `std` 命名空间，所以 `<cstdio>` 里的 `FILE` 在 C++ 中写成 `std::FILE`
> - `std::FILE*` 就是"指向 FILE 结构体的指针"，是 C 语言操作文件的句柄
>
> **C++ 包裹 C 头文件的惯例**：
>
> | C 头文件 | C++ 头文件 | 类型/函数 | 说明 |
> |---------|-----------|---------|------|
> | `<stdio.h>` | `<cstdio>` | `FILE` → `std::FILE`、`fopen` → `std::fopen` | 去掉 .h，加 c 前缀，放进 std |
> | `<stdlib.h>` | `<cstdlib>` | `malloc` → `std::malloc` | 同上 |
> | `<string.h>` | `<cstring>` | `strlen` → `std::strlen` | 同上 |
> | `<math.h>` | `<cmath>` | `sqrt` → `std::sqrt` | 同上 |
>
> 规则：C 的 `<xxx.h>` → C++ 的 `<cxxx>`，内容放进 `std` 命名空间。两种写法都能用，但 C++ 推荐用 `<cxxx>` + `std::` 前缀。
>
> **本例的关系**：
> ```cpp
> class File {                  // ← 我们写的 C++ RAII 类
> private:
>     std::FILE* handle_;       // ← C 标准库的文件句柄，被我们包装起来
> };
> // File 是 C++ 类，内部用 std::FILE* 这个 C 类型干活
> // 相当于给 C 的 FILE 加了一层 RAII 外壳
> ```
>
> **记忆口诀**：`File` 是我们写的"盒子"，`std::FILE*` 是盒子里装的"C 语言零件"。

> **新人疑问一：哪里体现了 RAII？**
>
> RAII 的核心是"**资源获取即初始化，资源释放即析构**"。对照代码看：
>
> | RAII 原则 | 对应代码 | 说明 |
> |-----------|---------|------|
> | 资源获取即初始化 | 构造函数里 `std::fopen(path, "rb")` | 对象**创建时**就打开文件 |
> | 资源释放即析构 | 析构函数里 `std::fclose(handle_)` | 对象**销毁时**就关闭文件 |
> | 作用域管理生命周期 | `File f("test.txt")` 在 `{}` 内 | 离开 `{}` 自动调用析构 |
>
> **为什么这就是 RAII**：你不需要手动调 `f.close()`，只要 `f` 离开作用域（`}`），C++ 保证析构函数一定执行，文件一定被关闭。即使中间抛异常也不会漏掉。
>
> 对比非 RAII 的写法（手动管理）：
> ```cpp
> // ❌ 非 RAII：手动打开/关闭，容易漏
> FILE* handle = std::fopen("test.txt", "rb");
> do_something();       // 如果这里抛异常...
> std::fclose(handle);  // ← 这行不会执行！文件句柄泄漏
>
> // ✅ RAII：构造打开，析构关闭
> {
>     File f("test.txt");  // 打开
>     do_something();      // 即使抛异常...
> }                        // 析构仍执行，自动关闭 ✅
> ```
>
> **RAII 的本质**：把"资源"绑定到"对象的生命周期"上，让 C++ 的作用域规则替你自动管理资源。这种思想可以推广到任何资源——文件、锁、内存、GPU 显存、CUDA Stream。

> **新人疑问二：`explicit` 是什么意思？**
>
> `explicit` 修饰构造函数，**禁止编译器做隐式类型转换**。
>
> ```cpp
> explicit File(const char* path)
> ```
>
> **没有 `explicit` 会发生什么**：
> ```cpp
> class File {
> public:
>     File(const char* path);  // 没有 explicit
>     // ...
> };
>
> void process(File f) {      // 函数参数是 File 类型
>     // ...
> }
>
> process("test.txt");  // ⚠️ 编译器会隐式转换！
> // 等价于：process(File("test.txt"))
> // 你可能根本不想打开文件，但编译器"好心"帮你开了一个
> ```
>
> **加了 `explicit` 后**：
> ```cpp
> class File {
> public:
>     explicit File(const char* path);  // 加了 explicit
>     // ...
> };
>
> process("test.txt");        // ❌ 编译错误！不能隐式转换
> process(File("test.txt"));  // ✅ 必须显式构造
> ```
>
> **什么时候用 `explicit`**：
> - 构造函数**只有一个参数**时，建议加 `explicit`
> - 防止意外的隐式转换导致 bug
> - 多参数构造函数一般不需要（C++11 后多参数也能隐式转换，但少见）
>
> **记忆口诀**：`explicit` = "**别替我自作主张**"。强迫调用者明确写出构造，避免编译器偷偷帮你转换。

### 4.3 RAII 的优势

| 手动管理 | RAII |
|---------|------|
| 忘记 delete → 泄漏 | 离开作用域自动释放 |
| 异常路径忘记 delete → 泄漏 | 异常时析构自动执行 |
| 重复 delete → 崩溃 | 一个对象只析构一次 |
| 心智负担重 | 像写 Python 一样省心 |

### 4.4 标准库已有的 RAII 类型

> **原则：优先用标准库已有的，不要重复造轮子。**

| 标准库类型 | 管理的资源 | 用途 |
|-----------|-----------|------|
| `std::vector` | 动态数组 | 替代 `new[]` / `delete[]` |
| `std::string` | 字符串 | 替代 `char*` |
| `std::fstream` | 文件 | 替代 `fopen` / `fclose` |
| `std::lock_guard` | 互斥锁 | 替代 `lock()` / `unlock()` |
| `std::unique_ptr` | 堆对象（唯一） | 替代 `new` / `delete` |
| `std::shared_ptr` | 堆对象（共享） | 引用计数管理 |

## 五、智能指针与所有权

### 5.1 为什么要智能指针

裸指针的问题：你不知道谁负责 `delete`。

> **什么是裸指针？**
>
> **裸指针（raw pointer）** 就是直接用 `*` 声明的普通指针，比如 `int*`、`float*`、`FILE*`。它**只是存一个地址**，不附带任何额外信息——不知道指向的对象多大、不知道该不该释放、不知道谁来释放。
>
> ```cpp
> int* p = new int(42);     // p 是裸指针，指向堆上的 int
> int x = 10;
> int* q = &x;              // q 是裸指针，指向栈上的 int
> ```
>
> **裸指针 vs 智能指针对比**：
>
> | 特性 | 裸指针 `int*` | 智能指针 `unique_ptr<int>` |
> |------|-------------|--------------------------|
> | 存什么 | 只存地址 | 存地址 + 所有权信息 |
> | 谁负责 delete | **不知道！** 要靠人记 | 自动 delete（离开作用域） |
> | 会不会忘记 delete | 很容易忘 | 不会（RAII 自动处理） |
> | 能不能知道是否有效 | 不能（可能悬空） | 能（unique_ptr 为空就是没有） |
> | 内存开销 | 零（就是一个地址） | 几乎零（unique_ptr 和裸指针一样大） |
>
> **裸指针的典型用法**（仍然合理的场景）：
> ```cpp
> // ① 函数参数：只借用，不拥有（这种用法 OK）
> void print(int* p) {
>     if (p) std::cout << *p << '\n';  // 只是读一下，不管 delete
> }
>
> // ② 指向栈变量的指针（不需要 delete）
> int x = 42;
> int* p = &x;  // p 指向栈变量，栈变量离开作用域自动销毁，不用 delete
>
> // ③ C 风格 API 返回的指针（如 malloc 的结果，要用 free 释放）
> int* arr = (int*)malloc(100 * sizeof(int));
> free(arr);  // C 风格，用 free 不是 delete
> ```
>
> **裸指针出问题的场景**（智能指针要解决的）：
> ```cpp
> // 问题 1：忘记 delete → 内存泄漏
> int* p = new int(42);
> // ... 用完忘了 delete
> // 这块内存永远不释放
>
> // 问题 2：delete 后继续使用 → 悬空指针
> int* p = new int(42);
> delete p;
> *p = 100;  // ❌ UB！p 指向的内存已释放
>
> // 问题 3：delete 两次 → 崩溃
> int* p = new int(42);
> delete p;
> delete p;  // ❌ 重复释放
>
> // 问题 4：所有权不明（最核心问题）
> int* p = new int(42);
> process(p);  // process 会 delete 吗？不知道！
> // 如果 process delete 了，这里再用 p 就崩
> // 如果 process 没 delete，这里不 delete 就泄漏
> ```

```cpp
// 谁该 delete p？调用者还是函数？
void process(int* p) {
    std::cout << *p << '\n';
    // delete p;  ← 该不该在这里 delete？
}

int main() {
    int* p = new int(42);
    process(p);
    // delete p;  ← 还是该在这里 delete？
    // 如果 process 里删了，这里再删就崩了
}
```

智能指针解决了这个问题：**类型即文档**——看到 `unique_ptr` 就知道所有权在它手里，看到 `shared_ptr` 就知道多个地方共享。

### 5.2 三种智能指针

```cpp
#include <memory>
#include <iostream>
#include <vector>   // 修复1：补充vector头文件

struct Buffer {
    Buffer(std::size_t n) : data(n) { std::cout << "Buffer 创建\n"; }
    ~Buffer() { std::cout << "Buffer 销毁\n"; }
    std::vector<float> data;
};
```

> **新人疑问：为什么用 `std::size_t` 而不是 `int`？**
>
> **核心原因**：`std::size_t` 是专门用来表示"大小/个数/索引"的**无符号整数**类型，比 `int` 更安全也更准确。
>
> **① 容量匹配——能表示更大的范围**
>
> | 类型 | 64 位系统大小 | 范围 | 能表示的最大数组大小 |
> |------|-------------|------|-------------------|
> | `int` | 4 字节 | -21 亿 ~ 21 亿 | 21 亿 |
> | `std::size_t` | **8 字节** | 0 ~ 1844 京 | 1844 京 |
>
> `std::vector::size()` 返回的就是 `std::size_t`。如果要存 30 亿个 `float`（约 12GB），`int` 装不下，`std::size_t` 可以。
>
> **② 语义匹配——"大小"不可能是负数**
>
> ```cpp
> Buffer(std::size_t n);   // n 是元素个数，永远 ≥ 0，用无符号类型 ✅
> Buffer(int n);           // n 理论上可能是 -5？毫无意义 ❌
> ```
>
> `std::size_t` 是**无符号**的，类型本身就在声明"这个值不可能是负数"。
>
> **③ 避免警告——和标准库 API 对齐**
>
> ```cpp
> int n = 1024;
> std::vector<float> data(n);   // ⚠️ 编译器警告：int → size_t 隐式转换
>                             // 如果 n 是负数，会变成巨大的无符号数！
> 
> std::size_t n = 1024;
> std::vector<float> data(n);   // ✅ 类型完全匹配，无警告
> ```
>
> **④ 避免比较时的坑**
>
> ```cpp
> std::vector<int> v = {1, 2, 3};
> 
> // ❌ 危险：int 和 size_t 比较
> for (int i = 0; i < v.size(); ++i) {  // 警告：有符号/无符号比较
>  std::cout << v[i];
> }
> 
> // ✅ 正确
> for (std::size_t i = 0; i < v.size(); ++i) {  // 类型一致
>  std::cout << v[i];
> }
> ```
>
> **最坑的场景——无符号下溢**：
>
> ```cpp
> std::vector<int> v;       // 空数组
> int n = v.size() - 1;     // ⚠️ 灾难！
> // v.size() = 0（size_t 无符号）
> // 0 - 1 = 18446744073709551615（无符号下溢）
> // 赋给 int 后变成 -1（如果幸运）
> ```
>
> **选择决策表**：
>
> | 场景 | 推荐 | 原因 |
> |------|------|------|
> | 数组/容器大小 | `std::size_t` | 和标准库对齐 |
> | 数组索引 | `std::size_t` | 和 `.size()` 类型一致 |
> | 内存字节数 | `std::size_t` | `malloc`/`sizeof` 都返回 `size_t` |
> | 可能是负数的值 | `int` | 比如温度、坐标差 |
>
> **记忆口诀**：**"大小用 size_t，正负用 int"**——凡是表示"多少个""多大"的用 `std::size_t`，可能为负的用 `int`。

**① `std::unique_ptr`——唯一所有权（默认首选）**

```cpp
void use_unique() {
    // 创建
    auto buf = std::make_unique<Buffer>(1024);  // buf 独占这个 Buffer

    // 使用
    buf->data[0] = 1.0f;

    // 不能复制
    // auto buf2 = buf;  // ❌ 编译错误！unique_ptr 不能复制

    // 可以移动（转移所有权）
    auto buf2 = std::move(buf);  // buf2 接管，buf 变成 nullptr

    // buf 现在是空的
    if (buf == nullptr) {
        std::cout << "buf 已经转移了\n";
    }
}  // buf2 离开作用域 → 自动 delete Buffer ✅
```

**② `std::shared_ptr`——共享所有权**

```cpp
void use_shared() {
    auto a = std::make_shared<Buffer>(512);  // 引用计数 = 1

    {
        auto b = a;  // 复制，引用计数 = 2
        std::cout << "引用计数: " << a.use_count() << '\n';  // 2
    }  // b 离开作用域，引用计数 = 1，不销毁

    std::cout << "引用计数: " << a.use_count() << '\n';  // 1
}  // a 离开作用域，引用计数 = 0 → 自动 delete Buffer ✅
```

**③ `std::weak_ptr`——不增加引用计数的观察者**

```cpp
void use_weak() {
    auto shared = std::make_shared<Buffer>(256);
    std::weak_ptr<Buffer> weak = shared;  // 不增加引用计数

    // 使用前要 lock() 检查是否还有效
    if (auto locked = weak.lock()) {  // locked 是 shared_ptr
        std::cout << "Buffer 还活着，大小: " << locked->data.size() << '\n';
    } else {
        std::cout << "Buffer 已被销毁\n";
    }

    shared.reset();  // 手动释放 shared_ptr，引用计数 = 0 → Buffer 销毁

    // 再检查
    if (auto locked = weak.lock()) {
        std::cout << "还活着\n";
    } else {
        std::cout << "Buffer 已被销毁\n";  // ← 走这里
    }
}
```

### 5.3 智能指针对比表

| 类型 | 语义 | 能复制吗 | 成本 | 适用场景 |
|------|------|---------|------|---------|
| `unique_ptr<T>` | 唯一拥有 | ❌ 只能移动 | 接近裸指针（零开销） | **默认首选** |
| `shared_ptr<T>` | 共享拥有 | ✅ | 有引用计数+控制块开销 | 多处需要持有同一对象 |
| `weak_ptr<T>` | 不拥有的观察者 | ✅ | 轻，配合 shared_ptr 使用 | 打破循环引用、缓存 |
| `T*` | 通常不拥有 | ✅ | 零 | 函数参数（只读/借用） |
| `T&` | 不拥有的别名 | ❌ | 零 | 函数参数（必须存在） |

### 5.4 循环引用问题

```cpp
#include <memory>

struct Node {
    std::shared_ptr<Node> next;  // 如果两个 Node 互相引用...
    ~Node() { std::cout << "Node 销毁\n"; }
};

void memory_leak() {
    auto a = std::make_shared<Node>();  // a 引用计数 = 1
    auto b = std::make_shared<Node>();  // b 引用计数 = 1

    a->next = b;  // b 引用计数 = 2
    b->next = a;  // a 引用计数 = 2  ← 循环引用！
}
// a 离开作用域 → a 引用计数 = 1（b->next 还指着 a）
// b 离开作用域 → b 引用计数 = 1（a->next 还指着 b）
// 两个都不会被销毁 → 内存泄漏！

// 解决方案：用 weak_ptr 打破循环
struct NodeFixed {
    std::shared_ptr<NodeFixed> next;       // 强引用
    std::weak_ptr<NodeFixed> prev;         // 弱引用（不增加计数）
};
```

> **常见误区：定义了 `weak_ptr` 但还是泄漏？**
>
> 新手常犯的错误：以为"只要结构体里有 `weak_ptr` 成员就自动防泄漏"，但实际还是用 `shared_ptr` 互相引用。
>
> **❌ 错误用法（还是泄漏）**：
> ```cpp
> struct NodeFixed {
>     std::shared_ptr<NodeFixed> next;
>     std::weak_ptr<NodeFixed> prev;  // 定义了但没用！
> };
>
> void memory_no_leak_wrong() {
>     auto a = std::make_shared<NodeFixed>();
>     auto b = std::make_shared<NodeFixed>();
>
>     a->next = b;   // shared_ptr：b 计数 = 2
>     b->next = a;   // ⚠️ 还是 shared_ptr！和 Node 一模一样，循环引用！
>                     // prev 根本没用，还是泄漏
> }
> ```
>
> **✅ 正确用法（一个方向用 shared，另一个方向用 weak）**：
> ```cpp
> void memory_no_leak_fixed() {
>     auto a = std::make_shared<NodeFixed>();  // a 计数 = 1
>     auto b = std::make_shared<NodeFixed>();  // b 计数 = 1
>
>     a->next = b;   // shared_ptr：a 持有 b → b 计数 = 2
>     b->prev = a;   // weak_ptr：b 弱引用 a → a 计数不变 = 1 ✅
>                     // weak_ptr 不增加引用计数！
>
>     std::cout << a.use_count() << '\n';  // 1（只有局部变量 a）
>     std::cout << b.use_count() << '\n';  // 2（局部变量 b + a->next）
> }
> // 函数结束：
> // 局部 a 销毁 → a 计数 1→0 → a 销毁 ✅ → a->next 释放 → b 计数 2→1
> // 局部 b 销毁 → b 计数 1→0 → b 销毁 ✅
> // 两个都销毁，无泄漏！
> ```
>
> **核心认知**：`weak_ptr` 不是"定义了就自动防泄漏"，而是**需要你正确使用**——把循环中的一条边改成 `weak_ptr`，而不是继续用 `shared_ptr`。
>
> | 用法 | 结果 |
> |------|------|
> | `a->next = b; b->next = a;`（两都用 shared） | ❌ 循环引用，泄漏 |
> | `a->next = b; b->prev = a;`（一强一弱） | ✅ 打破循环，无泄漏 |
>
> **典型场景**：双向链表、树结构的父指针。规则是"父→子用 `shared_ptr`，子→父用 `weak_ptr`"。

> **原则**：如果所有权没有共享需求，不要使用 `shared_ptr`。"避免思考所有权"不是共享所有权的合理理由。

## 六、拷贝、移动与 Rule of Zero

### 6.1 拷贝 vs 移动

```cpp
#include <vector>
#include <iostream>

std::vector<float> make_buffer() {
    std::vector<float> values(1'000'000, 0.0f);  // 100万个 float
    return values;  // 编译器通常消除复制（RVO），必要时可移动
}

int main() {
    auto a = make_buffer();  // a 拥有这 100万个 float

    auto b = std::move(a);   // 移动：b 接管内部数据，a 变成空壳

    std::cout << "a.size() = " << a.size() << '\n';   // 0（a 被掏空了）
    std::cout << "b.size() = " << b.size() << '\n';   // 1000000
}
```

**图解拷贝 vs 移动**：

```
拷贝（copy）：                 移动（move）：
  a: [1,2,3,4,...]              a: [1,2,3,4,...]
         │                            │
         ▼ 复制所有数据                ▼ 只改指针指向
  b: [1,2,3,4,...]              b: [1,2,3,4,...]
                                a: []  ← a 被掏空
  成本：O(n)                    成本：O(1)
```

### 6.2 Rule of Zero

> **如果你只使用标准库的 RAII 类型（vector, string, unique_ptr 等）作为成员变量，就不需要手写析构函数、拷贝构造、移动构造——编译器会自动生成正确的版本。这就是 Rule of Zero。**

```cpp
#include <vector>
#include <string>
#include <memory>

// ✅ Rule of Zero：不需要手写任何特殊函数
class GoodClass {
public:
    GoodClass(std::size_t n) : data_(n), name_("default") {}

private:
    std::vector<float> data_;     // vector 自己管内存
    std::string name_;            // string 自己管内存
    std::unique_ptr<int> extra_;  // unique_ptr 自己管堆对象
};
// 析构？编译器自动调用 data_、name_、extra_ 的析构
// 拷贝？编译器自动逐成员拷贝
// 移动？编译器自动逐成员移动
// 完美！

// ❌ 需要手写的场景：直接管理了原始资源
class RawClass {
public:
    RawClass(std::size_t n) : data_(new float[n]), size_(n) {}
    ~RawClass() { delete[] data_; }  // 必须手写析构

    // 还需要手写：拷贝构造、拷贝赋值、移动构造、移动赋值
    // 因为编译器不知道怎么正确复制/移动你的裸指针

private:
    float* data_;
    std::size_t size_;
};
```

### 6.3 不要乱用 std::move

```cpp
// ❌ 不要对返回值写 std::move
std::vector<int> make() {
    std::vector<int> v = {1, 2, 3};
    return std::move(v);  // 反而可能阻止返回值优化（RVO）！
}
// ✅ 直接 return
std::vector<int> make() {
    std::vector<int> v = {1, 2, 3};
    return v;  // 编译器自动 RVO 或移动
}

// ⚠️ 移动后不要假设源对象还有原内容
auto a = std::make_unique<int>(42);
auto b = std::move(a);
// a 现在是 nullptr，不要再用 *a
```

## 七、const：把不可修改写进接口

### 7.1 const 与指针的三种组合

```cpp
int value = 42;
int another = 99;

const float* p1;          // 指向 const float：不能通过 p1 改值
                          // ✅ p1 = &another;  ← 可以改指向
                          // ❌ *p1 = 3.14;     ← 不能改值

float* const p2 = nullptr; // const 指针：不能让 p2 改指向
                          // ❌ p2 = &another;  ← 不能改指向
                          // ✅ *p2 = 3.14;     ← 可以改值（如果 p2 有效）

const float* const p3 = nullptr;  // 两个都 const：都不能改
                          // ❌ p3 = &another;
                          // ❌ *p3 = 3.14;
```

**记忆口诀**：**const 在 `*` 左边修饰值，在 `*` 右边修饰指针**。

```
const float* p1;    // const 在 * 左边 → 不能改 *p1 的值
float* const p2;    // const 在 * 右边 → 不能改 p2 的指向
```

### 7.2 const 与成员函数

```cpp
class TensorView {
public:
    TensorView(std::size_t n) : size_(n) {}

    // const 成员函数：承诺不修改对象状态
    std::size_t size() const { return size_; }
    //                    ^^^^^ 这个 const 表示"这个函数不会修改对象"

    // 非 const 成员函数：可以修改对象状态
    void resize(std::size_t n) { size_ = n; }

private:
    std::size_t size_;
};

int main() {
    const TensorView tv(10);     // const 对象
    std::cout << tv.size() << '\n';  // ✅ 可以调用 const 成员函数
    // tv.resize(20);            // ❌ 不能调用非 const 成员函数

    TensorView tv2(10);          // 非 const 对象
    tv2.resize(20);              // ✅ 可以调用
}
```

### 7.3 const 的价值

| 价值 | 说明 |
|------|------|
| **编译器帮你检查** | 如果 const 函数里不小心改了成员变量，编译器报错 |
| **接口意图清晰** | `const T&` 参数告诉调用方"我不会改你的数据" |
| **性能提示** | 某些场景编译器可以更好优化 const 代码 |

> **注意**：const 有助于编译器和读者验证接口，但**不能自动解决并发安全**与底层别名问题。多线程下即使都是 const 也需要加锁。

## 八、未定义行为（UB）为什么危险

### 8.1 什么是未定义行为

**未定义行为（Undefined Behavior，UB）**：C++ 标准说"这种情况的行为没有定义"，编译器可以做任何事情——包括不做任何事。

常见 UB：

| UB 类型 | 代码示例 | 为什么危险 |
|---------|---------|-----------|
| 数组越界 | `values[4]` 当 values 只有 4 个元素 | 可能不报错，但读到垃圾数据 |
| 悬空引用 | 引用了已销毁的局部变量 | 可能读到旧数据或垃圾 |
| 有符号整数溢出 | `INT_MAX + 1` | 编译器可能假设不溢出，据此优化出错误结果 |
| 数据竞争 | 多线程不加锁读写同一变量 | 结果不可预测 |
| 使用未初始化变量 | `int x; std::cout << x;` | 读到任意值 |
| `delete` 后使用 | `delete p; *p = 5;` | 可能崩溃，可能静默出错 |

### 8.2 UB 为什么不是"报错"那么简单

```cpp
#include <vector>

int main() {
    std::vector<int> values(4);  // 4 个元素，下标 0-3
    values[4] = 1;               // ❌ 越界！UB！

    // 可能的结果：
    // 1. 程序正常继续运行（静默写入了不该写的内存）
    // 2. 程序崩溃（Segmentation Fault）
    // 3. 调试版正常，Release 版悄悄算错（编译器优化后行为不同）
}
```

**为什么调试版和 Release 版行为不同**：

编译器可以假设"UB 永远不会发生"，并据此做优化。例如：

调试版本：

```cpp
int* p = nullptr;
if (p != nullptr) {
    *p = 42; 
}
```

然后编译器就会认为：

```cpp
int* p = nullptr;
if (p != nullptr) {
    *p = 42;  // 编译器看到这行，认为 p 不会是 nullptr（否则是 UB）
}
// 优化后：编译器可能直接删除 if 判断，认为"p 肯定不是 nullptr"
```

然后会导致Release版本是这样的：

```cpp
int* p = nullptr;
*p = 42;
```

就会导致错误！

> **新人疑问：编译器会删掉正确的代码？这不是制造 bug 吗？**
>
> 不是"删掉正确的代码"，而是**编译器认为你写的代码不可能执行到 UB 分支**，所以把检查那个分支的判断删掉。问题在于：你的代码**本来就有 UB**，编译器的优化让 UB 的后果更明显了。
>
> **核心逻辑**（编译器的推理过程）：
>
> ```
> ① 你的代码里有 *p（解引用 p）
> ② 解引用 nullptr 是 UB
> ③ C++ 标准说："UB 不会发生"（程序员保证）
> ④ 所以 p 肯定不是 nullptr
> ⑤ 那 if (p != nullptr) 这个判断永远为 true
> ⑥ 永远为 true 的判断是多余的 → 删掉
> ```
>
> **具体示例一：空指针检查被删**
>
> ```cpp
> int* p = get_pointer();  // 可能返回 nullptr
> if (p != nullptr) {      // 你写的安全检查
>     *p = 42;
> }
> // 你以为：p 是 nullptr 时跳过赋值，安全
> // 编译器想：*p 不会是 UB → p 肯定不是 nullptr → if 没用 → 删掉
> // 优化后：*p = 42;  // 直接赋值！
> // 如果 p 实际是 nullptr → 崩溃！
> ```
>
> Debug 版可能保留 if 判断（不优化），所以能正常跳过；Release 版优化后删掉 if，直接崩。
>
> **具体示例二：数组越界检查被删**
>
> ```cpp
> int arr[10];
> int i = get_index();           // 可能返回 100
> if (i >= 0 && i < 10) {        // 你写的越界检查
>     arr[i] = 42;               // 如果越界，arr[i] 是 UB
> }
> // 编译器想：arr[i] 不会是 UB → i 肯定在 [0,10) → if 没用 → 删掉
> // 优化后：arr[i] = 42;  // 直接访问！
> // 如果 i 实际是 100 → 越界写入，可能崩或损坏其他内存
> ```
>
> **具体示例三：有符号整数溢出**
>
> ```cpp
> int x = INT_MAX;
> if (x + 1 < x) {        // 你以为溢出后会变小，检查溢出
>     handle_overflow();   // 处理溢出
> }
> // 编译器想：有符号溢出是 UB → 不会发生 → x+1 永远 >= x
> // 所以 if 永远为 false → 整个 if 块删掉
> // 优化后：什么都不做，溢出处理代码被删了！
> ```
>
> **Debug 版 vs Release 版为什么行为不同**：
>
> | 版本 | 优化级别 | if 判断 | 行为 |
> |------|---------|---------|------|
> | Debug (`-O0`) | 不优化 | 保留 | 按你写的执行，if 起作用 |
> | Release (`-O2`/`-O3`) | 优化 | 删掉 | 直接执行，UB 后果暴露 |
>
> **总结**：
> - 编译器**不是在制造 bug**，而是**暴露你代码里本来就有的 bug**
> - 你的代码有 UB，Debug 版碰巧"看起来正常"，Release 版优化后崩溃
> - 解决方案：**不要写 UB 代码**，用 Sanitizer 检测

### 8.3 用 Sanitizer 检测 UB

编译时加 `-fsanitize` 选项，运行时自动检测 UB：

```bash
# 编译时启用 AddressSanitizer + UndefinedBehaviorSanitizer
g++ -std=c++20 -Wall -Wextra -Wpedantic \
    -fsanitize=address,undefined -g main.cpp -o app

./app
# 如果有越界/UB，会输出详细错误信息
```

| Sanitizer | 检测什么 | 用法 |
|-----------|---------|------|
| AddressSanitizer (ASan) | 越界、use-after-free、双重释放 | `-fsanitize=address` |
| UndefinedBehaviorSanitizer (UBSan) | 有符号溢出、空指针解引用、整数除零 | `-fsanitize=undefined` |
| ThreadSanitizer (TSan) | 数据竞争（多线程不加锁） | `-fsanitize=thread` |

> **注意**：ASan 和 TSan **不能同时启用**（冲突）。调试时先用 ASan+UBSan，再单独用 TSan 测多线程。

### 8.4 完整调试示例

```cpp
// bug.cpp —— 故意制造 UB 来测试 Sanitizer

#include <iostream>

    // Bug 1: 数组越界
    std::vector<int> v(4);

    int* p = new int(10);

    std::cout << *p << '\n';  // 使用已释放的内存

    // Bug 3: 有符号整数溢出
    int max_val = 2147483647;  // INT_MAX
    int overflow = max_val + 1;  // UB！
    std::cout << overflow << '\n';
}
```

## 九、概念速查表

| 概念 | 一句话记忆 |
|------|------------|
| 指针 `T*` | 存地址，可以为空，可以改指向 |
| 引用 `T&` | 别名，不能为空，不能改指向 |
| `const T&` | 只读别名，避免复制，函数参数最常用 |
| 数组退化 | 数组传参会退化为指针，丢失长度 |
| `std::span` | 指针+长度的轻量视图（C++20） |
| 栈 | 自动管理，快，小，离开作用域自动销毁 |
| 堆 | 手动管理，慢，大，必须 delete |
| RAII | 构造获取资源，析构释放资源，异常安全 |
| `unique_ptr` | 唯一所有权，默认首选，零开销 |
| `shared_ptr` | 共享所有权，引用计数，有开销 |
| `weak_ptr` | 不增加计数的观察者，打破循环引用 |
| 拷贝 | 复制数据，O(n) |
| 移动 `std::move` | 转移所有权，O(1)，源对象被掏空 |
| Rule of Zero | 只用 RAII 成员就不需要手写析构/拷贝/移动 |
| `const` 成员函数 | 承诺不修改对象状态 |
| UB（未定义行为） | 越界、悬空指针、溢出等，编译器可假设不发生 |
| AddressSanitizer | 检测越界、use-after-free |
| UBSan | 检测整数溢出等未定义行为 |

## 十、现代 C++ 内存管理决策树

```
你需要一个对象/数据
│
├─ 大小固定且小（int, float 等）→ 直接用值，按值传递
│
├─ 动态数组 → std::vector（RAII，自动管理）
│
├─ 字符串 → std::string（RAII，自动管理）
│
├─ 单个堆对象
│   ├─ 只有一个拥有者 → std::unique_ptr
│   └─ 多个拥有者 → std::shared_ptr + std::weak_ptr 打破循环
│
├─ 函数参数（不拥有）
│   ├─ 必须存在且只读 → const T&
│   ├─ 必须存在且要修改 → T&
│   ├─ 可能为空 → const T*（或 T*）
│   └─ 连续数组 → std::span<T>（C++20）
│
└─ 管理外部资源（文件、锁、GPU内存）
    └─ 自己写 RAII 类，或用标准库的（fstream, lock_guard 等）
```

#### 第 2 块：访问详情

```

| 字段 | 含义 |
|------|------|
| `WRITE` | 是**写**操作（不是读）。对应代码 `v[4] = 42;` 中的赋值 |

| `at 0x502000000020` | 写入的目标地址 |
| `thread T0` | 主线程（`T0` = thread 0，主线程） |

**这一行告诉你**：程序试图往一个地址**写入 4 字节**（一个 int），而这个地址是越界的。

> **怎么判断是读还是写？**
> - `WRITE of size N` → 写越界（比如 `v[4] = 42;`）
> - `READ of size N` → 读越界（比如 `int x = v[4];`）
> 知道是读还是写，能帮你快速在代码里找到出错的那一行。

#### 第 4 块：内存位置关系（越界多少）

```
0x502000000020 is located 0 bytes after 16-byte region [0x502000000010,0x502000000020)
```

这一行信息量很大，逐段拆：

**① `0x502000000020`**：被越界访问的地址（和第 1、2 块里的地址一致）。

**② `is located 0 bytes after`**：位于……之后 0 字节。**"0 bytes after"** 的意思是——**紧挨着有效区域的末尾，第 0 字节**（即第一个越界字节）。

| 越界字节数 | 报告怎么说 | 含义 |
|-----------|-----------|------|
| 0 bytes after | 紧挨着末尾，第 1 个越界字节 | `v[4]`（越界 1 个元素） |
| 4 bytes after | 末尾之后第 4 字节 | `v[5]`（越界 2 个元素） |
| 8 bytes after | 末尾之后第 8 字节 | `v[6]`（越界 3 个元素） |

**③ `16-byte region`**：分配了 16 字节的合法内存。

**为什么是 16 字节？** 因为 `std::vector<int> v(4);` 分配了 4 个 `int`，每个 `int` 4 字节，`4 × 4 = 16` 字节。

**④ `[0x502000000010, 0x502000000020)`**：合法内存的地址区间，**左闭右开**。

- 起始地址 `0x502000000010`（`v[0]` 的地址）
- 结束地址 `0x502000000020`（`v[4]` 的地址，但这个地址已经越界了）
- 区间是左闭右开 `[)`，所以合法地址是 `0x010` ~ `0x01C`（`v[0]`~`v[3]`），`0x020` 不在合法范围内

**画个图**：

```
地址:    0x010  0x014  0x018  0x01C  | 0x020

                                     (0x502000000020)

`v[4]` 的地址正好是 `0x502000000020`，它**紧挨在合法区域末尾之后**（`0 bytes after`），所以 ASan 立刻报错。

> **ASan 怎么知道 `0x020` 是越界的？**
>
> ASan 在你每块分配的内存周围，会偷偷插入一些"陷阱字节"（叫 redzone，红区）。任何对红区的访问都会被立刻抓住。这就像图书馆在每本书前后各放一张"禁止触碰"的红标签，谁碰到红标签就报警。
>
> 也就是这里蓝色[fa]，可以推算出 箭头指向的 第一个 fa的地址是 0x50200000000, 第一个00的地址是0x50200000010，蓝色[fa]地址是 0x50200000020，以此类推 第二行的[fa]正好是 0x50200000080
>

**什么是 Shadow bytes（影子内存）？**

ASan 把你的程序内存**按 8 字节一组**映射到一张"影子表"上。每个应用字节对应 1/8 个影子字节，或者反过来说：**1 个影子字节描述 8 个应用字节的状态**。

**类比**：想象图书馆每 8 本书一组，门口贴一张小标签：
- 标签写 `00` → 这 8 本都是合法的，可以借
- 标签写 `fa` → 这 8 本是"红区"（陷阱），碰了就报警
- 标签写 `fd` → 这 8 本已经销毁了（被 delete 了），不能再碰

**怎么读上面那张表**：

1. 每行前面是**应用内存的起始地址**（如 `0x502000000000`）
2. 后面 16 个两字符就是**影子字节**，每个描述 8 字节应用内存
3. 所以一行描述 `16 × 8 = 128 字节`的应用内存
4. **`=>` 箭头**指向出错地址所在的那一行
5. **`[fa]` 加方括号**的就是出错位置对应的影子字节

**重点看 `=>` 那一行**：

```
=>0x502000000000: fa fa 00 00[fa]fa fa fa fa fa fa fa fa fa fa fa
```

把这 16 个影子字节和它描述的应用内存对应起来：

```
影子字节:   fa    fa    00    00    [fa]   fa    fa    ...
应用地址:   0x000 0x008 0x010 0x018  0x020  0x028 0x030 ...
含义:       红区  红区  合法  合法   红区!  红区  红区
对应 v:                 v[0..1] v[2..3]  ↑
                                  v[4] 越界写这里
```

**解读**：
- `0x010`~`0x01F`（影子字节 `00 00`）：合法区域，就是 `v[0]`~`v[3]` 所在的 16 字节
- `0x000`~`0x00F`（影子字节 `fa fa`）：**左侧红区**，分配内存时 ASan 在前面留的陷阱
- `0x020`~`0x027`（影子字节 `[fa]`）：**右侧红区**，`v[4]` 越界写到的就是这个红区 → 报警！

**所以 ASan 的工作原理**：
1. 每次 `new` 分配内存时，ASan 在前后各加一段红区（标记为 `fa`）
2. 同时建立影子表，记录每个 8 字节组的状态
3. 程序每次访问内存，ASan 先查影子表，如果对应影子字节是 `fa`/`fd` 等非法标记，立刻报错
4. 这样越界、use-after-free 都能被当场抓住，不用等程序崩溃

> **为什么 `00` 表示合法、`fa` 表示红区？**
>
> 没有特别的原因，就是 ASan 设计者选的编码。`fa` 是 "fence"（栅栏）的缩写，`fd` 是 "freed"（已释放），`f1`/`f2`/`f3` 是栈相关红区。完整的编码表就是下面第 8 块那张图例。

#### 第 8 块：图例 + ABORTING

```
Shadow byte legend (one shadow byte represents 8 application bytes):
  Addressable:           00
  Partially addressable: 01 02 03 04 05 06 07
  Heap left redzone:       fa
  Freed heap region:       fd
  Stack left redzone:      f1
  Stack mid redzone:       f2
  Stack right redzone:     f3
  Stack after return:      f5
  Stack use after scope:   f8
  Global redzone:          f9
  Global init order:       f6
  Poisoned by user:        f7
  Container overflow:      fc
  Array cookie:            ac
  Intra object redzone:    bb
  ASan internal:           fe
  Left alloca redzone:     ca
  Right alloca redzone:    cb
==3794==ABORTING
```

**图例（legend）**：把上面 Shadow bytes 表里出现的两字符编码翻译成人话。

**新手需要认识的几个**：

| 编码 | 含义 | 什么时候出现 |
|------|------|------------|
| `00` | 完全可访问 | 正常分配的内存 |
| `01`~`07` | 部分可访问（前 N 字节合法） | 内存对齐时末尾的"边角料" |
| `fa` | **堆红区**（fence） | `new` 出来的内存前后 ← 本例就是这个 |
| `fd` | **已释放的堆内存**（freed） | `delete` 后再访问 → use-after-free |
| `f1`~`f3` | 栈红区 | 局部变量越界 |
| `f9` | 全局变量红区 | 全局/静态数组越界 |

**最后一行**：

```
==3794==ABORTING
```

程序被 ASan 强制中止（调用 `abort()`），进程退出码非 0。**这就是为什么 Bug 2 和 Bug 3 没有被报出来**——见下一节。

### A.4 为什么只报了 Bug 1，Bug 2 和 Bug 3 哪去了？

源码里有三个 bug：

```cpp
// Bug 1: 数组越界
std::vector<int> v(4);
v[4] = 42;  // ← 第 8 行 ← ASan 在这里就 abort 了！

// Bug 2: use-after-free
int* p = new int(10);
delete p;
std::cout << *p << '\n';  // ← 第 13 行 ← 没机会执行

// Bug 3: 有符号整数溢出
int max_val = 2147483647;
int overflow = max_val + 1;  // ← 第 16 行 ← 没机会执行
```

**ASan 默认行为**：发现第一个错误就立刻 `abort()`，**不继续往下执行**。所以 Bug 2 和 Bug 3 根本没运行到，自然也不会被报告。

**怎么把所有 bug 都报出来？** 有两个办法：

**方法一：修一个跑一次**

最朴素——把 Bug 1 修掉（`v[4]` 改成 `v[3]`），重新编译运行，这次会报 Bug 2；再修 Bug 2，再跑，这次报 Bug 3。

**方法二：开启 `halt_on_error=0`**

让 ASan 报错但**不退出**，继续往下跑，把所有错误一次性全报出来：

```bash
# 编译时加 ASAN_OPTIONS 环境变量
ASAN_OPTIONS=halt_on_error=0:abort_on_error=0 ./bug_san
```

| 选项 | 含义 |
|------|------|
| `halt_on_error=0` | 发现错误不停止，继续执行 |
| `abort_on_error=0` | 不调用 `abort()`（否则即使 halt_on_error=0 也会退出） |

> ⚠️ 注意：`halt_on_error=0` 时，Bug 2（use-after-free）这种错误继续执行可能引发更奇怪的崩溃，因为内存已经被破坏了。**推荐还是用方法一**——修一个跑一次，最稳。

### A.5 阅读报告的"三步法"（总结）

拿到一份 ASan 报告，按这三步走：

```
第 1 步：找 SUMMARY 行（最快）
    SUMMARY: AddressSanitizer: heap-buffer-overflow /home/lenck/test/bug.cpp:8 in main
    ──────────────────────────  ────────────────────────  ────────────────────
    错误类型                    文件名:行号              函数名

第 2 步：看 ERROR 行的细节
    ==3794==ERROR: AddressSanitizer: heap-buffer-overflow on address 0x...
    WRITE of size 4 at 0x... thread T0
    ─────                ─────
    是写还是读           多大（4 字节 = 1 个 int）

第 3 步：看 located X bytes after
    0x... is located 0 bytes after 16-byte region [..., ...)
    ──────────────  ──────────────
    越界多少        合法区域多大
```

**三步走完，你已经知道**：
- 什么错误（heap-buffer-overflow）
- 在哪一行（bug.cpp:8）
- 是读还是写、多大（WRITE 4 字节）
- 越界多少（0 bytes after = 紧挨着末尾第 1 个字节）

足够你去源码里定位并修复了。

### A.6 常见 ASan 错误类型速查

| 错误类型 | 中文 | 典型代码 | 报告关键字 |
|---------|------|---------|-----------|
| `heap-buffer-overflow` | 堆越界 | `v[v.size()]` | `located X bytes after ... region` |
| `stack-buffer-overflow` | 栈越界 | `int a[4]; a[4]=0;` | `located X bytes after ... stack` |
| `heap-use-after-free` | 用了已释放内存 | `delete p; *p;` | `located X bytes inside of ... byte region [已释放]` |
| `stack-use-after-return` | 用了已返回函数的栈变量 | 返回局部变量引用 | `Shadow bytes: ... f5 ...` |
| `double-free` | 重复释放 | `delete p; delete p;` | `ERROR: AddressSanitizer: attempting double-free` |
| `alloc-dealloc-mismatch` | 分配释放不匹配 | `new[]` 但用 `delete` | `ERROR: AddressSanitizer: alloc-dealloc-mismatch` |
| `initialization-order-fiasco` | 全局对象初始化顺序问题 | 全局变量互相依赖 | `ERROR: AddressSanitizer: initialization-order-fiasco` |

> **UBSan 的报告格式不同**：UBSan 报错是 `runtime error: signed integer overflow: 2147483647 + 1 cannot be represented in type 'int'` 这种一行式描述，没有调用栈和 Shadow bytes。如果同时开了 ASan+UBSan，会先看到 UBSan 的简单报告，再看到 ASan 的详细报告。

### A.7 给新手的最后建议

1. **永远加 `-g`**：没有调试信息，ASan 报告里的行号全是 `??:0`，等于白报。
2. **先看 SUMMARY**：一句话就知道是什么错、在哪一行，比从上往下读快得多。
3. **忽略 `#1` 以后的运行时栈**：`__libc_start_call_main` / `_start` 这些是 C 运行时启动代码，跟你无关。
4. **分配栈从下往上找你的代码**：那一堆 `std::...` 中间层是标准库内部，跳过即可。
5. **Shadow bytes 看不懂没关系**：那是 ASan 的内部机制，新手能定位到行号就够用了。
6. **一次只修一个 bug**：ASan 默认第一个错误就 abort，别期望一次报完所有问题。
