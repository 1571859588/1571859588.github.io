## 一句话结论

Windows 推荐用 **MSYS2 + GCC**（最接近 Linux 体验，Sanitizer 支持好），Ubuntu 直接 `apt install`；服务器无 sudo 权限时改用 **Miniconda + conda-forge**（见 2.7 节）。配置好后验证标准：能编译 C++20、能跑 AddressSanitizer、能用 CMake。

## 零、为什么需要配置环境？

Python 装个解释器就能跑，C++ 不一样：

| 需要什么 | Python | C++ |
|---------|--------|-----|
| 运行代码 | Python 解释器 | C++ 编译器（g++/clang++/MSVC） |
| 构建工具 | pip | CMake + Make/Ninja |
| 调试工具 | traceback | GDB + Sanitizer |
| 包管理 | pip + PyPI | vcpkg / conan（可选） |

C++ 没有统一的"一键安装"，不同操作系统方案不同。

## 一、Windows 配置（推荐 MSYS2）

### 1.1 为什么选 MSYS2

Windows 上有三种主流方案：

| 方案 | 编译器 | Sanitizer 支持 | C++20 支持 | 推荐度 |
|------|--------|--------------|-----------|--------|
| **MSYS2 + GCC** | g++ (MinGW-w64) | ✅ 完整 | ✅ | ⭐⭐⭐⭐⭐ |
| Visual Studio | MSVC | ⚠️ 部分（ASan 需特定版本） | ✅ | ⭐⭐⭐ |
| WSL2 + Ubuntu | g++ | ✅ 完整 | ✅ | ⭐⭐⭐⭐ |

**推荐 MSYS2 的原因**：
- 命令行体验接近 Linux（pacman 包管理、bash 终端）
- Sanitizer 完整支持（MSVC 的 ASan 对 C++20 支持不全）
- 轻量（比 Visual Studio 小很多）
- 和笔记里的命令完全兼容（`g++`、`cmake`、`nm`、`ldd`）

> 如果你已经装了 WSL2，直接看第二节 Ubuntu 配置即可，跳过 MSYS2。

### 1.2 安装 MSYS2

**第一步：下载安装**

1. 打开 https://www.msys2.org/
2. 下载 `msys2-x86_64-xxxxxxxx.exe`
3. 安装到默认路径 `C:\msys64`（**路径不要有空格和中文**）
4. 安装完成后会自动打开 MSYS2 终端

**第二步：更新包数据库**

在 MSYS2 终端里运行：

```bash
pacman -Syu
```

如果提示关闭终端，关掉重新打开，再运行一次：

```bash
pacman -Su
```

**第三步：安装 GCC 工具链**

MSYS2 有多个环境，我们用 **UCRT64**（现代 Windows 推荐）：

```bash
pacman -S --needed \
    mingw-w64-ucrt-x86_64-gcc \
    mingw-w64-ucrt-x86_64-cmake \
    mingw-w64-ucrt-x86_64-gdb \
    mingw-w64-ucrt-x86_64-make \
    mingw-w64-ucrt-x86_64-clang
```

| 包 | 作用 |
|----|------|
| `gcc` | g++ 编译器 |
| `cmake` | 构建工具 |
| `gdb` | 调试器 |
| `make` | Make 构建工具 |
| `clang` | Clang 编译器（可选，某些 Sanitizer 更好） |

**第四步：验证安装**

打开 **UCRT64 终端**（开始菜单 → MSYS2 → UCRT64）：

```bash
g++ --version       # 应显示 g++ (Rev..., Built by MSYS2 project) 14.x.x
cmake --version     # cmake version 3.28+
gdb --version       # GNU gdb 14.x
```

如果都能显示版本号，安装成功。

### 1.3 配置 VS Code（推荐编辑器）

**第一步：安装 VS Code**

从 https://code.visualstudio.com/ 下载安装。

**第二步：装扩展**

打开 VS Code，装这两个扩展：
- **C/C++**（Microsoft 官方，提供 IntelliSense 和调试）
- **CMake Tools**（Microsoft 官方，CMake 集成）

**第三步：配置编译器路径**

1. 按 `Ctrl+Shift+P`，输入 `C/C++: Edit Configurations (UI)`
2. 在 `编译器路径` 里填：`C:/msys64/ucrt64/bin/g++.exe`
3. 在 `IntelliSense 模式` 里选：`windows-gcc-x64`

**第四步：用 VS Code 的终端**

1. 按 `Ctrl+Shift+P`，输入 `Terminal: Select Default Profile`
2. 选 `MSYS2 UCRT64`（如果没有，选 `Command Prompt`，手动加路径）

或者更简单——直接用 MSYS2 的 UCRT64 终端写命令，VS Code 只当编辑器用。

### 1.4 第一个程序验证

创建文件 `hello.cpp`：

```cpp
#include <iostream>
int main() {
    std::cout << "Hello, C++!\n";
    return 0;
}
```

在 UCRT64 终端里：

```bash
g++ -std=c++20 hello.cpp -o hello.exe
./hello.exe
# 输出：Hello, C++!
```

### 1.5 测试 Sanitizer（验证笔记里的 UB 例子）

创建 `bug.cpp`：

```cpp
#include <vector>
#include <iostream>

int main() {
    // Bug: 数组越界
    std::vector<int> v(4);
    v[4] = 42;  // 越界！下标 0-3 才有效

    // Bug: use-after-free
    int* p = new int(10);
    delete p;
    std::cout << *p << '\n';

    // Bug: 有符号整数溢出
    int max_val = 2147483647;
    int overflow = max_val + 1;
    std::cout << overflow << '\n';
}
```

```bash
# 正常编译——可能不报错（UB 静默执行）
g++ -std=c++20 bug.cpp -o bug.exe
./bug.exe
# 可能"正常"输出垃圾值

# 用 Sanitizer——立即报告错误
g++ -std=c++20 -fsanitize=address,undefined -g bug.cpp -o bug_san.exe
./bug_san.exe
# 输出详细的错误报告：
# ==12345==ERROR: AddressSanitizer: heap-buffer-overflow...
# runtime error: signed integer overflow...
```

### 1.6 测试 CMake 项目

创建项目结构：

```
test_cmake/
├── CMakeLists.txt
└── main.cpp
```

```cpp
// main.cpp
#include <iostream>
int main() {
    std::cout << "CMake works!\n";
    return 0;
}
```

```cmake
# CMakeLists.txt

project(test LANGUAGES CXX)
set(CMAKE_CXX_STANDARD 20)
add_executable(test main.cpp)
```

构建运行：

```bash
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
cmake --build build --parallel
./build/test.exe
# 输出：CMake works!
```

### 1.7 常见问题

| 问题 | 解决 |
|------|------|
| `g++: command not found` | 用 UCRT64 终端，不是默认的 MSYS2 终端 |
| `cannot find -lstdc++` | 检查是不是用了错误的终端环境 |
| 中文路径乱码 | 项目路径不要有中文和空格 |
| Sanitizer 报错但程序不崩 | 正常，ASan 默认继续执行，加 `-fno-omit-frame-pointer` 看更清楚的栈 |

## 二、Ubuntu 配置（apt 一键搞定）

### 2.1 安装编译工具链

Ubuntu 的包管理比 Windows 简单很多：

```bash
# 更新包列表
sudo apt update

# 一键安装全套
sudo apt install -y \
    build-essential \
    cmake \
    gdb \
    valgrind \
    clang
```

| 包 | 作用 |
|----|------|
| `build-essential` | 包含 g++、gcc、make |
| `cmake` | 构建工具 |
| `gdb` | 调试器 |
| `valgrind` | 内存检测（ASan 的替代/补充） |
| `clang` | Clang 编译器 |

> **⚠️ 可能遇到的报错**：执行 `sudo apt update` 时如果出现
> `E: dpkg was interrupted, you must manually run 'sudo dpkg --configure -a' to correct the problem.`
> 说明上次 apt/dpkg 被中断（WSL 常见：直接关窗口、Windows 重启没优雅关 WSL），包数据库停留在半配置状态。
> **一句话解法**：按顺序跑 `sudo dpkg --configure -a` → `sudo apt --fix-broken install -y` → `sudo apt update` → 重装目标包。
> 完整原因分析和兜底方案见 **附录 A：dpkg 被中断的修复**。

### 2.7 没有 sudo 权限怎么办（服务器/共享主机场景）

> 适用：学校服务器、公司开发机、HPC 节点——只有普通账号，无法 `sudo apt install`。
> 触发场景：在远程服务器上跑 `cpp/` 笔记代码，但 `sudo apt install` 报 `Permission denied` 或 `user is not in the sudoers file`。

**一句话结论：不一定需要 sudo。** sudo 的作用只是把包装进系统目录（`/usr/bin`、`/usr/lib` 等），完全可以装到自己的家目录（`~/.local`、`~/opt`、`~/miniconda3`），再改 `PATH` / `LD_LIBRARY_PATH` 让系统找得到。最推荐 **Miniconda + conda-forge**，一行命令搞定整套工具链；HPC 集群先试 `module load`。

#### 2.7.1 先判断是否真的无 sudo

```bash
# -n 表示非交互式：有免密 sudo 返回 0，否则返回非 0
sudo -n true 2>/dev/null && echo "有 sudo 免密" || echo "无 sudo 或需要密码"

# 或看当前用户是否在 sudo 组
groups | grep -o sudo
```

只是忘了密码、或有 sudo 但不想用，直接 sudo 装即可；下面针对"完全无 sudo"的情况。

#### 2.7.2 方案对比

| 方案 | 难度 | 版本新度 | 体积 | 适用场景 |
|------|------|---------|------|---------|
| **Miniconda + conda-forge** | ⭐ 最简 | ✅ 较新 | 大（~1GB） | 通用首选 |
| Homebrew on Linux | ⭐⭐ | ✅ 新 | 中 | 已在用 brew |
| apt download + dpkg -x | ⭐⭐ | 看系统源 | 小 | 只缺一两个包 |
| 官方预编译二进制 | ⭐⭐ | ✅ 最新 | 小 | 要指定版本（如 CMake 3.30） |
| 源码编译 GCC | ⭐⭐⭐⭐⭐ | ✅ 任意 | 大 | 极端情况，慎用（1-3 小时） |
| module load（HPC） | ⭐ | 看管理员 | 0 | 学校超算，先试这个 |

#### 2.7.3 推荐方案：Miniconda + conda-forge

**第一步：装 Miniconda（不需要 sudo，全程家目录）**

```bash
cd ~
wget https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh
# 没 wget 用 curl -O 替代：
# curl -O https://repo.anaconda.com/miniconda/Miniconda3-latest-Linux-x86_64.sh

bash Miniconda3-latest-Linux-x86_64.sh
# 交互式安装，路径默认 ~/miniconda3，全部在用户目录
```

**第二步：用 conda 装 C++ 工具链**

```bash
source ~/miniconda3/bin/activate   # 没自动激活时手动激活

# 一行装齐：g++、cmake、gdb、clang
conda install -c conda-forge gxx cmake gdb clang

# 验证
g++ --version
cmake --version
```

**第三步：可选——独立环境隔离（避免污染 base）**

```bash
conda create -n cpp -c conda-forge gxx=13 cmake gdb clang
conda activate cpp
```

> conda 装的 `g++` 在 `~/miniconda3/envs/cpp/bin/g++`，自动配好 `LD_LIBRARY_PATH`，不用手动设环境变量。

#### 2.7.4 备选方案：apt download + dpkg -x（只缺一两个包时）

`apt install` 必须写系统目录所以要 sudo，但 `apt download` 只下载 `.deb` 不安装，再用 `dpkg -x` 解压到家目录：

```bash
mkdir -p ~/local && cd /tmp
apt download g++ gcc make cmake         # 下载 .deb 到当前目录

for deb in *.deb; do dpkg -x "$deb" ~/local; done

# 加环境变量
echo 'export PATH=$HOME/local/usr/bin:$PATH' >> ~/.bashrc
echo 'export LD_LIBRARY_PATH=$HOME/local/usr/lib/x86_64-linux-gnu:$LD_LIBRARY_PATH' >> ~/.bashrc
source ~/.bashrc

g++ --version
```

> 缺点：依赖要逐个 download，`build-essential` 这种元包依赖很多、很繁琐。这种情况更推荐用 conda。

#### 2.7.5 备选方案：官方预编译二进制（要指定版本时）

**CMake 官方二进制**最常用，解压即用：

```bash
cd ~
wget https://github.com/Kitware/CMake/releases/download/v3.30.0/cmake-3.30.0-linux-x86_64.tar.gz
tar -xzf cmake-3.30.0-linux-x86_64.tar.gz -C ~/
echo 'export PATH=$HOME/cmake-3.30.0-linux-x86_64/bin:$PATH' >> ~/.bashrc
source ~/.bashrc
cmake --version   # 3.30.0
```

GCC 预编译可从 conda-forge 直接下 tarball，或参考 [GCC 官方 binaries 页面](https://gcc.gnu.org/install/binaries.html)。

#### 2.7.6 HPC 场景：先试 module load

学校超算 / 公司集群通常已装好多个版本，用 `module` 命令切换，零安装：

```bash
module avail gcc          # 看有哪些可用
module avail cmake
module load gcc/13.2.0    # 加载指定版本
module load cmake/3.28

g++ --version
which g++                 # 通常是 /opt/.../gcc-13.2.0/bin/g++
```

把 `module load` 写进 `~/.bashrc` 自动加载即可，连 conda 都不用。

#### 2.7.7 常见误区

| 误区 | 正解 |
|------|------|
| `apt install xxx` 不加 sudo 也能用 | 不行，apt 要写 `/var/cache/apt`、`/usr/bin` 等系统目录，必须 sudo |
| 没有 sudo 就装不了高版本 GCC | 错，conda / 预编译二进制 / 源码编译都行 |
| 源码编译 GCC 最通用 | 慎用，编译 1-3 小时，依赖 gmp/mpfr/mpc 等，先试 conda |
| 自己装的 g++ 找不到标准库 | 设好 `LD_LIBRARY_PATH`，或用 conda 自动管理 |

#### 2.7.8 复用判断（下次如何快速选方案）

- **学校/公司服务器无 sudo** → 直接 conda 装，5 分钟搞定
- **要装指定版本（如 GCC 14、CMake 3.30）** → conda 指定版本，或下官方二进制
- **HPC 集群** → 先 `module avail` 看有没有现成的，有就直接 load
- **完全离线机器** → 在有网机器上用 `conda-pack` 打包，传过去解压即可
- **只缺一两个包（如 cmake）** → `apt download` + `dpkg -x` 解压到家目录

## 四、环境对比与选择

| 维度 | MSYS2 (Windows 原生) | WSL2 (Windows 上的 Linux) | Ubuntu (原生 Linux) |
|------|---------------------|--------------------------|-------------------|
| 安装难度 | 中等 | 简单 | 最简单 |
| Sanitizer | ✅ 完整 | ✅ 完整 | ✅ 完整 |
| 文件系统 | Windows 原生 | 跨 Windows/Linux | Linux 原生 |
| 性能 | 原生 | 略有损耗 | 原生 |
| 适合场景 | 日常开发 | 需要完整 Linux | 服务器/CI |

**建议**：
- 日常学习用 **MSYS2**（快、原生）
- 需要完整 Linux 体验用 **WSL2**
- 生产环境用 **原生 Ubuntu**

## 五、常用编译命令速查

### 5.1 基础编译

```bash
# 单文件
g++ -std=c++20 main.cpp -o app

# 多文件
g++ -std=c++20 main.cpp add.cpp -o app

# 带优化
g++ -std=c++20 -O2 main.cpp -o app       # 优化级别 2
g++ -std=c++20 -O3 main.cpp -o app       # 优化级别 3（最快）

# 带警告
g++ -std=c++20 -Wall -Wextra -Wpedantic main.cpp -o app

# 带调试信息
g++ -std=c++20 -g main.cpp -o app        # 生成调试信息（给 gdb 用）
```

### 5.2 分阶段编译

```bash
g++ -E main.cpp -o main.ii       # 只预处理
g++ -S main.cpp -o main.s        # 生成汇编
g++ -c main.cpp -o main.o        # 生成目标文件
g++ main.o -o app                # 链接
```

### 5.3 Sanitizer

```bash
# AddressSanitizer + UBSan（测内存错误 + UB）
g++ -std=c++20 -fsanitize=address,undefined -g main.cpp -o app_san

# ThreadSanitizer（测数据竞争，单独用）
g++ -std=c++20 -fsanitize=thread -g main.cpp -o app_tsan

# MemorySanitizer（只 Clang 支持，测未初始化内存）
clang++ -std=c++20 -fsanitize=memory -g main.cpp -o app_msan
```

### 5.4 CMake

```bash
# 配置（生成构建文件）
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release

# 编译
cmake --build build --parallel

# 清理重建
rm -rf build && cmake -S . -B build -DCMAKE_BUILD_TYPE=Debug
```

### 5.5 调试

```bash
# 用 gdb 调试
g++ -std=c++20 -g main.cpp -o app    # -g 生成调试信息
gdb ./app
# (gdb) break main    # 设断点
# (gdb) run           # 运行
# (gdb) next          # 单步
# (gdb) print x       # 查看变量
# (gdb) quit          # 退出
```

## 八、附录

### 附录 A：dpkg 被中断的修复（WSL 高频问题）

> 触发位置：第二节 Ubuntu 配置，`sudo apt update` 或 `sudo apt install` 时。
> 现象：`E: dpkg was interrupted, you must manually run 'sudo dpkg --configure -a' to correct the problem.`

#### A.1 根因

dpkg（Debian 包管理底层工具）上次执行到一半被中断，导致包数据库（`/var/lib/dpkg/`）停留在"半配置"状态——有些包已经解压但配置脚本（postinst）没跑完。apt 检测到这种状态会拒绝继续，强制你先修复。

WSL 下高频触发场景：

| 场景 | 说明 |
|------|------|
| 直接关 WSL 窗口 | 窗口关闭不等于进程结束，apt 可能正在写数据库 |
| Windows 重启/关机 | 没有先 `wsl --shutdown`，WSL 被强杀 |
| 磁盘满 | `/var` 写不下，dpkg 中途失败 |
| 网络中断 | 下载到一半失败，残留半解压的包 |

#### A.2 标准修复四步法

按顺序执行，每步跑完看输出，没报错再跑下一步：

```bash
# 第 1 步：修复 dpkg 半配置状态（报错信息里指定的命令）
sudo dpkg --configure -a
# 逐个把上次没配置完的包跑完配置脚本
# 会输出一堆 "Setting up xxx ..."，正常现象，等它跑完

# 第 2 步：检查是否还有损坏的依赖关系
sudo apt --fix-broken install -y
# 上一步报依赖错误时，这步自动修复

# 第 3 步：再次更新包列表（验证 apt 恢复正常）
sudo apt update
# 这次应该不再报 "dpkg was interrupted"

# 第 4 步：重试原本要装的 C++ 工具链
sudo apt install -y build-essential cmake gdb valgrind clang
```

#### A.3 中途可能看到的良性警告（不用慌）

WSL 默认没有完整 systemd，配置脚本里尝试重启服务时会看到：

```
invoke-rc.d: could not determine current runlevel
Failed to open connection to "system" message bus:
  Failed to connect to socket /run/dbus/system_bus_socket: No such file or directory
invoke-rc.d: initscript dbus, action "force-reload" failed.
```

**这是良性的，不影响安装**。原因：WSL 没有 systemd（除非手动开启），`invoke-rc.d` 找不到 init 系统就报这几行，但包本身（g++/cmake/gdb 等）和 C++ 开发无关，配置照样完成。看到后直接忽略，等 `Setting up ...` 继续即可。

#### A.4 兜底方案（四步法搞不定时）

**情况 1：卡在某个包的 postinst 配置脚本**

某个包的配置脚本交互式提问（比如改不改配置文件）卡住：

```bash
sudo dpkg --configure -a --force-confdef --force-confold
# --force-confdef：有改动用包默认配置
# --force-confold：没默认时保留你现有的旧配置
# 组合起来 = 全部非交互式，不弹提问
```

**情况 2：磁盘满导致 dpkg 失败**

```bash
df -h /var            # 先看 /var 是不是满了
sudo apt clean        # 清理 apt 下载缓存（/var/cache/apt/archives/）
sudo rm -rf /var/cache/apt/archives/*.deb
sudo dpkg --configure -a   # 清出空间后再修复
```

**情况 3：apt/dpkg 锁文件被占用**

报 `Could not get lock /var/lib/dpkg/lock-frontend`：

```bash
ps aux | grep -E 'apt|dpkg'   # 先确认是不是真有进程在跑
# 如果有 apt/dpkg 进程在跑，等它跑完，别强删锁
# 如果确认没有进程但仍报锁占用（上次崩溃残留）：
sudo rm /var/lib/dpkg/lock-frontend
sudo rm /var/lib/dpkg/lock
sudo dpkg --configure -a
```

> ⚠️ 删锁前务必确认没有 apt/dpkg 进程在跑，否则会损坏包数据库。

#### A.5 预防

| 习惯 | 说明 |
|------|------|
| 关 WSL 窗口前确认无 apt/dpkg 进程 | `ps aux \| grep apt` 看一眼 |
| Windows 关机/重启前先 `wsl --shutdown` | 在 PowerShell 里跑，优雅关闭所有 WSL 实例 |
| 不要在 apt 跑的时候关终端 | 等命令提示符回来再关 |
| 定期 `sudo apt clean` | 避免磁盘被缓存撑满 |

#### A.6 复用判断

- `apt update` / `apt install` 报 `dpkg was interrupted` → 直接四步法
- 报 `Could not get lock` → A.4 情况 3
- 配置脚本卡住弹提问 → `--force-confdef --force-confold`
- 看到 `invoke-rc.d` / `dbus` 警告 → 忽略，WSL 无 systemd 的良性警告
- 装完后 `g++ --version` 能出版本号 → 修复成功
