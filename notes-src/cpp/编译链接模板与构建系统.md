## 一句话结论

C++ 程序从源代码到可执行文件要经过**预处理 → 编译 → 汇编 → 链接**四个阶段，每个阶段都可能出错。新手最容易混淆的是"编译错误"和"链接错误"——前者是语法/类型问题，后者是"找不到定义"。现代 C++ 项目用 **CMake** 管理构建，用 **模板** 在编译期生成代码，用 **STL 容器** 替代手写数据结构。

---

## 零、为什么 Python 不用操心这些？

先建立直觉——对比 Python 和 C++ 的构建流程：

| 维度 | Python | C++ |
|------|--------|-----|
| 源码到运行 | `.py` 直接运行 | `.cpp` 要编译链接成可执行文件 |
| 头文件 | 不需要 | 需要（声明 vs 定义分离） |
| 依赖管理 | `pip install` | CMake + 链接库 |
| 错误出现时机 | 运行时 | **编译时**（类型检查更严格） |
| 跨平台 | 解释器管 | 你要管编译器、操作系统、CPU 架构 |

```
Python:  write code.py → python code.py → 直接运行
C++:     write code.cpp → 预处理 → 编译 → 汇编 → 链接 → ./app
```

C++ 的"麻烦"换来的是：**编译时就能发现很多错误**，性能远超 Python。

## 一、从源代码到可执行文件

### 1.1 四个阶段总览

```
源文件 .cpp
  │
  ▼ ① 预处理（Preprocess）
  │   - 展开 #include（把头文件内容粘贴进来）
  │   - 展开宏（#define）
  │   - 处理条件编译（#ifdef）
  │   产物：.ii（预处理后的源码）
  │
  ▼ ② 编译（Compile）
  │   - 语法检查
  │   - 类型检查
  │   - 生成汇编代码
  │   产物：.s（汇编文件）
  │
  ▼ ③ 汇编（Assemble）
  │   - 汇编代码 → 机器码
  │   产物：.o（目标文件，二进制）
  │
  ▼ ④ 链接（Link）
  │   - 解析符号（把"调用了 add()"和"add() 的定义"对应起来）
  │   - 合并多个 .o 文件
  │   - 链接外部库
  │   产物：可执行文件（app）或库（.a / .so）
  │
  ▼
可执行文件 app
```

### 1.2 逐步观察每个阶段

**准备一个最简单的项目**——先建一个文件夹，放三个文件：

```
myproject/          ← 自己建一个文件夹，名字随意
├── add.h           ← 头文件：放声明（告诉编译器"有这个函数"）
├── add.cpp         ← 源文件：放定义（函数的真正实现）
└── main.cpp        ← 主程序：调用 add 函数
```

> **下面所有的命令，都在 `myproject/` 文件夹里执行**（即先 `cd myproject` 再跑命令）。

把下面三个文件的内容分别写进对应的文件（用 VS Code、记事本、vim 都行）：

```cpp
// add.h —— 头文件：放声明（告诉编译器"有这个函数"）
#pragma once  // 防止重复包含（同一个 .cpp 里 #include 两次也不会出错）
int add(int lhs, int rhs);  // 只写函数签名，不写函数体！末尾有分号
```

```cpp
// add.cpp —— 源文件：放定义（函数的真正实现）
#include "add.h"  // 引入自己的头文件（双引号表示"在当前目录找"）
int add(int lhs, int rhs) {  // 这里写函数体，真正实现
    return lhs + rhs;
}
```

```cpp
// main.cpp —— 主程序
#include <iostream>  // 标准库，用尖括号（在系统目录找）
#include "add.h"     // 自己的头文件，用双引号（在当前目录找）
                     // 告诉编译器：add 函数存在，签名是 int add(int, int)

int main() {
    std::cout << add(1, 2) << '\n';  // 调用 add，输出 3
    return 0;  // main 返回 0 表示程序正常结束
}
```

**逐步执行每个阶段**（在 `myproject/` 目录下，逐条复制到终端跑）：

```bash
# ========== ① 只预处理：展开 #include 和宏 ==========
g++ -E main.cpp -o main.ii
# 逐字解释：
#   g++         调用 C++ 编译器
#   -E          只做预处理（Preprocess），然后停下来
#   main.cpp    输入文件
#   -o main.ii  输出文件名（.ii 是预处理后的源码后缀）
#
# main.ii 是纯文本，可以用编辑器打开看
# 你会发现 <iostream> 的全部内容被粘贴进来了（好几万行！）
# 这就是"展开 #include"的含义——把头文件内容原地复制过来

# ========== ② 只编译：生成汇编代码 ==========
g++ -S main.ii -o main.s
#   -S  只做编译（Compile），停在汇编阶段
# main.s 是汇编代码（人类可读的机器指令前身）
# 用编辑器打开能看到 main:、call add 等汇编指令

# 提示：也可以直接从 .cpp 编到 .s，跳过手动预处理：
# g++ -S main.cpp -o main.s

# ========== ③ 只汇编：生成目标文件（二进制） ==========
g++ -c main.cpp -o main.o
#   -c  只做编译+汇编（Compile + Assemble），不链接
# main.o 是二进制目标文件，用编辑器打开是乱码
# 里面有你写的 main 函数的机器码
# 但 main 调用了 add，add 的定义在另一个文件里——这个符号还没解析

# 同样把 add.cpp 也编译成 .o
g++ -c add.cpp -o add.o
# 现在 myproject/ 里有 main.o 和 add.o 两个目标文件

# ========== ④ 链接：把 main.o 和 add.o 合并成可执行文件 ==========
g++ main.o add.o -o app
#   main.o add.o  要链接的目标文件（顺序无所谓）
#   -o app        输出可执行文件名叫 app
# 链接器把两个 .o 合并，把 main 里"调用 add"和 add.o 里"add 的定义"对应起来

# 运行
./app
# 输出：3
```

> **新人最常问**：为什么要拆成 `add.h` + `add.cpp` 两个文件？
> 答案见下一节 1.3。现在先记住：头文件是"菜单"，源文件是"厨房"。

### 1.3 为什么要分头文件和源文件？

| 文件 | 内容 | 谁需要包含它 |
|------|------|------------|
| `.h` 头文件 | **声明**（函数签名、类定义） | 所有调用它的 `.cpp` |
| `.cpp` 源文件 | **定义**（函数的真正实现） | 只在编译时需要，不暴露给其他文件 |

**类比**：
- 头文件 = **菜单**（告诉客人有什么菜）
- 源文件 = **厨房**（菜真正怎么做）

```cpp
// add.h —— 菜单：声明"有 add 这个菜"
int add(int lhs, int rhs);

// add.cpp —— 厨房：add 的真正做法
int add(int lhs, int rhs) {
    return lhs + rhs;
}

// main.cpp —— 顾客：看了菜单，点菜
#include "add.h"  // 看菜单
int main() {
    add(1, 2);     // 点菜
}
```

**为什么不直接把定义写在头文件里？**
- 如果多个 `.cpp` 都包含了这个头文件，函数会被定义多次 → 链接时报"重复定义"错误
- 例外：`inline` 函数、模板、`constexpr` 函数可以写在头文件里

## 二、三类错误：编译、链接、运行时

### 2.1 错误分类表

| 阶段 | 典型报错 | 常见原因 | 什么时候发现 |
|------|---------|---------|------------|
| **编译错误** | `not declared`、类型不匹配 | 缺声明、语法错误、模板实例化失败 | `g++ -c` 时 |
| **链接错误** | `undefined reference to 'add(int, int)'` | 只声明未定义、漏链接库 | `g++ *.o -o app` 时 |
| **运行时装载错误** | `cannot open shared object file` | 动态库不在搜索路径、架构不匹配 | `./app` 运行时 |

### 2.2 编译错误示例

> 这个例子只需要一个文件 `bug_compile.cpp`，单独建个文件夹放就行。

```cpp
// bug_compile.cpp —— 故意写一个会报编译错误的代码
#include <iostream>

int main() {
    std::cout << add(1, 2) << '\n';  // ❌ add 未声明
    return 0;
}
// 问题：main 里调用了 add，但从没告诉编译器 add 是什么
// 编译器不认识 add → 编译阶段就报错（还没到链接阶段）
```

```bash
# 在 bug_compile.cpp 所在目录执行：
g++ -c bug_compile.cpp
# 报错：
# error: 'add' was not declared in this scope
#    7 |     std::cout << add(1, 2) << '\n';
#      |                     ^~~
#
# 含义：'add' 在这个作用域里没声明过
# 时机：编译阶段（g++ -c 时），还没到链接
```

**修复**：加上声明（告诉编译器 add 存在）

```cpp
// bug_compile_fixed.cpp
#include <iostream>
int add(int lhs, int rhs);  // ← 加上声明（只要有签名就行，不需要函数体）

int main() {
    std::cout << add(1, 2) << '\n';  // 现在编译器认识 add 了
    return 0;
}
```

```bash
g++ -c bug_compile_fixed.cpp   # 编译通过 ✅（但链接还会出错，见下节）
```

### 2.3 链接错误示例

> 这个例子也只需要一个文件 `bug_link.cpp`。它和 2.2 的区别：**声明有了，但定义没有**。

```cpp
// bug_link.cpp
#include <iostream>
int add(int lhs, int rhs);  // ✅ 有声明：编译器知道 add 存在

int main() {
    std::cout << add(1, 2) << '\n';  // 编译通过
    return 0;
}
// 但没有 add 的定义（函数体）！
// 编译器说"OK，add 存在"，但链接器找不到 add 的真正实现 → 链接错误
```

```bash
g++ bug_link.cpp -o app
# 报错：
# /usr/bin/ld: /tmp/ccXXX.o: in function `main':
# bug_link.cpp:(.text+0x20): undefined reference to `add(int, int)'
# collect2: error: ld returned 1 exit status
#
# 含义：undefined reference = 找不到 add 的定义
# /usr/bin/ld = 链接器（linker）报的，不是编译器
# 时机：链接阶段（编译已经通过了）
```

**修复**：提供 add 的定义。最简单的方式是把 `add.cpp`（第一章 1.2 写的那个）一起编译：

```bash
# 假设 bug_link.cpp 和 add.cpp 在同一个文件夹里
# （如果没有 add.cpp，回到第一章 1.2 把 add.cpp 和 add.h 复制过来）
g++ bug_link.cpp add.cpp -o app   # 把 add.cpp 一起编译，链接器就能找到 add 的定义了
./app                              # 输出 3 ✅
```

> **编译错误 vs 链接错误的判断**（新人必记）：
> - 报错里有 `error:` 且提到 `not declared`、`expected` → **编译错误**，代码语法/类型有问题
> - 报错里有 `undefined reference`、`ld returned` → **链接错误**，声明了但找不到定义

### 2.4 运行时装载错误示例

```bash
# 编译时链接了动态库
g++ main.cpp -lmylib -o app

# 运行时找不到动态库
./app
# 报错：
# ./app: error while loading shared libraries: libmylib.so:
# cannot open shared object file: No such file or directory
```

**修复方案**：
```bash
# 方案一：设置 LD_LIBRARY_PATH（临时诊断用）
export LD_LIBRARY_PATH=/path/to/lib:$LD_LIBRARY_PATH
./app

# 方案二：编译时设置 RPATH（推荐，写入二进制）
g++ main.cpp -L/path/to/lib -lmylib -Wl,-rpath,/path/to/lib -o app

# 方案三：把库装到系统目录
sudo cp libmylib.so /usr/local/lib/
sudo ldconfig
```

### 2.5 调试符号的常用命令

```bash
# 查看 app 里的符号（函数名等），-C 反修饰 C++ 名称
nm -C build/app | less
# 输出示例：
# 0000000000001149 T add(int, int)    ← T 表示已定义
#                  U std::cout        ← U 表示未定义（外部符号）

# Linux：查看动态库依赖
ldd build/app
# 输出示例：
#   libstdc++.so.6 => /lib/x86_64-linux-gnu/libstdc++.so.6 (0x...)
#   libmylib.so => not found  ← 问题在这里

# 查看 ELF 文件的动态段（RPATH 等）
readelf -d build/app | grep -i path
# 0x000000000000000f (RPATH)  Library rpath [/path/to/lib]
```

> **macOS 对应**：`otool -L app` 查看动态库依赖（替代 `ldd`）。

## 三、静态库与动态库

### 3.1 对比表

| 维度 | 静态库（.a / .lib） | 动态库（.so / .dylib / .dll） |
|------|-------------------|---------------------------|
| 链接时机 | **编译时** | **运行时** |
| 代码位置 | 复制到可执行文件里 | 独立文件，运行时加载 |
| 产物大小 | 大（每个程序都复制一份） | 小（多个程序共享一份） |
| 更新库 | 要重新编译所有依赖的程序 | 替换 .so 文件即可 |
| 部署 | 简单（只发一个可执行文件） | 要带上 .so 文件 |
| 启动速度 | 快 | 略慢（要加载动态库） |

### 3.2 创建和使用静态库

> **完整可复现项目**——建一个文件夹 `mylib/`，放三个文件：

```
mylib/              ← 建一个新文件夹
├── math_lib.h      ← 头文件：声明 square 函数
├── math_lib.cpp    ← 源文件：实现 square 函数
└── main.cpp        ← 主程序：调用 square
```

> 下面所有命令都在 `mylib/` 文件夹里执行（先 `cd mylib`）。

```cpp
// math_lib.h —— 头文件：声明
#pragma once
int square(int x);  // 声明：返回 x 的平方
```

```cpp
// math_lib.cpp —— 源文件：实现
#include "math_lib.h"
int square(int x) { return x * x; }  // 定义：x * x
```

```cpp
// main.cpp —— 主程序：调用 square
#include <iostream>
#include "math_lib.h"  // 引入头文件，告诉编译器 square 存在

int main() {
    std::cout << square(5) << '\n';  // 输出 25
    return 0;
}
```

**创建静态库并使用**（在 `mylib/` 目录下逐条执行）：

```bash
# ① 编译 math_lib.cpp 成目标文件
g++ -c math_lib.cpp -o math_lib.o
#   -c  只编译+汇编，不链接
#   产物：math_lib.o（二进制目标文件）

# ② 打包成静态库（ar = archiver，归档工具）
ar rcs libmymath.a math_lib.o
#   ar       归档工具
#   r        插入/替换（把 math_lib.o 放进库）
#   c        创建（库不存在就创建）
#   s        写索引（让链接器能快速查找）
#   libmymath.a  库文件名（静态库以 lib 开头、.a 结尾是约定）
#   math_lib.o   要打包的目标文件

# ③ 编译 main.cpp 并链接静态库
g++ main.cpp -L. -lmymath -o app
#   main.cpp     主程序源文件
#   -L.          在当前目录（.）搜索库文件
#   -lmymath     链接名为 mymath 的库（自动加 lib 前缀和 .a 后缀 → libmymath.a）
#   -o app       输出可执行文件
#
# 注意：-lmymath 不是文件名 libmymath.a，而是去掉 lib 前缀和 .a 后缀的"库名"

# ④ 运行
./app
# 输出：25
```

### 3.3 创建和使用动态库

> **用同样的项目结构**（3.2 的 `mylib/` 文件夹和三个文件），只是编译命令不同。

**创建动态库并使用**（在 `mylib/` 目录下逐条执行）：

```bash
# ① 编译成位置无关代码（PIC）
g++ -c -fPIC math_lib.cpp -o math_lib.o
#   -fPIC  Position Independent Code（位置无关代码）
#          动态库必须加这个，因为 .so 加载到内存的位置不固定
#          静态库不需要，因为它在链接时直接复制进可执行文件

# ② 打包成动态库（-shared）
g++ -shared -o libmymath.so math_lib.o
#   -shared          生成动态库（.so）而非可执行文件
#   -o libmymath.so  输出文件名（动态库以 lib 开头、.so 结尾是约定）
#   math_lib.o       要打包的目标文件

# ③ 编译 main.cpp 并链接动态库（命令和静态库一样）
g++ main.cpp -L. -lmymath -o app
#   -L. -lmymath  和静态库完全一样的写法
#   链接器会优先找 .so（动态库），找不到才找 .a（静态库）

# ④ 运行时需要找到 .so 文件
./app
# 报错：./app: error while loading shared libraries: libmymath.so:
#       cannot open shared object file: No such file or directory
#
# 为什么？动态库是运行时加载的，系统不知道去 mylib/ 目录找 .so
# 解决：告诉系统去当前目录找

export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH
#   把当前目录（.）加到动态库搜索路径里
#   这是临时方案（只对当前终端有效）

./app
# 输出：25 ✅
```

> **静态库 vs 动态库对新人来说的关键区别**：
> - 静态库 `.a`：链接时把库代码**复制进**可执行文件，运行时不需要 .a 文件了
> - 动态库 `.so`：链接时只记一个"依赖关系"，运行时才去加载 .so，所以运行时必须能找到 .so

### 3.4 动态库搜索路径（Linux）

运行时找 `.so` 的顺序：

```
1. RPATH/RUNPATH（编译时写入二进制的路径）
2. LD_LIBRARY_PATH（环境变量，临时用）
3. /etc/ld.so.cache（系统缓存，由 ldconfig 生成）
4. /lib, /usr/lib（默认目录）
```

> **原则**：`LD_LIBRARY_PATH` 适合临时诊断，不应成为掩盖安装布局问题的长期方案。生产环境用 RPATH。

## 四、ABI：源码兼容 ≠ 二进制兼容

### 4.1 什么是 ABI（先看类比）

**先说结论**：新人现阶段大概率不会自己踩 ABI 的坑，但用到第三方库（特别是 Python C++ 扩展）时会遇到别人踩的坑。理解概念即可，不用死记细节。

**类比**：想象两个人要合作拼乐高——

- **源码兼容** = 两个人看的是同一份拼装图纸（都能看懂怎么拼）
- **二进制兼容（ABI）** = 两个人的乐高积木接口尺寸一样（能物理插到一起）

两个人图纸一样（源码兼容），但一个用标准乐高、一个用仿制乐高，接口尺寸差 0.1mm——拼不上（ABI 不兼容）。

**ABI（Application Binary Interface）** 就是编译后的二进制文件之间的"接口尺寸"，规定：
- 调用约定：函数参数怎么传（寄存器还是栈）、返回值放哪
- 名称修饰（name mangling）：`add(int, int)` 编译后在二进制里叫什么名字（比如 `_Z3addii`）
- 对象布局：一个 struct 在内存里各字段怎么排布、占多少字节
- 标准库类型实现：`std::string` 在内存里长什么样

**源码兼容**：你的代码能编译通过（图纸一样）。  
**二进制兼容**：编译好的 `.o` / `.so` 能直接链接使用（积木接口一样）。

> **新人现阶段的理解**：你写 C++ 代码用 `g++` 编译，通常不会遇到 ABI 问题。但当你要**链接别人编译好的 .so 库**，或者**做 Python C++ 扩展**时，可能遇到"代码明明对、编译也通过、运行时却报 undefined symbol"——这就是 ABI 不匹配。

### 4.2 什么时候会 ABI 不兼容

| 情况 | 说明 | 新人会遇到吗 |
|------|------|-------------|
| 编译器版本不同 | GCC 9 编译的库，GCC 12 可能链接不上 | 很少（通常用同一版本） |
| 标准库实现不同 | libstdc++（GCC）vs libc++（Clang） | 偶尔（Mac 上 Clang 用 libc++） |
| 编译选项不同 | `-D_GLIBCXX_USE_CXX11_ABI=0` vs `=1` | **常见**（Python 扩展高频坑） |
| Debug/Release 混用 | MSVC 的 `/MDd` vs `/MD` | Windows 上可能遇到 |
| CPU 架构不同 | x86_64 vs ARM64 | 很少（除非跨平台部署） |

### 4.3 实际案例：Python C++ 扩展（新人最可能遇到的场景）

```bash
# 场景：你用 pybind11 编译了一个 Python 扩展 myext.so
# 编译时用的是新版 GCC（默认新 ABI）

# 导入时报错
python -c "import myext"
# ImportError: undefined symbol: _ZNSt7__cxx1112basic_string...
#
# 那串乱码 _ZNSt7__cxx1112basic_string 是 name mangling 后的 std::string
# 报错含义：你的扩展用了"新版 std::string"，
#           但 Python 解释器（或某个依赖）用的是"旧版 std::string"
#           两者 ABI 不匹配，找不到对方的符号

# 解决：强制统一 ABI
g++ ... -D_GLIBCXX_USE_CXX11_ABI=1
# 这个宏控制 std::string 等 STL 类型用新 ABI 还是旧 ABI
# 新版 GCC 默认是 1（新 ABI），但某些老环境/老 Python 可能是 0（旧 ABI）
# 两边要一致
```

> **排查思路**："能编译但导入失败"时，除了检查 Python 路径，还要检查扩展和依赖库的 ABI、架构与符号。这在新人做 pybind11 扩展时是高频问题。

## 五、模板：编译期生成代码

### 5.1 什么是模板

**模板**：写一份"通用代码"，编译器根据你用的类型自动生成特化版本。

```cpp
// 写一个通用的 add，T 是类型参数
template <typename T>
T add(T lhs, T rhs) {
    return lhs + rhs;
}

// 使用时，编译器自动生成两个版本：
auto i = add(1, 2);        // 实例化 add<int>：int add(int, int)
auto f = add(1.0f, 2.0f);  // 实例化 add<float>：float add(float, float)
```

**类比**：
- Python 的 duck typing：`def add(a, b): return a + b` 运行时检查
- C++ 模板：`template<T> T add(T a, T b)` **编译时**生成具体类型版本

### 5.2 模板必须放在头文件里（新人高频踩坑）

> 这是一个**很容易踩的坑**：模板和普通函数不同，定义必须放头文件里。下面用完整的对比示例说明。

**❌ 错误写法：模板声明在 .h，定义在 .cpp（和普通函数一样拆分）**

```
bad_template/
├── add.h       ← 声明模板
├── add.cpp     ← 定义模板
└── main.cpp    ← 使用模板
```

```cpp
// add.h —— 声明模板
#pragma once
template <typename T> T add(T lhs, T rhs);  // 只声明
```

```cpp
// add.cpp —— 定义模板
#include "add.h"
template <typename T>
T add(T lhs, T rhs) { return lhs + rhs; }  // 定义
```

```cpp
// main.cpp —— 使用模板
#include "add.h"
int main() { return add(1, 2); }  // 调用 add<int>
```

```bash
g++ main.cpp add.cpp -o app
# 报错：
# undefined reference to `int add<int>(int, int)'
#
# 为什么？看下面的图解
```

**为什么报错？画个图就明白了**：

```
编译 main.cpp 时：
  编译器看到 add(1, 2) → 想生成 add<int> 的代码
  但 main.cpp 只 #include 了 add.h（只有声明，没有定义）
  编译器想："我没有函数体，怎么生成机器码？"
  → 编译器只能记下"需要 add<int> 这个符号"，等链接时找

编译 add.cpp 时：
  编译器看到 add 的模板定义
  但编译器不知道谁会用 add<int>（main.cpp 的事它不知道）
  → 编译器什么都没生成
  → add.o 里没有 add<int> 的机器码

链接时：
  main.o 说"我需要 add<int>"
  add.o 里没有 add<int>
  → undefined reference！
```

**✅ 正确写法：模板定义放头文件里**

```
good_template/
├── add.h       ← 声明 + 定义（模板放这里！）
└── main.cpp    ← 使用模板
```

```cpp
// add.h —— 模板的声明和定义都放头文件
#pragma once
template <typename T>
T add(T lhs, T rhs) {
    return lhs + rhs;  // 定义就在这里
}
```

```cpp
// main.cpp —— 使用模板
#include "add.h"   // #include 会把 add.h 的全部内容粘贴进来
                   // 所以编译 main.cpp 时，编译器同时看到了声明和定义
int main() {
    return add(1, 2);  // 编译器看到定义，直接生成 add<int> 的机器码
}
```

```bash
g++ main.cpp -o app   # 不需要 add.cpp 了！编译通过 ✅
./app; echo $?         # 输出 3（main 的返回值）
```

**对比普通函数和模板的区别**：

| | 普通函数 | 模板 |
|---|---------|------|
| 声明放哪 | .h | .h |
| 定义放哪 | .cpp（可以） | **必须放 .h**（新人记住这条） |
| 为什么 | 编译 .cpp 时生成固定类型的机器码，链接时能找到 | 编译器需要在"用到的点"看到完整定义才能生成具体类型的代码 |

> **新人记忆口诀**：普通函数可以"声明在 .h、定义在 .cpp"；**模板不行**，模板的定义必须跟着声明一起在 .h 里。

### 5.3 非类型模板参数

```cpp
#include <array>
#include <iostream>

// N 是编译期常量，不是类型
template <typename T, std::size_t N>
T sum(const std::array<T, N>& values) {
    T result{};                        // ← 为什么用 {}？见下方解释
    for (const T& value : values) {
        result += value;
    }
    return result;
}

int main() {
    std::array<int, 5> arr = {1, 2, 3, 4, 5};
    std::cout << sum(arr) << '\n';  // 15
    // 编译器生成 sum<int, 5>

    std::array<double, 3> darr = {1.5, 2.5, 3.0};
    std::cout << sum(darr) << '\n';  // 7.0
    // 编译器生成 sum<double, 3>
}
```

#### `T result{};` 为什么用 `{}` 而不是 `T result;`？

新人看到这行很容易疑惑：为什么不写 `T result;`，偏要加个 `{}`？这涉及 C++ 一个重要的坑。

**先看 `T result;`（不加 `{}`）会怎样**：

```cpp
int result;        // 未初始化！值是垃圾（可能是 0，也可能是 32767）
double result;     // 未初始化！值是垃圾
```

在 C++ 里，**局部变量如果不初始化，它的值是未定义的**（内存里上一个程序留下的垃圾值）。如果你写：

```cpp
T result;          // result 是垃圾值
result += value;   // 垃圾值 + value = 还是垃圾
```

那求和结果就完全错了。这是 C++ 新人最常见的 bug 之一——Python 不会这样（Python 变量必须赋值才能用），所以从 Python 转 C++ 的人特别容易踩。

**`T result{};` 的含义：值初始化（zero-initialization）**：

```cpp
int result{};      // result = 0
double result{};   // result = 0.0
bool result{};     // result = false
// 任何类型用 {} 初始化，都会被设为该类型的"零值"
```

`{}` 是 C++11 引入的**列表初始化**语法，对于基本类型（int、double 等）来说，它会初始化为零值。

**对比四种写法**：

| 写法 | 含义 | result 的值 |
|------|------|------------|
| `T result;` | 默认初始化（基本类型 = 不初始化） | **垃圾值**（未定义） |
| `T result{};` | 值初始化（列表初始化） | 零值（int→0, double→0.0） |
| `T result = 0;` | 拷贝初始化 | 0（但只对数值类型有效） |
| `T result = T{};` | 显式构造临时对象再拷贝 | 零值 |

**为什么模板里用 `{}` 而不是 `= 0`**：

因为 `T` 是模板参数，可能是 `int`、`double`、`float`，甚至可能是自定义类型。`T result = 0;` 只对数值类型有效（自定义类型可能不接受 `= 0`），但 `T result{};` 对所有类型都有效——基本类型初始化为零值，自定义类型调用默认构造函数。这就是模板代码里几乎总是用 `{}` 的原因。

> **新人记忆**：C++ 里声明局部变量时，**养成加 `{}` 的习惯**。`int x{};` 比 `int x;` 安全得多，永远不会有垃圾值。

**为什么有用**：N 是编译期常量，编译器可以据此优化（比如展开循环）。

### 5.4 模板在 CUDA 中的应用

CUDA 和高性能计算大量使用模板：

```cpp
// 数据类型特化
template <typename T>
__global__ void vector_add(T* a, const T* b, int n) {
    int i = blockIdx.x * blockDim.x + threadIdx.x;
    if (i < n) a[i] += b[i];
}
// 用 float、half、bfloat16 实例化

// 非类型参数：tile 大小
template <int TILE_SIZE>
__global__ void matrix_mul(float* C, const float* A, const float* B, int N) {
    __shared__ float tile_A[TILE_SIZE][TILE_SIZE];
    __shared__ float tile_B[TILE_SIZE][TILE_SIZE];
    // ...
}
// 实例化 matrix_mul<16>, matrix_mul<32>
```

### 5.5 模板错误的阅读方法

模板错误往往很长（几十行），阅读方法：

```
error: no matching function for call to 'add(int, double)'
  auto x = add(1, 2.5);
           ^~~
note: candidate template ignored: deduced conflicting types
      for parameter 'T' ('int' vs. 'double')
  T add(T lhs, T rhs) {
    ^
```

**阅读顺序**：
1. **最前面**：用户代码哪里出错了 → `add(1, 2.5)` 调用失败
2. **最底层**：真正的原因 → T 推导冲突，既想是 int 又想是 double
3. **忽略中间**几十层展开

**修复**：显式指定类型，或让类型一致

```cpp
auto x = add<int>(1, 2.5);     // 显式指定 T=int
auto y = add(1.0, 2.5);        // 都是 double
auto z = add(static_cast<double>(1), 2.5);  // 转换
```

## 六、STL 容器与迭代器

### 6.1 常用容器对比

| 容器 | 内存特征 | 访问 | 插入/删除 | 常见用途 |
|------|---------|------|----------|---------|
| `std::vector<T>` | 连续内存 | 随机访问 O(1) | 尾部 O(1)，中间 O(n) | **默认选择**，数值缓冲区 |
| `std::array<T,N>` | 连续内存 | 随机访问 O(1) | 不能改大小 | 编译期已知大小的小数组 |
| `std::deque<T>` | 分段存储 | 随机访问 O(1) | 两端 O(1) | 队列、滑动窗口 |
| `std::list<T>` | 链表 | 顺序 O(n) | 任意位置 O(1) | 频繁中间插删（少用） |
| `std::unordered_map<K,V>` | 哈希表 | 平均 O(1) | 平均 O(1) | 无序键值索引 |
| `std::map<K,V>` | 红黑树 | O(log n) | O(log n) | 需要有序遍历/范围查询 |
| `std::unordered_set<T>` | 哈希表 | 平均 O(1) | 平均 O(1) | 去重 |
| `std::string` | 连续内存 | 随机访问 O(1) | 尾部 O(1) | 字符串 |

### 6.2 vector 完整示例

```cpp
#include <vector>
#include <iostream>

int main() {
    // 创建
    std::vector<int> v1;              // 空
    std::vector<int> v2(5, 0);        // 5 个 0
    std::vector<int> v3 = {1, 2, 3};  // 列表初始化

    // 访问
    std::cout << v3[0] << '\n';       // 1（不检查越界）
    std::cout << v3.at(1) << '\n';    // 2（检查越界，抛异常）

    // 修改
    v3.push_back(4);                   // 尾部添加
    v3.pop_back();                     // 尾部删除
    v3.size();                         // 3
    v3.empty();                        // false

    // 遍历
    for (int x : v3) {                 // 只读遍历
        std::cout << x << ' ';
    }
    for (int& x : v3) {                // 可修改遍历
        x *= 2;
    }

    // 迭代器
    for (auto it = v3.begin(); it != v3.end(); ++it) {
        std::cout << *it << ' ';
    }
}
```

### 6.3 迭代器失效——重要陷阱

```cpp
std::vector<int> v = {1, 2, 3, 4, 5};

// ❌ 危险：遍历时 erase，迭代器失效
for (auto it = v.begin(); it != v.end(); ++it) {
    if (*it == 3) {
        v.erase(it);  // it 失效！下次 ++it 是 UB
    }
}

// ✅ 正确：用 erase 的返回值
for (auto it = v.begin(); it != v.end(); ) {
    if (*it == 3) {
        it = v.erase(it);  // erase 返回下一个有效迭代器
    } else {
        ++it;
    }
}
```

### 6.4 vector 扩容与迭代器失效

```cpp
std::vector<int> v = {1, 2, 3};
int* ptr = &v[0];  // 指向第一个元素

v.push_back(4);    // 可能触发扩容！
// 如果扩容了，ptr 指向的旧内存被释放
// *ptr = 99;  // ❌ UB！ptr 悬空了

// 预防：提前 reserve
std::vector<int> v2;
v2.reserve(1024);  // 预分配，避免后续扩容
```

### 6.5 map 和 unordered_map

```cpp
#include <map>
#include <unordered_map>
#include <string>
#include <iostream>

int main() {
    // unordered_map：哈希表，平均 O(1)
    std::unordered_map<std::string, int> ages = {
        {"Alice", 25}, {"Bob", 30}
    };
    ages["Charlie"] = 28;          // 插入
    std::cout << ages["Alice"] << '\n';  // 25

    // 遍历（无序）
    for (const auto& [name, age] : ages) {
        std::cout << name << ": " << age << '\n';
    }

    // map：红黑树，O(log n)，有序
    std::map<std::string, int> sorted_ages = {
        {"Alice", 25}, {"Bob", 30}
    };
    // 遍历（按 key 排序）
    for (const auto& [name, age] : sorted_ages) {
        std::cout << name << ": " << age << '\n';
        // Alice: 25, Bob: 30（按字母序）
    }
}
```

### 6.6 性能提示：缓存局部性

```cpp
// 理论上 list 插删 O(1)，vector 插删 O(n)
// 但实际 vector 往往更快！

// 原因：vector 连续内存 → CPU 缓存友好
// list 节点分散 → 缓存 miss 频繁

// 经验：除非需要在中间频繁插删且元素很大，
// 否则优先 vector
```

> **原则**：性能敏感代码除了看大 O 复杂度，还要看连续性、分配次数、缓存局部性。很多场景中，线性扫描连续 vector 比理论查找复杂度更低但指针跳转频繁的结构更快。

## 七、最小 CMake 项目

### 7.1 为什么用 CMake

| 手动 g++ | CMake |
|---------|-------|
| `g++ main.cpp add.cpp -o app` | 写一次 CMakeLists.txt，跨平台 |
| 文件多了命令很长 | 自动管理依赖 |
| 换机器要改命令 | `cmake -B build` 就行 |
| 没有增量编译 | 自动只编译改动过的文件 |

### 7.2 项目结构

> **完整可复现项目**——建一个文件夹 `vector_ops/`，按下面的结构放文件：

```
vector_ops/                 ← 建一个新文件夹
├── CMakeLists.txt          ← 构建配置（CMake 的"说明书"）
├── include/
│   └── vector_ops.h        ← 公开头文件（放声明）
└── src/
    ├── vector_ops.cpp      ← 实现（放定义）
    └── main.cpp            ← 主程序（7.5 节会加）
```

> 下面所有命令都在 `vector_ops/` 文件夹里执行（先 `cd vector_ops`）。

先创建目录结构：

```bash
mkdir -p vector_ops/include vector_ops/src
cd vector_ops
# 现在你在 vector_ops/ 里
```

把下面三个文件的内容分别写进对应文件：

```cpp
// include/vector_ops.h —— 公开头文件
#pragma once
#include <vector>

float sum(const std::vector<float>& values);  // 声明：对 vector 求和
```

```cpp
// src/vector_ops.cpp —— 实现
#include "vector_ops.h"   // 引入自己的头文件
#include <numeric>        // std::accumulate 在这里

float sum(const std::vector<float>& values) {
    // std::accumulate：从 begin 到 end 累加，初始值 0.0f
    return std::accumulate(values.begin(), values.end(), 0.0f);
}
```

> 注意：`src/vector_ops.cpp` 里 `#include "vector_ops.h"` 能找到 `include/` 下的头文件，是因为 CMakeLists.txt 里配了头文件搜索路径（见下节 `target_include_directories`）。

### 7.3 CMakeLists.txt 逐行解释

在 `vector_ops/` 根目录下创建 `CMakeLists.txt`（注意大小写，C 代表大写）：

```cmake
# 要求 CMake 最低版本 3.20（太老的 CMake 不支持某些语法）
cmake_minimum_required(VERSION 3.20)

# 项目名：vector_ops，语言：CXX（C++ 的缩写，C 语言是 C）
project(vector_ops LANGUAGES CXX)

# 创建一个静态库 target，名字叫 vector_ops，源文件是 src/vector_ops.cpp
# "target" 是 CMake 的核心概念：你定义一个"构建目标"
# target 可以是库（add_library）或可执行文件（add_executable）
add_library(vector_ops src/vector_ops.cpp)

# 设置头文件搜索路径（相当于 g++ 的 -I 参数）
# PUBLIC = 对内对外都生效（见下方关键字解释）
target_include_directories(vector_ops PUBLIC include)
# 意思：编译 vector_ops 库时，去 include/ 找头文件
#       而且链接这个库的人，也会自动获得这个搜索路径

# 要求 C++20 标准（相当于 g++ 的 -std=c++20）
# PUBLIC = 依赖此库的也要用 C++20
target_compile_features(vector_ops PUBLIC cxx_std_20)

# 对 GCC/Clang 启用警告（相当于 g++ 的 -Wall -Wextra -Wpedantic）
if(CMAKE_CXX_COMPILER_ID MATCHES "GNU|Clang")
  target_compile_options(vector_ops PRIVATE -Wall -Wextra -Wpedantic)
endif()
# PRIVATE = 只对 vector_ops 自己生效，链接它的人不受影响
```

**关键字解释**（新人最容易混淆的三个）：

| 关键字 | 含义 | 类比 |
|--------|------|------|
| `PUBLIC` | 对内对外都生效（自己编译要用，依赖我的人也要用） | 餐厅的公开菜单：自己和客人都能看 |
| `PRIVATE` | 只对自己生效（依赖我的人不需要） | 厨房的内部工具：只有厨师用 |
| `INTERFACE` | 只对依赖我的人生效（自己不用） | 只给客人看的规则：厨师不用遵守 |

> **新人最常问**：什么时候用 PUBLIC，什么时候用 PRIVATE？
> - 头文件路径：如果你在 `.h` 里 `#include` 了某个库，链接你的人也需要那个库的头文件 → `PUBLIC`
> - 头文件路径：如果你只在 `.cpp` 里 `#include`，链接你的人不需要 → `PRIVATE`
> - 编译选项（-Wall 等）：只对自己有意义 → `PRIVATE`

### 7.4 构建命令

在 `vector_ops/` 目录下执行（有 CMakeLists.txt 的那个目录）：

```bash

cmake -S . -B build -DCMAKE_BUILD_TYPE=Release
#   -S .            源码目录是当前目录（.）
#   -B build        构建目录是 build/（CMake 会自动创建）
#   -DCMAKE_BUILD_TYPE=Release  设置构建类型为 Release（启用优化）
# 执行后会在 build/ 里生成一堆文件，包括 Makefile

# ② 编译（执行 Makefile，实际调用 g++）
cmake --build build --parallel
#   --build build   在 build/ 目录里构建
#   --parallel      并行编译（利用多核，更快）
# 产物：build/libvector_ops.a（静态库）

# ③ 清理重建（改了 CMakeLists.txt 后可能需要）
rm -rf build/                              # 删掉 build 目录
cmake -S . -B build -DCMAKE_BUILD_TYPE=Release  # 重新配置
cmake --build build --parallel             # 重新编译
```

### 7.5 加一个可执行文件 target

> 目前 `vector_ops` 只是一个**库**，不能直接运行。我们加一个 `main.cpp` 来调用它，编译成**可执行文件**。

在 `src/` 下新建 `main.cpp`：

```cpp
// src/main.cpp —— 主程序，调用 vector_ops 库的 sum 函数

#include <iostream>
```

```cmake

#   - include/ 头文件路径（所以 #include "vector_ops.h" 能找到）
```

## 八、C++ 并发的最小认知

> **新人提示**：这章的代码都可以用一个文件跑。关键是编译时**必须加 `-pthread`**，否则运行时报错。
> 文件结构：建一个文件夹 `concurrency/`，每节的代码单独存成一个 `.cpp` 文件。

### 8.1 数据竞争

**先看一个会出错的代码**（两个线程同时改一个变量）：

```cpp
// data_race.cpp —— 数据竞争示例
#include <iostream>
#include <thread>

int counter = 0;  // 全局变量，两个线程都会改它

void increment() {
    for (int i = 0; i < 100000; ++i) {
        counter++;  // ❌ 两个线程同时执行这行 → 数据竞争
    }
}

int main() {
    std::thread t1(increment);  // 启动线程 1
    std::thread t2(increment);  // 启动线程 2

    t1.join();  // 等线程 1 跑完
    t2.join();  // 等线程 2 跑完

    std::cout << counter << '\n';  // 期望 200000，实际可能是 143287 等
    // counter++ 不是一步操作（读→加1→写），两个线程交错执行会丢更新
    return 0;
}
```

```bash
# 编译时必须加 -pthread！
g++ -std=c++20 -pthread data_race.cpp -o data_race
#   -pthread  链接线程库（不加会运行报错：Enable multithreading to use std::thread）

./data_race
# 输出可能是 143287（每次不一样，但通常不是 200000）
# 这就是数据竞争：结果不可预测
```

### 8.2 用互斥锁修复

```cpp
// mutex_fix.cpp —— 用互斥锁修复数据竞争
#include <iostream>
#include <thread>
#include <mutex>

int counter = 0;
std::mutex mtx;  // 互斥锁：保证同一时刻只有一个线程能访问

void increment() {
    for (int i = 0; i < 100000; ++i) {
        std::lock_guard<std::mutex> guard(mtx);  // 加锁
        counter++;
    }  // guard 离开这个大括号自动解锁（RAII！见 cpp/内存生命周期与资源管理.md）
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);

    t1.join();
    t2.join();

    std::cout << counter << '\n';  // 200000 ✅
    return 0;
}
```

```bash
g++ -std=c++20 -pthread mutex_fix.cpp -o mutex_fix
./mutex_fix
# 输出：200000
```

**`std::lock_guard` 就是 RAII**：构造时加锁，析构时解锁，即使中间抛异常也会解锁。不用手动 `mtx.lock()` / `mtx.unlock()`。

### 8.3 原子变量（简单场景）

```cpp
// atomic_fix.cpp —— 用原子变量替代锁（更轻量）
#include <atomic>
#include <iostream>
#include <thread>

std::atomic<int> counter{0};  // 原子变量：加法是不可分割的操作

void increment() {
    for (int i = 0; i < 100000; ++i) {
        counter++;  // 原子操作，不需要锁
    }
}

int main() {
    std::thread t1(increment);
    std::thread t2(increment);

    t1.join();
    t2.join();

    std::cout << counter << '\n';  // 200000 ✅
    return 0;
}
```

```bash
g++ -std=c++20 -pthread atomic_fix.cpp -o atomic_fix
./atomic_fix
# 输出：200000
```

**什么时候用原子 vs 锁**：

| 场景 | 推荐 |
|------|------|
| 单个变量的读写 | `std::atomic` |
| 多个变量要一起修改 | `std::mutex` + `lock_guard` |
| 简单计数器 | `std::atomic` |
| 复杂临界区 | `std::mutex` |

### 8.4 条件变量（等待状态变化）

**先说场景**：主线程启动两个工作线程后，要"等 1 秒再通知它们开始干活"。工作线程在收到通知前一直阻塞（不消耗 CPU）。

**条件变量解决什么问题**：让线程能"等待某个条件成立"，而不是用 `while (!ready);` 死循环空转浪费 CPU。

```cpp
// cond_var.cpp —— 条件变量示例
#include <iostream>
#include <thread>
#include <mutex>
#include <condition_variable>

std::mutex mtx;              // 互斥锁：保护 ready 变量的读写
std::condition_variable cv;  // 条件变量：用来"通知"和"等待"
bool ready = false;          // 共享状态：是否可以开始工作

void worker(int id) {
    // worker 线程：等主线程说"准备好了"才开始干活
    std::unique_lock<std::mutex> lock(mtx);    // 先加锁（条件变量要求配合 unique_lock）
    cv.wait(lock, [] { return ready; });        // 等待 ready 变 true
    //  ↑ 这一行做了三件事：
    //    1. 检查 lambda（[] { return ready; }）是否为 true
    //    2. 如果是 false：自动解锁 mtx，然后线程在这里"睡觉"（不占 CPU）
    //    3. 被 notify 唤醒后：自动重新加锁 mtx，再检查 lambda
    //    （循环检查直到 true，防止"虚假唤醒"）
    std::cout << "Worker " << id << " started\n";
    // 离开函数时 lock 析构，自动解锁
}

int main() {
    std::thread t1(worker, 1);  // 启动 worker 线程 1（会卡在 cv.wait 等 ready）
    std::thread t2(worker, 2);  // 启动 worker 线程 2（同上）

    std::this_thread::sleep_for(std::chrono::seconds(1));  // 主线程睡 1 秒
    //  模拟"准备工作"——这 1 秒里两个 worker 都在 cv.wait 处阻塞

    {
        std::lock_guard<std::mutex> lock(mtx);  // 加锁，准备改 ready
        ready = true;                           // 改成"准备好了"
    }  // 离开大括号，lock 析构自动解锁（一定要先解锁再 notify！）
    cv.notify_all();  // 唤醒所有在 cv.wait 等待的线程（t1 和 t2）

    t1.join();  // 等 t1 跑完
    t2.join();  // 等 t2 跑完
}
```

```bash
g++ -std=c++20 -pthread cond_var.cpp -o cond_var
./cond_var
# 输出（顺序可能不同）：
# Worker 1 started
# Worker 2 started
```

**逐行解读关键点**：

| 代码 | 为什么这么写 |
|------|-------------|
| `std::unique_lock<std::mutex>` | 条件变量的 `wait` 要求用 `unique_lock`（不是 `lock_guard`），因为 `wait` 内部需要解锁/加锁，`unique_lock` 支持这个 |
| `cv.wait(lock, [] { return ready; })` | 第二个参数是 lambda（判断条件）。`wait` 会循环检查它，只有返回 `true` 才继续往下执行 |
| `{ std::lock_guard lock(mtx); ready = true; }` | 改 `ready` 要加锁（防止和 worker 的读操作竞争），大括号保证离开时立即解锁 |
| `cv.notify_all()` | 必须在**解锁之后**调用，否则 worker 被唤醒后想加锁却加不上，又卡住了 |
| `notify_all` vs `notify_one` | `notify_all` 唤醒所有等待的线程；`notify_one` 只唤醒一个 |

**执行时间线**：

```
时间   主线程              worker 1           worker 2
 0s    启动 t1, t2        cv.wait(睡觉)     cv.wait(睡觉)
 0s    sleep 1 秒         ...                ...
 1s    加锁, ready=true   ...                ...
 1s    解锁                ...                ...
 1s    notify_all          被唤醒, 加锁       被唤醒, 加锁
 1s                        ready==true        ready==true
 1s                        输出 Worker 1      输出 Worker 2
 1s                        解锁, 结束         解锁, 结束
 1s    join 完成
```

### 8.5 volatile 不是线程同步工具

**先说 volatile 是什么**：

`volatile` 是 C++ 的一个关键字，它的本意是告诉编译器："这个变量的值可能在你看不到的地方被改变（比如被硬件寄存器改），所以每次用到它都要从内存重新读，不要把它缓存到寄存器里偷懒。"

```cpp
// 编译器可能这样优化：
int x = flag;   // 读一次 flag 到寄存器
if (x == 0) { ... }   // 用寄存器里的值
// 如果编译器觉得 x 没被改，下次用 flag 时可能不重新读内存

volatile int flag = 0;  // 加 volatile 后：
// 编译器每次用 flag 都从内存重新读，不缓存到寄存器
```

**新人常见的错误想法**："既然 volatile 让编译器每次都重新读内存，那用它修饰的变量在多线程里就是安全的吧？"

**答案是：不行！** 看这个例子：

```cpp
// ❌ 错误：volatile 不能保证线程安全
volatile int flag = 0;
// 线程1: flag = 1;
// 线程2: while (flag == 0);  // 可能永远循环，即使线程1已经把 flag 改成 1 了
```

**为什么不行**：`volatile` 只解决了"编译器不要缓存到寄存器"这一个问题，但它**不提供**多线程需要的三个保证：

| 需要的保证 | volatile 提供？ | std::atomic 提供？ |
|-----------|----------------|-------------------|
| **原子性**（读和写不可分割，不会读到"写了一半"的值） | ❌ 不提供 | ✅ 提供 |
| **内存序**（一个线程的写对另一个线程"可见"的顺序保证） | ❌ 不提供 | ✅ 提供 |
| **禁止编译器缓存**（每次从内存读） | ✅ 提供 | ✅ 提供 |

volatile 只提供了 3 个保证中的 1 个。特别是"可见性"——线程 1 把 `flag` 改成 1 后，线程 2 可能**永远看不到**这个变化（因为 CPU 缓存、指令重排等原因，volatile 在 C++ 里管不了这些）。

> **补充**：Java 的 `volatile` 和 C++ 的 `volatile` **不是一回事**。Java 的 `volatile` 确实提供了可见性保证，所以从 Java 转过来的人特别容易误用 C++ 的 `volatile`。

**正确做法**：用 `std::atomic`

```cpp
// ✅ 正确
std::atomic<bool> flag{false};
// 线程1: flag = true;        // 原子写，且保证对其他线程可见
// 线程2: while (!flag);      // 原子读，能看到线程1的修改
```

`std::atomic` 同时提供原子性 + 内存序 + 禁止缓存，才是多线程正确的工具。

> **新人记忆**：C++ 里看到 `volatile` 想到两个场景——① 裸机/嵌入式开发读写硬件寄存器（这是它本来的用途）；② 多线程同步——**不用 volatile，用 atomic**。

### 8.6 用 ThreadSanitizer 检测数据竞争

```bash

#   -fsanitize=thread  启用线程竞争检测
#   -g                 加调试信息（让报错信息有行号）

#   - 竞争的变量地址
#   - 两个线程的调用栈
```

#### WSL2 上 TSan 报 `unexpected memory mapping` 怎么办

在 WSL2（Windows Subsystem for Linux 2）里跑 TSan，大概率会遇到这个报错：

```
FATAL: ThreadSanitizer: unexpected memory mapping 0x5b0c8d7c3000-0x5b0c8d7c4000
```

**这不是你的代码有问题**，而是 WSL2 的已知限制。

**原因**：TSan 需要监控程序的内存映射，它假设 Linux 的内存布局符合一定规律。但 WSL2 是跑在虚拟机里的，它的虚拟内存映射方式和原生 Linux 不一样，TSan 的假设不成立，直接 `FATAL` 退出。这是 WSL2 + TSan 的已知不兼容，不是你的代码 bug。

**解法（按推荐顺序）**：

```bash
# ========== 解法 1：用 helgrind 替代（WSL2 里推荐） ==========
# helgrind 是 valgrind 套件里的线程错误检测工具，不依赖 TSan 的内存映射假设
# 8.1 装过 valgrind 的话直接用：
valgrind --tool=helgrind ./data_race
# 不需要重新编译（不用 -fsanitize=thread），用普通的 data_race 就行
# 输出会报告数据竞争的详细信息
# 缺点：比 TSan 慢很多（valgrind 是模拟执行）

# ========== 解法 2：在 Docker 容器里跑 TSan ==========
# Docker 容器用的是真正的 Linux 内核（不是 WSL2 的虚拟机内核），TSan 能正常工作
docker run --rm -it -v "$PWD:/work" -w /work ubuntu:22.04 bash
# 容器内：
apt update && apt install -y g++ && g++ -std=c++20 -pthread -fsanitize=thread -g data_race.cpp -o data_race_tsan
./data_race_tsan

# ========== 解法 3：升级 WSL2 内核（可能修复，不保证） ==========
# 在 PowerShell（管理员）里：
wsl --update
# 然后重启 WSL：wsl --shutdown
# 重新打开终端再试，部分版本能修复此问题

# ========== 解法 4：用真机 Linux ==========
# 如果你有双系统的 Linux 或远程 Linux 服务器，直接在那上面跑 TSan
```

> **总结**：WSL2 上 TSan 基本不可用，优先用 **helgrind**（解法 1，最方便）或 **Docker 容器**（解法 2，最接近原生）。ASan（AddressSanitizer）在 WSL2 上一般没问题，只有 TSan 有这个限制。

> **注意**：ThreadSanitizer 和 AddressSanitizer **不能同时启用**。先测内存错误（ASan），再单独测线程竞争（TSan 或 helgrind）。

## 附录 A：g++ 命令行参数详解

> 本附录系统梳理 `g++` 后面常用的参数，按功能分组，每组配示例。
> 同类参数在 `clang++` 上大多一致；MSVC（`cl.exe`）参数体系完全不同，不在本篇范围。

### A.1 分阶段控制参数（最常用）

这组参数控制"编译器停在哪一步"，对应第一章的四阶段流程：

| 参数 | 作用 | 产物 | 对应阶段 |
|------|------|------|---------|
| `-E` | 只预处理 | `.ii`（纯文本） | 预处理 |
| `-S` | 预处理 + 编译，停在汇编 | `.s`（汇编代码） | 编译 |
| `-c` | 预处理 + 编译 + 汇编，不链接 | `.o`（目标文件） | 汇编 |
| 无 | 全流程到链接 | 可执行文件 | 链接 |

```bash
g++ -E main.cpp -o main.ii     # 只预处理（看 #include 展开结果）
g++ -S main.cpp -o main.s      # 到汇编（看生成的汇编指令）
g++ -c main.cpp -o main.o      # 到目标文件（不链接，多个文件分开编）
g++ main.o add.o -o app        # 单独链接（把多个 .o 合并）
g++ main.cpp add.cpp -o app    # 一步到位（编译+链接）
```

**记忆口诀**：`-E` 看 macro、`-S` 看汇编、`-c` 出 `.o`、不加就出可执行。

### A.2 输出文件名：`-o`

```bash
g++ main.cpp -o app            # 输出文件名为 app
g++ main.cpp                   # 不加 -o，默认输出 a.out（Linux）/ a.exe（Windows）
```

`-o` 可以放任意位置，但习惯放在最后。如果和 `-c` 一起用，输出是 `.o` 文件名：

```bash
g++ -c main.cpp -o main.o      # 显式指定 .o 文件名
g++ -c main.cpp                # 默认输出 main.o（取源文件名换后缀）
```

### A.3 C++ 标准版本：`-std=`

指定用哪个 C++ 标准版本，这是现代 C++ 必加的参数。

| 参数 | 标准 | 典型特性 | GCC 支持版本 |
|------|------|---------|-------------|
| `-std=c++11` | C++11 | auto、lambda、智能指针、range-for | 4.7+ |
| `-std=c++14` | C++14 | 泛型 lambda、`std::make_unique` | 4.9+ |
| `-std=c++17` | C++17 | `std::optional`、结构化绑定、`if constexpr` | 5+ |
| `-std=c++20` | C++20 | concepts、ranges、`std::span`、协程 | 10+ |
| `-std=c++23` | C++23 | `std::print`、`std::expected`、deducing this | 13+（部分） |

```bash
g++ -std=c++20 main.cpp -o app         # 用 C++20 标准
g++ -std=gnu++20 main.cpp -o app       # gnu++ 扩展版（默认开启 GNU 扩展）
```

**`c++NN` vs `gnu++NN`**：`c++NN` 是纯标准，`gnu++NN` 额外开启 GNU 扩展（如 `__int128`、变长数组）。g++ 默认用 `gnu++NN`，写笔记/教学时一般用纯 `c++NN` 避免依赖扩展。

> **重要**：`-std=c++20` 和 `-std=gnu++20` 都**只是"语法标准"**，不等于"能用 `<span>`"。能否用某特性还取决于 GCC 版本是否实现了它。判断方法：用 `g++ --version` 看版本，对照 GCC 文档的"C++ 标准库支持状态"。

### A.4 警告参数：`-W`

控制编译器报多少警告。警告不是错误（默认不阻止编译），但应该尽量消除。

| 参数 | 作用 |
|------|------|
| `-w` | **关闭所有警告**（不推荐，掩盖问题） |
| `-Wall` | 开启常见警告（未使用变量、有符号比较、返回值等） |
| `-Wextra` | 在 `-Wall` 基础上再加一批（空基类、缺失字段初始化等） |
| `-Wpedantic` | 严格按标准，禁用 GNU 扩展警告（用了扩展会报） |
| `-Werror` | 把警告当错误（有警告就停止编译） |
| `-Wno-xxx` | 关闭某个具体警告（如 `-Wno-unused-variable`） |
| `-Wxxx` | 开启某个具体警告（如 `-Wshadow`、`-Wconversion`） |

```bash
# 推荐的开发组合：开全警告 + 当错误处理
g++ -std=c++20 -Wall -Wextra -Wpedantic -Werror main.cpp -o app

# 只想关掉某个烦人的警告
g++ -Wall -Wextra -Wno-unused-parameter main.cpp -o app
```

**`-Wall` 并不是"所有警告"**：名字有误导，它只开"最常见"的。`-Wextra` 才补上更多。想看全部可以用 `-Weverything`（仅 Clang）或 `-Wall -Wextra -Wpedantic -Wconversion -Wshadow -Wnon-virtual-dtor`。

### A.5 优化参数：`-O`

控制编译器优化级别，影响性能和调试体验。

| 参数 | 优化级别 | 效果 | 调试 |
|------|---------|------|------|
| `-O0` | 不优化（默认） | 代码和源码一一对应 | ✅ 最好 |
| `-O1` | 轻度优化 | 稍快，变量可能被重排 | ⚠️ 部分变量丢失 |
| `-O2` | 标准优化（Release 推荐） | 内联、循环展开、向量化 | ❌ 变量大量 `<optimized out>` |
| `-O3` | 激进优化 | 更激进的内联和向量化，可能增大体积 | ❌ 同上 |
| `-Os` | 优化体积 | 类似 `-O2` 但禁用增体积的优化 | ❌ |
| `-Ofast` | `-O3` + 不严格遵守标准 | 启用 `-ffast-math` 等，可能改变浮点结果 | ❌ |

```bash
# Debug 构建：不优化 + 调试信息
g++ -std=c++20 -O0 -g main.cpp -o app_debug

# Release 构建：标准优化
g++ -std=c++20 -O2 main.cpp -o app_release

# 性能极致（注意浮点行为可能变）
g++ -std=c++20 -Ofast main.cpp -o app_fast
```

**关键认知**：
- `-O2` 是 Release 的默认选择，平衡性能和安全
- `-O3` 不一定比 `-O2` 快，激进内联可能导致 icache miss
- 优化后崩溃排查：用 `-O2 -g`（保留调试信息），GDB 能看到调用栈但部分变量是 `<optimized out>`

### A.6 调试信息：`-g`

在二进制里嵌入调试信息（源文件名、行号、变量名），供 GDB 使用。

```bash
g++ -g main.cpp -o app               # 默认级别（DWARF 格式）
g++ -g -O0 main.cpp -o app_debug     # 调试最佳：不优化 + 调试信息
g++ -g -O2 main.cpp -o app           # 优化版也保留调试信息（排查 release 崩溃）
g++ -g3 main.cpp -o app              # g3 包含宏定义（GDB 里能展开宏）
```

| 级别 | 内容 |
|------|------|
| `-g` / `-g2` | 行号、变量名、函数名 |
| `-g3` | 额外包含宏定义（`#define` 的值，GDB 里 `macro expand` 可用） |
| `-g0` | 无调试信息 |

**注意**：调试信息让二进制变大（可能几倍），但**不影响运行性能**。发布版可以 `strip` 掉：

```bash
strip app                 # 删除调试符号，二进制变小
# 注意：strip 后无法用 GDB 调试，调用栈只有地址没有函数名
```

### A.7 头文件搜索路径：`-I`

告诉编译器去哪里找 `#include "xxx.h"`（双引号包含）。

```bash
# 项目结构：
# project/
# ├── include/       ← 公开头文件
# │   └── mylib.h
# └── src/main.cpp   ← #include "mylib.h"

g++ -Iinclude src/main.cpp -o app
# -Iinclude 表示把 include/ 加入头文件搜索路径
```

**搜索顺序**（双引号 `#include "x.h"`）：
1. 当前源文件所在目录
2. `-I` 指定的目录（按命令行顺序）
3. 系统目录（`/usr/include`、`/usr/local/include`、编译器自带目录）

**尖括号 `#include <x.h>`**：跳过第 1 步，直接从 `-I` 和系统目录开始找。所以自己的头文件用双引号，系统/第三方库头文件用尖括号。

```bash
# 多个搜索路径
g++ -Iinclude -Ithird_party/eigen -I/usr/local/cuda/include main.cpp -o app
```

### A.8 宏定义：`-D` / `-U`

在命令行定义/取消宏，等价于源码里的 `#define` / `#undef`。

```bash
# 定义 DEBUG 宏（不带值，等价于 #define DEBUG）
g++ -DDEBUG main.cpp -o app

# 定义带值的宏
g++ -DVERSION=\"1.0\" -DBUFFER_SIZE=1024 main.cpp -o app
# 注意：字符串要用 \" 转义，否则 shell 会吃掉引号

# 取消宏
g++ -UDEBUG main.cpp -o app     # 等价于 #undef DEBUG
```

**实际用途**：

```bash
# 区分 Debug/Release 构建
g++ -DDEBUG -g -O0 main.cpp -o app_debug      # 源码里 #ifdef DEBUG 开启调试日志
g++ -DNDEBUG -O2 main.cpp -o app_release       # NDEBUG 禁用 assert

# 控制日志级别
g++ -DLOG_LEVEL=2 main.cpp -o app              # 源码里 #if LOG_LEVEL >= 2 时输出详细日志

# CUDA 项目里常见
g++ -D_GLIBCXX_USE_CXX11_ABI=1 main.cpp -o app  # 强制新 ABI（见第四章 ABI）
```

### A.9 库搜索路径与链接：`-L` / `-l`

控制链接器去哪里找库文件。

```bash
# -L<path>    添加库搜索路径
# -l<name>    链接名为 lib<name>.a 或 lib<name>.so 的库

g++ main.cpp -L/usr/local/lib -lmylib -o app
# 在 /usr/local/lib 找 libmylib.so 或 libmylib.a
```

**搜索顺序**：
1. `-L` 指定的目录（按命令行顺序）
2. 编译时写入的 RPATH/RUNPATH
3. 系统默认目录（`/lib`、`/usr/lib`、`/usr/local/lib`）

**静态库 vs 动态库的选择**：如果同目录下同时有 `libmylib.a` 和 `libmylib.so`，默认链接动态库。强制用静态库：

```bash
g++ main.cpp -L. -l:libmylib.a -o app    # 用 -l: 显式指定文件名
# 或
g++ main.cpp -L. -Wl,-Bstatic -lmylib -Wl,-Bdynamic -o app
```

**链接顺序很重要**：被依赖的库要放在依赖它的后面：

```bash
# ❌ 错误：mylib 依赖 math，但 math 在前，链接器扫描到 mylib 时找不到 math 的符号
g++ main.cpp -lmath -lmylib -o app

# ✅ 正确：依赖者在前，被依赖者在后
g++ main.cpp -lmylib -lmath -o app
```

### A.10 动态库相关：`-fPIC` / `-shared` / `-rdynamic`

| 参数 | 作用 |
|------|------|
| `-fPIC` | 生成位置无关代码（Position Independent Code），编译动态库的目标文件时必须 |
| `-shared` | 生成动态库（.so）而非可执行文件 |
| `-rdynamic` | 导出可执行文件的符号表（让动态库能回调可执行文件里的函数） |

```bash
# 创建动态库
g++ -c -fPIC mylib.cpp -o mylib.o        # ① 编译成 PIC 目标文件
g++ -shared -o libmylib.so mylib.o        # ② 打包成 .so

# 使用动态库
g++ main.cpp -L. -lmylib -o app
export LD_LIBRARY_PATH=.:$LD_LIBRARY_PATH
./app

# -rdynamic 的用途：插件系统
g++ -rdynamic main.cpp -lplugin -o app
# main.cpp 里定义了 register_handler()，plugin.so 要回调它
# 不加 -rdynamic，plugin.so 找不到 register_handler 的符号
```

### A.11 链接器透传参数：`-Wl,`

把参数透传给底层链接器 `ld`。`-Wl,` 后面的内容按逗号分隔传给 ld。

```bash
# 设置 RPATH（把库路径写进二进制）
g++ main.cpp -L/opt/lib -lmylib -Wl,-rpath,/opt/lib -o app
# -Wl,-rpath,/opt/lib  →  传给 ld：-rpath /opt/lib

# 设置 RUNPATH（现代版 RPATH，优先级更低但更灵活）
g++ main.cpp -L/opt/lib -lmylib -Wl,--enable-new-dtags,-rpath,/opt/lib -o app

# 生成 map 文件（调试链接问题时看符号布局）
g++ main.cpp -o app -Wl,-Map,app.map

# 去掉未使用的段
g++ main.cpp -o app -Wl,--gc-sections -ffunction-sections -fdata-sections
```

**`-Wl,-rpath,<path>` vs `LD_LIBRARY_PATH`**：
- RPATH 写进二进制，部署后不用管环境变量（推荐生产）
- LD_LIBRARY_PATH 是运行时环境变量，临时诊断用，不要写进 `~/.bashrc` 长期依赖

### A.12 Sanitizer 参数

详见 `cpp/内存生命周期与资源管理.md` 和 `cpp/环境配置教程.md`。

| 参数 | 检测什么 | 不能和谁同用 |
|------|---------|-------------|
| `-fsanitize=address` | 内存错误（越界、use-after-free、泄漏） | TSan、MSan |
| `-fsanitize=undefined` | 未定义行为（整数溢出、空指针解引用等） | 无冲突 |
| `-fsanitize=thread` | 数据竞争 | ASan、MSan |
| `-fsanitize=memory` | 未初始化内存读取（仅 Clang） | ASan、TSan |
| `-fsanitize=leak` | 内存泄漏（ASan 的子集，开销更小） | TSan、MSan |

```bash
# 内存错误 + UB（最常用组合）
g++ -std=c++20 -fsanitize=address,undefined -g -O0 main.cpp -o app_asan

# 数据竞争（单独用）
g++ -std=c++20 -fsanitize=thread -g -O0 main.cpp -o app_tsan

# 常用附加参数（让错误报告更详细）
g++ -fsanitize=address -g -O0 -fno-omit-frame-pointer main.cpp -o app
# -fno-omit-frame-pointer：保留帧指针，栈回溯更清晰
```

### A.13 多线程与并行：`-pthread` / `-fopenmp`

| 参数 | 作用 |
|------|------|
| `-pthread` | 链接 pthread 库（用 `std::thread` 时必须加） |
| `-fopenmp` | 启用 OpenMP 指令（`#pragma omp parallel`） |

```bash
# 用 std::thread
g++ -std=c++20 -pthread main.cpp -o app

# 用 OpenMP
g++ -std=c++20 -fopenmp main.cpp -o app
```

**容易踩的坑**：用了 `std::thread` 但忘记 `-pthread`，编译通过但运行时报 `std::system_error: Enable multithreading to use std::thread`。`-pthread` 要同时加在编译和链接阶段（CMake 里 `target_link_libraries(app PRIVATE pthread)`）。

### A.14 CPU 架构优化：`-march`

让编译器针对特定 CPU 架构生成指令（AVX、SSE 等）。

```bash
g++ -march=native main.cpp -o app        # 针对当前机器的 CPU（自动检测）
g++ -march=haswell main.cpp -o app       # 针对 Haswell 架构
g++ -mavx2 -mfma main.cpp -o app         # 只开启 AVX2 和 FMA 指令集
```

**`-march=native` 的风险**：在本机编译的代码，拷到老 CPU 上可能报 `Illegal instruction` 崩溃。生产部署要么不指定 `-march`，要么明确指定一个保守的目标架构。

### A.15 其他常用参数

| 参数 | 作用 |
|------|------|
| `-v` | 显示详细的编译过程（预处理、编译、链接各阶段调用的命令） |
| `-save-temps` | 保留中间产物（`.ii`、`.s`、`.o`），调试编译问题时用 |
| `-pipe` | 用管道代替临时文件传中间结果，稍快 |
| `-j<N>` | g++ 本身没有并行编译，用 CMake 的 `--parallel` 或 `make -j` |
| `-fno-rtti` | 禁用 RTTI（`dynamic_cast`），减小体积（某些嵌入式场景用） |
| `-fno-exceptions` | 禁用异常，减小体积（嵌入式/内核开发用） |
| `-fvisibility=hidden` | 默认隐藏符号（默认不导出，减少符号表体积，做库时推荐） |
| `-flto` | 链接时优化（Link-Time Optimization），跨文件内联，慢但优化更彻底 |

```bash
# 排查"为什么找不到头文件"：加 -v 看搜索路径
g++ -v main.cpp 2>&1 | grep -A 20 'search starts here'

# 排查"编译器到底生成了什么"：保留中间产物
g++ -save-temps main.cpp -o app
ls *.ii *.s *.o    # 看到所有中间文件

# 生产库：隐藏符号 + 链接时优化
g++ -fvisibility=hidden -flto -O2 mylib.cpp -shared -o libmylib.so
```

## 附录 B：常用编译组合速查

按场景给出"拿来就能用"的命令组合。

### B.1 日常学习/调试

```bash
# 单文件，C++20，带警告和调试信息，不优化
g++ -std=c++20 -Wall -Wextra -g -O0 main.cpp -o app

# 跑 Sanitizer 抓内存错误
g++ -std=c++20 -Wall -Wextra -g -O0 \
    -fsanitize=address,undefined -fno-omit-frame-pointer \
    main.cpp -o app_asan
```

### B.2 多文件项目

```bash
# 分开编译（改一个文件只重编那一个）
g++ -std=c++20 -c main.cpp -o main.o
g++ -std=c++20 -c add.cpp -o add.o
g++ main.o add.o -o app

# 一步到位（简单项目）
g++ -std=c++20 main.cpp add.cpp -o app
```

### B.3 Release 构建

```bash
# 优化 + 警告，不带调试信息
g++ -std=c++20 -O2 -Wall -Wextra main.cpp -o app_release

# 发布版：优化 + strip 掉调试符号
g++ -std=c++20 -O2 main.cpp -o app_release
strip app_release
```

### B.4 创建和使用库

```bash
# 静态库
g++ -c mylib.cpp -o mylib.o
ar rcs libmylib.a mylib.o
g++ main.cpp -L. -lmylib -o app

# 动态库
g++ -c -fPIC mylib.cpp -o mylib.o
g++ -shared -o libmylib.so mylib.o
g++ main.cpp -L. -lmylib -Wl,-rpath,. -o app
# -Wl,-rpath,. 把当前目录写进 RPATH，运行时不用设 LD_LIBRARY_PATH
```

### B.5 多线程

```bash
# std::thread
g++ -std=c++20 -pthread -g main.cpp -o app

# OpenMP
g++ -std=c++20 -fopenmp -O2 main.cpp -o app

# 测数据竞争
g++ -std=c++20 -pthread -fsanitize=thread -g -O0 main.cpp -o app_tsan
```

### B.6 外部依赖

```bash
# 链接 OpenCV
g++ main.cpp -I/usr/include/opencv4 -lopencv_core -lopencv_imgproc -o app

# 链接 CUDA（编译自定义算子）
g++ -std=c++17 -I/usr/local/cuda/include \
    -L/usr/local/cuda/lib64 -lcudart -lcublas \
    main.cpp -o app

# pybind11 扩展
g++ -O3 -Wall -shared -std=c++20 -fPIC \
    $(python -m pybind11 --includes) \
    module.cpp -o module$(python3-config --extension-suffix)
```

### B.7 排查问题专用

```bash
# 看头文件搜索路径
g++ -v -E -x c++ /dev/null 2>&1 | grep -A 20 'search starts here'

# 看一个宏是否被定义、值是多少
g++ -dM -E -x c++ /dev/null | grep -i '__cplusplus'
# 输出：#define __cplusplus 201703L（表示默认 C++17）

# 保留中间产物排查"预处理后长什么样"
g++ -save-temps -std=c++20 main.cpp -o app
# 生成 main.ii（预处理后）、main.s（汇编）、main.o（目标文件）

# 看二进制依赖了哪些库
ldd app

# 看二进制导出/未定义的符号
nm -C app | grep ' T '      # T = 已定义的导出符号
nm -C app | grep ' U '      # U = 未定义（要外部链接）
```

## 附录 C：参数位置与顺序

`g++` 命令行的参数顺序有讲究，记不住时遵循这个模板：

```
g++ [编译选项] [源文件] [库选项] -o [输出文件]
```

**核心规则**：

1. **`-I`、`-D`、`-std`、`-W`、`-O`、`-g`、`-fsanitize`**：编译期选项，放在源文件前后都行，习惯放前面
2. **`-L`、`-l`**：链接期选项，必须放在源文件**后面**（因为链接器按顺序扫描，源文件的未定义符号要靠后面的 `-l` 解决）
3. **`-l` 之间的顺序**：被依赖的库放后面（A 依赖 B，则 `-lA -lB`）

```bash
# ✅ 正确：编译选项在前，源文件在中，库在后
g++ -std=c++20 -O2 -Iinclude main.cpp -Llib -lmylib -lm -o app

# ❌ 错误：-l 放在源文件前面，链接器扫描到 main.cpp 时还不知道去哪找库
g++ -std=c++20 -Llib -lmylib main.cpp -o app
```

**用 CMake 就不用操心这些**：CMake 会自动处理参数顺序，这也是用 CMake 的好处之一（见第七章）。
