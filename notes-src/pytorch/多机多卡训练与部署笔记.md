## 一句话结论

**能。** 两台 8×A100 服务器可以组成 16 卡集群做训练或部署，PyTorch 原生支持（DDP / RPC / TorchRec）。关键是：① 所有机器网络互通且延迟低（推荐万兆/IB）；② 用 `torchrun` 启动 + 环境变量传 rank；③ 代码用 `DistributedDataParallel`（DDP）包装模型。"多机多卡"= 多台服务器 × 每台多张 GPU 协同计算，本质是用更多算力+显存来训更大模型或加速训练。

## 一、什么是"多机多卡"？

### 1.1 概念拆解

| 术语 | 含义 | 例子 |
|------|------|------|
| **单机单卡** | 1 台机器，1 张 GPU | 笔记本跑 RTX 3060 |
| **单机多卡** | 1 台机器，多张 GPU | 1 台服务器 8×A100，`DataParallel`/`DistributedDataParallel` |
| **多机多卡** | 多台机器，每台多张 GPU，**通过网络协同** | 2 台服务器各 8 卡，共 16 卡协同训练 |

```
单机多卡（1 台机器内）：              多机多卡（多台机器协同）：
┌────────────────────┐              ┌──────────┐    ┌──────────┐
│  Server A          │              │ Server A │    │ Server B │
│  ┌──┐┌──┐┌──┐┌──┐ │              │ ┌──────┐ │    │ ┌──────┐ │
│  │G0││G1││G2││G3│ │              │ │8×A100│◄┼────┼►│8×A100│ │
│  └──┘└──┘└──┘└──┘ │              │ └──────┘ │网络│ └──────┘ │
│  PCIe/NVLink 互联   │              └──────────┘    └──────────┘
└────────────────────┘               每台机器内 NVLink 高速互联
                                     机器之间走网络（10G/IB/RoCE）
```

### 1.2 为什么需要多机多卡

| 痛点 | 单机单卡解决不了 | 多机多卡的价值 |
|------|------------------|----------------|
| **模型太大** | 70B 模型 ~140GB，单卡 40G 装不下 | 切分到多卡（张量并行/流水线并行） |
| **训练太慢** | 1 卡训 GPT 要几年 | 16 卡近线性加速 |
| **batch 太小** | 单卡 batch=2，BN 统计不稳 | 多卡汇总，等效大 batch |
| **数据太多** | TB 级数据单卡吞吐跟不上 | 多卡并行读数据 |

### 1.3 三种并行策略（核心概念，必背）

```
数据并行（Data Parallelism）       张量并行（Tensor Parallelism）     流水线并行（Pipeline Parallelism）
                                                                
  每张卡拿完整模型                   模型的一层切到多张卡              模型的不同层分到不同卡
  每张卡处理不同数据                 同一层的矩阵分块算                像流水线一样接力前向
                                                                
  卡0: 模型副本 + batch0            卡0: 算 W的一半                  卡0: Layer 1-10
  卡1: 模型副本 + batch1            卡1: 算 W的另一半                卡1: Layer 11-20
  ...                              ...                              ...
                                                                
  反向后 AllReduce 汇总梯度         通信拼接结果                      micro-batch 流水传递
```

| 策略 | 切什么 | 通信量 | 实现难度 | 典型场景 |
|------|--------|--------|----------|----------|
| **数据并行（DP/DDP）** | 切数据 | 中（梯度 AllReduce） | ⭐ 简单 | 大多数训练，**首选** |
| **张量并行（TP）** | 切模型层内的矩阵 | 大（每层都通信） | ⭐⭐⭐ 难 | 超大模型（百亿+参数） |
| **流水线并行（PP）** | 切模型的层 | 小（只在边界通信） | ⭐⭐ 中 | 超大模型分层 |

**实战建议**：先用 **DDP（数据并行）**，绝大多数场景够用。模型大到单卡装不下才上 TP/PP（用 DeepSpeed / Megatron-LM / FSDP）。

## 二、你的场景能不能做？硬件 & 软件前提

### 2.1 你的配置

- 2 台 Linux 服务器
- 每台 8 × A100 40G
- 合计 16 卡，640GB 显存

**结论**：完全够用，这是非常标准的多机多卡配置。

### 2.2 硬件前提检查清单

| 项目 | 要求 | 检查命令 |
|------|------|----------|
| GPU 驱动 | 两台都装，版本一致 | `nvidia-smi` |

| SSH 免密 | 主节点能免密 SSH 到从节点 | 见 `troubleshooting/远程连接与传输/SSH 免密登录` |
| 端口可用 | 训练用端口（如 29500）未占用 | `ss -tlnp \| grep 29500` |

### 2.3 软件前提检查清单

| 项目 | 要求 | 检查命令 |
|------|------|----------|
| PyTorch 版本 | 两台一致（≥ 1.8 推荐 2.x） | `python -c "import torch; print(torch.__version__)"` |
| Python 版本 | 两台一致 | `python --version` |
| 代码路径 | 两台**相同路径**放相同代码 | 比如 `/home/user/project` |
| 数据路径 | 两台**相同路径**放相同数据 | 或用共享存储（NFS） |
| 依赖包 | 两台 `pip list` 一致 | 用 `requirements.txt` + 同步安装 |

> **关键坑**：两台机器的代码、数据、Python 环境**必须完全一致**，否则各卡跑出不同结果。最稳的做法是用 Docker 镜像 + NFS 共享数据。

### 2.4 网络是最大的瓶颈

```
单机内：GPU 间走 NVLink，带宽 600 GB/s（A100）
多机间：走网卡，带宽 10~100 Gbps（1.25~12.5 GB/s）
```

**多机通信比单机慢 50~500 倍**，所以：
- 能单机跑完的，别上多机（得不偿失）
- 多机训练时，通信占比高，要优化（梯度压缩、overlap 通信、梯度累积）

## 三、实战：用 DDP 跑多机多卡训练

### 3.1 整体架构

```
┌─────────────────────────────────────────────────────┐
│  你在 Server A 上执行 torchrun 启动命令              │
│  torchrun 会通过 SSH 自动在 Server B 上启动对应进程  │
└─────────────────────────────────────────────────────┘
        │                                    │
        ▼                                    ▼
┌──────────────────┐                ┌──────────────────┐
│  Server A (rank 0)│                │  Server B (rank 1)│
│  node_rank=0      │                │  node_rank=1      │
│  8 个进程         │                │  8 个进程         │
│  local_rank 0-7   │                │  local_rank 0-7   │
│  global rank 0-7  │                │  global rank 8-15 │
└──────────────────┘                └──────────────────┘
        │                                    │
        └────────── NCCL 通过网络通信 ────────┘
                   （AllReduce 梯度）
```

**关键概念**：

| 概念 | 含义 | 你的场景 |
|------|------|----------|
| `world_size` | 全局总进程数 | 16（2 机 × 8 卡） |
| `rank` | 全局进程编号 | 0~15 |
| `local_rank` | 单机内进程编号 | 0~7 |
| `node_rank` | 机器编号 | 0（Server A）、1（Server B） |
| `master` | 主进程，rank 0 | Server A 上的 GPU 0 |

### 3.2 代码模板（DDP 标准写法）

```python
# train.py - 多机多卡通用训练脚本
import os
import torch
import torch.nn as nn
from torch.utils.data import DataLoader, DistributedSampler
from torch.nn.parallel import DistributedDataParallel as DDP
import torch.distributed as dist


def setup():
    """初始化分布式环境，torchrun 会自动注入环境变量"""
    dist.init_process_group(backend="nccl")
    # torchrun 自动设置 RANK、LOCAL_RANK、WORLD_SIZE 环境变量
    local_rank = int(os.environ["LOCAL_RANK"])
    torch.cuda.set_device(local_rank)   # 每个进程绑定一张卡
    return local_rank


def cleanup():
    dist.destroy_process_group()


def main():
    local_rank = setup()
    
    # 1. 模型：放到当前进程的 GPU，再用 DDP 包装
    model = MyModel().to(local_rank)
    model = DDP(model, device_ids=[local_rank])
    
    # 2. 数据：DistributedSampler 自动切分数据给各卡
    dataset = MyDataset()
    sampler = DistributedSampler(dataset)
    dataloader = DataLoader(dataset, batch_size=32, sampler=sampler,
                            num_workers=4, pin_memory=True)
    
    # 3. 损失 & 优化器
    criterion = nn.CrossEntropyLoss()
    optimizer = torch.optim.Adam(model.parameters(), lr=1e-3)
    
    # 4. 训练循环
    for epoch in range(num_epochs):
        sampler.set_epoch(epoch)        # 重要！保证每轮数据打乱不同
        model.train()
        for batch_x, batch_y in dataloader:
            batch_x = batch_x.to(local_rank)
            batch_y = batch_y.to(local_rank)
            
            optimizer.zero_grad()
            outputs = model(batch_x)
            loss = criterion(outputs, batch_y)
            loss.backward()            # DDP 自动在反向时做 AllReduce
            optimizer.step()
        
        # 只在主进程保存模型，避免 16 个进程同时写文件冲突
        if dist.get_rank() == 0:
            torch.save(model.module.state_dict(), f"model_epoch{epoch}.pth")
    
    cleanup()


if __name__ == "__main__":
    main()
```

**DDP 代码与单卡的 3 处关键差异**：

| 步骤 | 单卡 | DDP 多卡 |
|------|------|----------|
| 模型 | `model.to(device)` | `model = DDP(model, device_ids=[local_rank])` |
| 数据 | `DataLoader(dataset, shuffle=True)` | `DataLoader(dataset, sampler=DistributedSampler)` |
| 保存 | 直接 `torch.save(model.state_dict())` | `torch.save(model.module.state_dict())` 且只在 rank 0 |

### 3.3 启动命令（torchrun）

**方式一：单机多卡（先在 1 台机器上验证）**

```bash
# 在 Server A 上，用 8 张卡跑
torchrun \
    --nproc_per_node=8 \
    --nnodes=1 \
    --node_rank=0 \
    --master_addr=127.0.0.1 \
    --master_port=29500 \
    train.py
```

**方式二：多机多卡（2 台机器，16 张卡）**

在 **Server A（主节点，node_rank=0）** 上执行：

```bash
torchrun \
    --nproc_per_node=8 \
    --nnodes=2 \
    --node_rank=0 \
    --master_addr=<Server A 的内网IP> \
    --master_port=29500 \
    train.py
```

同时（或提前）在 **Server B（从节点，node_rank=1）** 上执行：

```bash
torchrun \
    --nproc_per_node=8 \
    --nnodes=2 \
    --node_rank=1 \
    --master_addr=<Server A 的内网IP> \
    --master_port=29500 \
    train.py
```

**参数解释**：

| 参数 | 含义 |
|------|------|
| `--nproc_per_node=8` | 每台机器启动 8 个进程（每张卡一个） |
| `--nnodes=2` | 总共 2 台机器 |
| `--node_rank=0` | 当前机器的编号（A=0, B=1） |
| `--master_addr` | 主节点的 IP（两台都填 Server A 的 IP） |
| `--master_port` | 主节点通信端口（两台一致） |

> **torchrun 会自动处理**：环境变量注入、进程拉起、容错重启（`--rdzv_backend`）。比老版本的 `torch.distributed.launch` 更好用。

### 3.4 一键启动多机（用 pdsh 或 SSH 自动化）

手动在两台机器分别敲命令很烦，可以用 SSH 自动化：

```bash
# 在 Server A 上，一条命令同时拉起两台
# Server A 自己跑
torchrun --nproc_per_node=8 --nnodes=2 --node_rank=0 \
    --master_addr=<A_IP> --master_port=29500 train.py &

# SSH 到 Server B 拉起
ssh user@<B_IP> "cd /home/user/project && \
    torchrun --nproc_per_node=8 --nnodes=2 --node_rank=1 \
    --master_addr=<A_IP> --master_port=29500 train.py" &

wait
```

或用 `pdsh` 批量执行（更优雅）：

```bash
pdsh -w user@<A_IP>,user@<B_IP> "cd /home/user/project && torchrun ..."
```

### 3.5 训练 vs 部署的区别

| 维度 | 训练（DDP） | 部署/推理 |
|------|-------------|-----------|
| 目标 | 多卡协同更新模型参数 | 多卡并行处理请求，提高吞吐 |
| 通信 | 反向时 AllReduce 梯度 | 一般不通信（各卡独立推理） |
| 框架 | DDP / FSDP / DeepSpeed | Triton Inference Server / vLLM / TGI |
| 模型 | 每卡一份完整副本 | 可切分（TP）或复制 |
| 延迟 | 不敏感（批处理） | 敏感（要低延迟） |

**部署多机的典型方案**：

1. **每卡一份模型副本**（最简单）：每台机器起 8 个推理进程，前面挂负载均衡（Nginx/Traefik）分发请求。适合模型能装进单卡的情况。
2. **张量并行 + 多机**（大模型）：用 vLLM / TGI，跨机切模型。需高速网络。
3. **Ray Serve / Triton**：专业推理服务框架，自动管理多机多卡。

## 四、模型太大单卡装不下怎么办？（进阶）

你的 A100 是 40G，单卡能装约 10B 参数模型（FP32）。如果要训 70B+ 模型，需要更高级的并行：

### 4.1 FSDP（Fully Sharded Data Parallel）

PyTorch 官方，DDP 的升级版，**把模型参数/梯度/优化器状态都切分到各卡**，用时再聚合。

```python
from torch.distributed.fsdp import FullyShardedDataParallel as FSDP

model = MyLargeModel().to(local_rank)
model = FSDP(model)          # 替换 DDP 为 FSDP
# 其余训练代码和 DDP 一样
```

**DDP vs FSDP**：

| | DDP | FSDP |
|--|-----|------|
| 模型存储 | 每卡完整副本 | 切分到各卡 |
| 显存占用 | 高（每卡存全模型） | 低（分摊） |
| 通信 | 反向时梯度 AllReduce | 前向/反向时参数 AllGather |
| 适合 | 模型能装进单卡 | 模型装不进单卡 |

### 4.2 DeepSpeed / Megatron-LM

第三方框架，支持 3D 并行（DP + TP + PP），训百亿/千亿模型用。

```python
# DeepSpeed 示例
import deepspeed

model_engine, optimizer, dataloader, _ = deepspeed.initialize(
    model=model,
    optimizer=optimizer,
    config="ds_config.json",
)

for batch in dataloader:
    loss = model_engine(batch)
    model_engine.backward(loss)
    model_engine.step()
```

## 五、网络优化（多机性能关键）

### 5.1 选对网络接口

多机训练默认走默认网卡，如果机器有多个网卡（比如 1G 管理网 + 100G 训练网），要指定用哪个：

```bash
# 指定 NCCL 用的网卡
export NCCL_SOCKET_IFNAME=eth1        # 用 eth1（100G）而不是 eth0（1G）

# 或用 IP 段匹配
export NCCL_SOCKET_IFNAME=^docker0,lo  # 排除 docker 和 lo
```

### 5.2 常用 NCCL 环境变量

```bash
# 调试通信问题
export NCCL_DEBUG=INFO                # 打印通信日志

# 性能优化
export NCCL_IB_DISABLE=0              # 启用 InfiniBand（如果有）
export NCCL_P2P_DISABLE=0             # 启用 GPU 间 P2P（NVLink）

# 多网卡负载均衡
export NCCL_NET_GDR_LEVEL=5           # RDMA 等级
```

### 5.3 测试网络带宽

## 六、常见报错速查

| 报错 | 原因 | 解决 |
|------|------|------|
| `Connectively error: Connection refused` | 从节点连不上主节点 | 检查 `master_addr` IP、防火墙、29500 端口 |
| `NCCL error: unhandled system error` | NCCL 通信失败 | 设 `NCCL_DEBUG=INFO` 看详情，多为网卡/驱动问题 |
| `RuntimeError: Address already in use` | 端口被占 | 换 `--master_port=29501` |
| 各卡 loss 不一致 | 数据没切分或 `set_epoch` 漏了 | 用 `DistributedSampler` + 每轮 `sampler.set_epoch(epoch)` |
| `world_size` 对不上 | 两台 `--nproc_per_node` 不一致 | 确保都是 8，`--nnodes=2` |
| 从节点卡住不动 | 主节点没启动或 SSH 不通 | 先起主节点，检查 SSH 免密 |
| `CUDA out of memory` | 单卡显存不够 | 减 batch / 用 FSDP / 混合精度 |

## 七、面试回答模板

### Q1: 什么是多机多卡训练？为什么要用？

> 多机多卡训练是指把深度学习训练任务分布到多台服务器、每台多张 GPU 上协同完成。单机单卡受限于显存和算力，训不动大模型或训得太慢。多机多卡通过数据并行（DDP）让每张卡处理不同数据、反向时 AllReduce 汇总梯度，实现近线性加速；通过模型并行（TP/PP/FSDP）把大模型切分到多卡，解决单卡装不下的问题。代价是机器间通信开销大，需要高速网络（万兆/IB）支撑。

### Q2: DDP 和 DP 的区别？

> `DataParallel`（DP）是单机多卡的旧方案，单进程多线程，受 GIL 限制，GPU0 负载不均衡（负责汇总），效率低。`DistributedDataParallel`（DDP）是多机多卡的标准方案，每张卡一个进程，基于 NCCL 通信，负载均衡，支持多机，效率高。新项目一律用 DDP，不用 DP。

### Q3: 多机训练怎么启动？

> 用 PyTorch 的 `torchrun` 启动。每台机器执行 `torchrun` 命令，指定 `--nproc_per_node`（每机进程数）、`--nnodes`（机器数）、`--node_rank`（当前机器编号）、`--master_addr`（主节点 IP）、`--master_port`（通信端口）。主从节点都要执行，torchrun 会通过环境变量自动初始化分布式环境。代码里用 `dist.init_process_group("nccl")` 初始化，模型用 `DDP` 包装，数据用 `DistributedSampler` 切分。

## 八、深入追问

### 追问1：AllReduce 是什么？为什么 DDP 要用？

AllReduce 是一种集合通信原语：所有进程各自有一个梯度，AllReduce 后每个进程都得到所有梯度的**和/平均**。DDP 反向传播时，每张卡只算了自己那份数据的梯度，必须 AllReduce 才能得到全局梯度来同步更新。常用 Ring AllReduce 算法，通信量恒定（与卡数无关），效率高。

### 追问2：为什么 `sampler.set_epoch(epoch)` 必不可少？

`DistributedSampler` 默认用 `epoch` 作为随机种子打乱数据。如果不调 `set_epoch`，每轮用相同种子，导致每轮各卡拿到的数据顺序完全一样，模型学不到新东西。这是新手最常踩的坑。

### 追问3：混合精度 + 多机多卡怎么配？

```python
from torch.cuda.amp import GradScaler, autocast

scaler = GradScaler()

with autocast():
    outputs = model(batch_x)
    loss = criterion(outputs, batch_y)
scaler.scale(loss).backward()
scaler.step(optimizer)
scaler.update()
```

DDP 兼容 AMP，注意 `scaler` 在多进程下每张卡各自维护一份，无需特殊处理。

## 九、易混淆点

### 易混淆1：DDP vs DP vs FSDP vs DeepSpeed

| 方案 | 机器数 | 模型存储 | 通信 | 难度 | 适合 |
|------|--------|----------|------|------|------|
| `DataParallel` (DP) | 单机 | 全副本 | 主卡汇总 | ⭐ | ❌ 已淘汰 |
| `DistributedDataParallel` (DDP) | 单/多机 | 全副本 | AllReduce 梯度 | ⭐⭐ | **首选，模型装得下单卡** |
| `FSDP` | 单/多机 | 分片 | AllGather 参数 | ⭐⭐⭐ | 模型装不进单卡 |
| DeepSpeed ZeRO | 单/多机 | 分片+卸载 | 类似 FSDP | ⭐⭐⭐ | 超大模型，省显存 |

### 易混淆2：`rank` vs `local_rank` vs `node_rank`

```
2 台机器各 8 卡，共 16 进程：

Server A (node_rank=0):
  GPU0: rank=0,  local_rank=0
  GPU1: rank=1,  local_rank=1
  ...
  GPU7: rank=7,  local_rank=7

Server B (node_rank=1):
  GPU0: rank=8,  local_rank=0
  GPU1: rank=9,  local_rank=1
  ...
  GPU7: rank=15, local_rank=7

world_size = 16
master = rank 0 (Server A 的 GPU0)
```

- `rank`：全局唯一编号，0~15
- `local_rank`：单机内编号，0~7（两台都有 0~7）
- `node_rank`：机器编号，0 或 1

### 易混淆3：训练多机 vs 推理多机

训练多机：所有卡协同算梯度，强耦合，需 NCCL 实时通信。
推理多机：各卡独立处理请求，松耦合，前面挂负载均衡即可，一般不通信。

## 十、踩坑记录

> 待实际使用后补充。预留常见坑方向：

- **坑1（预期）**：两台机器 PyTorch 版本不一致 → 通信协议不兼容，报 NCCL 错。务必版本对齐。
- **坑2（预期）**：数据路径不一致 → 从节点读不到数据。用 NFS 共享或同步数据。
- **坑3（预期）**：默认走 1G 管理网卡 → 训练慢 100 倍。设 `NCCL_SOCKET_IFNAME` 指定高速网卡。
- **坑4（预期）**：`DistributedSampler` 忘了 `set_epoch` → 每轮数据顺序相同，模型不收敛。
- **坑5（预期）**：所有进程都保存模型 → 文件冲突。只在 `rank == 0` 保存。
- **坑6（预期）**：防火墙没放 29500 端口 → 从节点连不上主节点。
- **坑7（预期）**：`batch_size` 没除以 `world_size` → 等效 batch 过大，学习率要相应调（线性缩放规则）。
- **坑8（预期）**：Linux 服务器开了 IPv6 → NCCL 优先走 IPv6 但不通，强制 IPv4 `NCCL_SOCKET_FAMILY=AF_INET`。

## 十一、操作清单（你的场景：2 台 8×A100）

```bash
# === 准备阶段（两台都做）===
# 1. 检查 GPU 和驱动
nvidia-smi

# 2. 检查 PyTorch 版本（两台必须一致）
python -c "import torch; print(torch.__version__, torch.cuda.is_available())"

# 3. 同步代码和数据到两台（路径要一致）
rsync -avz /home/user/project/ user@<B_IP>:/home/user/project/
rsync -avz /home/user/data/ user@<B_IP>:/home/user/data/

# 4. 配置 SSH 免密（A → B）
ssh-keygen -t ed25519
ssh-copy-id user@<B_IP>

# 5. 测试网络带宽（期望 ≥10Gbps）
iperf3 -s                          # B 上
iperf3 -c <B_IP> -t 10             # A 上

# 6. 放行端口（两台）
sudo ufw allow 29500/tcp           # Ubuntu
# sudo firewall-cmd --permanent --add-port=29500/tcp && sudo firewall-cmd --reload  # CentOS

# === 先单机验证（Server A，8 卡）===
cd /home/user/project
torchrun --nproc_per_node=8 --nnodes=1 --node_rank=0 \
    --master_addr=127.0.0.1 --master_port=29500 train.py

# === 多机启动（16 卡）===
# Server A（node_rank=0，先起）
torchrun --nproc_per_node=8 --nnodes=2 --node_rank=0 \
    --master_addr=<A_IP> --master_port=29500 train.py &

# Server B（node_rank=1，后起）
ssh user@<B_IP> "cd /home/user/project && \
    torchrun --nproc_per_node=8 --nnodes=2 --node_rank=1 \
    --master_addr=<A_IP> --master_port=29500 train.py"

# === 监控 ===
watch -n 1 nvidia-smi              # 两台都看，确认 16 卡都在用
```
