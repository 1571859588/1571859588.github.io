## 阶段一：RAG 系统深挖（04:31-08:09）

### Q1 [05:39] 相似度匹配用了什么算法？

**面试官原话**：
> "我看你刚才聊的那个 RAG 里面，你用到一些 embedding 啊，以及一些相似度匹配的，你们相似度匹配用的是什么算法？"

**候选人现场回答**：余弦相似度（COS）+ BM25 + TF-IDF 混合检索

**✅ 标准参考答案**：

> 我们采用的是**混合检索（Hybrid Search）**策略，由两路召回 + 一路融合组成：
>
> **第一路：稠密检索（Dense Retrieval）**
> - 将文档 chunk 和用户 query 分别通过 embedding 模型（如 BGE-large）编码为向量
> - 使用**余弦相似度（Cosine Similarity）**计算 query 向量与文档向量的夹角余弦值
> - 余弦相似度公式：`cos(A,B) = (A·B) / (||A|| × ||B||)`
> - 余弦值范围 [-1, 1]，值越接近 1 表示语义越相似
> - 向量维度归一化后，余弦相似度等价于点积，可以用向量数据库（FAISS/Milvus）的 ANN 索引加速
>
> **第二路：稀疏检索（Sparse Retrieval）**
> - 使用 **BM25** 做关键词级召回——基于词频（TF）和逆文档频率（IDF）计算相关性
> - BM25 相比 TF-IDF 的优势：对词频做了饱和处理（避免长文档过匹配），引入了文档长度归一化
>
> **融合：RRF（Reciprocal Rank Fusion）**
> - 两路结果各自排 top-K，用 RRF 公式 `Σ 1/(k + rank)` 融合排序
> - 融合后送入 reranker（如 BGE-reranker）做精排，最终返回 top-N 给 LLM 生成答案
>
> **为什么需要混合检索**：
>
> - 稠密检索擅长捕捉**语义相近**的内容（"怎么修复" vs "如何解决"）
> - 稀疏检索擅长**精确关键词匹配**（函数名、API 名称、错误码）
> - 在我们 EDA 工具场景，API 名称拼写和大小写非常关键，纯向量检索容易漏掉，必须混合

### Q2 [06:52] 余弦 vs 其他相似度算法，各自适用场景？

**面试官原话**：
> "余弦或者其他的一些相似度算法，它们的适用场景是什么？什么场景下用余弦，什么场景下改用其他的？"

**候选人现场回答**：余弦对语义相似好，但 EDA 场景中 API 名称大小写变化会导致余弦漏掉，所以需要 BM25 补充

**✅ 标准参考答案**：

> 三种主要相似度度量及其适用场景：
>
> **1. 余弦相似度（Cosine Similarity）**
> - 衡量的是**方向**而非距离：`cos(A,B) = (A·B)/(||A||·||B||)`
> - 适合 **文本/语义** 场景：两个文本的向量方向一致 = 语义相似，不关心向量的模长（即不关心文档长度）
> - 典型用途：文本 embedding 检索、推荐系统中用户-物品相似度
> - 局限性：对精确字符匹配不敏感，"GetUserName" 和 "get_user_name" 在语义空间里可能很近，但代码层面完全不同
>
> **2. 欧氏距离（Euclidean Distance）**
> - 衡量的是**绝对距离**：`d(A,B) = √Σ(Ai-Bi)²`
> - 适合 **数值型 / 低维特征** 场景：坐标距离、价格差异、传感器读数
> - 受向量模长（scale）影响大→ 使用前通常需要标准化
> - 在高维稀疏空间中效果差（维度灾难）
>
> **3. 点积 / 内积（Dot Product）**
> - `A·B = Σ Ai×Bi`，值越大越相似
> - 向量已 L2 归一化时，点积 = 余弦相似度
> - 未归一化时，受向量模长影响——长向量即使方向不那么一致也可能得分高
> - 常用于模型最后一层输出（logits）和注意力机制中的 Q·K
>
> **选择决策表**：
>
> | 场景 | 推荐算法 | 原因 |
> |------|---------|------|
> | 文本语义检索 | 余弦 | 只关心语义方向，不关心文档长度 |
> | 图像特征匹配 | 欧氏/余弦 | 取决于特征是否归一化 |
> | 推荐系统评分预测 | 点积 | 矩阵分解模型天然输出 |
> | 代码/API 精确匹配 | BM25（非向量） | 关键词精确匹配优于语义模糊匹配 |

### Q3 [08:00] Agent 做检索 vs 传统 RAG

**面试官原话**：
> "现在也有一种思路，就是用 Agent 的方式来做检索。这两种思路你怎么看？"

**候选人现场回答**：之前试过大模型判断 trunk/query 相关性迭代检索，但成本高，没上线；认为 Agent 是未来趋势但目前业务不需要

**✅ 标准参考答案**：

> 这是 **标准 RAG 和 Agentic RAG** 的对比问题：
>
> **传统 RAG（标准流程）**：
> ```
> Query → Embedding → ANN检索(一次) → Rerank → 拼接→ LLM生成 → 回答
> ```
> - 一次检索 + 一次生成，流程固定
> - 优点：延迟低（<1s）、成本可控、实现简单
> - 缺点：检索质量取决于 query 表达，复杂问题（多跳推理、需要多文档信息整合）表现差
>
> **Agentic RAG（智能检索）**：
> ```
> Query → Agent规划 → 子查询1 → 检索 → 评估 → 子查询2 → 检索 → 评估 → ... → 综合生成
> ```
> - Agent 可以**多次迭代检索**，根据中间结果调整策略
> - 核心技术：Self-RAG、CRAG（Corrective RAG）、ReAct
>
> **具体差异**：
> | 维度 | 传统 RAG | Agentic RAG |
> |------|---------|------------|
> | 检索次数 | 固定 1 次 | 动态，可能多次 |
> | Query 处理 | 直接用原始 query | Agent 可改写、拆分、补全 query |
> | 结果评估 | 无/只用 reranker | Agent 判断检索质量，不行就换策略 |
> | 多源检索 | 不支持 | 可同时查向量库+关键词库+API |
> | 延迟 | 低（<1s） | 高（可能 5-10s+） |
> | 成本 | 1 次 LLM 调用 | 多次 LLM 调用 |
> | 适用场景 | 简单 FAQ | 复杂推理、多步查询 |
>
> **我的判断**：
> - 不能一刀切说"Agent 更好"——它是一个**准确率换延迟和成本**的 trade-off
> - 实际推荐分两层：高频简单查询走传统 RAG（毫秒级），复杂问题检测后升级为 Agent 检索
> - 如果你们要做客服 Agent，大概率也会是这种分层架构

### Q4 [09:26] 只用 Agent 做文档检索，怎么做？

**面试官原话**：
> "比如说就是检索文档，基于关键词来检索文档，如果只用 Agent 来做的话，是怎么做的？"

**候选人现场回答**：两种方法——Agent 猜相似关键词 grep 文档；把特定 API 做成支持库让 Agent 匹配

**✅ 标准参考答案**：

> Agent 做文档检索的核心是 **Tool Use + ReAct 循环**：
>
> **Step 1：给 Agent 配备工具集**
> ```
> Tools = [
> semantic_search(query, top_k)  # 向量语义检索
> keyword_search(query, top_k)   # BM25 关键词检索
> grep_search(pattern, path)     # 精确文本搜索
> read_file(path, offset, limit) # 读取具体文件内容
> ]
> ```
>
> **Step 2：Agent 推理-行动循环（ReAct）**
> ```
> Thought: 用户在问 BGE 模型的 API 用法，这涉及精确的函数名
> Action: keyword_search("BGE model encode function")
> Observation: 返回了 5 个文档片段，但都没有给出具体的 API 调用方式
> 
> Thought: 关键词检索没找到，可能是函数名拼写或大小写不同
> Action: grep_search("def encode", "docs/bge/")
> Observation: 找到了 `def Encode(text, model_name)` 函数定义
> 
> Thought: 需要看完整的参数说明
> Action: read_file("docs/bge/api.md", 0, 200)
> Observation: 获取到完整文档
> 
> Answer: BGE 模型调用方式是...
> ```
>
> **Step 3：结构化输出**
>
> - Agent 整合多次检索结果，生成带引用的回答
> - 如果检索不到，诚实地告诉用户并建议替代方案
>
> **与纯 RAG 的区别**：
> - 纯 RAG 是**固定路径**：query → 一次检索 → LLM 回答
> - Agent 检索是**自适应路径**：query → 选择工具 → 检索 → 评估 → 不够就换工具再检索 → 回答
>
> **关键设计要点**：
> - 工具描述要写得清晰（Agent 需要理解每个工具干什么、参数是什么）
> - 加一个 max_iterations 防止无限循环
> - 如果能力有限（只搜文档不能执行代码），在 system prompt 里声明边界

## 阶段二：Agent 系统设计（10:55-24:58）

### Q5 [10:51] 设计一个天气查询 Agent

**面试官原话**：
> "就是现在让你设计一个天气查询的 Agent，你会怎么设计？"

**候选人现场回答**：用户输入自然语言 → 提取地点和时间 → 信息不全就追问 → 调用天气 API

**面试官追问 [12:10]**：Agent 的输出是什么？工具怎么结合？

**候选人现场回答**：把天气 API 封装成 function，给大模型 function 定义，模型返回结构化调用参数

**✅ 标准参考答案**：

> **整体架构（基于 Function Calling）**：
>
> ```
> 用户: "今天北京天气怎么样？"
>       │
>       ▼
> ┌─────────────────┐
> │   System Prompt  │  定义 Agent 角色 + 可用工具
> └────────┬────────┘
>       │
>       ▼
> ┌─────────────────┐
> │   LLM 推理       │  理解意图，决定是否调用工具
> └────────┬────────┘
>       │ (需要调用 get_weather)
>       ▼
> ┌─────────────────┐
> │   Function Call  │  输出: {name: "get_weather", args: {city:"北京", date:"今天"}}
> └────────┬────────┘
>       │
>       ▼
> ┌─────────────────┐
> │   执行函数        │  HTTP request → 天气 API → 返回 JSON
> └────────┬────────┘
>       │
>       ▼
> ┌─────────────────┐
> │   LLM 总结       │  将 API 返回的 JSON 转成自然语言
> └────────┬────────┘
>       │
>       ▼
>   "北京今天晴，气温 25-35°C，南风 2 级"
> ```
>
> **Function Schema 设计示例**：
> ```json
> {
> "name": "get_weather",
> "description": "查询指定城市和日期的天气信息",
> "parameters": {
>  "type": "object",
>  "properties": {
>    "city": {
>      "type": "string",
>      "description": "城市名称，如'北京'、'上海'"
>    },
>    "date": {
>      "type": "string",
>      "description": "日期，格式 YYYY-MM-DD。'今天'→当天日期，'明天'→次日"
>    }
>  },
>  "required": ["city"]
> }
> }
> ```
>
> **信息补全策略**：
>
> - 缺地点 → "请问你想查哪个城市的天气？"
> - 缺日期 → 默认当天
> - 地点模糊（如"朝阳"）→ "你是指北京朝阳区还是辽宁朝阳市？"
>
> **System Prompt 示例**：
>
> ```
> 你是一个天气查询助手。用户会向你询问天气信息。
> 当用户提供的查询信息不完整时（缺少地点），你需要追问补充。
> 调用 get_weather 函数获取天气数据后，用友好的语言回复用户。
> ```

### Q6 [14:25] Skill vs MCP

**面试官追问**：Skill 的目的是什么？什么时候用 Skill？

**候选人现场回答**：MCP 会给出大量参数占上下文，Skill 像目录更轻量

**✅ 标准参考答案**：

> **Skill 的定义与目的**：
> - Skill 是对工具能力的**高层抽象和封装**，本质是"意图 → 工具调用链"的映射
> - 目的：让 Agent 在众多工具中**快速路由到正确的工具**，而不是每次都把所有工具定义塞进 context
>
> **MCP（Model Context Protocol）**：
> - 一个标准化的工具注册协议，定义了工具的发现、描述、调用方式
> - 优点：标准化、跨平台
> - 缺点：所有注册工具的 schema 都要发给 LLM，工具多了会**严重占用上下文窗口**
>
> **Skill 的设计思路**：
> ```
> 用户说"查天气" → Skill路由 → 只加载 weather 相关的 tool schema
> 用户说"订机票" → Skill路由 → 只加载 flight 相关的 tool schema
> ```
> - Skill 像一个**工具箱管理器**：先识别用户意图，再按需加载对应工具
> - 避免一次性把所有工具的 function definition 塞进 system prompt
>
> **对比总结**：
> | 维度 | MCP（工具协议） | Skill（能力封装） |
> |------|----------------|------------------|
> | 层次 | 协议层，定义工具如何注册和描述 | 应用层，定义工具如何组织和路由 |
> | 关注点 | 工具的标准化描述 | 工具的智能调度 |
> | 关系 | 底层基础设施 | 上层调度逻辑 |
> | 类比 | USB 协议（设备如何连接） | 设备管理器（用哪个设备） |
>
> **使用场景**：
> - 工具数量少（<10 个）→ 直接用 MCP 给所有 tool schema
> - 工具数量多（>20 个）→ 用 Skill 做路由，按意图按需加载
> - 复杂业务场景 → Skill 可以封装多步工具调用链（如"订机票" Skill = 查航班 + 比价 + 下单 + 支付）

### Q7 [15:32] 多功能扩展：天气 + 出行 + 闲聊 + 旅游攻略

**面试官原话**：

> "我现在这个天气系统想要增加一些功能：闲聊、出行计划、旅游攻略……增加了这么多功能之后，你会怎么设计这个 Agent？"

**候选人现场回答**：意图识别 → 分发到不同处理逻辑；记忆按主题分层；不同意图切换不同记忆

**✅ 标准参考答案**：

> 这是一个**多意图 Agent 架构**问题，核心思路是**路由层 + 执行层分离**：
>
> **架构设计**：
>
> ```
>                ┌──────────────┐
> 用户输入 ──────▶│  意图识别器    │──────▶ 意图分类
>                │  (Router)    │
>                └──────┬───────┘
>                       │
>       ┌───────┬───────┼───────┬───────┐
>       ▼       ▼       ▼       ▼       ▼
>    ┌────┐ ┌────┐ ┌────┐ ┌────┐ ┌────┐
>    │天气│ │出行│ │闲聊│ │攻略│ │兜底│
>    │Agent│Agent│Agent│Agent│Agent│
>    └──┬─┘ └──┬─┘ └──┬─┘ └──┬─┘ └──┬─┘
>       │      │      │      │      │
>       ▼      ▼      ▼      ▼      ▼
>    各自独立 System Prompt + Tool Set + Memory
> ```
>
> **意图识别（Router）**：
> ```python
> # 方法一：LLM 分类（灵活但成本高）
> intent = llm.classify(query, categories=["天气","出行","闲聊","攻略"])
> 
> # 方法二：关键词 + 小模型（低成本高吞吐）
> if any(kw in query for kw in ["天气","气温","下雨"]):
>  intent = "天气"
> elif any(kw in query for kw in ["机票","火车","酒店"]):
>  intent = "出行"
> ```
>
> **每个子 Agent 的独立配置**：
>
> | Agent | System Prompt | 工具集 | 典型流程 |
> |-------|-------------|--------|---------|
> | 天气 | 天气查询助手 | `get_weather`, `get_air_quality` | 提取地点→查天气→回复 |
> | 出行 | 出行规划助手 | `search_flight`, `search_hotel`, `plan_route` | 多轮收集信息→搜索→方案对比 |
> | 闲聊 | 友好聊天助手 | 无（纯对话） | 直接生成回复 |
> | 攻略 | 旅游顾问 | `search_attractions`, `search_restaurants`, `get_weather` | 多步信息搜集→整合→生成攻略 |
> | 兜底 | 通用助手 | 全部工具 | 尝试理解→引导用户明确需求 |
>
> **话题切换与上下文管理**：
>
> - 检测到意图变化时，自动切换 Agent
> - 前一个 Agent 的关键信息（如用户当前所在城市）可传递给新 Agent
> - 避免闲聊内容污染天气 Agent 的记忆

### Q8 [19:50] 记忆管理

**面试官原话**：
> "记忆指的是什么？"

**候选人现场回答**：用户对话历史 + 模型回答

**✅ 标准参考答案**：

> 记忆（Memory）在 Agent 系统中分为三个层次：
>
> **1. 短期记忆（Short-term / Working Memory）**
> - **内容**：当前对话窗口内的所有消息（最近 N 轮）
> - **存储**：直接存在 LLM 的 context window 中
> - **生命周期**：当前 session
> - **问题**：窗口有限（4K-128K tokens），长了就丢
>
> **2. 长期记忆（Long-term Memory）**
> - **内容**：跨 session 的重要信息——用户偏好、历史决策、知识积累
> - **存储**：向量数据库（ChromaDB/Milvus + embedding）+ 结构化存储
> - **典型实现**：
>   - **Mem0**：自动提取用户偏好存入向量库，每次对话时检索相关记忆注入 context
>   - **MemGPT**：给 LLM 一个"虚拟内存"系统，自动在 context 和外部存储间 swap
> - **检索策略**：用当前 query 去向量库搜相关记忆 → 注入 system prompt
>
> **3. 工作记忆（Scratchpad）**
>
> - **内容**：当前任务进展中的中间状态——已完成的子任务、待处理的步骤、收集到的信息
> - **存储**：结构化 JSON 或 Markdown，每次 LLM 调用时 append 到 system prompt
> - **生命周期**：当前任务
> - **示例**：订机票时，工作记忆记录 `{出发地:北京, 目的地:已查, 日期:6月30日, 航班:待确认}`
>
> **记忆管理策略**：
> ```
> ┌──────────────────────────────────────────┐
> │              System Prompt                │
> │  ┌──────────┐ ┌──────────────────────┐   │
> │  │ 长期记忆  │ │     工作记忆(任务状态)  │   │
> │  │(向量检索) │ │     (结构化JSON)       │   │
> │  └──────────┘ └──────────────────────┘   │
> │  ┌──────────────────────────────────┐    │
> │  │        短期记忆(最近N轮对话)        │    │
> │  └──────────────────────────────────┘    │
> └──────────────────────────────────────────┘
> ```
>
> **面试亮点补充**：
> - 可以提**摘要式压缩**：超过窗口时，用 LLM 对历史对话做摘要，保留关键信息丢掉废话
> - 可以提**分级记忆**：重要信息（用户偏好）持久化，普通信息（闲聊）会话结束丢弃

### Q9 [23:29] Multi-Agent 架构

**面试官原话**：
> "意图识别之后，有没有考虑 Multi-Agent？"

**候选人现场回答**：每个主题独立 Agent + 结果整合，承认是"不计成本"的方式

**✅ 标准参考答案**：

> **Multi-Agent 的核心意义**：
> - 单个 Agent 的能力有上限——system prompt 太长会互相干扰、工具太多选不准
> - Multi-Agent 让每个 Agent 专注一个领域，通过协作解决复杂问题
>
> **常见 Multi-Agent 架构模式**：
>
> **模式一：路由模式（Router-Based）**
> ```
>          ┌──────────┐
> 用户 ───▶│  Router   │───▶ Agent A(天气)
>          │ (分发器)  │───▶ Agent B(出行)
>          └──────────┘───▶ Agent C(闲聊)
> ```
> - 最简单，适合独立任务
> - 各 Agent 之间不通信
>
> **模式二：流水线模式（Pipeline）**
> ```
> Agent A ──▶ Agent B ──▶ Agent C ──▶ 输出
> (规划)     (检索)      (生成)
> ```
> - 适合有明确步骤的任务
> - 类似 MapReduce：先拆解 → 各 Agent 处理子任务 → 汇总
>
> **模式三：辩论/协作模式（Debate/Collaboration）**
> ```
>          ┌──────────┐
> 用户 ───▶│  Orchestrator │
>          └──┬───┬───┘
>       ┌─────┘   └─────┐
>       ▼               ▼
>   Agent A ───讨论─── Agent B
>       │               │
>       └──────┬────────┘
>              ▼
>          共识/综合输出
> ```
> - 适合需要批判性思考的任务（代码审查、方案评估）
> - ChatDev、AutoGen 等框架用的就是这种模式
>
> **设计要点**：
> - **通信协议**：Agent 之间用什么格式交换信息（自然语言 vs 结构化 JSON）
> - **任务分配**：谁决定哪个 Agent 做什么（中心化 Orchestrator vs 去中心化协商）
> - **冲突解决**：Agent A 和 B 意见不一致怎么办（投票、Orchestrator 仲裁、置信度加权）
> - **成本控制**：Multi-Agent 会成倍增加 LLM 调用次数，需要加 early stop 和 budget 限制
>
> **实际落地的务实做法**：
>
> ```
> 第一阶段：单 Agent + 意图路由（覆盖 80% 场景）
> 第二阶段：复杂意图引入 Multi-Agent（如"帮我规划北京三日游"需要天气+出行+攻略协作）
> 第三阶段：Agent 间引入评价/反馈机制（Agent A 产出后 Agent B 审核）
> ```

### Q11 [25:35] Java 并发模型

**面试官原话**：
> "Java 里面的并发模型是啥样的？并发是怎么做的？"

**✅ 标准参考答案**：

> Java 并发模型基于 **共享内存 + 线程**：
>
> **基础层——Thread 与 Runnable**：
> ```java
> // 方式一：继承 Thread
> class MyThread extends Thread {
>     public void run() { /* 并发执行的代码 */ }
> }
> new MyThread().start();
>
> // 方式二：实现 Runnable（推荐，更灵活）
> new Thread(() -> { /* 并发执行的代码 */ }).start();
> ```
>
> **线程池——ThreadPoolExecutor**（避免频繁创建/销毁线程）：
> ```java
> ExecutorService pool = Executors.newFixedThreadPool(10);
> pool.submit(() -> doWork());
> ```
> 核心参数：corePoolSize（核心线程数）、maximumPoolSize（最大线程数）、keepAliveTime（空闲线程存活时间）、workQueue（任务队列）
>
> **同步机制**：
> - `synchronized`：互斥锁，保证同一时刻只有一个线程执行临界区
> - `ReentrantLock`：可重入锁，比 synchronized 更灵活（可定时、可中断、公平锁）
> - `volatile`：保证变量可见性（不保证原子性）
> - `AtomicInteger/CAS`：无锁并发，利用 CPU 的 CAS 指令实现原子操作
>
> **高级并发工具（JUC 包）**：
> | 工具 | 用途 |
> |------|------|
> | `ConcurrentHashMap` | 线程安全的 HashMap，分段锁 + CAS |
> | `CountDownLatch` | 等待 N 个线程完成 |
> | `CyclicBarrier` | N 个线程互相等待到齐后一起执行 |
> | `Semaphore` | 控制同时访问资源的线程数 |
> | `CompletableFuture` | 异步编程，链式组合多个异步任务 |
>
> **Java vs Python 的并发差异**：
> - Java：真多线程并行（JVM 线程映射到 OS 线程）
> - Python：GIL（全局解释器锁）限制同一时刻只有一个线程执行 Python 字节码，多线程适合 IO 密集型，CPU 密集用 multiprocessing

### Q13 [26:19] MySQL 索引数据结构

**面试官原话**：
> "MySQL 的索引的数据结构是什么样的？"

**⚠️ 常见误区**：不能回答"红黑树"！MySQL InnoDB 默认索引用的是 **B+ 树**。

**✅ 标准参考答案**：

> **MySQL InnoDB 索引 = B+ 树**
>
> **B+ 树结构**：
>
> ```
>                  [30 | 60]                    ← 非叶子节点：只存索引 key
>                 /    |    \
>        [10|20]    [40|50]    [70|80|90]        ← 非叶子节点
>        /  |  \     /  |  \     /  |  |  \
>      [1] [10] [20] [30] [40] [50] [60][70][80][90]  ← 叶子节点：存数据(双向链表连接)
>       ↓   ↓    ↓    ↓    ↓    ↓    ↓  ↓  ↓  ↓
>     data data data data data data data............
> ```
>
> **B+ 树的关键特性**：
> 1. **所有数据存在叶子节点**：非叶子节点只存 key 做索引，不存数据
> 2. **叶子节点形成有序双向链表**：范围查询极快（沿着链表直接扫）
> 3. **多叉树**：每个节点可以存很多 key（几百个），树高度低（3-4 层就能存千万级数据）
>
> **为什么不用红黑树/二叉树？**
> - 二叉树高度 = log₂(N)，千万数据 = 24 层 = 24 次磁盘 IO
> - B+ 树高度 = log_100(N)，千万数据 = 3-4 层 = 3-4 次磁盘 IO
> - 数据库瓶颈在**磁盘 IO**，树越矮越快
>
> **为什么不用 B 树？**
> - B 树的非叶子节点也存数据，节点能存的 key 更少 → 树更高
> - B 树不支持高效范围查询（需要中序遍历，不能沿着链表直接扫）
>
> **为什么不用 Hash 索引？**
> - Hash 只支持等值查询，不支持范围查询（`>`, `<`, `BETWEEN`, `ORDER BY`）
> - InnoDB 也支持自适应 Hash 索引（AHI），但只是辅助加速
>
> **聚集索引 vs 二级索引**：
> - **聚集索引**：索引的叶子节点存的是**完整的行数据**（主键索引），InnoDB 必须有主键
> - **二级索引**：叶子节点存的是**主键值**，查到后需要**回表**（再去聚集索引查完整数据）
> - **覆盖索引**：如果查询的列都在二级索引中，不需要回表，性能最优
>
> **记忆口诀**：B+ 树 = 矮胖子（多叉）+ 数据全在叶子 + 叶子串成链表

### Q16 [33:40] 长对话 AI 遗忘问题

**面试官原话**：
> "会话特别长，AI 写着写着就忘了之前自己要做什么事，有没有遇到过？怎么解决？"

**✅ 标准参考答案**：

> 这个问题的本质是**上下文窗口有限 + LLM 注意力衰减**：
>
> **为什么会忘**：
> 1. **窗口限制**：模型能"看到"的 token 有上限（如 128K），超了最早的消息就被截断
> 2. **Lost in the Middle**：研究表明 LLM 对 prompt 中间位置的注意力最弱——你放在中间的信息最容易"被忽略"
> 3. **上下文污染**：无关的历史信息挤占了关键信息的位置
>
> **解决策略（从简单到复杂）**：
>
> **策略一：摘要式压缩（Summary Compression）**
>
> ```
> 原始对话(10轮) → LLM 摘要 → "用户想要一个登录页面，用 React + TypeScript，已经实现了基本组件"
> ```
> - 定期对历史对话做摘要，只保留关键决策和约束
> - 摘要替代原始对话，节省 80%+ token
>
> **策略二：结构化关键信息提取**
> ```yaml
> # 不在对话中存，而是单独维护一个结构化记录
> task: "实现用户登录页面"
> tech_stack: ["React", "TypeScript", "Tailwind CSS"]
> decisions:
>   - "使用 JWT 认证"
>   - "登录表单包含邮箱和密码两个字段"
> todo:
>   - ✅ 创建 LoginForm 组件
>   - ✅ 添加表单验证
>   - ⏳ 对接后端 API
> ```
> - 每次对话开始前，把这个结构化记录注入 system prompt
> - 关键信息永不丢失
>
> **策略三：滑动窗口 + 关键帧**
> ```
> [完整摘要] [第5轮关键帧] [第6轮关键帧] [最近3轮完整对话]
> ```
> - 最近几轮保留完整
> - 中间轮只保留"关键帧"（重要的决策点、代码片段）
> - 最早的内容压缩为摘要
>
> **策略四：任务拆分**
> - 超长任务拆成多个子任务
> - 每个子任务用独立 session（干净的上下文）
> - 子任务之间通过结构化文档传递信息
>
> **策略五：长上下文模型 + RAG 混合**
> - 用长上下文模型（如 Gemini 1M token）作为基础
> - 同时用 RAG 把关键历史信息检索出来强化注入
> - 双重保障

## 阶段五：计算机基础继续（35:47-40:19）

### Q17 [36:38] TCP vs UDP

**面试官原话**：
> "TCP 跟 UDP 有什么区别？"

**✅ 标准参考答案**：

> | 维度 | TCP | UDP |
> |------|-----|-----|
> | 连接 | **面向连接**（三次握手建立，四次挥手释放） | **无连接**，直接发送 |
> | 可靠性 | **可靠**（确认+重传+序号+校验） | **不可靠**，不保证送达 |
> | 顺序 | **有序**（序号保证数据按序到达） | **无序**，先发的不一定先到 |
> | 速度 | 慢（握手+确认+流量控制开销大） | 快（无连接建立，无确认等待） |
> | 流量控制 | 有（滑动窗口） | 无 |
> | 拥塞控制 | 有（慢启动、拥塞避免、快重传） | 无 |
> | 数据边界 | **流式**（字节流，无消息边界） | **数据报**（保留消息边界） |
> | 头部大小 | 20-60 字节 | 固定 8 字节 |
> | 典型应用 | HTTP/HTTPS、SSH、FTP、邮件 | DNS、视频直播、VoIP、在线游戏 |
>
> **核心记忆点**：
>
> - TCP 像**打电话**——先拨号（建立连接），确认对方在听，按顺序说话
> - UDP 像**发短信**——直接发出去，不管对方收没收到
> - TCP 的可靠性代价 = 头部大 + 握手延迟 + 确认开销
> - UDP 的"不靠谱"在实时场景反而是优势（视频通话丢几帧无所谓，重传反而卡顿）

### Q18 [37:16] TCP 为什么三次握手？

**面试官原话**：
> "TCP 为什么三次握手？两次不行？四次可以吗？"

**✅ 标准参考答案**：

> **三次握手的过程**：
> ```
> 客户端                          服务端
>   │                               │
>   │─────── SYN (seq=x) ──────────▶│  ① 客户端：我要连接
>   │                               │
>   │◀── SYN+ACK (seq=y, ack=x+1) ──│  ② 服务端：收到，我准备好了
>   │                               │
>   │─────── ACK (ack=y+1) ────────▶│  ③ 客户端：确认收到
>   │                               │
>   │     连接建立，开始传数据         │
> ```
>
> **为什么两次不行？**
>
> **原因：防止已失效的连接请求报文突然传到服务端**
>
> ```
> 客户端                              服务端
>   │                                   │
>   │── SYN(seq=100) ──→ (网络延迟...)   │
>   │                                   │
>   │   ... 等了很久没收到 ACK ...        │
>   │                                   │
>   │── SYN(seq=200) ────────────────→  │  ← 客户端重发
>   │◀──   连接建立，正常通信    ──────→  │
>   │◀──   连接关闭              ──────→  │
>   │                                   │
>   │            此时 第一个 SYN(seq=100) 终于到达 ──→ │
>   │                                   │
>   │◀── SYN+ACK ──────────────────────│  ← 服务端以为新连接！
>   │                                   │
>   │  (客户端：我没请求连接啊，忽略)       │  服务端：傻等着数据...
>   │                                   │  资源浪费！
> ```
>
> - 如果只有两次握手，服务端收到过期的 SYN 就会直接建立连接
> - 三次握手中，客户端可以不发最后的 ACK，服务端就不会错误地建立连接
>
> **为什么四次可以但没必要？**
> - 第二次握手（SYN+ACK）已经完成了服务端的"响应确认"和"连接请求"两项功能
> - 拆成四次握手 = SYN → ACK → SYN → ACK，多了 1 个 RTT，毫无必要
> - 三次是**理论最小值（保证双方确认收发能力）+ 工程最优值**
>
> **记忆口诀**：
> - 第一次：客户端说"我要连"
> - 第二次：服务端说"我收到了，我也要连"
> - 第三次：客户端说"好的那开始吧"
> - 二次握手不能防止历史连接，四次浪费一轮 RTT

### Q19 [39:20] 进程 vs 线程

**面试官原话**：
> "进程跟线程有什么区别？"

**✅ 标准参考答案**：

> | 维度 | 进程（Process） | 线程（Thread） |
> |------|---------------|---------------|
> | 定义 | **资源分配**的最小单位 | **CPU 调度**的最小单位 |
> | 内存 | 独立地址空间，互不干扰 | 共享进程内存（堆、全局变量） |
> | 通信 | IPC（管道、消息队列、共享内存） | 直接读写共享变量 |
> | 创建开销 | 大（复制父进程地址空间） | 小（只需分配栈空间） |
> | 切换开销 | 大（切换页表、刷新 TLB） | 小（同进程内切换只换寄存器和栈） |
> | 独立性 | 一个进程崩溃不影响其他进程 | 一个线程崩溃可能拖垮整个进程 |
> | 并发性 | 多核真并行 | 多核真并行（C++/Java）/ Python 受 GIL 限制 |
>
> **经典比喻**：
> - 进程 = 一栋**独立房子**（有自己的厨房、卫生间）
> - 线程 = 房子里的**不同房间**（共享厨房，各有自己的书桌）
>
> **一个线程崩溃会影响进程吗？**
> - C/C++：如果不做信号处理，segfault 等异常可能导致整个进程终止
> - Java：未捕获异常只影响出问题的线程，其他线程继续运行
> - Go：goroutine panic 没有 recover 会崩溃整个进程
>
> **还有协程（Coroutine）**：
> - 比线程更轻量，用户态调度（不需要内核态切换）
> - Go 的 goroutine（几 KB 栈，几十万个轻松跑）
> - Python 的 asyncio（单线程事件循环，适合 IO 密集）
>
> | | 进程 | 线程 | 协程 |
> |--|------|------|------|
> | 调度者 | 操作系统 | 操作系统 | 用户程序 |
> | 切换成本 | 高 | 中 | 低 |
> | 内存 | 独立 | 共享 | 共享 |
> | Go 对应 | — | — | goroutine |

### Q20 [40:02] 进程间通信（IPC）

**面试官原话**：

> "进程之间的通信方式（是什么）？"

**✅ 标准参考答案**：

> **六大经典 IPC 方式**：
>
> | 方式 | 原理 | 特点 | 典型场景 |
> |------|------|------|---------|
> | **管道（Pipe）** | 内核缓冲区，一端写一端读 | 半双工、父子进程间 | `ls | grep txt` |
> | **命名管道（FIFO）** | 有文件名的管道 | 无亲缘关系进程也可用 | 日志收集 |
> | **消息队列** | 消息链表，按类型收发 | 异步、有边界 | 任务分发 |
> | **共享内存** | 多个进程映射同一块物理内存 | **最快**，但需同步 | 大数据量传输 |
> | **信号量（Semaphore）** | 计数器，控制访问资源数 | 主要用于同步 | 生产者-消费者 |
> | **Socket** | 网络通信接口 | 跨机器、通用 | 所有网络通信 |
> | **信号（Signal）** | 异步通知 | 简单但信息量少 | `kill -9`、`SIGTERM` |
>
> **速度排序**：共享内存 > 管道 ≈ 消息队列 > Socket
>
> **实际开发中最常见**：
>
> - **微服务之间**：HTTP/gRPC（基于 Socket）
> - **同机器进程间大数据量**：共享内存（如 Python multiprocessing.shared_memory）
> - **简单数据传递**：管道或 Redis 作为中间件
> - **分布式系统**：消息队列（Kafka/RabbitMQ）

### Q21 [40:19] 子数组和为 K

**题目**：给定一个整数数组 `nums` 和一个整数 `k`，返回和为 `k` 的连续子数组的个数。

> LeetCode 560: Subarray Sum Equals K 
>
> [560. 和为 K 的子数组 - 力扣（LeetCode）](https://leetcode.cn/problems/subarray-sum-equals-k/description/)

**✅ 解法：前缀和 + 哈希表**

```python
def subarraySum(nums: list[int], k: int) -> int:
    """
    时间 O(n)，空间 O(n)
    核心：pref[j] - pref[i-1] = k → pref[i-1] = pref[j] - k
    遍历到 j 时，看哈希表里有多少个 pref[j] - k
    """
    prefix_sum = 0
    count = 0
    prefix_count = {0: 1}  # 初始化 {0:1} 处理从开头开始的子数组
    
    for num in nums:
        prefix_sum += num
        if prefix_sum - k in prefix_count:
            count += prefix_count[prefix_sum - k]
        prefix_count[prefix_sum] = prefix_count.get(prefix_sum, 0) + 1
    
    return count
```

**推导过程**：

```
pref[j] - pref[i-1] = k   →   pref[i-1] = pref[j] - k

举例：nums = [1, 2, 3], k = 3
i=0, num=1: pre=1, pre-k=-2(无), {0:1, 1:1}
i=1, num=2: pre=3, pre-k=0(有1个!), count=1, {0:1, 1:1, 3:1}  → [1,2]
i=2, num=3: pre=6, pre-k=3(有1个!), count=2, {0:1, 1:1, 3:1, 6:1}  → [3]
答案: 2
```

**同类变体**：

| 变体 | 题目 | 差异 |
|------|------|------|
| 是否存在 | 子数组和是否等于 K | Set 替代 Map（不计数） |
| 最长子数组 | 和为 K 的最长子数组长度 | Map 只存首次出现位置 |
| 取模版 | 和可被 K 整除的子数组 | `(pre % K + K) % K` 处理负数 |

### Q22 两数之和（Two Sum）

**题目**：给定数组 `nums` 和目标值 `target`，找出和为 target 的两个数的下标。

> LeetCode 1，字节/阿里/腾讯高频
>
> [1. 两数之和 - 力扣（LeetCode）](https://leetcode.cn/problems/two-sum/description/?envType=study-plan-v2&envId=top-100-liked)

**✅ 解法：哈希表一次遍历**

```python
def twoSum(nums: list[int], target: int) -> list[int]:
    """
    时间 O(n)，空间 O(n)
    核心：a + b = target → b = target - a
    遍历时，对每个 a，看 target-a 是否在哈希表中
    """
    seen = {}  # 数值 → 下标
    
    for i, num in enumerate(nums):
        complement = target - num
        if complement in seen:
            return [seen[complement], i]
        seen[num] = i
    
    return []  # 题目保证有解
```

**推导过程**：

```
nums = [2, 7, 11, 15], target = 9

i=0, num=2: complement=7, 不在seen, seen={2:0}
i=1, num=7: complement=2, 在seen! 返回 [0, 1]
```

**面试追问**：

| 追问 | 答案 |
|------|------|
| 如果数组有序？ | 双指针，O(n) 时间 O(1) 空间 |
| 如果找三个数？ | 排序 + 固定一个 + 双指针 Two Sum，O(n²) |
| 如果返回所有可能的对（不重复）？ | 排序 + 双指针 + 跳过重复元素 |

**双指针版本（有序数组）**：

```python
def twoSum_sorted(nums: list[int], target: int) -> list[int]:
    """时间 O(n)，空间 O(1)，前提：数组已排序"""
    left, right = 0, len(nums) - 1
    while left < right:
        s = nums[left] + nums[right]
        if s == target:
            return [left, right]
        elif s < target:
            left += 1
        else:
            right -= 1
    return []
```

### Q23 二叉树的最近公共祖先（LCA）

**题目**：给定二叉树和两个节点 p、q，找 p 和 q 的最近公共祖先。

> LeetCode 236，字节/阿里/腾讯必考题
>
> [236. 二叉树的最近公共祖先 - 力扣（LeetCode）](https://leetcode.cn/problems/lowest-common-ancestor-of-a-binary-tree/description/)

**✅ 解法：递归后序遍历**

```python
class TreeNode:
    def __init__(self, x):
        self.val = x
        self.left = None
        self.right = None

def lowestCommonAncestor(root: TreeNode, p: TreeNode, q: TreeNode) -> TreeNode:
    """
    时间 O(n)，空间 O(h)（递归栈）
    核心：p 和 q 必在 LCA 的两侧（或其一就是 LCA）
    后序遍历：左子树找 → 右子树找 → 判断当前节点
    """
    # Base case: 空节点 / 找到 p / 找到 q
    if not root or root == p or root == q:
        return root
    
    # 递归搜索左右子树
    left = lowestCommonAncestor(root.left, p, q)
    right = lowestCommonAncestor(root.right, p, q)
    
    # 左右都有 → 当前节点就是 LCA
    if left and right:
        return root
    
    # 只有一边有 → LCA 在那一边
    return left if left else right
```

**递归过程图解**：

```
p = 5, q = 1

        3           ← left=5, right=1, 左右都有 → 返回3 ✓
       / \
      5   1         ← 左子树返回5(找到p), 右子树返回1(找到q)
     / \   \
    6   2   8
       / \
      7   4

另例：p = 5, q = 4
        3           ← left=5, right=None, 只有左边 → 返回5
       / \
  --> 5   1         ← left=None, right=4 → 返回4
     / \
    6   2   <--
       / \
      7   4(q)
```

**复杂度**：
- **时间**：O(n)，最坏遍历整棵树
- **空间**：O(h)，h 为树高。平衡树 O(log n)，退化成链表 O(n)

**面试追问**：

| 追问 | 答案 |
|------|------|
| 如果是 BST？ | 利用大小关系：都比 root 小 → 去左子树；都大 → 去右子树；否则 root 是 LCA。O(h) |
| 如果每个节点有父指针？ | 转换成两个链表求交点（p 向上走到根的路径与 q 的路径交叉点） |
| 如果需要频繁查询多对 LCA？ | Tarjan 离线算法 或 倍增法预处理，单次查询 O(1) |

**BST 版本（优化）**：

```python
def lowestCommonAncestor_BST(root: TreeNode, p: TreeNode, q: TreeNode) -> TreeNode:
    """只在 BST 有效，时间 O(h)"""
    cur = root
    while cur:
        if p.val < cur.val and q.val < cur.val:
            cur = cur.left    # 都比 root 小 → 去左边
        elif p.val > cur.val and q.val > cur.val:
            cur = cur.right   # 都大 → 去右边
        else:
            return cur        # 一个在左一个在右 → 当前就是 LCA
```

### 一、Harness（安全护栏工程）

#### 是什么

Harness 是给 AI Agent 输出加的**多层安全栅栏**。类比：自动驾驶的"安全冗余系统"——不是让车跑得更快，而是保证车**不出事**。

#### 核心原理

```
用户输入
   │
   ▼
┌──────────────┐
│  输入护栏      │  ← 过滤 Prompt Injection、恶意指令
│ (Input Guard) │
└──────┬───────┘
       │ 安全
       ▼
┌──────────────┐
│   LLM 推理    │
└──────┬───────┘
       │
       ▼
┌──────────────┐
│  输出护栏      │  ← 校验 Schema、拦截敏感内容、限制操作范围
│ (Output Guard)│
└──────┬───────┘
       │ 安全
       ▼
    对外输出
```

#### 三层护栏体系

| 层级 | 机制 | 示例 |
|------|------|------|
| **Schema 约束** | JSON Schema / Function Calling 强类型输出 | `response_format={"type":"json_object"}` |
| **规则过滤** | 关键词/正则黑名单、长度限制、格式校验 | 拦截含 `rm -rf` 的 shell 命令 |
| **沙盒执行** | Docker 隔离、文件系统只读、网络白名单 | Codex 的做法：AI 代码在隔离容器中运行 |
| **人工审核** | 高风险操作需人工确认 | 删除数据库记录前弹出确认框 |

#### Guardrails 框架选型

| 框架 | 定位 | 特点 |
|------|------|------|
| **NeMo Guardrails** | 对话安全 | YAML 规则定义，支持对话流控制 |
| **Guardrails-AI** | 结构化校验 | 基于 Pydantic/XML 的校验管道 |
| **LangChain Hub** | 可插拔 | 社区贡献的安全规则库 |
| **自建** | 完全可控 | 正则 + 分类模型 + 人工审核流程 |

#### 适用场景

- **必须用**：Agent 可执行 shell 命令、操作数据库、发送 HTTP 请求
- **推荐用**：对外暴露的 ChatBot、客服 Agent
- **可简化**：内部工具、只读查询 Agent

#### 面试话术

> "Harness 不是单一技术，而是一套多层防御体系。最外层是 Schema 约束——让 LLM 只能输出预定义格式；中间是规则引擎——用正则和关键词拦截危险操作；最内层是沙盒——即使前两层被绕过，代码也跑在隔离环境里。类似银行的金库，不是一道门而是很多道。"

### 二、Skill（能力封装/意图路由）

#### 是什么

Skill 是**工具能力的上层抽象**——把多个底层工具的调用逻辑封装成一个"技能包"，Agent 只需知道"我能做什么"而不用关心"每个工具怎么调"。

#### 核心原理

```
没有 Skill 的 Agent：
  System Prompt 里塞 50 个 tool schema
  → 上下文爆炸、选工具不准、token 浪费

有 Skill 的 Agent：
  System Prompt 里只有 5 个 Skill 描述
  → Router 识别意图 → 加载对应 Skill 的 tool schema → 按需调用
```

#### Skill 的三层结构

```
Skill "订机票"
  ├── 意图描述: "帮助用户搜索和预订机票"
  ├── 触发词: ["买机票", "订航班", "飞北京"]
  ├── 工具链:
  │   ├── search_flights(出发地, 目的地, 日期)
  │   ├── compare_prices(航班列表)
  │   └── book_ticket(航班ID, 乘客信息)
  ├── 执行流程: 搜索 → 比价 → 用户确认 → 下单
  └── 错误处理: 搜索无结果 → 建议改日期 / 无权限 → 引导联系人工客服
```

#### Skill vs MCP vs Function Calling

| 维度 | Function Calling | MCP | Skill |
|------|-----------------|-----|-------|
| **层次** | 单次工具调用 | 工具注册协议 | 工具编排 + 路由 |
| **粒度** | 一个 API | 一组 API 的标准化描述 | 一个业务能力（含多 API + 流程） |
| **谁管理工具选择** | LLM 从全部 tool schema 中选择 | MCP Server 暴露全部工具 | Skill Router 先筛选，LLM 只在子集中选 |
| **可组合性** | 无 | 低（工具描述统一但无编排） | 高（Skill 可以嵌套调用其他 Skill） |
| **类比** | 一个螺丝刀 | 一个工具箱（所有工具摊开） | 一个"修水管"服务（带工具+流程+经验） |

#### 适用场景

| 场景 | 推荐方案 |
|------|---------|
| 3-5 个简单工具 | 直接用 Function Calling |
| 10-30 个工具，多领域 | MCP + Skill 路由 |
| 复杂业务流程（多步串联） | Skill 编排（如 LangGraph 实现） |
| 对外发布给第三方调用 | MCP（标准化协议） |

#### 面试话术

> "Skill 解决的核心问题是**工具爆炸**——当 Agent 有 50 个工具时，全部 function schema 塞进 context 既不经济也不准确。Skill 做两件事：一是意图路由——先判断用户想干什么，只加载对应领域的工具；二是流程编排——把多步工具调用封装成可复用的业务能力。MCP 解决工具的标准化描述问题，Skill 解决工具的智能调度问题，两者互补。"

---

### 三、MCP（Model Context Protocol）

#### 是什么

MCP 是 Anthropic 提出的 **LLM 与外部工具/数据源之间的标准化通信协议**。类比：USB 协议——不管什么设备（鼠标、键盘、U 盘），只要符合 USB 标准就能即插即用。

#### 核心架构

```
┌──────────────────────────────────────────────┐
│                 MCP Host (IDE / App)          │
│  ┌─────────────┐  ┌─────────────┐            │
│  │  MCP Client │  │  MCP Client │  ...       │
│  └──────┬──────┘  └──────┬──────┘            │
└─────────┼───────────────┼────────────────────┘
          │               │
    JSON-RPC 2.0     JSON-RPC 2.0
          │               │
┌─────────▼───────────────▼────────────────────┐
│               MCP Servers                     │
│  ┌──────────┐ ┌──────────┐ ┌──────────────┐  │
│  │Filesystem│ │ Database │ │   Web Search │  │
│  │  Server  │ │  Server  │ │    Server    │  │
│  └──────────┘ └──────────┘ └──────────────┘  │
└──────────────────────────────────────────────┘
```

#### MCP 三大原语

| 原语 | 作用 | 示例 |
|------|------|------|
| **Resources**（资源） | 暴露数据（只读） | 文件内容、数据库记录、API 文档 |
| **Tools**（工具） | 暴露可执行操作（可写） | 创建文件、发送邮件、执行 SQL |
| **Prompts**（提示模板） | 预定义的交互模板 | "帮我分析这段代码"的 prompt 模板 |

#### MCP 的优劣

| 优势 | 劣势 |
|------|------|
| ✅ 标准化——一套协议对接所有工具 | ❌ 工具多了 context 爆炸（所有 tool schema 都要发给 LLM） |
| ✅ 生态——社区 MCP Server 越来越多 | ❌ 每个请求都走 JSON-RPC，延迟较高 |
| ✅ 安全——工具权限由 Server 管理 | ❌ 不支持工具编排——多个工具间的依赖关系需要应用层自己处理 |
| ✅ 多 Client 复用——一个 Server 多个 App 用 | ❌ 协议仍在快速迭代，不稳定 |

#### 适用场景

| 场景 | MCP 合适吗？ |
|------|------------|
| IDE 集成（Cursor 连数据库查 schema） | ✅ 标准场景 |
| 企业内部工具统一接入 | ✅ 减少重复开发 |
| 高频实时调用（每秒 100 次） | ❌ JSON-RPC 开销大 |
| 复杂多步业务流程 | ❌ 需要上层编排 |

### 四、Agent Loop（智能体执行循环）

> 你说的 "loop engineering" 在业内最常指 **Agent Loop / Agentic Loop**——也就是 Agent 的**思考-行动-观察-反思**的循环执行模式。也叫 ReAct Loop、Agent Execution Loop。

#### 核心问题

一个"聪明"的 Agent 和一次简单的 LLM 调用的本质区别是什么？

```
简单 LLM 调用： 用户输入 → LLM 推理 → 输出（一次就完）
Agent Loop：    用户输入 → 思考 → 行动 → 观察结果 → 再思考 → 再行动 → ... → 最终输出
```

Agent Loop 就是把这个"循环往复直到达成目标"的过程**工程化**。

#### 四种核心 Loop 模式

**模式 1：ReAct（Reasoning + Acting）—— 最基础**

```
        ┌──────────────────────────────────┐
        │          ReAct 循环               │
        │                                  │
        │  Thought: "我需要查天气"    ◄──┐  │
        │     │                         │  │
        │     ▼                         │  │
        │  Action: call get_weather()   │  │
        │     │                         │  │
        │     ▼                         │  │
        │  Observation: "北京 35°C"     │  │
        │     │                         │  │
        │     ▼                         │  │
        │  Thought: "不够，还要查空气质量" ─┘  │
        │     │                             │
        │     ▼                             │
        │  Action: call get_aqi()           │
        │     │                             │
        │     ▼                             │
        │  Observation: "AQI 45 优"         │
        │     │                             │
        │     ▼                             │
        │  Answer: "北京今天..."            │
        └──────────────────────────────────┘
```

实现要点：
- 每一步都要把历史 `Thought-Action-Observation` 序列回传给 LLM
- 需要设置 `max_iterations` 防止死循环
- 需要定义"终止条件"（给出 Final Answer 或达到最大步数）

**模式 2：Plan-and-Execute —— 先规划再执行**

```
用户: "帮我规划北京三日游"

Step 1 — Plan:
  LLM 生成计划:
  1. 查北京三日天气
  2. 搜热门景点
  3. 搜特色餐厅
  4. 按日期排列行程
  5. 生成最终攻略

Step 2 — Execute:
  逐条执行计划，结果存入工作记忆
  如果某步失败了 → 重试 / 跳过 / 修改后续计划

Step 3 — Finalize:
  整合所有结果 → 生成最终输出
```

vs ReAct 的区别：Plan-and-Execute 先做**全局规划**再执行，适合步骤多但依赖关系清晰的任务。ReAct 是**边想边做**，适合不确定性高的任务。

**模式 3：Reflexion —— 带自我反思的循环**

```
        尝试 → 失败 → 反思 → 改进 → 重试
                       │
                       ▼
              "上次失败的原因是检索词太宽泛，
               下次应该用更具体的 API 名称"
```

Reflexion 在标准 Loop 基础上加了一个**反思步骤**：每次执行完成后，用 LLM 分析为什么成功/失败，把教训存入长期记忆，下次类似任务时自动规避。

**模式 4：Self-Refine —— 迭代自我优化**

```
初稿 → 自我审视 → 发现问题 → 修改 → 再审视 → ... → 满意输出
```

适用场景：代码生成、文案撰写、翻译——凡是"可以迭代优化"的输出。

#### Loop Engineering 的工程要点

| 要点 | 说明 |
|------|------|
| **步数限制** | `max_iterations` 防止无限循环，超限后走降级策略 |
| **终止判断** | LLM 输出 `FINISH` 标记 或 达到预设条件 |
| **上下文压缩** | Loop 越长 context 越长，需要定期压缩历史 Observation |
| **并行执行** | 相互独立的 Action 可以并行调用（如同时查天气和酒店） |
| **错误恢复** | Tool 调用失败 → 重试(N 次) → 降级 → 人工介入 |
| **状态管理** | 维护结构化的工作记忆（已完成/进行中/待处理） |
| **成本控制** | 每次迭代 = 一次 LLM 调用，复杂任务可能有 10+ 次调用 |

#### 面试话术

> "Agent Loop 本质上是把单次 LLM 推理升级为**推理-行动-观察**的闭环。最基础的是 ReAct 模式——每步思考需要什么信息、调用对应工具、观察结果、再决定下一步。复杂场景下可以叠加 Plan-and-Execute（先做全局规划）和 Reflexion（失败后自我反思改进）。工程上最需要注意的是步数限制和上下文管理——Loop 越长，context 越长，token 消耗和延迟都呈线性增长。"

#### ⚠️ 重要更新：从 ReAct 到工程化 Agent Loop（2025-2026 演进）

> 上面讲的 ReAct / Plan-Execute / Reflexion / Self-Refine 四种模式是 **概念层**（2022 论文），回答的是"Agent 怎么思考和行动"。但 2025-2026 年业内真正火的 "Agent Loop" 是 **工程层**——把这些模式做成生产级框架。两者不是同一个东西，面试时要区分清楚，别把 ReAct 当成最新概念讲。

**演进时间线**：

| 时间 | 事件 | 意义 |
|------|------|------|
| 2022.10 | ReAct 论文（Yao et al.） | 理论起源：Thought-Action-Observation 循环 |
| 2024.12 | Anthropic《Building Effective Agents》 | 明确区分 **workflow vs agent**，Agent 的本质就是 loop |
| 2025 下半年 | Claude Agent SDK（前身 Claude Code SDK） | 把 Agent Loop **工程化**，提供 Turn / 上下文压缩 / Subagent |
| 2025 | OpenAI Agents SDK | `Runner.run()` + Handoff + Guardrails |
| 2025-2026 | Google Cloud agentic AI 架构指南 | 同样指向 Agent Loop |

**核心转变**：从"ReAct 是一种 prompting 技巧" → "Agent Loop 是一种工程架构"。ReAct 没有被淘汰，而是被**装进了工程框架**里，成为框架内部的一种行为模式。

**工程化 Agent Loop 的关键新概念**（笔记原内容缺失的部分）：

| 新概念 | 说明 | 所属 SDK |
|--------|------|----------|
| **Turn** | 一次完整的"LLM 输出 → 工具执行 → 结果反馈"循环，多个 Turn 串联成整个 Agent 执行 | Claude Agent SDK |
| **自动上下文压缩** | 上下文接近窗口上限时，SDK 自动把旧消息压缩成摘要，保留近期关键信息 | Claude Agent SDK |
| **Subagent**（独立上下文） | 子 Agent 拥有独立上下文窗口，只把最终结果返回主 Agent，不污染主上下文 | Claude Agent SDK |
| **Handoff**（上下文传递） | 一个 Agent 把控制权连同上下文一起交给另一个 Agent，适合流水线协作 | OpenAI Agents SDK |
| **Guardrails** | Loop 入口/出口的安全护栏，输入侧拦敏感信息，输出侧过滤不合规结果 | OpenAI Agents SDK |
| **Hooks** | Loop 中的检查点（PreToolUse/PostToolUse 等），在应用进程运行不消耗 token | Claude Agent SDK |
| **max_turns + max_budget_usd** | 双重成本控制，超限有明确 subtype（error_max_turns / error_max_budget_usd） | Claude Agent SDK |

**新老对比**（面试关键）：

| 维度 | 笔记原内容（老，概念层） | 2025-2026 Agent Loop（新，工程层） |
|------|---------------------------|------------------------------------|
| **本质** | Prompting 技巧（Thought-Action-Observation 文本交替） | 工程架构（消息流 + 工具系统 + 上下文管理） |
| **抽象单位** | "Thought / Action / Observation" 三元组 | **Turn**（LLM 输出 + 工具执行 + 结果反馈） |
| **上下文管理** | 只提"需要压缩历史" | **自动压缩** + Subagent 独立上下文 + effort 级别 |
| **多 Agent** | 没讲 | **Subagent**（独立上下文）vs **Handoff**（上下文传递）两种路线 |
| **安全** | 归到 Harness 章节 | **Guardrails**（入/出口护栏）+ **Hooks**（Loop 内检查点）+ Permission Mode |
| **成本控制** | 只说"每次迭代一次 LLM 调用" | **max_turns + max_budget_usd** 双限制，超限有 subtype |
| **错误恢复** | "重试 N 次" | 工具错误作为 UserMessage 反馈给 LLM 自行决策 |
| **何时用** | 没讲 | Anthropic 明确：**能预先定义步骤的用 workflow，不能的才用 Agent Loop** |

**Anthropic 的 workflow vs agent 判断标准**（面试高频，必背）：

> "大多数场景下，简单的 workflow 比复杂的 agentic loop 更合适。Agent Loop 适用于那些**无法预先定义完整执行路径**的任务——需要根据中间结果动态调整策略的场景。对于可以明确步骤的流程（数据管道、固定审批流、模板化内容生成），使用确定性 workflow 更稳定、更可预测。"
>
> 一句话：**workflow 是"你告诉系统怎么做"，Agent Loop 是"你告诉系统做什么，让它自己决定怎么做"。**

**更新的面试话术**（在原话术基础上加工程层）：

> "Agent Loop 的理论起源是 2022 年的 ReAct 论文，核心是 Thought-Action-Observation 循环。但 2025 年之后，Claude Agent SDK、OpenAI Agents SDK 把它工程化了——引入了 Turn 机制、自动上下文压缩、Subagent/Handoff 多 Agent 协作、Guardrails/Hooks 安全护栏、max_turns + max_budget_usd 成本控制。现在讲 Agent Loop 不应只停留在 ReAct 模式，还要讲工程化。另外 Anthropic 强调：能预先定义步骤的用 workflow，不能的才用 Agent Loop——这个判断标准比 loop 本身更重要。"

### 四者关系总结

```
                    ┌──────────────────────────────┐
                    │       Agent Loop               │
                    │  (ReAct → Plan-Execute → ...)  │
                    │  "Agent 怎么思考和行动"          │
                    └──────────┬───────────────────┘
                               │ 调用
              ┌────────────────┼────────────────┐
              ▼                ▼                ▼
       ┌──────────┐    ┌──────────┐     ┌──────────┐
       │  Skill   │    │   MCP    │     │ Harness  │
       │ "能做什么"│    │ "怎么对接"│     │ "安全边界"│
       └──────────┘    └──────────┘     └──────────┘
        工具编排+路由    工具标准化协议    多层安全护栏
```

- **Agent Loop**：定义 Agent 的**行为模式**（怎么思考、怎么行动、怎么迭代）
- **Skill**：定义 Agent 的**能力边界**（能做什么，怎么做，用什么工具）
- **MCP**：定义 Agent 和工具之间的**通信协议**（怎么描述工具、怎么调用）
- **Harness**：定义 Agent 的**安全边界**（什么能做、什么不能做、出错了怎么办）
