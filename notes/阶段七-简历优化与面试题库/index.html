    二、高频面试题（分模块，含参考答案）

    2.1 Linux & 日志类

    Q1: 程序被 OOM 杀死如何排查？

    参考答案：

    1. 先确认是否被 OOM Killer 杀掉：dmesg | grep -i "killed process" 或 dmesg -T | grep -i oom，会显示被杀的进程名和原因。
    2. 看进程内存占用：ps aux --sort=-%mem | head，找出占内存最高的进程。
    3. 看系统内存：free -h，确认是否物理内存不足或 swap 没配。
    4. 看应用日志：journalctl -u <service> --since "1 hour ago"，看是否有内存泄漏迹象。
    5. 解决：如果是大模型显存 OOM，用量化降显存或换更大显卡；如果是内存泄漏，加监控告警+定期重启。

    Q2: 如何快速抓取日志内的报错信息？

    参考答案：

    用文本三剑客组合：

    # 最基础
grep "ERROR" app.log

# 提取关键字段
cat app.log | grep ERROR | awk '{print $1, $2, $5}'

# 统计各错误出现次数
grep "ERROR" app.log | awk '{print $5}' | sort | uniq -c | sort -rn

# 按时间段过滤
awk '/2026-06-29 10:00/,/2026-06-29 11:00/' app.log | grep ERROR

    如果是 systemd 服务，用 journalctl -u <service> -p err。

    2.2 Docker & nvidia-docker2 类

    Q1: nvidia-docker2 和普通 Docker 的区别，容器如何调用宿主机 GPU？

    参考答案：

    普通 Docker 只能隔离 CPU、内存、磁盘资源，容器内看不到 GPU。nvidia-docker2（新版叫 nvidia-container-toolkit）通过在容器启动时把宿主机的 GPU 设备文件（/dev/nvidia*）和驱动库注入容器，让容器内的程序能调用 GPU。

    具体机制：

    • 它是一个 Docker runtime hook，在 docker run --gpus all 时触发
    • 把 /dev/nvidia0、/dev/nvidiactl 等设备文件挂载进容器
    • 把 libcuda.so 等驱动库挂载进容器
    • 容器内的 CUDA 程序就能像在宿主机上一样调用 GPU

    验证命令：docker run --rm --gpus all nvidia/cuda:12.1-base nvidia-smi，容器内能看到 GPU 信息就说明配置成功。

    2.3 K8s-GPU 类

    Q1: GPU-Operator 的作用是什么？K8s 如何给 Pod 分配指定 GPU？

    参考答案：

    GPU-Operator 是 NVIDIA 提供的 K8s 组件，作用是让 K8s 集群能识别和管理 GPU 硬件。它做了几件事：

    1. 自动在 GPU 节点上安装 NVIDIA 驱动、CUDA、container runtime
    2. 注册 nvidia.com/gpu 为 K8s 的扩展资源
    3. 运行 DCGM Exporter 采集 GPU 指标

    给 Pod 分配 GPU 的方式是在 YAML 里声明：

    resources:
  limits:
    nvidia.com/gpu: 1    # 申请 1 张 GPU

    K8s 调度器会把这个 Pod 调度到有可用 GPU 的节点上。GPU-Operator 底层的 device plugin 负责把具体的 GPU 设备分配给容器。

    2.4 CUDA 驱动类

    Q1: nvidia-smi 和 nvcc -V 版本不一致是否会影响模型运行？驱动与 CUDA 的兼容规则？

    参考答案：

    不一致是正常的，不一定影响运行。两者的区别：

    • nvidia-smi 显示的是驱动支持的最高 CUDA 版本
    • nvcc -V 显示的是实际安装的 CUDA Toolkit 版本

    兼容规则：驱动向下兼容 CUDA，即实际安装的 CUDA 版本 ≤ 驱动支持的最高版本即可。例如驱动显示 CUDA 12.2，实际安装 CUDA 12.1 是可以的；但如果实际装了 12.3 就会报错。

    影响模型运行的情况：实际 CUDA 版本 > 驱动支持的最高版本，或 PyTorch 要求的 CUDA 版本和实际安装的不匹配。

    2.5 vLLM 核心问题（必考题）

    Q1: vLLM 的 PagedAttention 原理，为什么比 TGI 显存利用率更高？

    参考答案：

    PagedAttention 借鉴了操作系统的虚拟内存分页机制来管理 KV Cache。

    传统推理的 KV Cache 按连续内存分配，不同请求长度不同，导致大量内存碎片，显存利用率通常只有 60% 左右。

    PagedAttention 把 KV Cache 分成固定大小的 block（如 16 token 一块），按需分配、不连续存储。类似操作系统的页表，通过 block table 映射逻辑位置到物理位置。这样显存利用率可以提升到 90%+，同时支持更大的 batch size，吞吐量比 TGI 高 2-4 倍。

    Q2: AWQ 和 GPTQ 量化的区别，分别适用什么场景？

    参考答案：

    维度AWQGPTQ
    原理基于激活值感知的权重量化基于二阶信息的逐层量化
    量化速度快慢（逐层计算 Hessian）
    推理速度快中
    精度损失小小

    AWQ 适用场景：生产部署首选，vLLM 原生支持，量化快、推理快、精度好。

    GPTQ 适用场景：研究对比、需要极致压缩率时。量化过程慢但压缩率略高。

    实际工程中 AWQ 是主流选择，因为综合性能更好。

    2.6 项目深挖题

    Q1: vLLM 服务突然宕机，你的完整排查流程是什么？

    参考答案：

    按顺序排查：

    1. 看服务状态：systemctl status vllm，确认是否崩溃
    2. 看日志：journalctl -u vllm --since "10 min ago"，找 ERROR/OOM/CUDA 报错
    3. 看 GPU：nvidia-smi，确认 GPU 是否还在、显存是否被占满
    4. 看系统资源：free -h（内存）、df -h（磁盘）、dmesg | grep killed（OOM Killer）
    5. 看端口：ss -tlnp | grep 8000，确认端口是否被占
    6. 看依赖：模型文件是否还在、CUDA 是否正常
    7. 恢复：systemd 的 Restart=always 会自动重启；如果显存被占满，先 nvidia-smi 找残留进程 kill 掉再重启

    Q2: 大模型显存溢出，你有哪些优化手段？

    参考答案：

    从易到难排列：

    1. 量化：AWQ/GPTQ 4-bit 量化，显存降 60%+
    2. 减小 max-model-len：限制最大上下文长度
    3. 减小 batch size：降低并发请求数
    4. KV Cache 量化：压缩 KV Cache
    5. Tensor Parallel：多卡切分模型
    6. CPU offload：把部分权重卸载到内存（牺牲速度）
    7. 换更大显存的显卡：A10 24G → A100 40G/80G
